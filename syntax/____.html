1.3 解析语法

本节介绍了 syntax-parse, 即 syntax/parse 库中用于解析语法的工具。 syntax-parse 和规范化工具–syntax classes, 都使用一种通用的 syntax patterns 语言,这在 语法模式 一书中有详细描述。

我们提供了两种解析形式: syntax-parse 和 syntax-parser 。

syntax

(syntax-parse stx-expr parse-option ... clause ...+)

 
parse-option = #:context context-expr
  | #:literals (literal ...)
  | #:datum-literals (datum-literal ...)
  | #:literal-sets (literal-set ...)
  | #:track-literals
  | #:conventions (convention-id ...)
  | #:local-conventions (convention-rule ...)
  | #:disable-colon-notation
     
literal = literal-id
  | (pattern-id literal-id)
  | (pattern-id literal-id #:phase phase-expr)
     
datum-literal = literal-id
  | (pattern-id literal-id)
     
literal-set = literal-set-id
  | (literal-set-id literal-set-option ...)
     
literal-set-option = #:at context-id
  | #:phase phase-expr
     
clause = (syntax-pattern pattern-directive ... body ...+)
 
  stx-expr : syntax?
  context-expr : 
(or/c syntax? symbol? #f
      (list/c symbol? syntax?))
  phase-expr : (or/c exact-integer? #f)
评估 stx-expr, 它应该产生一个语法对象,并按顺序与 clause 匹配。 如果某个子句的模式匹配,其属性将被绑定到语法对象的相应子项上,该子句的边际条件和 expr 被评估。 其结果是 expr 的结果。

每个子句由一个 syntax pattern 、 一个可选的 pattern directives 序列和一个非空的主体形式序列组成。

如果语法对象不能匹配任何一个模式(或者所有的匹配都不能满足相应子句的侧面条件),就会产生一个语法错误。

支持以下选项:

#:context context-expr
 
  context-expr : 
(or/c syntax? symbol? #f
      (list/c symbol? syntax?))

当存在时,在报告解析失败时使用 context-expr ;否则使用 stx-expr 。 如果 context-expr 求值为 (list who context-stx), 那么 who 作为引发错误的形式出现在错误信息中,而 context-stx 则作为术语使用。 如果 context-expr 评估为一个符号,那么它被用作 who , 而 stx-expr (要去结构化的语法)被用作 context-stx 。 如果 context-expr 被评估为一个语法对象,它被用作 context-stx , 而 who 被推断为 raise-syntax-error 。

current-syntax-context 参数也被设置为语法对象 context-stx 。

Examples:
> (syntax-parse #'(a b 3)
    [(x:id ...) 'ok])

a: expected identifier

  at: 3

  in: (a b 3)

> (syntax-parse #'(a b 3)
    #:context #'(lambda (a b 3) (+ a b))
    [(x:id ...) 'ok])

lambda: expected identifier

  at: 3

  in: (lambda (a b 3) (+ a b))

> (syntax-parse #'(a b 3)
    #:context 'check-id-list
    [(x:id ...) 'ok])

check-id-list: expected identifier

  at: 3

  in: (a b 3)

#:literals (literal ...)
 
literal = literal-id
  | (pattern-id literal-id)
  | (pattern-id literal-id #:phase phase-expr)
 
  phase-expr : (or/c exact-integer? #f)

与 syntax-case 不同, syntax-parse 要求所有字词都有一个绑定。 要通过符号名称来匹配标识符,请使用 #:datum-literals 或 ~datum 模式来代替。 #:literals 选项指定了应被视为 literals 而非 pattern variables 的标识符。 在 literals 列表中的一个条目有两个组成部分: 在模式中用来表示要匹配的位置的标识符 (pattern-id), 以及预期出现在这些位置的标识符 (literal-id)。 如果条目是一个单一的标识符,该标识符将用于这两个目的。

如果给了 #:phase 选项,那么就在 phase-expr 阶段对字面进行比较。 具体来说,在 phase-expr 的 literal-id 的绑定必须与输入在 phase-expr 的绑定一致。

换句话说, syntax-pattern 被解释为每个出现的 pattern-id 都被替换成以下模式:

(~literal literal-id #:phase phase-expr)

#:datum-literals (datum-literal ...)
 
datum-literal = literal-id
  | (pattern-id literal-id)

就像 #:literals 一样,但是 literals 是作为符号而不是作为标识符来匹配的。

换句话说, syntax-pattern 被解释为每个出现的 pattern-id 都被替换成了下面的模式:

(~datum literal-id)

#:literal-sets (literal-set ...)
 
literal-set = literal-set-id
  | (literal-set-id literal-set-option ...)
     
literal-set-option = #:at lctx
  | #:phase phase-expr
 
  phase-expr : (or/c exact-integer? #f)

许多字词可以通过一个或多个 literal sets 一次性声明,用 #:literal-sets 选项导入。 更多信息请参见 literal sets 。

如果给出 #:at 关键字, lctx 术语的词法上下文将被用来确定模式中的哪些标识符被视为字面意义; 这个选项主要对生成 syntax-parse 表达式的宏有用。

#:track-literals

如果指定了这个选项,每个最终的 body 表达式都被进一步限制为产生一个单一的值, 这个值必须是一个 syntax object, 它的 'disappeared-use syntax property 被自动扩展到包括作为模式匹配一部分的字面意义。 字面符号会从 #:literals, #:literal-sets 或 ~literal 的使用中自动追踪, 但也可以使用 syntax-parse-state-cons! 手动追踪。 该属性的添加或扩展方式与通过 syntax-parse-track-literals 添加的属性相同。

由于 body 形式的包装方式, 指定这个选项意味着最终的 body 形式相对于包围的 syntax-parse 形式将不再处于尾部位置。

Added in version 6.90.0.29 of package base.

#:conventions (conventions-id ...)

输入 convention ,为没有明确指定语法类的模式变量提供默认的语法类。

#:local-conventions (convention-rule ...)

使用指定的 conventions 。 与 #:conventions 相比, #:local-conventions 的优势在于, local conventions 可以在语法类参数绑定的范围内。例子见 conventions 部分。

#:disable-colon-notation

禁止为注释的模式变量使用 "冒号符号"。

Examples:
> (syntax-parse #'(a b c)
    [(x:y ...) 'ok])

syntax-parse: not defined as syntax class

  at: y

  in: (syntax-parse (syntax (a b c)) ((x:y ...) (quote ok)))

> (syntax-parse #'(a b c) #:disable-colon-notation
    [(x:y ...) 'ok])

'ok

syntax

(syntax-parser parse-option ... clause ...+)

与 syntax-parse 类似,但产生一个匹配过程。该过程接受一个参数,该参数应该是一个语法对象。

syntax

(define/syntax-parse syntax-pattern pattern-directive ... stx-expr)

 
  stx-expr : syntax?
syntax-parse 的定义形式。 也就是说,它将 stx-expr 的语法对象结果与 syntax-pattern 匹配, 并为 syntax-pattern 的属性创建模式变量定义。

Examples:
> (define/syntax-parse ((~seq kw:keyword arg:expr) ...)
    #'(#:a 1 #:b 2 #:c 3))
> #'(kw ...)

#<syntax:eval:7:0 (#:a #:b #:c)>

与 define/with-syntax 相比,它是一种类似的定义形式, 使用更简单的 syntax-case 模式。
