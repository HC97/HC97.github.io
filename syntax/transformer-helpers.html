5 宏转化器帮助程序

5.1 提取推理名称

 (require syntax/name) package: base

procedure

(syntax-local-infer-name stx [use-local?]) → any/c

  stx : syntax?
  use-local? : any/c = #t
与 syntax-local-name 类似, 只是 stx 会被检查出一个 'inferred-name 属性(它覆盖了任何推断出的名字)。 如果 syntax-local-name 和 'inferred-name 都没有产生一个名字, 或者 'inferred-name 属性的值是 #<void>, 那么就从 stx 中的源位置信息中构造一个名字,如果有的话。 如果不能构建名称,结果是 #f 。

为了支持在扩展过程中传播和合并一致的属性, 'inferred-name 属性的值可以是一个用 cons 形成的树,其中所有的叶子都是相同的。 例如, (cons 'name 'name) 等同于 'name, 而 (cons (void) (void)) 等同于 #<void> 。

如果 use-local? 是 #f, 那么就不会使用 syntax-local-name 。 提供 use-local? 为 #f ,以构建一个语法对象的名称,该名称不是当前正在展开的表达式。

5.2 支持 local-expand

 (require syntax/context) package: base

procedure

(build-expand-context v) → list?

  v : (or/c symbol? list?)
返回一个适合作为内部定义上下文的 local-expand 的上下文参数的列表。 v 参数代表用于扩展的直接上下文。 如果上下文列表是一个列表,那么它将建立在 (syntax-local-context) 之上。

procedure

(generate-expand-context [liberal-definitions?]) → list?

  liberal-definitions? : boolean? = #f
用一个生成的唯一值调用 build-expand-context 。 当 liberal-definitions? 为真时,该值是一个结构类型的实例, 其 prop:liberal-define-context 属性的值为真。

5.3 解析类似 define 的表单

 (require syntax/define) package: base

procedure

(normalize-definition defn-stx    
  lambda-id-stx    
  [check-context?    
  opt+kws?]) → 
identifier? syntax?
  defn-stx : syntax?
  lambda-id-stx : identifier?
  check-context? : boolean? = #t
  opt+kws? : boolean? = #f
接受一个定义形式,其形状类似于 define (尽管可能有不同的名字),并返回两个值: 被定义的标识符和右手边的表达式。

为了生成右边的表达式,这个函数可能需要插入 lambda 的使用。 lambda-id-stx 参数提供了一个合适的 lambda 标识符。

如果定义不符合要求,就会产生一个语法错误。 如果 check-context? 为真, 那么如果 (syntax-local-context) 表明当前上下文是一个表达式上下文,就会引发语法错误。 check-context? 为 #t 。

如果 opt+kws? 是 #t, 那么允许使用形式为 [id expr], keyword id 和 keyword [id expr] 的参数, 并且它们在扩展中被保留下来。

procedure

(normalize-definition/mk-rhs defn-stx 
  lambda-id-stx 
  check-context? 
  opt+kws? 
  err-no-body?) 
 → 
identifier? (-> syntax? syntax?) syntax?
  defn-stx : syntax?
  lambda-id-stx : identifier?
  check-context? : boolean?
  opt+kws? : boolean?
  err-no-body? : boolean?
normalize-definition 的助手,产生三个值: 被定义的标识符,一个接收主体语法并产生具有预期绑定结构的语法的函数, 最后是 normalize-definition 给前一个函数的右侧表达。

如果 err-no-body? 为真,那么必须有一个右手边的表达式,否则就是语法错误。 err-no-body? 参数对于 normalize-definition 的使用是真的。

Added in version 6.1.1.8 of package base.

5.4 扁平化 begin 形式

 (require syntax/flatten-begin) package: base

procedure

(flatten-begin stx) → (listof syntax?)

  stx : syntax?
从一个类似于 begin 的表格中提取子表达式,如果 stx 没有正确的形状(即语法列表),则报告错误。 由此产生的语法对象有来自 stx 的注释,使用 syntax-track-origin 。

Examples:
> (flatten-begin #'(begin 1 2 3))

'(#<syntax:eval:2:0 1> #<syntax:eval:2:0 2> #<syntax:eval:2:0 3>)

> (flatten-begin #'(begin (begin 1 2) 3))

'(#<syntax:eval:3:0 (begin 1 2)> #<syntax:eval:3:0 3>)

> (flatten-begin #'(+ (- 1 2) 3))

'(#<syntax:eval:4:0 (- 1 2)> #<syntax:eval:4:0 3>)

procedure

(flatten-all-begins stx) → (listof syntax?)

  stx : syntax?
从一个 begin 形式中提取子表达式, 并递归地平移嵌套在原形式中的 begin 形式。 如果 stx 不是一个 begin 形式,则会报告错误。 产生的语法对象的注释是使用 syntax-track-origin 从 stx 中转移过来的。

Examples:
> (flatten-all-begins #'(begin 1 2 3))

'(#<syntax:eval:5:0 1> #<syntax:eval:5:0 2> #<syntax:eval:5:0 3>)

> (flatten-all-begins #'(begin (begin 1 2) 3))

'(#<syntax:eval:6:0 1> #<syntax:eval:6:0 2> #<syntax:eval:6:0 3>)

Added in version 6.1.0.3 of package base.

5.5 展开类似 define-struct 的表单

 (require syntax/struct) package: base

procedure

(parse-define-struct stx orig-stx) → 
identifier?
(or/c identifier? false/c)
(listof identifier?)
syntax?
  stx : syntax?
  orig-stx : syntax?
将 stx 解析为 define-struct 形式, 但使用 orig-stx 来报告语法错误 (假设 orig-stx 与 stx 相同,或者它们至少共享子形式)。 结果是四个值:结构类型名称的标识符、超级名称的标识符或 #f 、字段的标识符列表, 以及检查器表达式的语法对象。

procedure

(build-struct-names name-id 
  field-ids 
  [#:constructor-name ctr-name] 
  omit-sel? 
  omit-set? 
  [src-stx]) 
 → (listof identifier?)
  name-id : identifier?
  field-ids : (listof identifier?)
  ctr-name : (or/c identifier? #f) = #f
  omit-sel? : boolean?
  omit-set? : boolean?
  src-stx : (or/c syntax? false/c) = #f
生成由 define-struct 约束的名称,给定结构类型名称的标识符和字段名称的标识符列表。 其结果是一个标识符的列表:

  • struct:name-id

  • ctr-name 或 make-name-id 如果 ctr-name 是 #f

  • name-id?

  • name-id-field, 对于 field-ids 中的每个 field 。

  • set-name-id-field! (getter和setter的名字交替出现)。

  • ....

如果 omit-sel? 为真,那么选择器名称将从结果列表中省略。 如果 omit-set? 为真,那么结果列表中就会省略 setter 的名字。

默认的 src-stx 是 #f; 它被用来为生成的标识符提供一个源位置。

procedure

(build-struct-generation name-id 
  field-ids 
  [#:constructor-name ctr-name] 
  omit-sel? 
  omit-set? 
  [super-type 
  prop-value-list 
  immutable-k-list]) 
 → (listof identifier?)
  name-id : identifier?
  field-ids : (listof identifier?)
  ctr-name : (or/c identifier? #f) = #f
  omit-sel? : boolean?
  omit-set? : boolean?
  super-type : any/c = #f
  prop-value-list : list? = '(list)
  immutable-k-list : list? = '(list)
接受与 build-struct-names 相同的参数, 并为使用 make-struct-type 的代码生成S表达式, 以便为 build-struct-names 创建的标识符生成结构类型和返回值。 可选的 super-type,prop-value-list 和 immutable-k-list 参数采取 S 表达式, 作为 make-struct-type 的相应参数表达式。

procedure

(build-struct-generation* all-name-ids 
  name-id 
  field-ids 
  [#:constructor-name ctr-name] 
  omit-sel? 
  omit-set? 
  [super-type 
  prop-value-list 
  immutable-k-list]) 
 → (listof identifier?)
  all-name-ids : (listof identifier?)
  name-id : identifier?
  field-ids : (listof identifier?)
  ctr-name : (or/c identifier? #f) = #f
  omit-sel? : boolean?
  omit-set? : boolean?
  super-type : any/c = #f
  prop-value-list : list? = '(list)
  immutable-k-list : list? = '(list)
与 build-struct-generation 类似, 但是给定由 build-struct-names 产生的名称,而不是重新生成它们。

procedure

(build-struct-expand-info name-id    
  field-ids    
  [#:omit-constructor? no-ctr?    
  #:constructor-name ctr-name    
  #:omit-struct-type? no-type?]    
  omit-sel?    
  omit-set?    
  base-name    
  base-getters    
  base-setters) → any
  name-id : identifier?
  field-ids : (listof identifier?)
  no-ctr? : any/c = #f
  ctr-name : (or/c identifier? #f) = #f
  no-type? : any/c = #f
  omit-sel? : boolean?
  omit-set? : boolean?
  base-name : (or/c identifier? boolean?)
  base-getters : (listof (or/c identifier? false/c))
  base-setters : (listof (or/c identifier? false/c))
接受与 build-struct-names 基本相同的参数,加上一个父类型的 identifier/#t/#f 和一个访问器和突变器标识符列表(可能以 #f 结尾), 并为扩展时代码生成一个 S 表达式,用于结构名称的绑定中。

如果 no-ctr? 为真,那么构造函数名称将从扩展时信息中省略。 同样地,如果 no-type? 为真,那么结构类型名称将被省略。

base-name 的 #t 示没有超级类型, #f 表示超级类型(如果有的话)是未知的,而标识符表示超级类型的标识符。

procedure

(struct-declaration-info? v) → boolean?

  v : any/c
如果 x 具有结构类型声明的扩展时间信息的形状,返回 #t ,否则返回 #f 。 参见 结构类型转化器绑定 。

procedure

(generate-struct-declaration orig-stx    
  name-id    
  super-id-or-false    
  field-id-list    
  current-context    
  make-make-struct-type    
  [omit-sel?    
  omit-set?]) → syntax?
  orig-stx : syntax?
  name-id : identifier?
  super-id-or-false : (or/c identifier? false/c)
  field-id-list : (listof identifier?)
  current-context : any/c
  make-make-struct-type : procedure?
  omit-sel? : boolean? = #f
  omit-set? : boolean? = #f
这个过程实现了一个 define-struct 扩展的核心。

generate-struct-declaration 过程被宏扩展器调用以生成扩展, 其中 name-id,super-id-or-false 和 field-id-list 参数提供了主要参数。 current-context 参数通常是 syntax-local-context 的结果。 orig-stx 参数用于处理语法错误。 可选的 omit-sel? 和 omit-set? 参数默认为 #f; #t 值会分别抑制字段选择器或变体的定义。

make-struct-type 过程是为了生成表达式以实际创建结构类型。 它的参数是 orig-stx, name-id-stx, defined-name-stxes 和 super-info 。 前两个参数与最初提供给 generate-struct-declaration 的参数一样, 第三个参数是由 build-struct-names 生成的名称集, 最后一个参数是通过解析 super-id-or-false (当它不是 #f, #f )获得的超级结构信息。

其结果应该是一个表达式,其值与 make-struct-type 的结果相同。 因此,下面是一个基本的 make-make-struct-type:

(lambda (orig-stx name-stx defined-name-stxes super-info)
  #`(make-struct-type '#,name-stx
                       #,(and super-info (list-ref super-info 0))
                       #,(/ (- (length defined-name-stxes) 3) 2)
                       0 #f))

但是一个实际的 make-make-struct-type 可能会做得更多。

5.7 控制语法模板

 (require syntax/template) package: base

procedure

(transform-template template-stx 
  #:save save-proc 
  #:restore-stx restore-proc-stx 
  [#:leaf-save leaf-save-proc 
  #:leaf-restore-stx leaf-restore-proc-stx 
  #:leaf-datum-stx leaf-datum-proc-stx 
  #:pvar-save pvar-save-proc 
  #:pvar-restore-stx pvar-restore-stx 
  #:cons-stx cons-proc-stx 
  #:ellipses-end-stx ellipses-end-stx 
  #:constant-as-leaf? constant-as-leaf?]) 
 → syntax?
  template-stx : syntax?
  save-proc : (syntax? . -> . any/c)
  restore-proc-stx : syntax?
  leaf-save-proc : (syntax? . -> . any/c) = save-proc
  leaf-restore-proc-stx : syntax? = #'(lambda (data stx) stx)
  leaf-datum-proc-stx : syntax? = #'(lambda (v) v)
  pvar-save-proc : (identifier? . -> . any/c) = (lambda (x) #f)
  pvar-restore-stx : syntax? = #'(lambda (d stx) stx)
  cons-proc-stx : syntax? = #'cons
  ellipses-end-stx : syntax? = #'values
  constant-as-leaf? : boolean? = #f
产生一个类似于 #`(syntax #,template-stx)的表达式, 但是像 save-proc 这样的函数可以收集可能被 syntax 丢失的信息 (比如当语法对象在字节码内被调集时的属性), 而像 restore-proc-stx 指定的运行时函数可以使用保存的信息或以其他方式处理由模板生成的语法对象。

save-proc 被应用于原始模板表示中的每个语法对象(即在 template-stx 中)。 如果 constant-as-leaf? 是 #t, 那么 save-proc 只应用于在子表单中至少包含一个模式变量的语法对象。 save-proc 的结果作为 restore-proc-stx 的第一个参数被提供回来, 它表示一个带有契约的函数 (-> any/c syntax any/c any/c); restore-proc-stx 的第二个参数是 syntax 生成的语法对象, 最后一个参数是已经被递归处理(由 restore-proc-stx 等函数)的数据, 通常会使用第二个参数的上下文、源和属性转换回一个语法对象。 请注意, save-proc 在扩展时工作(关于模板形式), 而 restore-proc-stx 表示在运行时调用的函数(对于模板形式), 从 save-proc 流向 restore-proc-stx 的数据通过 quote 跨阶段。

leaf-save-proc 和 leaf-restore-proc-stx 程序与 save-proc 和 restore-proc-stx 类似,但它们是应用于叶子的,所以对于递归处理的子表单没有第三个参数。 leaf-restore-proc-stx 所指示的函数应该具有契约 (-> any/c syntax? any/c) 。

leaf-datum-proc-stx 程序适用于不是语法对象的叶子,这种情况可能发生, 因为对和空列表并不总是被单独包装成语法对象。 该函数应该有契约 (-> any/c any/c) 。 当 constant-as-leaf? 为 #f 时,程序的唯一可能参数是 null 。

pvar-save 和 pvar-restore-stx 过程与 save-proc 和 restore-proc-stx 类似,但它们是应用于模式变量。 pvar-restore-stx 过程应该具有契约 (-> any/c syntax? any/c), 其中第二个参数对应于模式变量的替换。

cons-proc-stx 过程用于建立中间对, 包括传递给 restore-proc-stx 的对和不对应于语法对象的对。

ellipses-end-stx 过程是语法对象上的一个额外的过滤器, 它跟随模板中的 ... 省略号序列。 该过程应该具有契约 (-> any/c any/c) 。

下面的例子说明了使用 transform-template 来实现一个 syntax/shape 的形式, 它保留了原始模板中的 'paren-shape 属性,即使模板代码被编入字节码中。

(define-for-syntax (get-shape-prop stx)
  (syntax-property stx 'paren-shape))
 
(define (add-shape-prop v stx datum)
  (syntax-property (datum->syntax stx datum stx stx stx)
                   'paren-shape
                   v))
 
(define-syntax (syntax/shape stx)
  (syntax-case stx ()
    [(_ tmpl)
     (transform-template #'tmpl
                         #:save get-shape-prop
                         #:restore-stx #'add-shape-prop)]))

5.8 创建宏转化器

 (require syntax/transformer) package: base

procedure

(make-variable-like-transformer reference-stx 
  [setter-stx]) 
 → (and/c set!-transformer? (-> syntax? syntax?))
  reference-stx : (or/c syntax? (-> identifier? syntax?))
  setter-stx : (or/c syntax? (-> syntax? syntax?) #f) = #f
创建一个转化器,用 reference-stx 替换对宏标识符的引用。 在操作者位置上使用宏被解释为一个应用, reference-stx 为函数,参数为给定值。 如果 reference-stx 是一个过程,它将被应用到宏标识符上。

如果宏标识符被用作 set! 形式的目标,那么如果 setter-stx 是语法的话, set! 形式将扩展为 setter-stx 在 set! 表达式右侧的应用; 否则,标识符将被视为不可改变的,并且会出现语法错误。 如果 setter-stx 是一个过程,它将被应用于整个 set! 表达式。

Examples:
> (define the-box (box add1))
> (define-syntax op
    (make-variable-like-transformer
     #'(unbox the-box)
     #'(lambda (v) (set-box! the-box v))))
> (op 5)

6

> (set! op 0)
> op

0

Added in version 6.3 of package base.

procedure

(make-expression-transformer transformer)

 → (-> syntax? syntax?)
  transformer : (-> syntax? syntax?)
创建一个从 transformer 派生的转化器,确保它在表达式上下文中展开。 当在表达式上下文中调用时,它调用 transformer 。 当在任何其他上下文中调用时,新的转换函数用 #%expression 。

Added in version 7.7.0.9 of package base.

仅用于向后兼容; syntax-local-apply-transformer 是首选。

将 transformer 作为 syntax transformer 应用于当前扩展环境中的 stx 。 其结果类似于用 local-expand 将 identifier 绑定为 syntax transformer 绑定到 transformer 上进行扩展, 只是在应用一个宏转换后,扩展被保证停止(假设 transformer 没有明确强制进一步递归扩展)。

与简单地将 transformer 直接应用于 stx 不同, 使用 local-apply-transformer 引入了适当的 use-site scope 和 macro-introduction scope ,这些范围将由扩展器添加。

context 和 intdef-ctx 参数的处理方式与 local-expand 的相应参数相同。

Added in version 6.90.0.29 of package base.
