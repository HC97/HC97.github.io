interface

gl-context<%> : interface?

gl-context<%> 对象表示用 OpenGL 绘制特定 dc<%> 实例的背景。 要获得 gl-context<%> 对象,请调用目标绘图上下文的 get-gl-context 。

只有 dc<%> 和包含来自 make-gl-bitmap 的 bitmap-dc% 对象支持 OpenGL ( 在 Windows 和 Mac OS 上总是如此,有时在 Unix 上)。 普通 dc<%> 绘图和 OpenGL 绘图可以在 bitmap-dc% 中混合使用, 但是使用 'gl 风格来支持 OpenGL 的画布不能可靠地支持普通 dc<%> 绘图; 如果你需要混合绘图模式,请使用位图,并使用画布来最大化 OpenGL 性能。

当 bitmap-dc% 上下文的目标位图通过 set-bitmap 改变时, 相关的 gl-context<%> 也会改变。 画布上下文通常是双缓冲的,而位图上下文是单缓冲的。

racket/gui/base 库没有提供 OpenGL 例程。 相反,它们必须从一个单独的库中获得,如 sgl 。 racket/gui/base 中的设施只是管理当前的 OpenGL 上下文,将其连接到窗口和位图。

在所有线程和事件空间中,一次只能有一个 OpenGL 上下文处于活动状态。 OpenGL 上下文不受线程间干扰的保护;也就是说,如果一个线程选择了它的一个 OpenGL 上下文,那么其他线程可以通过 OpenGL 命令写入该上下文。 然而,如果所有线程都只在传递给 call-as-current 的 thunk 中发出 OpenGL 命令, 那么来自不同线程的绘图就不会受到干扰, 因为 call-as-current 使用锁来序列化 Racket 中所有线程的上下文选择。

method

(send a-gl-context call-as-current thunk    
  [alternate    
  enable-breaks?]) → any/c
  thunk : (-> any)
  alternate : evt? = never-evt
  enable-breaks? : any/c = #f
以这个 OpenGL 上下文作为 OpenGL 命令的当前上下文调用一个 thunk。

该方法阻塞以获得一个保护全局 OpenGL 上下文的锁,并在 thunk 返回或逃脱时释放该锁。 该锁是可重入的,因此在同一线程中以相同的 OpenGL 上下文嵌套使用该方法,不会获得或释放该锁。

该锁可以防止使用 OpenGL 的线程之间的干扰。 如果一个线程在持有上下文锁的情况下被终止,该锁将被释放。 继续跳转到 thunk 中不会获取锁或设置 OpenGL 上下文。 参见 gl-context<%> 获取更多关于干扰的信息。

该方法接受一个备用的 synchronizable event ,以便在阻塞上下文锁的时候使用; 也请参见 sync 。

方法调用的结果是 thunk 的结果(如果它被调用了),或者是备用事件的结果(如果它被选择代替上下文锁)。

如果在调用此方法时 ok? 返回 #f ， an exn:fail:contract exception is raised 。

如果 enable-breaks? 为 true,那么该方法将使用 sync/enable-break , 同时阻塞上下文设置的锁,而不是 sync 。

method

(send a-gl-context get-handle) → cpointer?

返回一个指向平台底层上下文的句柄。 指针所代表的值取决于平台:

  • Windows: HGLRC

  • Mac OS: NSOpenGLContext

  • Unix: GLXContext

请注意,这些值不一定是最 "低级" 的上下文对象,而是对 Racket 有用的对象。 例如,一个 NSOpenGLContext 包装了一个 CGLContextObj 。

method

(send a-gl-context ok?) → boolean?

如果这个上下文是可用的 OpenGL 绘图,则返回 #t ,否则返回 #f 。

如果 OpenGL 支持在编译时或运行时被禁用,如果该上下文与一个没有选定位图的 bitmap-dc% 或与一个单色选定位图相关联,如果该上下文是为一个不再存在的画布准备的, 或者如果在准备该上下文时有一个低级错误,则该上下文不可用。

method

(send a-gl-context swap-buffers) → void?

对于与画布相关的上下文,交换前面(可见)和后面( OpenGL 绘制)的缓冲区,对位图上下文没有影响。

该方法隐含地使用 call-as-current 来获得上下文锁。 然而,由于锁是可重入的, swap-buffers 方法可以在 call-as-current thunk 中安全使用。

procedure

(get-current-gl-context) → gl-context<%>

如果在调用 call-as-current 方法的动态范围内,返回当前上下文;否则返回 #f 。 这可能对缓存与上下文相关的状态或数据最有用,比如扩展字符串。 使用 make-weak-hasheq 创建此类缓存。

Added in version 1.3 of package draw-lib.
