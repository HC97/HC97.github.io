5 Scribbling 文档

本节介绍了Racket文档写作的良好风格。

5.1 散文和术语

在 defform, defproc 在 “This ...” 开头。 取而代之的是,以一个隐含的主题是被描述的形式或值的句子开始(但只在第一句中这样开始)。 将第一个词大写。因此,描述往往以 "返回" 或 "产生" 开始。用名字来指代参数和子表格。

不要用 "参数" 这个词来描述语法形式中的子形式;用 "子形式" 这个词来代替,将 "参数" 保留给函数调用中的值或表达。 提及库和语言,而不是 "模块"(尽管排版库或语言名称的形式被称为 racketmodname)。 不要把一个标识符(即一个语法元素)称为 "变量" 或 "符号"。 不要用 "表达式" 这个词来表示一个定义或可能是一个定义的形式;用 "形式" 这个词来代替。 最好使用 "函数" 而不是 "程序"。

只有当你指的是由空列表和 cons 单元组成的运行时值时,才使用 "列表" 这个词;如果你必须使用任何一个词,在其他情况下使用 "序列"。 例如,不要写 begin 有一个 "子表格的列表";相反,它有一个 "子表格的序列"。 同样地,不要在函数调用中提到 "参数列表";如果可能的话,就写 "参数",或者写 "参数表达式的序列"。 (不幸的是,“sequence” 也获得了特定的运行时含义, 但这种碰撞没有 Lisp 中列表和其他实体之间的历史混淆那么严重)。

避免对描述性文本进行剪切和粘贴。 如果两个函数是相似的,可以考虑用 deftogether 来记录它们。 要对描述进行抽象,可以考虑使用明确的散文抽象,比如 “x is like y, except that ...,” , 而不是对源头进行抽象并多次实例化;通常,散文抽象比文档实现中的隐藏抽象对读者来说更清晰。

将 "子表单" 和 "子表达式" 连在一起。

使用 "Windows"、"Mac OS" 和 "Unix" 来表示 Racket 所运行的三个 "平台"(而不是 "系统")。 使用 "Unix" 作为 Mac OS 以外的类 Unix 操作系统–特别是包括 Linux–的通用术语。 即使在 "Gtk" 或 "X11 窗口系统" 更准确的情况下,也要使用 "Unix",但在必要时使用 "X11" 作为形容词,如 "X11 显示"。 Racket 运行在一个平台上,而不是 "在" 一个平台下。

避免将谓词用作名词,代表满足谓词的值。 相反,使用 tech 和 deftech 来建立描述价值类别的英语单词或短语与谓词(或 contract)之间的联系。 例如,避免使用 “supply a path-string?”; 最好使用 “supply a path or string.”

5.2 排版代码

在 defform 中使用 id 或以 -id 结尾的名称,表示标识符, 而不是 identifier, variable, name 或 symbol 。 同样地,在语法形式中使用 expr 或以 -expr 结尾的东西表示表达式的位置。 对于内部定义位置的形式(定义或表达)使用 body , 在语法描述中总是以 ...+ 结尾。 不要对不完全是表达式的东西使用 expr ,对不完全是标识符的东西使用 id ,等等; 相反,使用 defform/subs 来定义一个新的非终端。

小心使用 deftogether 来定义一个语法形式或过程的多个变体, 因为每个 defform 或 defproc 都会创建一个定义点,但每个形式或过程应该有一个定义点。 (当一个绑定有多个定义点时,Scribble 会发出一个警告)。 相反,使用 defproc* 或 defform* 。

对于函数参数,使用 v 作为 “any value” 的元变量。 只对数值使用 x 作为元变量。 其他约定包括 lst 代表列表, proc 代表过程。

在使用 racket 时要注意标识符和元变量之间的区别, 特别是在 defproc 或 defform 之外。 在元变量前面加上 _; 例如、

  @racket[(rator-expr rand-expr ...)]

是指一个函数调用的语法的错误方式,因为它产生了 (rator-expr rand-expr ...), 其中 rator-expr 和 rand-expr 被排成了变量。 正确的描述是

  @racket[(_rator-expr _rand-expr ...)]

它产生 (rator-expr rand-expr ...), 其中 rator-expr 和 rand-expr 被类型化为元变量。 defproc, defform 等形式大大减少了描述中的这种负担, 因为它们为非文字标识符自动设置了元变量排版。 在 defform 中, 一定要在 #:literals 子句中包括字面标识符(即那些不是作为变量的标识符,而只是被定义的表单名称)。

要对一个没有特殊解释的标识符进行排版–语法、变量、元变量等等–使用 racketidfont (例如,如上面的 rand-expr )。 否则,使用 litchar, 而不仅仅是 racketfont 或 verbatim, 来指代一个特定的字符序列。

当一个语法形式从一个给定的标识符合成一个标识符时, 使用 racketidfont 和 racket 的组合来描述标识符。 例如,如果 id 与 is- 和 ? 组合成 is-id?, 那么将该标识符实现为 @racketidfont{is-}@racket[id]@racketidfont{?} 。

当使用 defform 来描述一个语法形式时,不要将 #:contracts 子句与语法规范混淆。 只对句法形式中的表达式使用 #:contracts ,而且合同是一个运行时约束–而不是句法约束, 比如要求子形式是一个标识符。对句法约束使用 defform/subs 。

当显示示例求值时,使用 REPL-snapshot 风格:

  @examples[

   (+ 1 2)

  ]

也请参见 scribble/example 库和 例子 。

使用四个点, ...., 以代替省略的代码,因为 ... 意味着重复。

5.3 散文排版

不要提及 "上面" 或 "下面" 的文档,而要用超链接指向正确的地方。

在散文中,使用 `` 和 '' 引号而不是 " 。 使用 --- 来代替 em 破折号,并且不要在两边加上空格。 引号和引号末尾的标点符号使用美式风格(即句子末尾的句号放在引号内)。 当然,这一规则不适用于作为代码一部分的引号。

不要将引文参考(如 cite 创建的)作为名词使用;将其作为注释使用。

不要用 Racket 变量名来开始一个句子,因为它通常是小写的。 例如,使用 “The thing argument is...” 而不是 “thing is...”

当 “etc.” 没有结束一个句子时,使用 etc ,并在 “etc.” 后面加上一个逗号, 除非它结束一个后面有其他标点的句子(如括号)。

5.4 章节标题

除冠词 (“the,” “a,” 等)、介词和不在标题开头的连接词外,其他所有单词均为大写。

手册的标题通常应该以一个合适的关键词或关键短语开始(如本手册的 "Scribble"),并以黑体字显示。 如果关键词主要是一个可执行的名称,则使用 exec 而不是 bold 。 可以选择在标题中的冒号后添加进一步的描述性文字,其中以冒号开始的文字不使用黑体。

5.5 索引

文档和章节的标题,用 defproc, defform 等记录的标识符被自动编入索引, 用 deftech 定义的术语也是如此。

符号不会被自动索引。 使用 indexed-racket 而不是 racket 来表示一个符号的实例,它大致定义了用途。 举个例子,尝试搜索 “truncate” 来找到 'truncate ,它与 open-output-file 一起使用。 不要使用类似 (index "'truncate") 的东西来索引一个符号,因为它不会正确排版(即用固定宽度的字体和字面的颜色)。

使用 index, as-index 和 section-index 作为最后手段。 为那些与其他索引的术语完全不同的术语创建索引条目。 不要为了改善搜索结果而试图对一个术语或短语的微小变化进行索引; 如果由于微小的变化而导致搜索失败,那么应该解决搜索算法的问题,而不是索引条目。

5.6 例子

努力在每一个功能和语法形式的文档中包含例子(使用 examples) 。 在写例子时,不要使用 “foo” 和 “bar” 这样的废话。例如,在记录 member 的时候,不要写成

> (member "foo" '("bar" "foo" "baz"))

'("foo" "baz")

而要写成如下内容

> (member "Groucho" '("Harpo" "Groucho" "Zeppo"))

'("Groucho" "Zeppo")
