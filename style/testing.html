2 测试

2.1 测试套件

我们大多数的集合都有测试套件。 这些测试套件通常存在于 PLT 资源库中的 collects/tests/ 中,尽管由于历史原因,有一些集合带有他们自己的本地测试套件。 如果你添加了一个新的集合,在 tests 集合中创建一个新的测试套件。

在你提交之前运行测试套件。 为了方便测试,我们敦促你在你的集合中添加一个 TESTME.txt 文件。 理想情况下,你可能也希望在这个目录下有一个运行基本测试的文件。 参见 2htdp, 它是有自己测试风格的集合之一。 该文件应该描述测试的位置,如何运行这些测试,以及在成功和失败方面应该注意什么。 这些文件是必要的,因为不同的集合对测试有不同的需求,而且测试在我们的历史上有许多不同的演变方式。

在你提交之后,注意并阅读(!) DrDr 的电子邮件。 不要 忽视它们。 如果你有已知失败的测试,并且修复这个问题需要大量的工作,考虑将你的测试目录分成两部分: success 和 failure 。 前者是现在应该成功的测试,后者是目前预计会失败的测试。 请看 Typed Racket testing arrangement 中的例子。 当你创建这种 failure 测试时,你可能希望像这样禁用 DrDr 的检查:

  git prop set drdr:command-line "" <file> ...

这是一个 Racket 特有的 git 命令。

2.2 经常测试!

当你调试一段现有的代码时,先制定一个测试案例。 把它放到组件的测试套件中,这样错误就不会被意外地重新引入,并添加一个指向问题报告的注释。 第二,修改代码以解决这个错误。 第二,确定你没有在你的测试中引入一个错误;很容易认为你已经修复了一个错误,而实际上你的新测试只是没有正确揭示旧的错误。 第三,重新运行测试套件,以确保错误被修复,没有现有的测试失败。

如果没有测试套件,而且你不确定如何建立一个,那么可以在开发者邮件列表中询问。 也许人们会解释为什么没有测试套件,或者他们会勾勒出如何创建一个测试套件。 请不要忽视这个问题。如果你不能建立一个测试套件,你有几个选择:

  1. 向库中添加功能以实现测试。当然,增加功能意味着增加外部文档。 Robby 和 Matthew 已经为 GUI 库做到了这一点,现在有一个大型的 DrRacket 自动测试套件。 所以即使是 GUI 程序也可以附带扩展的测试套件。

  2. 增加一个可能要由人验证的端到端测试。 例如,可能很难测试 Slideshow,所以你可以创建一个幻灯片集,并描述它应该是什么样子, 以便未来的维护者在 they 进行修改时进行验证。 不过,考虑到这是 last and least desirable 选项。

一些集合缺乏测试的情况不会在一夜之间消失。 但是,如果我们都能做出一点贡献,我们最终会扩大测试套件,以覆盖整个代码库,而未来的维护者将对此表示感谢。
