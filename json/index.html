JSON

Eli Barzilay
and Dave Herman


吴华均 译

 (require json) package: base

这个库提供了解析和产生 JSON 数据交换格式的数据到/从 Racket 值的实用程序。 关于 JSON 的更多信息请参见 JSON web site 和 JSON RFC 。

1 JS 表达式

procedure

(jsexpr? x [#:null jsnull]) → boolean?

  x : any/c
  jsnull : any/c = (json-null)
执行深度检查以确定 x 是否是一个 jsexpr 。

这个库定义了一个可以表示为 JSON 字符串的 Racket 值子集,这个谓词可以检查这些值。 一个 JS-Expression, 或称 jsexpr, 是以下的一种:

  • jsnull 的值,默认为 'null , 使用 eq? 可以识别

  • boolean?

  • string?

  • (or/c exact-integer? (and/c inexact-real? rational?))

  • (listof jsexpr?)

  • (hash/c symbol? jsexpr?)

Examples:
> (jsexpr? 'null)

#t

> (jsexpr? #t)

#t

> (jsexpr? "cheesecake")

#t

> (jsexpr? 3.5)

#t

> (jsexpr? (list 18 'null #f))

#t

> (jsexpr? #hasheq((turnip . 82)))

#t

> (jsexpr? (vector 1 2 3 4))

#f

> (jsexpr? #hasheq(("turnip" . 82)))

#f

> (jsexpr? +inf.0)

#f

parameter

(json-null) → any/c

(json-null jsnull) → void?
  jsnull : any/c
这个参数决定了对应于 JSON “null” 的默认 Racket 值。 默认情况下,它是 'null 符号。 在某些情况下,不同的值可能更适合你的需要,因此本库中的所有函数都接受一个 #:null 关键字参数, 用于表示 JSON “null”,这个参数默认为 (json-null) 。

注意 (json-null) 的值(或一个明确提供的 #:null 参数)可以用 eq? 来识别。

2 从 JS 表达式生成 JSON 文本

procedure

(write-json x    
  [out    
  #:null jsnull    
  #:encode encode    
  #:indent indent]) → void?
  x : jsexpr?
  out : output-port? = (current-output-port)
  jsnull : any/c = (json-null)
  encode : (or/c 'control 'all) = 'control
  indent : (or/c #f #\tab natural-number/c) = #f
将编码为 JSON 的 x jsexpr 写到 out 输出端口。

默认情况下,只有 ASCII 控制字符被编码为 “\uHHHH”。 如果 encode 为 'all, 那么除了 ASCII 控制字符外,非 ASCII 字符也会被编码。 如果你需要通过可能不支持 UTF-8 的通道传输文本,这可能很有用。 请注意, U+10000 及以上范围的字符被编码为两个 \uHHHH 转义,见 JSON RFC 的第 2.5 节。

如果提供了 indent,并且不是 #f,那么每个数组元素或者对象的键值对都会被写入新的一行, 并且 indent 的值指定了每一层嵌套所要添加的空白:可以是一个 #\tab 字符, 或者,如果 indent 是一个数字,那么是相应数量的 #\space 字符。

Examples:
> (with-output-to-string
    (λ () (write-json #hasheq((waffle . (1 2 3))))))

"{\"waffle\":[1,2,3]}"

> (with-output-to-string
    (λ () (write-json #hasheq((와플 . (1 2 3)))
                      #:encode 'all)))

"{\"\\uc640\\ud50c\":[1,2,3]}"

> (for ([indent (in-list '(#f 0 4 #\tab))])
  (newline)
  (write-json #hasheq((waffle . (1 2 3)) (와플 . (1 2 3)))
              #:indent indent)
  (newline))

application: procedure does not expect an argument with

given keyword

  procedure: write-json

  given keyword: #:indent

  arguments...:

   '#hasheq((waffle . (1 2 3)) (와플 . (1 2 3)))

   #:indent #f

procedure

(jsexpr->string x    
  [#:null jsnull    
  #:encode encode    
  #:indent indent]) → string?
  x : jsexpr?
  jsnull : any/c = (json-null)
  encode : (or/c 'control 'all) = 'control
  indent : (or/c #f #\tab natural-number/c) = #f
为 jsexpr x 生成一个 JSON 源字符串。

Example:
> (jsexpr->string #hasheq((waffle . (1 2 3))))

"{\"waffle\":[1,2,3]}"

procedure

(jsexpr->bytes x    
  [#:null jsnull    
  #:encode encode    
  #:indent indent]) → bytes?
  x : jsexpr?
  jsnull : any/c = (json-null)
  encode : (or/c 'control 'all) = 'control
  indent : (or/c #f #\tab natural-number/c) = #f
为 jsexpr x 生成一个 JSON 源字节串。 (字节串是用 UTF-8 编码的)。

Example:
> (jsexpr->bytes #hasheq((waffle . (1 2 3))))

#"{\"waffle\":[1,2,3]}"

3 将 JSON 文本解析为 JS 表达式

procedure

(read-json [in #:null jsnull]) → (or/c jsexpr? eof-object?)

  in : input-port? = (current-input-port)
  jsnull : any/c = (json-null)
从一个 JSON 编码的输入端口 in 中读取一个 jsexpr 作为 Racket(不可变)值,如果只剩下空白, 则产生 eof 。 和 read 一样,该函数在端口中留下了所有剩余的字符,以便第二次调用可以检索到剩余的 JSON 输入。 如果 JSON 输入本身没有划线(true, false, null),它们必须与下面的 JSON 输入用空格隔开。 如果 in 不在 EOF 处,并且以畸形的 JSON 开始(即 in 中没有初始字节序列可以被解析为 JSON), 则引发 exn:fail:read ;例子见下文。

Examples:
> (with-input-from-string
    "{\"arr\" : [1, 2, 3, 4]}"
    (λ () (read-json)))

'#hasheq((arr . (1 2 3 4)))

> (with-input-from-string
    "\"sandwich\""
    (λ () (read-json)))

"sandwich"

> (with-input-from-string
    "true false"
    (λ () (list (read-json) (read-json))))

'(#t #f)

> (with-input-from-string
    "true[1,2,3]"
    (λ () (list (read-json) (read-json))))

'(#t (1 2 3))

> (with-input-from-string
    "true\"hello\""
    (λ () (list (read-json) (read-json))))

'(#t "hello")

> (with-input-from-string
    "\"world\"41"
    (λ () (list (read-json) (read-json))))

'("world" 41)

> (with-input-from-string
    "sandwich sandwich" ; invalid JSON
    (λ () (read-json)))

string::1: read-json: bad input starting #"sandwich

sandwich"

> (with-input-from-string
    "false sandwich" ; valid JSON prefix, invalid remainder is not (immediately) problematic
    (λ () (read-json)))

#f

> (with-input-from-string
    "false42" ; invalid JSON text sequence
    (λ () (read-json)))

string::6: read-json: bad input starting #"false42"

> (with-input-from-string
    "false 42" ; valid JSON text sequence (notice the space)
    (λ () (list (read-json) (read-json))))

'(#f 42)

Changed in version 8.1.0.2 of package base: 调整了对空白的处理,拒绝不是 #\space, #\tab, #\newline 或 #\return 。

procedure

(string->jsexpr str [#:null jsnull]) → jsexpr?

  str : string?
  jsnull : any/c = (json-null)
将字符串 str 的可识别前缀解析为一个不可变的 jsexpr 。 如果前缀本身没有划线(true、false、null),那么它必须用空格与其余的字符隔开。 如果字符串是畸形的 JSON,会引发 exn:fail:read 。

Example:
> (string->jsexpr "{\"pancake\" : 5, \"waffle\" : 7}")

'#hasheq((pancake . 5) (waffle . 7))

procedure

(bytes->jsexpr str [#:null jsnull]) → jsexpr?

  str : bytes?
  jsnull : any/c = (json-null)
将字符串 str 的可识别前缀解析为一个不可变的 jsexpr 。 如果前缀本身没有分隔(true、false、null),它必须用空格与其余字节分隔。 如果字节串是畸形的 JSON,会引发 exn:fail:read 。

Example:
> (bytes->jsexpr #"{\"pancake\" : 5, \"waffle\" : 7}")

'#hasheq((pancake . 5) (waffle . 7))

4 关于设计的说明

4.1 JS 表达式数据类型

JSON 在语法上区分了 “null”、数组字面和对象字面, 因此有一个问题,即 Racket 值应该代表一个 JSON 的 “null”。 本库默认使用 Racket 的 'null 符号。 请注意,这是毫不含糊的,因为 Racket 符号只被用作对象键,而对象键在 JSON 中需要是字符串。

其他几个选项已经被不同的库所使用。 例如,Dave Herman 的 PLaneT 库(它一直是本库的基础)使用 #\nul 字符, Racket 和其他 Lisps 的库使用 (void), NIL NIL(有些也用于 JSON 的 “false”),等等。 本库采取的方法是对所有函数使用一个关键字参数, 由一个参数决定其默认值,这样就可以很容易地使用任何符合你需要的值。

JSON RFC 只指出对象字面表达式 "应该" 包含唯一的键,但并没有完全禁止它们。 从现有的实践来看,流行的 JSON 库在解析带有重复键的对象字面时, 只需从键值对中挑选一个,并丢弃具有相同键的其他键。 这种行为自然与 Racket 的哈希表相仿,使其成为自然的类似物。

最后, JSON RFC 对于键值对的顺序几乎完全没有提及。 虽然 RFC 只规定了 JSON 的语法,当然,JSON 必须将对象字词表示为一个有序的集合,但介绍中指出:

一个对象是零个或多个名/值对的无序集合,其中名是一个字符串,值是一个字符串、数字、布尔值、空、对象或数组。

在实践中,JSON 库在解析 JSON 文本时抛弃了对象字词的顺序,对生成的对象字词的顺序不做任何保证, 通常使用某种风味的哈希表作为自然选择。因此我们也是这样做的。

4.2 命名规则

本库中有些名字使用 "jsexpr",有些使用 "json"。 其理由是前者用于我们的表述,而后者则是作为从外界接收或发送到外界的信息。
