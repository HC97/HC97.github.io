4.1 Equality

平等性是指两个值是否 "相同" 的概念。Racket 默认支持几种不同的平等性, 尽管在大多数情况下 equal? 是首选。

procedure

(equal? v1 v2) → boolean?

  v1 : any/c
  v2 : any/c
两个值当且仅当它们是 eqv? 时才是 equal? ,除非对特定的数据类型另有规定。

对 equal? 有进一步说明的数据类型包括字符串、 字节字符串、对、可变对、向量、盒子、散列表和可检查结构。 在后六种情况下,平等是递归定义的;如果 v1 和 v2 都包含引用循环, 那么当值的无限展开将是平等的,它们就是平等的。 另见 gen:equal+hash 和 prop:impersonator-of 。

Examples:
> (equal? 'yes 'yes)

#t

> (equal? 'yes 'no)

#f

> (equal? (* 6 7) 42)

#t

> (equal? (expt 2 100) (expt 2 100))

#t

> (equal? 2 2.0)

#f

> (let ([v (mcons 1 2)]) (equal? v v))

#t

> (equal? (mcons 1 2) (mcons 1 2))

#t

> (equal? (integer->char 955) (integer->char 955))

#t

> (equal? (make-string 3 #\z) (make-string 3 #\z))

#t

> (equal? #t #t)

#t

procedure

(equal-always? v1 v2) → boolean?

  v1 : any/c
  v2 : any/c
表示 v1 和 v2 是否相等,并将永远保持相等,不受 mutations 的影响。 一般来说,要使两个值永远相等, v1 和 v2 中对应的不可变的值必须是 equal? , 而它们中对应的可变的值必须是 eq?。 这个操作符在其他语言中的先例包括 egal [Baker93] 。

当且仅当存在第三个值 v3 ,使得 v1 和 v2 都是 v3 的陪衬, 即 (chaperone-of? v1 v3) 和 (chaperone-of? v2 v3) 都是真的时候, 两个值 v1 和 v2 才是 equal-always? 。

对于不包括陪护者或其他冒名顶替者的值,如果 v1 和 v2 是 equal?, 只是 v1 和 v2 中对应的可变向量、盒子、哈希表、字符串、字节串、 mutable pairs 和可变结构必须是 eq?, 结构上的平等可以通过 gen:equal-mode+hash 专门用于 equal-always? 。

Examples:
> (equal-always? 'yes 'yes)

#t

> (equal-always? 'yes 'no)

#f

> (equal-always? (* 6 7) 42)

#t

> (equal-always? (expt 2 100) (expt 2 100))

#t

> (equal-always? 2 2.0)

#f

> (equal-always? (list 1 2) (list 1 2))

#t

> (let ([v (mcons 1 2)]) (equal-always? v v))

#t

> (equal-always? (mcons 1 2) (mcons 1 2))

#f

> (equal-always? (integer->char 955) (integer->char 955))

#t

> (equal-always? (make-string 3 #\z) (make-string 3 #\z))

#f

> (equal-always? (string->immutable-string (make-string 3 #\z))
                 (string->immutable-string (make-string 3 #\z)))

#t

> (equal-always? #t #t)

#t

Added in version 8.5.0.3 of package base.

procedure

(eqv? v1 v2) → boolean?

  v1 : any/c
  v2 : any/c
当且仅当两个值是 eq? 时,它们就是 eqv? ,除非对特定数据类型另有规定。

number 和 character 数据类型是唯一一个 eqv? 与 eq? 不同的类型。 当两个数字具有相同的精确性,精度,并且都是相等的非零,都是 +0.0, 都是 +0.0f0, 都是 -0.0, 都是 -0.0f0, 都是 +nan.0, 或者都是 +nan.f—在 complex numbers 的情况下分别考虑实部和虚部时, 这两个数字就是 eqv? 。 当两个字符 char->integer 结果相等时,它们就是 eqv? 。

一般来说, eqv? 与 equal? 相同, 只是前者不能递归比较复合数据类型(如列表和结构)的内容,也不能由用户定义的数据类型来定制。 我们不鼓励使用 eqv? ,而是使用 equal? 。

Examples:
> (eqv? 'yes 'yes)

#t

> (eqv? 'yes 'no)

#f

> (eqv? (* 6 7) 42)

#t

> (eqv? (expt 2 100) (expt 2 100))

#t

> (eqv? 2 2.0)

#f

> (let ([v (mcons 1 2)]) (eqv? v v))

#t

> (eqv? (mcons 1 2) (mcons 1 2))

#f

> (eqv? (integer->char 955) (integer->char 955))

#t

> (eqv? (make-string 3 #\z) (make-string 3 #\z))

#f

> (eqv? #t #t)

#t

procedure

(eq? v1 v2) → boolean?

  v1 : any/c
  v2 : any/c
如果 v1 和 v2 指的是同一个对象,返回 #t ,否则返回 #f 。 作为 numbers 中的一个特例,根据 eq? ,两个 = 的 fixnums 也是相同的。 参见 对象身份和比较。

Examples:
> (eq? 'yes 'yes)

#t

> (eq? 'yes 'no)

#f

> (eq? (* 6 7) 42)

#t

> (eq? (expt 2 100) (expt 2 100))

#f

> (eq? 2 2.0)

#f

> (let ([v (mcons 1 2)]) (eq? v v))

#t

> (eq? (mcons 1 2) (mcons 1 2))

#f

> (eq? (integer->char 955) (integer->char 955))

#t

> (eq? (make-string 3 #\z) (make-string 3 #\z))

#f

> (eq? #t #t)

#t

procedure

(equal?/recur v1 v2 recur-proc) → boolean?

  v1 : any/c
  v2 : any/c
  recur-proc : (any/c any/c -> any/c)
就像 equal?, 但使用 recur-proc 进行递归比较(这意味着不会自动处理引用循环)。 来自 recur-proc 的非 #f 结果在被 equal?/recur 返回之前被转换为 #t 。

Examples:
> (equal?/recur 1 1 (lambda (a b) #f))

#t

> (equal?/recur '(1) '(1) (lambda (a b) #f))

#f

> (equal?/recur '#(1 1 1) '#(1 1.2 3/4)
                (lambda (a b) (<= (abs (- a b)) 0.25)))

#t

procedure

(equal-always?/recur v1 v2 recur-proc) → boolean?

  v1 : any/c
  v2 : any/c
  recur-proc : (any/c any/c -> any/c)
就像 equal-always?,但使用 recur-proc 进行递归比较(这意味着不会自动处理引用循环)。 来自 recur-proc 的非 #f 结果在被 equal-always?/recur 返回之前被转换为 #t 。

Examples:
> (equal-always?/recur 1 1 (lambda (a b) #f))

#t

> (equal-always?/recur '(1) '(1) (lambda (a b) #f))

#f

> (equal-always?/recur (vector-immutable 1 1 1) (vector-immutable 1 1.2 3/4)
                       (lambda (a b) (<= (abs (- a b)) 0.25)))

#t

4.1.1 对象身份和比较

eq? 操作符比较两个 values ,当这两个值指向同一个 object 时,返回 #t 。 这种形式的平等适用于比较支持强制更新的对象(例如,确定通过一个引用修改一个对象的效果通过另一个引用是可见的)。 另外,一个 eq? 测试可以快速评估, 而且基于 eq? 散列比散列表中基于 equal? 的散列更轻量。

然而,在某些情况下, eq? 不适合作为比较运算符,因为 objects 的产生没有明确的定义。 特别是,对相同的两个精确整数应用 + , 可能会产生也可能不会产生 eq? 的结果,尽管结果总是 equal? 。 同样,对一个 lambda 形式的评估通常会生成一个新的过程 object, 但是它可能会重新使用之前由同一个源 lambda 形式生成的过程 object 。

一个数据类型相对于 eq? 的行为通常是由数据类型及其相关程序来指定的。

4.1.2 Equality and Hashing

所有可比较的值都至少有一个 hash code — 一个任意的整数 (更确切地说是一个 fixnum),通过对该值应用一个哈希函数来计算。 这些哈希代码的定义属性是, 相等的值有相等的哈希代码 。 请注意,反之亦然:两个不相等的值仍然可以有相等的哈希码。 哈希码对于各种索引和比较操作非常有用,特别是在实现 hash tables 的时候。 更多信息请参见 Hash Tables .

procedure

(equal-hash-code v) → fixnum?

  v : any/c
返回一个与 equal? 一致的 hash code 。 对于任何两个具有 equal? 值的调用,返回的数字是相同的。 即使 v 包含一个通过对、向量、盒子和/或可检查结构域的循环,也会计算出一个哈希码。 此外,用户定义的数据类型可以通过实现 gen:equal+hash 或 gen:equal-mode+hash 来定制这个哈希代码的计算方式。

对于任何可能由 read 产生的 v ,如果 v2 由 read 产生, 且输入字符相同,则 (equal-hash-code v) 与 (equal-hash-code v2) 相同– 即使 v 和 v2 不同时存在(因此不能通过调用 equal? 进行比较)。

Changed in version 6.4.0.12 of package base: 加强了对 read 的保证。

procedure

(equal-hash-code/recur v recur-proc) → fixnum?

  v : any/c
  recur-proc : (-> any/c exact-integer?)
就像 equal-hash-code,但在 v 中使用 recur-proc 进行递归散列。

Examples:
> (define (rational-hash x)
    (cond
      [(rational? x) (equal-hash-code (inexact->exact x))]
      [else (equal-hash-code/recur x rational-hash)]))
> (= (rational-hash 0.0) (rational-hash -0.0))

#t

> (= (rational-hash 1.0) (rational-hash -1.0))

#f

> (= (rational-hash (list (list (list 4.0 0.0) 9.0) 6.0))
     (rational-hash (list (list (list 4 0) 9) 6)))

#t

Added in version 8.8.0.9 of package base.

procedure

(equal-secondary-hash-code v) → fixnum?

  v : any/c
与 equal-hash-code 类似,但计算一个适合用于双重散列的二级 hash code 。

procedure

(equal-always-hash-code v) → fixnum?

  v : any/c
返回一个与 equal-always? 一致的 hash code 。 对于任何两个具有 equal-always? 值的调用,返回的数字是相同的。

当 equal-always-hash-code 遍历 v 时, v 中不可变的值用 equal-hash-code 散列, 而 v 中可变的值则用 eq-hash-code 散列。

procedure

(equal-always-hash-code/recur v recur-proc) → fixnum?

  v : any/c
  recur-proc : (-> any/c exact-integer?)
就像 equal-always-hash-code,但使用 recur-proc 来实现 递归散列在 v 中。

Added in version 8.8.0.9 of package base.

procedure

(equal-always-secondary-hash-code v) → fixnum?

  v : any/c
与 equal-always-hash-code 类似,但计算一个适合用于双重散列的二级 hash code 。

procedure

(eq-hash-code v) → fixnum?

  v : any/c
返回一个与 eq? 一致的 hash code 。 对于任何两个具有 eq? 值的调用,返回的数字是相同的。

相等的 fixnums 总是 eq? 。

procedure

(eqv-hash-code v) → fixnum?

  v : any/c
返回一个与 eqv? 一致的 hash code 。 对于任何两个具有 eqv? 值的调用,返回的数字是相同的。

4.1.3 Implementing Equality for Custom Types

value

gen:equal+hash : any/c

一个 generic interface (参见 通用接口) , 用于可以使用 equal? 进行平等比较的类型。 以下方法必须被实现:

  • equal-proc : (-> any/c any/c (-> any/c any/c boolean?) any/c) — 测试前两个参数是否相等,其中两个值都是通用接口所关联的结构类型(或结构类型的一个子类型)的实例。

    第三个参数是用于递归平等检查的 equal? 谓词; 使用给定的谓词而不是 equal? ,以确保正确处理数据循环, 并与 equal?/recur 一起工作(但要注意, 可以向 equal?/recur 提供任意函数用于递归检查, 这意味着提供给谓词的参数可能会暴露给任意的代码)。

    只有当一对结构不是 eq? 时, 并且只有当它们都有一个从同一结构类型继承的 gen:equal+hash 值时, 才会为它们调用 equal-proc 。 使用这种策略, equal? 接收两个结构的顺序并不重要。 这也意味着,默认情况下,一个结构子类型继承其父类的平等谓词,如果有的话。

  • hash-proc : (-> any/c (-> any/c exact-integer?) exact-integer?) — 为给定的结构计算一个哈希代码,就像 equal-hash-code 一样。 第一个参数是结构类型(或其子类型之一)的实例,通用接口与之相关。

    第二个参数是一个类似于 equal-hash-code 的程序,用于递归哈希码的计算; 使用给定的程序而不是 equal-hash-code ,以确保数据周期得到正确处理。

    尽管 hash-proc 的结果可以是任何精确的整数,但在大多数情况下, 它将被截断为一个 fixnum (例如,对于 equal-hash-code 的结果)。 大致上,截断使用 bitwise-and 来获取数字的低位。 因此,哈希码计算中的变化应该反映在 hash-proc 的结果中与 fixnum 兼容的位上。 哈希码的消费者应该适当地使用 fixnum 范围内的变化, 而生产者 没有 责任反映哈希码在适合 fixnum 的全部比特范围内的变化。

  • hash2-proc : (-> any/c (-> any/c exact-integer?) exact-integer?) — 为给定的结构计算一个二级哈希码。 这个过程就像 hash-proc ,但类似于 equal-secondary-hash-code 。

注意确保 hash-proc 和 hash2-proc 与 equal-proc 一致。 具体来说, hash-proc 和 hash2-proc 应该对任何两个结构产生相同的值, 而 equal-proc 则产生一个真值。

equal-proc 不仅用于 equal? , 还用于 equal?/recur 和 impersonator-of? 。 此外,如果结构类型没有可变字段, equal-proc 用于 equal-always? 和 chaperone-of? 。 同样,当结构类型没有可变字段时, hash-proc 和 hash2-proc 分别用于 equal-always-hash-code 和 equal-always-secondary-hash-code 。 这些方法的实例应该遵循 诚实的自定义平等 中的准则,合理地实现所有这些操作。 特别是,这些方法不应该访问可变数据,除非该结构被声明为可变。

当一个结构类型没有 gen:equal+hash 或 gen:equal-mode+hash 实现时, 那么透明结构(即具有由当前 inspector 控制的 inspector 的结构) 在它们是同一结构类型(不算子类型)的实例,以及它们具有 equal? 的字段值时,是 equal? 的。 对于透明结构, equal-hash-code 和 equal-secondary-hash-code (在没有可变字段的情况下)使用字段值导出哈希代码。 对于至少有一个可变字段的透明结构类型, equal-always? 与 eq? 相同, 并且 equal-secondary-hash-code 的结果只基于 eq-hash-code 。 对于不透明的结构类型, equal? 与 eq? 相同, equal-hash-code 和 equal-secondary-hash-code 的结果只基于 eq-hash-code 。 如果一个结构有一个 prop:impersonator-of 属性,那么如果该属性值的过程在应用于该结构时返回非 #f 值, 则 prop:impersonator-of 属性优先于 gen:equal+hash 。

Examples:
(define (farm=? farm1 farm2 recursive-equal?)
  (and (= (farm-apples farm1)
          (farm-apples farm2))
       (= (farm-oranges farm1)
          (farm-oranges farm2))
       (= (farm-sheep farm1)
          (farm-sheep farm2))))
 
(define (farm-hash-code farm recursive-equal-hash)
  (+ (* 10000 (farm-apples farm))
     (* 100 (farm-oranges farm))
     (* 1 (farm-sheep farm))))
 
(define (farm-secondary-hash-code farm recursive-equal-hash)
  (+ (* 10000 (farm-sheep farm))
     (* 100 (farm-apples farm))
     (* 1 (farm-oranges farm))))
 
(struct farm (apples oranges sheep)
  #:methods gen:equal+hash
  [(define equal-proc farm=?)
   (define hash-proc  farm-hash-code)
   (define hash2-proc farm-secondary-hash-code)])
 
(define eastern-farm (farm 5 2 20))
(define western-farm (farm 18 6 14))
(define northern-farm (farm 5 20 20))
(define southern-farm (farm 18 6 14))

 

> (equal? eastern-farm western-farm)

#f

> (equal? eastern-farm northern-farm)

#f

> (equal? western-farm southern-farm)

#t

Changed in version 8.7.0.5 of package base: 增加了一个检查,使省略 equal-proc, hash-proc 和 hash2-proc 中的任何一个现在都是一个语法错误。

value

gen:equal-mode+hash : any/c

一个 generic interface (参见 通用接口) ， 用于可能指定 equal? 和 equal-always? 之间的差异的类型。 以下方法必须被实现:

  • equal-mode-proc : (-> any/c any/c (-> any/c any/c boolean?) boolean? any/c) — 前两个参数是要比较的值,第三个参数是用于递归比较的平等函数,最后一个参数是模式: #t 表示 equal? 或 impersonator-of? 的比较, #f 表示 equal-always? 或 chaperone-of? 的比较。

  • hash-mode-proc : (-> any/c (-> any/c exact-integer?) boolean? exact-integer?) — 第一个参数是要计算散列代码的值,第二个参数是用于递归散列的散列函数,最后一个参数是模式: #t 表示 equal? 散列, #f 表示 equal-always? 散列。

hash-mode-proc 的实现既可用于一级哈希码,也可用于二级哈希码。

当实现这些方法时,请遵循 诚实的自定义平等 中的准则。 特别是,这些方法应该只在 "mode" 参数为真时访问可变数据, 以表示 equal? 或 impersonator-of? 。

实现 gen:equal-mode+hash 对指定 equal? 和 equal-always? 之间的区别的类型最有用, 比如用 getter 和 setter 程序包装可变数据的结构类型:

Examples:
> (define (get gs) ((getset-getter gs)))
> (define (set gs new) ((getset-setter gs) new))
> (struct getset (getter setter)
     #:methods gen:equal-mode+hash
     [(define (equal-mode-proc self other rec mode)
        (and mode (rec (get self) (get other))))
      (define (hash-mode-proc self rec mode)
        (if mode (rec (get self)) (eq-hash-code self)))])
> (define x 1)
> (define y 2)
> (define gsx (getset (lambda () x) (lambda (new) (set! x new))))

Added in version 8.5.0.3 of package base.
Changed in version 8.7.0.5: 增加了一个检查,使省略 equal-mode-proc 或 hash-mode-proc 现在是一个语法错误。

value

prop:equal+hash : struct-type-property?

一个 structure type property (参见 结构类型属性) ,它为结构类型提供了一个平等的谓词和散列函数。 使用 prop:equal+hash 属性是使用 gen:equal+hash 或 gen:equal-mode+hash generic interface 的一个替代方案。

一个 prop:equal+hash 属性值是一个由三个程序 (list equal-proc hash-proc hash2-proc) 或两个程序 (list equal-mode-proc hash-mode-proc) 组成的列表:

  • 三个程序的情况对应于 gen:equal-hash 的程序:

    • equal-proc : (-> any/c any/c (-> any/c any/c boolean?) any/c)

    • hash-proc : (-> any/c (-> any/c exact-integer?) exact-integer?)

    • hash2-proc : (-> any/c (-> any/c exact-integer?) exact-integer?)

  • 两个程序的情况对应于 gen:equal-mode-hash 的程序:

    • equal-mode-proc : (-> any/c any/c (-> any/c any/c boolean?) boolean? any/c)

    • hash-mode-proc : (-> any/c (-> any/c exact-integer?) boolean? exact-integer?)

在实现这些方法时,请遵循 诚实的自定义平等 中的指导方针。 特别是,这些方法只能在结构被声明为可变或模式为真时访问可变数据。

Changed in version 8.5.0.3 of package base: 增加了对双程序值的支持,以定制 equal-always? 。

4.1.4 诚实的自定义平等

由于 equal-proc 或 equal-mode-proc 不仅仅用于 equal?, 它们的实例应该遵循某些准则, 以确保它们在 equal-always?, chaperone-of? 和 impersonator-of? 中正确工作。

由于这些操作之间的差异,避免在这些操作中调用 equal? 。 取而代之的是,使用第三个参数在棋子上 "递归",这样可以使 equal?/recur 正常工作, 让其他操作在棋子上以各自不同的方式行事,并能进行一些循环检测。

good

(define (equal-proc self other rec)
  (rec (fish-size self) (fish-size other)))

bad

(define (equal-proc self other rec)
  (equal? (fish-size self) (fish-size other)))

不要在元素的计数上使用第三个参数来“recur”。 当一个数据结构关心离散的数字时,它可以对这些数字使用 = ,而不是 equal? 或 “recur”。 当来自 equal?/recur 的 “recur” 参数对某些范围内的数字过于宽容时,在计数上使用 “recur” 是不好的。

good

(define (equal-proc self other rec)
  (and (= (tuple-length self) (tuple-length other))
       (for/and ([i (in-range (tuple-length self))])
         (rec ((tuple-getter self) i)
              ((tuple-getter other) i)))))

bad

(define (equal-proc self other rec)
  (and (rec (tuple-length self) (tuple-length other))
       (for/and ([i (in-range (tuple-length self))])
         (rec ((tuple-getter self) i)
              ((tuple-getter other) i)))))

操作 equal? 和 equal-always? 应该是对称的, 所以当参数交换时, equal-proc 实例不应该改变其答案:

good

(define (equal-proc self other rec)
  (rec (fish-size self) (fish-size other)))

bad

(define (equal-proc self other rec)
  (<= (fish-size self) (fish-size other)))

然而, chaperone-of? 和 impersonator-of? 操作 不是 对称的, 所以当调用第三个参数对棋子进行 "递归" 时,要按棋子的顺序传递:

good

(define (equal-proc self other rec)
  (rec (fish-size self) (fish-size other)))

bad

(define (equal-proc self other rec)
  (rec (fish-size other) (fish-size self)))

操作 equal-always? 和 chaperone-of? 在突变时不应该改变, 所以 equal-proc 实例不应该访问潜在的可变数据。 这包括避免使用 string=? ,因为字符串可以是可变的。 针对不可变类型的平等函数,如 symbol=?, 是可以的。

fine

(define (equal-proc self other rec)
  ; symbols are immutable: no problem
  (symbol=? (thing-name self) (thing-name other)))

bad

(define (equal-proc self other rec)
  ; strings can be mutable: accesses mutable data
  (string=? (thing-name self) (thing-name other)))

将结构声明为可变,使得 equal-always? 和 chaperone-of? 避免使用 equal-proc, 所以如果结构被声明为可变, equal-proc 实例可以自由访问可变数据:

good

(struct mcell (value) #:mutable
  #:methods gen:equal+hash
  [(define (equal-proc self other rec)
     (rec (mcell-value self)
          (mcell-value other)))
   (define (hash-proc self rec)
     (+ (eq-hash-code struct:mcell)
        (rec (mcell-value self))))
   (define (hash2-proc self rec)
     (+ (eq-hash-code struct:mcell)
        (rec (mcell-value self))))])

bad

(struct mcell (box)
  ; not declared mutable,
  ; but represents mutable data anyway
  #:methods gen:equal+hash
  [(define (equal-proc self other rec)
     (rec (unbox (mcell-box self))
          (unbox (mcell-box other))))
   (define (hash-proc self rec)
     (+ (eq-hash-code struct:mcell)
        (rec (unbox (mcell-value self)))))
   (define (hash2-proc self rec)
     (+ (eq-hash-code struct:mcell)
        (rec (unbox (mcell-value self)))))])

结构控制访问易变数据的另一种方法是通过实现 gen:equal-mode+hash 而不是 gen:equal+hash 。 当模式为真时, equal-mode-proc 实例可以自由访问可变数据,而当模式为假时,它们不应该访问:

also good

(struct mcell (value) #:mutable
  ; only accesses mutable data when mode is true
  #:methods gen:equal-mode+hash
  [(define (equal-mode-proc self other rec mode)
     (and mode
          (rec (mcell-value self)
               (mcell-value other))))
   (define (hash-mode-proc self rec mode)
     (if mode
         (+ (eq-hash-code struct:mcell)
            (rec (mcell-value self)))
         (eq-hash-code self)))])

still bad

(struct mcell (value) #:mutable
  ; accesses mutable data ignoring mode
  #:methods gen:equal-mode+hash
  [(define (equal-mode-proc self other rec mode)
     (rec (mcell-value self)
          (mcell-value other)))
   (define (hash-mode-proc self rec mode)
     (+ (eq-hash-code struct:mcell)
        (rec (mcell-value self))))])
