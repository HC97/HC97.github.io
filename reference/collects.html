18.2 Libraries and Collections

一个 library 是供多个程序使用的 module 声明。 Racket 将库进一步分组为 collections 。 通常,集合是通过 packages 来添加的(见 Racket 中的包管理); 包管理器在 Racket 核心之外工作,但它通过 collection links files 来配置核心运行时系统。

集合中的库是通过 lib 路径(见 require) 或符号缩写来引用的。 例如,下面的模块使用了 "setup" 集合中的 "getinfo.rkt" 库模块和 "games" 集合的 "cards" 子集合中的 "cards.rkt" 库模块:

#lang racket
(require (lib "setup/getinfo.rkt")
         (lib "games/cards/cards.rkt"))
....

这个例子用符号速记法写得更紧凑、更普遍:

#lang racket
(require setup/getinfo
         games/cards/cards)
....

当标识符 id 被用于 require 形式时,它被转换为 (lib rel-string) , 其中 rel-string 是 id 的字符串形式。

(lib rel-string) 中的 rel-string 由一个或多个命名集合的路径元素组成, 然后是一个命名库文件的最后路径元素;路径元素之间用 / 分隔。 如果 rel-string 不包含 / ,那么 /main.rkt 将被隐含地附加到路径中。 如果 rel-string 包含 / 但没有以文件后缀结尾,那么 .rkt 将被隐含地附加到路径中。

库也可以通过 PLaneT 包发布。 这样的库是通过 planet 模块路径(见 require) 来引用的, 并且由 Racket 按需下载,而不是通过 collections 来引用。

planet 或 lib 路径到 module 声明的转换是由 module name resolver 决定的, 由 current-module-name-resolver 参数指定。

18.2.1 集合搜索配置

对于默认的 module name resolver, 集合的搜索路径由 current-library-collection-links 参数和 current-library-collection-paths 参数决定:

  • 最原始的基于 collection 的模块位于相对于 Racket 可执行文件的 "collects" 目录中。 一个集合的库被分组在一个目录中,这个目录的名字与集合的名字一致。 到 "collects" 目录的路径通常包含在 current-library-collection-paths 中。

  • 基于集合的库也可以安装其他目录,也许是用户专用的,其结构类似于 "collects" 目录。 这些额外的目录可以通过 racket 的命令行参数动态地包含在 current-library-collection-paths 参数中, 或者通过设置 PLTCOLLECTS 环境变量;见 find-library-collection-paths 。

  • Collection links files 提供了从顶级收藏集名称到目录的映射, 以及额外的类似于 "collects" 的目录(这些目录的名称与收藏集名称一致)。 每个要搜索的 collection links file 都由 current-library-collection-links 参数来引用; 该参数引用的是文件,而不是文件的内容,这样就可以检测到文件的变化,并影响以后的模块解析。 参见 find-library-collection-links 。

  • The current-library-collection-links 参数的值也可以包括哈希表, 提供与 collection links files 相同的内容:从符号形式的集合名称到集合路径列表的映射, 或者从 #f 到类似 "collects" 的路径列表的映射。

  • 最后, current-library-collection-links 参数的值包括 #f , 表示在搜索过程中, module-name resolver 应该检查 current-library-collection-paths 相对于 current-library-collection-links 中的文件和哈希表。

为了解决一个模块引用 rel-string , 默认的 module name resolver 从头到尾搜索 current-library-collection-links 中的集合链接, 找到第一个包含 rel-string 的目录, 在 current-library-collection-paths 中包含 #f 的地方进行拼接搜索。 链接表和搜索路径中每个元素的文件系统树实际上是与对应于同一集合的其他路径元素的文件系统树 spliced 在一起。 一些 Racket 工具依赖于模块路径名称的唯一解析,所以一个安装和配置不应该允许多个文件匹配相同的集合和文件组合。

current-library-collection-links 参数的值被 racket 可执行程序初始化为 (find-library-collection-links) 的结果, current-library-collection-paths 参数的值被初始化为 (find-library-collection-paths) 的结果。

18.2.2 藏品链接

Collection links files 被 collection-file-path, collection-path 和默认的 module name resolver 使用, 以便在尝试 (current-library-collection-paths) 搜索路径之前找到收藏品。 要使用的 collection links files 由 current-library-collection-links 参数决定, 它被初始化为 find-library-collection-links 的结果。

使用默认的阅读器参数设置 read 一个 collection links file ,以获得一个列表。 列表中的每个元素都必须是一个链接规范, 其形式为 (list string encoded-path), (list string encoded-path regexp), (list 'root encoded-path), (list 'root encoded-path regexp), (list 'static-root encoded-path), (list 'static-root encoded-path regexp) 。 一个 string 命名了一个 collection, 在这种情况下, encoded-path 描述了一个可以作为集合的路径 (直接,而不是由 string 命名的 encoded-path 的一个子目录)。 相比之下,一个 'root 就像 (current-library-collection-paths) 中的一个路径。 一个 'static-root 就像一个 'root ,但是目录的直接内容被认为不会改变, 除非 collection links file 改变。 每个 encoded-path 要么是一个字符串,要么是一个用 bytes->path 转换为路径的字节字符串, 要么是一个相对路径元素字节字符串、 'up 和 'same 指标的列表, 它们与 build-path 的字节字符串一起用 bytes->path-element 转换。 如果 encoded-path 描述了一个相对路径,它是相对于包含 collection links file 的目录的。 如果在链接中指定了 regexp , 那么只有在 (regexp-match? regexp (version)) 产生一个真实的结果时才会使用该链接。

在一个 collection links file 中,一个顶级集合可以有多个链接,并且可以出现任意数量的 'root 。 相应的路径被有效地拼接在一起,因为这些路径是为了定位一个文件或子集合而被尝试的。

raco link 命令-链接工具可以显示、安装和删除一个 collection links file 的链接。 参见 raco: Racket 命令行工具 中的 raco link: 库收藏链接 以获得更多信息。

Changed in version 8.1.0.6 of package base: 改变了 encoded-path 以允许字节字符串和列表。

18.2.3 采集路径和参数

procedure

(find-library-collection-paths [pre-extras    
  post-extras    
  config]    
  name) → (listof path?)
  pre-extras : (listof path-string?) = null
  post-extras : (listof path-string?) = null
  config : hash? = (read-installation-configuration-table)
  name : (get-installation-name config)
产生一个路径列表,通常用于初始化 current-library-collection-paths,如下所示:

  • 由 (build-path (find-system-path 'addon-dir) name "collects") 产生的路径是默认收集路径列表的第一个元素, 除非 use-user-specific-search-paths 参数的值是 #f 。

  • 在 pre-extras 中提供的额外目录被包含在默认的收集路径列表旁边,转换为相对于可执行文件的完整路径。

  • 如果 (find-system-path 'collects-dir) 指定的目录是绝对的, 或者它是相对的(相对于可执行文件)并且存在,那么它将被添加到默认收集路径列表的最后。

  • 在 post-extras 中提供的额外目录在默认的收集路径列表中最后包括,并转换为相对于可执行文件的完整路径。

  • 如果 config 有一个 'collects-search-dirs 的值, 那么它将被用来代替默认的收集路径列表(由前面三条子弹构建), 并且默认值被拼接到 'collects-search-dirs 列表中的任何 #f 中。 如果 config 没有 'collects-search-dirs 值,那么就使用默认的收集路径列表。

  • 如果定义了 PLTCOLLECTS 环境变量, 只要 use-user-specific-search-paths 的值为 true, 就会用 path-list-string->path-list 与默认列表相结合。 如果没有定义或者 use-user-specific-search-paths 的值为 #f, 则直接使用由前面四条子弹构建的集合路径列表。

    注意,在 Unix and Mac OS 上,路径用 : 分隔,而在Windows上用 ; 分隔。 另外, path-list-string->path-list 在空路径处拼接默认路径,例如,在许多 Unix shells 中, 你可以将 PLTCOLLECTS 设置为 ":‘pwd‘", "‘pwd‘:" 或 "‘pwd‘" , 以分别指定在默认路径之后、之前或代替默认路径搜索当前目录。

Changed in version 8.4.0.3 of package base: 增加了 config 和 name 参数。

procedure

(find-library-collection-links [config] name)

 → (listof (or/c #f (and/c path? complete-path?)))
  config : hash? = (read-installation-configuration-table)
  name : (get-installation-name config)
产生一个路径和 #f 的列表,通常用于初始化 current-library-collection-links, 如下所示:

  • 列表以 #f 开始,它使默认的 module name resolver, collection-file-path 和 collection-path 在 collection links files 之前尝试 current-library-collection-paths 中的路径。

  • 只要 use-user-specific-search-paths 和 use-collection-link-paths 的值为真, 结果列表中的第二个元素就是用户特定的 collection links file 的路径, 默认为 (build-path (find-system-path 'addon-dir) name "links.rktd") , 但它可以被 config 中的 'links-file 值替换。

  • 只要 use-collection-link-paths 的值为 true, 列表的其余部分就会包含一个类似 get-links-search-files 的结果,但如果提供了 config , 则使用 config ,而不是读取安装的 "config.rktd" 文件。 通常情况下,这个结果是一个带有单一路径的列表 (build-path (find-config-dir) "links.rktd") 。

Changed in version 8.4.0.3 of package base: 增加了 config 和 name 参数。

procedure

(collection-file-path file 
  collection ...+ 
  [#:check-compiled? check-compiled?]) 
 → path?
  file : path-string?
  collection : path-string?
  check-compiled? : any/c = (regexp-match? #rx"[.]rkt$" file)
(collection-file-path file    
  collection ...+    
  #:fail fail-proc    
  [#:check-compiled? check-compiled?]) → any
  file : path-string?
  collection : path-string?
  fail-proc : (string? . -> . any)
  check-compiled? : any/c = (regexp-match? #rx"[.]rkt$" file)
返回由 collection 指定的集合中 file 所指示的路径, 其中第二个 collection (如果有的话)命名了一个子集合,以此类推。 搜索使用 current-library-collection-links 和 current-library-collection-paths 的值。

也请参见 setup/collection-search 中的 collection-search 。

如果 file 没有找到,但是 file 以 ".rkt" 结尾, 并且存在一个后缀为 ".ss" 的文件,那么将使用 ".ss" 文件的目录。 如果没有找到 file ,并且 ".rkt"/".ss" 转换不适用, 但找到了与 collection 相对应的目录,那么将返回一个使用第一个这样的目录的路径。

如果 check-compiled? 为true, 那么搜索也取决于 use-compiled-file-paths 和 current-compiled-file-roots; 如果 file 没有找到,那么会以与默认的 compiled-load handler 相同的方式检查一个后缀为 ".zo" 的编译形式的 file 。 如果找到了一个编译的文件,来自 collection-file-path 的结果报告了该 file 本身在找到的编译文件中所占据的位置(如果它存在)。

最后,如果没有找到集合,并且提供了 fail-proc , 那么 fail-proc 将被应用于一个错误信息(没有以 "collection-file-path:" 开始, 或者以其他方式要求一个来源),其结果是 collection-file-path 的结果。 如果没有提供 fail-proc 并且没有找到集合,那么会引发 exn:fail:filesystem exception is raised 。

Examples:
> (collection-file-path "main.rkt" "racket" "base")

#<path:path/to/collects/racket/base/main.rkt>

> (collection-file-path "sandwich.rkt" "bologna")

collection-file-path: collection not found

  collection: "bologna"

  in collection directories:

   /home/ice/racket-8.9-2023-07-14/collects/

   ... [166 additional linked and package directories]

当多个目录对应于集合时,在搜索序列中发现的第一个目录(见 集合搜索配置) 被返回。

决定 collection links files 、附加路径和 current-library-collection-paths 的相对搜索顺序的参数, 用于通过默认的 module name resolver 寻找库(例如 require 中引用的), 以及通过 collection-path 和 collection-file-path 寻找路径。 更多信息见 集合搜索配置 。

parameter

(use-user-specific-search-paths) → boolean?

(use-user-specific-search-paths on?) → void?
  on? : any/c
参数决定了用户特定的路径,这些路径在 (find-system-path 'addon-dir) 产生的目录中, 是否被包括在集合和其他文件的搜索路径中。 例如,当这个参数的值为 #f 时, find-library-collection-paths 的初始值会省略用户特定的集合目录。

如果 -U 或 --no-user-path 参数给 racket, 那么 use-user-specific-search-paths 就被初始化为 #f 。

parameter

(use-collection-link-paths) → boolean?

(use-collection-link-paths on?) → void?
  on? : any/c
决定 collection links files 是否包含在 find-library-collection-links 的结果中的参数。

如果这个参数在启动时的值是 #f ,那么对于 Racket 进程来说, collection links files 实际上是永久禁用的。 特别是,如果提供一个空字符串作为 racket 的 -X 或 --collects 参数, 那么不仅 current-library-collection-paths 被初始化为空列表, use-collection-link-paths 也被初始化为 #f 。

procedure

(read-installation-configuration-table)

 → (and/c hash? immutable?)
只要安装的 "config.rktd" 文件(见 安装配置和搜索路径) 的内容是一个 hash table ,就返回该文件的内容,否则就返回一个空的哈希表。

Added in version 8.4.0.3 of package base.
