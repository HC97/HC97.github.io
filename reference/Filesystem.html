15.2 Filesystem

15.2.1 定位路径

procedure

(find-system-path kind) → path?

  kind : symbol?
返回由 kind 指定的标准类型路径的机器特定路径,它必须是下列之一:

  • 'home-dir — 当前 user’s home directory.

    在所有的平台上,如果 PLTUSERHOME 环境变量被定义为一个 complete 路径, 那么该路径就被用作用户的主目录。

    在 Unix 和 Mac OS 上,当 PLTUSERHOME 不适用时,用户的主目录是通过扩展路径 "~" 来确定的, 扩展时首先检查是否有 HOME 环境变量。 如果没有定义,则查询 USER 和 LOGNAME 环境变量(按顺序)以找到一个用户名, 然后查询系统文件以找到用户的主目录。

    在 Windows 上,当 PLTUSERHOME 不适用时,用户的主目录是由 Windows 注册表决定的用户特定的配置文件目录。 如果注册表由于某种原因不能提供一个目录,那么将使用 USERPROFILE 环境变量的值来代替, 只要它指向一个存在的目录。 如果 USERPROFILE 也失败了, 那么目录就是由 HOMEDRIVE 和 HOMEPATH 环境变量指定的。 如果这些环境变量没有被定义,或者指定的目录仍然不存在,那么包含当前可执行文件的目录将被作为主目录。

  • 'pref-dir — 用于存储当前用户的偏好的标准目录。偏好目录可能不存在。

    在 Unix 中,偏好目录通常是 XDG_CONFIG_HOME 指定的路径中的 "racket" 子目录, 如果 XDG_CONFIG_HOME 没有设置为绝对路径或者设置了 PLTUSERHOME , 则是 user’s home directory 中的 ".config/racket" 。 无论如何,如果该目录不存在,但在 user’s home directory 中存在一个 ".racket" 目录, 那么该目录就是首选项目录。

    在 Windows 上,如果由 PLTUSERHOME 决定, 首选项目录是 user’s home directory 中的 "Racket" , 否则就是 Windows 注册表指定的用户的应用数据文件夹; 应用数据文件夹通常是用户配置文件目录中的 "Application Data" 。

    在 Mac OS 上,偏好目录是 user’s home directory 下的 "Library/Preferences" 。

  • 'pref-file — 一个包含偏好值的符号键关联列表的文件。 该文件的目录路径总是与 'pref-dir 返回的结果一致。 文件名在 Unix 和 Windows 上是 "racket-prefs.rktd" , 在 Mac OS 上是 "org.racket-lang.prefs.rktd" 。 该文件的目录可能不存在。 参见 get-preference 。

  • 'temp-dir — 用于存储临时文件的标准目录。 在 Unix and Mac OS 上,这是 TMPDIR 环境变量指定的目录,如果它被定义, 否则就是 "/var/tmp", "/usr/tmp" 和 "/tmp" 中存在的第一个路径。 在 Windows 中,如果 TMP 或 TEMP 环境变量被定义, 那么结果就是 TMP 或 TEMP 环境变量所指定的目录,否则就是当前目录。

  • 'init-dir — 包含Racket可执行文件所使用的初始化文件的目录。

    在 Unix 中,初始化目录与 'pref-dir 返回的结果相同 —返回的结果,除非该目录不存在, 而 user’s home directory 中存在一个 ".racketrc" 文件,在这种情况下,主目录就是初始化目录。

    在 Windows 上,初始化目录与 user’s home directory 相同。

    在 Mac OS 上,初始化目录是 user’s home directory 中的 "Library/Racket" —除非那里没有 "racketrc.rktl" ,而主目录中确实存在 ".racketrc" 文件, 在这种情况下,主目录是初始化目录。

  • 'init-file — Racket可执行文件启动时加载的文件。 路径中的目录部分与 'init-dir 返回的路径相同。

    在 Windows 中,文件部分的名称是 "racketrc.rktl" 。

    在 Unix 和 Mac OS 上,文件部分的名称是 "racketrc.rktl"— 除非 'init-dir 返回的路径是 user’s home directory , 在这种情况下,文件部分的名称是 ".racketrc" 。

  • 'config-dir — 一个用于安装配置的目录。 这个目录由 PLTCONFIGDIR 环境变量指定, 它可以被 --config 或 -G 命令行标志所覆盖。 如果没有指定环境变量或标志,或者该值不是一个合法的路径名称, 那么这个目录就默认为相对于当前可执行文件的 "etc" 目录。 如果 (find-system-path 'config-dir) 的结果是一个相对路径, 那么它就是相对于当前可执行文件的。该目录可能不存在。

  • 'host-config-dir — 和 'config-dir 一样, 但是当选择了跨平台构建模式(通过 racket 的 -C or --cross 参数; 见 命令行), 其结果是指当前系统的安装目录,而不是目标系统的目录。

  • 'addon-dir — 一个用于用户特定的 Racket 配置、包和扩展的目录。 这个目录由 PLTADDONDIR 环境变量指定,它可以被 --addon 或 -A 命令行标志所覆盖。 如果没有指定环境变量或标志,或者该值不是一个合法的路径名称,那么这个目录就默认为一个平台特定的位置。 该目录可能不存在。

    在 Unix 上,默认的是 XDG_DATA_HOME 指定的路径中的 "racket" 子目录, 如果 XDG_CONFIG_HOME 没有设置为绝对路径或者设置了 PLTUSERHOME , 则是 user’s home directory 中的 ".local/share/racket" 。 如果该目录不存在,但在用户的主目录中存在 ".racket" 目录,那么 ".racket" 目录路径就是默认的。

    在 Windows 上,默认值与 'pref-dir 目录相同。

    在 Mac OS 上,默认的是 user’s home directory 下的 "Library/Racket" 。

  • 'cache-dir — 一个用于存储用户特定的缓存的目录。这个目录可能不存在。

    在 Unix 上,缓存目录通常是 XDG_CACHE_HOME 指定的路径中的 "racket" 子目录,如果 XDG_CACHE_HOME 没有设置为绝对路径或者设置了 PLTUSERHOME , 则是 user’s home directory 中的 ".cache/racket" 。 如果该目录不存在,但主目录中存在 ".racket" 目录,那么 ".racket" 目录就是缓存目录。

    在 Windows 上,缓存目录与 'addon-dir 返回的结果相同。

    在 Mac OS 上,缓存目录是 user’s home directory 下的 "Library/Caches/Racket" 。

  • 'doc-dir — 用于存储当前用户文档的标准目录。 在 Unix 中,它是 user’s home directory 。 在 Windows 中,如果由 PLTUSERHOME 决定,它就是 user’s home directory , 否则就是 Windows 注册表指定的用户的文档文件夹;文档文件夹通常是用户主目录下的 "My Documents" 。 在 Mac OS 上,它是 user’s home directory 下的 "Documents" 目录。

  • 'desk-dir — 当前用户桌面的目录。 在 Unix 中,它是 user’s home directory 。 在 Windows 上,如果由 PLTUSERHOME 决定,它就是 user’s home directory , 否则就是由 Windows 注册表指定的用户的桌面文件夹; 桌面文件夹通常是用户主目录中的 "Desktop" 。 在Mac OS上,它是 user’s home directory 下的 "Desktop" 。

  • 'sys-dir — 包含 Windows 操作系统的目录。在 Unix and Mac OS 上,其结果是 "/" 。

  • 'exec-file — 由操作系统为当前调用提供的 Racket 可执行文件的路径。 对于某些操作系统,路径可以是相对的。

    对于 GRacket 来说,可执行路径是 GRacket 可执行文件的名称。

  • 'run-file — 当前可执行文件的路径;这可能与 'exec-file 的结果不同, 因为通过 Racket (或 GRacket)可执行文件的 --name 或 -N 命令行标志提供了一个替代路径, 或者因为嵌入的可执行文件安装了一个替代路径。 特别是由 make-racket-launcher 创建的 "启动器 "脚本将此路径设置为脚本的路径。

  • 'collects-dir — 主库集合的路径(见 Libraries and Collections) )。 如果这个路径是相对的,那么它就是相对于 (find-system-path 'exec-file) 报告的可执行文件而言的 –尽管后者可能是一个软链接或相对于用户的可执行文件搜索路径, 所以这两个结果应该与 find-executable-path 合并。 'collects-dir 路径通常嵌入 Racket 可执行文件中, 但它可以被 --collects 或 -X 命令行标志所覆盖。

  • 'host-collects-dir — 与 'collects-dir 类似, 但当选择了跨平台构建模式(通过 racket 的 -C 或 --cross 参数; 见 命令行), 其结果是指当前系统安装的目录,而不是目标系统的目录。 在跨平台构建模式下,采集文件通常从目标系统的安装中读取, 但有些任务需要当前系统的目录(比如存放外国库的目录),这些目录是相对于主库采集路径配置的。

  • 'orig-dir — 启动时的当前目录, 这在将 (find-system-path 'exec-file) 或 (find-system-path 'run-file) 的相对路径结果转换为完整路径时很有用。

Changed in version 6.0.0.3 of package base: 增加了 PLTUSERHOME 。
Changed in version 6.9.0.1: 增加了 'host-config-dir 和 'host-collects-dir 。
Changed in version 7.8.0.9: 增加了 'cache-dir, 并改为在 Unix 上使用 XDG 目录作为首选, 以前的路径作为后备,对 Mac OS 也做了类似的调整。

procedure

(path-list-string->path-list str 
  default-path-list) 
 → (listof (or/c path? 'same))
  str : (or/c string? bytes?)
  default-path-list : (listof (or/c path? 'same))
解析一个包含路径列表的字符串或字节串,并返回一个路径列表。 在 Unix and Mac OS, path-list 字符串中的路径用 : 隔开;在 Windows 上,路径用 ; 隔开,字符串中的所有 " 被丢弃。 每当路径列表中包含一个空路径时,列表 default-path-list 会被拼接到返回的路径列表中。 str 中不构成有效路径的部分不包括在返回的列表中。 给定的 str 不能包含 nul 字符或 nul 字节。

Changed in version 8.0.0.10 of package base: 改为允许 default-path-list 中的 'same 。

procedure

(find-executable-path program    
  [related    
  deepest?]) → (or/c path? #f)
  program : path-string?
  related : (or/c path-string? #f) = #f
  deepest? : any/c = #f
查找可执行 program 的路径,如果找不到路径则返回 #f 。

在 Windows 中,如果 program 没有被找到,而且没有文件扩展名,那么搜索将重新开始, 并在 program 上添加 ".exe" , 只有当带有 ".exe" 的路径也找不到时,结果才是 #f 。 如果只找到带有扩展名的 program ,则结果包括扩展名 ".exe" 。

如果 related 不是 #f 那么它必须是一个相对路径字符串, 而且为 program 找到的路径必须是 related 文件或目录与可执行文件存在于同一目录中。 然后,结果是找到的 related 文件的完整路径,而不是可执行文件的路径。

这个过程被 Racket 可执行程序用来寻找标准库集合目录(见 Libraries and Collections)。 在这种情况下, program 是用来启动 Racket 的名字, related 是 "collects" 。 使用 related 参数是因为在 Unix and Mac OS, program 可能涉及一连串的软链接; 在这种情况下,related 决定了链中的哪个链接是相关的。

如果 related 不是 #f, 那么当 find-executable-path 没有找到一个 program 是另一个文件路径的链接时,搜索可以继续进行链接的目的地。 进一步的链接被检查,直到找到 related 或者到达链接链的末端。 如果 deepest? 是 #f (默认值),那么结果对应于找到 related 的链接链中的第一个路径(而进一步的链接实际上没有被探索); 否则,结果对应于找到 related 的链接链中的最后一个链接。

如果 program 是一个无路径的名字, find-executable-path 会得到 PATH 环境变量的值; 如果这个环境变量被定义了, find-executable-path 会尝试将 PATH 中的每个路径作为 program 的前缀, 使用上述针对含路径 program 的搜索算法。 如果没有定义 PATH 环境变量, program 的前缀是当前目录并用于上述搜索算法。 (在 Windows 中,当前目录总是隐含地成为 PATH 中的第一项, 所以find-executable-path 在 Windows 中首先检查当前目录)。

Changed in version 8.1.0.7 of package base: 在 Windows 上增加了带 ".exe" 的搜索。

15.2.2 文件

procedure

(file-exists? path) → boolean?

  path : path-string?
如果文件(不是目录) path 存在,返回 #t ,否则返回 #f 。

在 Windows 中, file-exists? 对于所有特殊文件名的变化 (例如, "LPT1","x:/baddir/LPT1" )都报告 #t 。

procedure

(link-exists? path) → boolean?

  path : path-string?
如果一个链接 path 存在,则返回 #t ,否则返回 #f 。

谓词 file-exists? 或 directory-exists? 对一个链接或一系列链接的最终目的地起作用, 而 link-exists? 只跟随链接来解决 path 的基本部分(即除了路径中最后一个名字以外的所有内容)。

这个过程从不引发 exn:fail:filesystem 。

在 Windows 上, link-exists? 对符号链接和结点都报告 #t 。

Changed in version 6.0.1.12 of package base: 增加了对 Windows 上链接的支持。

procedure

(file-or-directory-type path [must-exist?])

 → (or/c 'file 'directory 'link 'directory-link #f)
  path : path-string?
  must-exist? : any/c = #f
报告 path 是否指的是文件、目录、链接或目录链接(在 Windows 中;也可以参见 make-file-or-directory-link),假设 path 可以被访问。

如果 path 不能被访问,如果 must-exist? 是 #f , 结果是 #f ,否则会引发 exn:fail:filesystem exception is raised 。

Added in version 7.8.0.5 of package base.

procedure

(delete-file path) → void?

  path : path-string?
如果文件存在,则删除路径为 path 的文件,否则会引发 exn:fail:filesystem exception is raised 。 如果 path 是一个链接,则删除该链接而不是链接的目的地。

在 Windows 上,如果删除文件的初始尝试因权限错误而失败,并且 current-force-delete-permissions 的值为 true,那么 delete-file 会尝试改变文件的权限(允许写入),然后删除文件; 权限改变和删除是一个非原子序列,如果删除失败,不会尝试恢复权限改变。

在 Windows 上, delete-file 可以删除一个符号链接,但不能删除一个结点。 使用 delete-directory 来删除一个结点。

在 Windows 上,请注意,如果一个文件被删除,而它仍然被某些进程使用(例如,后台搜索索引器), 那么该文件的内容最终会消失,但该文件的名称仍然被占用,直到该文件不再被使用。 只要名称仍然被占用,试图打开、删除或替换该文件将触发一个权限错误(而不是一个文件存在的错误)。 避免这个陷阱的一个常用技巧是在删除文件之前将其移到一个生成的临时名称上。 参见 delete-directory/files 。

Changed in version 6.1.1.7 of package base: 改变了 Windows 的行为, 使用 current-force-delete-permissions 。

procedure

(rename-file-or-directory old    
  new    
  [exists-ok?]) → void?
  old : path-string?
  new : path-string?
  exists-ok? : any/c = #f
将路径为 old 的文件或目录重命名为路径为 new 的文件或目录–如果它存在的话。 如果文件或目录没有被成功重命名,会引发 exn:fail:filesystem exception is raised 。

这个过程可以用来将一个文件/目录移动到不同的目录(在同一个文件系统上),也可以用来重命名一个目录中的文件/目录。 除非 exists-ok? 被提供为真值,否则 new 不能指向一个现有的文件或目录, 但在 Unix 和 Mac OS 上,检查与重命名操作不一致。 即使 exists-ok? 为真,当 old 是一个目录时, new 也不能引用一个现有文件,反之亦然。

如果 new 存在并被替换,替换在 Unix 和 Mac OS 上是原子性的,但在 Windows 上不能保证是原子性的。 此外,如果 new 存在并且被任何进程打开进行读写,那么在 Windows 上试图替换它通常会失败。 也请参见 call-with-atomic-output-file 。

如果 old 是一个链接,该链接被重新命名,而不是链接的目的地, 并且它被算作一个文件来替换任何现有的 new 。

在 Windows 上,要注意如果一个目录中的任何文件都是开放的,则该目录不能被重命名。 如果搜索索引器在后台运行(如默认的 Windows 配置),这种限制就特别有问题。 一个可能的解决方法是将 copy-directory/files 和 delete-directory/files 结合起来,因为后者可以处理打开的文件, 尽管这个序列显然不是原子性的,而且会暂时重复文件。

procedure

(file-or-directory-modify-seconds path    
  [secs-n]) → exact-integer?
  path : path-string?
  secs-n : #f = #f
(file-or-directory-modify-seconds path    
  secs-n) → void?
  path : path-string?
  secs-n : exact-integer?
(file-or-directory-modify-seconds path    
  [secs-n    
  fail-thunk]) → any
  path : path-string?
  secs-n : (or/c exact-integer? #f) = #f
  fail-thunk : (-> any)
   = (lambda () (raise (make-exn:fail:filesystem ....)))
当 secs-n 没有提供或为 #f 时, 文件或目录的 返回 文件或目录的最后修改日期, 单位是自 the epoch 以来的秒数(也见 时间) 。

对于 Windows 的 FAT 文件系统,目录没有修改日期。 因此,目录的创建日期被返回,但文件的修改日期被返回。

如果提供了 secs-n 而不是 #f, path 的访问和修改时间将被设置为给定的时间。

如果出错(例如,如果没有这样的文件存在),那么 fail-thunk 将被调用(通过尾部调用), 产生 file-or-directory-modify-seconds 的调用结果。 如果没有提供 fail-thunk ,错误会引发 exn:fail:filesystem 。

15.2.4 检测文件系统变化

许多操作系统提供文件系统变化的通知,这些通知通过 filesystem change events 反映在 Racket 中。

  v : any/c

procedure

(filesystem-change-evt path [failure-thunk])

 → (or/c filesystem-change-evt? any)
  path : path-string?
  failure-thunk : (or/c (-> any) #f) = #f
创建一个 filesystem change event , 这是一个 synchronizable event , 在 path 发生变化后就可以 ready for synchronization :

  • 如果 path 指的是文件,当文件的内容或属性发生变化时,或者当文件被删除时, 该事件就成为 ready for synchronization 。

  • 如果 path 指的是一个目录,如果在该目录中添加、重命名或删除一个文件或子目录, 该事件将成为 ready for synchronization 。

如果事件被传递给 filesystem-change-evt-cancel,它也会成为 ready for synchronization 。

最后,根据操作系统提供的信息的精确性,该事件在其他情况下也可能成为 ready for synchronization 。 例如,在 Windows 中,当任何文件在与该文件相同的目录中发生变化时,该文件的事件就会准备就绪。

在一个 filesystem change event 成为 ready for synchronization 后, 它将保持 ready for synchronization 。 该事件的 synchronization result 就是事件本身。

如果当前平台不支持文件系统变化通知,那么如果没有提供 failure-thunk 这个过程, 就会引发 exn:fail:unsupported exception is raised ; 如果提供了 failure-thunk ,就会在尾部位置调用。 同样地,如果在创建事件时有任何操作系统错误(比如一个不存在的文件), 那么将引发 exn:fail:filesystem exception is raised 或调用 failure-thunk 。

创建一个文件系统变更事件会在操作系统层面分配资源。 当事件被同步化并 ready for synchronization 时, 当事件被 filesystem-change-evt-cancel 取消时, 或者当垃圾收集器确定文件系统变更事件无法到达时, 这些资源最迟会被释放。 参见 'fs-change 模式下的 system-type 。

当一个文件系统变化事件被创建时,它被置于 current custodian 的管理之下。 如果 custodian 被关闭, filesystem-change-evt-cancel 将应用于该事件。

15.2.5 声明运行时需要的路径

 (require racket/runtime-path) package: base
The bindings documented in this section are provided by the racket/runtime-path library, not racket/base or racket.

racket/runtime-path 库提供了在运行时访问文件和目录的形式,使用的路径通常是相对于包围的源文件。 与使用 collection-path 不同, define-runtime-path 将每个运行时路径暴露给工具, 如可执行文件和发行版创建者,因此运行时需要的文件和目录在发行版中被携带。

除了下面描述的绑定外, racket/runtime-path 在 phase level 1 提供了 #%datum , 因为字符串常量经常被用作 define-runtime-path 的编译时表达式。

syntax

(define-runtime-path id maybe-runtime?-id expr)

 
maybe-runtime?-id = 
  | #:runtime?-id runtime?-id
将 expr 既作为编译时(即 phase 1)表达式又作为运行时(即 phase 0) 表达式。 在这两种情况下, expr 都应该产生一个路径、一个代表路径的字符串、 一个形式为 (list 'lib str ...+) 的列表, 或者一个形式为 (list 'so str) 或 (list 'so str vers) 的列表。 如果提供了 runtime?-id ,那么在 expr 的上下文中,对于expr 的编译时实例, 它被绑定到 #f ,对于 expr 的运行时实例,它被绑定到 #t 。

对于运行时, id 被绑定到一个基于 expr 结果的路径上。 路径的计算通常是通过从 expr 中获取相对路径结果,并将其添加到包围文件的路径中(其计算方法如下所述)。 然而,像可执行文件创建者这样的工具也可以安排(通过与 racket/runtime-path 合作), 在生成的可执行文件中替换出不同的基本路径。 如果 expr 产生了一个绝对路径,它通常会直接返回,但也可能被可执行文件创建者替换。 在所有情况下,可执行文件创建者都会保留给定 package 内所有路径的相对位置(将任何包外的路径视为一起)。 当 expr 产生一个相对或绝对路径时,那么与 id 绑定的路径总是一个绝对路径。

如果 expr 产生一个形式为 (list 'lib str ...+) 的列表, 那么与 id 绑定的值是一个绝对路径。 该路径指的是一个基于集合的文件,类似于将该值作为 module path 。

如果 expr 产生一个形式为 (list 'so str) 或 (list 'so str vers) 的列表, 绑定在 id 上的值可以是 str 或绝对路径;当在 Racket 特定的共享对象库目录(由 get-lib-search-dirs 确定)中搜索到路径时,它是一个绝对路径。 这样一来,专门为 Racket 安装的共享对象库就会被带入到分发中。 搜索依次尝试每个目录;在一个目录中,搜索直接使用 str , 然后尝试添加 vers 指定的每个版本–默认为 '(#f) –以及一个平台特定的共享库扩展– 如 (system-type 'so-suffix) 产生的。 vers 可以是一个字符串,也可以是一个字符串和 #f 的列表。

如果 expr 产生了一个形式为 (list 'share str) 的列表, 与 id 绑定的值可以是 str 或绝对路径; 当在 find-user-share-dir 和 find-share-dir (按顺序)报告的目录中搜索路径时, 它是一个绝对路径。 这样一来,安装在 Racket 的 "share" 目录下的文件就会在分发时被带入。

如果 expr 产生一个 (list 'module module-path var-ref) 或 (list 'so str (list str-or-false ...)) 形式的列表, 绑定在 id 上的值是一个 module path index , 其中 module-path 被视为相对(如果它是相对的)模块, 是 variable reference var-ref 的家, 如果 module-path 是绝对的, var-ref 可以是 #f 。 在一个可执行文件中,相应的模块会被带上,包括其所有的依赖关系。

在编译时, expr 的结果会被可执行文件的创建者使用,但在编译包含的模块时却不是这个结果。 相反, expr 作为一个编译时表达式被保留在模块中(在 begin-for-syntax 的意义上)。 后来,在创建可执行文件时,模块的编译时部分被执行(再次), expr 的结果是可执行文件中包含的文件或目录。 额外的编译时执行的原因是 expr 的结果可能与平台有关,所以结果不应该存储在模块的(与平台无关的)字节码形式中; 然而,可执行文件创建时的平台与可执行文件运行时的平台相同。 注意 expr 仍然在运行时被评估;因此,要避免像 collection-path 这样的程序,它依赖于源安装, 而要使用相对路径和 (list 'lib str ...+) 这样的形式。

如果一个路径只在某些平台上需要,而在其他平台上不需要, 可以使用 define-runtime-path-list 和一个 expr , 在不需要该路径的平台上产生一个空列表。

请注意,如果 expr 在创建可执行文件时产生了一个目录的路径, 那么该目录的全部内容(包括任何子目录)就会包含在可执行文件或最终的发行中。

还要注意的是,在 phase level 不是 0 的情况下, define-runtime-path 不能与可执行文件创建者正常合作。 要解决这个限制,可以把 define-runtime-path 放在一个单独的模块中 –也许是一个由 module 创建的 submodule –然后导出定义, 然后包含定义的模块可以被 require 到任何阶段级别。 在 0 以外的 phase level 中使用 define-runtime-path 会在扩展时记录一个警告。

  • 如果根据 syntax-source-module, 表单有一个源模块, 那么源位置的确定是通过将原始表达式保留为一个语法对象, 在运行时提取其源模块路径(同样使用 syntax-source-module), 然后解析得到的模块路径索引。 请注意, syntax-source-module 是基于语法对象的 lexical information, 而不是其源位置。

  • 如果表达式没有源模块,则使用与表单相关的 syntax-source 位置,如果是一个字符串或路径。

  • 如果没有源模块,并且 syntax-source 没有产生路径,那么如果不是 #f, 则使用 current-load-relative-directory。 最后,如果其他都失败了,就使用 current-directory 。

在后两种情况下,路径通常以(平台特定的)字节形式保存, 但如果包围的路径对应于 collection-file-path 的结果,那么路径将被记录为相对于相应模块的路径。

syntax

(define-runtime-paths (id ...) maybe-runtime?-id expr)

与 define-runtime-path 类似,但一次声明并绑定多个路径。 expr 应该产生和 id 。

syntax

(define-runtime-path-list id maybe-runtime?-id expr)

与 define-runtime-path 类似,但 expr 应产生一个路径列表。

syntax

(define-runtime-module-path-index id maybe-runtime?-id module-path-expr)

与 define-runtime-path 类似,但 id 被绑定到一个 module path index 上, 该索引封装了 module-path-expr 相对于包围模块的结果。

使用 define-runtime-module-path-index 来绑定一个模块路径, 该路径被传递给像 dynamic-require 这样的反射函数,同时也为构建和分发可执行文件创建一个模块依赖。

syntax

(runtime-require module-path)

与 define-runtime-module-path-index 类似,但在不绑定模块路径索引的情况下创建分发依赖关系。 当 runtime-require 在一个具有相同 module-path 的模块中被多次使用时, 除了第一次使用外,其他都会扩展到一个空的 begin 。

syntax

(define-runtime-module-path id module-path)

与 define-runtime-path 类似,但 id 被绑定到一个 resolved module path 。 id 的 resolved module path 对应于 module-path (与 require 的模块路径的语法相同),它可以是相对于包围模块的。

通常倾向于使用 define-runtime-module-path-index 形式,因为它与被引用的模块建立了一个较弱的链接。 与 define-runtime-module-path-index 不同, define-runtime-module-path 形式创建了一个从封闭模块到 module-path 的 for-label 依赖关系。 由于这个依赖关系仅仅是 for-label ,当包围模块被 instantiate 或 visit 时, module-path 不会被 instantiate 或 visit (除非这种依赖关系是由其他 require 创建的),但是当包围模块被加载时,被引用的模块的代码会被加载。

syntax

(runtime-paths module-path)

这种形式主要是为诸如可执行文件生成器等工具所使用。 它扩展为一个带引号的列表,包含由 module-path 声明的运行时路径,返回声明 expr 的编译时结果,只是路径被转换为字节串。 包围的模块必须要求(直接或间接)由 module-path 指定的模块,这是一个没有引号的模块路径。 产生的列表 不 包括通过 define-runtime-module-path 绑定的模块路径。

procedure

(fold-files proc    
  init-val    
  [start-path    
  follow-links?]) → any

(or/c (path? (or/c 'file 'dir 'link) any/c
        . -> . any/c)

从 start-path 开始遍历文件系统,对发现的每个文件、目录和链接调用 proc 。 如果 start-path 是 #f, 那么遍历就从 (current-directory) 开始。

每个文件、目录或链接都有三个参数来调用 proc 程序:

  • 如果 start-path 是 #f, 第一个参数是一个相对于当前目录的路径名字符串。 否则,第一个参数是一个以 start-path 开头的路径名。 因此,为 start-path 提供 (current-directory) 与提供 #f 是不同的, 因为 proc 在前一种情况下接收完整路径,在后一种情况下接收相对路径。 另一个区别是,当 start-path 为 #f 时, 不会为当前目录调用 proc 。

proc 参数的使用方式与 foldl 的过程参数类似,其结果被用作新的累积结果。 对于目录的情况有一个例外(当第二个参数是 'dir 时): 在这种情况下,过程可以返回两个值,第二个值表示递归扫描是否应该包括给定的目录。 如果它返回一个值,那么该目录就被扫描了。 在文件或链接的情况下(当第二个参数是 'file 或 'link 时),第二个值是允许的,但被忽略。

(call-with-atomic-output-file

在与 file 相同的目录下打开一个供写入的临时文件,调用 proc 来写入临时文件, 然后原子式地(除了在 Windows 上)移动临时文件以取代 file 。 在 Unix 和 Mac OS 上,移动只需使用 rename-file-or-directory , 如果提供了 rename-fail-handler ,则在 Windows 上使用 rename-file-or-directory ; 否则,在 Windows 上,移动使用额外的重命名步骤(见下文),以避免因 file 的并发读者而出现问题。

调用 proc 函数时有一个临时文件的输出端口,加上临时文件的路径。 proc 的结果是 call-with-atomic-output-file 的结果。

call-with-atomic-output-file 函数安排在异常情况下删除临时文件。

从 (find-system-path 'pref-file) 指定的文件中提取偏好值, 或者从 filename 指定的文件中提取偏好值,如果它被提供并且不是 #f 。 在前一种情况下,如果偏好文件不存在, get-preferences 会尝试读取一个 old preferences file, 然后读取配置目录中的 "racket-prefs.rktd" 文件(由 find-config-dir 报告),而不是。 如果这些文件都不存在,偏好集就是空的。

偏好文件应该包含一个用默认参数设置编写的符号-值列表。 以 racket:, mzscheme:, mred: 和 plt: 开头的任何字母大小写的键都保留给 Racket 实现者使用。 如果偏好文件不包含符号-值列表,则通过 log-error 记录错误,并调用 failure-thunk 。

get-preference 的结果是与 name 相关的值, 如果它存在于关联列表中,否则就是调用 failure-thunk 的结果。

偏好设置在调用 get-preference 时被缓存(弱), 使用 (path->complete-path filename) 作为缓存键。 如果 flush-mode 被提供为 #f, 则使用缓存而不是重新查阅偏好文件。 如果 flush-mode 被提供为 'timestamp (默认), 那么只有当文件的时间戳与上次读取文件的时间戳相同时才会使用缓存。 否则,该文件将被重新访问。

在 preferences-lock-file-mode 返回 'file-lock 的平台上, 当 use-lock? 为 true 时,preferences-file 的读取由一个锁来保护; 多个读者可以共享这个锁,但写入者只能使用这个锁。 如果因为锁不可用而无法读取偏好文件,则会在锁文件的路径上调用 lock-there ； 如果 lock-there 是 #f,则会引发一个异常。 在尝试 timeout-lock-there 之前, 默认的 lock-there 处理程序重试了大约 5 次(每次尝试之间的延迟不断增加), 默认的 timeout-lock-there 会触发一个异常。

Old preferences files: 当没有提供 filename 且 (find-system-path 'pref-file) 指示的文件不存在时, 为了与旧版本的 Racket 兼容,会检查以下路径:

  • Windows: (build-path (find-system-path 'pref-dir) 'up "PLT Scheme" "plt-prefs.ss")

  • Mac OS: (build-path (find-system-path 'pref-dir) "org.plt-scheme.prefs.ss")

  • Unix: (expand-user-path "~/.plt-scheme/plt-prefs.ss")

安装一组偏好值,并将所有当前值写入由 (find-system-path 'pref-file) 指定的偏好文件中, 如果提供了文件名而不是 #f,则为 filename 。

names 参数提供了偏好值的名称, vals 的长度必须与 names 相同。 vals 中的每个元素必须是一个内置数据类型的实例, 其 write 输出是 read 的(也就是说,在写入偏好时, print-unreadable 参数被设置为 #f )。

当前的偏好值在更新前从偏好文件中读取,并且在文件读取前开始持有一个写锁,持续到偏好文件被更新后。 该锁是通过与偏好文件在同一目录下的文件的存在来实现的;更多信息请参见 preferences-lock-file-mode 。 如果偏好设置文件的目录不存在,就会被创建。

(preferences-lock-file-mode) → (or/c 'exists 'file-lock)

报告锁文件在当前平台上用于实现首选项文件锁定的方式。

'exists 模式目前在所有平台上使用,除了 Windows。 在 'exists 模式下,锁文件的存在表明持有一个写锁, 而读者不需要锁(因为偏好文件是通过 rename-file-or-directory 原子更新的)。

目前在 Windows 上使用的是 'file-lock 模式。 在 'file-lock 模式下,锁文件上的共享锁和独占锁(在 port-try-file-lock? 的意义上)反映了偏好文件内容上的读者锁和写者锁。 (首选项文件本身不被锁定,因为锁会干扰通过 rename-file-or-directory 替换文件的工作)。

(make-handle-get-preference-locked delay

  filename 
  #:lock-there lock-there 
  #:max-delay max-delay]) 
 → (path-string? . -> . any)

procedure

(call-with-file-lock/timeout filename    
  kind    
  thunk    
  failure-thunk    
  [#:lock-file lock-file    
  #:delay delay    
  #:max-delay max-delay]) → any
  filename : (or/c path-string? #f)
  kind : (or/c 'shared 'exclusive)
  thunk : (-> any)
  failure-thunk : (-> any)
  lock-file : (or/c #f path-string?) = #f
  delay : (and/c real? (not/c negative?)) = 0.01
  max-delay : (and/c real? (not/c negative?)) = 0.2
为文件名 lock-file 获取一个锁,然后调用 thunk 。 filename 参数指定了一个文件路径前缀,当 lock-file 是 #f 时, 该参数仅用于生成锁定文件名。 具体来说,当 lock-file 为 #f 时, call-with-file-lock/timeout 使用 make-lock-file-name 来建立锁文件名。 如果锁文件还不存在,它就会被创建;注意,锁文件 不 会被 call-with-file-lock/timeout 删除。

当 thunk 返回时, call-with-file-lock/timeout 释放锁,返回 thunk 的结果。 call-with-file-lock/timeout 函数会在 delay 秒后重试, 并以指数后退的方式继续重试,直到延迟达到 max-delay 。 如果 call-with-file-lock/timeout 不能获得锁,则会在尾部位置调用 failure-thunk 。 kind 参数指定了锁是 'shared 还是 'exclusive , 就像 port-try-file-lock? 一样。

Examples:
> (call-with-file-lock/timeout filename 'exclusive
    (lambda () (printf "File is locked\n"))
    (lambda () (printf "Failed to obtain lock for file\n")))

File is locked

> (call-with-file-lock/timeout #f 'exclusive
    (lambda ()
      (call-with-file-lock/timeout filename 'shared
        (lambda () (printf "Shouldn't get here\n"))
        (lambda () (printf "Failed to obtain lock for file\n"))))
    (lambda () (printf "Shouldn't get here either\n"))
    #:lock-file (make-lock-file-name filename))
