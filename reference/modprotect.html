14.10 Code Inspectors

与检查器控制对结构字段的访问一样(见 结构检查器), 检查器也控制对 module bindings 的访问。 以这种方式使用的检查器是 code inspectors 。 module bindings 的默认代码检查器由 current-code-inspector 参数决定, 而不是 current-inspector 参数。

当 module 声明被评估时, current-code-inspector 参数的值与模块声明相关。 当模块通过 require 或 dynamic-require 被调用时, 模块声明时检查器的一个子检查器被创建,这个子检查器与模块的调用相关联。 任何控制该子检查器的检查器(包括声明时检查器及其上级)都会控制模块的调用。 特别是,如果 current-code-inspector 的值从未改变, 那么对任何模块的调用都不会失去控制,因为模块的调用与 current-code-inspector 的一个子检查器相关。

当控制模块调用的检查器与 current-code-inspector 一起安装时, 它可以在模块上使用 module->namespace , 并且可以通过 dynamic-require 来访问模块的 protected 出口(即那些从模块导出的标识符, 带有 protect-out )。 一个模块不能 require 一个具有较弱的声明时代码检查器的模块。

当一个 module 被扩展或一个 namespace 被创建时, current-code-inspector 的值与模块或命名空间的顶层 lexical information 相关。 具有该 lexical information 的语法对象可以访问该检查器控制的任何模块的受保护和未导出的绑定。 在 module 的情况下,检查器坚持使用这样的语法对象,即使该语法对象是在一个不太强大的上下文中用于扩展代码; 此外,如果语法对象是一个被编译为变量引用的标识符,检查器坚持使用该变量引用, 即使它出现在一个用较弱的检查器评估(即声明)的模块形式。 当语法对象或变量引用出现在被打印的编译代码中时(见 打印编译后的代码), 相关的检查器不会被保留。

当以打印形式的编译代码被 read 时,没有检查器与该代码相关联。 当代码被 eval 时, 被实例化的语法对象字头和模块变量引用会获得 current-code-inspector 的值作为它们的检查器。

当一个模块实例连接到多个 namespaces 时,每个命名空间都有自己的 module registry, 模块调用的检查器可以是注册表专用的。 可以通过 namespace-unprotect-module 改变特定模块注册表中的调用检查器 (但改变检查器需要对旧的检查器进行控制)。

Changed in version 8.1.0.8 of package base: 增加了对具有较弱代码检查器的模块的 require 的约束。

parameter

(current-code-inspector) → inspector?

(current-code-inspector insp) → void?
  insp : inspector?
一个决定检查器的 parameter ,以控制对模块绑定和重定义的访问。

如果代码检查器从原来的值中被改变,那么由默认的 compiled-load handler 加载的字节码将被标记为不可运行。
