6.7 对象和类的契约

为一个班级产生一个合同。

在 class/c 表格中列出的契约有两大类:外部契约和内部契约。 当一个对象从一个类中实例化时,或者当方法或字段通过该类的一个对象被访问时,外部契约管理行为。 当方法或字段在类的层次结构中被访问时,内部契约管理行为。 这种分离允许为类客户提供更强的契约,为子类提供更弱的契约。

方法契约必须包含一个额外的初始参数,它与方法的隐式 this 参数相对应。 这允许在方法被调用时讨论对象的状态的契约(或者,对于依赖性契约,在契约的其他部分)。 替代的契约形式,例如 ->m, 被提供作为编写方法契约的速记。

在 absent 子句中列出的方法和字段必须 不 存在于类中。

一个类的契约可以用 #:opaque 关键字指定为 opaque 。 一个不透明的类契约将只接受一个完全定义了契约所指定的外部方法和字段的类。 如果被契约的类包含任何未被指定的方法或字段,将会产生一个契约错误。 如果提供了#:ignore-local-member-names , 那么具有本地成员名的方法或字段(即用 define-local-member-name 定义的)在这个检查中会被忽略。

外部契约的内容如下:

  • 一个没有标签的外部方法契约描述了 method-id 的实现在向契约类的一个对象发送方法时的行为。 这个契约将继续在子类中被检查,直到契约类的实现不再是动态调度的入口。

    如果只有字段名,这相当于只坚持该方法在类中存在。

    Examples:
    (define woody%
      (class object%
        (define/public (draw who)
          (format "reach for the sky, ~a" who))
        (super-new)))
     
    (define/contract woody+c%
      (class/c [draw (->m symbol? string?)])
      woody%)

     

    > (send (new woody%) draw #f)

    "reach for the sky, #f"

    > (send (new woody+c%) draw 'zurg)

    "reach for the sky, zurg"

  • 一个外部字段契约,被标记为 field,描述了从类之外访问该字段所含值的行为。 由于字段可以被变异,这些契约在字段的任何外部访问(通过 get-field) 和外部变异(通过 set-field!) 中被检查。

    如果只有字段名,这等同于使用契约 any/c (但它被更有效地检查)。

    Examples:
    (define woody/hat%
      (class woody%
        (field [hat-location 'uninitialized])
        (define/public (lose-hat) (set! hat-location 'lost))
        (define/public (find-hat) (set! hat-location 'on-head))
        (super-new)))
    (define/contract woody/hat+c%
      (class/c [draw (->m symbol? string?)]
               [lose-hat (->m void?)]
               [find-hat (->m void?)]
               (field [hat-location (or/c 'on-head 'lost)]))
      woody/hat%)

  • 一个初始化参数契约,标记为 init,描述了在类实例化过程中与该名称配对的值的预期行为。 同一个名字可以被提供多次,在这种情况下, class/c 形式的第一个这样的契约被应用于初始化参数列表中第一个被标记为这个名字的值,以此类推。

    如果只有初始化参数的名字存在,这就相当于使用合约 any/c (但它被更有效地检查)。

    Examples:
    (define woody/init-hat%
      (class woody%
        (init init-hat-location)

  • 列在 init-field 部分的契约被视为每个契约出现在 init 部分和 field 部分。

内部契约限制了类和它们的子类之间的方法调用行为;这种调用不受上述类契约的控制。

与外部契约一样,当一个方法或字段的名字被指定,但没有出现契约时,契约仅仅满足于相应字段或方法的存在。

  • 被标记为 inherit 的方法契约描述了该方法在契约类的任何子类中直接(即通过 inherit) 被调用时的行为。 这个契约,就像外部方法契约一样,在契约类的方法实现不再是动态调度的入口之前一直适用。

    Examples:
    > (new (class woody+c%
             (inherit draw)
             (super-new)
             (printf "woody sez: “~a”\n" (draw "evil dr porkchop"))))

    woody sez: “reach for the sky, evil dr porkchop”

  • 一个标记为 super 的方法契约描述了子类中的 super 形式调用 method-id 时的行为。 这个契约只影响子类中的 super 调用,这些子类调用了契约类的 method-id 的实现。

    这个例子展示了如何扩展 draw 方法,如果它被传递了两个参数, 它就会结合两次对原始 draw 方法的调用,但有一个契约来控制 super 方法必须被调用的方式。

    Examples:
    (define/contract woody%+s
      (class/c (super [draw (->m symbol? string?)]))
      (class object%
        (define/public (draw who)
          (format "reach for the sky, ~a" who))
        (super-new)))
     
    (define woody2+c%
      (class woody%+s
        (define/override draw
          (case-lambda
            [(a) (super draw a)]
            [(a b) (string-append (super draw a)
                                  " and "
                                  (super draw b))]))
        (super-new)))

     

    > (send (new woody2+c%) draw 'evil-dr-porkchop  'zurg)

    "reach for the sky, evil-dr-porkchop and reach for the sky, zurg"

    > (send (new woody2+c%) draw "evil dr porkchop" "zurg")

      given: "evil dr porkchop"

           (super (draw (->m symbol? string?))))

      contract from: (definition woody%+s)

      contract on: woody%+s

    最后一次调用发出了一个错误信号,指责 woody2%+c , 因为没有检查初始的 draw 调用的契约,而且超调用违反了它的契约。

  • 一个标记为 inner 的方法契约描述了该类对子类中的增强方法的期望行为。 这个契约会影响到子类中 method-id 的任何实现, 这些方法可以通过契约类的 inner 被调用。 这意味着通过 augment 或 overment 实现 method-id 的子类会阻止未来的子类受到契约的影响,因为进一步的扩展不能通过契约类达到。

  • 一个被标记为 override 的方法契约描述了被契约的类在直接调用 (即被应用程序 (method-id ...) )时对 method-id 的预期行为。 只有在子类中覆盖该方法将改变动态调度链的入口点时,才能使用这种形式(即该方法从未被增强过)。

    这一次,我们不需要重写 draw 来支持两个参数,而是可以建立一个新的方法 draw2 , 它接收两个参数并调用 draw 。 我们还添加了一个契约,以确保重写 draw 不会破坏 draw2 。

    Examples:
    (define/contract woody2+override/c%
      (class/c (override [draw (->m symbol? string?)]))
      (class woody+c%
        (inherit draw)
        (define/public (draw2 a b)
          (string-append (draw a)
                         " and "
                         (draw b)))
        (super-new)))
     
    (define woody2+broken-draw
      (class woody2+override/c%
        (define/override (draw x)
          'not-a-string)
        (super-new)))

     

    > (send (new woody2+broken-draw) draw2
            'evil-dr-porkchop
            'zurg)

    draw: contract violation

      expected: string?

      given: 'not-a-string

      in: the range of

          the draw method in

          (class/c

           (override (draw (->m symbol? string?))))

      contract from:

          (definition woody2+override/c%)

      contract on: woody2+override/c%

      blaming: top-level

       (assuming the contract is correct)

  • 一个被标记为 augment 或 augride 的方法契约描述了当从子类中直接调用 method-id 时,由契约类提供的行为。 这些形式只能在该方法之前是可增强的情况下使用,这意味着任何增强或覆盖的实现都不会改变动态调度链的入口点。 augment 用于子类可以增强该方法,而 augride 用于子类可以覆盖当前的增强。

  • 用 inherit-field 标记的字段契约描述了在契约类的任何子类中直接 (即通过 inherit-field) 访问该字段所含值的行为。 由于字段可能被突变,这些契约在这些子类中发生的任何访问和/或字段的突变上被检查。

  • Changed in version 6.1.1.8 of package base: 如果提供了额外的关键字,不透明的 class/c 现在可以选择忽略本地成员名。

syntax

(absent absent-spec ...)

参见 class/c ;在 class/c 形式之外使用是一个语法错误。

syntax

(->m dom ... range)

与 -> 类似,除了产生的契约的域比声明的域多包含一个元素,其中第一个(隐含的)参数是用 any/c 契约的。 当不需要检查 this 的属性时,这种契约对于编写更简单的方法契约很有用。

syntax

(->*m (mandatory-dom ...) (optional-dom ...) rest range)

类似于 ->*, 除了产生的契约的强制域比声明的域多包含一个元素,其中第一个(隐含的)参数是用 any/c 契约的。 当 this 的属性不需要被检查时,这种契约对于编写更简单的方法契约很有用。

syntax

(case->m (-> dom ... rest range) ...)

类似于 case->, 除了产生的契约的每个 case 的强制域比声明的域多一个元素, 其中第一个(隐含的)参数是用 any/c 契约的。 当 this 的属性不需要被检查时,这种契约对于编写更简单的方法契约很有用。

syntax

(->dm (mandatory-dependent-dom ...)
      (optional-dependent-dom ...)
      dependent-rest
      pre-cond
      dep-range)
与 ->d 类似,只是所产生的契约的强制域比声明的域多了一个元素, 其中第一个(隐含的)参数是用 any/c 契约的。 此外, this 在契约的主体中被适当地绑定。 当 this 的属性不需要被检查时,这种契约对于编写更简单的方法契约很有用。

field-spec = field-id

生成一个对象的契约。

与老式的 object-contract 不同,但与 class/c 一样,允许任意的契约表达。 而且, object/c 的方法契约遵循 class/c 的契约。 用 object/c 包装的对象的行为就像它的类被用等价的 class/c 契约包装一样。

生成一个对象的契约。

每个方法的契约都与相应的函数契约具有相同的语义, 但是方法契约的语法必须直接写在对象契约的主体中–就像类定义中的方法使用与普通函数定义相同的语法, 但不能是任意的程序。 与 class/c 的方法契约不同,隐含的 this 参数不是契约的一部分。 为了允许在依赖性契约中使用 this , ->d 契约隐含地将 this 绑定到对象本身。
