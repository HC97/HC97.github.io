3.26 懒惰地导入模块: lazy-require

 (require racket/lazy-require) package: base
The bindings documented in this section are provided by the racket/lazy-require library, not racket/base or racket.

syntax

(lazy-require [module-path (fun-import ...)] ...)

 
fun-import = fun-id
  | (orig-fun-id fun-id)
将每个 fun-id 定义为一个函数,当被调用时, 动态地要求从 module-path 指定的模块中导出名为 orig-fun-id 的函数并以相同的参数调用它。 如果没有给出 orig-fun-id ,它默认为 fun-id。

如果包围的相对相位不是 0,那么 module-path 也会被放在一个子模块中 (在子模块的相位 0 处使用 define-runtime-module-path-index )。 引入的子模块的名称是 lazy-require-auxn-m, 其中 n 是相位数, m 是一个数字。

当使用一个懒惰要求的函数触发模块加载时, 它也触发了 register-external-module 的使用, 以声明一个间接的编译依赖(以防该函数在编译模块的过程中被使用)。

Examples:
> (lazy-require
    [racket/list (partition)])
> (partition even? '(1 2 3 4 5))

'(2 4)

'(1 3 5)

> (module hello racket/base
    (provide hello)
    (printf "starting hello server\n")
    (define (hello) (printf "hello!\n")))
> (lazy-require
    ['hello ([hello greet])])
> (greet)

starting hello server

hello!

syntax

(lazy-require-syntax [module-path (macro-import ...)] ...)

 
macro-import = macro-id
  | (orig-macro-id macro-id)
就像 lazy-require 一样,但对宏而言。 也就是说,它将每个 macro-id 定义为一个宏,当使用时, 动态地从给定的 module-path 加载宏的实现。 如果没有给出 orig-macro-id ,它默认为 macro-id。

在具有大型复杂宏的库的 实现 中使用 lazy-require-syntax , 以避免库的客户端对宏 "编译器 "的依赖。 请注意,只有具有特别大的编译时组件的宏(如 Typed Racket,它包括一个类型检查器和优化器) 才会从 lazy-require-syntax 中受益;一般的宏不会。

警告: lazy-require-syntax 破坏了 Racket 的模块加载器和链接器所依赖的不变性; 这些不变性通常确保宏产生的代码中的引用在代码运行前被加载。 安全使用 lazy-require-syntax 需要在宏的实现中采用一种特殊的结构。 (特别是,不能简单地在客户代码中引入 lazy-require-syntax )。 宏的实现必须遵循这些规则:

  1. 接口模块必须 require 运行时间支持模块

  2. 编译器模块必须通过 绝对 模块路径而不是 相对 路径来 require 运行时间支持模块

为了解释 "接口、编译器和运行时支持模块" 的概念,这里有一个导出宏的模块例子:
(module original racket/base
  (define (ntimes-proc n thunk)
    (for ([i (in-range n)]) (thunk)))
  (define-syntax-rule (ntimes n expr)
    (ntimes-proc n (lambda () expr)))
  (provide ntimes))
假设我们想使用 lazy-require-syntax 来懒散地加载 ntimes 宏转换器的实现。 原始模块必须分成三个部分:
(module runtime-support racket/base
  (define (ntimes-proc n thunk)
    (for ([i (in-range n)]) (thunk)))
  (provide ntimes-proc))
(module compiler racket/base
  (require 'runtime-support)
  (define-syntax-rule (ntimes n expr)
    (ntimes-proc n (lambda () expr)))
  (provide ntimes))
(module interface racket/base
  (require racket/lazy-require)
  (require 'runtime-support)
  (lazy-require-syntax ['compiler (ntimes)])
  (provide ntimes))
运行时支持模块包含了宏所指的函数和值的定义。 编译器模块包含宏定义本身–在编译后 "消失" 的那部分代码。 接口模块懒散地加载宏转换器,但它通过正常要求运行时支持模块来确保运行时的定义。 当然,在一个更大的例子中,运行时支持和编译器可能都由多个模块组成。

在这里,当我们不把运行时支持模块分离成一个单独的模块时会发生什么。

> (module bad-no-runtime racket/base
    (define (ntimes-proc n thunk)
      (for ([i (in-range n)]) (thunk)))
    (define-syntax-rule (ntimes n expr)
      (ntimes-proc n (lambda () expr)))
    (provide ntimes))
> (module bad-client racket/base
    (require racket/lazy-require)
    (lazy-require-syntax ['bad-no-runtime (ntimes)])
    (ntimes 3 (printf "hello?\n")))
> (require 'bad-client)

require: namespace mismatch;

 reference to a module that is not instantiated

  module: 'bad-no-runtime

  phase: 0

当接口模块没有引入对运行时支持模块的依赖时,会出现类似的错误。
