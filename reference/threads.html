11.1 线程

并发与同步 in Racket 指南 introduces threads.

关于 Racket 线程模型的基本信息,见 线程 。 另见 Futures 和 Places 。

当一个线程被创建时,它被置于 current custodian 的管理之下,并被添加到当前的 thread group 。 一个线程可以通过 thread-resume 添加任意数量的托管人管理者。 一个线程所做的分配被记入该线程的托管经理。 参见 custodian-limit-memory 的例子。

如果一个没有终止的线程无法到达并被暂停, 或者无法到达并通过 semaphore-wait, semaphore-wait/enable-break, channel-put, channel-get, sync, sync/enable-break 或 thread-wait 等函数只对无法到达的事件进行阻塞,那么它可以被垃圾回收(见 垃圾回收) 。 然而,要注意对 place-channel 阻塞的限制;见 Places 部分的 caveat 。

在GRacket中,当一个事件空间的处理线程的事件队列为空时,它将被封锁在一个内部信号上。 因此,当事件空间不可达时,处理程序线程是可收集的,并且不包含可见的窗口或运行的计时器。

一个线程可以被用作一个 synchronizable event (参见 事件)。 当 thread-wait 不会阻塞时,一个线程就 ready for synchronization ; 一个线程的同步结果就是 the synchronization result of a thread is the thread itself 本身。

11.1.1 创建线程

procedure

(thread thunk) → thread?

  thunk : (-> any)
在一个新的控制线程中调用 thunk ,没有参数。 thread 过程立即返回一个 thread descriptor 的值。 当调用 thunk 的过程返回时,为调用 thunk 而创建的线程就会终止。

procedure

(thread? v) → thread?

  v : any/c
如果 v 是一个 thread descriptor , 返回 #t ,否则返回 #f 。

procedure

(current-thread) → thread?

返回当前正在执行的线程的 thread descriptor 。

procedure

(thread/suspend-to-kill thunk) → thread?

  thunk : (-> any)
和 thread 一样,只是通过 kill-thread 或 custodian-shutdown-all 来 "杀死 "线程, 只是暂停该线程,而不是终止它。

procedure

(call-in-nested-thread thunk [cust]) → any

  thunk : (-> any)
  cust : custodian? = (current-custodian)
创建一个由 cust 管理的嵌套线程来执行 thunk 。 (嵌套线程的当前托管人是由创建线程继承的,与 cust 参数无关)。 当前线程会阻塞,直到 thunk 返回, 而 call-in-nested-thread 的调用结果就是 thunk 返回的结果。

嵌套线程的异常处理程序被初始化为一个过程,该过程跳转到线程的开头并将异常转移到原始线程。 处理程序因此终止了嵌套线程,并在原线程中重新引发了异常。

如果由 call-in-nested-thread 创建的线程在 thunk 返回之前死亡, exn:fail exception is raised 就会在原线程中被引发。 如果原始线程在 thunk 返回之前就被杀死了,那么嵌套线程就会排队休息。

如果在嵌套线程运行时,原线程(通过 break-thread)有一个中断排队,那么这个中断会被重定向到嵌套线程。 如果在创建嵌套线程时,一个中断已经在原线程上排队了,那么这个中断就会被移到嵌套线程上。 如果当嵌套线程完成时,一个断点仍然在嵌套线程上排队,那么断点会被移到原始线程上。

如果由 call-in-nested-thread 创建的线程在调用 call-in-nested-thread 时死亡, 那么调用 call-in-nested-thread 的外部调用会等待最里面的嵌套线程完成, 并且任何在内部线程上等待的中断会被转移到原始线程。

11.1.2 暂停、恢复和杀戮线程

procedure

(thread-suspend thd) → void?

  thd : thread?
如果 thd 正在运行,则立即暂停它的执行。 如果线程已经终止或已经被暂停, thread-suspend 不起作用。 线程保持挂起(即,它不执行),直到它被 thread-resume 恢复。 如果 current custodian 不单独管理 thd (即 thd 的某些托管人不是当前托管人或下属),将引发 exn:fail:contract exception is raised ,并且线程不会挂起。

procedure

(thread-resume thd [benefactor]) → void?

  thd : thread?
  benefactor : (or/c thread? custodian? #f) = #f
如果 thd 被暂停,并至少有一个托管人(可能是通过 benefactor ,如下文所述), 则恢复 thd 的执行。 如果线程已经终止,或者线程已经在运行并且没有提供 benefactor , 或者线程没有监护人并且 benefactor 没有提供,那么 thread-resume 将不起作用。 否则,如果提供了 benefactor ,它会触发多达三个额外的操:

  • 如果 benefactor 是一个线程,那么将来当它从挂起状态恢复时, thd 会恢复。 (恢复 thd 可能会触发之前通过 thread-resume 附加到 thd 的其他线程的恢复。)

  • 新的托管人可能会被添加到 thd 的经理集合中。 如果 benefactor 是一个线程,那么所有线程的托管人都被添加到 thd 中。 否则, benefactor 是一个托管人,它被添加到 thd (除非管理员已经关闭)。 如果 thd 由托管人和一个或多个下属管理,多余的下属从 thd 中移除。 如果 thd 被暂停,并添加托管,然后在添加后再恢复 thd 。

  • 如果 benefactor 是一个线程,当它在未来接收一个新的管理托管人时, 那么 thd 也会接收这个托管人。 (向 thd 添加保管人可能会触发将保管人添加到之前通过 thread-resume 附加到 thd 。)

procedure

(kill-thread thd) → void?

  thd : thread?
立即终止指定的线程,如果 thd 是使用 thread/suspend-to-kill 创建的,则挂起该线程。 终止主线程退出应用程序。 如果 thd 已经终止, kill-thread 什么也不做。 如果 current custodian 不管理 thd (并且它的下属都不管理 thd), 则会引发 exn:fail:contract exception is raised, 并且线程不会被杀死或挂起。

除非另有说明,球拍(和 GRacket)提供的程序是安全的和悬挂安全的; 也就是说,杀死或挂起一个线程永远不会干扰其他线程中过程的应用。 例如,如果线程在从输入端口提取字符时被杀死,则该字符要么完全被消耗,要么不被消耗,其他线程可以安全地使用该端口。

procedure

(break-thread thd [kind]) → void?

  thd : thread?
  kind : (or/c #f 'hang-up 'terminate) = #f
Registers 指定线程的中断,其中 kind 可选地指示要注册的中断类型。 如果在 thd 中禁用了换行符,则该换行符将被忽略,直到重新启用换行符(参见 中断)。

procedure

(sleep [secs]) → void?

  secs : (>=/c 0) = 0
使当前线程休眠,直到它开始休眠后至少经过 secs 秒。 secs 的零值只是作为允许其他线程执行的提示。 secs 的值可以是非整数,以请求任意精度的睡眠持续时间;实际睡眠时间的精度未知。

procedure

(thread-running? thd) → any

  thd : thread?
如果 thd 未终止且未挂起则返回 #t ,否则返回 #f 。

procedure

(thread-dead? thd) → any

  thd : thread?
如果 thd 结束,则返回 #t ,否则返回 #f 。

11.1.3 线程状态同步

procedure

(thread-wait thd) → void?

  thd : thread?
阻塞当前线程的执行,直到 thd 结束。 注意: (thread-wait (current-thread)) 会死锁当前线程, 但是如果启用了 break,并且线程是主线程或以其他方式可访问,则 break 可以结束死锁;参见 中断 。

除非 thd 是与 thread/suspend-to-kill 一起创建的,否则即使 thd 无法访问, (thread-wait thd) 也可能继续,因为 custodian 关闭可能会终止线程。 因此,使用 thread-wait 的线程阻塞通常不能被垃圾收集(参见 垃圾回收)。 然而,作为一种特殊情况,如果 thd 是当前线程, (thread-wait thd) 会阻塞而不阻止线程的垃圾收集,因为只有当一个 break 从等待中逃脱时,线程才能继续。

procedure

(thread-dead-evt thd) → evt?

  thd : thread?
当且仅当 thd 终止时, 返回一个 ready for synchronization 的 synchronizable event (参见 事件) 。 然而,与直接使用 thd 不同的是, 保留对事件的引用并不会阻止 thd 被垃圾收集(参见 垃圾回收)。 The synchronization result of a thread-dead event is the thread-dead event itself.

等待 (thread-dead-evt thd) 结果的线程本身通常不能被垃圾回收, 除非 thd 是通过 thread/suspend-to-kill 创建的, 与通过 thread-wait 等待相同。 但是,等待 (thread-dead-evt thd) 的结果没有特殊情况,其中 thd 是当前线程。

对于给定的 thd, thread-dead-evt 总是返回相同的(即 eq?) 结果。

procedure

(thread-resume-evt thd) → evt?

  thd : thread?
返回一个 synchronizable event (见 事件) , 当 thd 运行时,它将成为 ready for synchronization 。 (如果 thd 已经终止,该事件永远不会准备好)。 如果 thd 运行并在调用 thread-resume-evt 后被暂停,那么结果事件仍然是就绪的; 每次暂停 thd 后,都会产生一个新的事件,由 thread-resume-evt 返回。 该事件的结果是 thd ,但如果 thd 从未被恢复, 那么对该事件的引用并不妨碍 thd 被垃圾回收(参见 垃圾回收)。

procedure

(thread-suspend-evt thd) → evt?

  thd : thread?
返回一个 synchronizable event (见 事件) , 当 thd 被暂停时,该事件将成为 ready for synchronization 。 (如果 thd 已经终止,该事件将永远不会解封)。 如果 thd 被暂停,然后在调用 thread-suspend-evt 后恢复,那么结果事件仍然是就绪的; thd 的每次恢复都会创建一个新的事件,由 thread-suspend-evt 返回。 该事件的结果是 thd , 但如果 thd 是用 thread (而不是 thread/suspend-to-kill) 创建的, 并且从未被恢复,那么对该事件的引用不会阻止 thd 被垃圾收集(参见 垃圾回收)。

如果 thd 是用 thread/suspend-to-kill 创建的, 那么等待 (thread-suspend-evt thd) 会阻止等待中的线程被垃圾收集,这与 (thread-dead-evt another-thd) 对通过 thread 创建的 another-thd 的作用相同。 此外,由于事件的结果是 thd, 等待 (thread-suspend-evt thd) 可以防止 thd 的垃圾回收。

11.1.4 线程邮箱

每个线程都有一个 mailbox ,它可以通过这个邮箱接收任意的消息。 换句话说,每个线程都有一个内置的异步通道。

See also 缓冲的异步通道.

procedure

(thread-send thd v [fail-thunk]) → any

  thd : thread?
  v : any/c
  fail-thunk : (or/c (-> any) #f)
   = (lambda () (raise-mismatch-error ....))
将 v 作为消息排到 thd ,而不阻塞。 如果消息被排队,结果是 #<void> 。 如果 thd 在消息被排队之前停止运行–如 thread-running?—那么如果它是一个产生结果的过程, 就会调用 fail-thunk (通过尾部调用), 如果 fail-thunk 是 #f ,则返回 #f 。

procedure

(thread-receive) → any/c

接收并取消为当前线程排定的消息,如果有的话。 如果没有可用的消息, thread-receive 会阻塞,直到有可用的消息。

procedure

(thread-try-receive) → any/c

如果有的话,接收并取消对当前线程排队的消息,如果没有消息,则立即返回 #f 。

procedure

(thread-receive-evt) → evt?

返回一个恒定的 synchronizable event (见 事件) ,当同步线程有消息要接收时,该事件将成为 ready for synchronization 。 The synchronization result of a thread-receive event is the thread-receive event itself.

procedure

(thread-rewind-receive lst) → void?

  lst : list?
将 lst 中的元素推回到当前线程队列的前面。 这些元素被逐一推送,因此第一个可用的消息是 lst 的最后一个元素。
