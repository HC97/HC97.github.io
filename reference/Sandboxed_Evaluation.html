14.12 Sandboxed Evaluation

 (require racket/sandbox) package: sandbox-lib
The bindings documented in this section are provided by the racket/sandbox library, not racket/base or racket.

racket/sandbox 模块提供了创建 "沙盒" 评估器的工具, 它以一种特殊的方式配置,可以限制资源(内存和时间),文件系统和网络访问,以及更多。 沙盒化的评估器可以通过许多参数进行配置–而且默认值是为沙盒非常有限的常见使用情况而设置的。

procedure

(make-evaluator 
  language 
  input-program ... 
  [#:requires requires 
  #:allow-for-require allow-for-require 
  #:allow-for-load allow-for-load 
  #:allow-read allow-read 
  #:allow-syntactic-requires allow-syntactic-requires])

  language : 
(or/c module-path?
      (list/c 'special symbol?)
      (cons/c 'begin list?))
  input-program : any/c
  requires : 
(listof (or/c module-path? path-string?
              (cons/c 'for-syntax (listof module-path?))))
   = null

  allow-for-load : (listof path-string?) = null
  allow-read : (listof (or/c module-path? path-string?)) = null
  allow-syntactic-requires : (or/c #f (listof module-path?))
   = #f

  #:readers readers 
  #:allow-for-require allow-for-require 
  #:allow-for-load allow-for-load 
  #:allow-read allow-read 
  #:allow-syntactic-requires allow-syntactic-requires]) 
 → (any/c . -> . any)
  module-decl : (or/c syntax? pair? path? input-port? string? bytes?)
  lang : (or/c #f module-path?) = #f
  readers : (or/c #f (listof module-path?))
   = (and lang (default-language-readers lang))
  allow-for-require : (listof (or/c module-path? path?)) = null
  allow-for-load : (listof path-string?) = null
  allow-read : (listof (or/c module-path? path-string?)) = null
  allow-syntactic-requires : (or/c #f (listof module-path?))
   = #f
The make-evaluator 函数用 language 和 requires 规范创建一个评估器, 并开始评估给定的 input-program 。 make-module-evaluator 函数创建一个评估器,在给定的模块背景下工作。 这两种情况下的结果都是一个用于进一步评估的函数。

返回的评估器在一个孤立的和有限的环境中工作。 特别是,文件系统的访问受到限制,这可能会干扰到使用文件系统中不在 collection 。 关于 allow-for-require,allow-for-load 和 allow-read 参数的信息见下文; 基于集合的模块文件通常不需要包括在这些列表中。 当 language 是一个模块路径或提供 requires 时, 指定的模块会被隐含地包括在 allow-for-require 列表中。 当 allow-syntactic-requires 不是 #f 时, 它限制了可以在模块中直接引用的模块集合;更多信息见下文。 (为了向后兼容,非 module-path? 的字符串在 requires 这样的参数中是允许的;在加入 allow-for-require 之前,它们被隐含地转换为路径。)

每个 input-program 或 module-decl 参数提供以下形式的程序:

  • 一个用于读取程序的输入端口;

  • 一个持有完整输入的字符串或字节串;

  • 一个路径,用来命名存放输入的文件;或

  • 一个S-表达式或一个 syntax object, 它被评估为 eval (参见 get-uncovered-expressions)。

在上述前三种情况下,使用 sandbox-reader 程序,启用行数以获得合理的错误信息, 并以 'program 为源(用于测试覆盖率)。 在最后一种情况下,预计输入是完整的程序,并被转换为一个 syntax object (使用 'program 作为源),除非它已经是一个 syntax object 。

返回的评估器函数接受额外的表达式(每次被调用时),其形式基本相同: 容纳表达式序列的字符串或字节串、容纳表达式的文件路径、S 表达式或 syntax object 。 如果评估器收到一个 eof 值,它就会被终止,此后会引发错误。也请参见 kill-evaluator , 它终止评估器而不引发异常。

对于 make-evaluator ,多个 input-program 被有效地串联起来,形成一个单一的程序。 input-program 的评估方式取决于 language 参数:

  • language 参数可以是一个模块路径(即符合 require 的 module-path 语法的数据)。

    在这种情况下, input-program 被自动包装在一个 module 中, 结果评估器在结果模块的命名空间内工作。

  • language 参数可以是一个以 'special 开头的列表,它表示一种具有特殊输入配置的内置语言。 可能的值是 '(special r5rs) 或表示教学语言的值: '(special beginner), '(special beginner-abbr), '(special intermediate), '(special intermediate-lambda) 或 '(special advanced) 。

    在这种情况下, input-program 被自动包装在一个 module 中, 所产生的评估器在所产生的模块的命名空间内工作。 此外,某些参数(如如 read-accept-infix-dot) 被设置为自定义从字符串和端口读取程序。

    这个选项主要是为老的测试系统提供的。 使用 make-module-evaluator ,输入以 #lang 开头,一般来说会更好。

  • 最后, language 可以是一个列表,其第一个元素是 'begin 。

    在这种情况下,使用 sandbox-namespace-specs 创建一个新的命名空间, 它默认使用 sandbox-make-namespace 创建一个新的命名空间 (反过来,它使用 make-base-namespace 或 make-gui-namespace , 取决于 sandbox-gui-available 和 gui-available?) 。

    在新的命名空间中, language 被评估为一个表达式,以进一步初始化命名空间。

requires 列表为 input-program 的模块或命名空间增加了额外的导入, 即使在 require 没有通过 language 提供的情况下。 allow-syntactic-requires 参数,如果不是 #f, 当 language 参数意味着一个 module 包装器时, 它限制了在模块中扩展的 require 引用;更确切地说, 它限制了在向 module name resolver 提供语法对象时可以解析的模块路径, 这将包括由宏扩展创建的 require 形式。 使用 submod 后面的 "." 或 ".." 的相对子模块路径总是被允许的。

下面的例子说明了将程序放入模块的评估器和仅仅初始化顶层命名空间的评估器之间的区别:

> (define base-module-eval
    ; a module cannot have free variables...
    (make-evaluator 'racket/base '(define (f) later)))

program:1:0: later: unbound identifier

  in: later

> (define base-module-eval
    (make-evaluator 'racket/base '(define (f) later)
                                 '(define later 5)))
> (base-module-eval '(f))

5

> (define base-top-eval
    ; non-module code can have free variables:
    (make-evaluator '(begin) '(define (f) later)))
> (base-top-eval '(+ 1 2))

3

> (base-top-eval '(define later 5))
> (base-top-eval '(f))

5

make-module-evaluator 函数本质上是对 make-evaluator r的限制, 其中程序必须是一个模块,所有导入都是程序的一部分。 在某些情况下,限制程序成为一个模块,在其语言位置使用一个特定的模块是很有用的; 使用可选的 lang 参数来指定这样的限制,其中 #f 表示不执行限制。 readers 参数同样限制了可以跟随 #lang 或 #reader 的路径, 如果它不是 #f 的话,默认是基于 lang 。 allow-syntactic-requires 参数的处理方式与 module-wrapper 情况下的 make-evaluator 相同。

当程序被指定为一个路径时,那么这个路径就被隐含地加入到 allow-for-load 列表中。

(define base-module-eval2
  ; equivalent to base-module-eval:
  (make-module-evaluator '(module m racket/base
                            (define (f) later)
                            (define later 5))))

make-module-evaluator 函数可以方便地测试模块文件: 传入一个文件名的路径值,你会得到一个模块上下文中的评估器,你可以用你喜欢的测试工具来使用。

在所有情况下,评估器在一个孤立和有限的环境中运行:

  • 它使用一个新的托管人和命名空间。 当 gui-available? 和 sandbox-gui-available 产生 true 时,它也在自己的事件空间中运行。

  • 评估器在 sandbox-security-guard 下工作,它限制了文件系统和网络访问。

  • 评估器包含在一个内存受限的环境中, 每个评估都被包裹在一个 call-with-limits 中(当内存核算可用时); 另见 sandbox-memory-limit, sandbox-eval-limits 和 set-eval-limits 。

注意,这些限制也适用于沙盒环境的创建–因此,例如,如果创建沙盒所需的内存高于限制,那么 make-evaluator 将以内存限制异常而失败。

allow-for-require 和 allow-for-load 参数调整文件系统的权限,以扩大评估器可使用的文件集。 在一个集合中的模块可以自动访问,但是 allow-for-require 参数列出了额外的模块, 这些模块可以通过文件系统路径与它们的导入一起被 require 。 allow-for-load 参数也同样列出了可以被 load 的文件。 (require 和 load 所需的精确权限可能不同)。 allow-read 参数只是为了向后兼容; allow-read 的每个 module-path? 元素都被有效地移到 allow-for-require, 而其他元素被移到 allow-for-load 。

沙盒环境被很好地隔离了,评估器函数基本上是向它发送一个表达式并等待结果。 这种通信形式使得对单个评估器的嵌套(或并发)调用是不可能的。 通常这不是一个问题,但在某些情况下,你可以让评价器函数在沙箱代码内可用,比如说:
> (let ([e (make-evaluator 'racket/base)])
    (e `(,e 1)))

evaluator: nested evaluator call with: 1

在这种情况下,将发出一个错误信号。

如果在创建沙盒时, sandbox-propagate-exceptions 的值为true(默认值), 那么异常(包括语法和运行时)会像往常一样传播给评估函数的调用者(即用 with-handlers 捕捉它们)。 关于直接使用引发的异常,请看下面的注意事项。 如果在创建沙盒时, sandbox-propagate-exceptions 的值是 #f , 那么在沙盒评估中未捕获的异常会导致错误被打印到沙盒的错误端口,而评估的调用者会收到 #<void> 。

在使用从沙盒返回的值或由沙盒引发的异常时要注意。 该值可能是一个冒名顶替者,也可能是一个结构类型重定向平等比较或打印操作的结构。 为了安全地处理由沙盒产生的未知值,请在沙盒中操作它,可能使用 call-in-sandbox-context 。

评估器一次只能由一个线程使用,检测到的并发使用会触发一个异常。 小心在非主线程中使用评估器,因为 sandbox-make-plumber 的默认值在当前管道中注册了一个回调, 以冲刷评估器的管道,这意味着冲刷当前管道(比如Racket进程即将退出时)意味着使用评估器。

Changed in version 1.2 of package sandbox-lib: 增加了 #:readers 和 #:allow-syntactic-require 参数。

14.12.1 安全方面的考虑

尽管沙盒的设计是为执行 Racket 程序提供一个安全的环境,限制对系统资源的访问,但在沙盒中执行不受信任的程序仍有一定的风险。 因为恶意程序可以从 Racket 运行时和已安装的程序集中行使任意功能,发现 Racket 或已安装的程序集的漏洞的攻击者可能会逃脱沙箱。

为了减轻这种风险,使用沙盒的程序应该尽可能地采取额外的预防措施。 建议的措施包括:

  • 为 make-evaluator 或 make-module-evaluator 提供一个自定义的模块语言, 让不受信任的代码只访问它绝对需要的语言结构。

  • 如果不受信任的代码需要访问已安装的集合,只安装你的程序需要的集合。

  • 使用操作系统级的安全功能来提供深度防御,以防运行沙盒的进程被破坏。

  • 确保你的Racket安装和安装的软件包是最新的版本。

14.12.2 自定义评估器

make-evaluator 创建的沙盒评估器可以通过许多参数进行定制。 大多数配置参数会影响新创建的评估器;改变它们对已经运行的评估器没有影响。

默认的配置选项是为一个非常有限的沙盒环境设置的–一个可以安全地公开使用的环境。 如果需要更多的权限,或者你想要更严格的限制,可能需要进一步定制。 定制评估器的另一个有用的方法是,从一个相对不受限制的配置开始,添加所需的限制。 这种方法是由 call-with-trusted-sandbox-configuration 函数实现的。

沙盒环境使用两个概念来限制评价所需的时间: shallow time 和 deep time 。 Shallow time 指的是表达式的立即执行。 例如,5 秒的 shallow time 限制将限制 (sleep 6) 和其他需要超过 5 秒的计算。 Deep time 指的是表达式以及表达式所创建的所有线程和子进程的总执行。 例如,5秒的 deep time 限制会限制 (thread (λ () (sleep 6))), 而 shallow time 不会, 以及 shallow time 会限制的所有表达。 默认情况下,大多数沙盒只限制 shallow time ,以方便使用线程的表达式。

procedure

(call-with-trusted-sandbox-configuration thunk) → any

  thunk : (-> any)
在沙盒配置参数被设置为最小限制的情况下调用 thunk 。 更确切地说,没有内存或时间限制,并使用现有的 inspectors, security guard, exit handler, logger, plumber 和 environment variable set 。(注意,I/O 端口的设置并不包括在内)。

(sandbox-init-hook) → (-> any)

一个 parameter ,决定了为初始化一个新的评估器所要调用的 thunk。 该钩子在程序在新创建的评估器上下文中被评估之前被调用。 它可以用来设置与读、写、评估等有关的环境参数。 某些语言 ('(special r5rs) 有特定于该语言的初始化;钩子在初始化之后使用,所以它可以覆盖设置。

(sandbox-path-permissions)

 →

一个 parameter ,通过列出允许的路径和访问模式来配置默认沙盒安全防护的行为。 这个参数的内容是一个规格列表,每个规格都是一个访问模式和被授予这种访问权的路径的字节重合表达式。

访问模式的符号是以下之一: 'execute, 'write, 'delete, 'read 或 'exists 。 这些符号的顺序是递减的:每个符号也意味着对以下模式的访问(例如, 'read 允许读取或检查是否存在)。

一个额外的模式符号 'read-bytecode 不属于这些模式的线性顺序。 指定这个模式与指定 'read 相似,但它不被任何其他模式所暗示。 (例如,即使你为某个路径指定了 'write ,你也需要指定 'read-bytecode 来授予这个权限)。 沙盒通常在下层代码检查器(见 sandbox-make-code-inspector )的背景下工作, 它可以防止加载不受信任的字节码文件–沙盒被设置为允许从指定为 'read-bytecode 的文件中加载字节码。 该规范默认为 Racket 集合层次结构(包括用户特定的库)和在 #:allow-read 参数中明确指定的库。 (注意,这只适用于加载字节码文件,在较低的代码检查器下,仍然无法使用受保护的模块绑定(见 Code Inspectors)。)

默认值是 null,但是当一个评估器被创建时,它被 'read-bytecode 权限所增强, 使其有可能使用集合库(包括 sandbox-override-collection-paths)。 更多信息见 make-evaluator 。

parameter

(sandbox-memory-limit) → (or/c (>=/c 0) #f)

(sandbox-memory-limit limit) → void?
  limit : (or/c (>=/c 0) #f)
一个 parameter ,它决定了沙盒的总内存限制,单位是兆字节(它可以容纳一个有理数或一个浮点数)。 当超过这个限制时,沙盒被终止。这个值在沙盒创建时使用,此后不能再更改。 它的默认值是30MB。参见 sandbox-eval-limits ,了解每个评价的限制,以及这两个限制如何协同工作的描述。

注意,(当内存核算被启用时)内存被归属于引用它的最高托管人。 这意味着,如果你检查沙盒评估在沙盒外返回的值,你自己的监护人将被收取费用。 为了确保它被计入沙盒,你应该在代码完成检查后删除对这种值的引用。

这个策略对沙盒内存限制与 sandbox-eval-limits 指定的每个表达式限制的交互方式有影响: 可从沙盒以及交互中获得的值将计入沙盒限制。 例如,在这段代码的最后一个交互中
(define e (make-evaluator 'racket/base))
(e '(define a 1))
(e '(for ([i (in-range 20)]) (set! a (cons (make-bytes 500000) a))))
内存块是在交互限制内分配的,但由于它们是与定义的变量连锁的,它们也可以从沙盒中到达–所以它们将被计入沙盒内存限制, 但不计入交互限制(更确切地说,没有超过一个块被计入交互限制)。

parameter

(sandbox-eval-limits) → 
(or/c (list/c (or/c (>=/c 0) #f)
              (or/c (>=/c 0) #f))
      #f)
(sandbox-eval-limits limits) → void?
  limits : 
(or/c (list/c (or/c (>=/c 0) #f)
              (or/c (>=/c 0) #f))
      #f)
一个 parameter ,决定了 每次 使用 make-evaluator 函数的默认限制,包括对输入程序的初始评估。 它的值应该是两个数字的列表;其中第一个是以秒为单位的 shallow time 值, 第二个是以兆字节为单位的内存限制(注意,它们不一定是整数)。 任何一个都可以是 #f ,用于禁用相应的限制;另外,该参数可以设置为 #f , 用于禁用所有的按评价的限制(在未来的版本中出现更多的限制种类时很有用)。默认是 (list 30 20) 。

注意,这些限制也适用于沙盒环境的创建–如果限制足够严格,甚至 (make-evaluator 'racket/base) 也会失败。 比如说、
(parameterize ([sandbox-eval-limits '(0.25 5)])
  (make-evaluator 'racket/base '(sleep 2)))
会抛出一个错误而不是创建一个评估器。因此,为了避免意外,你需要捕捉沙盒创建时发生的错误。

当限制被设置时, call-with-limits (见下文)被包裹在评价器的每次使用中,所以消耗过多的时间或内存会导致异常。 使用 set-eval-limits 来改变一个正在运行的评估器的限制。

一个托管人的限制只在垃圾回收后被检查,除了在某些单独大于托管人限制的大型分配中也可能被检查。

这个参数指定的内存限制适用于每个单独的评估,但不适用于整个沙盒–该限制通过 sandbox-memory-limit 指定。 当超过全局限制时,沙盒会被终止,但当超过每个评估的限制时,会引发 exn:fail:resource? 。 例如,假设你评估一个表达式,如
(for ([i (in-range 1000)])
  (set! a (cons (make-bytes 1000000) a))
  (collect-garbage))
然后,假设有足够小的限制,

  • 如果设置了全局限制,但没有设置每个评价的限制,那么沙盒最终将被终止,不可能再进行评价;

  • 如果有每个评价的限制,但是没有全局的限制,那么评价将以错误的方式中止,并且可以再次使用–具体来说, a 仍然持有一些块,你可以再次评价同一个表达式,这将为它增加更多的块;

  • 如果两个限制都设置了,全局的限制大于每个评价的限制,那么评价将中止,你可以重复评价, 但是这样做几次最终会终止沙盒(这将由错误信息和 evaluator-alive? 谓词表示)。

14.12.3 与评价器交互

除了使用沙盒评估器评估代码外,以下函数还用于与沙盒评估器进行交互。

procedure

(evaluator-alive? evaluator) → boolean?

  evaluator : (any/c . -> . any)
确定评价器是否还活着。

procedure

(kill-evaluator evaluator) → void?

  evaluator : (any/c . -> . any)
通过关闭评价器的监护人来释放由 evaluator 持有的资源。 试图在杀死后使用一个评估器会引发一个异常,试图杀死一个死的评估器会被忽略。

杀死一个评价器类似于向评价器发送一个 eof 值,只是 eof 值会立即引发一个错误。

procedure

(break-evaluator evaluator) → void?

  evaluator : (any/c . -> . any)
向正在运行的评估器发送一个中断。 其效果就像在评估器当前执行时输入 Ctrl-C 一样,将中断传播到评估器的上下文。

procedure

(get-user-custodian evaluator) → void?

  evaluator : (any/c . -> . any)
检索 evaluator 的顶层监护人。 这将返回一个与 (evaluator '(current-custodian)) 或 (call-in-sandbox-context evaluator current-custodian) 不同的值 –每个沙盒交互都被包裹在自己的监护器中,这就是这些将返回的值。

(这个监护人的一个用途是与 current-memory-use, 每个交互的子监护人将不会被收取整个沙箱的内存)。

procedure

(set-eval-limits evaluator secs mb) → void?

  evaluator : (any/c . -> . any)
  secs : (or/c exact-nonnegative-integer? #f)
  mb : (or/c exact-nonnegative-integer? #f)
将 evaluator 使用的每个表达式的限制改为 secs 秒的 shallow time 和 mb 兆字节(任何一个都可以是 #f, 表示没有限制)。

这个过程应该用来修改现有的评估器限制,因为改变 sandbox-eval-limits 参数并不影响现有的评估器。 参见 call-with-limits 。

procedure

(set-eval-handler evaluator handler) → void?

  evaluator : (any/c . -> . any)
  handler : (or/c #f ((-> any) . -> . any))
改变 evaluator 在每次交互中使用的每个表达式处理程序。 一个 #f 值意味着不使用处理程序。

这个过程应该被用来修改现有的评价器处理程序,因为改变 sandbox-eval-handlers 参数不会影响现有的评价器。 参见 call-with-custodian-shutdown 和 call-with-killing-threads 提供的两个有用的处理程序。

procedure

(call-with-custodian-shutdown thunk) → any

  thunk : (-> any)
(call-with-killing-threads thunk) → any
  thunk : (-> any)
这些函数对于作为一个评估处理程序是很有用的。 call-with-custodian-shutdown 将在一个新的托管机构中执行 thunk , 然后关闭该托管机构,确保 thunk 不可能留下任何资源。 call-with-killing-threads 与此类似,只是它杀死了被留下的线程,但其他资源则保持原样。

  evaluator : (any/c . -> . any)
  i/o : (or/c bytes? string? eof-object?)
如果在创建评估器时 (sandbox-input) 是 'pipe , 那么这个过程可以用来检索管道的输出端口端(当使用时没有参数),或者将一个字符串或一个字节字符串添加到管道中。 它还可以和 eof 一起使用,关闭管道。

procedure

(get-output evaluator) → (or/c #f input-port? bytes? string?)

  evaluator : (any/c . -> . any)
(get-error-output evaluator)
 → (or/c #f input-port? bytes? string?)
  evaluator : (any/c . -> . any)
返回 evaluator 的输出或错误输出, 其方式取决于创建评估器时的 (sandbox-output) 或 (sandbox-error-output) 的设置:

  • 如果是 'pipe, 那么 get-output 就会返回创建的管道的输入端口端;

  • 如果是 'bytes 或 'string, 那么结果是累积的输出,并且输出端口被重置, 所以每次调用都会返回评估器的不同输出 (注意,结果只在评估器终止之前可用,并且任何输出的分配都受到沙盒内存限制的影响)

  • 否则,它将返回 #f 。

procedure

(get-uncovered-expressions evaluator    
  [prog?    
  src]) → (listof syntax?)
  evaluator : (any/c . -> . any)
  prog? : any/c = #t
  src : any/c = default-src
从评估器中检索未覆盖的表达式,只要在创建评估器时 sandbox-coverage-enabled 参数的值为真。 否则,会产生一个异常,表明没有覆盖信息可用。

prog? 参数指定了是获取只对原始输入程序进行评估后发现的表达式 (#t) ,还是评估器的所有后续使用 (#f)。 使用 #t 检索的是一个在输入程序被评估后、评估器被使用前保存的列表,所以结果总是一样的。

prog? 的 #t 值对测试学生程序很有用,可以发现提交的程序是否有足够的测试覆盖率。 #f 值对编写程序的测试套件很有用,可以确保你的测试覆盖整个代码。

第二个可选参数, src, 指定对结果进行过滤, 只保留来源与 src 匹配的 syntax objects 。 默认的是程序代码中使用的源码,如果有的话。 注意,如果输入的程序是以 S 表达式或字符串的形式给出的, 那么 'program 将被用作源值(在这些情况下,它将成为过滤的默认值)。 如果给出 #f, 结果是未经过滤的表达式列表。

产生的 syntax objects 列表在每个位置和跨度上最多有一个表达式。 因此,内容可能是不可靠的,但位置信息是可靠的(也就是说,它总是表示源代码,在 DrRacket 中使用覆盖信息时,会被涂成红色)。

注意,如果输入的程序是一连串的语法值,要么确保它们有 'program 作为源字段,要么使用 src 参数。 使用S-表达式序列(而不是 syntax objects) 作为输入程序会导致不可靠的覆盖率结果, 因为每个表达式可能被分配到一个源位置。

procedure

(call-in-sandbox-context evaluator    
  thunk    
  [unrestricted?]) → any
  evaluator : (any/c . -> . any)
  thunk : (-> any)
  unrestricted? : boolean? = #f
在沙盒评估器的背景下调用给定的 thunk 。 该调用是在用于评估表达式的资源限制和评估处理程序下执行的,除非 unrestricted? 被指定为 true。

这个过程通常类似于 (evaluator (list thunk)), 只是它不依赖于基于 sexpr 的语法的常见含义,将列表表达式作为函数应用(这在所有语言中并不正确)。 请注意,这对元级操作(如命名空间操作)更有用,它并不打算作为安全评价的替代品(即照常使用沙箱评价器)。

此外,你可以通过使用你自己的权限来避免一些沙盒化的限制,比如说、
(let ([guard (current-security-guard)])
  (call-in-sandbox-context
    ev
    (lambda ()
      (parameterize ([current-security-guard guard])
        ; can access anything you want here
        (delete-file "/some/file")))))

value

gui? 的值不再由 racket/sandbox 本身使用。 相反, gui-available? 和 sandbox-gui-available 在沙盒评估器被创建时被检查。

procedure

(call-with-limits secs mb thunk) → any

  secs : (or/c exact-nonnegative-integer? #f)
  mb : (or/c exact-nonnegative-integer? #f)
  thunk : (-> any)
在有内存和时间限制的情况下执行给定的 thunk : 如果执行消耗的内存超过 mb 兆字节或超过 secs 的 shallow time 秒, 那么计算将被中止,并引发 exn:fail:resource? 。 否则,thunk 的结果会像往常一样被返回(一个值,多个值,或者一个异常)。 两个限制中的每一个都可以用 #f 来表示没有限制。 有关内存限制的信息,请参见 custodian-limit-memory 。

为了执行限制, thunk 会在一个新的线程中运行。 像往常一样,新线程开始时的参数值与调用 call-with-limits 的那条线程相同。 与往常 不 同的是,当 thunk 完成时, 用于运行 thunk 的线程的参数值会被复制回调用 call-with-limits 的线程。

根据 call-with-limits 的设置和 sandbox-eval-limits 的使用, 沙盒评估器使用 set-eval-limits: 每个表达式的评估都被保护起来,避免超时和内存问题。 直接使用 call-with-limits 只能限制整个测试会话,而不是每个表达式。

syntax

(with-limits sec-expr mb-expr body ...)

call-with-limits 的宏版本。

procedure

(call-with-deep-time-limit secs thunk) → any

  secs : exact-nonnegative-integer?
  thunk : (-> any)
在有 deep time 限制的情况下执行给定的 thunk ,并返回 thunk 生成的值。

给定的 thunk 会在一个新的线程中运行。 如果它出错或者线程终止时返回一个值,那么就会返回 (values) 。

Changed in version 1.1 of package sandbox-lib: 如果 thunk 正常完成,则修改为返回 thunk 的结果。
