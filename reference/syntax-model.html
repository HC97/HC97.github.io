1.2 语法模型

Racket 程序的语法是由以下部分定义的

  • read 通道,将字符流处理成一个 syntax object; 以及

  • expand 通道,它处理一个语法对象以产生一个完全解析的语法对象。

关于 read 通道的细节,请参见 阅读器 。 源代码通常是在 read-syntax 模式下读取的,它产生一个 syntax object。

expand 通道递归地处理 syntax object,产生一个完整的程序 parse 。 syntax object 中的 Binding 信息驱动 expansion 过程, 当 expansion 过程遇到 binding 形式时,它用新的绑定信息为子表达式扩展语法对象。

1.2.1 标识符、绑定和范围

标识符和绑定 in Racket 指南 introduces binding.

一个 identifier 是一个源程序实体。 解析(即扩展)Racket 程序会发现,有些 identifiers 对应于 variables, 有些指的是 syntactic forms (比如 lambda, 它是函数的 syntactic form), 有些指的是用于宏扩展的 transformers ,有些被引用来产生 symbols 或 syntax objects。 当前者被解析为一个 variable 或语法形式,而后者被解析为对前者的 reference 时, 一个标识符就会 binds 另一个标识符(即,它是一种 binding)； 后者被 bound。

例如,作为来源的一个片段,文本

(let ([x 5]) x)

包括两个 identifiers: let 和 x (出现两次)。 当这个源码在 let 有其通常含义的上下文中被解析时, 第一个 x binds 了第二个 x。

绑定和引用是通过 scope sets 确定的。 一个 scope 对应于程序的一个区域,这个区域要么是源的一部分,要么是通过对源的阐述而合成的。 嵌套的绑定上下文(如嵌套的函数)创造了嵌套的 scopes,而宏扩展创造了以更复杂方式重叠的作用域。 从概念上讲,每个 scope 都由一个唯一的标记表示,但这个标记不能直接访问。 相反,每个 scope 由一个值来表示,这个值是程序的内部表示。

一个 form 是一个程序的片段,比如一个标识符或一个函数调用。 一个 form 被表示为一个 syntax object ,每个语法对象都有一组相关的 scopes (即一个 scope set)。 在上面的例子中, x 的表示包括与 let 形式相对应的 scope 。

当一个 form 解析为一个特定标识符的绑定时,解析会更新一个全局表, 该表将标识符的 symbol 和 scope set 的组合映射到它的意义上: 一 个variable, 一个 syntactic form 或一个 transformer。 当引用的符号和标识符的符号相同,并且引用的 scope set 是绑定的 scope set的超集时, 一个标识符就指的是一个特定的绑定。 对于一个给定的标识符,多个绑定的 scope sets 可能是标识符的子集; 在这种情况下,标识符指的是作用域集是所有其他绑定的超集的那个绑定; 如果不存在这样的绑定,引用是模糊的(如果它被解析为表达式,会触发一个语法错误)。 一个绑定会 shadows 任何具有相同符号但有一个作用域子集的 binding (即它 shadowing 任何 binding) 。

例如,在

(let ([x 5]) x)

在 let 对应于通常的 syntactic form 的情况下, let 的解析为 x 的绑定引入了一个新的范围。 由于第二个 x 作为 let 主体的一部分收到了这个范围, 所以第一个 x binds 了第二个 x 。 在更复杂的情况下

(let ([x 5])
  (let ([x 6])
    x))

内部 let 为第二个 x 创建了第二个作用域, 所以它的 scope set 是第一个 x 的 scope set 的超集 –这意味着第二个 x 的绑定会 shadows 第一个 x 的绑定, 而第三个 x 指的是第二个绑定所创建的绑定。

top-level binding 是来自顶层定义的 binding ; module binding 是来自模块中的定义的绑定; 所有其他绑定都是 local bindings 。 在一个模块内,不允许引用 top-level bindings。一个没有绑定的标识符是 unbound。

在整个文档中, identifiers 的类型表明了它们被解析的方式。 像 lambda 这样的超链接标识符表示对一个句法形式或变量的引用。 像 x 这样的普通标识符是一个 variable 或对一个未指定的 top-level variable 的引用。

每个绑定都有一个可以被引用的 phase level , 其中 phase level 通常对应于一个整数(但特殊 label phase level 不对应于一个整数)。 Phase level 对应于封闭模块的运行时间(或顶级表达式的运行时间)。 处于 phase level 0 的绑定构成 base environment。 Phase level 1 对应于封闭模块(或顶层表达式)被展开的时间; phase level 1 中的绑定构成 transformer environment。 对应于不同模块的运行时间,对于该模块, 包围模块在 phase level 1 (相对于导入模块)被导入使用; phase level -1 中的绑定构成 template environment。 label phase level 不对应任何执行时间; 它被用来跟踪绑定(例如,在文档中的标识符)而不意味着执行依赖。

一个标识符在不同的 phase levels 可以有不同的绑定。 更确切地说,与一个 form 相关的 scope set 在不同的阶段级别可以是不同的; 顶层或模块上下文在每个阶段级别都意味着一个不同的作用域, 而来自宏扩展或其他语法形式的作用域在所有阶段都被添加到一个表单的 scope sets 。 每个绑定和引用的上下文决定了哪个 phase level的 scope set 是相关的。

binding space 是一个惯例,它通过对空间有一个特定的 scope 来区分绑定; 如果一个标识符的绑定在其 scope set 中包括了空间的范围,那么它就是 "绑定在一个空间中"。 一个空间的作用域可以通过使用 make-interned-syntax-introducer来间接访问; 也就是说,一个空间只是具有与该空间的名称互为作用域的绑定的集合, 其中 default binding space 对应于没有互为作用域。 require 和 provide 表单包括通过 for-space 和 only-space-in等子表单对绑定空间的支持。 racket 模块没有提供其他的形式来绑定或引用指定空间中的标识符; 这种形式打算由新的宏来实现。 根据惯例,当一个标识符被绑定在一个空间中时,一个相应的标识符也应该被绑定在默认的绑定空间中; 这个惯例有助于避免进口之间的不匹配或由于本地绑定只在某些空间中产生阴影而导致的不匹配。

Changed in version 6.3 of package base: 修改了本地绑定,使其具有特定的相位,就像顶层绑定和模块绑定。
Changed in version 8.2.0.3: 增加 binding spaces。

1.2.2 语法对象

语法对象 in Racket 指南 introduces the use of syntax objects.

syntax object 将一个较简单的 Racket 值(如符号或对)与 lexical information, source-location 信息, syntax properties 以及该语法对象是否有 tainted 结合起来。 syntax object 的 lexical information 包括一组 scope sets , 每个 phase level 都有一个。 特别是,一个 identifier 被表示为一个包含 symbol 的语法对象, 其 lexical information 可以与全局绑定表相结合, 以确定其在每个 phase level 的 binding (如果有)。

例如,一个 car identifier 可能有 lexical information， 指定它是 racket/base 语言中的 car (即内置的 car)。 同样地,一个 lambda 标识符的 lexical information 可能表明它代表一个过程形式。 其他一些 identifier 的 lexical information 可能表明它引用了一个 top-level variable。

当一个 syntax object 代表一个比 identifier 或简单常量更复杂的表达式时, 它的内部组件可以被提取出来。 即使对于提取的标识符,关于绑定的详细信息也大多是间接可用的; 两个标识符可以通过比较来确定它们是否引用了相同的绑定(即 free-identifier=?), 或者标识符是否具有相同的作用域集, 以便如果一个标识符处于绑定位置,而另一个处于表达位置, 则每个标识符都会绑定另一个(即 bound-identifier=?)。

例如,当程序写为

(let ([x 5]) (+ x 6))

表示为一个 syntax object ,那么就可以为这两个 x 提取两个 syntax objects 。 free-identifier=? 和 bound-identifier=? 谓词都将表明这两个 x 是相同的。 相反, let identifier 对任何一个 x 都不是 free-identifier=? 或 bound-identifier=? 。

syntax object 的 lexical information 是独立于 syntax object 的其他部分的, 它可以和任意的其他 Racket 值一起被复制到新的语法对象中。 因此, syntax object 中的标识符 binding 信息是以 identifier 的符号名称以及标识符的 lexical information 为前提的; 同一个问题,如果有相同的 lexical information ,但基础值不同,就会产生不同的答案。

例如,将上面程序中 let 的词法信息与 'x 相结合, 不会产生一个对两个 x 都是 free-identifier=? 的标识符, 因为它没有出现在 x 绑定的范围内。 相反,将 6 的词法信息与 'x 结合起来, 会产生一个对两个 x 都是 bound-identifier=? 的标识符。

quote-syntax 形式连接了程序的评估和程序的表示。 具体来说, (quote-syntax datum #:local) 产生一个语法对象, 它保留了 datum 作为 quote-syntax 形式的一部分被解析时的所有词法信息。 请注意, (quote-syntax datum) 形式是类似的, 但它从 datum 的 scope sets中删除了某些 scopes ; 更多信息参见 quote-syntax 。

1.2.3 扩展 (Parsing)

Expansion 是以递归方式处理特定阶段级别的 syntax object , 从 phase level 0 开始。 syntax object 的 lexical information 的 Bindings 推动了扩展过程, 并导致子表达式的词法信息被引入新的绑定。 在某些情况下,子表达式被扩展到比包围表达式更深的阶段(具有更大的阶段编号)。

1.2.3.1 完全展开的程序

一个完整的扩展会产生一个与下列语法相匹配的 syntax object:

请注意,完全展开的程序中标识符的符号名称可能与语法中的符号名称不符。 只有绑定(根据 free-identifier=?) 才重要。

  | (#%expression expr)
  | 
(module id module-path
  (#%plain-module-begin
   module-level-form ...))
  | (begin top-level-form ...)
  | (begin-for-syntax top-level-form ...)

  | 
(module* id #f
  (#%plain-module-begin
   module-level-form ...))

  | (define-values (id ...) expr)
  | (define-syntaxes (id ...) expr)
  | (#%require raw-require-spec ...)

  | (case-lambda (formals expr ...+) ...)
  | (if expr expr expr)

  | (begin0 expr expr ...)
  |

  expr ...+)
  |

  expr ...+)

  | (quote datum)

  | (quote-syntax datum #:local)
  | (with-continuation-mark expr expr expr)
  | (#%plain-app expr ...+)
  | (#%top . id)
  | (#%variable-reference id)
  | (#%variable-reference (#%top . id))
  | (#%variable-reference)

  | (id ...+ . id)

一个 fully-expanded syntax object 对应于一个程序的 parse (即一个被 parsed 的程序), 其 identifiers 的 lexical information 表示该 parse.

更具体地说,上述语法中标识符的排版是很重要的。 例如, expr 的第二种情况是一个 syntax-object 列表, 其第一个元素是一个 identifier, 其中 identifier 的 lexical information 指定了与 racket/base 语言的 #%plain-lambda 的绑定(即 identifier 是 free-identifier=? ,对其绑定是 #%plain-lambda)。 在所有情况下,上面排版为句法形式的标识符指的是 句法形式 中定义的绑定。 N 在一个 base phase 为 0 的命名空间的完全扩展程序中,如果程序中的绑定有 N 个周围的 begin-for-syntax 和/或 define-syntaxes 表单，则该绑定的相关阶段级别为 N– 不算任何包裹 module 或 module* 主体的 begin-for-syntax 表单, 除非 module* 形式的 id 后面有 #f 代替 module-path 。 在一个 datum 形式中的 quote-syntax 在所有 phase level 上都保留其信息。

在一个完全展开的程序中, 对一个 local binding 的引用有一个与它的绑定标识符完全匹配的 scope set 。 额外的 scopes, 如果有的话,会被删除。 因此, bound-identifier=? 可以用来将局部绑定标识符与引用标识符联系起来, 而 free-identifier=? 必须用来将引用与 module bindings 或 top-level bindings 联系起来。

除了上面的语法, #%expression 可以出现在完全局部展开的表达位置。 例如,当停止列表为空时, #%expression 可以出现在 local-expand 的结果中。 只有 当local-expand 的停止列表为空时, 参考标识符 scope sets 才会在局部展开的表达式中减少。

Changed in version 6.3 of package base: 添加了 quote-syntax 的 #:local 变体; 删除了 letrec-syntaxes+values 可能出现在完全 local-expanded 形式中。

1.2.3.2 扩展步骤

在递归扩展中,在某一 phase level 平上扩展 syntax object 的每一个单一步骤都取决于被扩展的 syntax object 的直接形状。:

  • 如果它是一个 identifier (即一个语法对象符号), 那么就由 identifier 的 lexical information 来确定一个 binding 。 如果该 identifier 有一个 binding, 该 binding 将被用于继续。 如果 identifier unbound, 就会使用 identifier 的 lexical information 创建一个新的 syntax-object 符号 '#%top ,并使用 implicit-made-explicit properties; 如果这个 #%top identifier 没有 binding, 那么解析就会出现 exn:fail:syntax 异常。否则,新的 identifier 会在一个新的 syntax-object 对中与原来的 identifier 相结合(也使用与原来 identifier 相同的 lexical information), 并且使用 #%top binding 来继续。

    Changed in version 6.3 of package base: 改变了 #%top 在顶层上下文中的引入, 仅用于 unbound 标识符。

  • 如果它是一个 syntax-object 对,其第一个元素是一个 identifier, 如果该 identifier 有一个 binding ,而不是作为一个 top-level variable , 那么该 identifier 的 binding 将被用来继续。

  • 如果它是任何其他形式的 syntax-object 对, 那么一个新的 syntax-object 符号 '#%app 将使用具有implicit-made-explicit properties 的对的 lexical information 创建。 如果生成的 #%app identifier 没有绑定, 解析就会失败,出现 exn:fail:syntax 异常。 否则,新的 identifier 就会与原来的对结合起来, 形成一个新的 syntax-object 对(也使用与原来的对相同的 lexical information), 并且 #%app binding 被用来继续。

  • 如果是其他的语法对象,那么一个新的 syntax-object 符号 '#%datum 将被创建, 使用原始 syntax object 的 lexical information 和 implicit-made-explicit properties。 如果生成的 #%datum identifier 没有 binding, 解析就会失败,出现 exn:fail:syntax 异常。 否则,新的 identifier 将与原始 syntax object 组合成一个新的 syntax-object 对(使用与原始对象对相同的 lexical information ),并使用 #%datum binding 来继续。

因此,没有失败的可能性会导致一个具有特定 binding 的 identifier。 这个绑定指的是三件事中的一件:

  • 一个 transformer, 比如由 define-syntax 或 let-syntax 引入的。 如果关联的值是一个只有一个参数的过程, 那么这个过程就会被作为一个 syntax transformer 来调用(如下所述), 然后用 syntax-object 的结果重新开始解析。 如果 transformer 的是任何其他类型的值,解析就会以 exn:fail:syntax 异常而失败。 对 syntax transformer 的调用被 parameterize , 以将 current-namespace 设置为一个与被用来扩展的名字空间共享 bindings 和 variables 的 namespace ,只不过它的 base phase 要大一个。

  • variable binding,例如由模块级 define 或 let。 在这种情况下,如果被解析的形式只是一个 identifier, 那么它就被解析为对相应 variable 的引用。 如果被解析的表单是一个 syntax-object 对, 那么就会在 syntax-object 对的前面加上一个 #%app , 方法与 syntax-object 对中的第一项不是标识符时相同(前面枚举的第三种情况),然后继续解析。

  • 一个核心 syntactic form (通常缩写为 core form), 按照 句法形式 中对每个形式的描述进行解析。 对核心语法形式的解析通常涉及子形式的递归解析, 并可能引入决定子形式解析的 bindings。

当一个 #%top, #%app 或 #%datum 标识符被扩展器添加时,它被赋予 implicit-made-explicit properties: 一个 'implicit-made-explicit syntax property ,其值为 #t, 以及一个隐藏属性,表示隐式标识符是 syntax-original? 意义上的原创, 如果赋予标识符其 lexical information 的语法对象具有该属性。

Changed in version 7.9.0.13 of package base: 增加 implicit-made-explicit properties。

1.2.3.3 扩展语境

每个扩展步骤都发生在一个特定的 context 中, 转化器和核心语法形式对于不同的 contexts可能会有不同的扩展。 例如, module 形式只允许在 top-level context 或 module context中使用, 而在其他上下文中则不能使用。可能的 contexts 如下:

  • top-level context : 在任何模块、定义或表达式之外,但顶层 begin 形式的子表达式也被扩展为顶层形式。

  • module-begin context : 在一个模块的主体中,是该模块中唯一的形式。

  • module context : 在模块的主体中(在模块开始层内)。

  • internal-definition context : 在一个允许定义和表达的嵌套上下文中。

  • expression context : 在一个只允许表达式的上下文中。

不同的核心 syntactic forms 使用不同的 contexts 来解析子形式。 例如, let 形式总是在 expression context 中解析绑定的右侧表达式, 但是它在 internal-definition context 中开始解析主体。

1.2.3.4 引入绑定

在 expansion 过程中,当遇到某些核心语法形式时,会引入 Bindings :

  • 当在顶层或模块层遇到 require 形式时,该形式指定的每个符号都与规范的 scope set 配对,以引入新的绑定。 如果 require 表单中没有其他说明,绑定将在输出模块指定的 phase level 上被引入: 每个正常 provide 的 phase level 为 0 , 每个 for-syntax provide 的 phase level 为 1,依此类推。 for-meta provide 形式允许在任意 phase level 上的输出 (只要在该 phase level上的模块中存在一个绑定)。

    require 中的 for-syntax 子表单的导入方式类似, 但生成的绑定物的 phase level 比导出的 phase levels 多一个, 当导出的 label phase level 仍被导入到 label phase level。 更一般地说, require 中的 for-meta 子表单以指定的 phase level 转移导入; 如果指定的转移是 #f, 或者如果 for-label 用来导入, 那么所有绑定都被导入到 label phase level。

  • 当在顶层或模块层遇到 define, define-values, define-syntax 或 define-syntaxes 形式时, 为每个被定义的标识符在 phase level 0 增加一个绑定(即 base environment 被扩展)。

  • 当在顶层或模块层遇到 begin-for-syntax 形式时, 与 define-values 和 define-syntaxes 的绑定被引入, 但在 phase level 1 (即, transformer environment 被扩展)。 更广泛地说, begin-for-syntax 形式可以被嵌套, 每个 begin-for-syntax 将其主体移到一个 phase level。

  • 当遇到 let-values 形式时, let-values 形式的主体被扩展(通过创建新的 syntax objects) , 有一个新的 scope。 该 scope 被添加到标识符本身, 所以在绑定位置的标识符在完全扩展的表单中是 bound-identifier=? 的用途, 所以它们不会被 bound-identifier=? 到其他标识符。 新的绑定是在 let-values 形式被扩展的 phase level 上。

  • 当遇到 letrec-values 或 letrec-syntaxes+values 形式时, 与 let-values 一样,绑定被添加,只是右侧的表达式也被扩展了新的 scope。

  • internal-definition contexts 中的定义会引入新的作用域和绑定, 如 内部定义中所述。

例如,在

(let-values ([(x) 10]) (+ x y))

为 x 引入的绑定适用于主体中的 x , 因为一个新的 scope 被创建并添加到绑定 x 和引用 x 上。 同样的作用域被添加到 y 上,但由于它的符号与绑定的 x不同, 所以它并不引用新的绑定。 在这个 let-values 形式之外的任何 x 都不会收到新的 scope ,因此不会引用新的绑定。

1.2.3.5 变换器绑定

在 top-level context 或 module context 中, 当扩展器遇到 define-syntaxes 表单时, 它为定义的标识符引入的绑定是一个 transformer 绑定。 binding 的 value 存在于扩展时,而不是运行时(尽管这两个时间可以重叠), 尽管绑定本身是在 phase level 0 引入的(即在 base environment 中)。

绑定的 value 是通过评估 define-syntaxes 形式的表达式得到的。 这个表达式在被评估之前必须被 expand (即解析), 而且它是在 phase level 1 (即在 transformer environment中) 而不是 phase level 0 展开。

如果产生的 value 是一个只有一个参数的过程, 或者是过程上的 make-set!-transformer 的结果, 那么它就被用作一个 syntax transformer (又称 macro)。 该过程被期望接受一个语法对象并返回一个语法对象。 对绑定的使用(在 phase level 0) 会触发扩展程序对 syntax transformer 的调用; 参见 扩展步骤。

在扩展器将 syntax object 传递给转化器之前, syntax object 被扩展了一个新的 macro-introduction scope (适用于所有子syntax objects) ,以区分宏使用地点的 syntax objects 和由宏引入的 syntax objects ; 在转化器的结果中,该 scope 的存在被翻转,因此引入的 syntax objects 保留该 scope, 而使用地点的 syntax objects 没有该作用。 此外,如果转化器的使用与它的绑定处于相同的定义上下文中, 则使用地 syntax object 会被扩展为一个额外的新鲜 use-site scope ,该范围在转化器的结果中不会被翻转, 因此只有使用地 syntax objects 才有 use-site scope。

宏扩展的 scope 介绍过程有助于保持扩展后的程序中的绑定与源程序的词汇结构一致。 例如,程序的扩展形式是

(define x 12)
(define-syntax m
  (syntax-rules ()
    [(_ id) (let ([x 10]) id)]))
(m x)

是

(define x 12)
(define-syntax m ....)
(let ([x 10]) x)

但是,最后一个表达式的结果是 12, 而不是 10 。 原因是绑定到 m 的变压器引入了绑定 x, 但引用 x 存在于变压器的参数中。 引入的 x 只剩下一个新 scope, 而引用的 x 有一个不同的新 scope, 所以绑定的 x 不是 bound-identifier=? 到物体 x。

绑定标识符上的 use-site scope 在引入 use-site scope 的上下文中时被忽略。 这种 use-site scopes 的特殊处理允许宏扩展为可见定义。 例如,程序的扩展形式

(define-syntax m
  (syntax-rules ()
    [(_ id) (define id 5)]))
(m x)
x

是

(define-syntax m ....)
(define x 5)
x

define 形式中的 x 有一个 use-site scope ,在最后的 x 中不存在。 最后的 x 仍然是指定义,因为在安装定义的绑定之前, use-site scope 已经被有效地删除了。 相比之下,

(define-syntax m
  (syntax-rules ()
    [(_ id) (let ([x 4])
              (let ([id 5])
                x))]))
(m x)

的展开式是

(define-syntax m ....)
(let ([x 4])
  (let ([x 5])
    x))

第二个 x 有一个 use-site scope , 阻止它绑定最后一个 x。 在这种情况下,use-site scope 不会被忽略, 因为绑定不是 (m x) 展开的定义上下文的一部分。

set! 形式使用 make-set!-transformer 和 prop:set!-transformer 属性来支持转换 set! 表达式的 assignment transformers 。 assignment transformer 包含一个过程, 该过程由 set! 应用,与扩展器应用普通变压器的方式相同。

make-rename-transformer 过程或 prop:rename-transformer 属性创建的值也由扩展器和 set! 专门处理,作为变压器绑定的值。 当 id 绑定到 make-rename-transformer 生成的 rename transformer 时, 它将被传递给 make-rename-transformer的目标标识符替换。 此外,只要目标标识符没有 'not-free-identifier=? syntax property 的真值, 绑定表就会被扩展,以表明 id 是 rename transformer 中标识符的别名。 free-identifier=? 函数遵循混叠链来确定绑定是否相等, identifier-binding 函数类似地遵循混叠链, 而 provide 表单导出 id 作为目标标识符。 最后,即使没有安装绑定别名, syntax-local-value 函数也会跟随 rename transformer 链。

除了使用作用域跟踪引入的标识符外, 扩展器还通过 'origin 等 syntax properties 跟踪表单的扩展历史。 有关更多信息,请参见 语法对象属性 。

展开器对 letrec-syntaxes+values 的处理类似于对 define-syntaxes 的处理。 letrec-syntaxes+values 可以在任意阶段级别 n (而不仅仅是 0)中展开, 在这种情况下,transformer 绑定的表达式在 phase level n+1。

begin-for-syntax 形式的表达式的展开和求值方式与 define-syntaxes 相同。 但是,在 begin-for-syntax 中从定义引入的任何绑定都处于 phase level 1 (而不是 phase level 0 的 transformer 绑定)。

1.2.3.6 本地绑定背景信息

尽管 identifier 的 binding 可以从其 lexical information 和全局绑定表的组合中惟一地确定,但扩展器还维护一个local binding context , 该上下文记录有关 local bindings 的附加信息, 以确保它们不会在绑定所在的词法区域之外使用。

由于本地绑定表单(如 let )同时为绑定 identifiers 和主体表单添加了新的 scope ， identifier 引用 local binding 而不出现在 let的主体中通常是不可能的。 但是,如果宏使用编译时状态来保存绑定的 identifiers, 或者使用 local-expand 从扩展的绑定表单中提取 identifiers ,则可能违反此约束。 例如,下面的 stash-id 和 unstash-id 宏合作将引用移动到本地绑定的 x identifier 之外的词法区域:

> (begin-for-syntax
    (define stashed-id #f))
> (define-syntax (stash-id stx)
    (syntax-case stx ()
      [(_ id)
       (begin
         (set! stashed-id #'id)
         #'(void))]))
> (define-syntax (unstash-id stx)
    stashed-id)
> (let ([x 42])
    (stash-id x)
    (unstash-id))

42

> (unstash-id)

eval:5:0: x: identifier used out of context

  in: x

通常, identifier 的 lexical information 不足以知道其 binding 在封闭的上下文中是否可用, 因为存储在 stashed-id 中的 identifier 的 scope set 明确地引用了全局绑定表中的绑定。 这可以通过这样的事实观察到: identifier-binding 绑定产生的是 'lexical, 而不是 #f:

> (define-syntax (stashed-id-binding stx)
    #`'#,(identifier-binding stashed-id))
> (stashed-id-binding)

'lexical

然而,在上面的程序中 (unstash-id) 产生的引用仍然是非法的,即使它在技术上没有被解除绑定。 为了记录 x 的 binding 只在其对应 let 形式的主体内的作用域内, 展开器在展开 let 主体时将 x 的 binding 添加到 local binding context 。 更一般地,展开器将所有 local variable bindings 添加 local binding context , 同时展开对 variable 的引用合法的表达式。 当展开器遇到绑定到 local variable 的 identifier , 而相关的 binding 不在当前 local binding context 中时,将引发语法错误。

local binding context 也以类似的方式跟踪本地 transformer bindings (即由 let-syntax等形式绑定的绑定), 只不过上下文还存储与 transformer 相关的编译时值。 当本地绑定为 transformer 的 identifier 在应用程序位置中用作 syntax transformer 时, 或者使用 syntax-local-value 查找其编译时值时, 将查阅 local binding context 以检索该值。 如果 binding 在作用域中,则使用其关联的编译时值;否则,展开器将引发语法错误。

Examples:
> (define-syntax (stashed-id-local-value stx)
    #`'#,(syntax-local-value stashed-id))
> (let-syntax ([y 42])
    (stash-id y)
    (stashed-id-local-value))

42

> (stashed-id-local-value)

syntax-local-value: identifier is not bound to syntax:

#<syntax:eval:11:0 y>

1.2.3.7 局部扩张

在某些上下文中,如 internal-definition context 中或 module context, partial expansion 用于确定表单是否表示定义、表达式或其他声明形式。 当相关绑定是针对基本语法形式时,部分展开通过切断正常的递归展开来工作。

作为一种特殊情况,当展开本应向表达式添加 #%app, #%datum 或 #%top 标识符,而当绑定结果为原语 #%app, #%datum 或 #%top 形式时, 展开停止而不添加标识符。

1.2.3.8 内部定义

internal-definition context 支持局部定义和表达式的混合。 允许内部定义的表单使用 body 元变量记录这些位置。 内部定义上下文中的定义相当于通过 letrec-syntaxes+values 进行本地绑定; 宏展开将内部定义转换为 letrec-syntaxes+values 形式。

展开依赖于内部定义序列中每个 body 的 partial expansion 。 每个 body 的部分展开产生符合下列情况之一的形式:

  • define-values 形式： 绑定表立即为 define-values 表单添加绑定。 定义的进一步扩展将被推迟,部分扩展将在主体的其余部分继续进行。

  • define-syntaxes 形式: 对右侧进行展开和求值(就像 letrec-syntaxes+values 表单一样), 并为主体序列安装转换器绑定,然后对主体的其余部分继续进行部分展开。

  • 除了 begin 之外的基本表达式形式: 表达式的进一步展开被推迟,部分展开继续与正文的其余部分一起进行。

  • begin 形式: begin 的子表单被拼接到内部定义序列中, 部分展开继续使用新拼接的第一个表单(或者下一个表单,如果 begin 没有子表单)。

在部分展开所有主体形式之后,如果没有遇到任何定义,则将表达式收集到 begin 形式中,作为内部定义上下文的展开。 否则,至少一个表达式必须出现在最后一个定义之后,任何出现在定义之间的 expr 都将转换为 (define-values () (begin expr (values))); 然后,定义以 letrec-syntaxes+values 形式转换为绑定, 最后一个定义之后的所有表达式都成为 letrec-syntaxes+values 形式的主体。

在开始部分扩展之前,内部定义上下文的扩展首先在内部定义上下文的内容上引入一个新的 outside-edge scope 。 这个外围作用域有效地标识以原始形式出现的语法对象。 inside-edge scope 也被创建并添加到原始内容中; 此外,将内边缘范围添加到任何部分展开的结果中。 这个内缘作用域确保由内部定义上下文引入的所有绑定都有一个共同的特定作用域。

1.2.3.9 模块扩展、阶段和访问

module 表单的 expansion of an internal-definition context 类似: 为原始模块内容创建一个 outside-edge scope , 并向原始模块和在部分扩展模块顶级表单期间出现的任何表单添加一个 inside-edge scope ,以揭示定义和导入。

require 表单不仅在扩展时引入 bindings , 而且在扩展程序遇到被引用的模块时 visits 它。 也就是说,扩展器在 begin-for-syntax 中实例化模块中定义的任何变量, 并且它还为 define-syntaxes transformer 绑定计算所有表达式。

模块 visits 以与模块 instantiation 相同的方式传播 require 。 此外,当一个模块在 phase 0 被 visit 时, 它 require for-syntax 的任何模块都将在 phase 1 被 instantiate , 而进一步 require for-template 引导回 phase 0 , 这将导致所需的模块在 phase 0 被访问(即不被 instantiate)。

在编译过程中,会隐式地 visit 模块上下文的顶层。 因此,当扩展器遇到 (require (for-syntax ....)) 时, 除了在 phase level 1 (即transformer environment)添加绑定之外, 它还会立即在 phase 1 instantiate 所需的模块。 类似地,展开器会立即计算在 begin-for-syntax 中遇到的任何表单。

超出0 Phases 的模块将根据需要被 visit 。 例如,当 phase-0 的 let-syntax 的右侧要被扩展时, 就会访问 phase 1 段的 available 模块。 更一般地说,在 phase n 启动扩展会 visit phase n 的模块,这反过来又会 instantiates phase n+1 的模块。 这些 visits 和 instantiations 适用于包围 namespace 的 module registry 表中的 available 模块; 每个注册表锁防止多个线程同时实例化和访问可用模块。 对可用模块的按需实例化使用与 namespace-call-with-registry-lock 相同的可重入锁。

当扩展器在 module context 中遇到 require 和 (require (for-syntax ....)) 时, 所产生的 visits 和 instantiations 是特定于包围模块的扩展, 并与从 top-level context 或从不同模块的扩展中触发的 visits 和 instantiations 保持分离。 按照同样的思路, 当一个模块通过 namespace-attach-module 被附加到一个命名空间时, 它所 require 的模块会被过渡性地附加, 但实例只在命名空间的 base phase 或更低的阶段被附加。

1.2.3.10 宏引入的绑定

当一个顶层定义绑定一个源于宏扩展的标识符时,由于为扩展生成的新 scope, 该定义只捕获由同一扩展产生的标识符的使用。

Examples:
> (define-syntax def-and-use-of-x
    (syntax-rules ()
      [(def-and-use-of-x val)
       ; x below originates from this macro:
       (begin (define x val) x)]))
> (define x 1)

       (begin (define x val) x)]))

对于一个顶层定义(在模块之外),评估的顺序会影响生成的定义对生成的标识符使用的绑定。 如果使用在定义之前,那么使用将被解决与当时的绑定,这将不包括来自随后的宏生成的定义的绑定。 (在模块中不存在这种顺序上的依赖,因为模块的绑定涵盖了整个模块体。) 为了支持标识符在使用前的声明,如果 define-syntaxes 声明的主体产生了零结果, define-syntaxes 形式就会避免绑定标识符。

> (define bucket-1 0)
> (define bucket-2 0)

    (syntax-rules ()
      [(def-and-set!-use-of-x val)

         (define (odd x) (if (zero? x) #f (even (sub1 x))))

宏生成的 require 和 provide 句还引入和引用了特定于生成的绑定(由于增加了 scope) ,其排序效果与定义相同。 绑定取决于附加到表单特定部分的 scope set :

  • 在 require, 对于 (rename-in [orig-id bind-id]) 或 (only-in .... [orig-id bind-id]) 这样的 require-spec , bind-id 提供了绑定的 scope set 。 在对其它 require-spec 的 require 中, require-spec 的生成器决定了 scope set。

  • 在 provide 中,对于形式为 id 的 provide-spec , 导出的标识符是绑定 id 的标识符,但外部名称是 id 的纯符号部分。 all-except-out 的例外情况也是类似的, 就像 rename-out 形式的 orig-id 绑定一样,外部名称使用的是普通符号。 对于 all-defined-out, 只有定义只具有 (all-defined-out) 形式的范围的标识符被导出; 外部名称是定义中的普通符号。

1.2.4 编译

在对扩展代码进行评估之前,首先要对其进行 compiled。 编译后的形式与相应的扩展形式的信息基本相同,尽管内部表示自然省去了句法形式和局部绑定的标识符。 一个重要的区别是,编译后的形式几乎是完全不透明的, 所以它所包含的信息不能被直接访问(这就是为什么一些标识符可以被放弃)。 同时,一个编译过的表单可以被编入或编出一个字节串,所以它适合于保存和重新加载代码。

虽然有单独的读取、扩展、编译和评估操作,但这些操作往往是自动组合的。 例如, eval 程序接收一个语法对象并对其进行扩展、编译和评估。

1.2.5 命名空间

参见 命名空间 ,了解操作命名空间的函数。

一个 namespace 既是解析的起点,也是运行 compiled 代码的起点。 一个 namespace 也有一个 module registry , 将模块名称映射到模块声明(参见 模块和模块级变量)。 这个注册表是所有 phase level 共享的, 它既适用于解析也适用于运行 compiled 后的代码。

作为解析的起点, namespace 提供了作用域(每个 phase level 一个, 外加一个跨越所有 phase levels)的作用域)。 像 namespace-require 这样的操作使用命名空间的 scopes 创建初始 bindings , 而命名空间的进一步扩展和评估可以创建额外的 bindings。 对带有命名空间的表单的评价总是将命名空间的特定阶段 scopes 添加到表单和扩展顶层表单的结果中;因此,每个绑定标识符至少有一个 scope 。 命名空间的额外作用域只有在请求时才会被添加 (例如,通过使用 eval 而不是 eval-syntax); 如果请求,额外的作用域会被添加到所有的 phase levels。 除了由模块生成的命名空间(参见 module->namespace), 每个命名空间都使用与添加到所 有phase levels 的命名空间相同的 scope , 而特定于一个 phase level 的 scopes 总是不同的。

作为评估 compiled 代码的起点, 每个命名空间在不同的 phases 封装了一组不同的顶级变量, 以及在每个 phase 可能不同的模块实例。 也就是说,即使所有 phase levels 的模块声明是共享的, 但每个 phase 的模块实例是不同的。 每个命名空间都有一个 base phase, 它对应于反思性操作(如 eval 和 dynamic-require )所使用的阶段。 特别是,在 require 形式上使用 eval 会在命名空间的 base phase instantiates 一个模块。

命名空间创建后,现有命名空间的模块实例可以被附加到新的命名空间。 就评估模型而言,来自不同名字空间的顶层变量基本上对应于具有不同前缀的定义, 但是附加模块时,在附加模块的名字空间中,模块的定义使用相同的前缀。 评估任何编译后的表达式的第一步是将其顶层变量和模块级变量的引用与名字空间中的特定变量联系起来。

在评估过程中的任何时候,一些名称空间都被指定为 current namespace。 然而,当前的名字空间与用于展开正在执行的代码的名字空间, 或者与用于链接当前正在评估的代码的编译形式的名字空间没有特殊关系。 特别是,在评估过程中改变当前名称空间并不改变执行表达式所指向的变量。 当前的名字空间只决定了展开代码和开始评估已展开/已编译代码的反射性操作的行为。

Examples:
> (define x 'orig) ; define in the original namespace
; The following let expression is compiled in the original
; namespace, so direct references to x see 'orig.
> (let ([n (make-base-namespace)]) ; make new namespace
    (parameterize ([current-namespace n])
      (eval '(define x 'new)) ; evals in the new namespace
      (display x) ; displays 'orig
      (display (eval 'x)))) ; displays 'new

orignew

如果一个 identifier 被绑定到语法或导入, 那么将该 identifier 定义为一个 variable 就会在未来的环境使用中对语法或导入产生阴影。 同样,如果一个 identifier 被绑定到一个 top-level variable, 那么将该标识符绑定到语法或导入就会对该变量产生阴影; 然而,该变量的值保持不变,并且可以通过先前评估的表达式进行访问。

像顶层 namespace 一样,每个 module 表单都有一个相关的 scope , 以跨越模块内容的所有 phase levels ,加上每 phase level 的 scope 。 后者被添加到每个表单中,原始的或通过部分宏扩展出现的,在模块的直接主体中。 这些相同的作用域被传播到由 module->namespace 为模块创建的命名空间。 同时,解析一个 module 的表单时, 首先要删除所有对应于包围的顶层或(在 submodules 的情况下) module 和 module* 表单的作用域。

1.2.6 推断值的名称

为了改进错误报告,在编译时对某些类型的值(如过程)进行了名称推断。例如,评估下面的表达式:

(let ([f (lambda () 0)]) (f 1 2 3))

产生一个错误信息,因为向过程提供了太多的参数。 该错误信息能够报告 f 作为过程的名称。 在这种情况下,Racket决定在编译时将所有由 let-bound lambda 创建的过程命名为 'f 。

参见 procedure-rename ,在运行时覆盖一个存储过程的推断名称。

只要有可能,程序的名称都是推断出来的。 离表达式较近的名称优先。例如,在

(define my-f
  (let ([f (lambda () 0)]) f))

绑定到 my-f 的过程将具有推断的名称 'f。

当 'inferred-name 属性被附加到表达式的语法对象上时 (参见 语法对象属性), 该属性值被用于命名表达式,并且它优先于从表达式的上下文中推断出的任何名称。 通常情况下,属性值应该是一个符号。 'inferred-name 的属性值 #<void> 隐藏了否则会从上下文推断出的名称(也许是为了避免暴露自动生成的绑定的标识符)。

为了在展开过程中支持一致属性的传播和合并, 'inferred-name 属性的值可以是一个由 cons 组成的树, 其中所有的叶子都是相同的。 例如, (cons 'name 'name) 等价于 'name, (cons (void) (void)) 等价于 #<void>。

当推断的名称不可用,但源位置可用时,将使用源位置信息构造名称。

语法转换器也可以通过 syntax-local-name 使用推断的和属性赋值的名称。

1.2.7 跨阶段持久模块声明

只有当一个模块符合以下语法(使用来自 完全展开的程序的非结束符), 只有当它包含 (#%declare #:cross-phase-persistent), 只有它不包含 quote-syntax 或 #%variable-reference 的使用, 并且只有当没有 set! 模块级绑定时,它才是 cross-phase persistent 。

  cross-module = 
(module id module-path
  (#%plain-module-begin
    cross-form ...))
     
  cross-form = (#%declare #:cross-phase-persistent)
  | (begin cross-form ...)
  | (#%provide raw-provide-spec ...)
  | submodule-form
  | (define-values (id ...) cross-expr)
  | (#%require raw-require-spec ...)
     
  cross-expr = id
  | (quote cross-datum)
  | (#%plain-lambda formals expr ...+)
  | (case-lambda (formals expr ...+) ...)
  | (#%plain-app cons cross-expr ...+)
  | (#%plain-app list cross-expr ...+)
  | (#%plain-app make-struct-type cross-expr ...+)
  | 
(#%plain-app make-struct-type-property
             cross-expr ...+)
  | (#%plain-app gensym)
  | (#%plain-app gensym string)
  | (#%plain-app string->uninterned-symbol string)
  | 
(#%plain-app variable-reference-from-unsafe?
             (#%variable-reference))
     
  cross-datum = number
  | boolean
  | identifier
  | string
  | bytes
  | ()

这种语法适用于 expansion 之后,但是因为 cross-phase persistent 模块只从其他跨阶段持久模块导入,所以惟一相关的扩展步骤是隐式引入 #%plain-module-begin, 隐式引入 #%plain-app, 隐式引入和/或扩展 #%datum。
