10.2 异常情况

异常 in Racket 指南 introduces exceptions.

关于 Racket 异常模型的信息请参见 异常 。 它是基于 Friedman, Haynes 和 Dybvig [Friedman95] 的提议。

每当 Racket 中发生原始错误时,就会引发一个异常。 传递给当前 exception handler 的原始错误的值总是 exn 结构类型的一个实例。 每个 exn 结构值都有一个 message 字段,是一个字符串,即原始错误信息。 默认的异常处理程序识别带有 exn? 谓词的异常值, 并将错误信息传递给当前的错误显示处理程序(参见 error-display-handler)。

接受具有特定所需节数的过程参数的原始过程(例如, call-with-input-file, call/cc) 立即检查参数的节数,如果节数不正确,则引发 exn:fail:contract 。

10.2.1 错误信息约定

Racket的 error message convention 是产生具有以下形状的错误信息:

‹srcloc›: ‹name›: ‹message›;
 ‹continued-message› ...
  ‹field›: ‹detail›
  ...

消息以一个可选的源位置 ‹srcloc› 开始,如果有的话,后面会有一个冒号和空格。 消息以一个可选的 ‹name› 继续, 这个名称通常是指投诉的函数、语法形式或其他实体,但也可以指被投诉的实体; ‹name› 后面也有冒号和空格。

‹message› 应该相对简短,它应该在很大程度上与触发错误的具体数值无关。 需要多行的更详细的解释应该继续,每行缩进一个空格,在这种情况下, ‹message› 应该以分号结束(但是如果 ‹continued-message› 不存在,分号应该被省略)。 信息文本应该小写,必要时使用分号来分隔句子,尽管长的解释可能最好推迟到额外的领域。

触发错误的具体数值或其他有用的信息应该出现在单独的 ‹field› 行中,每行缩进两个空格。 如果一个 ‹detail› 特别长或需要多行,它应该在 ‹field› 标签之后开始自己的一行, 并且每一行都应该缩进三个空格。字段名应全部为小写。

如果字段提供了相对详细的信息,在普通情况下可能会分散注意力,但在其他情况下却很有用, 则 ‹field› 名称应以 ... 结尾。 例如,当一个函数的某个参数被报告为合同失败时,该函数的其他参数可能会被显示在 "其他参数... "字段中。 其目的是,在DrRacket这样的环境中,名字以 ... 结尾的字段可能会被默认隐藏。

使 ‹field› 名称尽可能简短, 依靠 ‹message› 或 ‹continued message› 文本来阐明字段的含义。 例如,最好用 "given" 而不是 "given turtle" 作为字段名, 其中 ‹message› 是 "given turtle is too sleepy",以说明 "given" 是指一只乌龟。

10.2.2 引发异常

procedure

(raise v [barrier?]) → any

  v : any/c
  barrier? : any/c = #t
引发一个异常,其中 v 代表被引发的异常。 v 参数可以是任何东西;它被传递给当前的 exception handler 。

如果 barrier? 为真, 那么对 exception handler 的调用就会受到 continuation barrier 的保护, 因此不可能有多个返回/转义。 所有由 racket 函数引发的异常都有效地使用 raise 和 #t 值的 barrier? 。

从异常被引发开始,到异常处理程序获得控制权为止,中断是被禁用的, 处理程序本身也是 parameterize-break 的,以便最初禁用中断; 关于中断的更多信息,参见 中断 。

Examples:
> (with-handlers ([number? (lambda (n)
                             (+ n 5))])
    (raise 18 #t))

23

> (struct my-exception exn:fail:user ())
> (with-handlers ([my-exception? (lambda (e)
                                   #f)])
    (+ 5 (raise (my-exception
                 "failed"
                 (current-continuation-marks)))))

#f

> (raise 'failed #t)

uncaught exception: failed

procedure

(error message-sym) → any

  message-sym : symbol?
(error message-str v ...) → any
  message-str : string?
  v : any/c
(error who-sym format-str v ...) → any
  who-sym : symbol?
  format-str : string?
  v : any/c
引起异常 exn:fail,它包含一个错误字符串。 不同的形式以不同的方式产生错误字符串:

  • (error message-sym) 通过连接 "error: " 与 message-sym 的字符串形式创建一个消息字符串。 尽量少使用这种形式。

  • (error message-str v ...) 通过连接 message-str 和 v 的字符串版本(由当前错误值转换处理程序产生;参见 error-value->string-handler)创建一个消息字符串。 在每个 v 之前插入一个空格。 少用这种形式,因为它不符合 Racket 的 error message conventions; 考虑用 raise-arguments-error 代替它。

  • (error who-sym format-str v ...) 创建了一个信息字符串,相当于通过以下方式创建的字符串

    (format (string-append "~s: " format-str) who-sym v ...)

    在可能的情况下,使用 raise-argument-error 等函数, 这些函数构造的信息遵循Racket的 error message conventions 。

在所有情况下,构建的信息字符串都会被传递给 make-exn:fail, 并且产生的异常被引发。

Examples:
> (error 'failed)

error: failed

> (error "failed" 23 'pizza (list 1 2 3))

failed 23 'pizza '(1 2 3)

> (error 'method-a "failed because ~a" "no argument supplied")

method-a: failed because no argument supplied

procedure

(raise-user-error message-sym) → any

  message-sym : symbol?
(raise-user-error message-str v ...) → any
  message-str : string?
  v : any/c
(raise-user-error who-sym format-str v ...) → any
  who-sym : symbol?
  format-str : string?
  v : any/c
像 error 一样,但是用 make-exn:fail:user 而不是 make-exn:fail 构造一个异常。 默认的 error display handler 不显示 exn:fail:user 异常的 "堆栈跟踪" (参见 Continuation Marks),所以 raise-user-error 应该用于针对最终用户的错误。

procedure

(raise-argument-error name expected v) → any

  name : symbol?
  expected : string?
  v : any/c
(raise-argument-error name    
  expected    
  bad-pos    
  v ...) → any
  name : symbol?
  expected : string?
  bad-pos : exact-nonnegative-integer?
  v : any/c
创建一个 exn:fail:contract 值,并将其作为一个异常 raise 。 name 参数在错误信息中作为源过程的名称使用。 expected 参数被用作对预期契约的描述(即,作为一个字符串,但是这个字符串的目的是包含一个契约表达式)。

在第一种形式中, v 是程序收到的不具有预期类型的值。

在第二种形式中,坏参数由一个索引 bad-pos (从 0 开始计算)表示, 所有的原始参数 v 都被提供(按顺序)。 由此产生的错误信息命名了坏参数,同时也列出了其他的参数。 如果 bad-pos 不小于 v 的数量,就会引发 exn:fail:contract exception is raised。

raise-argument-error 产生的错误信息通过 error-contract->adjusted-string 调整, 然后 error-message->adjusted-string 使用默认的 'racket realm 。

Examples:
> (define (feed-machine bits)
    (if (not (integer? bits))
      (raise-argument-error 'feed-machine "integer?" bits)
      "fed the machine"))
> (feed-machine 'turkey)

feed-machine: contract violation

  expected: integer?

  given: 'turkey

> (define (feed-cow animal)
    (if (not (eq? animal 'cow))
      (raise-argument-error 'feed-cow "'cow" animal)
      "fed the cow"))
> (feed-cow 'turkey)

feed-cow: contract violation

  expected: 'cow

  given: 'turkey

> (define (feed-animals cow sheep goose cat)
    (if (not (eq? goose 'goose))
      (raise-argument-error 'feed-animals "'goose" 2 cow sheep goose cat)
      "fed the animals"))
> (feed-animals 'cow 'sheep 'dog 'cat)

feed-animals: contract violation

  expected: 'goose

  given: 'dog

  argument position: 3rd

  other arguments...:

   'cow

   'sheep

   'cat

procedure

(raise-argument-error* name realm expected v) → any

  name : symbol?
  realm : symbol?
  expected : string?
  v : any/c
(raise-argument-error* name    
  realm    
  expected    
  bad-pos    
  v ...) → any
  name : symbol?
  realm : symbol?
  expected : string?
  bad-pos : exact-nonnegative-integer?
  v : any/c
像 raise-argument-error, 但使用给定的 realm 来调整错误信息。

procedure

(raise-arguments-error name    
  message    
  field    
  v ...    
  ...) → any
  name : symbol?
  message : string?
  field : string?
  v : any/c
创建一个 exn:fail:contract 值,并将其作为一个异常 raise 。 name 在错误信息中作为源过程的名称。 message 是错误信息;如果 message 包含换行符, 每一个额外的行应该适当缩进(每行的开始有一个额外的空格),但它不应该以换行符结束。 每个 field 必须有一个相应的 v ,两者在错误信息中被呈现在各自的行上; 每个 v 都使用错误值转换处理程序(见 error-value->string-handler) 进行格式化, 除非 v 是一个 unquoted-printing string, 在这种情况下, 字符串内容被 display ,而不使用错误值转换处理程序。

raise-result-error 产生的错误信息通过 error-contract->adjusted-string 调整, 然后 error-message->adjusted-string 使用默认的 'racket realm 。

Example:
> (raise-arguments-error 'eat
                         "fish is smaller than its given meal"
                         "fish" 12
                         "meal" 13)

eat: fish is smaller than its given meal

  fish: 12

  meal: 13

  ...) → any

  message : string?
  field : string?

procedure

(raise-range-error name    
  type-description    
  index-prefix    
  index    
  in-value    
  lower-bound    
  upper-bound    
  [alt-lower-bound]) → any
  name : symbol?
  type-description : string?
  index-prefix : string?
  index : exact-integer?
  in-value : any/c
  lower-bound : exact-integer?
  upper-bound : exact-integer?
  alt-lower-bound : (or/c #f exact-integer?) = #f
创建一个 exn:fail:contract 值并 raise 了 它作为一个异常来报告一个超出范围的错误。 type-description 字符串描述了索引要选择一个元素的值, index-prefix 是 “index” 的前缀。 index 参数是被拒绝的索引。 in-value 参数是指索引所指向的值。 lower-bound 和 upper-bound 参数指定了索引的有效范围,包括在内; 如果 upper-bound 低于 lower-bound, 该值就被定性为 "空"。 如果 alt-lower-bound 不是 #f, 并且如果 index 在 alt-lower-bound 和 upper-bound 之间, 那么错误将被报告为 index 小于 "起始" index lower-bound 。

由于 upper-bound 是包容性的,一个典型的值是 小于 集合的大小–例如, (sub1 (vector-length vec)), (sub1 (length lst)) ,等等。

由 raise-range-error 产生的错误信息通过 error-message->adjusted-string 调整, 使用默认的 'racket realm 。

Examples:
> (raise-range-error 'vector-ref "vector" "starting " 5 #(1 2 3 4) 0 3)

vector-ref: starting index is out of range

  starting index: 5

  valid range: [0, 3]

  vector: '#(1 2 3 4)

> (raise-range-error 'vector-ref "vector" "ending " 5 #(1 2 3 4) 0 3)

vector-ref: ending index is out of range

  ending index: 5

  valid range: [0, 3]

  vector: '#(1 2 3 4)

> (raise-range-error 'vector-ref "vector" "" 3 #() 0 -1)

vector-ref: index is out of range for empty vector

  index: 3

> (raise-range-error 'vector-ref "vector" "ending " 1 #(1 2 3 4) 2 3 0)

vector-ref: ending index is smaller than starting index

  ending index: 1

  starting index: 2

  valid range: [0, 3]

  vector: '#(1 2 3 4)

  [alt-lower-bound]) → any

procedure

(raise-arity-error name arity-v arg-v ...) → any

  name : (or/c symbol? procedure?)
  arity-v : 
(or/c exact-nonnegative-integer?
      arity-at-least?
      (listof
       (or/c exact-nonnegative-integer?
             arity-at-least?)))
  arg-v : any/c
创建一个 exn:fail:contract:arity 值,并将其作为一个异常 raise 。 name 用于错误信息中的源过程的名称。

arity-v 值必须是 procedure-arity 的可能结果, 只是它不需要被规范化(关于规范化 arity 的细节,请看 procedure-arity? ); raise-arity-error 将规范化 arity 并在错误消息中使用规范化的形式。 如果 name 是一个过程,它的实际 arity 将被忽略。

The arg-v 参数是实际提供的参数,在错误信息中显示 (使用错误值转换处理程序;见 error-value->string-handler); 另外,提供的 arg-v 的数量在信息中被明确提及。

由 raise-arity-error 生成的错误信息通过 error-message->adjusted-string 调整, 使用默认的 'racket realm 。

Example:
> (raise-arity-error 'unite (arity-at-least 13) "Virginia" "Maryland")

unite: arity mismatch;

 the expected number of arguments does not match the given

number

  expected: at least 13

  given: 2

  arguments...:

   "Virginia"

   "Maryland"

(raise-arity-error* name    
  realm    
  arity-v    
  arg-v ...) → any
  name : (or/c symbol? procedure?)
  realm : symbol?
  arity-v : 
(or/c exact-nonnegative-integer?
      arity-at-least?
      (listof
       (or/c exact-nonnegative-integer?
             arity-at-least?)))

procedure

(raise-syntax-error name    
  message

  sub-expr    
  extra-sources

创建一个 exn:fail:syntax? 值,并将其作为一个异常 raise 。 宏使用这个过程来报告语法错误。

当提供 expr 时, name 参数通常是 #f ;下面将详细介绍。 message 被用作错误信息的主体;如果 message 包含换行符, 每个新行都应该适当缩进(在开始时有一个空格),而且不应该以换行符结束。

可选的 expr 参数是错误的源语法对象或 S 表达式 (但是表达式 #f 本身不能被表示;它必须被包装成一个 syntax object)。 可选的 sub-expr 参数是 expr 中的一个语法对象或 S 表达式(同样, #f 不能代表其本身), 它可以更精确地定位错误。 如果 error-print-source-location 是 #t ,两者都可以出现在生成的错误信息文本中。 当至少有一个是语法对象且 error-print-source-location 为 #t 时, 错误信息文本中的源位置信息也同样从 sub-expr 或 expr 中提取。

如果提供了 sub-expr 并且没有 #f, 那么它就会被用于生成的异常记录的 exprs 字段(以语法形式), 否则就会使用 expr ,如果提供了 expr 并且没有 #f 。 在任何一种情况下,语法对象都会被 cons 到 extra-sources 上以产生 exprs 字段, 或者如果既没有提供 expr 也没有提供 sub-expr 并且没有 #f,那么 extra-sources 将直接用于 exprs 。 在不寻常的情况下, exprs 中包含的 sub-expr 或 expr 不能被转换为语法对象(因为它包含一个循环), extra-sources 参数也会直接用于 exprs 。

生成的错误信息中使用的形式名称是通过 name, expr 和 sub-expr 参数的组合来确定的:

  • 当 name 是 #f, 并且 expr 是一个标识符或者一个包含标识符作为其第一个元素的语法对时, 那么错误信息中的形式名称就是标识符的符号。

  • 当 name 是 #f , 并且 expr 不是一个标识符或者一个包含标识符作为其第一个元素的语法对时, 那么错误信息中的形式名称是 "?" 。

  • 当 name 是一个符号时,那么该符号将被用作生成的错误信息中的形式名称。

message-suffix 字符串被附加到错误信息的末尾。 如果不是 "", 它通常应该以换行和两个空格开始,以增加信息的额外字段(见 错误信息约定)。

如果指定, exn 应该是一个构造函数或函数,其签名与 exn:fail:syntax 构造函数相同。

Examples:
> (raise-syntax-error #f "bad syntax" '(bad syntax))

?: bad syntax

  in: (bad syntax)

> (raise-syntax-error #f "unbound identifier" 'unbound-id #:exn exn:fail:syntax:unbound)

?: unbound identifier

  in: unbound-id

10.2.3 处理异常情况

procedure

(call-with-exception-handler f thunk) → any

  f : (any/c . -> . any)
  thunk : (-> any)
将 f 安装为调用 thunk 的 dynamic extent 的 exception handler 。 如果在评估 thunk 的过程中(在没有自己的异常处理程序的当前延续的扩展中)有一个异常被引发, 那么 f 会被应用到 raise 调用的 continuation 中的 raise 值 (但是 continuation 通常是用 continuation barrier 扩展的; 见 提示、限定的延续和障碍物 和 raise)。

任何需要一个参数的过程都可以是一个异常处理程序。 通常情况下,异常处理程序通过 abort-current-continuation 或其他逃避机制从 raise 调用的上下文中逃脱。 为了将异常传播给 "前一个 "异常处理程序– 即在被调用的异常处理程序与延续相关联之后与延续的其余部分相关联的异常处理程序– 异常处理程序可以简单地返回一个结果而不是逃逸,在这种情况下, raise 调用将值传播给前一个异常处理程序 (仍然在调用 raise 的动态范围内,并且在同一障碍下,如果有的话)。 如果一个异常处理程序返回一个结果,并且没有前一个处理程序可用, 那么就会使用 uncaught-exception handler 。

对一个异常处理程序的调用是 parameterize-break ,以禁止中断, 并且用 call-with-exception-handler 来安装一个异常处理程序, 通过 error display handler 报告原始和新引发的异常,然后通过 error escape handler 逃脱。

parameter

(uncaught-exception-handler) → (any/c . -> . any)

(uncaught-exception-handler f) → void?
  f : (any/c . -> . any)
一个parameter ，当相关的续集没有安装 call-with-exception-handler 或 with-handlers 的异常处理程序时,它决定了 uncaught-exception handler 被 raise 使用。 与使用 call-with-exception-handler 安装的异常处理程序不同, uncaught-exception 处理程序在被 raise 调用时必须不返回一个值; 如果它返回,就会引发一个异常(由报告原始异常和新引发的异常的异常处理程序来处理)。

除非处理程序的参数是 exn:break:hang-up 的实例, 否则默认的未捕获异常处理程序会使用当前的 error display handler (参见 error-display-handler)打印出一条错误信息。 如果处理程序的参数是 exn:break:hang-up 或 exn:break:terminate 的实例, 那么默认的 uncaught-exception 处理程序会调用带有 1 的 exit handler 的退出处理程序, 通常会退出或逃逸。 对于任何参数,默认的 uncaught-exception 处理程序会通过调用当前的 error escape handler (参见 error-escape-handler)来逃脱。 对每个处理程序的调用都是 parameterize ,将 error-display-handler , 将 error display handler ,并且它是 parameterize-break 的,以禁用中断。 对 error escape handler 的调用被进一步参数化, 将 error-escape-handler 设置为默认的 error escape handler; 如果 error escape handler 返回,那么就调用默认的 error escape handler 。

当当前的 error display handler 是默认处理程序时,那么 error-display 调用被参数化, 以安装一个紧急错误显示处理程序,记录错误(见 log-error) ,并且永不失败。

syntax

(with-handlers ([pred-expr handler-expr] ...)
  body ...+)
按照指定的顺序评估每个 pred-expr 和 handler-expr , 然后在其动态范围内用一个新的异常处理程序评估 body 。

新的异常处理程序只有在其中一个 pred-expr 程序应用于异常时返回真值时才会处理该异常, 否则异常处理程序会从 with-handlers 表达式的延续部分被调用(通过再次引发该异常)。 如果一个异常被 handler-expr 程序处理, 整个 with-handlers 表达式的结果就是处理程序的返回值。

当body 的评估过程中出现异常时,每个谓词过程 pred-expr 被应用到异常值上; 如果一个谓词返回一个真值,相应的 handler-expr 过程被调用,异常作为参数。 谓词是按照它们被指定的顺序来尝试的。

在任何谓词或处理程序被调用之前,整个 with-handlers 表达式的延续被恢复, 但也有 parameterize-break 来禁用断裂。 因此,在谓词和处理程序中,默认情况下禁用中断(见 中断), 异常处理程序是来自 with-handlers 表达式的延续部分。

exn:fail? 程序作为一个处理程序谓词是很有用的,可以捕捉所有的错误异常。 避免使用 (lambda (x) #t) 作为谓词, 因为 exn:break 异常通常不应该被捕获(除非它将被重新提出来合作破解)。 还要注意捕捉和丢弃异常,因为丢弃错误信息会给调试带来不必要的困难; 与其丢弃错误信息,不如考虑通过 log-error 或由 define-logger 创建的日志形式记录它。

Examples:
> (with-handlers ([exn:fail:syntax?
                   (λ (e) (displayln "got a syntax error"))])
    (raise-syntax-error #f "a syntax error"))

got a syntax error

> (with-handlers ([exn:fail:syntax?
                   (λ (e) (displayln "got a syntax error"))]
                  [exn:fail?
                   (λ (e) (displayln "fallback clause"))])
    (raise-syntax-error #f "a syntax error"))

got a syntax error

syntax

(with-handlers* ([pred-expr handler-expr] ...)
  body ...+)
和 with-handlers 一样,但是如果调用了 handler-expr 过程, 则不会显式地禁用 break,而且相对于 with-handlers* 形式,handler 调用处于尾部位置。

prop:exn:srclocs : struct-type-property?

一个属性,用于识别提供 srcloc 值列表的结构类型。 该属性通常被附加到用于表示异常信息的结构类型。 该属性的值必须是一个过程,该过程接受一个单一的值–从中提取源位置的结构类型实例, 并返回一个 srcloc 的列表。 一些 error display handlers 只使用第一个返回的位置。

作为一个例子,
#lang racket
 
;; We create a structure that supports the
;; prop:exn:srcloc protocol.  It carries
;; with it the location of the syntax that
;; is guilty.
(struct exn:fail:he-who-shall-not-be-named exn:fail
  (a-srcloc)
  #:property prop:exn:srclocs
  (lambda (a-struct)
    (match a-struct
      [(exn:fail:he-who-shall-not-be-named msg marks a-srcloc)
       (list a-srcloc)])))
 
;; We can play with this by creating a form that
;; looks at identifiers, and only flags specific ones.
(define-syntax (skeeterize stx)
  (syntax-case stx ()
    [(_ expr)
     (cond
       [(and (identifier? #'expr)
             (eq? (syntax-e #'expr) 'voldemort))
        (quasisyntax/loc stx
          (raise (exn:fail:he-who-shall-not-be-named
                  "oh dear don't say his name"
                  (current-continuation-marks)
                  (srcloc '#,(syntax-source #'expr)
                          '#,(syntax-line #'expr)
                          '#,(syntax-column #'expr)
                          '#,(syntax-position #'expr)
                          '#,(syntax-span #'expr)))))]
       [else
        ;; Otherwise, leave the expression alone.
        #'expr])]))
 
(define (f x)
  (* (skeeterize x) x))
 
(define (g voldemort)
  (* (skeeterize voldemort) voldemort))
 
;; Examples:
(f 7)
(g 7)
;; The error should highlight the use
;; of voldemort in g.

10.2.7 境界和错误信息调整器

realm 确定了一个命名函数和指定函数参数和结果的契约的惯例。 境界的目的是帮助改善在 Racket 之上实现的语言之间的分层和互操作性。

领域主要使一种语言能够识别并重写由较低层的实现所产生的错误信息。 例如,一种语言的 "数组" 实现可能直接使用 Racket 向量,但当向量的对象类型或基元边界检查失败时, 生成的错误信息会提到 "向量",可能还有 vector? 这样的契约和 vector-ref 这样的函数名。 由于这些错误信息被识别为来自 'racket/primitive 领域, 语言实现可以寻找 'racket/primitive 来检测和重写错误信息, 而将应用程序的其他部分(可能用新语言实现)的错误信息混杂在一起的危险降到最低。

每个程序和每个模块也有一个领域。 一个过程的境界是相关的,例如,当它被应用于错误的参数数量时;在这种情况下,arity-error 消息本身来自 'racket/primitive 境界,但错误消息也应该包括过程的名称,它可以来自一些不同的境界。 按照类似的思路, continuation-mark-set->context 可以报告与一个 continuation 中的每一帧(程序)相关的领域,这对于识别边界的跨越可能是有用的。

错误信息的构建必须与错误信息的调整明确合作。 最基本的合作方式是通过 error-message->adjusted-string 和 error-contract->adjusted-string 这样的函数, 通过 current-error-message-adjuster 参数运行错误信息调整器, 以及使用 error-message-adjuster-key 作为 continuation-mark 键与当前 continuation 相关的其他调整器。 像 raise-argument-error 和 raise-arity-error 这样的函数使用 error-message->adjusted-string 和 error-contract->adjusted-string , 默认境界为 'racket 。 像 raise-argument-error* 和 raise-arity-error* 这样的函数接受一个明确的境界参数。

不是所有的错误函数都会自动与错误信息调整合作。 例如, raise-reader-error 和 raise-syntax-error 函数不调用调整器, 因为它们报告的是与语法密切相关的错误(以及,沿着这些思路,更静态的错误)。

procedure

(error-message->adjusted-string name    
  name-realm    
  message    
  message-realm) → string?
  name : (or/c symbol? #f)
  name-realm : symbol?
  message : string?
  message-realm : symbol?
将 name (如果不是 #f) 与 ": " 结合起来,然后 message 一个错误信息字符串,但首先给错误信息调整器一个机会来调整 name 和/或 message 。

任何与当前续集相关的调整器函数作为带有 error-message-adjuster-key 的 continuation mark 被首先运行;调整器按由浅入深的顺序运行。 然后,使用 current-error-message-adjuster 的调整器值。

每个调整器都先用 'message 协议进行尝试。 如果调整器对 'message 的响应是 #f ,那么就会尝试 'name 协议。 参见 current-error-message-adjuster 以了解有关协议的信息。 一个对两者都回应为 #f 的调整器将被跳过, 任何使用 error-message-adjuster-key 作为延续标记的值也被跳过,该值不是一个接受一个参数的过程。 此外,如果(可能的调整) name 是 #f,则跳过 'name 协议。

parameter

(current-error-message-adjuster)

 → (symbol? . -> . (or/c procedure? #f))
(current-error-message-adjuster proc) → void?
  proc : (symbol? . -> . (or/c procedure? #f))
一个确定错误信息调整器的 parameter , 在通过 error-message-adjuster-key 与当前延续相关的任何调整器之后应用。

调整器过程接收一个识别协议的符号,它必须返回 #f 或一个通过该协议执行调整的过程。 目前定义了以下协议,但将来可能会增加更多的协议:

  • 'name: 该过程接收两个参数,一个名称符号和一个境界符号; 它返回一个调整后的名称符号和一个调整后的境界符号。

  • 'message: 该过程接收四个参数:一个名字符号或 #f (这意味着在消息上没有名字的前缀), 一个名字-境界符号,一个消息字符串,以及一个消息-境界符号;它返回四个调整后的值。

  • 'contract: 该过程接收两个参数,一个合同字符串和一个境界符号; 它返回一个调整的合同字符串和一个调整的境界符号。

一个新的库或语言也可以引入额外的模式符号。 为了避免冲突,在模式符号前加一个集合或库的名称,后面加 / 。

如果一个调整程序为一个协议返回 #f ,这与返回一个不执行调整并返回其参数的函数是一样的。 这个参数的默认值对于任何符号参数都会返回 #f ,除了上面列出的协议,对于这些协议, 它会返回一个检查其参数并返回参数的过程,不进行调整。

Added in version 8.4.0.2 of package base.
