3.2 进口和出口: require 和 provide

输入：require in Racket 指南 introduces require.

syntax

(require require-spec ...)

 
require-spec = module-path
  | (only-in require-spec id-maybe-renamed ...)
  | (except-in require-spec id ...)
  | (prefix-in prefix-id require-spec)
  | (rename-in require-spec [orig-id bind-id] ...)

在 top-level context 中, require 将模块 instantiates (参见 模块和模块级变量)。 在 top-level context 或 module context 中, require 的扩展 visits 模块(参见 模块扩展、阶段和访问)。 在这两种上下文中,无论是评估还是扩展, require 都会将绑定引入 namespace 或模块(见 引入绑定)。 在 expression context 或 internal-definition context 中的 require 形式是一个语法错误。

require-spec 指定了在导入上下文中要绑定的一组特定标识符。 每个标识符都被映射到一个特定模块的特定出口; 要绑定的标识符可能与最初出口的标识符的符号名称不同。 每个标识符也在一个特定的 phase level 和 binding space 中进行绑定。

在给定的 phase level 和 binding space 的组合中, 没有一个标识符可以通过导入而被多次绑定,除非所有的绑定都是指同一模块中的同一个原始定义。 在 module context 中,一个标识符既可以被导入, 也可以为给定的 phase level 和 binding space 进行定义, 但不能同时进行。

require-spec 的语法可以通过 define-require-syntax 来扩展, 当在一个 require 中指定了多个 require-spec 时, 每个 require-spec 的绑定对扩展后面的 require-spec 是可见的。 预先定义的形式(由 racket/base 导出)如下:

module-path

从命名的模块中导入所有导出的绑定,使用导出名称作为本地标识符。 (关于 module-path 见下文)。 module-path 形式的词法上下文决定了引入标识符的上下文, 为特定 binding space 中的导出添加空间范围, 并在每个导出的 phase level 中。

如果由 module-path 提供的任何标识符的符号形式是 uninterned, 则该标识符不被导入(即,不可能为未加约束的符号导入一个约束)。 这个限制是为了避免编译差异,这取决于模块是否被保存到文件中(参见 打印编译后的代码)。

syntax

(only-in require-spec id-maybe-renamed ...)

和 require-spec 一样, 但仅限于那些需要绑定的标识符与 id-maybe-renamed 相匹配的出口: 作为 id 或作为 [orig-id bind-id] 中的 orig-id 。 如果任何 id-maybe-renamed 的 id 或 orig-id 不在 require-spec 所描述的集合中,将报告一个语法错误。

Examples:
> (require (only-in racket/tcp
                    tcp-listen
                    [tcp-accept my-accept]))
> tcp-listen

#<procedure:tcp-listen>

> my-accept

#<procedure:tcp-accept>

> tcp-accept

tcp-accept: undefined;

 cannot reference an identifier before its definition

  in module: top-level

syntax

(except-in require-spec id ...)

与 require-spec 类似,但省略了那些以 id 为标识符的导入; 如果任何 id 不在 require-spec 描述的集合中,将报告一个语法错误。

Examples:
> (require (except-in racket/tcp
                      tcp-listen))
> tcp-accept

#<procedure:tcp-accept>

> tcp-listen

tcp-listen: undefined;

 cannot reference an identifier before its definition

  in module: top-level

syntax

(prefix-in prefix-id require-spec)

和 require-spec 一样,但是通过在每个标识符前加上 prefix-id 来调整它。 prefix-id 的词法上下文被忽略,而是在前缀之前从标识符中保留下来。

Examples:
> (require (prefix-in tcp: racket/tcp))
> tcp:tcp-accept

#<procedure:tcp-accept>

> tcp:tcp-listen

#<procedure:tcp-listen>

syntax

(rename-in require-spec [orig-id bind-id] ...)

与 require-spec 类似,但用 orig-id 替换了绑定 bind-id 的标识符; 如果任何 orig-id 不在 require-spec 描述的集合中,将报告一个语法错误。

Examples:
> (require (rename-in racket/tcp
                      (tcp-accept accept)
                      (tcp-listen listen)))
> accept

#<procedure:tcp-accept>

> listen

#<procedure:tcp-listen>

syntax

(combine-in require-spec ...)

require-spec 的联合。如果两个或多个来自 require-spec 的导入具有相同的标识符名称,但它们并不指向相同的原始绑定,将报告一个语法错误。

Examples:
> (require (combine-in (only-in racket/tcp tcp-accept)
                       (only-in racket/tcp tcp-listen)))
> tcp-accept

#<procedure:tcp-accept>

> tcp-listen

#<procedure:tcp-listen>

syntax

(relative-in module-path require-spec ...)

与 require-spec 的联合一样, 但 require-spec 中的每个相对模块路径都被视为相对于 module-path 而不是包围的上下文。

实现 relative-in 的 require transformer 设置了 current-require-module-path 来调整 require-spec 中的模块路径。

syntax

(only-meta-in phase-level require-spec ...)

就像 require-spec 的组合一样,但要移除任何不属于 phase-level 的绑定, 其中 #f 代表 phase-level ,对应于 label phase level。

下面的例子只导入了 phase level 1 绑定,即转换阶段:

> (module nest racket
    (provide (for-syntax meta-eggs)
             (for-meta 1 meta-chicks)
             num-eggs)
    (define-for-syntax meta-eggs 2)
    (define-for-syntax meta-chicks 3)
    (define num-eggs 2))
> (require (only-meta-in 1 'nest))
> (define-syntax (desc stx)
    (printf "~s ~s\n" meta-eggs meta-chicks)
    #'(void))
> (desc)

2 3

> num-eggs

num-eggs: undefined;

 cannot reference an identifier before its definition

  in module: top-level

下面的例子只导入了 phase level 0 的绑定,即正常阶段。

> (require (only-meta-in 0 'nest))
> num-eggs

2

syntax

(only-space-in space require-spec ...)

就像 require-spec 的组合,但要删除任何未被 space 提供的 binding space 标识符的绑定–通常是一个标识符, 但 #f 为 space 对应的是 default binding space。

Added in version 8.2.0.3 of package base.

(for-meta phase-level require-spec ...)

和 require-spec 的组合一样, 但是每个 require-spec 所指定的绑定是通过 phase-level 来转移的。 label phase level 对应于 #f , 涉及 #f 的移位组合产生 #f。

Examples:
> (module nest racket
    (provide num-eggs)
    (define num-eggs 2))
> (require (for-meta 0 'nest))
> num-eggs

2

> (require (for-meta 1 'nest))
> (define-syntax (roost stx)
    (datum->syntax stx num-eggs))
> (roost)

2

(for-syntax require-spec ...)

与 (for-meta 1 require-spec ...) 相同。

(for-template require-spec ...)

与 (for-meta -1 require-spec ...) 相同。

(for-label require-spec ...)

与 (for-meta #f require-spec ...)相同。 如果任何一个 require-spec中的标识符被绑定在一个以上的阶段级别, 将报告一个语法错误。

(for-space space require-spec ...)

和 require-spec 的组合一样,但是每个 require-spec 所指定的绑定被移到 space 所指定的 binding space ,space 通常是一个标识符,但是 #f 代表 space 对应于 default binding space。

绑定被移动到新的空间,方法是移除原先由 require-spec 暗示的空间的范围(如果有的话), 并添加 space 的范围(如果有的话)。

Added in version 8.2.0.3 of package base.

derived-require-spec

关于扩展 require-spec 形式的信息, 请参见 define-require-syntax 。

模块路径 in Racket 指南 introduces module paths.

module-path 标识一个模块,可以是根模块, 也可以是在另一个模块中以词法声明的 submodule 。 根模块可以通过标识符形式的具体名称标识, 也可以通过可以触发模块声明自动加载的间接名称标识。 除了下面的 (quote id) 情况外, 根模块路径的实际解析取决于当前 module name resolver (参见 current-module-name-resolver), 下面的描述对应于默认的 module name resolver。

(quote id)

指以前以 id 为名声明的子模块或以前以 id 为名交互声明的模块。 当 id 指的是一个子模块时, (quote id) 等同于 (submod "." id)。

Examples:
; a module declared interactively as test:
> (require 'test)

rel-string

一个相对于包含源的路径 (由 current-load-relative-directory 或 current-directory 决定)。 无论当前平台如何, rel-string 总是被解析为Unix格式的相对路径: / 是路径分隔符(不允许有多个相邻的 /, .. 访问父目录, . 访问当前目录。 路径不能为空,也不能包含前导或尾部斜线,最后一个之前的路径元素不能包含文件后缀 (即,在 . 或 .. 之外的元素中的 . ), 唯一允许的字符是 ASCII 字母、ASCII 数字、 -, +, _, ., / 和 % 。 此外, % 只允许在两个小写的十六进制数字之后,而且这些数字必须构成一个数字, 而不是字母、数字、 -, + 或 _。

% 规定的目的是支持任意字符串作为路径元素的一对一编码(在UTF-8编码之后)。 这样的编码不会被解码以得出文件名,而是保留在文件访问中。

如果 rel-string 以 ".ss" 后缀结尾, 它将被转换为 ".rkt" 后缀。 如果不存在 ".rkt" 文件而存在 ".ss" ， compiled-load handler 可能会逆转这种转换。

Examples:
; a module named "x.rkt" in the same
; directory as the enclosing module's file:
> (require "x.rkt")
; a module named "x.rkt" in the parent directory
; of the enclosing module file's directory:
> (require "../x.rkt")

syntax

(lib rel-string ...+)

一个安装到 collection 中的模块的路径 (参见 Libraries and Collections)。 lib 中的 rel-string 与 rel-string 的情况类似,有一个额外的约束,即一个 rel-string 不能包含 . 或 .. 目录指示符。

路径的具体解释取决于 rel-string 的数量和形状:

  • 如果提供了一个单一的 rel-string , 并且如果它由单一元素(即没有 /) 组成,没有文件后缀 (即没有 .), 那么 rel-string 命名了一个 collection, "main.rkt" 是库文件名。

    Examples:
    ; the main swindle library:
    > (require (lib "swindle"))
    ; the same:
    > (require (lib "swindle/main.rkt"))

  • 如果提供了一个单一的 rel-string , 并且如果它由多个 / 分隔的元素组成, 那么直到最后一个元素都会命名一个 collection、子集合等, 最后一个元素命名一个文件。如果最后一个元素没有文件后缀, ".rkt" 将被添加,而 ".ss" 后缀将被转换为 ".rkt"。

    Examples:
    ; "turbo.rkt" from the "swindle" collection:
    > (require (lib "swindle/turbo"))
    ; the same:
    > (require (lib "swindle/turbo.rkt"))
    ; the same:
    > (require (lib "swindle/turbo.ss"))

  • 如果提供了一个单一的 rel-string , 并且如果它由一个 带有 文件后缀(即带有 .) 的单一元素组成, 那么 rel-string 命名了 "mzlib" collection 中的一个文件。 后缀为 ".ss" 的文件被转换为 ".rkt" 。 (这个约定是为了与旧版本的 Racket 兼容)。

    Examples:
    ; "tar.rkt" module from the "mzlib" collection:
    > (require (lib "tar.ss"))

  • 否则,当提供多个 rel-string 时, 第一个 rel-string 被有效地移到其他的 rel-string 之后, 并且所有的 rel-string 都附加了 / 分隔符。 产生的路径命名一个 collection ,然后是子集合,等等,最后是一个文件名。 没有后缀被自动添加,但 ".ss" 后缀被转换为 ".rkt" 。 (这个惯例是为了与旧版本的 Racket 兼容)。

    Examples:
    ; "tar.rkt" module from the "mzlib" collection:
    > (require (lib "tar.ss" "mzlib"))

id

lib 形式的简称,它有一个单一的 rel-string , 其字符与 id 的符号形式相同。 除了 lib rel-string 的约束外, id 还不能包含 .。

Example:
> (require racket/tcp)

syntax

(file string)

类似于普通 rel-string 的情况, 但 string 是一个路径,可能是绝对的, 使用当前平台的路径惯例和 expand-user-path 。 后缀 ".ss" 被转换为 ".rkt"。

Example:
> (require (file "~/tmp/x.rkt"))

syntax

(planet id)

(planet string)
(planet rel-string (user-string pkg-string vers)
        rel-string ...)
指定一个通过 PLaneT 服务器可用的库。

第一种形式是对最后一种形式的速记, 其中 id 的字符序列必须符合以下 ‹spec› 语法:

 

‹spec›

 ::= 

‹owner› / ‹pkg› ‹lib›

 

‹owner›

 ::= 

‹elem›

 

‹pkg›

 ::= 

‹elem›  |  ‹elem› : ‹version›

 

‹version›

 ::= 

‹int›  |  ‹int› : ‹minor›

 

‹minor›

 ::= 

‹int›  |  <= ‹int›  |  >= ‹int›  |  = ‹int›

 

  |  

‹int› - ‹int›

 

‹lib›

 ::= 

‹empty›  |  / ‹path›

 

‹path›

 ::= 

‹elem›  |  ‹elem› / ‹path›

其中 ‹elem› 是一个非空的字符序列,是 ASCII 字母、ASCII 数字、 -, +, _ 或 % , 后面是小写十六进制数字(不编码其他允许的字符之一),而 ‹int› 是一个非空的 ASCII 数字序列。 随着这种速记的展开, ".plt" 扩展名被添加到 ‹pkg› 中, ".rkt" 扩展名被添加到 ‹path› 中; 如果不包括 ‹path› , "main.rkt" 被用于展开。

(planet string) 表格与 (planet id) 表格一样, 标识符被转换为字符串,只是 string 可以选择以文件扩展名 (即,一个 .) 作为 ‹path› 结尾。 一个 ".ss" 文件扩展名被转换为 ".rkt"。

在 planet 模块路径的更一般的最后一种形式中, rel-string 与 lib 形式类似, 只是 (user-string pkg-string vers) 命名了一个基于 PLaneT 的包, 而不是一个 collection。 一个版本规范可以包括一个可选的主要版本和次要版本, 其中次要版本可以是一个特定的数字,也可以是一个约束条件: (nat nat) 指定了一个包容性的范围, (= nat) 指定了一个精确的匹配, (+ nat) 指定了一个最小的版本, 相当于只有 nat, 而 (- nat) 指定了一个最大的版本。 在一个次要版本约束中的 =, + 和 - 标识符是以符号形式识别的。

Examples:
; "main.rkt" in package "farm" by "mcdonald":
> (require (planet mcdonald/farm))
; "main.rkt" in version >= 2.0 of "farm" by "mcdonald":
> (require (planet mcdonald/farm:2))
; "main.rkt" in version >= 2.5 of "farm" by "mcdonald":
> (require (planet mcdonald/farm:2:5))
; "duck.rkt" in version >= 2.5 of "farm" by "mcdonald":
> (require (planet mcdonald/farm:2:5/duck))

syntax

(submod root-module-path submod-path-element ...)

(submod "." submod-path-element ...)
(submod ".." submod-path-element ...)
识别 root-module-path 指定的模块中的一个 submodule , 或者在 (submod "." ....) 的情况下相对于当前模块, 其中 (submod ".." submod-path-element ...) 等同于 (submod "." ".." submod-path-element ...)。 子模块有符号名称,作为 submod-path-element 的标识符序列决定了一个具有给定名称的连续嵌套子模块的路径。 作为 submod-path-element 的 ".." 命名了一个子模块的包围模块, 它旨在用于 (submod "." ....) 和 (submod ".." ....) 的形式中。

当 require 准备处理一个 require-spec 序列时, 它将一条 “预取” 消息记录在 current logger 的 'info 级别, 使用名称为 'module-prefetch ,并包括包含两个元素的消息数据: 一个似乎已导入的 module paths 列表,以及一个用于相对模块路径的目录路径。 记录的模块路径列表可能是不完整的,但是编译管理器可以使用近似的预取信息来启动并行的编译。

Changed in version 6.0.1.10 of package base: 增加了预取记录。

syntax

(local-require require-spec ...)

和 require 一样,但在 internal-definition context 中使用,只是导入到本地上下文。 只有来自 phase level 0 的绑定被导入。

Examples:
> (let ()
    (local-require racket/control)
    fcontrol)

#<procedure:fcontrol>

> fcontrol

fcontrol: undefined;

 cannot reference an identifier before its definition

  in module: top-level

输出：provide in Racket 指南 introduces provide.

syntax

(provide provide-spec ...)

 
provide-spec = id
  | (all-defined-out)
  | (all-from-out module-path ...)

宣布从一个模块中输出。 provide 形式必须出现在 module context 或 module-begin context 中。

一个 provide-spec 表示要提供一个或多个绑定 。 对于每个导出的绑定,外部名称是一个符号,可以与模块内绑定的标识符的符号形式不同。 另外,每个导出都是从一个特定的 phase level 中抽取的,并在同一 phase level 中导出; 默认情况下,相关的阶段级别是包围 provide 形式的 begin-for-syntax 形式的数量。 最后,每个出口都来自一个 binding space, 并在同一 binding space 输出。

provide-spec 的语法可以通过绑定来扩展, 以 provide transformers 或 provide pre-transformers, 比如通过 define-provide-syntax, 但预先定义的形式如下。

id

输出 id, 它必须在相关的 phase level 和 binding space 中被 bound 在模块内(即,要么定义,要么导入)。 id 的符号形式被用作外部名称, 而定义的或导入的标识符的符号形式必须匹配(否则,外部名称可能是模糊的)。

Examples:
> (module nest racket
    (provide num-eggs)
    (define num-eggs 2))
> (require 'nest)
> num-eggs

2

如果 id 有一个转化器与 rename transformer 绑定, 那么该转化器会影响导出的绑定。 参见 make-rename-transformer 获取更多信息。

(path-up rel-string ...)

指定由 rel-string 命名的模块的路径,与直接使用 rel-string 类似, 只是如果没有找到相对于包围源的所需模块文件, 就会在父目录中搜索,然后在祖目录中搜索,等等,一直到根目录。 发现的相对于包围源的路径成为扩展形式的一部分。

这个表格在设置 "项目环境" 时很有用。 例如,在你项目的根目录下使用以下 "config.rkt" 文件:
#lang racket/base
(require racket/require-syntax
         (for-syntax "utils/in-here.rkt"))
 
(provide utils-in)
(define-require-syntax utils-in in-here-transformer)
并在同一根目录下使用 "utils/in-here.rkt" :
#lang racket/base
(require racket/runtime-path)
(provide in-here-transformer)
(define-runtime-path here ".")
(define (in-here-transformer stx)
  (syntax-case stx ()
    [(_ sym)
     (identifier? #'sym)
     (let ([path (build-path here (format "~a.rkt" (syntax-e #'sym)))])
       (datum->syntax stx `(file ,(path->string path)) stx))]))
那么 path-up 对项目目录下的任何其他模块都有效,可以找到 "config.rkt":
(require racket/require
         (path-up "config.rkt")
         (utils-in foo))
注意,示例中require的顺序很重要,因为前两个都绑定了下面使用的标识符。

在这种情况下,另一种方法是直接使用 path-up 来寻找实用模块:
(require racket/require
         (path-up "utils/foo.rkt"))
但随后被称为 "utils" 的子目录将覆盖项目根目录中的那个。 换句话说,前面的方法只需要一个唯一的名字。

sub-path = rel-string
  | id
指定从目录或集合的层次结构中需要的多个文件。 所需的模块路径集被计算为 subs 组的笛卡尔乘积, 其中每个 sub-path 与其他 sub-path 按顺序使用 / 分隔符组合。 作为 subs 的 sub-path 等同于 (sub-path)。 在一个给定的 multi-in 形式中的所有 sub-path 必须是字符串或标识符。

Examples:

(require (multi-in racket (dict list)))

   is equivalent to 

(require racket/dict racket/list)

 

(require (multi-in "math" "matrix" "utils.rkt"))

   is equivalent to 

(require "math/matrix/utils.rkt")

 

(require (multi-in "utils" ("math.rkt" "matrix.rkt")))

   is equivalent to 

(require "utils/math.rkt" "utils/matrix.rkt")

 

(require (multi-in ("math" "matrix") "utils.rkt"))

   is equivalent to 

(require "math/utils.rkt" "matrix/utils.rkt")

 

(require (multi-in ("math" "matrix") ("utils.rkt" "helpers.rkt")))

   is equivalent to 

(require "math/utils.rkt" "math/helpers.rkt"
         "matrix/utils.rkt" "matrix/helpers.rkt")

3.2.2 额外 provide 形式

 (require racket/provide) package: base
The bindings documented in this section are provided by the racket/provide library, not racket/base or racket.

syntax

(matching-identifiers-out regexp provide-spec)

与 provide-spec 类似, 但只包括具有匹配 regexp 的外部名称的绑定的输出。 regexp 必须是一个字面的正则表达式(参见 正则表达式)。

syntax

(filtered-out proc-expr provide-spec)

类似于 filtered-in, 但用于过滤和重命名出口。

For example,
(provide (filtered-out
          (lambda (name)
            (and (regexp-match? #rx"^[a-z-]+$" name)
                 (regexp-replace
                  #rx"-" (string-titlecase name) "")))
          (all-defined-out)))
只输出符合模式 #rx"^[a-z-]+$" 的绑定,并将名称转换为 “camel case。”
