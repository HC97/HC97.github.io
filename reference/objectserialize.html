6.9 对象序列化

syntax

(define-serializable-class* class-id superclass-expr
                                     (interface-expr ...)
  class-clause ...)
将 class-id 绑定到一个类,其中 superclass-expr, interface-expr 和 class-clause 与 class* 中的一样。

这种形式只能在顶层使用,无论是在模块内部还是外部。 class-id 标识符被绑定到新的类上, 并且 deserialize-info:class-id 也被定义; 如果定义在一个模块内,那么后者是通过 module+ 从 deserialize-info 子模块中提供。

类的序列化以两种方式中的一种工作:

  • 如果该类实现了内置接口 externalizable<%>, 那么通过调用其 externalize 方法来序列化一个对象; 其结果可以是任何可序列化的东西(但显然不应该是对象本身)。 反序列化创建了一个没有初始化参数的类的实例,然后用 externalize 的结果 (或者更准确地说,是之前调用的序列化结果的反序列化版本)调用该对象的 internalize 方法。

    为了支持这种形式的序列化,该类必须是可实例化的,没有初始化参数。 此外,只涉及该类(和其他此类类)的实例的循环不能被序列化。

  • 如果该类没有实现 externalizable<%>, 那么该类的每个超类都必须是可序列化的或透明的(即,有 #f 作为其检查器)。 序列化和反序列化是完全自动的,并且可能涉及实例的循环。

    为了支持实例的循环,反序列化可能会创建一个调用所有字段为未定义值的实例,然后变异对象以设置字段值。 串行化支持并不以其他方式使对象的字段可变。

在第二种情况下,可序列化的子类可以实现 externalizable<%>, 在这种情况下, externalize 方法负责所有的序列化(也就是说,子类的实例会失去自动序列化)。 在第一种情况下,所有可序列化的子类都实现了 externalizable<%>, 因为一个子类实现了它的父类的所有接口。

在这两种情况下,如果一个对象是一个子类的直接实例(该子类本身不是可序列化的), 那么该对象被序列化,就像它是可序列化类的直接实例一样。 特别是,对于不可序列化的子类的实例, externalize 方法的覆盖声明被忽略。

syntax

(define-serializable-class class-id superclass-expr
  class-clause ...)
类似于 define-serializable-class*, 但没有接口表达式(类似于 class)。

interface

externalizable<%> : interface?

externalizable<%> 接口只包括 externalize 和 internalize 方法。 更多信息参见 define-serializable-class* 。
