3.1 Modules: module, module*, ...

module 表 in Racket 指南 introduces module.

syntax

(module id module-path form ...)

声明一个顶层模块或一个 submodule 。 对于一个顶层模块,如果设置了 current-module-declare-name 参数, 参数值被用于模块名称, id 被忽略, 否则 (quote id) 是声明的模块名称。 对于一个 submodule, id 是子模块的名称, 作为 submod 模块路径中的一个元素使用。 在 expression context 或 internal-definition context 文中不允许使用 module 形式。

对于在顶层或模块主体以外的定义上下文中工作的类似 module 的形式, 有 define-package, 但使用一个单独的模块或 submodule 通常更好。

module-path 的形式必须和 require 一样, 它为主体 form 形式提供初始绑定。 也就是说,它被视为在 form 前的 (require module-path) 前缀, 除了 module-path 引入的绑定可以被模块主体 form 中的定义和 require 所掩盖。

如果提供了一个单一的 form , 那么它在 module-begin context 的上下文中被部分展开。 如果扩展导致 #%plain-module-begin , 那么 #%plain-module-begin 的主体就是该模块的主体。 如果部分扩展导致任何其他原始形式,那么该形式被 #%module-begin 包裹, 使用模块主体的词法上下文;该标识符必须被初始 module-path 导入绑定, 其扩展必须产生一个 #%plain-module-begin 来提供模块主体。 最后,如果提供了多个 form , 它们会被 #%module-begin 包裹, 就像单个 form 没有扩展到 #%plain-module-begin 的情况那样。

在这种包装之后,如果有的话,在任何扩展之前, 一个 'enclosing-module-name 属性被附加到 #%module-begin 语法对象上 (参见 语法对象属性);该属性的值是一个与 id 对应的符号。

每个 form 在 module context 中被部分展开(参见 局部扩张) 。 进一步的行动取决于表单的形状:

  • 如果它是一个 begin 形式,子形式就会被平铺到模块的主体中, 并立即代替 begin 进行处理。

  • 如果是 define-syntaxes 形式, 那么右手边会被评估(在 phase 1), 并且绑定会被立即安装,以便在模块内进一步部分扩展。 右手边的评估被 parameterize , 以设置 current-namespace ,就像在 let-syntax 中一样。

  • 如果它是一个 begin-for-syntax 形式, 那么主体将被展开(在 phase 1) 并被评估。 在 begin-for-syntax 表单内的扩展与 module 主体的部分扩展过程相同, 但在更高的 phase, 并保存所有阶段的 #%provide 表单, 直到 module 的扩展结束。 对主体的评估是 parameterize ,与 let-syntax 中一样, 设置 current-namespace 。

  • 如果表单是 #%require 表单,则立即引入绑定, 并根据情况对导入的模块进行 instantiate 或 visit 。

  • 如果该表格是 #%provide 表格,那么它将被记录在正文的其他部分之后进行处理。

  • 如果表单是一个 define-values 表单, 那么绑定会被立即安装,但是右侧的表达式不会被进一步扩展。

  • 如果该表单是一个 module 表单, 那么它将被立即展开,并在当前顶层包围模块的范围内声明。

  • 如果该表单是一个 module* 表单,那么它不会被进一步展开。

  • 同样地,如果表单是一个表达式,那么它也不会被进一步展开。

在所有的 form 都以这种方式被部分展开后, 然后剩余的表达式(包括定义的右侧)在表达式上下文中被展开。 在所有表达式之后, #%provide 形式按照它们在扩展的模块中出现的顺序 (与 phase)无关)进行处理。 最后,所有的 module* 形式被依次展开, 这样每个形式都可以被随后的 module* 形式使用;包围的模块本身也可以被 module* submodules 使用。

所有导入的标识符的范围包括整个模块体, 除了嵌套的 module 和 module* 形式 (假设后者是一个非 #f module-path )。 在模块体内定义的任何标识符的范围同样涵盖整个模块体, 除了这种嵌套的 module 和 module* 形式。 语法定义的排序不影响语法名称的范围; A 的转化器可以产生包含 B 的表达式, 而 B 的转化器则产生包含 A 的表达式, 不管 A 和 B 的声明顺序如何。 然而,产生语法定义的语法形式在使用前必须被定义。

除了通过 define-values 或 define-syntaxes 的定义可以通过 #%require 影射导入,只要前面的 #%declare 形式不包括 #:require=defined , 那么任何标识符在单个模块的任何 phase level 都不能被导入或定义超过一次。 每个导出的标识符都必须被导入或定义。 任何表达式都不能引用一个 top-level variable。 一个 module* 的形式,其中包围模块的绑定是可见的 (也就是说,一个嵌套的 module* 用 #f 代替 module-path) ,可以定义或导入绑定,以 shadow 包围模块的绑定。

对 module 形式的评估并不评估模块主体中的表达式 (有时对重新声明除外;参见 模块的重新声明) 。 评估只是声明一个模块,其全名同时取决于 id 或 (current-module-declare-name)。

只有当模块通过 require 或 dynamic-require 被显式 instantiate 时,模块主体才会被执行。 在调用时,导入的模块按照它们被 require 到模块中的顺序被实例化(尽管较早的实例化或反式 require 可以在给定模块中的顺序之前触发一个模块的实例化)。 然后,表达式和定义按照它们在模块中出现的顺序被评估。 表达式或定义的每一次评估都被默认 prompt tag 的继续提示 (参见 call-with-continuation-prompt) 所包裹, 并使用提示处理程序重新中止并将其参数传播到下一个包围的提示。 每次对定义进行评估时,都会在提示符之外检查定义的每个变量是否有值; 如果跳过提示符限定的延续部分来安装值,就会引发 exn:fail:contract:variable? exception is raised。

在更高的阶段,模块主体的部分与运行时的部分有类似的界限。 例如,在模块展开时和被访问时, 模块在 begin-for-syntax 内的部分都由一个继续提示划定。 define-syntaxes 形式的评估是有界限的, 但与 define-values 不同,没有检查语法定义是否完成。

在一个 module-level variable被定义之前访问它, 就像访问一个未定义的全局变量一样,是一个运行时错误的信号。 如果一个模块(以其完全展开的形式)不包含一个在模块内定义的标识符的 set! ,那么这个标识符在被定义后就是一个 constant ; 它的值在之后不能被改变,甚至不能通过反射机制。 然而, compile-enforce-module-constants 参数可以用来禁用常量的执行。

当代表 module 形式的 syntax object 附加了 'module-language syntax property , 并且该属性值是一个由三个元素组成的向量, 其中第一个元素是模块路径(在 module-path?的意义上), 第二个元素是一个符号,那么该属性值在相应的编译和/或声明的模块中被保留下来。 矢量的第三个分量应该是可打印和可 read , 这样它就可以在 marshaled 字节码中被保留下来。 racket/base 和 racket 语言将 '#(racket/language-info get-info #f) 附加到一个 module 的形式。 参见 module-compiled-language-info, module->language-info 和 racket/language-info。

参见 模块和模块级变量, 模块扩展、阶段和访问 和 扩展模块的信息。

Example:
> (module duck racket/base
    (provide num-eggs quack)
    (define num-eggs 2)
    (define (quack n)
      (unless (zero? n)
        (printf "quack\n")
        (quack (sub1 n)))))

Changed in version 6.3 of package base: 修改了 define-syntaxes 和 define-values ,使其对前面的导入有阴影, 并放弃了在嵌套 module 或 module* 形式上使用 'submodule syntax property 。

syntax

(module* id module-path form ...)

(module* id #f form ...)

子模块 in Racket 指南 introduces module*.

和 module 一样,但只适用于在一个模块中声明一个 submodule , 以及可能 require 包围模块的子模块。

在 id 后面没有 module-path , #f 表示所有来自包围模块的绑定在子模块中是可见的。 在这种情况下,包裹 module* 形式的 begin-for-syntax 形式会转移包围模块的绑定相对于子模块的 phase level 。 宏扩展器处理这种嵌套的方法是转移 module* 形式的 phase level , 使其主体从 phase level 0 开始,扩展,然后恢复 phase level 的转移; 注意这个过程会使 syntax objects 作为 'origin syntax property 值与扩展后的模块不同步。

当一个 module* 形式有一个 module-path 时, 子模块的扩展开始于移除包围模块的 scopes ,与 module 形式相同。 没有移位补偿任何可能包裹子模块的 begin-for-syntax 形式。

syntax

(module+ id form ...)

main 和 test 子模块 in Racket 指南 introduces module+.

声明和/或添加到一个名为 id 的 submodule 。

每个 id 的加法都被合并,以形成整个子模块, 使用 (module* id #f ....) 在包围模块的末尾。 如果一个给定的 id 只有一个 module+ , 那么 (module+ id form ...) 等同于 (module* id #f form ...), 但仍然移到包围模块的末尾。

在 module* 表单上有一个 syntax property, 其键值为 'origin-form-srcloc,记录每个贡献的 module+ 表单的 srcloc。

当一个模块包含多个用 module+ 声明的子模块时, 那么每个子模块的初始 module+ 声明的相对顺序决定了包围模块末尾的 module* 声明的相对顺序。

一个子模块不能用 module+ 和 module 或 module* 来定义。 也就是说,如果一个子模块是由 module+ 片构成的, 那么它必须 只 由 module+ 片构成。

Changed in version 8.9.0.1 of package base: 增加了 'origin-form-srcloc 语法属性。

syntax

(#%module-begin form ...)

只在 module begin context 的情况下合法, 并由 module 和 module* 形式处理。

racket/base 的 #%module-begin 形式包裹了每个顶层表达式, 使用 current-print 打印非 #<void> 结果。

racket/base 的 #%module-begin 形式也声明了一个 configure-runtime 子模块(在任何其他 form 之前), 除非某些 form 是直接 module 或 module* 形式, 名称是 configure-runtime 。 如果一个 configure-runtime 子模块被添加, 该子模块会调用 racket/runtime-config 的 configure 函数。

syntax

(#%printing-module-begin form ...)

只在 module begin context 中合法。

就像 #%module-begin,但没有添加 configure-runtime 子模块。

syntax

(#%plain-module-begin form ...)

只在 module begin context 中合法, 并由 module 和 module* 形式处理。

syntax

(#%declare declaration-keyword ...)

 
declaration-keyword = #:cross-phase-persistent
  | #:empty-namespace
  | #:require=define
  | #:unsafe
  | #:realm identifier
影响模块的运行时间或反射属性的声明:

  • #:cross-phase-persistent — 宣布该模块为 cross-phase persistent, 如果该模块不符合 cross-phase persistent 模块的导入或句法限制,则报告语法错误。

  • #:empty-namespace — 声明该模块的 module->namespace 应该产生一个没有绑定的命名空间; 以这种方式限制命名空间支持,可以减少必须为模块保留的 lexical information 。

  • #:require=define — 声明不允许紧随模块主体的后续定义影射 #%require (或 require) 绑定。 该声明不影响模块初始导入的阴影(即模块的语言)。

  • #:unsafe — 声明该模块在编译时可以不进行可能触发 exn:fail:contract 的检查,并且在 exn:fail:contract 应该被提出的情况下,所产生的行为是未指定的;也请看 不安全的操作 。 例如, car 的使用可以被编译为 unsafe-car 的使用, 而 unsafe-car 被应用到非对上时,其行为是未指定的。 只有当当前的 code inspector 是初始检查器时, 才允许使用 #:unsafe 声明关键字。 宏可以生成有条件的不安全代码,这取决于扩展环境,通过扩展到使用 (variable-reference-from-unsafe? (#%variable-reference))。

  • #:realm identifier — 声明模块和模块中的任何程序都被赋予一个境界,它是 identifier 的符号形式, 有效地覆盖了 current-compile-realm 的值。

#%declare 形式必须出现在 module context 或 module-begin context 中。 每个 declaration-keyword 在一个 module 体中最多可以声明一次。

Changed in version 6.3 of package base: 增加 #:empty-namespace。
Changed in version 7.9.0.5: 增加 #:unsafe。
Changed in version 8.4.0.2: 增加 #:realm。
Changed in version 8.6.0.9: 增加 #:require=define。
