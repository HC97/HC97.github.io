13.4 Reading

procedure

(read [in]) → any

  in : input-port? = (current-input-port)
从 datum 读取并返回单个 in 。 如果 in 有一个通过 port-read-handler 与其关联的处理程序,则调用该处理程序。 否则,将使用默认读取器,由 current-readtable 参数以及许多其他参数参数化。

有关默认读卡器的信息,请参见 阅读器 ,有关 read 协议的信息, 参见 通过扩展阅读 。

procedure

(read-syntax [source-name in]) → (or/c syntax? eof-object?)

  source-name : any/c = (object-name (current-input-port))
  in : input-port? = (current-input-port)
和 read 一样,但产生一个带有源位置信息的 syntax object 。 source-name 被用作语法对象的源字段;它可以是一个任意的值,但它通常应该是源文件的路径。

关于 read-syntax 模式下的默认阅读器的信息请参见 阅读器 , 关于 read-syntax 的协议请参见 通过扩展阅读 。

通常情况下,应该为 in 启用行计数,以便 语法对象中的源位置是以字符为单位,而不是以字节。 也请参见 计算位置、行和列 。

See also 语法对象 in Racket 指南.

procedure

(read/recursive [in start readtable graph?]) → any

  in : input-port? = (current-input-port)
  start : (or/c char? #f) = #f
  readtable : (or/c readtable? #f) = (current-readtable)
  graph? : any/c = #t
类似于调用 read ,但通常是在 read 的动态范围内, 在 read 的扩展过程中使用(见 读者扩展程序) 。 使用 read/recursive 而不是 read 的主要影响是, 嵌套的read中的图结构注释(见 读数图结构) 被认为是整个 read 的一部分, 至少在 graph? 参数为真时是这样;但是由于结果被包裹在占位符中,所以不能直接检查。

如果提供了 start ,而不是 #f, 那么它实际上是预置在读的 in 的流的开头。 (要给多个字符加前缀,请使用 input-port-append 。)

readtable 参数用于顶层解析,以满足读取请求, 包括各种内置的顶层形式的定界符(例如用于读取哈希表的括号和 . ); 读取中的递归解析(例如,读取一个列表的元素)则使用由 current-readtable 参数决定的当前读取表。 读取器宏可以用一个字符和 readtable 调用 read/recursive ,以有效地调用 readtable 对该字符的行为。 如果 readtable 是 #f, 默认的 readtable 被用于顶层解析。

当 graph? 为 #f 时,读取的数据中的图结构注释是数据的局部。

当在 read 的动态范围内调用时,当输入流的第一个非空白内容解析为注释时, read/recursive 过程可以产生一个特殊的注释值(见 特殊注释) 。

参见 读取表 ,了解一个使用 read/recursive 的扩展示例。

Changed in version 6.2 of package base: 调整了 readtable 的使用,使其更一致地适用于内置表格的定界符。

procedure

(read-syntax/recursive [source-name    
  in    
  start    
  readtable    
  graph?]) → any
  source-name : any/c = (object-name in)
  in : input-port? = (current-input-port)
  start : (or/c char? #f) = #f
  readtable : (or/c readtable? #f) = (current-readtable)
  graph? : any/c = #t
类似于调用 read/recursive, 但是产生的值封装了带有源位置信息的S-表达式结构。 和 read/recursive 一样, 当 read-syntax/recursive 在 read-syntax 的动态范围内使用时, read-syntax/recursive 的结果是一个特殊的注释值,文件结束, 或者不透明的图形结构占位符(不是一个语法对象)。 占位符可以被嵌入到一个 S 表达式或由阅读器宏返回的语法对象中,等等, 在最外层的 read-syntax 返回之前,它将被替换成实际的语法对象。

在 read-syntax 的动态范围内使用 read/recursive 并不允许将用于阅读的图结构包含在外层 read-syntax 解析中, 在 read 的动态范围内使用 read-syntax/recursive 也不允许。 在这些情况下, read/recursive 和 read-syntax/recursive 产生的结果与 read 和 read-syntax 一样, 只是当输入流以注释开始时(在空白处之后)会返回一个特殊的注释值。

参见 读取表 ,了解一个使用 read-syntax/recursive 的扩展例子。

Changed in version 6.2 of package base: 调整了 readtable 的使用方式, 与 read/recursive 的方式相同。

procedure

(read-language [in fail-thunk])

 → (or/c (any/c any/c . -> . any) #f)
  in : input-port? = (current-input-port)
  fail-thunk : (-> any) = (lambda () (error ...))
以与 read 相同的方式从 in 读取, 但一旦确定有 reader language (或没有读者语言)就会停止, 并使用 current namespace 来加载一个读者模块, 而不是其 root namespace (如果它们不同的话)。

reader language 由 #lang 或 #! (见 通过扩展阅读) 在输入的开头指定, 尽管可能在注释形式之后。 默认的读表是由 read-language 使用的(而不是 current-readtable), #reader 形式(可能产生注释)在 #lang 或 #! 之前是不允许的。

See also 源处理配置 in Racket 指南.

当它发现一个 #lang 或 #! 规范时, read-language 不会像 read 和 read-syntax 那样派发到 read 或 read-syntax 函数, 而是派发到同一模块输出的 get-info 函数(如果有的话)。 get-info 的参数与 通过扩展阅读 中描述的 read 的参数相同。 get-info 函数的结果是 read-language 的结果,如果它是一个两个参数的函数; 如果 get-info 产生任何其他类型的结果,就会引发 exn:fail:contract exception is raised 。 如果没有导出 get-info 函数, read-language 会返回 #f 。

get-info 产生的函数反映了关于输入流的预期语法的信息。 该函数的第一个参数作为此类信息的关键;可接受的关键和结果的解释由外部工具决定,如 DrRacket(见 the DrRacket documentation)。 如果没有给定键的信息,结果应该是第二个参数。

Examples:
> (define scribble-manual-info
    (read-language (open-input-string "#lang scribble/manual")))
> (scribble-manual-info 'color-lexer #f)

#<procedure:scribble-inside-lexer>

> (scribble-manual-info 'something-else #f)

#f

get-info 函数本身适用于五个参数:被读取的输入端口, get-info 函数提取的模块路径, 以及 #lang 或 #! 形式开始的源行(正精确整数或 #f)、 列(非负精确整数或 #f)和位置(正精确整数或 #f) 。 get-info 函数可以进一步从给定的输入端口读取数据以确定其结果,但它不应该读取超过必要的数据。 get-info 函数在返回一个函数之后,不应该再从端口中读取。

如果 in 以一个 reader language 规范开始, 但相关模块没有输出 get-info (但也许输出了 read 和 read-syntax), 那么 read-language 的结果是 #f 。

如果 in 有一个 #lang 或 #! 的规范, 但解析和解决这个规范会引发一个异常,那么这个异常会被 read-language 传播。 至少有 #l 或 #! (在注释和空白之后)才算开始了 #lang 或 #! 的规范。

如果 in 没有用 #lang 或 #! 指定一个 reader language , 那么就会调用 fail-thunk 。 默认的 fail-thunk 会引发 exn:fail:read 或 exn:fail:read:eof 。

parameter

(read-case-sensitive) → boolean?

(read-case-sensitive on?) → void?
  on? : any/c
一个控制符号解析和打印的 parameter 。 当这个参数的值为 #f 时,阅读器会对符号进行大小写折叠 (例如,当输入是 hi, Hi, HI 或 hI 中的任何一个时, 会产生 'hi )。 该参数也会影响到 write 打印包含大写字母的符号的方式;如果该参数的值是 #f, 那么打印的符号就会用大写字母引号 \ 或 | 。 该参数的值被引号 \ 或 | 的竖条引号以及 #cs 和 #ci 前缀所覆盖; 更多信息请参见 阅读符号 。 当一个模块被加载时,该参数被设置为 #t (见 current-load) 。

procedure

(call-with-default-reading-parameterization thunk) → any

  thunk : (-> any)
在 parameterize 的 tail position 调用 thunk , 将上面所有的读取器 parameters 设置为默认值。

使用默认的参数值可以确保一致性,而且当从不被信任的来源读取时,它也提供了安全性, 因为默认值禁止通过 #lang 或 #reader 评估任意代码。

parameter

(read-on-demand-source)

 → (or/c #f #t (and/c path? complete-path?))
(read-on-demand-source mode) → void?
  mode : (or/c #f #t (and/c path? complete-path?))
一个 parameter ,它使编译代码的懒惰解析成为可能,这样, 闭合体和语法对象就会按要求从打包的编译代码中提取(并进行验证)。 通常情况下,当 load-on-demand-enabled 为 #t 时, 这个参数由默认的 load handler 设置。

read-on-demand-source 的 #f 值禁用了对编译代码的懒惰解析。 一个 #t 的值可以启用懒惰解析。 当 PLT_DELAY_FROM_ZO 环境变量在启动时被设置为(任何值)时, 一个 path 值更能使懒惰地从磁盘检索,而不是将未编译的代码保留在内存中。

如果在启用从磁盘进行延迟检索时,在解析延迟代码之前,处于 mode 作为 path 的文件发生了更改, 则按需解析很可能会遇到垃圾,从而导致异常。

procedure

(port-read-handler in) → 
(case->
 (input-port? . -> . any)
 (input-port?  any/c . -> . any))
  in : input-port?
(port-read-handler in proc) → void?
  in : input-port?
  proc : 
(case->
 (input-port? . -> . any)
 (input-port? any/c . -> . any))
获取或设置 in 的 port read handler 。 当内置的 read 或 read-syntax 过程被应用到端口时, 被调用的处理程序将从端口读取数据。 (端口读取处理程序不用于 read/recursive 或 read-syntax/recursive 。)

一个端口读处理程序可以应用于一个参数或两个参数:

  • 当端口与 read 一起使用时,会提供一个参数;该参数是被读取的端口。 返回值是从端口读出的值(或文件的末端)。

  • 当端口与 read-syntax 一起使用时,需要提供两个参数; 第一个参数是被读取的端口,第二个参数是一个表示源的值。 返回值是一个从端口读取的语法对象(或文件结束)。

默认的端口读取处理程序用 Racket 的内置解析器读取标准的 Racket 表达式(见 阅读器)。 它处理来自自定义输入端口(见 make-input-port) 的特殊结果, 将其视为一个单一的表达式,除了特殊的注释值(见 特殊注释) 被视为空白。

默认的端口读取处理程序本身可以通过读表来定制;更多信息请参见 读取表 。
