6.2 创建类

类和对象 in Racket 指南 introduces classes and objects.

value

object% : class?

一个内置的类,没有方法字段,只实现自己的接口 (class->interface object%) , 并且是透明的(即，它的检查器是 #f ,并且是透明的(即。它的检查器是 equal?)。 所有其他的类都是从 object% 派生的。

syntax

(class* superclass-expr (interface-expr ...)
  class-clause
  ...)
 
class-clause = (inspect inspector-expr)
  | (init init-decl ...)
  | (init-field init-decl ...)
  | (field field-decl ...)
  | (inherit-field maybe-renamed ...)
  | (init-rest id)
  | (init-rest)
  | (public maybe-renamed ...)
  | (pubment maybe-renamed ...)
  | (public-final maybe-renamed ...)
  | (override maybe-renamed ...)
  | (overment maybe-renamed ...)
  | (override-final maybe-renamed ...)
  | (augment maybe-renamed ...)
  | (augride maybe-renamed ...)
  | (augment-final maybe-renamed ...)
  | (private id ...)
  | (abstract id ...)
  | (inherit maybe-renamed ...)
  | (inherit/super maybe-renamed ...)
  | (inherit/inner maybe-renamed ...)
  | (rename-super renamed ...)
  | (rename-inner renamed ...)

  | (case-lambda (formals expr ...+) ...)
  | (#%plain-lambda formals expr ...+)
  | 
(let-values ([(id) method-procedure] ...)

  | 
(letrec-values ([(id) method-procedure] ...)

  |

  id)
  | 
(chaperone-procedure method-procedure wrapper-proc
                     other-arg-expr ...)
生成一个类值。

当 class* 表达式被评估时, superclass-expr 被评估。 结果必须是一个类值(可能是 object%),否则会引发 exn:fail:object exception is raised 。 superclass-expr 表达式的结果是新类的超类。

当评估 class* 表达式时,在评估superclass-expr 之后, interface-expr 表达式也被评估。 每个 interface-expr 的结果必须是一个接口值,否则会引发 exn:fail:object exception is raised 。 由 interface-expr 返回的接口都是由类实现的。 对于每个接口中的每个标识符,该类(或其祖先之一)必须声明一个具有相同名称的公共方法, 否则将引发 exn:fail:object exception is raised 。 该类的超类必须满足每个接口的实现要求,否则会引发 exn:fail:object exception is raised 。

inspect class-clause 为类的扩展选择了一个检查器(见 结构检查器)。 当 class* 形式被评估时, inspector-expr 必须评估为一个 inspector 或 #f。 就像对结构类型一样,检查器控制对类的字段的访问,包括私有字段,同时也影响使用 equal? 进行的比较。 如果没有提供 inspect 子句,对类的访问将由当前 inspector 的父级控制(见 结构检查器)。 如果指定了一个以上的 inspect 子句,会报告一个语法错误。

其他的 class-clause 定义初始化参数、公共和私有字段以及公共和私有方法。 对于 public, override, augment, pubment, overment, augride, public-final, override-final, augment-final 或 private 子句中的每个 id 或 maybe-renamed ,必须有一个 method-definition 。 所有其他的定义 class-clause 都会创建私有字段。 所有剩下的 expr 都是初始化表达式,当类被实例化时将被评估(参见 创建对象)。

class* 表达式的结果是一个新类,派生自指定的超类并实现指定的接口。 类的实例是用 instantiate 形式或 make-object 过程创建的, 如 创建对象 中所述。

每个 class-clause 都被(部分地)宏展开以显示其形状。 如果一个 class-clause 是一个 begin 表达式, 那么它的子表达式将从 begin 中提取出来,并被视为 class-clause , 就像将 begin 用于顶级和嵌入式定义一样。 每个 class-clause 在扩展前都将 syntax property 'class-body 设置为 true。

在新类实例的 class* 表单中, this 被绑定到对象本身; this% 绑定到对象的类; super-instantiate, super-make-object 和 super-new 被绑定到表单来初始化超类中的字段(参见 创建对象); super 可用于调用超类方法(参见 方法的定义); 和 inner 可用于调用方法的子类扩充(参见 方法的定义)。

Changed in version 8.8.0.10 of package base: 增加了 'class-body 语法属性到类主体形式。

syntax

(class superclass-expr class-clause ...)

类似于 class* ,但省略了 interface-expr ,因为不需要接口表达式。

Example:
(define book-class%
  (class object%
    (field (pages 5))
    (define/public (letters)
      (* pages 500))
    (super-new)))

syntax

this

在 class* 表单中, this 指的是当前对象(即被初始化的对象或其方法被调用的对象)。 在 class* 的主体之外使用是一个语法错误。

Examples:
(define (describe obj)
  (printf "Hello ~a\n" obj))
(define table%
  (class object%
    (define/public (describe-self)
      (describe this))
    (super-new)))

 

> (send (new table%) describe-self)

Hello #(struct:object:table% ...)

syntax

this%

在 class* 表单中, this% 指的是当前对象的类(即被初始化的对象或其方法被调用的对象)。 在 class* 的主体之外使用是一个语法错误。

Examples:
(define account%
  (class object%
    (super-new)
    (init-field balance)
    (define/public (add n)
      (new this% [balance (+ n balance)]))))
(define savings%
  (class account%
    (super-new)
    (inherit-field balance)
    (define interest 0.04)
    (define/public (add-interest)
      (send this add (* interest balance)))))

 

> (let* ([acct (new savings% [balance 500])]
         [acct (send acct add 500)]
         [acct (send acct add-interest)])
    (printf "Current balance: ~a\n" (get-field balance acct)))

Current balance: 1040.0

syntax

(class/derived original-datum
  (name-id super-expr (interface-expr ...) deserialize-id-expr)
  class-clause
  ...)
类似于 class*, 但包含一个子表达式,用作类定义中所有语法错误的源。 例如,define-serializable-class 扩展为 class/derived , 这样类主体中的错误就会根据 define-serializable-class 而不是 class 来报告。

original-datum 是用于报告错误的原始表达式。

name-id 用于命名产生的类;如果它是 #f, 则推断出类的名字。

super-expr, interface-expr 和 class-clause 与 class* 一样。

如果 deserialize-id-expr 不是 #f, 不是 make-deserialize-info 产生的 deserialize-info 结构。 deserialize-id-expr 应该产生一个适合作为 make-serialize-info 的第二个参数的值, 它应该指向一个出口,其值是 deserialize-info 结构。

未来的可选形式可以被添加到目前以 deserialize-id-expr 结束的序列中。

6.2.1 初始化变量

一个类的初始化变量,用 init, init-field 和 init-rest 声明,为一个类的每个对象实例化。 初始化变量可以被用于字段的初始值表达式、初始化参数的缺省值表达式以及初始化表达式中。 只有用 init-field 声明的初始化变量可以从方法中访问; 从方法中访问任何其他初始化变量都是一个语法错误。

绑定到初始化变量的值是

  • instantiate 提供的参数或传递给 make-object 的参数, 如果对象是作为类的直接实例创建的;或者,

  • 如果对象是作为派生类的实例创建的,那么传递给超类初始化形式或过程的参数。

如果一个初始化变量没有提供初始化参数,而该变量有一个相关的 default-value-expr, 那么 default-value-expr 将被评估以获得该变量的值。 只有当没有为其变量提供参数时, default-value-expr 才被评估。 default-value-expr 的环境包括所有的初始化变量、所有的字段和类的所有方法。 如果多个 default-value-expr 被评估,它们将从左到右被评估。 对象的创建和字段的初始化将在 创建对象 中详细描述。

如果一个初始化变量没有 default-value-expr, 那么对象创建或超类初始化调用必须为该变量提供一个参数,否则会引发 exn:fail:object exception is raised 。

初始化参数可以通过名称或位置来提供。 初始化变量的外部名称可以与 instantiate 或超类初始化形式一起使用。 这些形式也接受按位置的参数。 make-object 过程和超类初始化过程只接受 by-position 参数。

通过位置提供的参数将使用 init 和 init-field 子句的顺序以及每个子句中变量的顺序被转换为副名参数。 当一个 instantiate 的形式同时提供按位置和按名称的参数时,转换后的参数被放在按名称的参数之前。 (顺序可能很重要,请参见 创建对象 。)

除非一个类包含一个 init-rest 子句,否则当副位置参数的数量超过声明的初始化变量的数量时, 超类中变量的顺序(以此类推,沿着超类链向上)决定副名的转换。

如果一个类表达式包含一个 init-rest 子句,必须只有一个,而且必须是最后一个。 如果它声明了一个变量,那么这个变量会以列表的形式接收额外的 by-position 初始化参数(类似于过程中带点的 "休息参数")。 init-rest 变量可以接收派生类的旁位初始化参数,这些参数是在派生类的旁位转换中留下的。 当一个派生类的超类初始化提供了更多的旁置参数时,它们会被前置到迄今为止积累的旁置参数上。

如果向对象创建或超类初始化提供的副位置初始化参数太少或太多,那么会引发 exn:fail:object exception is raised 。 同样地,如果向一个有 init-rest 子句的类提供了额外的副位置参数, 那么 exn:fail:object exception is raised 异常就会被引发。

未使用的(按名称)参数将被传播到超类,如 创建对象 中所述。 如果类的派生包含了多个声明(在不同的类中)的初始化变量的名称,那么多个初始化参数可以使用相同的名称。 更多细节请参见 创建对象 。

关于内部和外部名称的信息,也请参见 内部和外部名称 。

6.2.2 字段

类中的每个 field, init-field 和非方法 define-values 子句都为该类声明一个或多个新字段。 用 field 或 init-field 声明的字段是公共的。 公有字段可以被子类使用 inherit-field 访问和修改。 公有字段也可以在类外通过 class-field-accessor 进行访问, 并且可以通过 class-field-mutator 进行变异(参见 字段和方法访问)。 用 define-values 声明的字段只能在类内访问。

用 init-field 声明的字段既是一个公共字段也是一个初始化变量。 关于初始化变量的信息请参见 初始化变量 。

inherit-field 声明使一个由超类定义的公共字段可以在类表达式中直接访问。 如果指定的字段没有在超类中定义,那么当类表达式被评估时就会引发 exn:fail:object exception is raised 。 超类中的每个字段都存在于派生类中,即使它在派生类中没有用 inherit-field 声明。 inherit-field 子句并不控制继承,而只是控制类表达式中的词法范围。

当一个对象第一次被创建时,它的所有字段都有 #<undefined> 值(见 Void)。 一个类的字段在评估该类的初始化表达式的同时被初始化;更多信息请参见 创建对象 。

关于内部和外部名称的信息,请参见 内部和外部名称 。

6.2.3 方法
6.2.3.1 方法的定义

类中的每个 public, override, augment, pubment, overment, augride, public-final, override-final, augment-final 和 private 子句都声明了一个或多个方法名。 每个方法名必须有一个相应的 method-definition 。 public 等子句及其相应定义的顺序(在它们之间,以及相对于类中其他子句)并不重要。

如 class* 的语法所示,方法定义在语法上被限制为某些程序形式, 如 method-procedure 的语法所定义;在 method-procedure 的后两种形式中, 主体 id 必须是 let-values 或 letrec-values 所约束的 id 之一。 method-procedure 表达式不被直接评估。 相反,对于每个方法,一个特定于类的方法过程被创建;它需要一个初始的对象参数, 以及如果 method-procedure 表达式被直接评估,该过程将接受的参数之外的参数。 过程的主体被转换为通过对象参数访问方法和字段。

用 public, pubment 或 public-final 声明的方法在一个类中引入了一个新的方法。 这个方法必须不存在于超类中,否则当类的表达式被评估时就会引发 exn:fail:object exception is raised 异常。 一个用 public 声明的方法可以在一个使用 override, overment 或 override-final 的子类中被覆盖。 一个用 pubment 声明的方法可以在一个使用 augment, augride 或 augment-final 的子类中被增强。 一个用 public-final 声明的方法不能在子类中被重载或增强。

用 override, overment 或 override-final 声明的方法会覆盖已经存在于超类中的定义。 如果该方法不存在,当类表达式被评估时,将引发 exn:fail:object exception is raised 异常。 用 override 声明的方法可以在使用 override, overment 或 override-final 的子类中再次被覆盖。 一个用 overment 声明的方法可以在一个使用 augment, augride 或 augment-final 的子类中被增强。 一个用 override-final 声明的方法不能在子类中被进一步重载或增强。

用 augment, augride 或 augment-final 声明的方法是对已经存在于超类中的定义进行增强。 如果该方法不存在,当类表达式被评估时,将引发 exn:fail:object exception is raised 。 用 augment 声明的方法可以在使用 augment, augride 或 augment-final 的子类中被进一步增强。 一个用 augride 声明的方法可以在一个使用 override, overment 或 override-final 的子类中被覆盖。 (这样的覆盖只是替换了增强,而不是替换了被增强的方法)。 一个用 augment-final 声明的方法不能在子类中被重写或进一步增强。

用 private 声明的方法在类表达式之外是不可访问的,不能被重载,也不能重载超类中的方法。

当一个方法用 override, overment 或 override-final 声明时, 那么该方法的超类实现可以用 super 形式调用。

当一个方法用 pubment, augment 或 overment 声明时,那么子类的增强方法就可以用 inner 形式调用。 public-final 和没有相应的 inner 的 pubment 之间的唯一区别是, public-final 阻止了会被忽略的增强方法的声明。

用 abstract 声明的方法必须是没有实现的。 子类可以通过 override, overment 或 override-final 形式实现抽象方法。 任何包含或继承任何抽象方法的类都被认为是抽象的,不能被实例化。

syntax

(super id arg ...)

(super id arg ... . arg-list-expr)
总是访问超类的方法,与该方法是否在子类中被再次重载无关。 在 class* 之外使用 super 形式是一个语法错误。 每个 arg 和 #%app 一样: 要么是 arg-expr 要么是关键字 keyword arg-expr 。

第二种形式类似于在过程中使用 apply ; arg-list-expr 不能是一个括号内的表达式。

syntax

(inner default-expr id arg ...)

(inner default-expr id arg ... . arg-list-expr)
如果对象的类没有提供增强方法,那么 default-expr 会被评估,而 arg 表达式不会被评估。 否则,就用 arg 结果作为参数来调用增强方法,而 default-expr 不被评估。 如果对于一个特定的方法没有评估 inner 调用,那么由子类提供的增强方法就不会被使用。 在 class* 之外使用 inner 形式是一个语法错误。

第二种形式类似于在过程中使用 apply ; arg-list-expr 不能是一个括号内的表达式。

6.2.3.2 继承和超类方法

每个 inherit, inherit/super, inherit/inner, rename-super 和 rename-inner 子句都声明了一个或多个在类中定义的方法,但在超类中必须存在。 rename-super 和 rename-inner 声明很少使用,因为 inherit/super 和 inherit/inner 提供了相同的访问。 另外,超类和增量方法通常是通过同样声明了这些方法的类中的 super 和 inner 来访问,而不是通过 inherit/super, inherit/inner, rename-super 或 rename-inner 。

用 inherit, inherit/super 或 inherit/inner 声明的方法名在运行时访问覆盖声明(如果有的话)。 用 inherit/super 声明的方法名也可以用 super 形式来访问超类的实现, 用 inherit/inner 声明的方法名也可以用 inner 形式来访问一个增强的方法(如果有的话)。

用 rename-super 声明的方法名在运行时总是访问超类的实现。 用 rename-inner 声明的方法访问子类的增强方法(如果有的话),并且必须以如下形式调用

(id (lambda () default-expr) arg ...)

因此,当没有可用的增加方法时,可以使用 default-expr 来计算。 在这种形式中, lambda 是一个文字标识符,用于分隔 default-expr 和 arg 。 当扩展方法可用时,它将 arg 表达式的结果作为参数接收。

存在于超类中但没有用 inherit, inherit/super 或 inherit/inner 或 rename-super 声明的方法在该类中不能直接访问(尽管它们可以用 send 调用)。 超类中的每个公有方法都存在于派生类中,即使它在派生类中没有用 inherit 声明; inherit 子句并不控制继承,而只是控制类表达式中的词法范围。

如果一个用 inherit, inherit/super, inherit/inner, rename-super 或 rename-inner 声明的方法在超类中不存在,当类表达式被评估时就会引发 exn:fail:object exception is raised 。

6.2.3.3 内部和外部名称

当 (internal-id external-id) 用于声明方法时,每个用 public, override, augment, pubment, overment, augride, public-final, override-final, augment-final, inherit, inherit/super, inherit/inner, rename-super 和 rename-inner 声明的方法可以有单独的内部和外部名称。 内部名称用于在类表达式中直接访问该方法(包括在 super 或 inner 形式中),而外部名称则用于 send 和 generic (参见 字段和方法访问)。 如果为一个方法声明提供了一个单一的 id ,那么这个标识符将同时用于内部和外部名称。

方法的继承、覆盖和增强都只基于外部名称。 rename-super 和 rename-inner 需要单独的内部和外部名称(主要出于历史原因)。

每个 init, init-field, field 或 inherit-field 变量同样有一个内部和一个外部名称。 内部名称在类内用于访问该变量,而外部名称在类外用于提供初始化参数(例如, instantiate)、 继承字段或外部访问字段(例如,使用 class-field-accessor)。 至于方法,当用 inherit-field 继承一个字段时,外部名称与超类中的外部字段名相匹配, 而内部名称在 class 表达式中被绑定。

一个标识符可以被用作内部标识符和外部标识符,而且有可能将同一个标识符用作不同绑定的内部和外部标识符。 此外,在一个单一的类中,一个单一的名字可以被用作外部方法名、外部字段名和外部初始化参数名。 总的来说,每个内部标识符必须与所有其他内部标识符不同,每个外部方法名必须与所有其他方法名不同, 每个外部字段名必须与所有其他字段名不同,而每个初始化参数名必须与所有其他初始化参数名不同。

默认情况下,外部名称没有词法范围,这意味着, 例如,一个外部方法名称在 send 的所有使用中匹配相同的语法符号。 define-local-member-name 和 define-member-name 形式引入了范围性的外部名称。

当 class 表达式被编译时,用于替代外部名称的标识符必须是符号上的不同(当相应的外部名称被要求是不同的), 否则会报告语法错误。 当没有外部名称被 define-member-name 绑定时, 那么当 class 表达式被评估时,实际的外部名称被保证是不同的。 当任何外部名称被 define-member-name 绑定时,如果实际的外部名称不是不同的, class 会引发 exn:fail:object exception is raised 。

syntax

(define-local-member-name id ...)

除非它作为顶层定义出现,否则将每个 id 绑定,以便在定义的范围内, 每个 id 作为外部名称的使用都被解析为由 define-local-member-name 声明生成的隐藏名称。 因此,用这种外部名称 id 声明的方法、字段和初始化参数只能在 define-local-member-name 声明的范围内访问。 作为一个顶层定义, define-local-member-name 将 id 与它的符号形式结合起来。

由 define-local-member-name 引入的绑定是一种语法绑定,可以通过 module 导出和导入。 每次对 define-local-member-name 声明的评估都会产生一个独特的隐藏名称(除了作为顶层定义)。 interface->method-names 程序不会暴露隐藏的名字。

Examples:
(define-values (r o)
  (let ()
    (define-local-member-name m)
    (define c% (class object%
                 (define/public (m) 10)
                 (super-new)))
    (define o (new c%))
 
    (values (send o m)
            o)))

 

> r

10

> (send o m)

send: no such method

  method name: m

  class name: c%

syntax

(define-member-name id key-expr)

将一个单一的外部名称映射到一个由表达式决定的外部名称。 key-expr 的值必须是 member-name-key 表达式或 generate-member-key 调用的结果。

syntax

(member-name-key identifier)

在 member-name-key 表达式的环境中为 id 产生一个外部名称的表示。

procedure

(generate-member-key) → member-name-key?

产生一个隐藏的名字,就像 define-local-member-name 的绑定。

procedure

(member-name-key? v) → boolean?

  v : any/c
对于由 member-name-key 和 generate-member-key 产生的值返回 #t , 否则返回 #f 。

procedure

(member-name-key=? a-key b-key) → boolean?

  a-key : member-name-key?
  b-key : member-name-key?
如果成员名键 a-key 和 b-key 代表相同的外部名称, 产生 #t ,否则产生 #f 。

procedure

(member-name-key-hash-code a-key) → integer?

  a-key : member-name-key?
产生一个与 member-name-key=? 比较一致的整数哈希代码,类似于 equal-hash-code 。

Examples:
(define (make-c% key)
  (define-member-name m key)
  (class object%
    (define/public (m) 10)
    (super-new)))

 

> (send (new (make-c% (member-name-key m))) m)

10

> (send (new (make-c% (member-name-key p))) m)

send: no such method

  method name: m

  class name: make-c%

> (send (new (make-c% (member-name-key p))) p)

10

 

(define (fresh-c%)
  (let ([key (generate-member-key)])
    (values (make-c% key) key)))
 
(define-values (fc% key) (fresh-c%))

 

> (send (new fc%) m)

send: no such method

  method name: m

  class name: make-c%

> (let ()
    (define-member-name p key)
    (send (new fc%) p))

10
