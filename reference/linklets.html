14.14 小链接和核心编译器

 (require racket/linklet) package: base

linklet 是编译、字节码编排和评估的一个原始元素。 Racket 的模块、宏和顶层评估的实现都是建立在 linklet 之上。 Racket 程序员一般不会直接接触到 linklet,但 racket/linklet 库提供了对 linklet 设施的访问。

一个单一的 Racket 模块(或顶层形式的集合)通常由多个 linklet 实现。 例如,一个模块中存在的每个评估阶段都由一个单独的 linklet 来实现。 一个 linklet 也被用于元数据,如一个模块的 require 的 module path index 。 这些小链接,加上其他一些元数据,被组合成一个 linklet bundle 。 这些小链接,加上其他一些元数据, 被组合成一个 linklet bundle 中的信息是由一个符号或一个 fixnum 键入的。 包含小 linklet 的 linklet bundle 可以通过 write 和 (启用 read-accept-compiled ) read 的方式被编入字节流,也可以从字节流中编入。

当一个 Racket 模块有子模块时,该模块和子模块的 linklet bundles 被分组到一个 linklet directory 中。 一个 linklet directory 目录可以有嵌套的linklet目录。 linklet目录中的信息以 #f 或符号为键, 其中 #f 必须被映射到一个 linklet bundle (如果有的话), 每个符号必须被映射到一个 linklet directory 。 一个 linklet directory 可以等效地被看作是一个从符号列表到 linklet bundle 的映射。 像 linklet bundles 一样, linklet directory 可以通过 write 和 read 方式,从一个字节流中调入和调出; 调入的形式允许单独的 linklet bundles 被加载。

一个 linklet 由一组变量定义和表达式、定义的变量名称的导出子集、一组尽管没有相应定义却要从 linklet 中导出的变量, 以及一组提供其他变量供 linklet 使用的导入。 为了运行一个 linklet,它被实例化为 linklet instance (或简称为 instance) 。 当一个 linklet 被实例化时,它接收其他 linklet instances 作为其进口, 并提取一组指定的变量,这些变量从每个给定的实例中导出。 新创建的 linklet instance 提供其导出的变量, 供其他 linklet 使用或通过 instance-variable-value 直接访问。 一个 linklet instance 可以直接用 make-instance 来合成。

一个 linklet 是通过编译其源的丰富的 S 表达式表示法而创建的。 由于 linklet 存在于宏和语法对象层之下,所以linklet的编译不使用 syntax objects 。 相反,linklet 编译使用 correlated objects , 它就像没有词法上下文信息的 syntax objects ,没有内容被强制到相关对象的约束。 使用 S 表达式或 correlated object ,由 compile-linklet 识别的 linklet 的语法是

(linklet [[imported-id/renamed ...] ...]
         [exported-id/renamed ...]
  defn-or-expr ...)
 
imported-id/renamed = imported-id
  | (external-imported-id internal-imported-id)
     
exported-id/renamed = exported-id
  | (internal-exported-id external-exported-id)

每个导入集 [imported-id/renamed ...] 指的是一个单一的导入实例, 每个 import-id/renamed 对应于该实例的一个变量。 如果单独指定了 external-imported-id 和 internal-imported-id , 那么 external-imported-id 是该实例导出的变量名称, 而 internal-imported-id 是在 defn-or-expr 中用来引用该变量的名称。 对于出口,单独的 internal-exported-id 和 external-exported-id 名称分别对应于在 defn-or-expr 中引用的出口变量名称。

defn-or-expr 的语法类似于 expander 的完全扩展表达式的语法(见 完全展开的程序),但有一些例外: quote-syntax 和 #%top 是不允许的; #%plain-lambda 被拼成 lambda; #%plain-app 被省略(即、 application是隐式的); lambda, case-lambda, let-values 和 letrec-values 只能有一个主体表达式;数字、布尔运算、字符串和字节字符串是自引号。 基元可以直接通过名称访问,在 linklet 形式中不允许对基元名称 (见 linklet-body-reserved-symbol?)、导入的变量、定义的变量或局部变量进行阴影处理。

当 exported-id/renamed 在 defn-or-expr 中没有相应的定义时, 那么该变量实际上被定义为未初始化;引用该变量将触发 exn:fail:contract:variable, 与在定义前引用一个变量一样。 当一个目标实例被提供给 instantiate-linklet 时,任何现有的同名变量将被保持原样,而不是被设置为未定义。 这种对未初始化变量的处理方式为顶层评估提供了核心支持,在顶层评估中,变量可以被引用,然后在单独的编译元素中定义。

Added in version 6.90.0.1 of package base.

接受一个 S 表达式或 correlated object 的 linklet 形式并产生一个 linklet 。 只要在 options 中包含 'serializable , 当产生的 linklet 是 linklet bundle 的一部分(可能在 linklet directory 中)时,它可以被编入字节流中。

可选的 name 与小链接相关联,用于调试目的,并作为小链接实例的默认名称。

可选的 import-keys 和 get-import 参数支持跨 linklet 优化。 如果 import-keys 是一个向量,它必须有与 form 上的进口集合一样多的元素。 如果编译器对优化对一个导入变量的引用感兴趣,它就会将 import-keys 中对应于该变量导入集的元素传回给 get-import (如果非 #f) 。 然后, get-import函数可以返回一个代表实例的 linklet 或实例,当编译的 linklet 最终被实例化时, 它将被提供给编译的 linklet; 确保报告的 linklet 或实例和最终的实例之间的一致性是由 compile-linklet 的调用者决定。 如果 get-import 返回 #f 作为它的第一个值,编译器将被阻止对导入的实例做任何假设。 get-import 的第二个结果是一个可选的键的向量, 以提供关于返回的 linklet 的导入的横向信息(对于返回的实例来说是不允许的); 返回的向量必须有与 linklet 的导入相同数量的元素。 当向量元素是 eq? 和非 #f 时,编译器可以认为它们对应于同一个运行时实例。 get-import 的 #f 值等同于一个总是返回两个 #f 结果的函数。

当 import-keys 不是 #f 时,则允许编译器为 linklet 增加或缩减导入的实例集。 结果向量指定了返回的小链接的进口键。 然而,任何 #f 或 linklet instance 的键都必须被完整地保留下来。

如果 options 中包含 'unsafe ,那么小链接将以 unsafe mode , 那么小链接将以不安全模式编译:小链接中安全操作的使用可以转换为不安全操作,前提是相关的契约得到满足。 例如, car 被转换为 unsafe-car 。 一些被替换的不安全操作可能没有直接可访问的名称, 比如在 unsafe mode 下可以替换的 in-list 的不安全变体。 只有当一个不安全操作的(未检查的)契约被安全操作的契约所覆盖时,才会被替换。 linklet 以 unsafe mode 编译的事实可以通过 variable-reference-from-unsafe? 暴露出来, 该变量引用由模块主体中的 #%variable-reference 形式产生。

如果 'static 被包含在 options 中,那么 linklet 必须只被实例化一次; 如果 linklet 被序列化,那么从序列化形式中读取的任何单个实例必须最多实例化一次。 用 'static 编译的目的是为了提高 linklet 中对定义和导入的变量的引用的性能。

如果 'quick 包含在 options 中, 那么 linklet 的编译可能会用运行时的性能来换取编译时的性能–也就是说, 花更少的时间来编译 linklet,但得到的 linklet 可能运行得更慢。

如果 'use-prompt 包含在 options 中, 那么实例化产生的 linklet 总是在 linklet 中的每个定义和即时表达式周围包裹一个提示。 否则,提供 #t 作为 instantiate-linklet 的 use-prompt? 参数, 可能只会在整个实例化过程中包裹一个提示。

如果 'uninterned-literal 被包含在 options 中,那么在编译或加载 linklet 时, form 中的字词不一定会通过 datum-intern-literal 被内部化。 禁用 datum-intern-literal 对于 linklet 包括一个不打算共享的大字符串或字节串常量来说特别有用。

options 中的符号必须是不同的,否则会引发 exn:fail:contract exception is raised 。

procedure

(eval-linklet linklet) → linklet?

  linklet : linklet?
返回一个为 JIT 编译准备的 linklet 的变体, 这样以后使用 instantiate-linklet 的结果 linklet 都会共享 JIT 生成的代码。 然而, eval-linklet 的结果不能作为 linklet bundle 的一部分被编入字节流, 也不能与 recompile-linklet 一起使用。

procedure

(instantiate-linklet linklet    
  import-instances    
  [target-instance?    
  use-prompt?]) → instance?
  linklet : linklet?
  import-instances : (listof instance?)
  target-instance? : #f = #f

(instantiate-linklet linklet    
  import-instances    
  target-instance    
  [use-prompt?]) → any
  linklet : linklet?
  import-instances : (listof instance?)
  target-instance : instance?
  use-prompt? : any/c = #t
通过运行其定义和表达式来实例化 linklet ,使用给定的 import-instances 作为其导入。 import-instances 中的实例数量必须与 linklet 中的导入集数量一致。

如果 target-instance 是 #f或者没有提供,结果是 linklet 的一个新实例。 如果 target-instance 是一个实例,那么该实例被用于 linklet 的定义和表达式, 并被修改,结果是 linklet 中最后一个表达式的值。

linklet导出的变量可以在结果实例中或在 target-instance 中使用 linklet 的外部名称来访问每个导出的变量。 如果 target-instance 被提供为非 #f ,其现有的变量如果没有被 linklet 定义修改,则保持不变。

如果 use-prompt? 为真,那么一个 prompt 将被包裹在 linklet 实例中,方法与模块主体中的表达式相同。 如果 linklet 包含多个定义或直接表达式,那么提示符可能被包裹在每个定义或表达式中,也可能不被包裹; 向 compile-linklet 提供 'use-prompt 以确保提示符被用于每个定义和表达式中。

procedure

(instance-set-variable-value! instance    
  name    
  v    
  [mode]) → void?
  instance : instance?
  name : symbol?
  v : any/c
  mode : (or/c #f 'constant 'consistent) = #f
设置或创建 instance 中作为 name 导出的变量,这样它的值就是 v , 只要该变量还没有作为常量存在。 如果 name 的变量作为常量存在,会引发 exn:fail:contract exception is raised 。

如果 mode 是 'constant 或 'consistent ,那么变量将被创建或改变为常量。 此外,当实例被报告给 linklet 的导入时,通过对 compile-linklet 的 get-import 回调, 编译器可以假设该变量在未来所有用于满足 linklet 导入的实例中是常量。

如果 mode 是 'consistent, 当实例通过回调报告给 compile-linklet 时, 编译器可以进一步假设该变量的值在未来的实例中是相同的。 出于编译的目的,"相同 "可以指一个过程的值将有相同的算术和实现细节, 一个 structure type 的值将有相同的配置,一个 marshalable 常量将与当前值 equal? ,等等。

  instance : instance?
  name : symbol?
改变 instance ,使其不将变量导出为 name, 只要 name 不作为常量变量存在。 如果 name 的变量作为常量存在,会引发 exn:fail:contract exception is raised 。

procedure

(instance-describe-variable! instance    
  name    
  desc-v) → void?
  instance : instance?
  name : symbol?
  desc-v : any/c
在 instance 中注册关于 name 的信息,这对于编译 linklet 可能很有用, 因为实例是通过 get-import 回调到 compile-linklet 的。 desc-v 描述可以是任何值;公认的描述取决于虚拟机,但可能包括以下内容:

  • `(procedure ,arity-mask) — 该值总是一个没有被冒名的过程,而不是一个结构, 它的arity在 procedure-arity-mask 的风格中是 arity-mask 。

  • `(procedure/succeeds ,arity-mask) — 的风格中是 `(procedure ,arity-mask) 一样,但是对于一个从未引发异常或捕获或逃避调用环境的过程。

  • `(procedure/pure ,arity-mask) — 类似于 `(procedure/succeeds ,arity-mask), 但是没有可观察的副作用,所以对过程的调用可以被重新排序。

  ref-site? : any/c = #f
如果 ref-site? 是 #f,则提取 varref 变量所在的实例, 如果 ref-site? 是true,则返回 varref 本身所在的实例。 这个 variable reference 的概念与模块级的概念相同,可以反映出实现模块实例的特定阶段的 linklet 实例。

当 ref-site? 是 #f 时, 当 varref 来自 (#%variable-reference) 没有标识符时,结果是 #f 。 如果 varref 指的是一个基元,则结果是一个符号。

与 syntax?, syntax-source, syntax-line, syntax-column, syntax-position, syntax-span, syntax-e, syntax->datum, datum->syntax, syntax-property 和 syntax-property-symbol-keys 类似,但用于 correlated objects 。

与 datum->syntax 不同, datum->correlated 并不通过给定的 S 表达式进行递归并将片段转换为 correlated objects 。 相反,一个 correlated object 被简单地包裹在即时值上。 与此相反, correlated->datum 通过它的参数(不一定是 correlated object) 来发现任何 correlated objects 并将它们转换为普通的S表达式。
