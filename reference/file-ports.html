13.1.5 文件端口

由 open-input-file, open-output-file, subprocess 和相关函数创建的端口是一个 file-stream port 。 racket 中的初始输入、输出和错误端口也是文件流端口。 file-stream-port? 谓词可以识别文件流端口。

当一个输入或输出的文件流端口被创建时,它将被置于当前监护人的管理之下(见 Custodians)。 在输出端口的情况下,一个 flush callback 被注册到 current plumber 处,以冲洗该端口。

procedure

(open-input-file path    
  [#:mode mode-flag    
  #:for-module? for-module?]) → input-port?
  path : path-string?
  mode-flag : (or/c 'binary 'text) = 'binary
  for-module? : any/c = #f
打开由 path 指定的文件进行输入。 mode-flag 参数指定了文件的字节在输入时的翻译方式:

  • 'binary — 字节从端口返回,与从文件中读出的字节完全一致。

  • 'text — 从文件中读出的回车和换行字节 (10 和 13) 会被 port 以特定平台的方式过滤掉:

    • Unix and Mac OS: 不进行过滤。

    • Windows: 文件中的回车-换行组合被 port 作为单个换行返回; 对于后面没有换行的回车字节, 或前面没有回车的换行字节, 则不进行过滤。

在 Windows 上, 'text 模式只适用于普通文件; 试图在其他类型的文件中使用 'text 会触发 exn:fail:filesystem 。

否则,由 path 指定的文件不一定是普通文件。 它可能是一个通过文件系统连接的设备,比如 Windows 上的 "aux" 或 Unix 上的 "/dev/null" 。 在所有情况下,端口都是默认的缓冲区。

open-input-file 产生的端口应该被明确地关闭, 可以通过 close-input-port 或者通过 custodian-shutdown-all 间接关闭, 以释放操作系统级的文件柄。 如果输入端口可用于垃圾收集,它将不会被自动关闭(见 垃圾回收); 一个 will 可以与输入端口相关联,以便更自动地关闭它(见 遗嘱和执行者)。

一个 path 值是 path 的 cleanse 版本,被用作打开端口的名称。

如果由于文件系统的错误导致打开文件失败, 那么 exn:fail:filesystem:errno exception is raised—将被引发–只要 for-module? 是 #f, current-module-path-for-load 有一个非 #f 值, 或者文件系统错误没有被识别为 file-not-found 错误。 否则,当 for-module? 为真, current-module-path-for-load 有一个非 #f 值, 并且文件系统错误被识别为 file-not-found 错误,那么引发的异常是 exn:fail:syntax:missing-module (如果 current-module-path-for-load 的值是一个 syntax object) 或 exn:fail:filesystem:missing-module (否则)。

Changed in version 6.0.1.6 of package base: 增加 #:for-module? 。

Examples:
> (with-output-to-file some-file
    (lambda () (printf "hello world")))
> (define in (open-input-file some-file))
> (read-string 11 in)

"hello world"

> (close-input-port in)

procedure

(open-output-file path 
  [#:mode mode-flag 
  #:exists exists-flag 
  #:permissions permissions] 
  #:replace-permissions? replace-permissions?) 
 → output-port?
  path : path-string?
  mode-flag : (or/c 'binary 'text) = 'binary
  exists-flag : 
(or/c 'error 'append 'update 'can-update
      'replace 'truncate
      'must-truncate 'truncate/replace)
   = 'error
  permissions : (integer-in 0 65535) = #o666
  replace-permissions? : #f
打开由 path 指定的文件进行输出。 mode-flag 参数指定了写到端口的字节在写到文件时的转换方式::

  • 'binary — 字节被写到文件中,与写到端口中的完全一样。

  • 'text — 在 Windows 上, 写到端口的换行字节 (10) 会被翻译成文件中的回车-换行组合;对回车不进行过滤。

在 Windows 上, 'text 模式只适用于普通文件; 试图对其他类型的文件使用 'text 会触发 exn:fail:filesystem 。

exists-flag 参数指定如何处理/要求已经存在的文件:

  • 'error — 如果文件存在,引发 exn:fail:filesystem 。

  • 'replace — 删除旧文件,如果它存在,并写入一个新文件。

  • 'truncate — 如果文件存在,删除所有旧数据。

  • 'must-truncate — 删除现有文件中的所有旧数据; 如果文件不存在,会引发 exn:fail:filesystem exception is raised 。

  • 'truncate/replace — 尝试 'truncate; 如果它失败了(也许是由于文件的权限),尝试 'replace 。

  • 'update — 打开一个现有的文件而不截断它; 如果文件不存在,就会引发 exn:fail:filesystem exception is raised 。 使用 file-position 来改变当前的读/写位置。

  • 'can-update — 打开一个现有的文件而不截断它,如果该文件不存在,则创建该文件。

  • 'append — 附加到文件的末尾,不管它是否已经存在; 在 Windows 上, 'append 等同于 'update, 只是不要求文件存在,打开文件后,文件位置立即被设置为文件的末端。

当由 path 指定的文件被创建时,permissions 指定了所创建文件的权限, 其中权限的整数表示法与 file-or-directory-permissions 的处理方法相同。 在 Unix 和 Mac OS 上,这些权限位与进程的 umask 相结合。 在 Windows 上, permissions 的唯一相关属性是它是否有 #o2位的写入权限设置。 注意,一个只读的文件可以用 open-output-file 创建,在这种情况下,只有在以后试图打开该文件时才禁止写入。 如果 replace-permissions? 是一个真值,那么与被打开的文件是否是新创建的无关, permissions 的值会应用于被打开的文件,而且在 Unix 和 Mac OS 上它的应用与进程的 umask 无关。

由 path 指定的文件不一定是一个普通的文件。 它可能是一个通过文件系统连接的设备,例如 Windows 上的 "aux" 或Unix上的 "/dev/null" 。 默认情况下,输出端口是块缓冲的,除非该文件对应的是一个终端,在这种情况下,默认是行缓冲的。 在 Unix 和 Mac OS 上,如果文件是一个 fifo,那么端口将阻断写入,直到 fifo 的读取器可用; 参见 port-waiting-peer? 。

open-output-file 产生的端口应该被明确地关闭, 可以通过 close-output-port 或者通过 custodian-shutdown-all间接地关闭, 以释放操作系统级的文件柄。 如果输出端口可用于垃圾收集,它将不会被自动关闭(见 垃圾回收); 可以将 will 与输出端口相关联,以便更自动地关闭它(见 遗嘱和执行者)。

一个 path 值是 path 的 cleanse 版本,被用作打开的端口的名称。

如果由于底层文件系统的错误导致打开文件失败,则会引发 exn:fail:filesystem:errno exception is raised 。

Examples:
> (define out (open-output-file some-file))
> (write "hello world" out)
> (close-output-port out)

Changed in version 6.9.0.6 of package base: 在 Unix 和 Mac OS 上,让 'truncate/replace 在权限错误时进行替换。 在 Windows 上,让 'replace 总是替换,而不是像 'truncate/replace 那样截断。
Changed in version 7.4.0.5: 在 Unix 和 Mac OS 上改变了对 fifo 的处理,使端口阻断输出,直到 fifo 有一个读者。
Changed in version 8.1.0.3: 增加了 #:permissions 参数。
Changed in version 8.7.0.10: 增加了 #:replace-permissions? 参数。

procedure

(open-input-output-file 
  path 
  [#:mode mode-flag 
  #:exists exists-flag 
  #:permissions permissions] 
  #:replace-permissions? replace-permissions?) 
 → 
input-port? output-port?
  path : path-string?
  mode-flag : (or/c 'binary 'text) = 'binary
  exists-flag : 
(or/c 'error 'append 'update 'can-update
      'replace 'truncate
      'must-truncate 'truncate/replace)
   = 'error
  permissions : (integer-in 0 65535) = #o666
  replace-permissions? : #f
像 open-output-file 一样,但产生两个值:一个输入端口和一个输出端口。 这两个端口是相连的,因为它们共享基础文件描述符。 这个过程是为了用于只能由一个进程打开的特殊设备,如 Windows 中的 "COM1" 。 对于普通文件,共享文件描述符可能会引起混淆。 例如,使用一个端口不会自动刷新另一个端口的缓冲区,而且在一个端口的读写会移动另一个端口的文件位置(如果有的话)。 对于普通文件,使用单独的 open-input-file 和 open-output-file 调用以避免混淆。

Changed in version 8.1.0.3 of package base: 增加了 #:permissions 参数。
Changed in version 8.7.0.10: 增加了 #:replace-permissions? 参数。

procedure

(call-with-input-file path    
  proc    
  [#:mode mode-flag]) → any
  path : path-string?
  proc : (input-port? . -> . any)
  mode-flag : (or/c 'binary 'text) = 'binary
用 path 和 mode-flag 参数调用 open-input-file ,并将结果端口传给 proc 。 proc 的结果是调用 call-with-input-file 的结果,但 proc 返回时新打开的端口被关闭。

Examples:
> (with-output-to-file some-file
    (lambda () (printf "text in a file")))
> (call-with-input-file some-file
    (lambda (in) (read-string 14 in)))

"text in a file"

procedure

(call-with-output-file 
  path 
  proc 
  [#:mode mode-flag 
  #:exists exists-flag 
  #:permissions permissions] 
  #:replace-permissions? replace-permissions?) 
 → any
  path : path-string?
  proc : (output-port? . -> . any)
  mode-flag : (or/c 'binary 'text) = 'binary
  exists-flag : 
(or/c 'error 'append 'update 'can-update
      'replace 'truncate
      'must-truncate 'truncate/replace)
   = 'error
  permissions : (integer-in 0 65535) = #o666
  replace-permissions? : #f
类似于 call-with-input-file, 但将 path, mode-flag, exists-flag 和 permissions 传递给 open-output-file 。

Examples:
> (call-with-output-file some-file
    (lambda (out)
      (write 'hello out)))
> (call-with-input-file some-file
    (lambda (in)
      (read-string 5 in)))

"hello"

Changed in version 8.7.0.10: 增加了 #:replace-permissions? 参数。

procedure

(port-try-file-lock? port mode) → boolean?

  port : file-stream-port?
  mode : (or/c 'shared 'exclusive)
试图使用当前平台的文件锁设施来获取文件的锁。 多个进程可以获得一个文件的 'shared , 但最多只有一个进程可以持有一个 'exclusive 锁, 而且 'shared 和 'exclusive 锁是相互排斥的。 当 mode 为 'shared 时,那么 port 必须是一个输入端口; 当 mode 为 'exclusive 时,那么 port 必须是一个输出端口。

如果请求的锁被获取,其结果是 #t ,否则是 #f 。 当一个锁被获取时,它将被保持到用 port-file-unlock 释放或者端口被关闭(也许是因为进程终止)。

根据不同的平台,锁可能仅仅是建议性的(即,锁只影响进程获得锁的能力),或者它们可能对应于强制锁,防止对被锁文件的读写。 具体来说,锁在 Windows 上是强制性的,在其他平台上是咨询性的。 在一个端口上多次尝试 'shared 锁可以成功;在 Unix 和 Mac OS 上,一次 port-file-unlock 就可以释放锁, 而在其他 Windows 上,每次成功的 port-file-unlock 都需要一次 port-try-file-lock? 。 在 Unix 和 Mac OS 上,多次尝试 'exclusive 可以成功, 一次 port-file-unlock 就可以释放锁, 而在 Windows 上,如果某个端口已经持有锁,则该端口的 'exclusive 尝试会失败。

从 open-input-output-file 获得的输入端口的锁可以通过相应的输出端口的 port-file-unlock 释放, 反之亦然。 如果 open-input-output-file 的输出端口持有一个 'exclusive 锁, 相应的输入端口仍然可以获得一个 'shared 锁,甚至多次; 在 Windows 上,每次成功的锁尝试都需要一个 port-file-unlock , 而在 Unix 和 Mac OS 上,一个 port-file-unlock 就可以平衡锁的尝试。 在 Unix 和 Mac OS 上,输入端口上的 'shared 锁可以通过相应的输出端口升级为 'exclusive 锁, 在这种情况下,单个 port-file-unlock (在任何一个端口)就可以释放该锁,而在 Windows 上不允许这种升级。

通常只支持文件端口的锁,试图用其他类型的文件流端口获取锁会引发 exn:fail:filesystem 。

procedure

(port-file-unlock port) → void?

  port : file-stream-port?
释放当前进程在 port 的文件上持有的锁。

procedure

(port-file-identity port) → exact-positive-integer?

  port : file-stream-port?
返回 一个数字,代表 port 所读或写的设备和文件的身份。 对于开放时间重叠的两个 port, 当且仅当这两个 port 访问相同的设备和文件时, port-file-identity 的结果对这两个 port 是相同的。 对于开放时间不重合的端口, 不能保证端口的身份 (即使这些端口实际访问了同一个文件) –除了可以通过与其他端口的关系推断出来之外。 如果 port 被关闭,将引发 exn:fail exception is raised 。 在 Windows 95、98 和 Me 上,如果 port 连接到一个管道而不是一个文件, 则会引发 exn:fail:filesystem exception is raised 。

Examples:
> (define file1 (open-output-file some-file))
> (define file2 (open-output-file some-other-file))
> (port-file-identity file1)

586587131106055747404

> (port-file-identity file2)

715714339622022608716

> (close-output-port file1)
> (close-output-port file2)
