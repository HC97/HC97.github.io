17 不安全的操作

 (require racket/unsafe/ops) package: base

所有由 racket/base 和 racket 提供的函数和形式都会检查它们的参数, 以确保参数符合契约和其他约束。 例如, vector-ref 检查它的参数,以确保第一个参数是一个向量, 第二个参数是一个精确的整数,并且第二个参数在 0 和比向量的长度少 1 之间,包括在内。

由 racket/unsafe/ops 提供的函数是 unsafe 。 它们有一定的约束条件,但这些约束条件不被检查,这使得系统可以生成和执行更快的代码。 如果参数违反了不安全函数的约束,函数的行为和结果是不可预测的,整个系统会崩溃或损坏。

racket/unsafe/ops 的所有导出绑定都是在 protect-out 意义上的保护, 所以可以通过调整代码检查器来防止对不安全操作的访问(见 Code Inspectors) 。

procedure

(unsafe-set-immutable-car! p v) → void?

  p : pair?
  v : any/c

procedure

(unsafe-set-immutable-cdr! p v) → void?

  p : pair?
  v : any/c
正如它们矛盾的名字所暗示的那样,使用这些函数没有 no generally correct way 。 然而,作为最后的手段,它们可能是有用的,在成对使用受限的情况下, 以及在对 Racket 的实现(包括对编译器优化的限制)做出正确假设时。

使用 unsafe-set-immutable-car! 和 unsafe-set-immutable-cdr! 的一些误区:

  • 消耗一对的函数可能会利用不变性,例如计算一次列表的长度并期望列表保持该长度,或者根据合约检查列表并期望合约此后保持。

  • 一个对的 list? 的结果可能是内部缓存的,因此将一个对的 cdr 从列表改为非列表, 或者反过来,都可能导致 list? 产生错误的值–对于被改变的对或达到被改变的对的另一个对来说。

  • 编译器可能会重新排序,甚至优化对 car 或 cdr 的调用,理由是对是不可变的,在这种情况下, unsafe-set-immutable-car! 或 unsafe-set-immutable-cdr! 可能不会对 car 或 cdr 的使用产生影响。

Added in version 7.9.0.18 of package base.

与 impersonate-procedure 类似,但假设 replacement-proc 调用 proc 本身。 当 unsafe-impersonate-procedure 的结果应用于参数时,参数会直接传递给 replacement-proc , 而忽略 proc 。 同时,当给出 unsafe-impersonate-procedure 和 proc 的结果时, impersonator-of? 报告 #t 。

如果 proc 本身是一个从 impersonate-procedure* 或 chaperone-procedure* 派生的冒充者, 请注意 replacement-proc 将无法正确调用它。 具体来说,由 unsafe-impersonate-procedure 将不会被传递给提供给 impersonate-procedure* 或 chaperone-procedure* 以产生 proc 的包装过程。

最后,与 impersonate-procedure 不同, unsafe-impersonate-procedure 没有将 impersonator-prop:application-mark 作为一个 prop 进行特别处理。

unsafe-impersonate-procedure 的不安全性仅限于上述与 impersonate-procedure 的区别。 当参数被提供时, unsafe-impersonate-procedure 的参数上的契约被检查。

举个例子,假设 f 接受一个参数,并且不是从 impersonate-procedure* 或 chaperone-procedure* 派生的,那么
(λ (f)
  (unsafe-impersonate-procedure
   f
   (λ (x)
     (if (number? x)
         (error 'no-numbers!)
         (f x)))))
相当于
(λ (f)
  (impersonate-procedure
   f
   (λ (x)
     (if (number? x)
         (error 'no-numbers!)
         x))))

同样,在对 f 的假设相同的情况下,下面两个程序 wrap-f1 和 wrap-f2 几乎是等价的; 它们的区别仅仅在于当它们的参数是返回多个值的函数时产生的错误信息(以及它们更新不同的全局变量)。 使用 unsafe-impersonate-procedure 的版本将在 let 表达式中发出关于多个返回值的错误信号, 而使用 impersonate-procedure 的版本则从 impersonate-procedure 发出关于多个返回值的错误信号。
(define log1-args '())
(define log1-results '())
(define wrap-f1
  (λ (f)
    (impersonate-procedure
     f
     (λ (arg)
       (set! log1-args (cons arg log1-args))
       (values (λ (res)
                 (set! log1-results (cons res log1-results))
                 res)
               arg)))))
 
(define log2-args '())
(define log2-results '())
(define wrap-f2
  (λ (f)
    (unsafe-impersonate-procedure
     f
     (λ (arg)
       (set! log2-args (cons arg log2-args))
       (let ([res (f arg)])
         (set! log2-results (cons res log2-results))
         res)))))

Added in version 6.4.0.4 of package base.

17.6 不安全断言

procedure

(unsafe-assert-unreachable) → none/c

就像 assert-unreachable ,但 unsafe-assert-unreachable 的契约永远不会被满足, "不安全" 的含义是,如果对 unsafe-assert-unreachable 的调用被达到,任何事情都可能发生。

编译器可以利用其自由度来选择方便或有效的行为,以代替对 unsafe-assert-unreachable 的调用。例如,表达式

(lambda (x)
  (if (pair? x)
      (car x)
      (unsafe-assert-unreachable)))

可能会被编译成相当于

(lambda (x) (unsafe-car x))

因为选择使 (unsafe-assert-unreachable) 的行为与 (unsafe-car x) 的行为相同, 使得 if 的两个分支都相同,然后 pair? 测试就可以取消了。

Added in version 8.0.0.11 of package base.

17.7 不安全的未定义

 (require racket/unsafe/undefined) package: base
The bindings documented in this section are provided by the racket/unsafe/undefined library, not racket/base or racket.

常数 unsafe-undefined 在内部被用作占位符的值。 例如,它被 letrec 用作一个尚未被赋值的变量的值。 然而,与 racket/undefined 导出的 undefined 值不同, unsafe-undefined 值不应该作为安全表达式的结果泄露, 也不应该作为一个过程的可选参数传递(因为它可能被算作 "没有提供值")。 有可能产生 unsafe-undefined 的表达式结果可以通过 check-not-unsafe-undefined 来保护, 这样就可以引发一个异常,而不是产生一个 undefined 值。

unsafe-undefined 值总是 eq? 到自己。

Added in version 6.0.1.2 of package base.
Changed in version 6.90.0.29: 有可选参数的程序有时会在内部使用 unsafe-undefined 值来表示 "没有提供参数"。

value

unsafe-undefined : any/c

不安全的 "未定义" 常量。

关于使用 unsafe-undefined 的重要限制见上文。

procedure

(check-not-unsafe-undefined v sym)

 → (and/c any/c (not/c (one-of/c unsafe-undefined)))
  v : any/c
  sym : symbol?
检查 v 是 unsafe-undefined ,如果是,则引发 exn:fail:contract:variable , 并给出类似于 “sym: undefined; use before initialization.”的错误信息。 如果 v 不是 unsafe-undefined, 那么 v 被返回。

procedure

(check-not-unsafe-undefined/assign v sym)

 → (and/c any/c (not/c (one-of/c unsafe-undefined)))
  v : any/c
  sym : symbol?
与 check-not-unsafe-undefined 相同, 只是错误信息(如果有的话)是按照 “sym: undefined; assignment before initialization.” 的思路。

procedure

(chaperone-struct-unsafe-undefined v) → any/c

  v : any/c
如果 v 是一个结构(通过一些 inspector 查看),则陪衬 v 。 对结构中一个字段的每一次访问都会被检查,以防止返回 unsafe-undefined 。 同样地,对结构中的字段的每次赋值都会被检查(除非如下所述禁用检查), 以防止赋值给当前值为 unsafe-undefined 的字段。

当一个字段的访问会产生 unsafe-undefined ,或者当一个字段的赋值会取代 unsafe-undefined 时, exn:fail:contract 异常就会被引发。

只要 (continuation-mark-set-first #f prop:chaperone-unsafe-undefined) 返回 unsafe-undefined ,监护人的字段赋值检查就会被禁用。 因此,一个字段初始化赋值–旨在替换字段的 unsafe-undefined 值– 应该用 (with-continuation-mark prop:chaperone-unsafe-undefined unsafe-undefined ....) 来包装。

value

prop:chaperone-unsafe-undefined : struct-type-property?

一个 structure type property ,它使结构类型的构造函数产生一个实例的 chaperone , 其方式与 chaperone-struct-unsafe-undefined 相同。

该属性的值应该是一个用作字段名的符号列表,但该列表应该与结构的字段顺序相反。 当一个字段的访问或赋值会产生或取代 unsafe-undefined 时, 如果结构属性的值提供了一个字段名,则会引发 exn:fail:contract:variable 异常,否则会引发 exn:fail:contract 异常。
