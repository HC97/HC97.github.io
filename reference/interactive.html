18.5 交互式模块加载

racket/rerequire 和 racket/enter 库提供了对加载、重新加载和使用模块的支持。

18.5.1 进入模块

 (require racket/enter) package: base
The bindings documented in this section are provided by the racket/enter and racket/init libraries, which means that they are available when the Racket executable is started with no command-line arguments. They are not provided by racket/base or racket.

syntax

(enter! module-path)

(enter! #f)
(enter! module-path flag ...+)
 
flag = #:quiet
  | #:verbose-reload
  | #:verbose
  | #:dont-re-require-enter
打算在 REPL 中使用,比如在交互式模式下启动 racket 时。 当提供了一个 module-path (与 require 的意义相同), 相应的模块会被加载或通过 dynamic-rerequire 调用, 并且当前的 namespace 会通过 module->namespace 改变为模块的主体。 当 #f 被提供时,当前的 namespace 将被恢复到原来的名字空间。

额外的 flag 可以自定义 enter! 的各个方面:

  • #:verbose, #:verbose-reload 和 #:quiet 的标志对应于 dynamic-rerequire 的 'all, 'reload 和 'none 。默认值对应于 #:verbose-reload 。

  • 在将名字空间切换到指定的模块后, enter! 会自动要求 racket/enter 进入名字空间, 这样就可以用 enter! 来再次切换名字空间。 在某些情况下,要求 racket/enter 可能是不可取的(例如,在一个使用 racket/enter 的工具中), 使用 #:dont-re-require-enter 标志来禁用要求。

procedure

(dynamic-enter! mod    
  [#:verbosity verbosity    
  #:re-require-enter? re-require-enter?]) → void?
  mod : (or/c module-path? #f)
  verbosity : (or/c 'all 'reload 'none) = 'reload
  re-require-enter? : any/c = #t
enter! 的程序变体,其中 verbosity 被传递给 dynamic-rerequire 和 re-require-enter? 决定 dynamic-enter! 是否需要在新进入的命名空间中使用 racket/enter 。

Added in version 6.0.0.1 of package base.

18.5.2 加载和重新加载模块

 (require racket/rerequire) package: base
The bindings documented in this section are provided by the racket/rerequire library, not racket/base or racket.

procedure

(dynamic-rerequire module-path    
  [#:verbosity verbosity]) → (listof path?)
  module-path : module-path?
  verbosity : (or/c 'all 'reload 'none) = 'reload
类似于 (dynamic-require module-path 0),但支持重载。 dynamic-rerequire 函数是为了在交互环境中使用,特别是通过 enter! 。

如果调用 module-path 需要加载任何文件,那么文件的修改日期将被记录。 如果文件被修改,那么稍后的 dynamic-rerequire 就会从源码重新加载模块; 另见 模块的重新声明 。 同样地,如果后来的 dynamic-rerequire 间接 require 一个修改过的模块, 那么需要的模块就会被重新载入。 重新加载的支持只适用于首次通过 dynamic-rerequire 加载的模块(无论是直接加载还是通过传递性 require 间接加载)。

返回的列表包含了此次调用 dynamic-rerequire 时被重新加载的模块的绝对路径。 如果返回的列表为空,说明没有模块被改变或加载。

当 enter! 从文件中加载或重新加载一个模块时,它可以向 (current-error-port) 打印一条信息, 这取决于 verbosity: 'all 打印所有加载和重新加载的信息, 'reload 只打印重新加载的模块的信息, 'none 则禁止打印输出。
