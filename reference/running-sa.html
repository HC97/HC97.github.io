18.1 运行 Racket 或 GRacket

核心的Racket运行时系统有两个主要的变体:

  • Racket,它提供了赖以实现 racket/base 的基元库。 在 Unix 和 Mac OS 上,该可执行文件被称为 racket 。 在 Windows 上,可执行文件被称为 Racket.exe.

  • GRacket,它是 racket 的一个 GUI 变体,在系统上可以区分它们。 在 Unix 上,可执行文件被称为 gracket, 单实例标志和 X11 相关的标志会被处理,并特别传达给 racket/gui/base 库。 在 Windows 上,可执行文件被称为 GRacket.exe, 它是一个 GUI 应用程序(相对于控制台应用程序),实现了单实例支持。 在 Mac OS 上, gracket 脚本会启动 GRacket.app 。

18.1.1 初始化

在启动时,顶层环境不包含任何绑定–甚至不包含 #%app 的函数应用。 名字以 #% 开头的原始模块被定义,但它们并不是为了直接使用,而且这类模块的集合可以改变。 例如, '#%kernel 模块最终被用来引导 racket/base 的实现。

Racket 或 GRacket 的第一个动作是将 current-library-collection-paths 初始化为 (find-library-collection-paths pre-extras extras) 的结果, 其中 pre-extras 通常为 null , extras 是在命令行中用 -S/--search 依次提供的额外目录路径。 从 Racket 或 GRacket 可执行文件中创建的可执行文件可以嵌入作为 pre-extras 的路径。

Racket 和 GRacket 接下来分别需要 require racket/init 和 racket/gui/init ,但只有当命令行在任何 eval, load 或 read-eval-print-loop 标志 (-e/--eval, -f/--load, -r/--script, -m/--main, 或 -i/--repl) 之前没有指定 require 标志 (-t/--require, -l/--lib 或 -u/--require-script) 。 初始化库可以通过 -I configuration option 来改变。 初始化库的 configure-runtime 子模块或初始化库的语言的 'configure-runtime 属性在库实例化之前被使用;见 语言运行时配置 。

在潜在地加载初始化模块之后,表达式 eval 、文件 load 和模块 require 按照命令行上提供的顺序执行。 如果任何一个引发了未捕获的异常,那么其余的 eval、load 和 require 将被跳过。 如果第一个 require 在任何 eval 或 load 之前,因此初始化库被跳过, 那么在模块实例化之前,所需模块的 configure-runtime 子模块或所需模块的库语言的 'configure-runtime 属性被使用;参见 语言运行时配置 。

在运行完所有的命令行表达式、文件和模块后,如果没有提供 configuration options以外的命令行标志, Racket 或 GRacket 会启动一个 read-eval-print 循环,进行交互式评估。 对于 Racket 来说,read-eval-print 循环是通过调用 racket/repl 的 read-eval-print-loop 来运行的。 对于 GRacket, read-eval-print 循环是通过从 racket/gui/base 调用 graphical-read-eval-print-loop 来运行的。 如果提供了任何不是 configuration option 的命令行参数,那么 read-eval-print-loop 就不会被启动, 除非在命令行中提供 -i/--repl 标志来特别重新启用它。

此外,就在 read-eval-print 循环启动之前,Racket 运行 racket/interactive, GRacket 运行 racket/gui/interactive, 除非在 (find-config-dir) 找到的安装的 "config.rktd" 文件中指定了不同的 interactive 文件, 或者在 (find-system-path 'addon-dir) 找到 "interactive.rkt" 文件。 如果在命令行中指定了 -q/--no-init-file 标志,那么就不会运行交互式文件。

最后,在 Racket 或 GRacket 退出之前,它会调用主线程中 executable-yield-handler 的当前值的程序, 除非指定 -V/--no-yield 命令行标志。 要求 racket/gui/base 设置这个参数调用 (racket 'yield) 。

Changed in version 6.7 of package base: 运行 racket/interactive 文件而不是直接运行 (find-system-path 'init-file) 。
Changed in version 6.90.0.30: 通过使用 racket/repl 或 racket/gui/base 而不是 racket/base 或 racket/gui/init 来运行一个read-eval-print循环。

18.1.2 退出状态

如果在命令行 eval (通过 -e等)、 load (通过 -f、 -r等)或 require (通过 -l、 -t 等)过程中发生错误—或者,更广泛地说, 如果围绕这些评估的 prompt 的中止处理程序被调用,Racket 或 GRacket 进程的默认退出状态为非零 —但只有在没有启动读-评估-打印循环的情况下。否则,默认的退出状态是 0 。

在所有情况下,对 exit 的调用(当默认的 exit handler 到位时)可以用一个特定的状态值结束进程。

18.1.3 启动库

 (require racket/init) package: base
racket/init 库是 Racket 的默认启动库。 它重新输出了 racket, racket/enter 和 racket/help 库, 并将 current-print 设置为使用 pretty-print 。

 (require racket/interactive) package: base
racket/interactive 是 REPL 开始时的默认启动库。 如果指定了 -q/--no-init-file ,它就不会运行。 可以通过修改 (find-config-dir) 中找到的 "config.rktd" 文件中的 'interactive-file 来改变互动文件。 另外,如果文件 "interactive.rkt" 存在于 (find-system-path 'addon-dir) 中, 它将被运行而不是安装范围内的交互式模块。
默认的交互式模块启动 xrepl 并运行用户主目录下的 (find-system-path 'init-file) 文件。 一个不同的交互式文件可以通过要求 racket/interactive 来保持这种行为。

Added in version 6.7 of package base.

 (require racket/language-info) package: base
racket/language-info 库提供了一个 get-info 函数,它接收任何值并返回另一个函数; 返回的函数接收一个键值和一个默认值,如果键值是 'configure-runtime , 它返回 '(#(racket/runtime-config configure #f)) ,否则返回默认值。

See also 模块处理配置 in Racket 指南.

向量 '#(racket/language-info get-info #f) 适合于附加到一个模块作为其语言信息, 以获得与 racket/base 语言一样的语言信息。

 (require racket/runtime-config) package: base
racket/runtime-config 库提供了一个 configure 函数,它接受任何值并将 print-as-expression 设为 #t 。

向量 #(racket/runtime-config configure #f) 适合作为运行时配置规范列表的成员 (如模块的语言信息函数返回的键 'configure-runtime) , 以获得与 racket/base 语言一样的运行时配置。

18.1.4 命令行

Racket 和 GRacket 可执行文件可以识别以下命令行标志:

  • 文件和表达式选项:

    • -e ‹expr› 或 --eval ‹expr› : eval ‹expr› 。 评估的结果会通过current-print 打印出来。

    • -f ‹file› 或 --load ‹file› : load ‹file›; 如果 ‹file› 是 "-",那么表达式将从标准输入读取和评估。

    • -t ‹file› 或 --require ‹file› : require ‹file›, 然后 require (submod (file "‹file›") main) 如果有的话。

    • -l ‹path› 或 --lib ‹path› : require (lib "‹path›"), 然后 require (submod (lib "‹path›") main) 如果有的话。

    • -p ‹package› : require (planet "‹package›"), 然后 require (submod (planet "‹package›") main) 如果有的话。

    • -r ‹file› 或 --script ‹file› : load ‹file› 尽管有这个名字, --script 通常不用于Unix的脚本。 关于脚本的更多信息,请参见 脚本 。 作为一个脚本。这个标志就像 -f ‹file› 加上 -N ‹file› 来设置程序名,以及 -- 使所有进一步的命令行元素都被当作非标志参数处理。

    • -u ‹file› 或 --require-script ‹file› : require ‹file› 作为一个脚本; 这个标志就像 -t ‹file› 加上 -N ‹file› 来设置程序名称, 并且 -- 导致所有进一步的命令行元素被当作非标志参数来处理。

    • -k ‹n› ‹m› ‹p› : 从文件位置 ‹n› 到 ‹m› 以及从 ‹m› 到 ‹p› 加载嵌入可执行文件的代码。 (在 Mac OS 上, ‹n›, ‹m› 和 ‹p› 是相对于可执行文件中的 __PLTSCHEME 段。 在 Windows 上,它们是相对于 257 类型和 ID 为 1 的资源而言的。 在使用ELF的Unix上,它们是相对于可执行文件中的 .rackprog 段的。) 第一个范围在每个新的 place 都被加载, 在这个范围内声明的任何模块都被认为是 module-predefined? 的意义上的预定义。 这个选项通常被嵌入到一个独立的二进制文件中,同时也嵌入了 Racket 代码。

    • -Y ‹file› ‹n› ‹m› ‹p› : 和 -k ‹n› ‹m› ‹p› 一样, 但是从 ‹file› 中读取(没有对段或资源偏移进行任何调整)。

    • -m 或 --main : 评估在顶层环境中对 main 的调用。 所有没有作为选项处理的命令行参数(即放在 current-command-line-arguments 中的参数) 都作为参数传递给 main 。 调用的结果通过 current-print 打印出来。

      对 main 的调用被构造成一个表达式 (main arg-str ...) , 表达式的词法上下文将 #%app 和 #%datum 绑定为 #%plain-app 和 #%datum, 但 main 的词法上下文是顶级环境。

  • 交互选项:

    • -i 或 --repl : 在显示 (banner) 和加载 (find-system-path 'init-file) 之后, 使用 read-eval-print-loop (Racket) 或 graphical-read-eval-print-loop (GRacket) 运行一个交互式的 read-eval-print 循环。 如果是 Racket, (read-eval-print-loop) 后面是 (newline) 。 对于 GRacket,提供 -z/--text-repl 配置选项来使用 read-eval-print-loop (和 newline) 而不是 graphical-read-eval-print-loop 。

    • -n 或 --no-lib : 在没有禁用的情况下, 跳过初始化库(即 racket/init 或 racket/gui/init, 除非用 -I 标志改变它)的要求。

    • -v 或 --version : 显示 (banner) 。

    • -K 或 --back : 仅适用于 Mac OS;使应用程序处于后台。

    • -V --no-yield : 跳过最后的 executable-yield-handler 的动作, 对于使用 racket/gui/base 的程序,通常会在 eventspace 中等待所有帧被关闭等,然后退出。

  • 配置选项:

    • -y 或 --make : 启用自动生成和更新初始命名空间中加载的模块的编译 ".zo" 文件。具体来说, (make-compilation-manager-load/use-compiled-handler) 的结果会在其他模块加载动作之前作为 compiled-load handler 安装。 请注意： 这个标志是为了在交互式设置中使用;在脚本中使用它可能是个坏主意, 因为脚本的并发调用可能会在试图更新编译文件时发生冲突,或者可能有文件系统许可问题。 使用 -c/--no-compiled 会取消 -y/--make 的效果。

    • -c 或 --no-compiled : 通过将 use-compiled-file-paths 初始化为 null,禁止加载已编译的 ".zo" 文件。 谨慎使用:这将有效地忽略所有 "compiled" 子目录的内容,因此任何使用过的模块都会被编译, —甚至包括 racket/base 和它的依赖项—这将导致昂贵的运行时间。

    • -q 或 --no-init-file : 跳过加载 (find-system-path 'init-file) 的 -i/--repl 。

    • -z 或 --text-repl : 仅限 GRacket;改变 -i/--repl ， 使用 textual-read-eval-print-loop ,而不是 graphical-read-eval-print-loop 。

    • -I ‹path› : 设置 (lib "‹path›") 为 require 的路径, 以初始化命名空间,除非命名空间初始化被禁用。 使用这个标志可以有效地设置read-eval-print循环和其他顶层评估的语言。

    • -X ‹dir› 或 --collects ‹dir› : 通过使 (find-system-path 'collects-dir) 产生 ‹dir›, 将 ‹dir› 设置为主要库集合的路径。 如果 ‹dir› 是一个空字符串,那么 (find-system-path 'collects-dir) 会返回 ".", 但是 current-library-collection-paths 被初始化为空列表, use-collection-link-paths 被初始化为 #f 。

    • -S ‹dir› 或 --search ‹dir› : 将 ‹dir› 添加到主集合目录之后的默认图书馆集合搜索路径。 如果多次提供 -S/--dir 标志,搜索顺序与提供的一样。

    • -G ‹dir› 或 --config ‹dir› : 设置由 (find-system-path 'config-dir) 返回的目录。

    • -A ‹dir› 或 --addon ‹dir› : 设置由 (find-system-path 'addon-dir) 返回的目录。

    • -U 或 --no-user-path : 在搜索集合、C 库等时, 通过将 use-user-specific-search-paths参数初始化为 #f ,省略用户特定的路径。

    • -A ‹dir› 或 --addon ‹dir› : 设置由 (find-system-path 'addon-dir) 返回的目录。

    • -R ‹paths› 或 --compiled ‹paths› : 设置 current-compiled-file-roots 参数的初始值, 优先于任何 PLTCOMPILEDROOTS 设置。 ‹paths› 参数的解析方式与 PLTCOMPILEDROOTS 相同 (见 current-compiled-file-roots) 。

    • -C 或 --cross : 选择跨平台构建模式, 使 (system-type 'cross) 报告 'force, 并将 (find-system-path 'config-dir) 和 (find-system-path 'collects-dir) 的当前配置分别设定为 (find-system-path 'host-config-dir) 和 (find-system-path 'host-collects-dir) 的结果。 如果多次提供 -C 或 --cross ,只有第一个实例有影响。

    • -N ‹file› 或 --name ‹file› : 将 (find-system-path 'run-file) 告的可执行文件的名称设为 ‹file› 。

    • -E ‹file› 或 --exe ‹file› : 将 (find-system-path 'exec-file) 报告的可执行文件的名称设为 ‹file› 。

    • -J ‹name› 或 --wm-class ‹name› : 仅限 GRacket,Unix;将 WM_CLASS 程序类别设置为 ‹name› (而 WM_CLASS 程序名称是由可执行文件名称或 -N/--name 参数派生的)。

    • -j 或 --no-jit : 通过将 eval-jit-enabled 参数设置为 #f 来禁用本地代码的即时编译器。

    • -M 或 --compile-any : 通过设置 current-compile-target-machine 参数为 #f 来启用独立于机器的字节码。

    • -d 或 --no-delay : 通过设置 read-on-demand-source 参数为 #f ,禁止按需解析编译后的代码和语法对象。

    • -b 或 --binary : 要求对进程的输入、输出和错误端口采用二进制模式,而不是文本模式。 这个标志目前没有作用,因为总是使用二进制模式。

    • -W ‹levels› 或 --warn ‹levels› : 设置将事件写入原始错误端口的日志级别。 可能的 ‹level› 值与 PLTSTDERR 环境变量的值相同。 更多信息请参见 记录 。

    • -O ‹levels› 或 --stdout ‹levels› : 设置将事件写到原始输出端口的日志级别。 可能的 ‹level› 值与 PLTSTDOUT 环境变量相同。 更多信息见 记录 。

    • -L ‹levels› 或 --syslog ‹levels› : 设置将事件写入系统日志的日志级别。 可能的 ‹level› 值与 PLTSYSLOG 环境变量相同。 更多信息见 记录 。

  • 元选项:

    • -Z : 这个标志后面的参数被忽略。 在一些贫困的脚本环境中,这个标志可以方便地取代或取消另一个命令行参数。

    • -- : 此标志后面的参数本身不作为标志使用。

    • -h or --help : 显示关于命令行标志和启动过程的信息,并退出,忽略所有其他标志。

如果至少提供了一个命令行参数,并且如果任何 configuration option 之后的第一个参数不是一个标志, 那么在第一个非标志参数之前会隐含地添加一个 -u/--require-script 标志。

如果除了 configuration options 之外没有提供任何命令行参数, 那么 -i/--repl 标志将被有效地加入。

对于 Unix 上的 GRacket 来说,当以下标志出现在命令行的开头时,就会被识别出来, 它们也算作配置选项(也就是说,它们不会禁用 read-eval-print 循环或者阻止插入 -u/--require-script):

  • -display ‹display› : 设置要使用的 X11 显示器。

  • -geometry ‹arg›, -bg ‹arg›, -background ‹arg›, -fg ‹arg›, -foreground ‹arg›, -fn ‹arg›, -font ‹arg›, -iconic, -name ‹arg›, -rv, -reverse, +rv, -selectionTimeout ‹arg›, -synchronous, -title ‹arg›, -xnllanguage ‹arg› 或 -xrm ‹arg› : 标准的 X11 参数,大部分被忽略,但为了与其他 X11 程序兼容而接受。 -synchronous 标志的行为与通常的方式相同。

  • -singleInstance : 如果现有的 GRacket 已经在同一个 X11 显示器上运行, 如果它是在具有相同主机名的机器上启动的, 如果它是以 (find-system-path 'run-file) 报告的相同名称启动的– 可能用 -N/--name 命令行参数设置– 那么所有非选项的命令行参数被视为文件名并通过应用程序文件处理程序发送到现有 GRacket 实例 (见 application-file-handler) 。

同样地,在 Mac OS 上,以 -psn_ 开头的前导开关被视为一个特殊的配置选项。 它表示 Finder 启动了应用程序,所以当前的输入、输出和错误输出被重定向到一个 GUI 窗口。

只要第一个开关不是 -) ,多个单字母开关(前面有一个 -- 的开关)可以通过串联字母的方式折叠成一个开关。 每个开关的参数被放在折叠的开关之后(按照开关的顺序)。 比如说

  -ifve ‹file› ‹expr›

和

  -i -f ‹file› -v -e ‹expr›

是等价的。如果一个折叠的 -- 出现在同一折叠集的其他折叠开关之前,它就会被隐含地移到折叠集的最后。

最后一个选项后面的额外参数可以从 current-command-line-arguments 参数中获得。

Changed in version 7.1.0.5: 添加了 -M/--compile-any 。
Changed in version 7.8.0.6: 添加了 -Z 。
Changed in version 8.0.0.10: 添加了 -E 。
Changed in version 8.0.0.11: 添加了 -Y 。
Changed in version 8.4.0.1: 添加了 -y/--make 。

18.1.5 语言运行时配置

See also 模块处理配置 in Racket 指南.

一个模块可以有一个 configure-runtime 子模块,当一个模块是一个程序的主模块时, 该子模块在模块本身之前是 dynamic-require 。 通常, configure-runtime 子模块由模块的语言添加到模块中 (即由 module 的初始绑定中的 #%module-begin 形式)。 configure-runtime 子模块的主体通常设置参数,可能包括 current-interaction-info 。

另外,或者除此之外,还有一个更古老的协议。 当一个模块使用 #lang 实现时, #lang 之后的语言可以指定当使用该语言的模块是程序的主模块时要执行的配置动作。 该语言通过以下方式指定运行时配置

  • 将 'module-language syntax property 附加到从源码读取的模块上 (见 module 和 module-compiled-language-info);

  • 让 'module-language syntax property 指示的函数识别 'configure-runtime 键,它返回一个向量列表; 每个向量必须具有 (vector mp name val) 的形式,其中 mp 是一个 module path, name 是一个符号, val 是一个任意的值;和

  • 以 ((dynamic-require mp name) val) 的形式调用每个函数, 配置运行时环境,通常是通过设置参数,如 current-print 。

'configure-runtime 查询返回一个向量列表,而不是直接配置环境, 这样,在创建独立可执行文件时,指定的模块将与程序捆绑在一起; 见 raco: Racket 命令行工具 中的 raco exe: 创建独立的可执行文件 。

关于定义一个新的 #lang 语言的信息,见 syntax/module-reader 。

18.1.6 语言扩展配置

一个模块 lang 可以有一个 configure-expand 子模块, 在另一个实现为 (module name lang ....) 的模块扩展之前, dynamic-require 。 该子模块被加载在 root namespace 中,与读者模块相同。 该子模块应该提供 enter-parameterization 和 exit-parameterization 两个过程, 每个过程都不需要参数并返回一个 parameterization:

  • lang 的 enter-parameterization 在一个模块的扩展开始时被调用 (module name lang ....), 参数化通过 call-with-parameterization 包裹了模块的扩展。

  • 如果 (module name lang ....) 的扩展触发了其他模块的扩展, 通常是因为这些模块是被扩展的模块所 require, 那么就为 lang 调用 exit-parameterization 。 在这种情况下, exit-parameterization 被调用以获得一个参数化, 这个参数化是围绕调用 enter-parameterization 为新被扩展的模块的语言而设置的。

current-parameterization 程序对 enter-parameterization 和 exit-parameterization 都有默认作用。

enter-parameterization 产生的参数化通常设置影响扩展期间错误报告的参数, 例如 error-syntax->string-handler 。 exit-parameterization 产生的参数化通常应该恢复 enter-parameterization 所做的任何改变, 同时保持其他参数值不变(如 current-load-relative-directory)。 要从 enter-parameterization 的使用到 exit-parameterization 的嵌套使用进行沟通,请使用一个私有 parameter 。

enter-parameterization 和 exit-parameterization 程序应该建立在当前的参数化之上, 但是它们一般不应该改变当前的参数,因为这种改变会超出使用返回的参数化的范围。 相反,使用 parameterize 来创建一个新的参数化,并更新参数值。 enter-parameterization 和 exit-parameterization 也不应该在当前 namespace 上操作,因为这可能会干扰模块的扩展。
