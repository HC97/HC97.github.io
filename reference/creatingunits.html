7.1 创建单位

syntax

(unit
  (import tagged-sig-spec ...)
  (export tagged-sig-spec ...)
  init-depends-decl
  unit-body-expr-or-defn
  ...)
 
tagged-sig-spec = sig-spec
  | (tag id sig-spec)
     
sig-spec = sig-id
  | (prefix id sig-spec)
  | (rename sig-spec (id id) ...)
  | (only sig-spec id ...)
  | (except sig-spec id ...)
     
init-depends-decl = 
  | (init-depend tagged-sig-id ...)
     
tagged-sig-id = sig-id
  | (tag id sig-id)
产生一个封装了其 unit-body-expr-or-defn 的单元。 unit 体中的表达式可以引用由 import 子句的 sig-spec 约束的标识符, 并且单元体必须为 export 子句中 sig-spec 的每个标识符包括一个定义。 被导出的标识符在定义单元和导入单元中都不能被 set! ,尽管初始化变量的隐式赋值可能作为突变可见。

每个导入或导出的 sig-spec 最终指的是一个 sig-id , 它是一个通过 define-signature 绑定到签名的标识符。 通过 sig-id 导入的每个标识符的 lexical information 从 sig-id 的词法信息开始; 更多信息见 define-signature 形式。

在一个特定的导入或导出位置,一个特定的 sig-id 所约束或要求的标识符集可以通过几种方式进行调整:

  • (prefix id sig-spec) 作为导入,与 sig-spec 的绑定方式相同, 只是每个绑定都有 id 作为前缀。 作为导出,这种形式使使用 id 前缀的定义满足 sig-spec 所要求的导出。

  • (rename sig-spec (id id) ...) 作为一个导入,与 sig-spec 的绑定方式相同, 只是在绑定时使用了第一个 id 而不是第二个 id (其中 sig-spec 本身必须暗示一个与第二个 id 绑定的 bound-identifier=? 的绑定)。 作为一个输出,这种形式使第一个 id 的定义满足 sig-spec 中第二个 id 所命名的输出。

  • (only sig-spec id ...) 作为导入,与 sig-spec 的约束相同, 但只限于列出的 id (其中 sig-spec 本身必须意味着对每个 id 的约束是 bound-identifier=? ) 这种形式在出口中是不允许的。

  • (except sig-spec id ...) 作为一个导入,与 sig-spec 的绑定相同, 但不包括所有列出的 id (其中 sig-spec 本身必须暗示对每个 id 的 bound-identifier=? ) 这种形式在出口中是不允许的。

正如语法所建议的,对签名的这些调整可以任意嵌套。

一个单元的声明的导入与实际提供的导入按签名匹配。 也就是说,链接时向单元提供进口的顺序是不相关的;重要的是每个提供的进口所实现的签名。 必须为每个声明的导入提供一个实际的导入。 同样地,当一个单元实现了多个签名时,出口签名的顺序并不重要。

为了支持同一签名的多个进口或出口,一个进口或出口可以使用 (tag id sig-spec) 的形式被标记。 当一个单元的导入声明被标记时,那么当该单元被链接时,一个实际的导入必须被赋予相同的标签(具有相同的签名)。 同样地,当一个单元的出口声明被标记时,那么对该特定出口的引用必须明确地使用该标记。

一个单元在语法上被禁止导入两个不一样的签名,除非它们有不同的标签; 两个签名只有在它们没有通过 extends 共享祖先时才是 distinct 。 同样的句法限制也适用于导出的签名。 此外,一个单元在语法上被禁止两次导入同一个标识符(在 sig-spec 上重命名和其他转换之后), 两次导出同一个标识符(同样在重命名之后),或者导出一个被导入的标识符。

当单元被链接时,被链接单元的主体将按照链接站点指定的顺序执行。 一个可选的 (init-depend tagged-sig-id ...) 声明通过指定当前单元必须在提供相应导入的单元之后被初始化, 从而限制了允许的链接顺序。 init-depend 声明中的每个 tagged-sig-id 必须在 import 子句中有一个相应的 import。

syntax

(define-signature sig-id extension-decl
  (sig-elem ...))
 
extension-decl = 
  | extends sig-id
     
sig-elem = id
  | (define-syntaxes (id ...) expr)
  | (define-values (id ...) expr)
  | (define-values-for-export (id ...) expr)
  | (contracted [id contract] ...)
  | (open sig-spec)
  | (struct id (field ...) struct-option ...)
  | (sig-form-id . datum)
     
field = id
  | [id #:mutable]
     
struct-option = #:mutable
  | #:constructor-name constructor-id
  | #:extra-constructor-name constructor-id
  | #:omit-constructor
  | #:omit-define-syntaxes
  | #:omit-define-values
将一个标识符 sig-id 绑定到一个签名上,该签名指定了一组用于导入或导出的绑定关系:

  • 签名声明中的每个 id 意味着实现该签名的单元必须为该 id 提供一个变量定义。 也就是说, id 可供导入签名的单元使用,而 id 必须由导出签名的单元定义。

  • 签名声明中的每个 define-syntaxes 形式都引入了一个宏,可供导入该签名的任何单元使用。 定义的 expr 中的自由变量首先指的是签名中的其他标识符, 如果签名不包括标识符,则指的是 define-signature 表单的上下文。

  • 签名声明中的每一个 define-values 形式都引入了代码,这些代码有效地在每一个导入签名的单元前缀。 定义的 expr 中的自由变量的处理方式与 define-syntaxes 相同。

  • 签名声明中的每一个 define-values-for-export 形式都会引入代码, 有效地给每个输出签名的单元加上后缀。 定义的 expr 中的自由变量与 define-syntaxes 的处理方式相同。

  • 签名声明中的每个 contracted 形式意味着输出该签名的单元必须为该形式中的每个 id 提供一个变量定义。 如果签名是导入的,那么在单元内对 id 的使用就会受到适当的契约的保护,使用该单元作为否定的责备。 如果签名是导出的,那么导出的值由适当的合约保护,这些合约使用单位作为正责,但是导出的标识符的内部使用不受保护。 contract 表达式中的变量与 define-syntaxes 的处理方式相同。

  • 每个 (open sig-spec) 都向签名中添加由 sig-spec 指定的一切。

  • 每个 (struct id (field ...) struct-option ...) 添加所有将被 (struct id (field ...) field-option ...) 绑定的标识符,其中额外的选项 #:omit-constructor 省略了构造器标识。

  • 每个 (sig-form-id . datum) 都以 sig-form-id 定义的方式扩展签名, 必须由 define-signature-form 绑定。 一个这样的绑定是针对 struct/ctc 的。

当 define-signature 形式包括一个 extends 子句时, 那么定义签名会自动包括扩展签名中的所有内容。 此外,新签名的任何实现都可以作为扩展签名的实现。

签名中每个 id 的 lexical information 与 sig-id 的词法信息进行比较。 id 相对于 sig-id 的额外作用域被记录为 id 。 当 sig-id 被用作引用时(例如,在 unit 的 import 子句中), 通过从引用的 sig-id 的词法信息开始,然后添加 id 的额外作用域, 为引用的上下文创建一个 id 的变体。
