15.1 Paths

当 Racket 程序需要一个文件系统路径作为参数时,路径可以以字符串或 path 数据类型的实例形式提供。 如果提供的是字符串,就用 string->path 将其转换为路径。 请注意,有些路径可能无法用字符串表示;更多信息请参见 Unix 的路径表示法 和 Windows的路径表示法 。 一个生成文件系统路径的 Racket 过程总是会生成一个 path 值。

默认情况下,路径是为当前平台创建和操作的,但是仅仅操作路径的过程(不使用文件系统)可以使用其他支持的平台的约定来操作路径。 bytes->path 过程接受一个可选的参数,表示路径的平台,可以是 'unix 或 'windows 。 对于其他函数,如 build-path 或 simplify-path, 其行为对所提供的路径类型是敏感的。 除非另有规定,一个需要路径的过程只接受当前平台的路径。

当两个 path 值使用相同的约定类型,并且它们的字节串表示是 equal? 时, 它们就是 equal? 。 路径字符串(或字节字符串)不能是空的,也不能包含一个空字符或字节。 当一个空字符串或一个包含 nul 的字符串被作为路径提供给除 absolute-path?, relative-path? 或 complete-path? 之外的任何过程时,会引发 exn:fail:contract exception is raised 。

大多数接受路径的 Racket 基元在使用路径之前都会先对其进行 cleanse 。 除了 cleanse-path, expand-user-path 和 simplify-path 外,构建路径或仅仅检查路径形式的程序不对路径进行清理。 关于路径清理的更多信息和其他特定平台的细节,请参见 Unix and Mac OS 路径 和 Windows 路径 。
