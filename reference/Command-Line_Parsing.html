15.9 Command-Line Parsing

 (require racket/cmdline) package: base
The bindings documented in this section are provided by the racket/cmdline and racket libraries, but not racket/base.

syntax

(command-line optional-name-expr optional-argv-expr
              flag-clause ...
              finish-clause)
 
optional-name-expr = 
  | #:program name-expr
     
optional-argv-expr = 
  | #:argv argv-expr
     
flag-clause = #:multi flag-spec ...
  | #:once-each flag-spec ...
  | #:once-any flag-spec ...
  | #:final flag-spec ...
  | #:usage-help string ...
  | #:help-labels string ...
  | #:ps string ...
     
flag-spec = (flags id ... help-spec body ...+)
  | (flags => handler-expr help-expr)
     
flags = flag-string
  | (flag-string ...+)
     
help-spec = string
  | (string-expr ...+)
     
finish-clause = 
  | #:args arg-formals body ...+
  | #:handlers handlers-exprs
     
arg-formals = rest-id
  | (arg ...)
  | (arg ...+ . rest-id)
     
arg = id
  | [id default-expr]
     
handlers-exprs = finish-expr arg-strings-expr
  | finish-expr arg-strings-expr help-expr
  | 
finish-expr arg-strings-expr help-expr
unknown-expr
根据 flag-clause 中的规范解析命令行。

name-expr, 如果提供的话,应该产生一个路径或字符串,作为程序名称,以便在命令行格式不正确时报告错误。 它的默认值是 (find-system-path 'run-file) 。 当提供一个路径时,只有路径的最后一个元素被用来报告错误。

argv-expr, 如果提供的话,必须评估为一个列表或一个字符串的向量。 它的默认值是 (current-command-line-arguments) 。

命令行被分解为标志,每个标志可能有特定的参数,然后是(非标志)参数。 以 - 或 + 开头的命令行字符串被解析为标志,但标志的参数从不被解析为标志, 以 - 或 + 开头的整数和十进制数字不被视为标志。 命令行中的非旗语参数必须出现在所有旗语和旗语的参数之后。 任何超过第一个非标志参数的命令行字符串都不会被解析为一个标志。 内置的 -- 标志是命令行标志的终点;任何超过 -- 标志的命令行字符串都被解析为非标志参数。

一个 #:multi, #:once-each, #:once-any 或 #:final 子句引入了一组命令行标志的规范。 子句标签表明该标志可以在命令行中出现多少次:

  • #:multi — 这组中指定的每个标志可以在命令行中代表任意次数; 也就是说,这组中的标志是独立的,每个标志可以被多次使用。

  • #:once-each — 集合中指定的每个标志都可以在命令行中表示一次; 也就是说,集合中的标志是独立的,但每个标志最多只能指定一次。 如果一个标志在命令行中被代表一次以上,就会引发 exn:fail exception is raised 。

  • #:once-any — 在命令行中只能表示该集合中指定的一个标志;也就是说,该集合中的标志是相互排斥的。 如果该集合在命令行中被表示了一次以上,就会引发 exn:fail exception is raised 。

  • #:final — 和 #:multi 一样,只是在标志后面的参数不会被当作标志。 注意,如果多个 #:final 标志的名字很短,那么可以指定多个 #:final 标志; 例如,如果 -a 是一个 #:final 标志, 那么 -aa 在一个命令行参数中结合了两个 -a 的实例。

一个正常的标志说明有四个部分:

  • flags — 一个标志字符串,或一组标志字符串。 如果提供了一组标志,所有的标志都是等同的。 每个标志字符串必须是 "-x" 或 "+x" 的形式,代表某个字符 x, 或 "--x" 或 "++x" 代表某个字符序列 x 。 一个 x 不能只包含数字或数字加上一个小数点,因为简单的(有符号的)数字不被当作标志。 此外,标志 "--", "-h" 和 "--help" 是预定义的,不能被改变。

  • id — 绑定到标志的参数的标识符。 标识符的数量决定了在命令行中可以提供多少个参数,这些标识符的名称将出现在描述该标志的帮助信息中。 id 被绑定到处理该标志的 body 中的字符串值。

  • help-spec — 一个描述该标志的字符串或字符串序列。 这个字符串用于内置的 -h (或 --help)标志的处理程序所产生的帮助信息中。 可以提供一个单一的字面字符串,也可以提供产生字符串的任何数量的表达式; 在后一种情况下,第一个表达式之后的字符串将显示在后续的行中。

  • bodys — 当命令行上出现一个 flags 时被评估的表达式。 标志是从左到右解析的,当遇到相应的标志时,每一串 body 都被评估。 当 body 被评估时,前面的 id 被绑定到命令行上为标志提供的参数。

一个使用 => 的标志说明会转到指定处理程序和帮助字符串的更通用方法。 在这种情况下,由 handler-expr 和 help-expr 返回的处理程序和帮助字符串列表被用作 parse-command-line 的 table 参数。

一个 #:usage-help 子句在用法行之后立即插入文本行。 子句中的每个字符串都提供了一个单独的文本行。

一个 #:help-labels 子句在命令行标志的帮助表中插入文本行。 子句中的每个字符串都提供了一个单独的文本行。

一个 #:ps 子句在帮助输出的末尾插入文本行。子句中的每个字符串都提供了一个单独的文本行。

在flag子句之后,最后一个子句处理没有被解析为 flag 的命令行参数:

  • 不提供结束子句与提供 #:args () (void) 相同。

  • 对于 #:args 结束子句, arg-formals 中的标识符被绑定到剩余的命令行字符串上, 就像标识符被绑定到 lambda 表达式上一样。 因此,指定一个 id (不带括号)就可以将所有剩下的参数收集到一个列表中。 arg-formals 规范的有效节数决定了用户可以提供的额外命令行参数的数量, 而 arg-formals 中的标识符的名称会在帮助字符串中使用。 当命令行被解析时,如果提供的参数数量不能与 arg-formals 中的标识符相匹配,就会引发 exn:fail exception is raised 。 否则, args 子句的 body 被评估以处理剩余的参数, 最后一个 body 的结果是 command-line 表达的结果。

  • 一个 #:handlers 结束子句会转到一个更通用的方法来处理剩余的参数。 在这种情况下,表达式的值就像最后两到四个参数一样被使用 parse-command-line 。

例子：

(define verbose-mode (make-parameter #f))
(define profiling-on (make-parameter #f))
(define optimize-level (make-parameter 0))
(define link-flags (make-parameter null))
 
(define file-to-compile
  (command-line
   #:program "compiler"
   #:once-each
   [("-v" "--verbose") "Compile with verbose messages"
                       (verbose-mode #t)]
   [("-p" "--profile") "Compile with profiling"
                       (profiling-on #t)]
   #:once-any
   [("-o" "--optimize-1") "Compile with optimization level 1"
                          (optimize-level 1)]
   ["--optimize-2"        (; show help on separate lines
                           "Compile with optimization level 2,"
                           "which includes all of level 1")
                          (optimize-level 2)]
   #:multi
   [("-l" "--link-flags") lf ; flag takes one argument
                          "Add a flag <lf> for the linker"
                          (link-flags (cons lf (link-flags)))]
   #:args (filename) ; expect one command-line argument: <filename>
   ; return the argument as a filename to compile
   filename))

procedure

(parse-command-line name    
  argv    
  table    
  finish-proc    
  arg-help-strs    
  [help-proc    
  unknown-proc]) → any
  name : (or/c string? path?)
  argv : (or/c (listof string?) (vectorof string?))
  table : (listof (cons/c symbol? list?))
  finish-proc : ((list?) () #:rest list? . ->* . any)
  arg-help-strs : (listof string?)
  help-proc : (string? . -> . any) = (lambda (str) ....)
  unknown-proc : (string? . -> . any) = (lambda (str) ...)
使用 table 中的规范解析一个命令行。 关于命令行解析的概述,请参阅 command-line 形式,它为大多数目的提供了更方便的符号。

这个程序形式的 table 参数编码了 command-line 子句中的信息,除了 args 子句。 相反,参数是由 finish-proc 过程处理的,关于非 flag 参数的帮助信息在 arg-help-strs 中提供。 此外, finish-proc 过程还接收在解析 flag 时积累的信息。 help-proc 和 unknown-proc 参数允许自定义,这在 command-line 中是不可能的。

当没有更多的标志时, finish-proc 会被调用, 并带有为命令行标志(见下文)和来自命令行的剩余非标志参数积累的信息列表。 finish-proc 的arity 决定了从命令行接受和要求的非 flag 参数的数量。 例如,如果 finish-proc 接受两个或三个参数,那么必须在命令行中提供一个或两个非 flag 参数。 finish-proc 过程可以有任何 arity(见 procedure-arity) , 除了 0 或 0 的列表(也就是说,该过程必须至少接受一个或多个参数)。

arg-help-strs 参数是一个字符串列表,用于识别预期的(非 flag)命令行参数,每个参数一个。 如果允许任意数量的参数, arg-help-strs 中的最后一个字符串代表所有的参数。

如果命令行中包含 -h 或 --help 标志,那么 help-proc 过程将被调用,并带有帮助字符串。 如果遇到一个未知的标志, unknown-proc 过程会像处理标志的过程一样被调用(如下所述); 它必须至少接受一个参数(未知标志),但也可以接受更多的参数。 默认的 help-proc 会显示字符串并退出,默认的 unknown-proc 会引发 exn:fail 。

一个 table 是一个标志规范集的列表。 每个集被表示为一对两个项目:一个模式符号和一个帮助字符串或标志规范的列表。 模式符号是 'once-each, 'once-any, 'multi, 'final, 'help-labels, 'usage-help 或 'ps 中的一个,其含义与 command-line 中的相应子句标签相同。 对于 'help-labels, 'usage-help 或 'ps 模式,将提供一个帮助字符串列表。 对于其他模式,提供了一个标志规格的列表,每个规格将一些标志映射到一个处理程序。 一个规范是一个包含三个项目的列表:

  • 规范所定义的标志的字符串的列表。 参见 command-line 以了解关于标志字符串格式的信息。

  • 当在命令行上发现其中一个标志时,一个处理该标志及其参数的过程。 这个处理过程的arity决定了标志所消耗的参数数量:处理过程被调用时,标志字符串加上来自命令行的下几个参数,以匹配处理过程的 arity。 处理程序必须接受至少一个参数来接收标志。 如果处理过程接受任意多的参数,所有剩余的参数都会被传递给处理过程。 一个处理程序的 arity 必须是一个数字或者一个 arity-at-least 值。

    处理程序的返回值被添加到一个列表中,最终被传递给 finish-proc 。 如果处理程序返回 #<void>, 则没有任何值被添加到这个列表中。 对于所有由处理程序返回的非 #<void> 值,列表中的值的顺序与命令行中的参数顺序相同。

  • 一个非空的列表,用于构建规范的帮助信息。 列表的第一个元素描述了标志;它可以是一个字符串或一个非空的字符串列表,在后一种情况下,每个字符串都显示在自己的行中。 主列表的其他元素必须是字符串,以命名该标志的预期参数。 为一个规范提供的额外帮助字符串的数量必须与该规范的处理程序所接受的参数数量相匹配。

下面的例子与 command-line 的核心例子相同,翻译成程序形式:

(parse-command-line "compile" (current-command-line-arguments)
  `((once-each
     [("-v" "--verbose")
      ,(lambda (flag) (verbose-mode #t))
      ("Compile with verbose messages")]
     [("-p" "--profile")
      ,(lambda (flag) (profiling-on #t))
      ("Compile with profiling")])
    (once-any
     [("-o" "--optimize-1")
      ,(lambda (flag) (optimize-level 1))
      ("Compile with optimization level 1")]
     [("--optimize-2")
      ,(lambda (flag) (optimize-level 2))
      (("Compile with optimization level 2,"
        "which implies all optimizations of level 1"))])
    (multi
     [("-l" "--link-flags")
      ,(lambda (flag lf) (link-flags (cons lf (link-flags))))
      ("Add a flag <lf> for the linker" "lf")]))
   (lambda (flag-accum file) file)
   '("filename"))
