6 使用 Git 开发包

当 Git 仓库被指定为包的来源时,仓库内容的副本会被安装为包的实现。 这种安装模式是为包的消费者设计的,他们通常使用一个包而不对其进行修改。 然而,安装的软件包副本不适合软件包作者的开发,因为安装的不是 Git 仓库的完整克隆。 Racket 包管理器提供了不同的安装模式来支持使用 Git 仓库克隆的包作者。

6.1 将 Git 签出作为一个目录进行链接

由于 Git 仓库签出是一个目录,因此可以像 链接和开发新包 中描述的那样,将其作为一个包进行链接。 在这种情况下,任何在本地进行的修改都会立即对包的安装生效,包括任何来自 git pull 的更新。 然而,开发者必须明确地将任何远程更新拉到仓库,包括当需要更新来满足依赖包的要求时。

在下一节中,我们描述了一个替代方案,它使 raco pkg update 意识到签出目录作为版本库克隆的状态。 此外,通过 raco pkg update ,一个目录链接的包可以被提升为克隆链接的包。

6.2 将 Git 签出链接为一个克隆文件

当一个软件包被安装到

  raco pkg install --clone ‹dir› ‹git-pkg-source›

来安装软件包,而不是将其作为仓库源码的一个单纯的副本来安装,该软件包是通过创建一个 ‹git-pkg-source› 的 Git 克隆作为 ‹dir› 来安装的。 克隆的签出与目录的链接方式相同,但与普通的目录链接不同,Racket 包管理器会跟踪仓库的连接。 ‹git-pkg-source› 必须是一个 Git 或 GitHub 的包源,或者必须是一个目录映射到 Git 或 GitHub 包源的包名; 如果源URL包含一个片段,它必须命名一个分支或标签(而不是一个原始提交)。 如果 ‹git-pkg-source› 通过 HTTPS 指向一个仓库,但没有 .git 后缀, 请使用 git+https:// 来指代该仓库。

当 ‹git-pkg-source› 的仓库被改变,以至于源代码有了新的校验和, 那么 raco pkg update 为该软件包从该仓库拉取提交到本地克隆中。 换句话说, raco pkg update 可以作为 git pull --ff-only 的替代品来拉取软件包的更新。 此外, raco pkg update 可以在检查依赖关系时拉动本地软件包库的更新。 例如, raco pkg update --all 会拉取所有链接的软件包库的更新。

一个用 --clone 提供的 package source 可以包括一个分支和/或进入仓库的路径。 分支规范会影响初始检出所使用的分支,而非空的路径会导致检出软件包的一个子目录被链接。

假设一个开发者要处理大量的软件包,并且只开发其中的几个。预定的工作流程如下:

  • 用 raco pkg install 安装所有相关的软件包。

  • 对于每一个从特定的 Git 仓库中开发出来的包,以 ‹pkg-name› 命名,用以下方式更新安装

      raco pkg update --clone ‹dir› ‹pkg-name›

    来更新安装,这样就可以丢弃软件包的原始安装,用本地克隆的 ‹dir› 来替换它。

    为了方便起见,当 ‹git-pkg-source› 和 ‹dir› 的最后一个元素相同时, ‹pkg-name› 就可以省略。 换句话说, --clone 的参数可以是一个指向 ‹pkg-name› 的路径:

      raco pkg update --clone ‹path-to›/‹pkg-name›

    作为进一步的便利,当从 Racket 主源码库从头开始构建时,Git 配置忽略了一个顶级的 "extra-pkgs" 目录。 这个目录是用来作为 --clone 的目标的:

      raco pkg update --clone extra-pkgs/‹pkg-name›

    如果 "extra-pkgs" 子目录不存在,它就会创建。

  • 如果一个软件包的当前安装不是从 Git 仓库中提取的(例如,它是从一个发行版或快照的内置软件包目录中提取的), 但 ‹catalog› 将软件包名称映射到正确的 Git 仓库中,那么将 --clone 与 --lookup 和 --catalog 结合起来:

      raco pkg update --lookup --catalog ‹catalog› --clone ‹path-to›/‹pkg-name›

    一个合适的 ‹catalog› 可能是 https://pkgs.racket-lang.org 。

  • 新克隆的软件包将以指定的(或现有安装的)仓库作为其 Git origin 。 如果你想从不同的仓库推送和拉取更新–例如,你自己的软件包源码的分叉– 那么就用 git 命令来添加或改变你的克隆的 origin 到其他仓库。例如,命令

      git remote set-url origin ‹url-of-other-repo›

    会使 git pull 和 git push 拉动和推送到指定的 ‹url-of-other-repo› 。

    你可以通过设置一个 upstream 远程来保留克隆与中央仓库的连接, 例如, git remote add upstream ‹url-of-central-repo› 。 这让你可以选择用 git pull --ff-only upstream 从中央仓库定期拉取提交内容。

    或者,先用 git 克隆目标 ‹url› ,然后在 ‹dir› 中提供本地克隆的路径。

      raco pkg update --clone ‹dir› ‹pkg-name›

    无论哪种方式,当 raco pkg update 将更新拉到克隆的时候,它仍然会从 ‹pkg-name› 的老源头对应的仓库中拉取, 而不是从 git remote ‹url› 中拉取。 通常,这就是软件包的开发者想要的;当他们不主动修改软件包时,其他开发者的更新应该从软件包的主仓库拉取。 如果 ‹url› 是 raco pkg update 的首选更新源,那么在 ‹url› 中使用

      raco pkg update --clone ‹dir› ‹url›

    然而,请注意,当提供另一个 ‹url› 时, raco pkg update 可能不太能够检测到多个软件包之间的仓库共享 (并保持软件包安装与特定克隆的一致关联)。

  • 以通常的方式用 git 工具管理每个已开发软件包的变化,但 raco pkg update 也可用于更新,包括大规模更新。

6.3 git 和 raco pkg 之间的互动

git 和 raco pkg 工具以特定的方式互动:

  • 与建立链接的

      raco pkg install --clone ‹dir› ‹git-pkg-source›

    或 raco pkg update 的情况下,如果本地仓库已经以 ‹dir› 的形式存在, 那么它将被留在原地,任何新的提交将从 ‹git-pkg-source› 获取。 软件包管理器并不试图检查预先存在的 ‹dir› 是否与 ‹git-pkg-source› 一致; 它只是开始获取 ‹dir› 的新提交,稍后的 git pull --ff-only 会发现任何不匹配。

    多个 ‹git-pkg-source› 可以提供给 raco pkg install, 这在多个软件包来自同一仓库,因此可以共享 ‹dir› 时是有意义的。 无论是通过单一的 raco pkg 使用,还是通过同一 --clone ‹dir› 的多次使用, 来自同一软件库的软件包应该从同一本地克隆中链接(假设它们在同一软件库中,因为它们应该被一起修改)。 软件包系统本身并不要求软件包之间的克隆共享,但由于非共享或不一致的安装模式可能会引起混淆, raco pkg install 和 raco pkg update 会报告非共享或不一致的安装。 在典型的情况下,默认的 --multi-clone ask 模式可以自动修复不一致的地方。

  • 当向有本地副本的仓库拉取修改时, raco pkg update 默认以相当于 git pull --ff-only 的方式拉取修改。 如果提供 --pull rebase ,则会以相当于 git pull --rebase 的方式拉取更改。 提供 --pull try 会尝试用 git pull --ff-only 拉动,但失败会被忽略

  • 当 raco pkg update 被指定为更新的目标时,它使用相当于 git merge --ff-only ‹checksum› 或 git merge --rebase ‹checksum› 。 这种方法旨在保留本地对软件包所做的任何修改, 但它意味着软件包不能仅仅通过指定 raco pkg update 的提交来 "降级" 到一个较早的提交; 任何已经在本地仓库中的较新的提交将被保留。

  • 在每次安装或更新后,安装包数据库会记录从源码库中提取的最新提交。 仓库签出中的当前提交仅在合并到拉出的提交时才被查询。 因此,在用 git push 推送版本库变化后, raco pkg update 更新有意义的是将软件包安装数据库与远程版本库状态同步(这时与本地版本库状态相同)。

  • 在检查 raco pkg install 或 raco pkg update 请求的依赖性和冲突时, 只有在当前检查包括目标提交时,才会直接使用克隆目录的内容。

    否则,会先用 git fetch, 获取提交内容,并在临时目录下创建一个额外的本地克隆。 如果整个安装或更新被认为对该副本中的远程提交(不一定是每个本地版本库中的当前提交)是成功的,那么对链接的版本库签出的更新就会进行。 最后,在所有检出成功后,其他软件包的安装和更新也会完成并被记录。 如果签出失败(例如,由于冲突或未提交的更改),那么版本库的签出将处于失败状态,但所有软件包的操作都会被取消。

  • 用 raco pkg remove 移除一个软件包,在移除软件包链接的同时,保留版本库签出的状态。
