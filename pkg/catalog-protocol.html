8 包目录协议

一个 package catalog 是由三种形式之一的 URL 指定的:

  • http:// 或 https:// — 一个远程 URL

  • file:// 以 .sqlite 结尾 — 一个本地 SQLite 数据库

  • file:// 不含 .sqlite — 一个本地目录

8.1 远程和目录

在远程 URL 或本地目录命名为 package catalog 的情况下,URL/路径被扩展如下,以获得关于包的信息:

  • pkg 和 ‹package› 路径元素,其中 ‹package› 是一个 package name, 在远程 URL 的情况下,加上一个 version=‹version› 查询(其中 ‹version› 是 Racket 的版本号)。

    这种URL/路径形式被用来获取关于 ‹package› 的信息。 对远程 URL 的 HTTP 请求应该用一个 read 哈希表来响应,如下所述。 在本地目录中通过添加 "pkg" 和 ‹package› 形成的路径应该指向一个类似于包含 read 哈希表的文件。

    这个哈希表应该提供以下的键:

    • 'source (required) — 一个 package source 字符串,通常是一个远程 URL。 如果这个源是一个相对的URL,那么它将被视为相对于目录的。

      Changed in version 6.0.1.7: 为目录服务器的客户端增加了相对路径支持。

    • 'checksum (必选) — 一个用于 checksum 的字符串。

    • 'name (可选) — 一个与 ‹package› 相同的字符串。

    • 'author (可选) — 一个字符串,代表包的作者,通常是一个电子邮件地址。

    • 'description (可选) — 一个描述软件包的字符串。

    • 'tags (可选) — 描述包的分类的字符串列表。

    • 'dependencies (可选) — 一个软件包的依赖性列表, 形状与 包的元数据 中描述的 deps "info.rkt" 字段相同。

    • 'modules (可选) — 由包提供的模块路径的列表; 每个模块路径应该按照 collapse-module-path 的意思进行规范化。

    • 'versions (可选) — 一个映射版本字符串和 'default 的哈希表, 其中每个特定版本的哈希表提供了覆盖主哈希表中的映射, 'default 适用于任何没有其他映射的版本。

      远程目录的客户可以请求特定版本的信息,但他们也应该检查目录响应中的 'versions 条目, 以防具有特定版本映射的目录被实现为目录或由文件服务的 HTTP 服务器实现。 同时,一个 'default 映射允许主哈希表提供适合 5.3.6 和更早版本的客户端的信息(不检查 'versions)。

    • 'ring (可选) — #f 或一个环号。参见 get-pkg-ring 以获得更多信息。

  • pkgs 路径元素: 获得一个由 package catalog 映射的包名称的列表。 对一个远程 URL 的 HTTP 请求应该以一个 read 字符串列表来响应。 在本地目录中通过添加 "pkgs" 形成的路径应该指向一个同样包含 read 字符串列表的文件。

    这种 URL/路径形式被 raco pkg catalog-copy 和允许用户浏览目录的工具所使用。

    在本地目录的情况下,如果没有 "pkgs" 文件,就会通过列出 "pkg" 目录下的所有文件来创建一个列表。

  • pkgs-all 路径元素: 获得一个哈希表,将软件包名称映射到软件包细节。 对一个远程 URL 的 HTTP 请求应该响应一个 read 哈希表,将字符串映射到哈希表。 通过添加 "pkgs-all" 形成的本地目录中的路径应该指向一个类似于包含 read 哈希表的文件。

    这种URL/路径形式是 pkgs URL/路径形式与每个软件包的 pkgs/‹package› 查询相结合的一种快捷方式。

    在本地目录的情况下,如果没有 "pkgs-all" 文件,就会从 "pkg" 目录中的文件创建一个列表。

注意,只要 "pkgs" 和 "pkgs-all" 文件存在, 通过 HTTP 服务器以文件形式提供的本地目录就可以作为远程 URL 使用(因为这些文件对本地来说是可选的,但对 HTTP 来说是必须的)。

PLT 托管的 package catalog 的来源是在 pkg-index 包中。

8.2 SQLite 目录

一个 SQLite 数据库 package catalog 是为了使用 pkg/db 库来构建和查询的, 但是只要数据库包含以下的表,就可以用任何方式来构建:

  • 一个 catalog 表,其格式为

      (id SMALLINT,

       url TEXT,

       pos SMALLINT)

    通常,这个表中只有一行是 (0, "local", 0), 但记录了一组其他目录内容的数据库也可以作为一个目录,在这种情况下,每一行都代表一个目录; id 字段是每个目录的唯一标识符, url 字段是目录的 URL, 而 pos 列是目录相对于其他目录的排序(其中 pos 越低越优先)。

  • 一个 pkg 表的格式是

      (name TEXT,

       catalog SMALLINT,

       author TEXT,

       source TEXT,

       checksum TEXT,

       desc TEXT)

    catalog 字段通常为 0; 在数据库反映了多个其他目录的情况下, catalog 字段表示软件包条目的源目录。

    pkg 和 catalog 字段共同决定了该表中的唯一行。

  • 一个 tags 表的形式是

      (pkg TEXT,

       catalog SMALLINT,

       tag TEXT)

    其中 pkg 和 catalog 的组合确定了 pkg 中的唯一行。

  • 一个 modules 表,其形式为

      (name TEXT,

       pkg TEXT,

       catalog SMALLINT,

       checksum TEXT)

    其中 pkg 和 catalog 组合确定了 pkg 中的唯一一行, 而 name 是打印的模块路径。

    这个表目前没有被任何 raco pkg 命令使用,但它可以被用来建议安装软件包以提供一个特定的库。

  • dependencies 表的形式为

      (onpkg TEXT,

       onversion TEXT,

       onplatform TEXT,

       pkg TEXT,

       catalog SMALLINT,

       checksum TEXT)

    其中 pkg 和 catalog 组合标识了 pkg 中的唯一行, onpkg, onversion 和 onplatform 代表依赖关系; 如果依赖关系没有版本或平台说明, onversion 或 onplatform 是一个空字符串。

    这个表目前没有被任何 raco pkg 命令使用。

  • ring 形表,其形式为

      (pkg TEXT,

       catalog SMALLINT,

       ring SMALLINT)

    其中 pkg 和 catalog 的组合标识了 pkg 中的唯一行。

    Added in version 6.10.0.3.
