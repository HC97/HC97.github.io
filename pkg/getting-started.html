1 包的入门

有两种方法来管理 Racket 包的安装:

  • 包管理器的图形界面。

    大多数用户通过 DrRacket 访问软件包管理器的图形界面, 它在 File 菜单中提供了 Package Manager... 项目。

    你也可以安装 gui-pkg-manager 包, 它提供了一个 Racket Package Manager 应用程序 (可以在命令行环境下作为 racket-package-manager 启动)。

  • raco pkg 命令行工具。

    raco 可执行文件可以启动各种 Racket 命令行工具, 而 raco pkg 命令则将各种包管理子命令分组。 不同的 raco pkg 子命令在 使用 raco pkg 中有所记载。

在这里我们将使用 raco pkg 命令来描述软件包管理的操作,但图形界面也允许同样的操作。

1.1 什么是包?

一个 package 不是你在 Racket 程序中直接引用的东西。 相反, package 是一组适合于 collection 层次结构的库, 你通过基于 collection 的路径来引用库。 在层次结构中相近的库可能由不同的包提供, 而一个包可能提供在层次结构中相距甚远的库(但在概念上有某种联系)。

Racket 文档会告诉你哪个包提供了一个给定的库。 例如, pict/face 库的文档说它是由 pict-lib 包提供的。 如果你在网络浏览器中阅读这篇文章, 点击 pict/face 就可以直接进入它的文档。

随着时间的推移,包可能会被重构,从而使一个库转移到一个不同的包中,但原来的包也应该继续提供这个库,通过声明对新包的依赖。 更一般地说,一个包的目的是有一个接口,只在库、绑定和功能方面有所增长,这提供了一个基本的向后兼容性。 不兼容的变化应该在一个新的包中实现。

1.2 检查你的安装

要查看你已经安装的软件包,请使用 raco pkg show 子命令:

  raco pkg show

除非你有一个特别小的 Racket 安装,否则你会有已经安装的包,可能在 "安装范围" 部分列出。 事实上,如果你有一个典型的 Racket 安装, 那么 raco pkg show 最初会显示一个 main-distribution 包和一个 racket-lib 包:

  Installation-wide:

   Package              Checksum          Source

   main-distribution    01..........ef    (catalog main-distribution)

   racket-lib           fe..........01    (catalog racket-lib)

  User-specific for installation ....:

   [none]

“Checksum” 一栏报告了所安装的每个软件包的具体实现。 一个软件包可以有一个更传统意义上的 version , 但就软件包系统而言, checksum 就是 “version” 。 当你请求更新时,如果软件包的当前实现与所安装的软件包有不同的 checksum , 那么无论软件包作者是否调整了软件包的 version ,软件包的安装都会被更新。

“Source” 一栏表示每个软件包是如何安装的。 catalog 来源表示该软件包是通过查阅 package catalog 安装的。 catalog 后面的名字表示从目录中请求的包的名字,这通常是(但不一定是)你的安装中存在的包的名字。 我们在 安装软件包 中讨论了 “Source” 的其他可能性。

无论是 main-distribution 包还是 racket-lib 包实际上都没有单独提供任何库,但它们都声明了对其他包的依赖。 racket-lib 包依赖于本地库包,如果有的话,适用于你的平台。 main-distribution 包依赖于许多被选入 Racket 主发行版的包。 如果你在 raco pkg show 中提供了 --all 标志, 那么你可以看到在安装 main-distribution 和 racket-lib (或者你明确选择的安装包)后自动安装的包。

  raco pkg show --all

每一个被 “auto-installed” 的软件包的名字旁边都有一个星号,以满足其依赖性。 所有自动安装的包和明确安装的包一样,可以供你使用,但通常你的代码应该只引用你明确安装的包。 自动安装的包和明确安装的包之间的区别在于各种命令,例如 raco pkg show, 如何对待这个包。 如果你特别要求安装一个自动安装的包,那么这个包就会被提升,然后被当作显式安装的包。

1.3 寻找软件包

PLT package catalog 在

https://pkgs.racket-lang.org

提供了一个可用的 Racket 软件包的集中列表。 PLT package catalog 通常是你寻找软件包时首先要查看的地方。

还有其他方法来分发和引用软件包。 例如,一个软件包可以直接从 ".zip" 文件中安装–可以在本地获得,也可以从网站上获得–或者从 Git 仓库中获得。 当一个软件包还没有准备好广泛分发,或者当它永远不会被广泛的受众所关注时,这种直接引用是有意义的。 因此,你可能会在邮件列表的帖子中发现非目录参考,由你的朋友推荐,或在垃圾邮件中宣传。

除了 PLT 的,可能还有其他的 package catalog 服务。 请注意,即使你从 PLT 的 package catalog 中发现了一个包的名字,你的安装可能被配置为查阅不同的 package catalog 来定位该包的实现(例如,获得该包的预建版本), 但你应该期望安装配置的 package catalog 能够提供 PLT package catalog 上描述的包。

1.4 安装软件包

如果你从 package catalog 中找到一个包的名字,那么就用 raco pkg install 来使用这个包的名字:

  raco pkg install ‹pkg-name›

如果这个包依赖于其他你还没有安装的包,那么 raco pkg install 会提醒你并询问是否也要安装它们。 使用 --auto 可以跳过这个问题,让依赖的软件包自动安装。 无论哪种方式,为满足依赖关系而安装的包都会被标记为自动安装,这使得它们更容易被卸载, 同时也使得它们在 raco pkg show 中被默认隐藏(因为为依赖关系而安装的包是一个实现细节,你通常并不关心)。

你提供给 raco pkg install 的参数不一定是 package catalog 所能识别的软件包名称。 一般来说, raco pkg install 的每个参数都是一个 package source 。 一个 package source 可以指一个 ".zip" 文件, 一个 ".tar" 文件,一个 Git 仓库,一个目录结构的网站,或者其他一些可能性。 在每一种情况下, package name 都是由 package source 推断出来的。 在软件包被安装后,你可以在其他 raco pkg 命令中使用该软件包的名称来引用已安装的软件包。

事实上, package catalog 实际上并不提供软件包的实现。 它只是将每个 package name 映射到一个 package source 。 当包管理器查询 package catalog 时,它会得到实际包实现的 package source , 所以每个从 package catalog 安装的包实际上是从 ".zip" 文件、Git 仓库等安装的。 在 package catalog 中注册只是一种使你的软件包更容易被找到和更新的方式。

1.5 更新软件包

如果你的软件包安装过时了,你可以用 raco pkg update 来更新软件包:

  raco pkg update ‹pkg-name›

要么指定个别软件包进行更新,要么使用 --all 来更新所有已安装的、有新 checksum 的软件包。

软件包管理器发现更新的方式取决于软件包的安装方式。 如果它是通过 package catalog 解析的 package name 安装的, 那么 package catalog 会被再次查询,以获得该包的当前 checksum , 如果 checksum 与当前安装不匹配,该包将被更新。 如果软件包是直接从 Git 参考中安装的,那么就会查询 Git 仓库以获得特定分支或标签的当前提交, 如果提交标识符与当前安装的 checksum 不匹配,就会更新该软件包。

在某些情况下,更新一个软件包可能需要更新该软件包的某个依赖项。 这应该只发生在软件包需要一个新的绑定、功能或依赖软件包的错误修复时,因为软件包是以一种向后兼容的方式发展的。 包的 versions 为包的作者提供了一种声明(以及让包管理器检查)这些依赖关系的方法。 最终的结果是, raco pkg update 可能会报告一个版本不匹配的错误,迫使你请求比你最初要求的更多的软件包更新。

通常情况下,你会向 raco pkg update 提供 package names 。 更广泛地说,你可以向 raco pkg update 提供一个 package source 。 在这种情况下,必须已经安装了一个同名的软件包,已安装的软件包将被替换成指定的软件包。 用一个新的 package source 替换一个包, 是对在先前指定的包源处获取一个具有新的 checksum 的替换包的一种概括。

1.6 移除软件包

正如你所期望的, raco pkg remove 会移除一个软件包:

  raco pkg remove ‹pkg-name›

如果一个软件包的安装触发了其他软件包的自动安装,那么移除该软件包并 不 会自动移除自动安装的软件包。 为 raco pkg remove 提供 --auto 标志, 无论是本身还是在卸载软件包时,都可以删除任何没有依赖关系的自动安装的软件包。

如果其他已安装的软件包依赖于它, raco pkg remove 命令将不会删除该软件包,除非你强制删除。 如果你想把一个包从明确安装的包降级为自动安装的包(为了以后的清理,也许是在其他包被删除的时候), 那么给 raco pkg remove 提供 --demote 标志。

1.7 创建软件包

一个包通常从一个包含模块文件的目录开始,逐渐成长为一个在 package catalog 中注册的 Git 仓库。

1.7.1 自动创建

为了方便起见, raco pkg new 可以自动创建一个 single-collection package 。 要创建 ‹pkg-name›:

  raco pkg new ‹pkg-name›

1.7.2 手动创建

要手动创建一个包,首先要建立一个目录,并选择它的名字 ‹pkg-name›:

  mkdir ‹pkg-name›

尽管一个包可以提供任何数量的 collections 中的库,但通常一个包只提供与包名匹配的集合中的库。 如果你的包是这种情况, 那么实现 ‹pkg-name› 集合中的模块的文件将直接进入你所创建的 ‹pkg-name› 目录。

如果你的包实现了多个 collections , 那么你就需要在 ‹pkg-name› 目录下添加一个基本的 "info.rkt" 文件:

  cd ‹pkg-name›
  echo "#lang info" > info.rkt
  echo "(define collection 'multi)" >> info.rkt

collection 定义告诉软件包管理器,该软件包在多个集合中实现了库, 每个集合由一个子目录表示,其名称与集合名称一致。 每个集合的库都放在该集合的目录中。

你可以从一个 single-collection package 开始, 然后通过重组包的目录把它改成一个 multi-collection package ,所以当你开始的时候不用太担心这个选择。

1.7.3 链接和开发新包

不管是创建一个 single-collection package 还是 multi-collection package 下一步都是将你的开发目录作为一个本地安装的包来链接。 在 ‹pkg-name› 目录下使用 raco pkg install :

  raco pkg install

如果你在这时使用 raco pkg show ,你会看到 ‹pkg-name› 的一行。 "Source "一栏将显示它是一个链接包,而 "Checksum" 一栏将显示 #f, 这意味着没有校验和。 像 raco pkg update 这样的子命令不会对一个链接包起作用, 因为只要你修改了包的实现,就会对包进行 “updates” 。

最后,在 ‹pkg-name› 目录下,添加目录和/或文件来实现你的软件包所提供的集合和/或模块。 例如,一个提供 games/tic-tac-toe/main 和 data/matrix 库的 tic-tac-toe multi-collection package 的开发者可以创建这样的目录和文件:

  mkdir -p games/tic-tac-toe
  touch games/tic-tac-toe/info.rkt
  touch games/tic-tac-toe/main.rkt
  mkdir -p data
  touch data/matrix.rkt

一个 "info.rkt" 文件对于没有依赖关系的 single-collection package 来说是没有必要的, 但你可能希望创建一个,以保存依赖关系的声明。 每个包都至少依赖于 base, 它提供了一个最小Racket安装的集合和库。 为了使你的软件包对其他用户发挥最大作用,你最终需要声明所有的依赖关系。 (幸运的是, raco setup 可以检查依赖关系,并帮助你找出需要声明的依赖关系)。

即使是一个 single-collection package, 你也可能想创建 "info.rkt" 并包括以下定义

(define collection "‹pkg-name›")

这个定义似乎是多余的,因为 ‹pkg-name› 可以作为包围目录的名称, 但是明确地声明集合名称可以防止你的包的实现的意义取决于实现被引用的方式。

最后,在 multi-collection package 的情况下, 注意 ‹pkg-name› 中的 "info.rkt" 文件是针对包的,而不是针对一个集合。 像 scribblings 或 raco-commands 这样的定义只在一个集合的 "info.rkt" 中起作用。 对于一个 single-collection package, "info.rkt" 文件对包和集合有双重作用。

1.8 分享软件包

在你的软件包准备好部署后,选择 GitHub 部署 或 手动部署, 然后转到 帮助他人发现你的软件包 。

1.8.1 GitHub 部署

首先,在GitHub上 创建一个免费账户， 然后为 你的软件包创建一个仓库 。 之后,你的 package source 就是:

https://github.com/‹user›/‹package›.git

如果你希望软件包在 ‹branch› 或 ‹tag› 而不是默认的分支, 那么在软件包源的末尾添加 "#‹branch›" 或 "#‹tag›" 。 如果你的包是版本库内的子目录 ‹path› , 则在包源的末尾添加 "?path=‹path›" 。

每当你

  git push

的时候,你的改动就会被那些使用 raco pkg update 的人自动发现, 然后从你基于 GitHub 的 package source 。

支持GitHub以外的服务需要 Racket 6.1.1.1 或更高版本。 只要服务器支持 HTTP(S) 协议或本地 Git 协议(但后者要使用 git:// 路径), 其他 Git 仓库服务可以和 GitHub 一样工作–包括 GitLab 或 BitBucket。

Racket 包管理器为基于 Git 的开发提供了更多支持,而不仅仅是部署。更多信息请参见 使用 Git 开发包 。

1.8.2 手动部署

另外,你也可以通过在你控制的 URL 上发布你的包来进行部署。 如果你这样做,最好先从你的包目录中创建一个存档:

  raco pkg create ‹package›

然后,将存档和它的 checksum 传到你的网站:

  scp ‹package›.zip ‹package›.zip.CHECKSUM your-host:public_html/

你的 package source 就会变成这样

http://your-host/~‹user›/‹package›.zip

每当你想提供一个新版本的软件包时,重新创建并重新上传软件包存档(和 checksum)。 当使用你的软件包源的人使用 raco pkg update 时,你的改变将自动被他们发现。

默认情况下, raco pkg create 会生成一个 ".zip" 档案。 关于更多的选项,请参考 raco pkg create 文档。 如果你想通过其他方式生成一个归档文件,只需将你在本节第一部分做的东西归档即可。 关于更正式的细节,请参考 package 的定义。

1.8.3 帮助他人发现你的软件包

通过使用 GitHub 部署 或 手动部署, 任何人都可以通过参考你的 package source 来安装你的包。 然而,在它被列入 package catalog 之前,他们将不能用简单的名字来指代它。

如果你想使用 PLT package catalog, 请浏览 https://pkgs.racket-lang.org/ 并上传一个新的软件包。 你需要先创建一个账户并登录。

你只需要去这个网站 一次 就可以列出你的软件包。 该服务器将定期检查你指定的软件包来源是否有更新。

如果你使用这个服务器,如果你使用公共的 Git 仓库进行部署,那么你将永远不需要打开网络浏览器来为终端用户更新你的软件包。 你只需要推送到你的 Git 仓库,然后在 24 小时内, PLT package catalog 会注意到, 并且 raco pkg update 更新会在你用户的机器上工作。

1.8.4 命名和设计软件包

我们建议在命名和设计软件包时采用以下惯例:

  • 软件包不应该包括制作它们的作者或组织的名字,而应该根据软件包的内容来命名。 例如, data-priority-queue 比 johns-amazing-queues 要好。

  • 提供到国外库或服务的接口的包应该与服务的命名相同。 例如,cairo 优于 Racket-cairo 或类似的名字。

  • 包的名称中一般不应包含类似版本的元素,最初。相反,当有必要进行向后不兼容的修改时,应该添加类似版本的元素。 例如,data-priority-queue 优于 data-priority-queue1 。 例外的情况包括为外部的、有版本的东西提供接口的包, 例如 sqlite3 或 libgtk2 。

  • 一个软件包的 version 声明只被其他软件包实现者用来有效地声明对所提供功能的依赖性。 这样的声明允许 raco pkg install 和 raco pkg update 来帮助检查依赖关系。 声明和改变版本是可选的, package catalog 忽略了版本声明; 特别是,当一个包的 checksum 发生变化时,它就是一个更新的候选者, 与该包的版本是否变化无关,甚至与版本变化的方向无关。 我们建议使用一个小于 "1.0" 的版本来表示一个包的接口是不稳定的, 当你准备承诺向后兼容时,再把它改为 "1.0" 。

  • 软件包不应该将大量的实用程序库与其他功能结合起来。 例如,一个包含 "racket" 集合的许多扩展的包, 像 "racket/more-lists.rkt" 和 "racket/more-bools.rkt" 不应该同时包含完整的应用程序, 因为其他对 "racket/more-bools.rkt" 库感兴趣的包将不希望依赖这样的应用程序。

  • 包通常应该包括文档和实现。 为了使一个包的实现与它的文档分开(用于本地文档没有用处的环境),可以定义一个包 ‹pkg-name›-lib ,只保存实现, ‹pkg-name›-doc 保存文档,而 ‹pkg-name› 则依赖于这两者, 并且用一个 implies 声明 “re-exports” 这两者(见 包的元数据 )。 如果你想把测试分开,可以把它们放在 不 依赖 ‹pkg-name› 的 ‹pkg-name›-test 包里。 类似地,使用 ‹pkg-name›-exe 表示可执行文件。

  • 包一般应该提供一个集合,其名称与包的名称相似。 例如, libgtk1 应该提供一个名为 "libgtk" 的集合。 例外情况包括对现有集合的扩展,例如 "data" 集合的新数据结构、DrRacket 工具、PLT 游戏的新游戏等等。

1.8.5 与 Racket 5.3.5 和 5.3.6 兼容的软件包

从 5.3.5 版本开始,Racket 中加入了一个测试版的软件包系统。 到 6.0 版本发布时,又增加了一些功能。

通过只使用某些较早的功能,可以制作一个可以用于 Racket 5.3.5, 5.3.6, 6.0 和更新版本的包。

在你的 info.rkt 中,你应该:

  • 使用 #lang setup/infotab (而不是 #lang info) 。

  • 使用 (define collection 'multi) 。 即使你的包有一个单一的集合,也要把它放在一个子目录下,做成一个多集合包。

  • 如果你依赖于另一个软件包的特定版本,使用 (other-package-name required-version) 表格(而不是带有 #:version 的表格)来说明。

最后,当在 https://pkgs.racket-lang.org 列出你的软件包时, 你应该使用 github://github.com/‹user›/‹repo›/‹rev›[/‹path›] 的 URL 格式提供一个 GitHub 源(而不是 git:// 或 http:// 格式)。

1.8.5.1 版本例外情况

为了使支持多版本的 Racket 更容易, package catalog 软件支持 version exception 。 版本例外允许软件包作者在使用特定版本的 Racket 安装特定的软件包时,指定其他的 package source 来使用。

例如,一个使用 Racket 6.0 特定功能的软件包可以为 Racket 5.3.6 提供一个 version exception , 在软件包的 GitHub 仓库中使用一个不同的分支或标签,或者一个不同的压缩包作为软件包的来源。 从 Racket 6.0 安装软件包的用户将使用该软件包的默认源, 而使用 Racket 5.3.5 的用户将从另一个分支、标签或压缩包中安装。

更多细节,请参阅 包目录协议 。
