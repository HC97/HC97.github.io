4 包的元数据

包的元数据,包括对其他包的依赖性,由包内的 "info.rkt" 模块报告。 这个模块必须用 info 语言实现。

例如,一个基本的 "info.rkt" 文件可以是

#lang info
(define version "1.0")
(define deps (list "base"))

以下 "info.rkt" 字段被包管理器使用:

当一个包是一个 single collection package 时,它的 "info.rkt" 文件可以指定额外的字段, 用于 Scribble 文档系统或其他工具。许多这些字段在 用 "info.rkt" 文件控制 raco setup 中被描述。

  • collection — 或者是 'multi 来实现 multi-collection package , 或者是一个字符串,或者是 'use-pkg-name 来实现一个 single-collection package 。 如果 collection 被定义为字符串,那么这个字符串将被用作包所实现的集合的名称。 如果 collection 被定义为 'use-pkg-name, 那么包的名称将被用作包的集合名称。

    请注意,省略 collection 或将其定义为 'use-pkg-name 意味着包的内容会随着包名的改变而改变。 一个包的内容通常应该与包的名字无关,因此对于一个 single-collection package 来说, 将 collection 定义为一个字符串是比较好的。

  • version — 一个 version 符串。一个包的默认 version 是 "0.0" 。

  • deps — 依赖关系的列表,每个依赖关系有以下形式之一:

    • 一个 package source 的字符串。

    • 格式的列表
      (list package-source-string
            keyword-and-spec ...)
      其中每个 keyword-and-spec 都有一个不同的关键字,形式为
        keyword-and-spec = '#:version version-string
        | '#:platform platform-spec
           
        platform-spec = string
        | symbol
        | regexp

      一个 version-string 指定了所需包的可接受 version 的下限。

      platform-spec 表示该依赖关系仅适用于平台,当 platforms-spec 是一个符号时, 它与 (system-type) 匹配,当 platform-spec 是一个字符串或正则表达式时, 它与 (path->string (system-library-subpath #f)) 匹配。参见 matching-platform? 。 例如,特定平台的二进制文件可以放在自己的包中,每个支持的平台都有一个单独的包和一个依赖关系。

    • 格式的列表

      (list package-source-string version-string)

      该列表已被废弃,等同于

      (list package-source-string '#:version version-string)

    deps 列表中的每个元素都决定了对 package 的依赖, 其名称是由 package source 推断出来的(也就是说,依赖是对包名的依赖,而不是对包源的依赖), 而 package source 指出了如果需要满足依赖,从哪里获得包。

    使用包名 "racket" 指定对 Racket 运行时系统的依赖,当依赖中包括一个版本时,这可能是有用的。 在大多数情况下,最好是指定对 "base" 的版本依赖性。

    参见 软件包的依赖性检查 。

  • build-deps — 和 deps 一样, 但是对于可以从 binary package 中省略的依赖性,它不包括源代码; 参见 源码、二进制和内置包 和 软件包的依赖性检查 。 build-deps 和 deps 列表是附加的, 而 raco pkg create 在转换 --binary 模式的软件包时,会剥去 build-deps 。

  • implies — 一个列表,其中每个元素是一个字符串或 'core 。 每个字符串指的是在 deps 中列出的软件包,并表示对当前软件包的依赖也算作对指定软件包的依赖; 例如,定义 gui 包是为了确保对 gui-lib 提供的所有库的访问, 所以 gui 的 "info.rkt" 文件在 implies 中列出了 "gui-lib" 。 列在 implies 列表中的包在更新时被特别对待:每当暗示包被更新时,暗示包就会自动更新。 特殊值 'core 是为适当的 base 包使用的,用来声明它是 Racket 核心库的代表。

  • update-implies — 一个字符串的列表。 每个字符串指的是在 deps 或 build-deps 中列出的软件包, 并表示每当暗示软件包被更新时,暗示的软件包会被自动更新。

  • setup-collects — 路径字符串和/或路径字符串的列表, 在软件包安装后通过 raco setup 来设置集合名称,或者用 'all 来表示需要设置所有集合。 默认情况下,只有软件包中的集合被设置(加上全局文档索引和链接的集合)。

  • license — license S-expression 指定软件包的许可证。 license S-expression 代表一个 SPDX license expression ，是一个带引号的数据:

      license-sexp = license-id
      | (license-id WITH exception-id)
      | (license-sexp AND license-sexp)
      | (license-sexp OR license-sexp)

    请看 下面 关于 license-id 和 + 操作符的进一步细节。

    其中:

    • a license-id 是 SPDX License List 中的一个简短的标识符, 例如 . LGPL-3.0-or-later, Apache-2.0 或 BSD-3-Clause; 以及

    • exception-id 是 SPDX License Exceptions 列表中的一个标识符, 例如 . Classpath-exception-2.0 。

    例如,Racket主发行版中的包将 license 定义为:
    (define license
      '(Apache-2.0 OR MIT))

    license S-expressions 的语法设计是这样的: (format "~s" license) 产生一个符合 SPDX Specification v2.2.2 的 Annex D 和 Annex E 的语法的字符串,该语法是以字符序列的形式指定的。

    如果使用 + 运算符, 它必须写成 license-id 的一部分,例如 . AFL-2.0+ 。 请注意, SPDX工作组已经取消了 (在 "允许许可证的后期版本" 下) 在 GNU 许可证中使用 + 操作符: 因此,人们写 AFL-2.0 或 AFL-2.0+ , 但写 GPL-3.0-only 或 GPL-3.0-or-later (而且 GPL-3.0 和 GPL-3.0+ 都不正确)。

  • distribution-preference — 要么是 'source,要么是 'built, 要么是 'binary,表示该软件包最适合的发布模式(但不保证它将以这种方式发布)。 如果软件包除了 "info.rkt" 文件之外没有 ".rkt" 或 ".scrbl" 文件, 并且有任何 ".so", ".dll", ".dylib" 或 ".framework" 文件, 没有这个定义就意味着 'binary ;否则就意味着 'built 。

  • package-content-state — 一个包含两个项目的列表; 第一个项目是 'binary, 'binary-lib 或 'built, 第二个项目是 #f 或一个字符串, 代表编译内容的 Racket 版本。 这一信息被 raco pkg install 或 raco pkg update 使用, 并带有 --source, --binary 或 --binary-lib ,以确保软件包的内容与要求的转换一致; 参见 源码、二进制和内置包 。 没有这个定义的情况下,与 (list 'source #f) 的处理方式相同。
