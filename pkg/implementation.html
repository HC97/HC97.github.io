12 包的安装和分发如何进行

包管理器建立在三个主要的基础设施上:

  • Racket 运行系统所支持的 Collection links files 。

    安装一个包会安装集合链接,因此可以找到该包的集合来编译和加载使用该包的模块。

    如果你使用 raco link -l 来查看已安装的链接,你会看到由包系统放置的链接。 很明显,你不应该直接修改这些链接。

  • raco setup 工具用于建立已安装的集合,包括其文档。

    raco setup 具驱动 raco make 将 Racket 源代码编译成字节码形式。 重新编译是由文件的时间戳、SHA-1哈希值和 "dep" 文件中记录的依赖关系的变化决定的。

    由于软件包的安装被反映为集合的链接,对集合的 raco setup 操作隐含着处理软件包。 raco setup 工具只在有限的范围内 “aware” 软件包: 它使用 path->pkg 这样的函数来打印软件包的进度信息, 并使用类似的软件包检查函数来连接模块和软件包,并检查实际的依赖性与声明的软件包依赖性。

  • racket-index 包,它扩展了 raco setup 来驱动 Scribble ,以获得基于集合的文档。

    racket-index 包实现了 raco make 的文档渲染类似功能, 检测文档声明的变化,并根据需要重新渲染文档,以获取交叉引用的变化。 racket-index 包还实现了 HTML 渲染文档的入口(即所有已安装文档的列表)、 HTML 搜索页面、本地重定向页面(服务器搜索链接到本地安装的文件)等特殊文档。

这三个层次中的每一个都适应 user 和 installation package scopes, 其中每个范围的细节往往不同。 一般来说, installation 范围内的引用必须实现为相对引用,这样就可以将 Racket 的原地安装转移到不同的位置。 同时, user 作用域中的引用可以在某种程度上直接引用到安装; 大多数引用是相对于集合或相对于安装的,所以包的内容可以在 user 作用域中构建, 然后组装成一个 built package 或 binary package ,以便在其他地方安装。

12.1 相互参照

像 path->collects-relative 和 path->main-collects-relative 这样的函数被用来将路径序列化为相对形式, 然后可以用 collects-relative->path 和 main-collects-relative->path 这样的函数来反序列化路径。 make-path->relative-string 函数支持相对于一组给定目录的序列化和反序列化。

raco make 生成的 ".dep" 文件中的依赖关系尽可能地使用集合的相对路径,对于集合中的依赖关系,应该总是可以的。 同样地,文档的交叉引用信息也尽可能使用与集合相关的路径。

在一个 collection links file 中,路径是相对于链接文件的目录而言的。 然后,当一个原地安装被移动时,全安装范围的链接就能发挥作用。

在整个安装范围的文档的交叉参考信息中,路径可以相对于安装的 "doc" 目录来存储。 对于在用户范围内构建的文档,构建的文档中的交叉引用信息通过 render% 的 root-path 初始化参数相对于文档的目录进行记录; 交叉引用信息可以被解包到不同的目的地, 其中使用时间的路径被提供给 load-xref 和/或 make-data+root 结构的 #:root 参数。

12.2 独立的文档渲染

与模块引用不同,模块引用必须不产生引用循环,而文档可以有引用循环。 文档也往往没有代码那么紧凑,虽然我们试图尽量减少代码中的模块依赖,但文档应该自由地引用任何其他相关的文档。 最后,文档的引用没有模块的引用那么静态; 例如,一个文档通过引用 racket/base 来引用 cons , 而文档系统必须弄清楚哪个其他文档定义了 cons 。 一个天真的文档渲染实现会加载所有的文档来渲染任何一个文档,这在时间和空间上都是非常昂贵的。

Scribble 通过对交叉引用信息进行汇总和解汇总来支持单独的文档渲染。 raco setup 的 racket-index 扩展将一个文档的信息存储在 ".sxref" 文件中。 有些文档,比如 reference, 导出了大量的交叉引用信息, 所以 raco setup 将一个文档导出的交叉引用信息分解成多个 "out‹n›.sxref" 文件。 关于 “imported” 的交叉引用信息–即上次建立文档时使用的交叉引用–被保存在 "in.sxref" 文件中。 最后,为了检测哪些 "out‹n›.sxref" 文件在构建文档时需要被加载, 交叉引用键与 "out‹n›.sxref" 文件的映射被保存在一个 SQLite 数据库中, "docindex.sqlite" 。 对 "out‹n›.sxref" 文件的懒惰加载是通过 load-xref 的 #:demand-source 参数来实现的, 它提供了一个函数来查询 "docindex.sqlite" 以将一个键映射到一个交叉引用文件。

交叉引用文件中的各种路径以不同的相对路径约定被存储。 安装中的 "docindex.sqlite" 文件可以在不修改安装的情况下被移动。 用户范围包中的 "docindex.sqlite" 文件是不可移植的(在任何包之外), 而 "in.sxref" 和 "out‹n›.sxref" 文件可以原样包含在 binary package 和 built package 中。

12.3 跨文档的 HTML 引用

为 Scribble 文档生成的 HTML 需要相对链接。 然而,与 Racket 代码不加掩饰的数据不同,没有办法把相对于各种安装目录的路径变成浏览器能理解的路径–至少,不能只用 HTML。 因此,为文档生成的 HTML 依赖于 JavaScript 来重写某些引用,并通过服务器的后备路径使文档也能作为普通 HTML 使用。

单个文档内的引用在 HTML 中被呈现为相对链接。 从一个文档到另一个文档的引用被呈现为一个查询,例如, http://docs.racket-lang.org/ 。 然而,每个文档也引用 "local-redirect.js" 和(在用户特定集合的文档中) "local-user-redirect.js" 。 当本地目标可用时,这些 JavaScript 片段动态地将查询引用重写为直接的文件系统引用,分别指向整个安装系统和用户特定的目标。 当本地目标不可用时,查询链接将不被修改而通过服务器。

"local-redirect.js" 和 "local-user-redirect.js" 文件将文档-目录名映射到特定路径。 大多数查询引用包含一个文档-目录名称和该目录内的相对路径,在这种情况下,从目录名称到路径的映射就足够了。 间接链接,比如那些由 (seclink #:indirect? #t ...) 创建的间接链接,嵌入了一个交叉引用键, 因此 "local-redirect.js" 和 "local-user-redirect.js" 也必须嵌入交叉引用数据库的一个部分。 (这个数据库的副本被分解成多个文件,每个文件都是按需加载的)。 "local-redirect.js" 和 "local-user-redirect.js" 文件是作为特殊的 "local-redirect" 文件的一部分生成的,该文件由 racket-index 包实现。

通过 "local-redirect.js" 和 "local-user-redirect.js" 的转接, 将相对链接的问题减少到引用这两个文件的问题上。 它们在用户特定的文件构建中被引用为绝对路径。 为了创建一个包含文档的 built package 或 binary package , 每个 ".html" 文件都必须被修改以移除绝对路径,然后每个 ".html" 文件都必须在安装时再次修改, 将目标安装的路径放入路径中。

racket-index 包对 raco setup 的扩展来构建 Scribble 文档, 使用来自 scribble/html-render 的 render-mixin 的 set-external-tag-path 方法, 将这些间接性放在了适当的位置。 http://docs.racket-lang.org/ 路径不是硬连接的, 而是基于 get-doc-search-url 报告的安装配置。 这种配置可以在构建 Racket 发行版时确定;PLT 的主要发行版将 URL 设置为特定版本的网站, 这样即使在新的 Racket 版本发布后,搜索也能正常进行,而快照同样将 URL 设置为特定快照网站。

12.4 HTML 文档搜索和起始页

racket-index 包提供了一个特殊的文档来实现安装 HTML 文档的初始页面。 该文档使用 "info.rkt" 文件 scribblings 标志来依赖所有文档的标题。

racket-index 包还提供了一个特殊的文档来实现搜索。 该搜索文档使用 JavaScript 和交叉引用数据库的副本(类似于 "local-redirect.js") 来实现交互式搜索。

如果有任何用户特定的集合被安装,那么 racket-index 就会生成两份开始和搜索文档:一份用于安装,另一份专门针对用户。 用户页面是安装页面的延伸。 用户专用的搜索页面读取整个安装系统的搜索页面的数据库,这既避免了重复,又使搜索能够接收到安装系统的任何新增内容, 而不需要重新建立用户专用的搜索页面。 相比之下,用户专用的启动页必须在任何安装范围内的新增内容后重建,以获取新增内容。

当 DrRacket 或 raco docs 在浏览器中打开文档时,它会打开用户专用的开始或搜索页面(如果存在)。 如果这些页面因任何原因被访问,浏览器本地存储或(如果不支持本地存储)安装一个 cookie。 本地存储键或 cookie 被命名为 “PLT_Root.‹version›,” 它指向用户特定文档的位置。 此后,使用 cookie 的本地值,在任何文档页面中搜索或进入 “top” 页面,都会进入用户特定的页面,甚至从整个安装页面。
