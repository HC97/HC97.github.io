2 包的概念

一个 package 是一些 collections 中的一组模块。 使用 Racket 包管理器安装的模块和其他模块一样是 require 。 例如,如果软件包 tic-tac-toe 在 "data" 集合中包含 "matrix.rkt" 模块, 那么在安装了 tic-tac-toe 之后、

(require data/matrix)

导入该模块。在 require 中没有提到包的名字,因为包是管理库集合的一种方式,而不是引用它们的一种方式。 然而,一个包实现的集合的名称与包的名称相同是很常见的–在这种情况下, require 可能看起来是在引用一个 package, 但它实际上是在引用一个由 package 提供的 collection 。

每个 package 都有相关的 package metadata:

  • package name — 由 a through z, A through Z, 0 through 9, _, and - 这些字符组成的字符串。

  • checksum — 一个字符串,用于识别一个包的不同版本。 当一个软件包的 checksum 发生变化时,无论其 version 是否发生变化,它都可以被更新。 校验和通常可以被计算成软件包内容的 SHA1(见 openssl/sha1)。

  • version — 一个 ‹maj›.‹min›, ‹maj›.‹min›.‹sub› 或 ‹maj›.‹min›.‹sub›.‹rel› 的字符串, 对版本号的约束与 version/utils 一致,并迫使版本号采用规范的形式。 例如,一个可能的版本字符串 "4.3.0" 必须写成 "4.3", "4.3.1.0" 必须写成 "4.3.1", 而 "4" 必须写成 "4.0" 。 其中 ‹maj›, ‹min›, ‹sub› 和 ‹rel› 都是自然数的典型十进制表示, ‹rel› 不是 0, ‹sub› 不是 0 ，除非提供 ‹rel› , ‹min› 不超过2位, 而 ‹sub› 和 ‹rel› 不超过3位。 版本旨在反映软件包的可用功能,不应该与 checksum 所表示的软件包的不同版本相混淆。

  • 依赖关系列表 — 一个要同时安装的软件包列表,可以选择对每个软件包的 version 进行下限设定。

一个 package 通常由一个与该包同名的目录来表示。 checksum 通常是隐含的。 包目录可以包含一个名为 "info.rkt" 的文件来声明其他元数据(见 包的元数据)。

2.1 单个集合和多个集合的包

一个 package 可以是一个 single-collection package 或一个 multi-collection package:

  • 一个 single-collection package 的目录是 collection 目录的两倍。 默认情况下,包的名字也是作为集合的名字,但是如果包有一个 "info.rkt" 文件, 将 collection 定义为一个字符串,那么这个字符串就被用作包的集合的名字。

  • 一个 multi-collection package 的目录包含子目录, 每个子目录都是由包提供的 collection (其中目录名被用作集合名)。 一个 multi-collection package 必须有一个 "info.rkt" 文件, 将 collection 定义为 'multi 。

2.2 包的来源

一个 package source 标识了一个 package 的代表。 每个包源类型都有不同的方式来存储 checksum 并提供包的内容 (通常有 single-collection package 和 multi-collection package 的变体)。

package source 的类型有:

  • 命名档案的本地文件路径(作为普通路径或 file:// URL) — 包的名称是档案文件的基本名称。 档案 "f.‹ext›" 的 checksum 由文件 "f.‹ext›.CHECKSUM" 给出。 有效的存档格式是(目前) ".zip", ".tar", ".tgz", ".tar.gz" 和 ".plt" 。 除了 type 查询(会影响到下面描述的推理)外, file:// URL 的任何查询或片段部分都会被忽略。

    例如, "~/tic-tac-toe.zip" 是一个归档包源, 它的 checksum 将在 "~/tic-tac-toe.zip.CHECKSUM" 内。

    归档文件表示包的内容类似于目录,但如果归档文件的内容包含在一个单一的顶级目录中, 那么该目录的内容(而不是整个归档文件的内容)就被用作包的内容。 ".plt" 格式既不能容纳额外的目录层,也不能容纳 single-collection package 的表示。

    只有当包源的后缀与有效的归档格式相匹配,并且以 file:// 开头或不以 :// 后面的字母字符开头时, 才能推断出它是指归档文件。 如果软件包来源以 file:// 开头,它必须是一个没有 type 查询的 URL, 或者 type 查询值为 file 。 推断出的包名是不含后缀的文件名。

    Changed in version 6.0.1.12: 改变了对包含顶层目录内所有内容的档案的处理。
    Changed in version 6.1.1.5: 改变了 file:// 的解析,以接受一个一般的 URL, 识别一个 type 查询,并忽略任何其他查询或片段。

  • 一个本地目录(作为一个普通路径或 file:// URL) — 包的名称就是目录的名称。 checksum 是不存在的。 除了 type 查询(会影响下面描述的推理)外, file:// URL 的任何查询或片段部分都会被忽略。

    例如, "~/tic-tac-toe/" 是一个目录包源。

    只有当一个包源没有文件存档的后缀,不符合包名的语法, 并且以 file:// 开头或不以 :// 后面的字母字符开头时,才会被推断为指一个目录。 在包源以 file:// 开头的情况下,它必须是一个没有 type 查询的 URL, 或者 type 查询值为 dir, link 或 static-link 。推断出来的包名是目录名。

    当软件包的来源是 file:// URL ,其 type 查询值为 link 或 static-link 时, 那么该软件包将作为目录链接安装,这与向 raco pkg install 或 raco pkg update 提供 --link 或 --static-link 的情况相同。

    Changed in version 6.1.1.5: 改变了 file:// 的解析,接受一般的 URL,识别 type 查询,并忽略任何其他查询或片段。

  • 命名一个存档的远程 URL — 这种类型遵循与本地文件路径相同的规则,但存档和 checksum 文件是通过 HTTP(S) 访问。

    例如, "http://game.com/tic-tac-toe.zip" 是一个远程 URL 包源, 其 checksum 在 "http://game.com/tic-tac-toe.zip.CHECKSUM" 找到。

    一个包源只有在 http:// 或 https:// 开始时才被推断为一个 URL, 当 URL 以一个可以推断为文件存档的路径元素结束时,它被推断为一个文件 URL。 推断出的包名来自于 URL 的文件名,与文件包源的方式相同。

  • 命名一个目录的远程 UR — 远程目录必须包含一个名为 "MANIFEST" 的文件,其中列出了所有的应急文件。 这些文件被下载到本地目录,然后遵循本地目录路径的规则。 然而,如果远程目录包含一个名为 ".CHECKSUM" 的文件,那么它将被用来确定 checksum 。

    例如, "http://game.com/tic-tac-toe/" 是一个目录 URL 包源, 它的 checksum 在 "http://game.com/tic-tac-toe/.CHECKSUM" 找到。

    一个包源被推断为目录或文件的 URL 是一样的,当它不以具有归档文件后缀或 ".git" 后缀的路径元素结束时,它被视为目录 URL。 推断出的包名就是目录名。

    Changed in version 6.1.1.1: 增加了 ".git" 后缀的特殊大小写。

  • 命名 Git 仓库的远程 URL — 这种 URL 的格式是:

    ‹scheme›://‹host›/.../‹repo›[.git][/][?path=‹path›][#‹rev›]

    其中 ‹scheme› 是 git, http, https, git+http 或 git+https, 除非 ‹scheme› 是 git 而 ‹host› 是 github.com (它被更具体地视为 GitHub 的引用)。 ‹path› 可以包含多个 / 分隔的元素来形成仓库内的路径,默认为空路径。 ‹rev› 可以是一个分支、标签或提交,它默认使用服务器报告的默认分支。

    由于 Git 协议的特性,当 ‹rev› 指的是一个分支或标签(即使它被写成一个提交)时,档案的访问效率可能更高。 在这些情况下,通常可以在不下载不相关历史的情况下获得内容。

    例如, "http://bitbucket.org/game/tic-tac-toe#main" 是一个 Git 包的源代码。

    在 ‹rev› 处签出仓库提供了包的内容, ‹scheme› 决定了用于克隆仓库的协议。 如果 ‹rev› 是一个分支或标签,软件包的 checksum 的哈希值, 否则 ‹rev› 就是识别 ‹rev› 本身就是 checksum 。

    当一个包的来源以 git:// 开头且主机不是 github.com 时,它被推断为是一个 Git 引用。 当一个包源以 http:// 或 https:// 开头, 且最后一个非空的路径元素以 .git 结尾时,也会被推断为 Git 参考; 如果该源被指定为 Git 参考,则会添加一个 .git 后缀。 最后,当一个包源以 git+https:// 或 git+http:// 开头时,将被推断为 Git 引用, 在这种情况下,路径中不需要 .git 后缀来指定该源为 Git 引用(也不会隐含地添加 .git 后缀)。 推断出的包名是 ‹path› 的最后一个元素,如果它不是空的,否则推断出的名字是 ‹repo› 。

    Changed in version 6.1.1.1: 增加了对 Git 仓库的支持。
    Changed in version 8.0.0.13: 增加了 git+https:// 和 git+http:// 支持。

  • 命名 GitHub 仓库的远程 URL — 此类 URL 的格式与以 git:// 开始的 Git 仓库引用相同, 但以 github.com 作为主机:

    git://github.com/‹user›/‹repo›[.git][/][?path=‹path›][#‹rev›]

    例如, "git://github.com/game/tic-tac-toe#main" 是一个 GitHub 软件包源。

    一个以 git:// 开头的 Github 仓库源, 获得的内容与 github.com 不被特殊处理时的访问内容相同。 由于历史原因,特别是结合 PLT_USE_GITHUB_API ,这种特殊处理被保留下来。

    为了向后兼容,也支持旧的格式:

    github://github.com/‹user›/‹repo›/‹rev›[/‹path›]

    仓库的 zip 格式的存档(由 GitHub 为任何提交生成)被用作远程 URL 存档路径。 如果 ‹rev› 是一个分支或标签, checksum 就是识别 ‹rev› 的哈希值,否则 ‹rev› 本身就是 checksum 。

    当一个包源以 git://github.com/ 或 github:// 开头时,将被推断为 GitHub 引用; 以其他方式被指定为 GitHub 引用的包源将自动以 git://github.com/ 为前缀。 如果 ‹path› 非空,推断出的软件包名称是 ‹path› 的最后一个元素,否则推断出的名称是 ‹repo› 。

    如果设置了 PLT_USE_GITHUB_API 环境变量,则使用 GitHub API 协议而不是 Git 协议获取 GitHub 包。

    Changed in version 6.3: 改变了对 GitHub 源的处理,默认使用 Git 协议。

  • package name — 一个 package catalog 被查阅以确定包的来源和 checksum 。

    例如, tic-tac-toe 是一个包的名字,可以作为包的来源。

    当一个包源符合包名的语法时,它就被推断为是一个包名,这意味着它只有 a through z, A through Z, 0 through 9, _, and - 等字符。

2.3 包目录

package catalog 是一个服务器或数据库,它将软件包名称转换为其他软件包来源。 一个 package catalog 由一个代表 URL 的字符串来识别,其中 http:// 或 https:// URL 表示一个远程服务器,而 file:// URL 表示一个 SQLite 数据库或目录树形式的本地目录。

PLT 支持两个默认启用的 package catalog 服务器: https://pkgs.racket-lang.org ,用于新的软件包; http://planet-compats.racket-lang.org ,用于自动生成老的 PLaneT 软件包的软件包。 任何人都可以托管一个 package catalog, 任何提供文件服务的 HTTP 主机都可以作为一个基本的 package catalog 服务器。 关于如何从目录中提取软件包信息的信息,请参见 包目录协议 。

2.4 显式安装与自动安装

当一个软件包被安装时,它的原始安装源会被记录下来,以及该安装是否是一个 automatic installation 。 automatic installation 是指因为它是其他软件包的依赖关系而被安装的(而不是由用户明确安装的)。

2.5 包的冲突

如果两个软件包包含相同的模块,它们就处于 conflict 之中。 例如,如果软件包 tic-tac-toe 包含模块文件 "data/matrix.rkt" , 而软件包 factory-optimize 包含模块文件 "data/matrix.rkt", 那么 tic-tac-toe 和 factory-optimize 就发生了冲突。

如果一个包包含一个属于 Racket 基本实现的模块文件,也可能与 Racket 本身发生冲突。 例如,任何包含 "racket/list.rkt" 的包都与 Racket 冲突。

就冲突而言,模块是以 ".rkt", ".ss" 或 ".scrbl" 结尾的文件, 但名为 "info.rkt" 的文件除外。

2.6 包的更新

包 A 是包 B 的 package update ,如果 (1) B 已经安装、 (2) A 和 B 有相同的名字,并且 (3) A 的 checksum 和与 B 的不同。 一个 single-collection package 可以是一个 multi-collection package 的 package update , 反之亦然。

请注意,在确定 package update 时不考虑包的 version , 尽管包的 version 变化(无论哪个方向)意味着 checksum 的变化, 因为校验和是从包的源头计算出来的,而指定版本的元数据是源头的一部分。

2.7 包范围

package scope 决定了包安装、更新等对不同用户和 Racket 安装的影响。 默认的 package scope 可以被配置,但通常是 user, 这使得操作针对当前用户和安装的名称/版本(在 get-installation-name 的意义上)。 installation 范围意味着软件包操作会影响Racket安装的所有用户。

一个目录路径可以被用作 package scope, 在这种情况下,包操作会影响目录中的包安装集合。 一个安装可以被配置为在其搜索路径中包括该目录的安装包(见 安装配置和搜索路径)。 当一个目录路径被用作 package scope 时,依赖性检查等操作将使用配置的搜索路径中的所有路径, 从被设计为 package scope 的路径开始;如果指定的路径不在配置的搜索路径中,那么该目录本身就被用作搜索路径。

冲突检查不允许在不同的作用域中安装相同或冲突的包,但如果强制进行这样的配置, 则首先在具有 user package scope 的包中找到集合。 然后按照配置好的顺序进行搜索,其中 installation package scope 通常在其他目录 package scopes 之前。

默认的包作用域是通过首先检查 'user 作用域的配置,然后检查更广泛的作用域如 'installation 中的配置来确定的。 如果默认包的作用域在任何作用域中都没有被配置,那么它就默认为 'user 。
