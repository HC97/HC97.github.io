10 FAQ

本节将回答关于软件包管理器的一些常见问题。

10.1 软件包的安装是否与 Racket 的版本有关?

大多数 Racket 的安装配置为:为特定的用户和特定的 Racket 版本安装一个包。 也就是说, package scope 是针对用户和版本的。 更确切地说,它是针对用户和安装名称的,其中安装名称通常是指 Racket 版本。

你可以用 raco pkg config -i --set default-scope installation 来改变默认的 package scope (针对特定的 Racket 安装),在这种情况下,软件包操作适用于 Racket 安装中的所有用户。 你也可以在特定的 raco pkg 命令中使用 -i 或 --installation 标志, 而不是为所有使用 raco pkg 的人改变默认范围。 请注意,一个安装范围的软件包并不完全是针对版本的, 因为如果一个安装的版本对应于一个定期更新和重建的源代码检查,那么它的版本就会改变。

如果你改变了默认的 package scope, 你可以在特定的 raco pkg 命令中使用 -u 或 --user 标志来执行针对用户的 package scope 的命令。

10.2 包安装在哪里以及如何安装?

用户专用和 Racket 版本专用的软件包在 (find-user-pkgs-dir), 而安装范围的软件包在 (find-pkgs-dir) 。 它们被作为集合(对于单集合包)或集合根(对于多集合包)用 raco link 连接。

10.3 特定用户的包和安装范围的 package scopes 是如何进行冲突检查的?

特定于用户的包会针对安装范围的包检查包名冲突和提供的模块冲突。 安装范围内的包只针对用户特定的包检查所提供的模块的冲突。

请注意,一个用户的无冲突的安装范围内的修改可能会给另一个用户带来冲突。

10.4 当我用错误修复等方法更新软件包时,我需要改变软件包的版本吗?

如果你对一个软件包有新的代码,那么它应该有一个新的 checksum 。 当搜索软件包更新时,安装的软件包的校验和与源码的校验和进行比较,如果它们不同,那么源码就会被重新安装。 这使得代码的变化可以被分发。 你不需要声明一个更新的版本号,除了允许其他软件包实现者表明对特定功能的依赖性 (其中一个错误修复可能被认为是一个功能,但通常没有必要这样考虑)。

10.5 如果一个包的接口发生了变化,而我需要一个 old 的版本,我怎样才能指定我所依赖的版本?

在这种情况下,包的作者发布了一个向后不兼容的包的版本。 包管理器没有提供任何帮助来处理这种情况(当然,除了不安装 "更新" 之外)。 因此,软件包作者不应该对软件包做向后不兼容的修改。 相反,他们应该发布一个有新名字的新包。 例如, libgtk 包可能变成 libgtk2 。 这些软件包也应该被设计成不会相互冲突。

10.6 我怎样才能将我的安装固定在一组特定的软件包实现或 checksums 上?

软件包只有在你运行 raco pkg update 这样的工具时才会被更新,所以软件包永远不会被隐含地更新。 此外,你可以快照一组软件包档案,并从这些档案中进行安装,而不是依赖通过 package catalog 进行的软件包名称解析。

如果你想控制软件包名称的解析(包括特定的 checksum ), 但没有必要保留所有软件包代码的副本(假设旧的 checksum 仍然可用,比如通过 GitHub), 你可以通过使用 raco pkg catalog-copy 创建 package name 到 package source 映射的快照。 比如说

  raco pkg catalog-copy --from-config /home/joe/snapshot.sqlite

创建一个当前软件包名称解析的快照 "/home/joe/snapshot.sqlite" ,然后

  raco pkg config --set catalogs file:///home/joe/snapshot.sqlite

将所有软件包名称解析引向该快照。你可以通过编辑快照来配置特定包名的解析。

你还可以进一步使用

  raco pkg catalog-archive --from-config /home/joe/snapshot/

更进一步,它不仅可以获取目录的快照,还可以获取所有软件包来源的快照(这样你就不会依赖原始来源)。

10.7 我怎样才能安装一个没有文档的软件包?

如果软件包是以 built package 的形式提供的, 那么你可以使用 raco pkg install --binary-lib 来在安装前剥离软件包的源代码、测试和文档。

https://pkgs.racket-lang.org/ 上的主要软件包目录指的是 source packages, 而不是 built packages 。 其他目录提供的是 built packages:

  • 对于与 Racket 发行版相关的包(比如当前的发行版), 主持发行版的网站通常也会主持 built packages—但只针对已经包含在发行版中的包。

  • https://pkgs.racket-lang.org/ 包的构建变体目前由目录 https://pkg-build.racket-lang.org/server/built/catalog/ 提供(仅针对当前版本)。 更多信息请参见 the package-build service 。

一些软件包已经在源码层面被分割成独立的库、测试和文档包。 例如, net-lib 提供了诸如 net/cookie 等模块, 但没有文档,而 net-doc 提供了文档, net-test 提供了测试。 net 包依赖于 net-lib 和 net-doc, 并且它意味着 net-lib, 所以你可以在一个最小的 Racket 发行版中安装 net 来获得带有文档的库(以及大量支持文档的附加包), 或者安装 net-lib 来获得仅仅是库。

如果你开发的软件包被广泛使用,或者在一个受限的安装环境中特别有用, 那么将你的软件包拆分成 -lib, -doc 和 -test 三个部分可能是值得的。 最有可能的是,你应该把软件包放在一起,放在一个源代码库中,并使用元数据, 如 implies 和 update-implies (见 包的元数据),这样软件包就能同步更新。

10.8 为什么包管理器与 PLaneT 如此不同 ?

PLaneT 和这个软件包管理器之间有两个基本的区别。

首先是 PLaneT 使用 "内部链接",而目前的软件包管理器使用 "外部链接"。 例如,一个单独的模块在 require 语句中直接需要一个 PLaneT 包:

(require (planet game/tic-tac-toe/data/matrix))

而使用包管理器,该模块将简单地要求感兴趣的模块:

(require data/matrix)

并将依赖于外部系统是否安装了 tic-tac-toe 包。

这个变化是好的,因为它使模块的起源更加灵活–这样代码就可以从核心中迁移出来,包可以很容易地被拆分、合并,或者被其他作者接管,等等。

这种变化是不好的,因为它使你的程序的意义依赖于系统的状态。

第二个主要的区别是, PLaneT 致力于保证软件包之间不会发生冲突,因此,同一软件包的任何数量的主要和次要版本都可以同时安装和使用。 软件包管理器不同意这一承诺,所以软件包作者和用户必须注意潜在的冲突,并围绕它们进行规划。

这个变化是好的,因为它更简单,降低了维护的负担(只要大多数软件包不冲突。)

这个变化是不好的,因为用户必须围绕潜在的冲突进行规划。

一般来说,软件包管理器的目标是成为一个较低级别的系统,更像操作系统使用的软件包系统。 PLaneT 的目标并不坏,但我们认为它们的需求并不频繁, 而且像 PLaneT 这样的系统可以更容易地建立在软件包管理器之上,而不是相反。

特别是,我们为减轻这些变化的弊端而制定的计划在 短期计划 中有所记载。
