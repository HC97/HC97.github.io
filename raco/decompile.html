7 raco decompile: 对字节码进行反编译

raco decompile 命令接收字节码文件(通常文件扩展名为 ".zo") ) 或带有相关字节码文件的源文件(通常由 raco make 创建)的路径,并将字节码文件的内容转换为 Racket 代码的近似值。 如果 "字节码" 文件包含机器码,如 Racket 的 CS 变体, 那么它就不能被转换为 Racket 的近似值,但安装 "disassemble" 包可以实现机器码的反汇编。 反编译主要用于检查编译器对源程序的转换和优化。

raco decompile 命令接受以下命令行标志:

  • --force — 跳过对给定文件的路径和相关的 ".zo" 文件(如果有的话)进行修改日期比较

  • -n ‹n› 或 --columns ‹n› — 为具有 ‹n› 列的显示格式输出

  • --linklet — 只对 linklets 进行反编译, 而不是将小链接解码为近似的Racket module 形式

  • --no-disassemble — 以字节串的形式显示机器代码,而不是尝试反汇编。

  • --partial-fasl — 保留更多字节码文件的原始结构,而不是专注于过程体。

Changed in version 1.8: 添加了 --no-disassemble 。
Changed in version 1.9: 添加了 --partial-fasl 。

7.1 Racket CS 反编译

Racket CS 字节码的反编译主要显示了围绕过程的机器码实现的模块结构。

  • #%machine-code 形式对应的是未被反汇编的机器码,其中机器码在一个字节串中。

  • #%assembly-code 形式对应于被分解的机器代码,其中汇编代码显示为一串字符串。

  • #%interpret 形式对应于一个大程序的编译形式,其中只有较小的嵌套程序被编译成机器码。

7.2 Racket BC 反编译

Racket BC 字节码的结构与 Racket 的核心语言足够接近,所以它更多的时候可以被转换为 Racket 代码的近似形式。 在可以转换回来的程度上,反编译后的代码中的许多形式与以往的含义相同, 如 module, define 和 lambda 。 其他的形式和转换是针对字节码的渲染的,它们反映了特定的执行模式:

  • 顶层变量、模块内定义的变量以及从其他模块导入的变量都以 _ 为前缀, 这有助于揭示本地变量与其他变量的使用之间的区别。 此外,从其他模块导入的变量有一个以 @ 开头的后缀,表示源模块。 最后,导入的具有恒定性的变量有一个中缀: :c 表示所有实例中的恒定形状, :f 表示初始化后的固定值, :p 表示一个过程, :P 表示一个返回时保留延续标记的过程, :t 表示一个结构类型, :mk 表示一个结构构造器, :? 表示一个结构谓词, :ref 表示一个结构访问器,或者 :set! 表示一个结构突变器。

    非本地变量总是通过一个隐含的 #%globals 或 #%modvars 变量间接访问, 该变量位于值堆栈中(否则包含本地变量)。 当编译器无法证明变量在访问前会被定义时,变量的访问会被进一步用 #%checked 包裹。

    对核心基元的使用没有前面的 _, 而且它们从不被 #%checked 包裹。

  • 对局部变量的访问可以用 #%sfs-clear 来包装,这表明存放变量的堆栈位置将被清除, 以防止变量的值被垃圾收集器保留。 名字以 unused 开头的变量实际上从未存储在堆栈中,所以它们从来没有 #%sfs-clear 注释。 (字节码编译器通常会消除这样的绑定,但有时它不能这样做, 因为它不能证明右手边产生了正确数量的值,或者编译器发现变量是未使用的时候已经太晚了。)

    使用 #%box, #%unbox 和 #%set-boxes! (可同时对多个盒子起作用),可改变的变量被转换为明确的盒子值。 一个 set!-rec-values 操作构建了相互递归的闭包,并同时更新了绑定闭包的相应变量栈位置。 在局部变量被闭包捕获之前,总是对其使用 set!, set!-values 或 set!-rec-values 形式;这种排序反映了闭包如何在变量栈位置捕获值,而不是堆栈位置。

  • 在 lambda 形式中,如果由 lambda 产生的过程有一个名称 (可以通过 object-name 访问)和/或源位置信息,那么它将在过程主体的开始部分显示为一个带引号的常量。 之后,如果 lambda形式从其上下文中捕获了任何绑定,这些绑定也会显示为一个带引号的常量。 当闭包被调用时,这两个常量都不对应于计算,尽管当 lambda 形式本身被评估时,捕获的绑定列表对应于闭包分配。

    一个没有绑定的 lambda 形式被 #%closed 加上一个绑定在闭包上的标识符所包裹。 绑定的范围涵盖了整个反编译的输出,它可以在程序的其他部分直接引用; 绑定对应于一个共享的常量闭包值,它甚至可以包含对自身或其他常量闭包的循环引用。

  • 一个表单 (#%apply-values proc expr) 等同于 (call-with-values (lambda () expr) proc), 但是运行时系统会避免为 expr 分配一个闭包。 类似地, #%call-with-immediate-continuation-mark 的调用等同于 call-with-immediate-continuation-mark 的调用,但避免了闭包分配。

  • 一个 define-values 形式的表达式可能有 (begin '%%inline-variant%% expr1 expr2) , 在这种情况下, expr2 是正常的结果,但是 expr1 可以被内联,以便从其他模块调用该定义。 没有 '%%inline-variant%% 的函数定义,永远不会被跨模块内联。

  • 已知具有特定类型的函数参数和局部绑定,其名称都嵌入了已知的类型。 例如,一个参数可能有一个以 argflonum 开头的名字, 或者一个局部绑定可能有一个以 flonum 开头的名字,表示一个 flonum 值。

7.4 解析字节码的 API

 (require compiler/zo-parse) package: zo-lib

除了 zo-parse 之外, compiler/zo-parse 模块还重新输出了compiler/zo-structs 。

procedure

(zo-parse [in]) → (or/c linkl-directory? linkl-bundle?)

  in : input-port? = (current-input-port)
解析一个包含字节码的端口(通常是打开一个 ".zo" 文件的结果)。 请注意,用于表示字节码的结构类型在不同的 Racket 版本中会经常变化。

解析后的字节码以 linkl-directory 或 linkl-bundle 结构返回–后者仅用于编译不包含子模块的模块。

除了linkl bundle 或目录结构, zo-parse 的结果还包含依赖于编译字节码的机器的linklet:

  • 对于一个与机器无关的字节码文件,一个 linklet 被表示为 faslable-correlated-linklet 。

  • 对于 Racket CS 字节码文件,linklet是不透明的,因为它主要是机器代码, 但 decompile 可以提取一些信息,并有可能反汇编机器代码。

  • 对于 Racket BC 字节码来说,字节码可以被解析成如下所述的结构。

本节的其余部分是专门针对 BC 字节码的。

在 linklet 中,表达式的字节码表示更接近于 S 表达式,而不是传统的、扁平的控制字符串。 例如,一个 if 表单由一个 branch 结构表示,它有三个字段: 一个测试表达式、一个 "那么 "表达式和一个 "其他 "表达式。 类似地,一个函数调用由一个 application 结构表示,它有一个参数表达式的列表。

局部变量或中间值(如函数调用的参数)的存储是以堆栈的形式明确指定的。 例如, application 结构的执行在堆栈上为每个参数结果保留空间。 同样地,当一个 let-one 结构(对于一个简单的 let )被执行时, 通过评估右侧表达式得到的值被推到堆栈上,然后评估主体。 本地变量总是以当前堆栈位置的偏移量来访问。 当一个函数被调用时,它的参数被传递到栈上。 闭包是通过将值从堆栈转移到一个平坦的闭包记录中来创建的,当闭包被应用时, 保存的值在堆栈中被恢复(尽管可能是以不同的顺序,并且可能是以比它们被捕获时更紧凑的布局)。

当一个子表达式产生一个值时,堆栈指针被恢复到评估子表达式之前的位置。 例如,评估 let-one 结构的右侧大小可能会暂时将值推到堆栈上, 但在推送结果值之前,堆栈会恢复到 let-one 之前的位置,并继续进行正文。 此外,尾部调用将堆栈指针重置到包围函数的参数之后的位置,然后尾部调用继续将尾部调用的函数的参数推入堆栈。

全局变量和模块级变量的值不直接放在堆栈中,而是存储在 "桶" 中,一个可访问的桶的数组被保存在堆栈中。 当一个闭包体需要访问全局变量时,闭包体会以捕获和恢复局部变量的相同方式捕获并随后恢复该桶阵列。 可变的局部变量与全局变量类似地被装箱,但单个装箱可以从栈和闭包中引用。

7.6.1 前缀

struct

(struct linkl-directory zo (table)
    #:extra-constructor-name make-linkl-directory
    #:prefab)
  table : (hash/c (listof symbol?) linkl-bundle?)

struct

(struct linkl-bundle zo (table)
    #:extra-constructor-name make-linkl-bundle
    #:prefab)
  table : (hash/c (or/c symbol? fixnum?) (or linkl? any/c))
包裹着编译后的代码。

模块和顶层编译会产生一个或多个小链接,代表在特定阶段的独立评估。 即使是一个顶层表达式或只有运行时代码的模块也会产生多个小链接,以实现元数据和语法数据。 一个没有子模块的模块直接由 linkl-bundle 表示,而任何其他的编译形式则由 linkl-directory 表示。

一个 linkl-bundle 将一个整数映射到代表形式的 linklet,以便在整数指示阶段进行评估。 符号被映射到元数据上,比如一个模块的编译名称或一个实现字面语法对象的 linklet。 一个linklet 目录通常将 '() 映射到一个模块的主要 linklet 包或一个单一的顶层形式; 然而,对于一个对应于顶层形式序列的linklet目录,没有 "主要" linklet 包,整数的符号形式被用来排列 linkets。

对于一个有子模块的模块,linklet 目录将子模块的路径(作为符号列表)映射到相应子模块的 linklet 包。

只有当源字节码文件为 Racket BC 时,单个的 linkl 才被表示为linkl。 一个 CS 字节码的 linklet 将由一个不透明的 linklet 来表示 (在 racket/linklet 的 linklet? 意义上)。 一个独立于机器的 linklet 被表示为一个 faslable-correlated-linklet 结构。

struct

(struct linkl zo (name
    importss
    import-shapess
    exports
    internals
    lifts
    source-names
    body
    max-let-depth
    need-instance-access?)
    #:extra-constructor-name make-linkl
    #:prefab)
  name : symbol?
  importss : (listof (listof symbol?))
  import-shapess : 
(listof (listof  (or/c #f 'constant 'fixed
                       function-shape?
                       struct-shape?)))
  exports : (listof symbol?)
  internals : (listof (or/c symbol? #f))
  lifts : (listof symbol?)
  source-names : (hash/c symbol? symbol?)
  body : (listof (or/c form? any/c))
  max-let-depth : exact-nonnegative-integer?
  need-instance-access? : boolean?
代表一个 linklet,它对应于一个模块体或一个顶层序列在一个阶段。

linklet 的 name 是为了调试的目的,类似于 lambda 形式的推断名称。

importss 列表描述了 linklet 的进口。 out 列表中的每个元素都对应于一个进口源,而内部列表中的每个元素都是该源的一个出口的符号名称。 import-shapess 列表与 imports 并行; 它反映了编译器的优化假设,这些假设被字节码验证器使用,并在 linklet 实例化时被检查。

exports 列表描述了被导出的 linklet 的定义名称。 internals 列表描述了 linket 中的额外定义,但它们不能从 linklet 或其实例之一的外部访问; 一个 #f 可以出现在已经被删除的未引用的内部定义的位置。 lifts 列表是 internals 的一个扩展,用于被编译器解除的程序; 这些程序有某些属性,可以被字节码验证器检查。

在 exports, internals 和 lifts 中的每个符号必须与这些列表中的任何其他符号不同。 source-names 表将 exports, internals 和 lifts 中的符号映射到其他符号上,这些符号可能不是独立的,对应于定义的原始源名称。 source-names 表只用于调试。

当一个 linklet 被实例化时,对应于 importss, exports, internals 和 lifts 列表的扁平化的变量被放置在一个数组中(按照这个顺序),以便通过 toplevel 引用进行访问。 初始槽被保留给一个类似变量的引用,该引用强烈地保留了与它所包围的 linklet 实例的连接。

bodys 列表是小链接的可执行内容。 当linklet被实例化时, bodys 中最后一个元素的值可能被返回,这取决于它被实例化的方式。

max-let-depth 字段表示将由任何 body 创建的栈的最大尺寸。

need-instance-access? 布尔值表示 linklet 是否包含一个位置为 0 的 toplevel 。 如果 linklet 主体中不存在这样的引用,则允许使用 #t (但次优)。

struct

(struct lam expr (name
    flags
    num-params
    param-types
    rest?
    closure-map
    closure-types
    toplevel-map
    max-let-depth
    body)
    #:extra-constructor-name make-lam
    #:prefab)
  name : (or/c symbol? vector?)
  flags : 
(listof (or/c 'preserves-marks 'is-method 'single-result
              'only-rest-arg-not-used 'sfs-clear-rest-args))
  num-params : exact-nonnegative-integer?
  param-types : (listof (or/c 'val 'ref 'flonum 'fixnum 'extflonum))
  rest? : boolean?
  closure-map : (vectorof exact-nonnegative-integer?)
  closure-types : (listof (or/c 'val/ref 'flonum 'fixnum 'extflonum))
  toplevel-map : (or/c #f (set/c exact-nonnegative-integer?))
  max-let-depth : exact-nonnegative-integer?
  body : (or/c expr? seq? any/c)
代表一个 lambda 形式。 name 字段是一个用于调试的名称。 num-params 字段表示过程接受的参数数量,不包括休息参数; rest? 字段表示是否接受额外的参数并将其收集到一个 "休息" 变量中。 param-types 列表包含 num-params 符号,表示每个参数的类型, 可以是普通参数的 'val ,也可以是盒式参数的 'ref (代表一个可变的局部变量), 还可以是 flonum 参数的 'flonum ,或者 extflonum 参数的 'extflonum 。

closure-map 字段是一个堆栈位置的向量,在评估 lambda 形式以创建一个闭包时被捕获。 closure-types 字段提供了一个相应的类型列表,但是没有区分正常值和盒装值; 此外,这个信息是多余的,因为它可以由通过 closure-map 引用的绑定来推断。

当一个闭包捕获顶层或模块级的变量或引用语法对象常量时, 这些变量和常量在闭包中通过捕获前缀(在 prefix 的意义上)来表示。 toplevel-map 字段指出哪些顶层变量(即 linklet 导入和定义) 实际上被闭包使用(这样,如果前缀中的变量变得不使用,运行时系统就可以对其进行修剪), 以及是否有语法对象被使用(这样,语法对象作为一个组可以被类似地修剪)。 一个 #f 值表示没有捕捉到前缀,或者前缀中的所有变量和语法对象都应该被视为已使用。 否则,集合中的数字表示哪些变量和解除的变量被使用。 变量按前缀中的位置从 0 开始连续编号, 但是与非提升变量的数量相等的数字对应的是语法对象(即,如果使用了任何语法对象常量,该数字就包括在内)。 被提升的变量的编号紧随其后–这意味着,如果前缀包含任何语法对象, 被提升的变量的编号相对于 toplevel 来说会向下移动, 移动量为前缀中语法对象的数量(这使得 toplevel-map 集更加紧凑)。

当函数被调用时,其余的参数列表(如果有的话)被推到堆栈中,然后是正常参数的倒序,然后是闭合捕获值的倒序。 因此,当 body 被运行时,堆栈上的第一个值是被 closure-map 数组捕获的第一个值,以此类推。

max-let-depth 字段表示由 body 创建的最大堆栈深度,加上推入堆栈的参数和闭包捕获的值。 body 字段是闭包体的表达式。

Changed in version 6.1.1.8 of package zo-lib: 为 toplevel-map 添加了一个数字, 以指示是否使用了任何语法对象,如果有任何语法对象在前缀中,解除的变量的数字会向上移动一个。

struct

(struct let-one expr (rhs body type unused?)
    #:extra-constructor-name make-let-one
    #:prefab)
  rhs : (or/c expr? seq? any/c)
  body : (or/c expr? seq? any/c)
  type : (or/c #f 'flonum 'fixnum 'extflonum)

将一个未初始化的槽推入堆栈,评估 rhs 并将其值放入槽中,然后运行 body 。 如果 type 不是 #f, 那么 rhs 必须产生一个相应类型的值, 并且该槽必须被期望该类型的 localref 访问。 如果 unused? 是 #t, 那么这个槽必须不被使用, 并且 rhs 的值实际上不会被推入堆栈(但是 rhs 被限制在产生一个单一的值)。

(struct let-rec expr (procs body)

(struct boxenv expr (pos body)

struct

(struct localref expr (unbox? pos clear? other-clears? type)
    #:extra-constructor-name make-localref
    #:prefab)
  unbox? : boolean?
  pos : exact-nonnegative-integer?

struct

(struct toplevel expr (depth pos const? ready?)
    #:extra-constructor-name make-toplevel
    #:prefab)
  depth : exact-nonnegative-integer?
  pos : exact-nonnegative-integer?
  const? : boolean?
  ready? : boolean?
代表对一个导入的或在 linklet 中定义的变量的引用。 depth 字段表示要跳过多少个堆栈槽才能到达前缀数组,而 pos 是数组中的偏移。

当 toplevel 是一个表达式时,如果 const? 和 ready? 都是 #t, 那么这个变量肯定会被定义,它的值保持不变,而且这个常量对于每个模块的实例化都是有效的。 如果只有 const? 是 #t, 那么它的值是恒定的,但在不同的实例中可能会有所不同。 如果只有 ready? 是 #t, 那么这个变量肯定会被定义,但是它的值可能会改变。 如果 const? 和 ready? 都是 #f, 那么就需要进行检查以确定变量是否被定义。

当 toplevel 是 def-values 的左手边时,那么 const? 就是 #f 。 如果 ready? 是 #t, 那么该变量在被定义后会被标记为不可变的。

 (require compiler/faslable-correlated) package: zo-lib

警告: compiler/faslable-correlated 库暴露了 Racket 字节码抽象的内部信息。 与其他Racket库不同, compiler/faslable-correlated 库在不同 Racket 版本中会有不兼容的变化。
