2 raco exe: 创建独立的可执行文件

为了实现更快的启动时间,与其尝试 raco exe,不如使用更小的基础语言–比如 #lang racket/base 而不是 #lang racket 。 另外,确保字节码文件是通过使用 raco make 来编译的。

由 raco make 产生的编译代码依赖于 Racket 可执行文件来为编译代码提供运行时支持。 然而, raco exe可以将代码与它的运行时支持打包成一个可执行文件, 而 raco distribute可以将可执行文件打包成一个可以在其他机器上运行的发行文件。 运行一个由 raco exe 产生的可执行文件不会比 raco make 提高性能。

raco exe 命令将一个模块,从源代码或字节码嵌入到 racket 可执行文件的副本中。 (在 Unix 中,嵌入的可执行文件实际上是一个包装可执行文件的副本)。 创建的可执行文件在启动时调用嵌入式模块。 --gui 标志使程序被嵌入到 gracket 可执行文件的副本中。 如果嵌入的模块通过 require 引用了其他模块,那么其他模块也会包含在嵌入的可执行文件中。

例如,命令

  raco exe --gui hello.rkt

产生 "hello.exe" (Windows), "hello.app" (Mac OS) 或 "hello" (Unix), 其运行与在 gracket 中运行 "hello.rkt" 模块相同。

动态引用的库模块或其他文件–通过 eval, load 或 dynamic-require— 不会自动嵌入到创建的可执行文件中。 这样的模块可以使用 raco exe 的 ++lib 标志明确地包含在内。 另外,使用 define-runtime-path 在可执行文件中嵌入对运行时文件的引用; 然后在创建发行版时将这些文件复制并与可执行文件打包在一起(如 raco distribute: 共享独立的可执行文件 所述)。 如果一个子模块的包围模块被包含,并且该子模块包含一个名为 declare-preserve-for-embedding 的子模块(其中子模块的实现被忽略),则该子模块被包含。

只通过 #lang 使用的语言阅读模块也不会被自动嵌入。 为了支持 #lang 与语言规范的动态使用,向 raco exe 提供 ++lang 标志。 ++lang 后面的参数可以是一个语言名称,但更普遍的是,它可以是出现在 #lang 后面的文本。 例如, at-exp racket/base 作为 ++lang 的参数是有意义的, 它允许 at-exp 与 racket/base 结合作为动态加载模块的语言。

直接由扩展实现的模块–即从 (build-path "compiled" "native" (system-library-subpath)) 自动加载以满足 require 的扩展–被当作其他运行时文件对待: 生成的可执行文件从其原始位置使用它们,并且在创建分发时将它们复制并打包在一起。

当一个模块被嵌入到一个可执行文件中时,它得到一个符号名称,而不是它原来基于文件系统的名称。 模块名称解析器在嵌入的可执行文件中被配置为将基于集合的模块路径映射到嵌入的符号名称,但没有为文件系统路径创建这种映射。 默认情况下,一个模块的符号名是以一种未指定但确定的方式生成的,名字以 #%embedded: 开头, 除了主模块以 #%mzc: 为前缀。 对于那些对模块名称敏感的语言结构,例如序列化,模块名称的相对缺乏规范可能是一个问题。 要对模块的符号名称进行更多的控制,可以使用 ++named-lib 或 ++named-file 参数来指定一个前缀, 该前缀被附加在模块的基本名称之前以生成一个符号名称。

raco exe 命令只对基于模块的程序起作用。 compiler/embed 库为嵌入机制提供了一个更通用的接口。

一个独立的可执行文件是 "独立的",因为你可以在不启动 racket, gracket 或 DrRacket 的情况下运行它。 然而,该可执行文件可能依赖于 Racket 共享库,也可能依赖于通过 define-runtime-path 声明的其他运行时文件。 在 Windows 上使用 --embed-dlls 或在 Unix 上使用 --orig-exe 可能会产生一个比其他方式更独立的可执行文件。 构建 Racket 时使用的选项会影响可执行文件的独立程度。 标准发行版使用的选项可以使可执行文件尽可能独立。 对于 Unix 系统来说,配置 --enable-shared 会使可执行文件不那么独立。 对于 Mac OS 来说,如果不配置 --enable-embedfw ,就会使非 GUI 的可执行文件不那么独立。 在任何情况下,可执行文件都可以与支持库一起打包, 用 raco distribute 创建一个独立的发行版,如 raco distribute: 共享独立的可执行文件 所述。

raco exe 命令接受以下命令行标志:

  • -o ‹file› — 将可执行文件创建为 ‹file›, 根据平台和可执行文件类型在 ‹file› 上添加一个合适的后缀。 在 Mac OS 的 --gui 模式下, ‹file› 实际上是一个 bundle 目录,但它在 Finder 中显示为一个文件。

  • --gui — 创建一个基于 gracket 而不是 racket 的图形化可执行文件。

  • -l 或 --launcher — 创建一个 launcher (见 安装专用启动器), 而不是一个独立的可执行文件。 诸如 --config-path, --collects-path 和 --lib 等标志对启动器没有影响。 要注意的是,默认的命令行标志会阻止对安装在用户范围内的软件包的访问; 使用 --exf -U 可以从启动器中访问用户范围的软件包。

  • --embed-dlls — 在 Windows 下,对于一个独立的可执行文件,将任何需要的 DLL 复制到可执行文件中。 嵌入 DLL 使产生的可执行文件真正成为独立的,如果它不依赖于其他外部文件。 并非所有的 DLL 都可以嵌入,限制主要与线程本地存储和资源有关, 但 Racket 主发行版中的所有 DLL 都可以使用 --embed-dlls 。

  • --config-path ‹path› — 将可执行文件中的 ‹path› 设置为 configuration directory 的路径;如果路径是相对的,它将被视为与可执行文件相对的。 默认路径是 "etc", 希望在运行时没有这样的目录存在。

  • --collects-path ‹path› — 在可执行文件中设置 ‹path› 作为 main collection directory 的路径;如果路径是相对的,它将被视为相对于可执行文件。 默认情况是没有路径,这意味着当可执行文件启动时, current-library-collection-paths 和 current-library-collection-links 参数被初始化为 null 。 请注意,其他各种目录在默认情况下是相对于 main collection directory 的(见 安装配置和搜索路径), 所以安装 ‹path› 可能会让其他目录被找到–无论是否有意。

  • --collects-dest ‹path› — 将模块与可执行文件一起写入 ‹path› (相对于当前目录),而不是嵌入可执行文件中。 --collects-dest 标志通常只在与 --collects-path 相结合时才有意义。 这种模式目前不对未引用的子模块进行修剪(并且会拉动子模块的任何依赖关系)。

  • --ico ‹.ico-path› — 在Windows下, 将生成的可执行文件的图标设置为从 ‹.ico-path› 中提取的图标; 参见 create-embedding-executable 对 'ico 辅助关联的使用, 以了解关于预期图标尺寸和转换的更多信息。

  • --icns ‹.icns-path› — 在 Mac OS 上, 将生成的可执行文件的图标设置为 ‹.icns-path› 的内容。

  • --orig-exe — 在 Unix 上,生成一个基于原始 racket 或 gracket 可执行文件的可执行文件, 而不是一个重定向到原始可执行文件的包装器。 如果原始可执行文件是静态链接到 Racket 运行库的,那么生成的可执行文件同样是独立的。 但要注意的是,如果原始可执行文件以共享库的形式链接到 Racket, 那么 raco distribute 就不能处理以 --orig-exe 创建的可执行文件 (因为当可执行文件被分发到不同机器上时,包装可执行文件通常会负责寻找共享库)。

  • --cs — 生成一个基于 Racket 的 CS 实现的可执行文件,这是默认的, 除非运行一个基于 BC 实现的 raco exe 。

  • --3m — 生成基于 Racket 的 3m 变体的可执行文件, 只有在运行基于 BC 实现的 3m 变体的 raco exe 时才是默认的。

  • --cgc — 生成一个基于 Racket 的 CGC 变体的可执行文件, 只有在运行基于 BC 实现的 CGC 变体的 raco exe 时才默认为可执行文件。

  • ++aux ‹file› — 基于 ‹file› 的后缀将信息附加到可执行文件中; 参见 extract-aux-from-path 以获得认可的后缀列表和含义, 参见 create-embedding-executable 对辅助关联的使用以获得关于每种文件如何被使用的更具体信息。

  • ++lib ‹module-path› — 在可执行文件中包括 ‹module-path› , 即使它没有被主程序引用,这样它就可以通过 dynamic-require 来使用。

  • ++lang ‹lang› — 包括动态加载 #lang ‹lang› 开始的模块所需的模块。 ‹lang› 不一定是一个普通的语言或模块名称;它可能是一个更一般的文本序列, 例如 at-exp racket/base ,以支持像 at-exp 这样的语言构造器。 读作 ‹lang› 的 module 的初始 require 必须通过语言阅读器的 get-info 函数和 'module-language 键来实现; 用 syntax/module-reader 实现的语言自动支持该键。

  • ++named-lib ‹prefix› ‹module-path› — 类似于 ++lib, 但是嵌入式模块的符号名称被指定为 ‹prefix› ,附加在库文件的基本名称之前。 指定一个模块的符号名对那些反射性地依赖模块名的语言结构是很有用的, 比如序列化格式(其中模块名被记录下来,以便以后可以找到一个函数进行反序列化)。

  • ++named-file ‹prefix› ‹file-path› — 在可执行文件中包括 ‹file-path› ,即使它没有被主程序引用, 并在文件的基本名称前使用 ‹prefix› 作为嵌入式模块的符号名称。 由于嵌入式模块的符号名是可预测的,所以在运行时可以通过动态查询 dynamic-require 来访问该模块。 一个可预测的模块名称也可以帮助处理序列化的数据,其方式与 ++named-lib 相同。

  • ++exf ‹flag› — 在启动时向嵌入式 racket 或 gracket 提供 ‹flag› 命令行参数。

  • --exf ‹flag› — 从启动时提供给嵌入式 racket 或 gracket 的命令行参数中删除 ‹flag› 。

  • --exf-clear — 删除启动时提供给嵌入式 racket 或 gracket 的所有命令行参数。

  • --exf-show — 显示(不改变)启动时提供给嵌入式 racket 或 gracket 的命令行参数。

  • -v — 报告进展情况,以节略方式进行。

  • --vv — 比 -v 更粗略地报告进度。

2.1 创建可执行文件的API

 (require compiler/embed) package: base
compiler/embed 库提供了一个将 Racket 代码嵌入到 Racket 或 GRacket 副本中的功能, 从而创建一个独立的 Racket 可执行文件。 要将可执行文件打包成独立于 Racket 安装的发行版,请使用 compiler/distribute 中的 assemble-distribution 。

嵌入在模块依赖关系图中行走,以找到一些初始的顶级模块集所需要的所有模块,如果需要的话, 对它们进行编译,并将它们组合成一个 "模块捆绑"。 除了模块代码外,捆绑包还扩展了模块名称解析器,因此,模块可以用它们的原始名称来 require , 它们将从捆绑包而不是文件系统中被检索。

create-embedding-executable 函数将 bundle 与一个可执行文件(Racket 或 GRacket)结合起来。 write-module-bundle 函数将 bundle 打印到当前的输出端口; 只要 read-accept-compiled 参数为 true,这个流就可以直接被运行中的程序 load 。

复制 Racket (如果 gracket? 和 mred? 是 #f) 或 GRacket (否则)的二进制文件, 将代码嵌入复制的可执行文件中,在启动时加载。 在 Unix 上,这个二进制文件实际上是一个 exec 原始文件的包装器; 也可以参见 aux 的 'original-exe? 标签。

嵌入的可执行文件被写入 dest, 如果它已经存在(作为一个文件或目录),则被覆盖。

嵌入的代码由模块声明和附加的(任意的)代码组成。当一个模块被嵌入时,它所导入的每个模块也被嵌入。 库模块被嵌入后,它们可以通过初始命名空间中的 lib 路径访问。

#:modules 参数 mod-list 指定了要被嵌入的模块,如下所述。 #:early-literal-expressions, #:literal-files 和 #:literal-expressions 参数指定了要复制到可执行文件中的字面代码: early-literal-sexps 中的每个元素被依次复制, 然后是 literal-files 每个文件的内容(没有中间的空格),然后是 literal-sexps 的每个元素。 literal-files 文件或 early-literal-sexps 或 literal-sexps 列表可以包含已编译的字节码,而且有可能 literal-files 文件的内容只有在串联时才能解析; 在嵌入过程中,文件和表达式不会被编译或以任何方式检查。 请注意初始命名空间不包含任何绑定;使用编译后的表达式来引导命名空间。 #:literal-expression (单数)参数是为了向后兼容。

如果 #:configure-via-first-module? 参数被指定为 true, 那么 mod-list 中第一个模块的语言会在 #:literal-files 和 #:literal-expressions 添加的表达式被评估之前, 但在 #:early-literal-expressions 的表达式之后用于配置运行时环境。 请参见 语言运行时配置 。

#:cmdline 参数 cmdline 包含命令行字符串, 这些字符串是提供给嵌入可执行文件的任何实际命令行参数的前缀。 一个评估表达式或加载文件的命令行参数将在嵌入代码加载后被执行。

#:modules 参数 mod-list 的每个元素都是一个 2 或 3 项的列表,其中第一项是模块名称的前缀, 第二项是一个模块路径数据(是默认模块名称解析器理解的格式),第三项是一个子模块名称的列表,如果它们可用的话,将被包括在内。 前缀可以是一个符号, #f 表示没有前缀,或者 #t 表示一个自动生成的前缀。例如、

'((#f "m.rkt"))

嵌入文件 "m.rkt" 中的模块 m ,没有前缀的模块名称; 配合上述的 literal-sexpr 参数可能是 '(require m) 。 当子模块可用并被包含时,通过将子模块路径的 write 形式加到包围模块的名称上,子模块被赋予一个名称。

当一个嵌入式模块没有被列在 #:modules 参数中,或者没有被赋予一个前缀,一个嵌入式模块的符号名称会被自动生成。 这些名字是以一种确定的但未指定的方式生成的,所以它们不方便访问。 生成的名字可能取决于 mod-list 的第一个元素的路径。 通过基于集合的路径包含的模块在运行时仍然可以通过其基于集合的路径(通过为嵌入的可执行文件安装的模块名称解析器)进行访问。

模块在被嵌入到目标可执行文件之前通常会被编译; 参见下面的 #:compiler 和 #:src-filter 。 当一个模块通过 define-runtime-path 声明运行时路径时, 生成的可执行文件会记录该路径(用于即时执行和创建包含可执行文件的分发)。

如果 collects-dest 是一个路径而不是 #f,那么不是将基于集合的模块嵌入到可执行文件中, 而是将模块(仅以编译的形式)复制到 collects-dest 目录下的集合中。

可选的 #:aux 参数是一个用于平台特定选项的关联列表 (即,它是一个配对列表,其中配对的第一个元素是一个关键符号,第二个元素是该关键的值)。 另见 build-aux-from-path 。目前支持的键值如下:

  • 'icns (Mac OS) : 图标文件路径(后缀 ".icns") ,用于可执行文件的桌面图标。

  • 'ico (Windows) : 一个图标文件路径(后缀为 ".ico") ,用于可执行文件的桌面图标。

    Changed in version 6.3 of package base: 可执行文件中的所有图标都被替换成文件中的图标, 而不是只设置某些尺寸和深度。

  • 'creator (Mac OS) : 提供一个4字符的字符串作为应用程序的签名。

  • 'file-types (Mac OS) : 提供了一个关联列表,应用程序处理的每种类型的文件都有一个; 每个关联是一个双元素列表,其中第一个(关键)元素是 Finder 识别的字符串, 第二个元素是 plist 值(见 xml/plist)。 参见 "drracket" 集合中的 "drracket.filetypes",以获得一个例子。

  • 'uti-exports (Mac OS) : 提供一个关联列表, 每个由可执行文件导出的 Uniform Type Identifier (UTI) 都有一个; 每个关联是一个双元素列表,其中第一个(关键)元素是 UTI 声明中认可的字符串, 第二个元素是一个 plist 值(见 xml/plist) 。 参见 "drracket" 集合中的 "drracket.utiexports" ,以获得一个例子。

  • 'resource-files (Mac OS) : 额外的文件, 复制到生成的可执行文件的 "Resources" 目录中。

  • 'config-dir : 包含配置信息的目录的字符串/路径, 例如 "config.rtkd" (见 安装配置和搜索路径)。 如果没有提供值,路径将保持原样,并根据需要转换为绝对形式。 如果提供了 #f ,路径将保持原样(可能是相对形式)。 请注意,如果 collects-path 被提供为一个空列表, 那么配置目录的路径就不会被 Racket 的启动过程所使用 (与正常的 Racket 启动相反,配置目录会被用来查询集合链接文件的信息)。

  • 'framework-root (Mac OS) : 一个字符串, 用来给可执行文件到 Racket 和 GRacket 框架的路径加前缀(包括一个分隔的斜线); 注意,当前缀启动 "@executable_path/" 对基于 Racket 的应用程序有效时, 对应的基于 GRacket 的应用程序的前缀启动是 "@executable_path/../../../"; 如果提供 #f ,可执行文件的框架路径保持不变, 否则原始可执行文件到一个框架的路径将转换为绝对路径,如果它是相对路径。

  • 'dll-dir (Windows) : 一个包含可执行文件所需的 Racket DLL 目录的字符串/路径, 例如 "racket‹version›.dll", 或者一个布尔值; 路径可以是相对于可执行文件的;如果提供了 #f ,路径将保持原样; 如果提供了 #t ,路径将被删除(因此 DLL 必须在系统目录或用户的 PATH 中); 如果没有提供值,原始可执行文件的 DLL 路径将转换为绝对路径(如果它是相对路径)。

  • 'embed-dlls? (Windows) : 一个布尔值,表示是否将 DLL 复制到可执行文件中,其默认值为 #f 。 嵌入式DLLs是通过内部链接步骤实例化的,绕过了一些操作系统的设施, 所以它不会对所有的 Windows DLL 起作用,但典型的 DLL 会作为嵌入式的工作。

  • 'subsystem (Windows) : 一个符号,可以是控制台应用的 'console , 也可以是无控制台应用的 'windows ; 默认是基于 Racket 的应用的 'console ,基于 GRacket 的应用的 'windows ; 也可以参见下面的 'single-instance? 。

  • 'single-instance? (Windows) : 一个布尔值,用于基于 GRacket 的应用程序; 默认值是 #t, 这意味着应用程序在启动时寻找自己的实例,只是把另一个实例带到前面; #f 意味着期待多个实例。

  • 'forget-exe? (Unix, Windows, Mac OS) : 一个布尔值; #t 用于启动器(见下面的 launcher? ), 不会为 (find-system-path 'exec-file) 保留原始的可执行文件名; 一个后果是,库集合将被找到,相对于启动器而不是原始可执行文件。

  • 'original-exe? (Unix) : 一个布尔值; #t 表示嵌入使用原始的 Racket 或 GRacket 可执行文件,而不是 exec 原始文件的封装二进制文件; 默认为 #f 。

  • 'relative? (Unix, Windows, Mac OS) : 一个布尔值; #t 意味着,在生成的可执行文件必须引用另一个文件的情况下, 它可以使用相对路径(所以可执行文件可以一起移动,但不能单独移动), 它意味着 #f 用于 'config-dir, 'framework-dir 和'dll-dir, 除非明确提供这些; #f 值(默认)意味着应该使用绝对路径(所以生成的可执行文件可以被移动)。

  • 'wm-class (Unix) : 一个字符串;用作程序窗口的默认 WM_CLASS 程序类。

如果 #:collects-path 参数是 #f, 那么创建的可执行文件将保持其内置的(相对)路径到主 "collects" 目录–这将是运行可执行文件时 (find-system-path 'collects-dir) 的结果– 加上一个潜在的其他目录列表,用于寻找图书馆收藏–这将用于结合 PLTCOLLECTS 环境变量来初始化 current-library-collection-paths 列表。 否则,该参数将指定一个替换;它必须是一个路径、字符串或路径和字符串的列表。 在最后一种情况下,第一个路径或字符串指定主收集目录, 其余的是收集搜索路径的附加目录(按顺序放在用户特定的 "collects" 目录之后, 但在主 "collects" 目录之前;然后搜索列表与 PLTCOLLECTS 相结合,如果它被定义的话)。 如果列表为空,那么 (find-system-path 'collects-dir) 将返回可执行文件的目录, 但 current-library-collection-paths 被初始化为空列表, use-collection-link-paths 被设置为false以禁止使用 collection links files 。

如果 #:launcher? 参数是 #t, 那么 mod-list 应该是空的, literal-files 应该是空的,而 literal-sexp 应该是 #f 。 嵌入的可执行文件的创建方式是: (find-system-path 'exec-file) 产生源 Racket 或 GRacket 路径, 而不是嵌入的可执行文件(但 (find-system-path 'run-file) 的结果仍然是嵌入的可执行文件), 除非 'forget-exe? 与 aux 中的一个真值相关。

#:variant 参数表示要使用哪种原始二进制文件的变体进行嵌入。 默认是 (system-type 'gc); 也可参见 current-launcher-variant 。

#:compiler 参数用于编译将包含在可执行文件中的模块的源代码(当编译后的形式还不可用时)。 它应该接受一个单一的参数,该参数是一个 module 形式的语法对象。 默认程序使用 compile 参数化,将当前命名空间设置为 expand-namespace 。

#:expand-namespace 参数为扩展额外模块选择一个命名空间(以及使用默认的 compile-proc) 。 额外模块的扩展是需要的,以检测包含的模块中的运行时路径声明, 这样路径解析就可以被引导到当前的位置(并最终被重定向到发行版中的副本)。

#:src-filter src-filter 参数接收一个路径, 如果相应的文件源应该以源的形式(而不是编译的形式)包含在嵌入的可执行文件中,则返回 true,否则返回 #f 。 默认情况下,对所有路径都返回 #f 。 注意,当过滤过程被调用时,当前的输出端口可能会被重定向到结果可执行文件。 给予 src-filter 的每个路径都对应于实际的文件名 (例如, ".ss"/".rkt" 的转换已经根据需要应用到了现有的文件中)。

如果 #:on-extension 参数是一个过程,当模块依赖关系的遍历到达一个扩展(即 DLL 或共享对象)时,该过程被调用。 默认情况下, #f 会导致一个单模块扩展(在其当前位置)的引用被嵌入到可执行文件中。 该过程被调用时有两个参数:一个是扩展的路径,另一个是 #f (出于历史原因)。

#:get-extra-imports extras-proc 参数需要一个源路径名和要包含在可执行文件中的每个模块的已编译模块。 它返回一个带引号的模块路径列表(绝对路径,而不是相对于模块的路径), 这些模块将被包含在可执行文件中,除了源模块 require 的模块之外。 例如,这些模块可能对应于解析模块所需的阅读器扩展,只要通过绝对模块路径引用阅读器,该模块就会作为源文件被包含。 给予 extras-proc 的每个路径都对应于实际的文件名 (例如, ".ss"/".rkt" 的转换已根据需要应用于引用现有文件)。

Changed in version 7.3.0.6: 改变了嵌入式模块的符号名称的生成,使其具有确定性。

2.2 安装专用启动器

launcher 启动器类似于一个独立的可执行文件,但启动器通常更小,而且可以更快创建, 因为它永久依赖于本地 Racket 安装和程序的源代码。在 Unix 的情况下, 启动器只是一个运行 racket 或 gracket 的 shell 脚本。 启动器 不能 用 raco distribute 打包成一个发行版。 raco exe 命令在指定了 -l 或 --launcher 标志后会创建一个启动器。

 (require launcher/launcher) package: base

2.2.1 创建启动器

procedure

创建启动器 dest ,它以 args 中指定的字符串形式的命令行参数启动 GRacket。 在运行时传递给启动器的额外参数被附加到这个列表中(除特殊的 Unix/X 标志处理外,如下所述)并传递给 GRacket。 如果 dest 已经存在,作为一个文件或目录,它将被替换。

可选的 aux 参数是一个用于平台特定选项的关联列表 (即,它是一个配对列表,其中配对的第一个元素是一个关键符号,第二个元素是该关键的值)。 另见 build-aux-from-path 。 参见 create-embedding-executable , 该列表同时适用于 Windows 和 Mac OS GRacket 上的独立可执行文件和启动器; 以下额外的关联适用于启动器:

  • 'independent? (Windows) — 一个布尔值; #t 创建一个旧式的启动器,可以与任何 Racket 或 GRacket 二进制文件一起使用,比如 raco.exe 。 其他 aux 关联都不用于旧式启动器。

  • 'exe-name (Mac OS, 'script-3m, 'script-cgc ak 'script-cs 变体) — 提供 '3m-/'cgc-/'cs 启动器的基本名称,脚本会忽略 args 而调用它。 如果没有提供这个名字,脚本将像往常一样通过 GRacket 可执行程序。

  • 'exe-is-gracket (当使用 'exe-name 时) — 表示 'exe-name 指的是 GRacket 可执行文件,它有可能在 "lib" 子目录下, 而不是与其他 GUI 应用程序一起。

  • 'relative? (所有平台) — 一个布尔值,其中 #t 表示生成的启动器应该通过一个相对路径找到基本的 GRacket 可执行文件。

  • 'install-mode (Windows, Unix) — 'main, 'user, 'config-tethered 或 'addon-tethered , 表示启动器被安装到整个安装区域、用户特定区域、嵌入配置路径的整个安装区域或嵌入 add-directory 路径的特定区域; 安装模式反过来决定了是否以及在哪里记录 'start-menu, 'extension-registry, 和/或 'desktop 信息。

  • 'start-menu (Windows) — 一个布尔值或实数; #t 表示安装程序包括启动器时,启动器应该在 Start 菜单中。 数字值的处理方式与 #t 类似,但也要求安装程序自动启动该应用程序, 其中数字决定了相对于其他可能要求启动的启动程序的优先级。 'start-menu 值只在 'install-mode 也被指定时使用。

  • 'extension-register (Windows) — 一个文件类型的列表,用于安装程序执行的文件扩展注册。 每个文件类型由一个包含六个项目的列表描述:

    • 一个描述文档类型的可读字符串,如 "Racket Document";

    • 一个作为文档类型的键的字符串,如 "Racket.Document";

    • 一个字符串的列表,每个字符串都是不含点的文件扩展名,比如 '("rkt" "rktl" "rktd");

    • 一个提供图标的文件的路径,如 "doc.ico";

    • 一个字符串,表示处理具有匹配扩展名的文件的命令行,如 "\"%1\"", 其中该字符串将以启动器的路径为前缀,而 %1 将被替换为文件路径

    只有当 'install-mode 也被指定时,才会使用 'extension-registry 值。

  • 'desktop (Unix) — 一个包含启动器的 ".desktop" 文件内容的字符串, 其中 Exec 和 Icon 条目会被自动添加。 如果字符串中存在 Exec 条目,并且其值以非空的字母数字 ASCII 字符序列开始, 然后是空格,那么空格和其余的值将被附加到自动生成的值中。 ".desktop" 文件被写入由 (find-apps-dir) 或 (find-user-apps-dir) 产生的目录中。 只有当 'install-mode 也被指定时, 'desktop 值才会被使用。

  • 'png (Unix) : 一个由 ".desktop" 文件(如果有的话)引用的图标文件路径 (后缀为 ".png") ; 'png 值比 'ico 值优先, 但除非 'desktop 值也存在,否则两者都不使用。

  • 'ico (Unix, 除了更普遍的 Windows 使用之外): 一个图标文件路径(后缀为 ".ico") ,如果没有 'png 值,它的使用方式与 'png 相同。

对于 Unix/X, make-mred-launcher 创建的脚本会检测并处理 X Windows 标志,当它们作为脚本的初始参数出现时。 这些参数不是附加在 args 的末尾,而是拼接在 args 中已经列出的任何 X Windows 标志之后。 剩下的参数(即最后一个 X Windows 标志或参数之后的所有脚本标志和参数)会被附加在拼接的 args 之后。

tether-mode 参数表示在多大程度上保留当前安装的配置目录和/或基于 (find-addon-tether-console-bin-dir) 和 (find-config-tether-console-bin-dir) 的addon目录的拴住。 'addon 模式允许完全拴住, 'config 模式只允许配置目录拴住,而 #f 模式则禁用拴住。

Changed in version 6.5.0.2 of package base: 增加了 #:tether-mode 参数。

2.2.2 发射器路径和平台惯例

procedure

  [#:user? user?    
  #:tethered? tethered?    
  #:console? console?]) → path?
  name : string?
  user? : any/c = #f
  tethered? : any/c = #f
  console? : any/c = #f
返回一个名为 name 的可执行文件的路径名,该文件在

  • Racket 安装 — 当 user? 是 #f ,并且 tethered? 是 #f 时;

  • 用户的 Racket 可执行目录 — 当 user? 是 #t 并且 tethered? 是 #f 时;

  • 一个额外的可执行目录,用于将可执行文件拴在一个特定的配置目录下 — 光 user? 是 #f 并且 tethered? 是 #t 时;或

  • 一个额外的可执行目录,用于连接到一个特定的附加组件和配置目录的可执行文件 — 当 user? 是 #t 并且 tethered? 是 #t 时。

对于 Windows, ".exe" 后缀会自动附加到 name 上。 对于 Unix 来说, name 被改为小写,空白被改为 -, 并且路径包括 Racket 安装的 "bin" 子目录。 对于 Mac OS, ".app" 的后缀会附加到 name 上。

如果 console? 为true,那么路径就在 console 可执行目录中, 比如 (find-console-bin-dir) 报告的目录,而不是 GUI 可执行目录, 比如 (find-gui-bin-dir) 报告的目录。
