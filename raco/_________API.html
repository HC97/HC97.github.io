1.4 用于生成字节码的 API

 (require compiler/cm) package: base
compiler/cm 模块实现了 raco make 和 raco setup 所使用的编译和依赖性管理。

procedure

(make-compilation-manager-load/use-compiled-handler 
  [delete-zos-when-rkt-file-does-not-exist? 
  #:security-guard security-guard]) 
 → (path? (or/c symbol? false/c) . -> . any)
  delete-zos-when-rkt-file-does-not-exist? : any/c = #f
  security-guard : (or/c security-guard? #f) = #f
返回一个适合作为 current-load/use-compiled 参数值的过程。 返回的过程将其参数传递给 current-load/use-compiled 过程, 该过程在调用 make-compilation-manager-load/use-compiled-handler 时被安装, 但首先它会自动将一个源文件编译成 ".zo" 文件,如果

  • 该文件被认为包含一个模块(即处理程序的第二个参数是一个符号);

  • (current-eval), (current-load) 和 (namespace-module-registry (current-namespace)) 的值与调用 make-compilation-manager-load/use-compiled-handler 时相同;

  • use-compiled-file-paths 的值包含在调用 make-compilation-manager-load/use-compiled-handler 时出现的第一个路径;

  • current-load/use-compiled 的值是这个过程的结果;并且

  • 以下情况之一成立:

    • 源文件比 use-compiled-file-paths 中列出的第一个子目录中的 ".zo" 文件要新 (在 make-compilation-manager-load/use-compiled-handler 被调用时), 并且要么不存在 ".dep" 文件,要么它记录的源文件 SHA-1 哈希值与当前版本和源文件SHA-1哈希值不同;

    • 在 ".zo" 文件旁边不存在 ".dep" 文件;

    • 在 ".dep" 文件中记录的版本与 (version) 的结果不一致;

    • 在 ".dep" 文件中记录的目标机器与 (current-compile-target-machine) 的结果不一致;

    • 在 ".dep" 文件中记录的源码哈希值与当前源码哈希值不一致;

    • 在 ".dep" 文件中列出的一个文件的 ".zo" 时间戳比目标 ".zo" 的时间戳新, 并且 use-compiled-file-check 被设置为 'modify-seconds;

    • 在 ".dep" 文件中记录的依赖关系的组合哈希值与 ".dep" 文件中记录的组合哈希值不一致。

如果 SHA-1 哈希值覆盖了基于时间戳的重新编译文件的决定,那么目标 ".zo" 文件的时间戳将更新为当前时间, 除非 use-compiled-file-check 参数没有设置为 'modify-seconds 。

在处理程序编译一个 ".zo" 文件后,它会创建一个相应的 ".dep" 文件, 其中列出了当前的版本和编译文件中模块直接 require 的每个文件的标识。 额外的依赖关系可以在编译期间通过 compiler/cm-accomplice 安装。 ".dep" 文件还记录了模块源代码的 SHA-1 哈希值,它还记录了所有依赖关系的组合 SHA-1 哈希值,包括其递归依赖关系。 如果一个字节码文件是通过重新编译以前被编译为独立于机器的字节码文件而产生的, 那么 ".dep" 文件也记录了独立于机器的形式的 SHA-1 哈希值,因为重新编译的模块的行为应该是完全一样的。

(cross-installation?) 或 (current-multi-compile-any) 为 #t, (current-compile-target-machine) 为 #f, (current-compiled-file-roots) 有两个或多个元素的特殊组合会触发一种特殊的编译模式。 运行中的 Racket 特有的字节码被写入由 (current-compiled-file-roots) 的第一个元素决定的目录中。 (cross-installation?) 的交叉编译目标或 (current-multi-compile-any) 的机器独立格式的字节码被写入由 (current-compiled-file-roots) 的第二个元素决定的目录。 通过配置 (current-compiled-file-roots) ,使第一个元素在构建树之外, 第二个元素在构建树之内,交叉编译可以创建一个适合目标机器的构建树,同时构建和加载字节码 (用于宏扩展等),这在当前机器上是可用的。 这种模式对于开始时只有源代码和与机器无关的字节码的构建目录来说是正确的。

处理程序在检查 ".dep" 文件时,会缓存时间戳,并且在调用同一处理程序时保持缓存。 在比较即时源文件和其 ".zo" 文件时,不会查阅缓存, 这意味着缓存行为与默认模块名称解析器的缓存一致(见 current-module-name-resolver) 。

如果在 make-compilation-manager-load/use-compiled-handler 被调用时, use-compiled-file-paths 包含一个空列表, 那么就会引发 exn:fail:contract 。

如果 delete-zos-when-rkt-file-does-not-exist? 参数是一个真值, 那么返回的处理程序将在没有相应的原始源文件时删除 ".zo" 文件。

如果提供了 security-guard 参数,那么在创建 ".zo" 文件、 ".dep" 文件和 "compiled/" 目录时,以及在调整现有文件的时间戳时,会使用该参数。 如果它是 #f, 那么在创建文件时使用 current-security-guard 中的安全防护(而不是在调用 make-compilation-manager-load/use-compiled-handler 时的安全防护)。

一个模块的继续编译被标记为 managed-compiled-context-key 和该模块的源路径。

当当前命名空间包含可能需要重新编译的已经加载的模块版本时, 不要 安装 make-compilation-manager-load/use-compiled-handler 的结果 –除非已经加载的模块从未被尚未加载的模块引用过。 对已经加载的模块的引用可能会产生时间戳不一致的编译文件和/或信息不正确的 ".dep" 文件。

处理程序将消息记录到主题 'compiler/cm 的 'info 级别。 这些消息是一个 compile-event 预制结构的实例:

(struct compile-event (timestamp path type) #:prefab)

timestamp 字段是事件发生的时间,单位是自历时以来的毫秒。 path 字段是正在编译的文件的路径,该事件是关于该文件的。 type 字段是一个描述事件所对应的动作的符号。 目前记录的值是 'locking, 'start-compile, 'finish-compile 和 'already-done 。

Changed in version 6.1.1.8 of package base: 增加了通过 managed-compiled-context-key 识别编译上下文的功能。
Changed in version 6.6.0.3: 增加了对源代码的 SHA1 哈希值的检查,以补充时间戳的检查, 后者可以通过 use-compile-file-check 禁用。

procedure

(managed-compile-zo file    
  [read-src-syntax    
  #:security-guard security-guard]) → void?
  file : path-string?
  read-src-syntax : (any/c input-port? . -> . syntax?)
   = read-syntax
  security-guard : (or/c security-guard? #f) = #f
将给定的模块源文件编译为 ".zo" ,在文件编译的同时安装一个编译管理器处理程序(以便所需的模块也被编译), 并创建一个 ".dep" 文件来记录用于编译源文件的即时文件的时间戳(即源文件中 require 的文件)。

编译是通过将一个模块加载到当前命名空间来触发的,所以如果作为 file 依赖的模块已经被加载到当前命名空间, 那么该模块不一定会被(重新)编译。 用于触发编译的处理程序是由 make-compilation-manager-load/use-compiled-handler 创建的, 所以那里的所有规则和约束都适用。

如果 file 是从源代码编译的,那么 read-src-syntax 的使用方式与 read-syntax 相同, 以读取源代码模块。 然而,任何需要的文件都会使用普通的 read-syntax 。

如果 security-guard 不是 #f, 那么在创建 "compiled/" 目录、 ".dep" 和 ".zo" 文件,以及调整现有文件的时间戳时,将使用所提供的安全保护。 如果是 #f,那么就使用创建文件时的 current-security-guard 中的安全防护 (而不是调用 managed-compile-zo 时的安全防护)。

在编译 file 时, error-display-handler 参数被设置为 (make-compilation-context-error-display-handler (error-display-handler)), 因此来自未捕获异常的错误会报告编译环境。

Changed in version 6.1.1.8 of package base: 增加了 error-display-handler 配置。

parameter

(manager-skip-file-handler)

 → (-> path? (or/c (cons/c number? promise?) #f))
(manager-skip-file-handler proc) → void?
  proc : (-> path? (or/c (cons/c number? promise?) #f))
一个参数,其值为每个被加载并需要重新编译的文件所调用。 如果该过程返回一对,那么该文件将被跳过(即不被编译);一对中的数字被用作该文件字节码的时间戳, 并且该承诺可以被 force 获得一个字符串,作为已编译文件及其依赖关系的哈希值。 如果该过程返回 #f, 那么该文件就像往常一样被编译。默认是 (lambda (x) #f) 。

parameter

(current-path->mode)

 → (or/c #f (-> path? (and/c path? relative-path?)))
(current-path->mode path->mode) → void?
  path->mode : (or/c #f (-> path? (and/c path? relative-path?)))
 = #f
由 make-compilation-manager-load/use-compiled-handler 和 make-caching-managed-compile-zo 使用,以覆盖 use-compiled-file-paths ,决定将编译后的 ".zo" 文件写入何处。 如果它是 #f, 那么使用 use-compiled-file-paths 的第一个元素。 如果它不是 #f, 那么就用原始源文件的位置来调用它, 其结果与 use-compiled-file-paths 的第一个元素的处理相同。

注意,这个参数不被 current-load/use-compiled 使用。 因此,如果该参数导致 ".zo" 文件被放置在不同的目录中, 那么正确的 ".zo" 文件仍然必须通过 use-compiled-file-paths 来传达, 一种方法是覆盖 current-load/use-compiled 来删除那些会导致在加载前选择错误的 ".zo" 文件。

parameter

(parallel-lock-client)

 → 
(or/c #f
      (->i ([command (or/c 'lock 'unlock)]
            [file bytes?])
           [res (command) (if (eq? command 'lock)
                              boolean?
                              void?)]))
(parallel-lock-client proc) → void?
  proc : 
(or/c #f
      (->i ([command (or/c 'lock 'unlock)]
            [file bytes?])
           [res (command) (if (eq? command 'lock)
                              boolean?
                              void?)]))
创建一个并行编译锁客户端,由 make-compilation-manager-load/use-compiled-handler 的结果使用, 以防止并行构建者之间的编译竞赛。

当 proc 为 #f 默认值)时,不会对并行编译进行检查 (因此,通过 make-compilation-manager-load/use-compiled-handler 运行编译的多个线程或地方将有可能破坏彼此的 ".zo" 文件)。

当 proc 是一个函数时,它的第一个参数是命令 'lock pr 'unlock, 表示调用者是否要锁定或解锁目标 zo-path, 第二个参数是目标 zo-path(表示为一个字节字符串)。

当 proc 对 'lock 命令返回 #t 时,当前构建者已经获得了 zo-path 的锁。 zo-path 的编译完成后, 构建者进程必须用完全相同的 zo-path 调用 proc 'unlock 来释放锁。

当 proc 的 'lock 命令返回 #f 时, 另一个并行构建程序首先获得了锁,并且已经编译了目标。 并行构建器应该继续进行,而不编译 zo-path 。 (在这种情况下, make-compilation-manager-load/use-compiled-handler 的结果不会用 'unlock 调用 proc 。)

Example:
> (let* ([lc (parallel-lock-client)]
         [zo-name  #"collects/racket/compiled/draw_rkt.zo"]
         [locked? (and lc (lc 'lock zo-name))]
         [ok-to-compile? (or (not lc) locked?)])
    (dynamic-wind
      (lambda () (void))
      (lambda ()
        (when ok-to-compile?
          (printf "Do compile here ...\n")))
      (lambda ()
        (when locked?
          (lc 'unlock zo-name)))))

Do compile here ...

procedure

(compile-lock->parallel-lock-client pc 
  [cust 
  current-shutdown-evt]) 
 → (-> (or/c 'lock 'unlock) bytes? boolean?)
  pc : place-channel?
  cust : (or/c #f custodian?) = #f
  current-shutdown-evt : (-> evt?) = (lambda () never-evt)
返回一个遵循 parallel-lock-client 协议的函数,通过 pc 进行通信, 其中 pc 必须是 make-compile-lock 的一个结果。

当 cust 为 #f 时,这个通信协议的实现是不安全的。 要使协议具有杀伤力,需要一个足够强大的监护人(即,除非所有参与编译的人也被终止,否则不会被终止)作为 cust 提供。 给定的监护人被用来创建一个线程,监控正在进行编译的线程。 如果其中一个线程被终止了,监护人的存在会让另一个线程继续进行。 (监护人也被用来创建一个管理线程安全表的线程)。

仅仅检查线程终止并不总是足以释放一个锁, 因为用 thread/suspend-to-kill 创建的线程只是通过移除其运行能力而暂停。 current-shutdown-evt 参数返回一个 synchronizable event , 监控线程在等待线程终止的同时也在等待这个事件。 如果该事件变得就绪,那么监控器就会释放一个锁,就像线程被终止一样。 例如, current-shutdown-evt 可能会返回一个 custodian box , 以检测托管的关闭。

Changed in version 8.1.0.7 of package base: 增加了 current-shutdown-evt 参数。
