6.9 读取 "info.rkt" 文件的 API

 (require setup/getinfo) package: base
setup/getinfo 库提供了访问 "info.rkt" 文件中字段的函数。 "info.rkt" 文件的格式在 "info.rkt" File Format 中有所说明。

procedure

(get-info collection-names 
  [#:namespace namespace 
  #:bootstrap? bootstrap?]) 
 → 
(or/c
 (symbol? [(-> any)] . -> . any)
 #f)
  collection-names : (listof string?)
  namespace : (or/c namespace? #f) = #f
  bootstrap? : any/c = #f
接受一个命名集合或子集合的字符串列表,并调用 get-info/full 与命名集合对应的完整路径和 namespace 参数。

procedure

(get-info/full path 
  [#:namespace namespace 
  #:bootstrap? bootstrap?]) 
 → 
(or/c (->* (symbol?) ((-> any)) any)
      #f)
  path : path-string?
  namespace : (or/c namespace? #f) = #f
  bootstrap? : any/c = #f
接受一个目录的路径。 如果它找到一个格式良好的 "info.rkt" 文件或者 "info.ss" 文件 (优先考虑 "info.rkt" 文件),它返回一个接受一个或两个参数的 info 过程。 info 过程的第一个参数总是一个符号化的名字,其结果是 "info.rkt" 文件中该名字的值,如果该名字被定义的话。 可选的第二个参数 thunk 是一个不需要参数的过程,当名称没有定义时被调用; 在这种情况下,info过程的结果就是 thunk 的结果。 如果名称没有被定义,并且没有提供 thunk ,那么就会产生一个异常。

如果目录中没有 "info.rkt" (或 "info.ss") 文件, get-info/full 函数返回 #f 。 如果有一个 "info.rkt" (或 "info.ss") 文件的形状不对 (即不是使用 info 或 setup/infotab 的模块), 或者 "info.rkt" 文件加载失败,则会产生一个异常。 如果 "info.rkt" 文件加载了, get-info/full 返回 get-info 文件。 如果 "info.rkt" 文件不存在,那么 get-info/full 对 "info.ss" 文件做同样的检查, 要么引发一个异常,要么返回 "info.ss" 文件中的 get-info 函数。

如果 "info.rkt" (或 "info.ss") 模块不是 #f 的话, 它将被加载到 namespace ,否则就是一个私有的、弱化的命名空间。

如果 bootstrap? 为真,那么在读取 "info.rkt" (或 "info.ss") 时, use-compiled-file-paths 被设置为 '() ,以防止现有的编译文件被破坏。 此外,在试图加载 "info.rkt" (或 "info.ss")之前, info 和 setup/infotab 模块从 get-info/full 的命名空间被附加到 namespace 。

随着模块的加载, environment variable set 被修剪成只包含 PLT_INFO_ALLOW_VARS 环境变量中列出的环境变量, 该变量包含一个 ;分隔的名称列表。 默认情况下,允许的变量名称列表为空。

Changed in version 6.5.0.2 of package base: 增加了环境变量修剪和 PLT_INFO_ALLOW_VARS 支持。

procedure

(find-relevant-directories syms [mode]) → (listof path?)

  syms : (listof symbol?)
  mode : (or/c 'preferred 'all-available 'no-planet 'no-user)
   = 'preferred
返回一个识别集合和安装的 PLaneT 包的路径列表,这些包的 "info.rkt" 文件定义了一个或多个给定的符号。 这个结果是基于 raco setup 所计算的缓存。

请注意,当你调用 get-info/full 时,缓存可能已经过期, 所以不要假设每个返回目录的 "info.rkt" 文件都会提供一个所需的符号。

结果是按规范顺序排列的(按目录名的词法排序),它返回的路径适合于提供给 get-info/full 。

如果指定了 mode ,它必须是 'preferred (默认), 'all-available, 'no-planet 或 'no-user 。 如果 mode 是 'all-available, find-relevant-directories 将返回所有已安装的目录, 这些目录的信息文件包含指定的符号–例如, 如果指定 'all-available ,所有已安装的 PLaneT 包的所有版本都将被搜索到。 如果 mode 是 'preferred, 那么只有 "preferred" 软件包的子集会被搜索到: 只有包含任何PLaneT软件包最新版本的目录会被返回。 如果 mode 是 'no-planet, 那么 PLaneT 软件包将不包括在搜索中。 如果 mode 是 'no-user, 那么只搜索整个安装目录,这意味着省略 PLaneT 软件包目录。

来自安装范围的 collection links file 或安装范围的软件包的链接被缓存在安装的主 "lib" 目录下, 而来自特定用户的 collection links file 和软件包的链接在所有版本的情况下被缓存在特定用户的目录 (build-path (find-system-path 'addon-dir) "collects") ， 在特定版本的情况下被缓存在 (build-path (find-system-path 'addon-dir) (version) "collects") 。

procedure

(find-relevant-directory-records syms key)

 → (listof directory-record?)
  syms : (listof symbol?)
  key : (or/c 'preferred 'all-available 'no-planet 'no-user)
与 find-relevant-directories 类似, 但返回 directory-record 结构而不是 path? 。

struct

(struct directory-record (maj min spec path syms)
    #:extra-constructor-name make-directory-record)
  maj : integer?
  min : integer?
  spec : any/c
  path : path?
  syms : (listof symbol?)
一个结构,它记录了关于一个集合或已经安装的 PLaneT 包的信息。 集合的主要版本是 1 ,次要版本是 0 。 spec 字段是一个带引号的模块规格; path 字段是这个集合或 PLaneT 包的 info.rkt 文件在文件系统中的位置; syms 字段持有该文件中定义的标识符。

procedure

(reset-relevant-directories-state!) → void?

重置 find-relevant-directories 使用的缓存。
