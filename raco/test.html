13 raco test: 运行测试

raco test 命令要求并运行与命令行上给出的每个路径相关的(默认) test 子模块。 命令行标志可以控制运行哪个子模块,如果没有找到子模块是否运行主模块, 以及是否直接运行测试,在单独的进程中(默认),或在单独的地方。 在运行文件之前,当前目录被设置为一个测试文件的目录。

当一个参数路径指向一个目录时, raco test 会递归地发现并运行该目录中所有以模块后缀 (见 get-module-suffixes, 但后缀总是包括 ".rkt", ".scrbl", ".ss" 和 ".scm") 结尾的文件, 或者有一个由 "info.rkt" 文件中 test-command-line-arguments 提供的命令行参数列表(可能为空), 或者按照 "info.rkt" 文件中 test-include-paths 指示。 同时, raco test 测试按照 "info.rkt" 文件中 test-omit-paths 的指示, 省略一个目录中的文件和目录。

如果一个测试通过 test-log! 记录失败的测试代码,导致 Racket 以非零退出代码退出, 或者(当指定 -e 或 --check-stderr 时)在错误端口产生输出,则被视为失败。

raco test 命令接受几个标志:

  • -c 或 --collection — 将参数解释为集合,其内容应被测试(与目录内容的方式相同), 并使 --process 成为默认的测试模式。

  • -p 或 --package — 将参数解释为包,其内容应被测试(与目录内容的方式相同)。 所有的包范围都会被搜索到第一个,最具体的 package scope 。 这个标志也使 --process 成为默认的测试模式。

  • -l 或 --lib — 参数解释为应该被测试的库。 每个参数 ‹arg› 被视为一个模块路径 (lib "@nonterm{arg}") 。 如果指定了一个模块,默认的测试模式是 --direct ,如果指定了多个模块,则是 --process 。

  • -m 或 --modules — 不仅将参数解释为路径(这是默认模式),而且将它们与在目录中找到的路径一样对待, 这意味着忽略没有模块扩展名的文件参数, 或者没有通过 "info.rkt" 文件中的 test-command-line-arguments 或 test-include-paths 明确启用的文件参数; 同时,可以通过 "info.rkt" 文件中的 test-omit-paths 禁用在其他方面启用的路径。 如果指定了一个单一的路径,默认的测试模式是 --direct , 如果指定了多个路径,则是 --process 。

  • --drdr — 配置默认值以模仿 DrDr 连续测试系统:忽略非模块, 在独立的进程中运行测试(除非指定 --thread 或 --direct ), 使用尽可能多的可用处理器(除非指定 --jobs ,将默认超时设置为 90 秒(除非指定 --timeout , 为每个测试创建一个新的 PLTUSERHOME 和 TMPDIR , 将 stderr 输出作为测试失败计算,使程序输出安静,提供空程序输入,并打印一个结果表。

  • -s ‹name› 或 --submodule ‹name› — 需要 submodule ‹name› 而不是 test 。 提供 -s 或 --submodule 来运行多个子模块,或者将多个子模块与 --first-avail 结合起来, 运行列出的模块中第一个可用的。请注意,如果你多次使用 -s ,但只提供一个模块文件作为参数, 默认模式仍然是 --direct (这可能意味着比 --process 或 --place 模式更少的新模块实例化)。

  • -r 或 --run-if-absent — 如果一个文件中没有相关的子模块,则要求使用顶级模块。这是默认模式。

  • -x 或 --no-run-if-absent — 如果相关的子模块不存在,则忽略一个文件。

  • --first-avail — 当用 -s 或 --submodule 提供多个子模块名称时,只运行第一个可用的子模块。

  • --configure-runtime — 在运行模块或子模块之前,运行每个指定模块的 configure-runtime 子模块(如果有的话)。 当只提供一个模块或指定 --process 或 --place 模式时,这种模式是默认的, 除非通过 -s 或 --submodule 提供一个子模块名称。

  • --direct — 在一个线程中运行每个测试,使用一个命名空间的模块注册表来加载所有的测试。 如果指定一个文件,这种模式是默认的。 多个测试可以通过退出、使用阻塞的不安全操作(从而防止超时)等方式相互干扰和整个测试运行。

  • --process — 在一个单独的操作系统进程中运行每个测试。 如果指定了多个文件,或者指定了一个目录、集合或包,这种模式是默认的。

  • --place — 在一个 place 运行每个测试, 而不是在一个操作系统进程中。

  • -j ‹n› 或 --jobs ‹n› — 并行运行最多 ‹n› 个测试文件。

  • --timeout ‹seconds› — 将默认的超时时间(超过这个时间,测试算作失败)设置为 ‹seconds› 。 使用 +inf.0 来允许测试无限制地运行,但允许 timeout 子模块配置。 如果任何测试由于超时而失败, raco test 的退出状态是 2(相对于只有非超时失败的 1 或成功的 0)。 如果没有通过 --timeout 或 --drdr 指定,默认超时与 +inf.0 相对应。

  • --fresh-user — 当在一个单独的进程中运行测试时,创建一个新的目录并设置 PLTUSERHOME 和 TMPDIR 。 PLTADDONDIR 环境变量也会被设置, 这样 add-on 目录(例如安装软件包的地方)就 不会 因为每个测试进程而改变。

  • --empty-stdin — 为每个测试程序提供一个空的 stdin。

  • -Q 或 --quiet-program — 抑制每个测试程序的输出。

  • -e 或 --check-stderr — 将任何 stderr 输出算作测试失败。

  • --deps — 如果将参数视为包,也要检查包的依赖性。

  • ++ignore-stderr ‹pattern› — 如果 stderr 输出与 ‹pattern› 相匹配,则不将其算作测试失败。 这个标志可以多次使用,只要 stderr 输出与任何一个 ‹pattern› 匹配,就被视为成功。

  • --errortrace — 在运行测试前动态加载 errortrace 。 注意,已经编译的文件将不包括跟踪信息。

  • -y 或 --make — 启用自动生成和更新编译后的 ".zo" 文件。 具体来说, (make-compilation-manager-load/use-compiled-handler) 的结果会在模块加载动作之前作为 current-load/use-compiled 的值安装。

  • -q 或 --quiet — 禁止输出进度信息、责任方和变化输出(见 负责任的一方和变化的输出记录) 。

  • --heartbeat — 在测试运行至少 5 秒后,定期报告该测试仍在运行。

  • --table 或 -t — 在所有测试后打印一个汇总表。如果一个测试使用了 rackunit, 或者如果一个测试至少使用了来自 rackunit/log 的 test-log! 来记录成功和失败, 该表会根据日志报告测试和失败次数。

  • ++arg ‹argument› — 将 ‹argument› 添加到被调用的测试模块的参数列表中, 以便被调用的模块在其 current-command-line-arguments 中看到 ‹argument› 。 这些参数与 "info.rkt" 中指定的任何参数结合在一起,由 test-command-line-arguments 。

  • ++args ‹arguments› — 与 ++arg 相同,但 ‹arguments› 被视为一个以空格分隔的要添加的参数列表。 要在一个典型的 shell 中使用这个标志指定多个参数 ‹arguments› 必须用引号括起来。

  • --output 或 -o ‹file› — 所有 stdout 和 stderr 输出保存到 ‹file› 。 如果目标 ‹file› 已经存在,它将被覆盖。

Changed in version 1.1 of package compiler-lib: 增加了 --heartbeat.
Changed in version 1.4: 改变了对模块后缀的识别,使用 get-module-suffixes, 这意味着可以识别 ".ss" 和 ".rkt" 。
Changed in version 1.5: 增加了 ++ignore-stderr 。
Changed in version 1.6: 增加了 ++arg 和 ++args 。
Changed in version 1.8: 增加了 --output 和 -o 。
Changed in version 1.11: 增加了 --make/-y 。
Changed in version 1.12: 增加了 --errortrace 。

13.1 子模块的测试配置

当 raco test 在一个子模块中运行一个测试时, config 子模块可以为运行测试提供额外的配置。 config 子模块应该使用 info 模块语言来定义以下标识:

  • timeout — 一个以秒为单位的实数,用于覆盖测试的默认超时,只在超时被启用时适用。

  • responsible — 一个字符串、符号或符号和字符串的列表,用于标识测试失败时应被通知的责任方。 参见 负责任的一方和变化的输出记录 。

  • lock-name — 一个字符串,命名一个锁文件,用于序列化测试(即,具有相同锁名称的测试不会同时运行)。 锁文件的位置由 PLTLOCKDIR 环境变量决定,或者默认为 (find-system-path 'temp-dir) 。 锁定文件的最长等待时间由 PLTLOCKTIME 环境变量决定,或者默认为 4 小时。

  • ignore-stderr — 一个字符串、字节串或 regexp value, 为一个模式,如果输出与该模式匹配,则导致错误输出不被视为失败。

  • random? — 如果为真,表示测试的输出预期会变化。见 负责任的一方和变化的输出记录 。

为了防止为测试目的评估一个文件,只需创建一个子模块,不执行任何测试,不触发包围模块的评估。因此,举例来说,一个文件可能看起来像这样:

#lang racket
 
(/ 1 0)
 
; don't run this file for testing:
(module test racket/base)

Changed in version 1.5 of package compiler-lib: 增加了 ignore-stderr 支持。

13.2 通过 "info.rkt" 进行测试配置

基于子模块的测试配置是首选(见 子模块的测试配置) 。 特别是,为了防止 raco test 运行一个特定的文件,通常该文件应该包含一个不采取任何行动的子模块。

然而,在某些情况下,添加一个子模块是不方便或不可能的(例如,因为文件不会总是编译)。 因此, raco test 也会查询候选测试文件目录中的任何 "info.rkt" 文件。 在一个集合中的文件的情况下,任何封闭的集合目录中的 "info.rkt" 文件也会被用于 test-omit-paths 和 test-include-paths 。 最后,对于一个包中的文件,包的 "info.rkt" 文件会被查阅 pkg-authors 以设置该包内所有文件的默认责任方(参见 负责任的一方和变化的输出记录)。

以下 "info.rkt" 字段是可以识别的:

  • test-omit-paths — 路径字符串的列表(相对于包围目录)和 regexp 值 (省略包围目录中与表达式匹配的所有文件),或者 'all 省略包围目录中的所有文件。 当一个路径字符串指向一个目录时,该目录中的所有文件都会被省略。

  • test-include-paths — 一个路径字符串的列表(相对于包围目录) 和 regexp 值(包括包围目录中与表达式匹配的所有文件), 或者 'all 包括包围目录中的所有文件。 当一个路径字符串指向一个目录时,该目录中的所有文件都被包括在内。

  • test-command-line-arguments — 一个 (list module-path-string (list argument-path-string ...)) 的列表,其中 current-command-line-arguments 被设置为一个向量, 在运行 module-path-string 时包含 argument-path-string 。

  • test-timeouts — 一个 (list module-path-string real-number) 的列表, 用来覆盖 module-path-string 的默认超时时间(秒)。

  • test-responsibles — (list module-path-string party) 或 (list 'all party) 的列表, 分别用于覆盖 module-path-string 或目录内所有文件的默认负责方(除非被覆盖)。 每个 party 是一个字符串、符号或符号和字符串的列表。参见 负责任的一方和变化的输出记录 。

  • test-lock-names — 一个 (list module-path-string lock-string) 的列表, 用于声明 module-path-string 的锁定文件名。 请看 子模块的测试配置 中的 lock-name 。

  • test-ignore-stderrs — 一个 (list module-path-string pattern) 或 (list 'all pattern) 的列表,用于声明标准错误输出的模式, 允许 module-path-string 或目录内的所有文件不出现故障。 每个 pattern 必须是一个字符串、字节字符串或 regexp value 。 见 子模块的测试配置 中的 ignore-stderr 。

  • test-randoms — 一个路径字符串的列表(相对于封闭的目录),用于输出不同的模块。 参见 负责任的一方和变化的输出记录 。

  • module-suffixes 和 doc-module-suffixes — 通过 get-module-suffixes 间接使用。

Changed in version 1.5 of package compiler-lib: 增加了 test-ignore-stderrs 支持。

13.3 负责任的一方和变化的输出记录

当一个测试有一个声明的责任方时,那么测试的输出会以

  raco test:‹which› @(test-responsible '‹responsible›)

行为前缀,其中 ‹which› 是一个空格,后面是一个精确的非负数,表示启用并行时的一个并行任务(否则为空), 而 ‹responsible› 是一个字符串、符号或列表数据集。

当一个测试的输出(写到 stdout )被期望在不同的运行中变化时–除了具有与 time 产生的相同形式的变化输出之外–那么它应该被声明为变化的。 在这种情况下,测试的输出以

  raco test:‹which› @(test-random #t)

行作为前缀。
