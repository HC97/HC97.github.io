1.6 语法转换器的编译管理器挂钩

 (require compiler/cm-accomplice) package: base

procedure

(register-external-file file    
  [#:indirect? indirect?]) → void?
  file : (and path? complete-path?)
  indirect? : any/c = #f
将一条消息(见 log-message) 记录到当前级别为 'info 的记录器中, 主题为 'cm-accomplice 。 消息数据是一个有两个字段的 file-dependency预制结构类型; 第一个字段的值是 file ,第二个字段的值是 #f (表示非模块依赖性)。 如果 indirect? 参数为真, 那么数据更确切的说是一个 file-dependency/options 预制结构类型的实例, 它是 file-dependency 的一个子类型,有一个额外的字段: 一个哈希表映射 'indirect 到 #t 。

由 compiler/cm 实现的编译管理器寻找这样的信息来注册一个外部依赖关系。 作为回应,编译管理器记录(在 ".dep" 文件中)该路径有助于当前正在编译的模块的实现。 之后,如果注册的文件被修改,编译管理器将知道重新编译该模块。 间接依赖对重新编译没有影响,但它可以向其他工具发出信号, 例如包-依赖检查器,该依赖是间接的(不应该暗示是直接的包依赖)。

例如, include 宏在扩展 include 形式时,用被包含文件的路径调用这个程序。

procedure

(register-external-module file    
  [#:indirect? indirect?]) → void?
  file : (and path? complete-path?)
  indirect? : any/c = #f
像 register-external-file 样, 但记录了一个带有 file-dependency 预制结构类型的信息,其第二个字段是 #t 。

由 compiler/cm 实现的编译管理器可以识别这个消息, 以注册一个模块的依赖关系(这意味着对该模块的所有依赖关系,等等)。
