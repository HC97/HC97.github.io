6.3 用 "info.rkt" 文件控制 raco setup

为了将一个集合的文件编译成字节码, raco setup 使用 compile-collection-zos 过程。 反过来,该过程会咨询集合的 "info.rkt" 文件,如果它存在的话,以获得编译集合的具体说明。 关于它所使用的 "info.rkt" 字段的更多信息,请参阅 compile-collection-zos ; 关于 "info.rkt" 文件的格式,请参阅 "info.rkt" File Format 文件格式。

其他字段由 Racket package manager 使用, 并在 包的元数据 中记录。 raco test 命令也识别额外的字段, 这些字段在 通过 "info.rkt" 进行测试配置 中有所记载。

可选的 "info.rkt" 字段会触发 raco setup 的额外动作:

  • scribblings : (listof (cons/c string? list?)) — 一个要建立的文件列表。列表中的每个文档本身表示为一个列表,其中每个文档的列表以一个字符串开始, 这个字符串是该文档源文件的集合相关路径。 文档名称(默认情况下是从源模块的名称派生出来的)旨在以与包或模块名称相同的方式实现全球唯一。

    更确切地说,一个 scribblings 条目必须是一个可以从匹配以下 entry 语法的表达式生成的值:

      entry = (list doc ...)
         
      doc = (list src-string)
      | (list src-string flags)
      | (list src-string flags category)
      | (list src-string flags category name)
      | (list src-string flags category name out-k)
      | (list src-string flags category name out-k order-n)
         
      flags = (list mode-symbol ...)
         
      category = (list category-string-or-symbol)
      | (list category-string-or-symbol sort-number)
         
      name = string
      | #f

    一个文档的列表可以选择继续提供关于如何构建该文档的信息。 如果一个文档的列表包含第二个项目,即 flags, 它必须是一个模式符号的列表(如下所述)。 如果一个文档的列表包含第三项, category, 它必须是一个对文档进行分类的列表(在下面进一步描述)。 如果一个文档的列表包含第四项, name, 它是一个用于生成文档的名字,而不是默认的源文件的名字(无扩展名), 其中 #f 意味着使用默认值;一个非 #f 的 name 值必须符合 collection-name-element? 检查的 collection-name 元素的语法。 如果一个文档的列表包含第五项, out-k, 它被用来提示文档的交叉引用信息所要使用的文件数量;见下文。 如果一个文档的列表包含第四项, order-n, 它被用来提示渲染的顺序;见下文。

    flags 中的每个模式符号可以是下列之一,其中只有 'multi-page 是常用的:

    • 'multi-page : 生成多页 HTML 输出,而不是默认的单页格式。

    • 'main-doc : 表示生成的文档应该被写入主安装目录,而不是一个用户特定的目录。 这种模式对于本身位于主安装目录下的集合来说是默认的。

    • 'user-doc : 表示生成的文档应该被写入一个用户特定的目录。 对于本身不在主安装中的集合,这种模式是默认的。

    • 'depends-all : 表示如果任何其他文档被重建, 该文档应该被重建–除了那些有 'no-depend-on 标志的文档。

    • 'depends-all-main : 表示如果任何其他文件被重建,并且被安装在主安装中, 那么该文件应该被重建–除了有 'no-depend-on 标志的文件之外。

    • 'depends-all-user : 表示如果任何其他文件被重建,并且被安装到用户的空间中, 该文件应该被重建–除了有 'no-depend-on 标志的文件。

    • 'always-run : 在每次运行 raco setup 的时候都建立文档,即使它的依赖关系没有改变。

    • 'no-depend-on : 移除该文件以考虑其他的依赖关系。 此外,从该文档到其他文档的引用总是直接的,而不是潜在的间接的(即,在文档浏览时解决,并可能被重定向到一个远程站点)。

    • 'main-doc-root : 指定了主要安装的根文件。 目前具有这种模式的文件应该是唯一具有这种模式的文件。

    • 'user-doc-root : 指定用户特定的文档目录的根文件。 目前拥有该模式的文档应该是唯一拥有该模式的文档。

    • 'keep-style : 保持文档的风格不变,而不是强加给手册的文档风格。

    • 'no-search : 建立没有搜索框的文档。

    • 'every-main-layer : 与 'main-doc 一起, 表示文档应该在每个安装层单独呈现( 分层安装)。

    category 列表指定如何在根目录中显示该文档。 列表必须以类别开始,这决定了手册在根文档页中出现的位置。 类别可以是一个字符串或一个符号。如果它是一个字符串,那么这个字符串就是根页面上的类别标签。 如果它是一个符号,那么就使用默认的类别标签。 可用的符号和根文档页上的类别顺序如下:

    • 'getting-started : 高水平的介绍性文件,与其他类别的标题排版相同。

    • 'language : 著名编程语言的文档。

    • 'tool : 一个可执行文件的文档。

    • 'gui-library : GUI 和图形库的文档。

    • 'net-library : 网络库的文档。

    • 'parsing-library : 解析库的文档。

    • 'tool-library : 编程工具库的文档 (也就是说,对于更突出的 'tool 类别来说,不够重要)。

    • 'interop : 互操作性工具和库的文档。

    • 所有字符串类别按 string<=? 排序。

    • 'library : 库的文档;这个类别是默认的,用于不被承认的类别符号。

    • 'drracket-plugin : DrRacket 插件文档。

    • 'legacy : 废弃的库、语言和工具的文档。

    • 'experimental : 实验性语言或库的文档。

    • 'other : 其他文档。

    • 'omit : 不应该在根页面上列出的文档,也不应该为了搜索而被索引。

    • 'omit-start : 不应该被列在根页面上的文档,但应该被索引以进行搜索。

    如果类别列表有第二个元素,它必须是一个实数,指定手册在类别中的排序位置;具有相同排序位置的手册是按字母顺序排列。 对于一对排序号为 n 和 m 的手册,如果 (truncate (/ n 10)) 和 (truncate (/ m 10)) 不同,手册的组别就用空格隔开。

    out-k 规范是关于是否将文档的交叉引用信息分成多个部分的提示, 这可以减少将交叉引用解析到文档中的时间和内存使用。 它必须是一个正的、精确的整数,默认是 1.

    order-n 规范是对文档构建的排序提示,因为文档引用可以是相互递归的。 顺序提示可以是任何实数。 如果数值为 -10 或更小,就不能与其他文档并行运行。 主 Racket 引用的值为 -11, 搜索页的值为 10, 默认值为 0 。

    预渲染文档的目录从源文件名开始计算,从 "info.rkt" 文件的目录开始,加上 "doc", 然后使用文档名(通常是不带后缀的源文件名); 如果这样的目录存在并且没有 "synced.rktd" 文件,那么它被视为预渲染文档并移动到位, 这种情况下文档源文件不需要存在。 将文档移动到地方可能根本不需要移动,这取决于包围集合的安装方式, 但移动包括添加一个 "synced.rktd" 文件来代表安装。

    Changed in version 6.4: 允许一个类别是一个字符串而不是一个符号。

  • release-note-files : (listof (cons/c string? (cons/c string? list?))) — 从主文档页面链接的 release-notes 文本文件的列表。 每个注释本身表示为一个列表,该列表可以指定与主注释分组的辅助性注释。

    一个 release-note-files 条目必须是一个可以从匹配以下 entry 语法的表达式中生成的值:

      entry = (list note ...)
         
      doc = (list label-string note-path)
      | (list label-string note-path order-integer)
      | 
    (list label-string note-path order-integer
          (list sub-note ...))
         
      sub-note = (list label-string note-path)

    order-integer 用于对笔记进行排序,默认为 0 。

  • racket-launcher-names : (listof string?) — 在安装的可执行目录中生成的可执行名称列表, 用于运行由集合实现的基于 Racket 的程序。 一个平行的库名列表必须由 racket-launcher-libraries 或 racket-launcher-flags 提供。

    对于每个名字,使用 make-racket-launcher 设置一个启动可执行文件。 参数是 -l- 和 ‹colls›/.../‹file›, 其中 ‹file› 是由 racket-launcher-libraries 命名的文件, ‹colls›/... 是 "info.rkt" 文件的集合(以及子集合)。

    此外、

    (build-aux-from-path
     (build-path (collection-path ‹colls› ...) ‹suffixless-file›))

    是提供给 make-racket-launcher 的可选 aux 参数(用于图标等), 其中 ‹suffixless-file› 是没有后缀的 ‹file› 。

    如果提供了 racket-launcher-flags ,它将被用作传递给 racket 的命令行参数列表, 而不是上述默认值,允许任意的命令行参数。 如果 racket-launcher-flags 和 racket-launcher-libraries 一起被指定, 那么 flags 将覆盖 libraries,但是 libraries 仍然可以用来为 build-aux-from-path 指定一个名字(查找相关信息,如图标文件等)。

  • racket-launcher-libraries : (listof path-string?) — 与 racket-launcher-names 并行的库名称列表。

  • racket-launcher-flags : (listof string?) — 一个命令行标志列表,与 racket-launcher-names 并行。

  • mzscheme-launcher-names, mzscheme-launcher-libraries 和 mzscheme-launcher-flags — 向后兼容的 racket-launcher-names 的变体,等等。

  • gracket-launcher-names : (listof string?) — 类似于 racket-launcher-names, 但用于基于 GRacket 的可执行文件。 启动器名称列表与 gracket-launcher-libraries 和 gracket-launcher-flags 并行处理。

  • gracket-launcher-libraries : (listof path-string?) — 一个与 gracket-launcher-names 并行的库名列表。

  • gracket-launcher-flags : (listof string?) — 一个命令行标志列表, 与 gracket-launcher-names 并行。

  • mred-launcher-names, mred-launcher-libraries 和 mred-launcher-flags — 向后兼容的 gracket-launcher-names 的变体,等等。

  • copy-foreign-libs : (listof (and/c path-string? relative-path?)) — 要复制到 ffi-lib 找到的外国库的目录中的文件。 如果定义了 install-platform ,那么只有在当前平台符合定义的情况下才会复制这些文件。

    在 Mac OS 上,当复制一个 Mach-O 文件时,如果复制的文件包括一个以 @loader_path/ 开头的库引用, 并且如果引用的库存在于 (get-lib-search-dirs) 列出的路径中的不同位置,那么库引用将被更新为一个绝对路径。

    在 Unix 上,当 ELF 文件被复制时,如果被复制的文件包括 $ORIGIN 的 RPATH 设置, 并且该文件被安装到一个用户特定的位置, 那么该文件的 RPATH 将被调整为 $ORIGIN: 后面是由 (find-lib-dir) 报告的主安装库目录的路径。

    在 Windows 上,如果文件正在使用,删除先前安装的外国库可能会因为文件上的锁而变得复杂。 为了弥补这一点, raco setup 在删除外国库文件时, 首先将该文件重命名为前缀 "raco-setup-delete-"; 然后尝试删除重命名的文件,如果未能删除重命名的文件,则只发出警告。 同时,在 raco setup 删除未安装的库的模式下, 它试图删除 foreign-library 目录中名称以 "raco-setup-delete-" 开头的任何文件(试图清理之前的失败)。

  • move-foreign-libs : (listof (and/c path-string? relative-path?)) — 像 copy-foreign-libs 一样, 但原始文件在被复制后会被删除(这对预编译包有意义)。

  • copy-shared-files : (listof (and/c path-string? relative-path?)) — 要复制到发现共享文件的目录的文件。 如果定义了 install-platform ,那么只有在当前平台符合定义的情况下才会复制文件。

    在 Windows 上,卸载文件的删除方式与 copy-foreign-libs 相同, 名称前缀 "raco-setup-delete-" 也同样特殊。

  • move-shared-files : (listof (and/c path-string? relative-path?)) — 和 copy-shared-files 一样, 但原始文件在被复制后会被删除(这对预编译的软件包是有意义的)。

  • copy-man-pages : (listof (and/c path-string? relative-path? filename-extension)) — 要复制到 man 目录的文件。 文件的后缀决定了它的类别;例如, .1 应该被用于描述可执行文件的 man 页面。

    在 Windows 上,卸载文件的删除方式与 copy-foreign-libs 相同, 名称前缀 "raco-setup-delete-" 也同样特殊。

  • move-man-pages : (listof (and/c path-string? relative-path? filename-extension)) — 和 copy-man-pages 一样,但原始文件在被复制后会被删除(这对预编译包有意义)。

  • install-platform : platform-spec? 如果此规范与当前平台相匹配,与此软件包相关的国外库将被复制或移动到有用的位置。 参见 copy-foreign-libs, move-foreign-libs, copy-shared-files 和 move-shared-files 。 也请参见 matching-platform? 以了解规范与 (system-type) 和 (system-library-subpath #f) 的比较方式。

  • install-collection : path-string? — 一个相对于提供 installer 的集合的库模块。 installer 必须接受一个、两个、三个或四个参数:

    • 第一个参数是 Racket 安装的 "collects" 目录的父目录路径。

    • 第二个参数,如果接受的话,是指向集合自身目录的路径。

    • 第三个参数,如果接受,是一个布尔值, 表示该集合是作为用户专用 (#t) 还是安装范围 (#f)安装的。

    • 第四个参数,如果接受的话,是一个布尔值,表示该集合是否以安装范围的方式安装,并且应该避免修改安装; 当参数为 #t 时,不接受这个参数的 installer 过程不会被调用。 一个接受此参数的安装程序在调用时,会以 #t 为参数, 这样它就可以执行用户特定的工作,即使该集合是安装在整个系统中。

  • pre-install-collection : path-string? — 与 install-collection 类似,只是相应的安装程序会在正常的 ".zo" 构建 before , 而不是之后被调用。提供的程序是 pre-installer, 所以它可以由提供 installer 程序的同一个文件提供。

  • post-install-collection : path-string? — 与 install-collection 类似,用于在执行 install-collection过程后立即调用的过程。 可以向 raco setup 提供 --no-install 标志来禁用 install-collection 和 pre-install-collection ,但不能禁用 post-install-collection 。 因此, post-install-collection 功能被期望执行总是需要的操作,即使是在包含预编译文件的安装后。 提供的程序是 post-installer ,所以它可以由提供 installer 程序的同一个文件来提供。

  • assume-virtual-sources : any/c — 真值表示没有相应源文件的字节码文件不应从 "已编译 "compiled" 目录中删除, 并且在向 raco setup 传递 --clean 或 -c 标志时不应删除任何文件。

  • clean : (listof path-string?) — 在向 raco setup 传递 --clean 或 -c 标志时要删除的路径名列表。 这些路径名必须是相对于集合的。 如果任何路径命名了一个目录,该目录中的每个文件都会被删除,但该目录的子目录都不会被检查。 如果路径命名了一个文件,则该文件被删除。 如果没有指定这个标志,默认情况是删除 "compiled" 子目录中的所有文件, 以及当前平台的已编译目录中特定平台的子目录中的所有文件。

    正如编译 ".zo" 文件将编译被编译模块使用的每个模块一样, 删除一个模块的编译镜像将删除被该模块使用的每个模块的 ".zo" 。 更具体地说,在删除 ".dep" 文件时,使用过的模块将被确定, 该文件是在 raco setup 或 raco make 构建 ".zo" 时为 ".zo" 文件创建的(见 相关性文件)。 如果 ".dep" 文件指示了另一个模块, 那么该模块的 ".zo" 只有在它也有一个附带的 ".dep" 文件时才会被删除。 在这种情况下, ".dep" 文件被删除,其他被使用的模块根据被使用模块的 ".dep" 文件被删除,等等。 向 raco setup 提供一个特定的集合列表,可以禁止这种基于依赖关系的编译文件的删除。

  • compile-omit-paths, compile-omit-files 和 compile-include-files — 通过 compile-collection-zos 间接使用。

  • module-suffixes 和 doc-module-suffixes — 通过 get-module-suffixes 间接使用。
