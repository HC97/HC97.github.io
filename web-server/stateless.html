3 无状态 Servlet

 #lang web-server/base package: web-server-doc
 #lang web-server

一个无状态的 servlet 应该 provide 以下输出:

value

interface-version : (one-of/c 'stateless)

这表明该 Servlet 是一个无状态的 Servlet。

value

stuffer : (stuffer/c serializable? bytes?)

这是用于 Servlet 的 stuffer 。

如果没有提供,则默认为 default-stuffer 。

value

manager : manager?

这是将用于 Servlet 的 manager 。

如果没有提供,则默认为 (create-none-manager #f) 。

procedure

(start initial-request) → response?

  initial-request : request?
当这个 servlet 的一个实例被启动时,这个函数被调用。 其参数是启动该实例的 HTTP 请求。

一个 'stateless servlet 模块的例子:
#lang web-server
(require web-server/http)
(provide interface-version stuffer start)
(define interface-version 'stateless)
(define stuffer
  (stuffer-chain
   serialize-stuffer
   (md5-stuffer (build-path (find-system-path 'home-dir) ".urls"))))
(define (start req)
  (response/xexpr
   `(html (body (h2 "Look ma, no state!")))))

web-server/base 语言导出了 racket/base 的所有函数和语法, 而没有其他的。

web-server 语言从以下库中输出所有的函数和语法: racket,net/url, web-server/http, web-server/http/bindings, web-server/lang/abort-resume, web-server/lang/web, web-server/lang/native, web-server/lang/web-param, web-server/lang/web-cells, web-server/lang/file-box, web-server/lang/soft, web-server/dispatch 和 web-server/stuffers 。 其中一些将在下面的小节中记录。

3.2 使用注意事项

一个无状态的 Servlet 会自动执行以下过程:

  • 所有对 letrec 的使用都被删除,取而代之的是对 let 和 imperative 特性的同等使用。

  • 程序被转换为 ANF (A-Normal Form), 使所有的延续性变得明确。

  • 所有的延续和延续标记都被记录在它们作为延续的表达式的延续标记中。

  • 所有对外部模块的调用都被识别和标记。

  • 所有对 call/cc 的使用都被删除,并通过前面安装的延续标记用等价的延续聚集来代替。

  • 程序被去功能化,每个 lambda 都有一个可序列化的数据结构。

这个过程允许你的 servlet 所捕获的连续体被序列化。这意味着它们可以被存储在客户的浏览器或服务器的磁盘上。

这意味着你的 servlet 除了执行之外,对服务器没有任何成本。 如果你使用 Racket servlet 并遇到内存问题,这就非常有吸引力。

这意味着你的服务器可以在一个长期运行的网络交互中重新启动,而不需要与客户端共享的 URL 过期。 如果你使用 Racket servlets 并遇到会话超时的问题,这就非常有吸引力。

这个过程被定义在所有的 Racket 上,并且发生在宏扩展之后,所以你可以自由地使用 Racket 的所有有趣的功能。 然而,有一些注意事项是你必须做的。

首先,这个过程极大地改变了你的程序的结构。 它将创建大量的 lambdas 和你的程序通常不包含的结构。 这对性能的影响还没有在 Racket 中得到研究。

其次,去功能化过程对你的程序的语法结构很敏感。 因此,如果你以一种微不足道的方式改变你的程序,例如改变一个常数, 那么所有序列化的连续体都将被淘汰,并且在尝试反序列化的时候会出错。 这是一个特点,而不是一个错误 这是为了保护你的程序在有意义的情况下不会出现各种错误而付出的小代价。 如果你使用 default-stuffer 或 web-server/stuffers/hash, 那么每当你改变 servlet 的代码时,你就可以安全地删除所有保存的延续,因为它们不会再被使用。

第三,你的连续体的词法范围内的值必须是可序列化的,这样连续体本身才是可序列化的。 这意味着你必须使用 define-serializable-struct 而不是 define-struct, 并注意使用同样的模块。 同样地,你也不能使用 parameterize, 因为参数化是不可序列化的。

第四,也是相关的,这个过程只在你的代码上运行,而不是在你 require 的代码上。 因此,你的延续必须是可序列化的,而不是在另一个模块的上下文中。 例如,下面的例子将以 "unsafe context" 的异常而失败:

(define requests
  (map (lambda (rg) (send/suspend/url rg))
       response-generators))
因为 map 没有被进程转化。 然而,如果你定义了自己的 map 函数,就不会有问题了。 另一个解决方案是用 serial->native 和 native->serial 在服务器上存储延续的 map 部分:
(define requests
  (serial->native
   (map (lambda (rg) (native->serial (send/suspend/url rg)))
       response-generators)))

第五,存储 没有 被序列化。 如果你依赖存储,你将承担巨大的风险。 你将假设在服务器重启或内存被垃圾回收之前,序列化的延续会在同一个服务器上被调用。

这个过程来源于 Pettyjohn 等人在 2005 年发表的论文 Continuations from Generalized Stack Inspection , Jay McCarthy 在 2009 年发表的 Automatically RESTful Web Applications, Or Marking Modular Serializable Continuations 以及 Jay McCarthy 在 2010 年发表的 The Two-State Solution: Native and Serializable Continuations Accord, 我们感谢 Greg Pettyjohn 对这个算法的最初实现。

3.3 可序列化的延续

 (require web-server/lang/abort-resume)
  package: web-server-lib
无状态语言的主要目的是为你的 Servlet 提供可序列化的延续。

procedure

(call-with-serializable-current-continuation response-generator)

 → any
  response-generator : (continuation? . -> . any)
以可序列化的方式捕获当前的延续,并用它调用 response-generator ,返回结果。

如果 serial->native 和 native->serial 被用来捕获不可转换的上下文, 这可能会使用当前 servlet 的 manager 的资源。

syntax

(serial->native expr)

serial->native 通知序列化运行时, expr 可能是对未转换上下文的调用。 这为 native->serial 设置了必要的信息, 以便向 call-with-serializable-current-continuation 发出信号, 捕获上下文的本地(因此是不可序列化的)部分并将其存储在服务器上。

syntax

(native->serial expr)

native->serial 通知序列化运行时, expr 标志着从未转换的上下文返回后的第一个表达式。 这捕获了未转换的上下文, 从而使 call-with-serializable-current-continuation 可以将其存储在服务器上并从可序列化的连续中引用它。

例如、
(build-list
 3
 (lambda (i)
   (call-with-serializable-current-continuation
    (lambda (k) (serialize k)))))
将在运行时失败,因为 build-list 没有被转换。然而、
(serial->native
 (build-list
  3
  (lambda (i)
    (native->serial
     (call-with-serializable-current-continuation
      (lambda (k) (serialize k)))))))
将会成功,并且 k 将引用当前 servlet manager 中的一个单元格, 该单元格存储了 build-list 中的延续部分。

3.4 本地接口

 (require web-server/lang/native) package: web-server-lib
在整个程序中使用 serial->native 和 native->serial 有时是很不方便的。 本模块提供了一个创建封装器的宏。

syntax

(define-native (native arg-spec ...) original)

 
  arg-spec : ho
  arg-spec : _
在 original 周围建立一个名为 native 的接口, 这样对 native 的调用就被包裹在 serial->native 中, 所有在 arg-spec 中标记为 ho 的参数都被假定为程序, 并被包裹在 native->serial 中。

举例来说、

(define-native (build-list/native _ ho) build-list)

相当于
(define (build-list/native fst snd)
  (serial->native
   (build-list
    fst
    (lambda args
      (native->serial
       (apply snd args))))))

3.5 无状态网络交互

 (require web-server/lang/web) package: web-server-lib

procedure

(send/suspend/url response-generator) → request?

  response-generator : (url? . -> . response?)
捕获当前的续集。序列化它并把它塞进一个 URL 。 用这个 URL 调用 response-generator ,并将响应传递给客户端。 如果 URL 被调用,请求将被返回给这个 continuation。

procedure

(send/suspend response-generator) → request?

  response-generator : (string? . -> . response?)
像 send/suspend/url 一样,但有一个字符串的 URL 表示。

procedure

(send/suspend/hidden response-generator) → request?

  response-generator : (url? xexpr/c . -> . response?)
捕获当前的续集。序列化它并将其塞入一个隐藏的 INPUT 表单元素。 用这个 URL 和表单字段调用 response-generator ,并将响应传递给客户端。 如果用包含隐藏表单的表单数据调用 URL,则请求被返回给这个 continuation。

procedure

(send/suspend/url/dispatch make-response) → any

  make-response : (((request? . -> . any) . -> . url?) . -> . response?)
用一个函数调用 make-response ,当用 request? 到 any/c 的过程调用时, 将生成一个 URL, 当调用时将用 request? 对象调用该函数并将结果返回给 send/suspend/dispatch 的调用者。

procedure

(send/suspend/dispatch make-response) → any

  make-response : (((request? . -> . any) . -> . string?) . -> . response?)
就像 send/suspend/url/dispatch ,但有一个字符串的 URL 表示。

procedure

(redirect/get [#:headers hs]) → request?

  hs : (listof header?) = empty
参见 web-server/servlet/web 。

Changed in version 1.3 of package web-server-lib: 增加了 hs 参数,并改为使用 see-other 而不是 temporarily 。

3.7 文件框

 (require web-server/lang/file-box) package: web-server-lib
如前所述,由于 Web 语言 servlet 的部署场景,依赖 Web 语言 servlet 中的存储是很危险的。 这个模块提供了一个简单的 API,以安全的方式替换盒子。

procedure

(file-box? v) → boolean?

  v : any/c
检查 v 是否是一个文件箱。

procedure

(file-box p v) → file-box?

  p : path-string?
  v : serializable?
创建一个存储在 p 的文件盒,其默认内容为 v 。

procedure

(file-unbox fb) → serializable?

  fb : file-box?
返回 fb 里面的值

procedure

(file-box-set? fb) → boolean?

  fb : file-box?
如果 fb 包含一个值,则返回 #t 。

procedure

(file-box-set! fb v) → void

  fb : file-box?
  v : serializable?
将 v 保存在由 fb 代表的文件中。

Warning: 如果你打算使用一个负载均衡器,请确保你的文件盒在一个共享介质上。

3.8 无状态的网络参数

 (require web-server/lang/web-param)
  package: web-server-lib
在网络语言中使用 parameterize 是不容易的。 这个模块以一种可序列化的方式提供了(大致)相同的功能。 像 Web 语言中其他可序列化的东西一样,它们对源代码的修改是敏感的。

syntax

(make-web-parameter default)

扩展到以 default 为默认值的web参数的定义。 web 参数是一个过程,当以零参数调用时, 返回 default 或调用的动态环境中最后一个 web-parameterize 的值。

procedure

(web-parameter? v) → boolean?

  v : any/c
检查 v 是否是一个 web-parameter 。

syntax

(web-parameterize ([web-parameter-expr value-expr] ...) expr ...)

运行 (begin expr ...) , 这样 web-parameter-expr 评估到的 web 参数就被绑定到 value-expr 上。 从 value-expr 的角度看,这就像 let 。

3.9 软状态

 (require web-server/lang/soft) package: web-server-lib
有时你想从一个无状态程序中引用一个大的数据结构,而不对数据结构进行序列化,增加序列化的大小。本模块为这种情况提供支持。

procedure

(soft-state? v) → boolean?

  v : any/c
确定 v 是否是一个软状态记录。

procedure

(make-soft-state thnk) → soft-state?

  thnk : (-> any/c)
创建一块由 thnk 计算的软状态。这个值是可序列化的。

procedure

(soft-state-ref ss) → any/c

  ss : soft-state?
提取与 ss 相关的值。 如果该值不可用(也许是因为垃圾回收、在未初始化的进程中反序列化等原因), 那么与 ss 相关的 thunk 被调用,该值被缓存。

syntax

(soft-state expr ...)

相当于 (make-soft-state (lambda () expr ...)) 。

下面是一个使用软状态的 Servlet 的例子:
#lang web-server
 
(provide interface-version start)
(define interface-version 'stateless)
 
(define softie
  (soft-state
   (printf "Doing a long computation...\n")
   (sleep 1)))
 
(define (start req)
  (soft-state-ref softie)
  (printf "Done\n")
  (start
   (send/suspend
    (lambda (k-url)
      (response/xexpr
       `(html (body (a ([href ,k-url]) "Done"))))))))

当这个被运行并且链接被点击了几次后,输出结果是:

$ plt-web-server -p 8080

Doing a long computation...

Done

Done

Done

Done

如果服务器被重新启动,或者 URL 中的主机名被改成不同的主机,代码相同,并且点击了该 URL:

^Cuser break

$ plt-web-server -p 8080

Doing a long computation...

Done

3.10 基本组合器

 (require web-server/stuffers) package: web-server-lib

web-server 语言提供了可序列化的延续。 序列化功能被抽象为 stuffers ,控制它的操作方式。 当你写一个无状态的 Servlet 时,你可以提供你自己的(用这些功能构建的)。

3.10.1 基本组合器

 (require web-server/stuffers/stuffer)
  package: web-server-lib

struct

(struct stuffer (in out)
    #:extra-constructor-name make-stuffer)
  in : (any/c . -> . any/c)
  out : (any/c . -> . any/c)
stuffer 本质上是在这个结构中捕获的一个可逆函数。以下情况应该成立:
(out (in x)) = x
(in (out x)) = x

procedure

(stuffer/c dom rng) → contract?

  dom : any/c
  rng : any/c
构建一个 stuffer 的合同,其中 in 有合同 (-> dom rng) ， out 有合同 (-> rng dom) 。

value

id-stuffer : (stuffer/c any/c any/c)

相同的 stuffer 。

procedure

(stuffer-compose g f) → (stuffer/c any/c any/c)

  g : (stuffer/c any/c any/c)
  f : (stuffer/c any/c any/c)
组合 f 和 g ,即对 in 应用 f 然后 g , 对 out 应用 g 然后 f 。

procedure

(stuffer-sequence f g) → (stuffer/c any/c any/c)

  f : (stuffer/c any/c any/c)
  g : (stuffer/c any/c any/c)
stuffer-compose ，参数互换。

procedure

(stuffer-if c f) → (stuffer/c bytes? bytes?)

  c : (bytes? . -> . boolean?)
  f : (stuffer/c bytes? bytes?)
创建一个 stuffer ,如果输入 in 的 c 为真,则用 f 进行填充。 同样地,如果在 in 时应用了 f , 则在 out时应用 f (通过添加一个字节来记录)。

procedure

(stuffer-chain x ...) → stuffer?

  x : (or/c stuffer? (bytes? . -> . boolean?))
将 stuffer-sequence 和 stuffer-if 应用于 x 的连续尾部。

3.10.5 密钥/值存储

web-server/stuffers/hash stuffers 依赖于一个键/值存储。

 (require web-server/stuffers/store)
  package: web-server-lib

struct

(struct store (write read)
    #:extra-constructor-name make-store)
  write : (bytes? bytes? . -> . void)
  read : (bytes? . -> . bytes?)
以下情况应该成立:

(begin (write k v) (read k)) = v

procedure

(dir-store root) → store?

  root : path-string?
一个将 key 键的值存储在一个文件中的存储,该文件位于
(build-path
 root
 (bytes->string/utf-8 key))

使用这个接口来创建 SQLite、CouchDB 或 BerkeleyDB 等数据库的存储应该很容易。

3.10.6 杂凑地址存储

 (require web-server/stuffers/hash) package: web-server-lib

value

hash-fun/c : contract?

相当于 (-> bytes? bytes?) 。

procedure

(hash-stuffer H store) → (stuffer/c bytes? bytes?)

  H : hash-fun/c
  store : store?
一个内容寻址的存储 stuffer ,它将输入的字节, input, 与键 (H input) 一起存储在 store 中并返回键。 同样地,在 out 的时候,原始的字节被查找出来。

procedure

(md5-stuffer root) → (stuffer/c bytes? bytes?)

  root : path-string?
相当于 (hash-stuffer md5 (dir-store root))

3.10.7 HMAC-SHA1 签名

 (require web-server/stuffers/hmac-sha1)
  package: web-server-lib

procedure

(HMAC-SHA1 kb db) → bytes?

  kb : bytes?
  db : bytes?
使用 kb 作为密钥对 db 执行 HMAC-SHA1 计算。 结果被保证为 20 个字节。 (你可以将其与 hash-stuffer 一起使用,但与 md5 相比,这样做没有什么价值)。

procedure

(HMAC-SHA1-stuffer kb) → (stuffer/c bytes? bytes?)

  kb : bytes?
一个使用 HMAC-SHA1 对输入进行签名的 stuffer ， kb 是密钥。 stuffer 的结果是预加在输入数据上的哈希值。 当 stuffer 反向运行时,它检查前 20 个字节是否为其余数据的正确字节。

Warning: 你应该把它和 base64-stuffer 组合起来,以获得URL安全的字节。
Warning: 在没有明确规定的情况下,用户有可能修改他们通过其他 stuffers 发送的连续数据。 这个 stuffer 允许 servlet 证明填充的数据确实是由 servlet 生成的。 因此,如果你不使用 hash-stuffer,你 应该 使用这个。
Warning: 这个 stuffer 没有 对数据进行任何加密,所以用户仍然可以观察到填充的值。

3.10.8 辅助工具

 (require web-server/lang/stuff-url)
  package: web-server-lib

procedure

(is-url-too-big? v) → boolean?

  v : bytes?
确定在当前 Servlet 的 URL 中塞入 v 是否会导致 URL 对 Internet Explorer 来说太大。 (IE 只支持 2048 个字符以内的 URL。)

procedure

(make-default-stuffer root) → (stuffer/c serializable? bytes?)

  root : path-string?
构建一个序列化的 stuffer ,如果 URL 过大,则进行压缩(并进行 base64 编码), 如果 URL 仍然过大,则将其存储在以 root 为根的 MD5 索引的数据库中。

相当于:
(stuffer-chain
 serialize-stuffer
 is-url-too-big?
 (stuffer-chain
  gzip-stuffer
  base64-stuffer)
 is-url-too-big?
 (md5-stuffer root))

value

default-stuffer : (stuffer/c serializable? bytes?)

相当于:
(make-default-stuffer
 (build-path
  (find-system-path 'home-dir)
  ".urls"))
