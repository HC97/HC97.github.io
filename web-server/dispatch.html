5 基于 URL 的调度

 (require web-server/dispatch) package: web-server-lib

该库允许在永久 URL 和请求处理程序之间建立双向映射。

这个库的灵感来自 (planet untyped/dispatch) 包。

5.1 使用 web-server/dispatch

假设你正在编写一个博客应用程序,并希望为网站的不同视图提供漂亮的 URL 。 你会定义一些 URL 调度规则,如下所示:

> (define-values (blog-dispatch blog-url)
    (dispatch-rules
     [("") list-posts]
     [("posts" (string-arg)) review-post]
     [("archive" (integer-arg) (integer-arg)) review-archive]
     [else list-posts]))

并定义你的请求处理程序,如下所示:
> (define (list-posts req) `(list-posts))
> (define (review-post req p) `(review-post ,p))
> (define (review-archive req y m) `(review-archive ,y ,m))

现在当一个请求被发送到你的应用程序时,它将被引导到适当的处理程序:
> (define (url->request u)
    (make-request #"GET" (string->url u) empty
                  (delay empty) #f "1.2.3.4" 80 "4.3.2.1"))
> (blog-dispatch
   (url->request "http://www.chrlsnchrg.com"))

'(list-posts)

> (blog-dispatch
   (url->request "http://www.chrlsnchrg.com/"))

'(list-posts)

> (blog-dispatch
   (url->request
    "http://www.chrlsnchrg.com/posts/Extracurricular-Activity"))

'(review-post "Extracurricular-Activity")

> (blog-dispatch
   (url->request "http://www.chrlsnchrg.com/archive/1984/10"))

'(review-archive 1984 10)

> (blog-dispatch
   (url->request "http://www.chrlsnchrg.com/contact"))

'(list-posts)

你也可以从过程调用中生成这些漂亮的 URL:
> (blog-url list-posts)

"/"

> (blog-url review-post "Another-Saturday-Night")

"/posts/Another-Saturday-Night"

> (blog-url review-archive 1984 11)

"/archive/1984/11"

在掌握了博客软件的世界后,你决定用 Sum.com 把无处不在的 Add-Two-Numbers.com 淘汰掉:
> (define-values (sum-dispatch sum-url)
    (dispatch-rules
     [((integer-arg) ...) sum]
     [else (lambda (req) (sum req empty))]))
> (define (sum req is)
    (apply + is))
> (sum-dispatch (url->request "http://www.sum.com/"))

0

> (sum-dispatch (url->request "http://www.sum.com/2"))

2

> (sum-dispatch (url->request "http://www.sum.com/2/3/4"))

9

> (sum-dispatch (url->request "http://www.sum.com/5/10/15/20"))

50

> (sum-url sum empty)

"/"

> (sum-url sum (list 1))

"/1"

> (sum-url sum (list 2 3 5 7))

"/2/3/5/7"

当你使用 web-server/dispatch 与 serve/servlet 时, 你几乎总是想使用 #:servlet-regexp 参数,其值为 "" 以捕获所有顶级请求。 然而,如果你也在为静态文件提供服务,请确保你在规则中不包括 else ,否则,文件系统服务器将永远看不到这些请求。

5.2 API 参考

syntax

(dispatch-rules
 dispatch-clause ...
 maybe-else-clause)
 
dispatch-clause = [dispatch-pattern maybe-method dispatch-fun]
     
dispatch-pattern = ()
  | (string . dispatch-pattern)
  | (bidi-match-expander ... . dispatch-pattern)
  | (bidi-match-expander . dispatch-pattern)
     
maybe-method = 
  | #:method method
     
method = pat
     
maybe-else-clause = 
  | [else else-fun]
 
  else-fun : (request? . -> . any)
  dispatch-fun : (request? any/c ... . -> . any)
返回两个值:第一个是一个具有契约 (-> request? any) 的调度函数, 根据第一个与请求的 URL(和方法)相匹配的 dispatch-pattern 调用适当的 dispatch-fun ; 第二个是一个具有契约 (-> procedure? any/c ... string?) 的 URL 生成函数, 使用作为其第一个参数的 dispatch-fun 的 dispatch-pattern 来生成一个 URL 。

如果 else-fun 被遗漏了,就会提供一个调用 (next-dispatcher) 的函数来向 Web 服务器发出信号, 表明这个 dispatcher 并不适用。

method 语法被用于 match 表达式, 以匹配传入请求对象的 request-method 部分。 然而,由于 HTTP 允许方法使用任何大小写,来自 request-method 的字节字符串被规范化为小写的字符串。 因此,有效的模式是这样的: "get","post", "head", (or "get" "post"), 等等。

如果 method 被遗漏,它被认为适用于没有方法的请求和 GET 方法。

syntax

(dispatch-rules+applies
 dispatch-clause ...
 maybe-else-clause)
和 dispatch-rules 一样,除了返回一个带有契约的第三个值 (-> request? boolean?) , 如果 dispatching 规则适用于该请求,则返回 #t ,否则返回 #f 。

syntax

(dispatch-case
 dispatch-clause ...
 maybe-else-clause)
返回一个由 dispatch-rules 描述的调度函数。

syntax

(dispatch-url
 [dispatch-pattern dispatch-fun]
 ...)
 
  dispatch-fun : (request? any/c ... . -> . any)
返回一个由 dispatch-rules 描述的 URL 生成函数。

procedure

(serve/dispatch dispatch) → void

  dispatch : (request? . -> . can-be-response?)
用 #:servlet-regexp 参数 (#rx"") 调用 serve/servlet , 从而使每个请求都由 dispatch 处理。

5.3 强制性调度容器

dispatch-rules 是纯函数式的。 这提供了一个更加声明性的接口,但抑制了一些编程和模块化模式。 Containers 在 dispatch-rules 上提供了一个强制性的覆盖。

procedure

(container? x) → boolean?

  x : any/c
识别 containers 。

syntax

(define-container container-id (dispatch-id url-id))

定义 container-id 为容器,以及 dispatch-id 为其调度功能, url-id 为其 URL 查找功能。

syntax

(dispatch-rules! container-expr [dispatch-pattern dispatch-fun] ...)

与 dispatch-rules 类似,但必须将模式添加到 container-expr 所指定的容器中。 新的规则会在已经在容器中的任何规则 之前 被查阅。

5.4 内置的 URL 模式

web-server/dispatch 内置了一些有用的 URL 组件模式。

syntax

(number-arg)

一个 bi-directional match expander , 从 URL 中解析出一个 number? ,并生成一个以数字编码为字符串的 URL 。

syntax

(integer-arg)

一个 bi-directional match expander , 从 URL 中解析一个 integer? ,并生成一个以整数编码为字符串的 URL 。

syntax

(real-arg)

一个 bi-directional match expander , 从 URL 中解析出一个 real? ,并生成一个以实数为编码的 URL,作为一个字符串。

syntax

(string-arg)

一个 bi-directional match expander , 从 URL 中解析一个 string? 并生成一个包含该字符串的 URL 。

syntax

(symbol-arg)

一个 bi-directional match expander , 从 URL 中解析一个 symbol? ,并生成一个以符号编码为字符串的 URL 。

5.5 扩展 web-server/dispatch

 (require web-server/dispatch/extend)
  package: web-server-lib

你可以通过定义 bi-directional match expanders 来创建新的 URL 组件模式。

syntax

(define-bidi-match-expander id in-xform out-xform)

将 id 绑定到一个 bi-directional match expander 上, 其中 in-xform 是一个匹配扩展器(由 define-match-expander 定义), 在解析 URL 时使用, out-xform 是一个在生成 URL 时使用。

in-xform 和 out-xform 都应该使用语法 (xform arg ... id) , 其中 arg 是针对 id 的,并且与 in-xform 和 out-xform 兼容。 id 通常由 dispatch-rules 自动提供。

syntax

bidi-match-going-in?

一个 syntax parameter , 由 bi-directional match expanders 使用, 以确定是否正在解析或生成一个 URL 。

当定义新的模式时,你可能会发现使用这些辅助函数很有用:

syntax

(define-coercion-match-expander id test? coerce)

将 id 绑定到一个匹配扩展器, 该扩展器将 (id x) 扩展为 (? test? (app coerce x)) (即用 test? 来确定模式是否匹配,用 coerce 来转换绑定。)

procedure

(make-coerce-safe? coerce) → (any/c . -> . boolean?)

  coerce : (any/c . -> . any/c)
返回一个函数,如果 coerce 不会抛出一个异常或在其输入上返回 #t ,则返回 #f 。

Examples:
> (define string->number? (make-coerce-safe? string->number))
> (string->number? "1")

#t

> (string->number? "1.2")

#t

> (string->number? "+inf.0")

#t

> (string->number? "one")

#f
