1 概述

尽管使用 FFI 不需要编写新的 C 语言代码,但它对 C 语言程序员所面临的安全和内存管理问题几乎没有提供任何保护。 一个 FFI 程序员必须特别注意跨越 Racket-C 分界线的数据的内存管理问题。 因此,本手册在许多方面依赖于 Inside: Racket C API　中的信息: Racket C API 中的信息,它定义了 Racket 与 C API 的一般交互方式。

由于使用 FFI 需要考虑许多安全问题,而 Racket 程序员通常可以忽略这些问题, 所以库的名字中包含了　unsafe　。 导入库的宏应该被视为声明你的代码本身是不安全的,因此在出现 bug 时可能导致严重的问题:你有责任提供一个安全的接口。 如果你的库提供了一个不安全的接口,那么它的名字中也应该有 unsafe 。

关于 Racket FFI 的动机和设计的更多信息,见 [Barzilay04]　。

1.1 库、C 类型和对象

要使用 FFI,你必须考虑到

  • 一个特定的库,你想从它那里访问一个函数或值、

  • 由文件导出的特定符号,以及

  • 输出符号的 C 级类型(通常是一个函数类型)。

库对应于一个后缀为　".dll", ".so" 或 ".dylib" (取决于平台)的文件, 或者它可能是 Mac OS 上 ".framework"　目录下的一个库。

知道库的名称和/或路径往往是使用FFI的最棘手的部分。 有时,当使用一个没有路径前缀或文件后缀的库名时,库文件可以被自动定位,特别是在 Unix 上。 请看 ffi-lib 的建议。

ffi-lib 函数获得一个库的句柄。 要提取库的出口,最简单的方法是使用 ffi/unsafe/define 库中的 define-ffi-definer :

#lang racket/base
(require ffi/unsafe
         ffi/unsafe/define)
 
(define-ffi-definer define-curses (ffi-lib "libcurses"))

这个 define-ffi-definer 声明引入了一种 define-curses 形式,用于将 Racket 名称与从 "libcurses"　中提取的值结合起来–根据平台的不同, "libcurses"　可能位于 "/usr/lib/libcurses.so"　处。

为了使用 define-curses,我们需要 "libcurses"　中的函数名称和 C 类型。我们将从使用以下函数开始:

  WINDOW* initscr(void);

  int waddstr(WINDOW *win, char *str);

  int wrefresh(WINDOW *win);

  int endwin(void);

我们使这些函数可以从 Racket 中调用,如下所示:

按照惯例,下划线前缀表示 C 类型的代表(如 _int)或这种代表的构造函数(如 _cpointer)。

(define _WINDOW-pointer (_cpointer 'WINDOW))
 
(define-curses initscr (_fun -> _WINDOW-pointer))
(define-curses waddstr (_fun _WINDOW-pointer _string -> _int))
(define-curses wrefresh (_fun _WINDOW-pointer -> _int))
(define-curses endwin (_fun -> _int))

_WINDOW-pointer 的定义通过 _cpointer　创建了一个反映 C 类型的 Racket 值, 它为一个指针类型创建了一个类型表示–通常是一个不透明的。 'WINDOW 参数可以是任何值,但根据惯例,我们使用一个与 C 基本类型相匹配的符号。

每个 define-curses 表单都使用给定的标识符作为库导出的名称和 Racket 标识符来绑定。 define-curses 的可选 #:c-id 子句可以指定一个与 Racket 标识符不同的库导出名称来绑定。 每个定义的 (_fun ... -> ...) 部分描述了导出函数的 C 类型,因为库文件并没有为其导出函数编码该信息。 在 -> 左边列出的类型是参数类型,而在 -> 右边的类型是结果类型。 预定义的 _int 类型自然对应于 int C 类型, 而 _string 在作为字符串读取时对应于　char* 类型。

此时, initscr, waddstr, wrefresh　和 endwin 是 Racket 函数的正常绑定(刚好调用 C 函数),因此它们可以从定义模块导出或直接调用:

(define win (initscr))
(void (waddstr win "Hello"))
(void (wrefresh win))
(sleep 1)
(void (endwin))

1.2 函数类型的钟声和口哨声

我们最初使用 waddstr 这样的函数是很草率的,因为我们忽略了返回代码。 C 函数经常返回错误代码,而检查它们是一件很麻烦的事情。 更好的方法是在 waddstr 绑定中建立检查,并在代码为非零时引发一个异常。

_fun 函数类型构造函数包括许多选项,以帮助将 C 函数转换为更漂亮的 Racket 函数。 我们可以使用其中的一些功能来将返回代码转换为 #<void> 或异常:

(define (check v who)
  (unless (zero? v)
    (error who "failed: ~a" v)))
 
(define-curses initscr (_fun -> _WINDOW-pointer))
(define-curses waddstr (_fun _WINDOW-pointer _string -> (r : _int)
                             -> (check r 'waddstr)))
(define-curses wrefresh (_fun _WINDOW-pointer -> (r : _int)
                              -> (check r 'wrefresh)))
(define-curses endwin (_fun -> (r : _int)
                            -> (check r 'endwin)))

使用 (r : _int) 作为结果类型,给 C 函数的结果取一个本地名称　r 。 这个名字随后被用于结果后处理表达式中,该表达式在 _fun 形式的第二个　-> 之后被指定。

1.3 旁注参数

为了从 "libcurses"　获得鼠标事件,我们必须通过 mousemask 函数明确地启用它们:

  typedef unsigned long mmask_t;

  #define BUTTON1_CLICKED 004L

  

  mmask_t mousemask(mmask_t newmask, mmask_t *oldmask);

在掩码中设置 BUTTON1_CLICKED 可以启用按钮点击事件。 同时, mousemask 通过将其安装到作为其第二个参数提供的指针中来返回当前掩码。

由于这种通过引用调用的接口在 C 语言中很常见, _fun 与 _ptr 形式合作,自动为通过引用的参数分配空间,并提取 C 语言函数放在那里的值。 给予提取的值名称,以便在后处理表达式中使用。 后处理表达式可以将通过引用的结果与函数的直接结果结合起来(在这种情况下,它报告的是实际支持的给定掩码的子集)。

(define _mmask_t _ulong)
(define-curses mousemask (_fun _mmask_t (o : (_ptr o _mmask_t))
                               -> (r : _mmask_t)
                               -> (values o r)))
(define BUTTON1_CLICKED 4)
 
(define-values (old supported) (mousemask BUTTON1_CLICKED))

1.4 C 结构

假设支持鼠标事件, "libcurses"　库通过 getmouse 报告它们, 它接受一个指向 MEVENT 结构的指针来填充鼠标事件信息:

  typedef struct {

     short id;

     int x, y, z;

     mmask_t bstate;

  } MEVENT;

  

  int getmouse(MEVENT *event);

为了处理 MEVENT 值,我们使用 define-cstruct:

(define-cstruct _MEVENT ([id _short]
                         [x _int]
                         [y _int]
                         [z _int]
                         [bstate _mmask_t]))

这个定义以　define-struct 绑定许多名字的方式绑定了许多名字: _MEVENT 是一个代表结构类型的 C 类型, _MEVENT-pointer 是一个代表指向 _MEVENT　的指针的 C 类型, make-MEVENT　构造一个 _MEVENT 值, MEVENT-x 从一个 _MEVENT 值中提取 x 字段,依此类推。

通过这个 C 结构声明,我们可以定义　getmouse　的函数类型。 最简单的方法是定义　getmouse 接受一个 _MEVENT-pointer, 然后在调用 getmouse 之前明确分配 _MEVENT 值:

(define-curses getmouse (_fun _MEVENT-pointer -> _int))
 
(define m (make-MEVENT 0 0 0 0 0))
(when (zero? (getmouse m))
  ; use m...
  ....)

对于一个更像 Racket 的函数,使用 (_ptr o _MEVENT) 和一个后处理表达式:

(define-curses getmouse (_fun (m : (_ptr o _MEVENT))
                              -> (r : _int)
                              -> (and (zero? r) m)))
 
(waddstr win (format "click me fast..."))
(wrefresh win)
(sleep 1)
 
(define m (getmouse))
(when m
  (waddstr win (format "at ~a,~a"
                       (MEVENT-x m)
                       (MEVENT-y m)))
  (wrefresh win)
  (sleep 1))
 
(endwin)

_MEVENT-pointer 和 _MEVENT　之间的区别很关键。 使用　(_ptr o _MEVENT-pointer) 将只为一个指向 MEVENT 结构的指针分配足够的空间, 这对于一个 MEVENT 结构来说是不够的。

1.5 指针和手动分配

为了从用户那里获得文本而不是鼠标点击 "libcurses"　提供了 wgetnstr:

  int wgetnstr(WINDOW *win, char *str, int n);

当 waddstr 的 char* 参数被当作一个无尾的字符串时, wgetnstr　的 char* 参数被当作一个缓冲区,其大小由最后的 int　参数表示。 C类型的 _string 对这种缓冲区不起作用。

从 Racket 接近这个函数的一种方法是以最原始的形式描述参数, 使用普通的 _pointer 作为 wgetnstr　的第二个参数:

(define-curses wgetnstr (_fun _WINDOW-pointer _pointer _int -> _int))

要调用这个原始版本的 wgetnstr, 需要分配内存,将其清零, 然后将大小减去 1(留出一个 nul 终止符)传给 wgetnstr:

(define SIZE 256)
(define buffer (malloc 'raw SIZE))
(memset buffer 0 SIZE)
 
(void (wgetnstr win buffer (sub1 SIZE)))

当 wgetnstr 返回时,它已经向　buffer　写入了字节数。 在这一点上,我们可以使用 cast 将值从一个原始指针转换为一个字符串:

(cast buffer _pointer _string)

通过 _string 类型的转换导致原始指针所引用的数据被复制(和 UTF-8 解码), 因此不再需要 buffer 所引用的内存。 用 (malloc 'raw ...)　分配的内存必须用 free 来释放:

(free buffer)

1.6 指针和 GC 管理的分配

我们可以用 (malloc 'raw ...) 来分配 buffer , 而不是用 (malloc 'atomic ...) 来分配它:

(define buffer (malloc 'atomic SIZE))

用 'atomic 分配的内存是由垃圾收集器管理的, 所以当 buffer 引用的内存不再需要时, free 既不需要也不允许。 相反,当 buffer 变得不可访问时,分配的内存将被自动回收。

让垃圾收集器(GC)来管理内存通常是比较好的。 很容易忘记调用 free ,而异常或线程终止也很容易跳过 free 。

同时,使用 GC 管理的内存给程序员增加了不同的负担: 由 GC 管理的数据可能会被移到一个新的地址,因为 GC 会压缩分配的对象以避免碎片化。 同时,C 函数希望收到指向对象的指针,这些指针将保持不变。

幸运的是,除非 C 函数回调到 Racket 运行时系统(也许是通过一个作为参数提供的函数), 否则在 C 函数被调用和函数返回之间不会发生垃圾收集。

让我们看一下与分配和指针有关的几种可能性:

  • Ok:

    (define p (malloc 'atomic SIZE))
    (wgetnstr win p (sub1 SIZE))

    虽然 malloc 分配的数据可以移动,但 p 总是指向它, 在从 p 中提取地址传递给 wgetnstr 和 wgetnstr 返回之间不会发生垃圾收集。

  • Bad:

    (define p (malloc 'atomic SIZE))
    (define i (cast p _pointer _intptr))
    (wgetnstr win (cast i _intptr _pointer) (sub1 SIZE))

    在地址被转换为整数后, p 引用的数据可能会移动,在这种情况下, i 投回指针将是错误的地址。

    很明显,将一个指针铸成一个整数通常是一个坏主意,但铸成后模拟了另一种可能性, 那就是将指针传递给一个 C 函数,该函数将指针保留在它自己的私有存储中供以后使用。 这样的私有存储对 Racket GC 来说是不可见的,所以它的效果与把指针投给整数是一样的。

  • Ok:

    (define p (malloc 'atomic SIZE))
    (define p2 (ptr-add p 4))
    (wgetnstr win p2 (- SIZE 5))

    指针 p2 保留了原来的引用, 只是在调用　wgetnstr 前的最后一刻(也就是在允许垃圾回收的时间点之后)添加了 4 。

  • Ok:

    (define p (malloc 'atomic-interior SIZE))
    (define i (cast p _pointer _intptr))
    (wgetnstr win (cast i _intptr _pointer) (sub1 SIZE))

    假设 p 本身保持可访问性,这样它所引用的数据就不会被回收,这就可以了。 用 'atomic-interior 分配的数据放在一个特定的地址上,并保持在那里。 垃圾回收不会改变 p　中的地址,因此 i (铸回指针)将始终引用数据。

请记住,像 make-MEVENT 这样的 C 结构构造函数实际上与 (malloc 'atomic ...)　相同; 在垃圾回收期间,结果值可以在内存中移动。 用 make-bytes　分配的字节串也是如此, (作为一种方便)可以直接作为指针值使用(不像字符串,它总是被复制用于 UTF-8 编码或解码)。

关于内存管理和垃圾回收的更多信息,请参阅 Inside: Racket C API　中的 Racket CS 的内存管理 和 Memory Allocation (BC) 。

1.7 可靠的资源释放

使用 GC 管理的内存可以省去对普通内存块的手动 free　,但 C 库经常会分配资源,并要求匹配调用释放资源的函数。 例如,　"libcurses" 支持屏幕上的窗口,这些窗口用 newwin 创建,用 delwin　释放:

  WINDOW *newwin(int lines, int ncols, int y, int x);

  int delwin(WINDOW *win);

在一个足够复杂的程序中,确保每一个 newwin　与 delwin 配对可能是一个挑战, 特别是当这些函数被其他安全的函数所包裹,而这些函数是由一个库提供的。 比如说,一个打算在沙盒中安全使用的库,必须防止在沙盒中的程序行为不端或被终止时,Racket 进程整体上的资源泄漏。

ffi/unsafe/alloc　库提供了连接资源分配函数和资源释放函数的功能。 如果资源在明确释放之前变得不可访问(根据 GC),该库就会安排最终化来释放资源。 同时,该库处理棘手的原子性要求,以确保最终化被正确地注册,并且永远不会运行多次。

使用 ffi/unsafe/alloc, newwin 和 delwin 函数可以分别用 allocator 和 deallocator 包装器导入:

(require ffi/unsafe/alloc)
 
(define-curses delwin (_fun _WINDOW-pointer -> _int)
  #:wrap (deallocator))
 
(define-curses newwin (_fun _int _int _int _int
                            -> _WINDOW-pointer)
  #:wrap (allocator delwin))

一个 deallocator 包装器使一个函数取消了该函数参数的任何现有 finalizer。 一个 allocator 包装器指的是去分配器,这样去分配器就可以在必要时被终结器运行。

如果一个资源是稀缺的,或者对终端用户是可见的, 那么 custodian 管理就比 allocator　实现的单纯的最终化更合适。 参见　ffi/unsafe/custodian 库。

1.8 线程和位置

尽管老版本的 "libcurses" 不是线程安全的,但 Racket 线程并不对应于操作系统级别的线程, 所以使用 Racket 线程来调用 "libcurses" 函数不会产生特别的问题。

然而,　Racket place　对应于操作系统级别的线程。 当库是线程安全的时候,从多个地方使用一个外国库是可行的。 从多个地方调用一个非线程安全的库需要更加小心。

从多个地方使用一个非线程安全的库的最简单的方法是指定　_fun　的 #:in-original-place? #t 选项, 它通过原始 Racket 地方而不是调用地方来引导对该函数的每一次调用。 我们最初使用的 "libcurses" 中的大多数函数都可以简单地变成线程安全的:

(define-curses initscr
  (_fun #:in-original-place? #t -> _WINDOW-pointer))
(define-curses wrefresh
  (_fun #:in-original-place? #t _WINDOW-pointer -> _int))
(define-curses endwin
  (_fun #:in-original-place? #t -> _int))

waddstr 函数就不那么简单了。

(define-curses waddstr
  (_fun #:in-original-place? #t _WINDOW-pointer _string -> _int))

的问题是,在 waddstr 调用完成之前, waddstr 的字符串参数可能会在调用的地方移动到原来的地方。 为了安全地调用 waddstr, 我们可以使用一个　_string/immobile 类型, 用 'atomic-interior　为字符串参数分配字节:

(define _string/immobile
  (make-ctype _pointer
              (lambda (s)
                (define bstr (cast s _string _bytes))
                (define len (bytes-length bstr))
                (define p (malloc 'atomic-interior len))
                (memcpy p bstr len)
                p)
              (lambda (p)
                (cast p _pointer _string))))
 
(define-curses waddstr
  (_fun #:in-original-place? #t _WINDOW-pointer _string/immobile -> _int))

注意,传递用 'interior 分配的内存(相对于 'atomic-interior) 只对读取内存而不写入的函数是安全的。 外来函数不能从分配内存的地方向　'interior　分配的内存写入,这是因为要对写入进行特定的地方处理,以实现世代垃圾收集。
