5.8 Atomic Execution

 (require ffi/unsafe/atomic) package: base

Atomic mode 评估一个 Racket 表达式,不需要在 Racket 线程之间切换,对事件的支持也很有限。 这种意义上的原子计算对于其他地方来说 不 是原子的, 而只是对于一个 places 的其他 threads 来说是原子的。

原子模式是 unsafe, 因为当执行处于原子模式时, Racket 调度器无法操作;调度器不能切换线程或轮询某些类型的事件,这可能导致死锁或其他线程的饥饿。 请注意,许多操作都可能涉及到这种同步,例如向输出端口写入。 即使一个输出目标是已知无同步的,也要注意值可以通过 prop:custom-write 附加任意的打印程序。 成功使用原子模式需要详细了解在原子模式下可能达到的任何实现,以确保其终止,不涉及同步。

(start-atomic) → void?

procedure

(end-atomic) → void?

在 Racket 线程层面上禁用/重新启用上下文切换,同时暂停/恢复中断异常的传递(与 break-enabled 的结果无关)。 对 start-atomic 和 end-atomic 的调用可以被嵌套。

请注意,将 start-atomic 和 end-atomic 与 dynamic-wind 配对,只有在以下情况下才有用

  • 当前的 exception handler 已知可以安全地逃离原子模式, 否则所有可能的逃离都是通过已知的继续跳转或中止(因为中断被禁用,没有其他异常可能)来安全逃离的;并且

  • 异常结构,如果有的话,避免在异常消息中打印数值, 否则总是使用 error value conversion handler ,并且已知是安全的原子模式。

使用 call-as-atomic 比使用 start-atomic 和 end-atomic 要安全一些, 因为 call-as-atomic 可以捕获异常并在退出原子模式后重新提出, 而且它用 call-as-nonatomic 包装了对错误值转换处理程序的任何调用。 然而,后者对于一个特定的原子区域是安全的,只有当该区域可以被一个非原子的异常构造安全地打断时。

与 call-as-atomic 不同, start-atomic 和 end-atomic 可以从任何操作系统线程中调用, 如 ffi/unsafe/os-thread 支持的那样,尽管在这种情况下调用没有效果。

也请看注意事项, atomic mode is unsafe 。

procedure

(start-breakable-atomic) → void?

procedure

(end-breakable-atomic) → void?

像 start-atomic 和 end-atomic 一样,但中断异常的传递不会被暂停。

这些函数并不比 start-atomic 和 end-atomic 快很多,所以在禁止中断的情况下,它们不会带来任何好处。

procedure

(call-as-atomic thunk) → any

  thunk : (-> any)
以原子模式调用 thunk ,其中 call-as-nonatomic 可以在调用的动态范围内使用, 以恢复到嵌套计算的非原子模式。

当在 call-as-atomic 的动态范围内使用 call-as-atomic 时, 那么 thunk 会被直接作为一个非尾巴调用。

如果 thunk 引发了一个异常,那么这个异常会被捕获并在退出原子模式后重新引发。 任何对当前 error value conversion handler 的调用都被 call-as-nonatomic 有效地包裹起来。

也请看注意事项, atomic mode is unsafe 。

procedure

(call-as-nonatomic thunk) → any

  thunk : (-> any)
在调用 call-as-atomic 的动态范围内,以非原子模式调用 thunk 。 请注意,在 thunk 的执行过程中,当前线程可能会被其他线程暂停或终止。

如果不是在调用 call-as-atomic 的动态范围内使用, call-as-nonatomic 会引发 exn:fail:contract 。

procedure

(in-atomic-mode?) → boolean?

当在 atomic mode 下(在当前 place 内)返回 #t , 否则返回 #f 。
