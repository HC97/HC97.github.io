5.10 Thread Scheduling

 (require ffi/unsafe/schedule) package: base
ffi/unsafe/schedule 库提供了与线程调度器合作并对其进行操作的函数。 该库的操作是不安全的,因为回调是在 atomic mode 和未指定的线程中运行。

Added in version 6.11.0.1 of package base.

procedure

(unsafe-poller poll) → any/c

  poll : (evt? (or/c #f any/c) . -> . (values (or/c #f list?) evt?))
产生一个允许作为 prop:evt 值的 poller 值,尽管它不是一个过程或本身不是一个 evt? 。 poll 回调在 atomic mode 下在一个未指定的线程中被调用, 以检查事件是否准备好了,或者允许它注册一个唤醒触发器。

poll 的第一个参数总是被用作 synchronizable event 的对象, poller 是其 prop:evt 值。 让我们把这个值称为 evt 。

poll 的第二个参数是 #f ,当调用 poll 来检查事件是否准备好了。 其结果必须是两个值。 如果 evt 准备好了,第一个结果值是一个结果列表,如果 evt 没有准备好, 它就是 #f 。 如果 evt 准备好了,第二个结果值是 #f , 或者如果 evt 没有准备好,它是一个替换 evt 的事件(通常只是 evt 本身)。

当线程调度器确定Racket进程应该睡到外部事件或超时时,那么就会调用 poll , 其第二个参数是非 #f ,即 wakeups 。 在这种情况下,如果第一个结果值是一个列表,那么睡眠将被取消,但列表不会被记录为结果(并且 poll 很可能会被再次调用)。 除了返回 #f 初始值, poll 还可以在 wakeups 上调用 unsafe-poll-ctx-fd-wakeup, unsafe-poll-ctx-eventmask-wakeup 和/或 unsafe-poll-ctx-milliseconds-wakeup ,以注册唤醒触发器。

procedure

(unsafe-poll-fd fd mode [socket?]) → boolean?

  fd : exact-integer?
  mode : '(read write)
  socket? : any/c = #t
检查给定的文件描述符或套接字当前是否准备好进行读取或写入,如 mode 所选。

Added in version 7.2.0.6 of package base.

procedure

(unsafe-poll-ctx-fd-wakeup wakeups fd mode) → void?

  wakeups : any/c
  fd : fixnum?
  mode : '(read write error)
注册一个文件描述符(Unix 和 Mac OS)或套接字(所有平台),以便在文件描述符或套接字准备好进行读写或报错时, 导致 Racket 进程被唤醒并恢复轮询,如 mode 所选。 wakeups 参数必须是一个非 #f 值,由调度器传递给 unsafe-poller 包裹的过程。

procedure

(unsafe-poll-ctx-eventmask-wakeup wakeups    
  mask) → void?
  wakeups : any/c
  mask : fixnum?
在 Windows 上,注册一个 eventmask,以使 Racket 进程在掩码选择的事件可用时被唤醒并恢复轮询。

procedure

(unsafe-poll-ctx-milliseconds-wakeup wakeups    
  msecs) → void?
  wakeups : any/c
  msecs : flonum?
导致 Racket 进程在 (current-inexact-monotonic-milliseconds) 开始返回 msecs 以上的值时唤醒并恢复轮询。

Changed in version 8.3.0.9 of package base: unsafe-poll-ctx-milliseconds-wakeup 之前使用 current-inexact-milliseconds。

procedure

(unsafe-set-sleep-in-thread! foreground-sleep    
  fd) → void?
  foreground-sleep : (-> any/c)
  fd : fixnum?
将 foreground-sleep 注册为一个程序,当线程调度器确定进程将睡眠时,为 Racket 进程实现睡眠。 同时,在调用 foreground-sleep 期间,调度器的默认睡眠函数将在一个单独的操作系统级线程中运行。 当这个默认的睡眠函数被唤醒时,一个字节被写入 fd , 作为通知 foreground-sleep 它应该立即返回的一种方式。

这个函数在 Racket 实现中的操作系统级线程可用时起作用。它对 Mac OS 总是有效的。

procedure

(unsafe-signal-received) → void?

通过 foreground-sleep 或其触发的东西与 unsafe-set-sleep-in-thread! 一起使用, 导致默认的睡眠函数要求 foreground-sleep 返回。

procedure

(unsafe-make-signal-received) → (-> void?)

返回一个类似于 unsafe-signal-received 的函数, 但它可以在任何 place 或 ffi/unsafe/os-thread 支持的任何 OS 线程中被调用,以确保线程调度器在调用 unsafe-make-signal-received 的 place 进行后续轮询。

unsafe-make-signal-received 的结果和调度器之间的同步将确保在调用 unsafe-make-signal-received 产生的函数之前相当于 (memory-order-release) , 在调度器调用轮询器之前相当于 (memory-order-acquire) 。

Added in version 8.0.0.4 of package base.
