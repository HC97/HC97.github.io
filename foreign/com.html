5.17 COM (通用对象模型)

ffi/com 和 ffi/unsafe/com 库支持两层的 COM 交互。 安全的上层提供了创建 COM 对象和动态构建基于 COM 自动化的方法调用的函数 (即由对象提供的反射信息)。 不安全的下层提供了一种语法形式和函数,用于更直接地与 COM 对象和接口工作。

一个 COM object 实例化了一个特定的 COM class 。 一个 COM class 类可以用两种方式中的一种来指定:

  • CLSID (类标识),它被表示为一个 GUID 。 GUID (全球唯一标识符)是一个 16 字节的结构。 GUID 通常以字符串形式书写,如 "{A3B0AF9E-2AB0-11D4-B6D2-0060089002FE}" 。 string->guid 和 guid->string 在字符串和 GUID 形式之间转换。 string->clsid 函数与 string->guid 相同, 但它的使用表明产生的 GUID 将被作为 CLSID 使用。

  • ProgID 是一个人类可读的名字,例如 "MzCom.MzObj.5.2.0.7", 其中包括一个版本号。 版本号可以在 ProgID 中省略,在这种情况下,会使用最新的可用版本。 操作系统提供了 ProgIDs 和 CLSIDs 之间的映射, 可以通过 progid->clsid 和 clsid->progid 获得。

一个 COM object 对象可以在本地机器上或在远程机器上被实例化。 后者依赖于操作系统的 DCOM (分布式 COM)支持。

每个 COM object 都支持一定数量 COM interfaces 。 一个 COM interface 有一个程序化的名字,比如 IDispatch 对应于一个C层的协议。 每个接口也有一个 IID (接口ID),表示为一个 GUID , 如 "{00020400-0000-0000-C000-000000000046}" 。 直接调用 COM 方法需要使用 QueryInterface 和所需的 IID 从一个对象中提取一个合适的接口指针; 其结果是有效地将其铸成一个指向调度表指针的指针,其中调度表有一个静态的已知大小和外来函数内容。 define-com-interface 形式简化了对接口指针的描述和使用。 COM 自动化层在内部使用固定数量的反射接口,特别是 IDispatch, 通过名称和安全的参数处理来调用方法。
