5.11 端口

 (require ffi/unsafe/port) package: base
ffi/unsafe/port 库提供了处理端口、文件描述符和套接字的功能。 该库的操作是不安全的,因为没有对文件描述符和套接字进行检查,而滥用文件描述符和套接字会破坏其他对象。

Added in version 6.11.0.4 of package base.

procedure

(unsafe-file-descriptor->port fd name mode)

 → (or/c port? (values input-port? output-port?))
  fd : exact-integer?
  name : any/c
  mode : (listof (or/c 'read 'write 'text 'regular-file))

procedure

(unsafe-socket->port socket name mode) → 
input-port? output-port?
  socket : exact-integer?
  name : bytes?
  mode : (listof (or/c 'no-close))
为给定的文件描述符或套接字返回一个输入端口和/或输出端口。 在 Windows 上,一个 "文件描述符" 对应于一个文件 HANDLE ,而一个套接字对应于一个 SOCKET 。 在 Unix 中,socket 是一个文件描述符,但是使用 socket 专用的 unsafe-socket->port 可以启用 socket 专用的功能,比如通过 tcp-addresses 报告地址。

name 参数决定了由 object-name 报告的端口的名称。 该 name 必须是 UTF-8 编码,被转换为套接字名称的符号。

对于文件描述符, mode 列表必须至少包括 'read 或 'write 中的一个, 如果 mode 包括 'read 和 'write ,则返回两个端口。 'text 模式只影响 Windows 端口。 'regular-file 模式表示文件描述符对应于一个普通的文件(它有一个属性,比如说,读取时从不阻塞)。 关闭所有返回的文件描述符端口会关闭文件描述符。

对于一个套接字, mode 列表可以包括 'no-close, 在这种情况下,关闭返回的两个端口不会关闭套接字。

对于任何类型的结果端口,关闭所产生的端口都会为文件描述符或套接字读出和取消注册任何信号, 这些信号是之前用 unsafe-file-descriptor->semaphore 或 unsafe-socket->semaphore 创建的。

procedure

(unsafe-port->file-descriptor p) → (or/c exact-integer? #f)

  p : port?

procedure

(unsafe-port->socket p) → (or/c exact-integer? #f)

  p : port?
如果 port 有一个文件描述符(在 Windows 中是一个 HANDLE 值), 则返回它的文件描述符,否则返回 #f 。

在 Unix 和 Mac OS 上,如果 unsafe-port->file-descriptor 的结果是 #f , 那么它对应的端口正在等待其对等物,正如 port-waiting-peer? 所报告的那样,例如没有连接阅读器的 fifo 的写入端。 通过使用 sync 等待,直到这种情况准备好。

Changed in version 7.4.0.5 of package base: 通过返回 #f 来适应一个被封锁在读卡器上的 fifo 写端。

procedure

(unsafe-file-descriptor->semaphore fd mode)

 → (or/c semaphore? #f)
  fd : exact-integer?
  mode : (or/c 'read 'write 'check-read 'check-write 'remove)

procedure

(unsafe-socket->semaphore socket mode) → (or/c semaphore? #f)

  socket : exact-integer?
  mode : (or/c 'read 'write 'check-read 'check-write 'remove)
返回一个 semaphore,当 fd 或 socket 准备好读或写时, 该 semaphore 就准备好了,如 mode 所选。 具体来说,这些函数提供了一个一次性的、 edge-triggered 的指示器; 当以下任何一种情况 first time 成立时,信号就会被发布:

  • fd 或 socket 已经准备好进行读写(取决于 mode),

  • unsafe-file-descriptor->port 或 unsafe-socket->port 从 fd 或 socket 创建了端口,并且这些端口被关闭,或

  • 在相同的 fd 或 socket 下发生了后续的调用,并且使用了 'remove mode 。

如果当前平台或给定的文件描述符或套接字不支持转换为信号灯,则结果为 #f 。

'check-read 和 'check-write 模式类似于 'read 和 'write, 但是如果在指定的模式下没有为指定的文件描述符或 socket 生成信号,那么结果就是 #f 。

'remove 模式为指定的文件描述符或套接字读出并解除任何先前创建的信号灯的注册。 Semaphores 必须在文件描述符或套接字被关闭之前被取消注册。 注意,从 unsafe-file-descriptor->port 或 unsafe-socket->port 关闭一个端口也会准备好并解除对信号灯的注册。 然而,在所有这些情况下,信号灯都是异步就绪的,所以可能会有一个可察觉的延迟。

procedure

(unsafe-fd->evt fd mode [socket?]) → (or/c evt? #f)

  fd : exact-integer?
  mode : (or/c 'read 'write 'check-read 'check-write 'remove)
  socket? : any/c = #t
返回一个事件,当 fd 准备好进行读或写时,正如 mode 所选择的那样,该事件已经就绪。 具体来说,它返回一个多用的、 level-triggered 的指示器; 以下任何一种情况发生时,事件都已准备就绪:

  • fd 已经准备好进行读或写(取决于 mode),

  • 后续的调用发生在相同的 fd 和 'remove mode 模式下(一旦移除,该事件就永远准备好了)。

事件的同步结果就是事件本身。

'check-read 和 'check-write 模式就像 'read 和 'write, 但是如果在指定的模式下还没有为指定的文件描述符或 socket 生成事件,那么结果就是 #f 。

'remove 模式为指定的文件描述符或套接字准备并取消先前创建的任何事件的注册。 事件必须在文件描述符或套接字被关闭之前被取消注册。 与 unsafe-file-descriptor->semaphore 和 unsafe-socket->semaphore 的 semaphore 结果不同, unsafe-fd->evt 的事件结果不会因为关闭端口而被触发或取消注册, 即使是 unsafe-file-descriptor->port 或 unsafe-socket->port 的端口。

Added in version 7.2.0.6 of package base.
