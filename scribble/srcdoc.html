4.5 源码内文档

scribble/srcdoc 和 scribble/extract 库支持在记录的代码中与导出的契约一起编写文档, 类似于使用 JavaDoc 。 通过这种方法,一个单一的契约规范既可以用于运行时契约,也可以用于导出绑定的文档。

scribble/srcdoc 库提供了导出绑定和相关文档的形式。 scribble/extract 库用来把基于 scribble/srcdoc 的文档拉到一个Scribble文档中(也许是多个库的文档)。

尽管在使用 scribble/srcdoc 时,文档是和库的实现一起写的,但文档没有给库带来运行时的开销。 同样地,排版文档也不需要运行库。 这两个阶段(运行时间与文档时间)是分开的,就像模块系统将扩展时间的代码与运行时间的代码分开一样, 文档信息被记录在一个子模块中,可以从包围的模块中单独加载。

关于使用的例子,请看 blog.racket-lang.org 上的 this post 。

4.5.1 文档的源注释

 (require scribble/srcdoc) package: scribble-lib

由 scribble/srcdoc 表单生成的文档信息被累积到 srcdoc 子模块中。 生成的子模块被 scribble/extract 的绑定所访问。

syntax

(for-doc require-spec ...)

一个 require 子表单用于在文档时间(和文档-扩展时间等)而不是运行时间(和扩展时间等)需要的绑定。 for-doc 导入对库的正常使用没有影响,它只影响文档提取。

通常,一个使用 scribble/srcdoc 的库至少包括 (require (for-doc scribble/base scribble/manual)) , 以获得核心 Racket 形式和基本的 Scribble 函数,用于文档表达。

每个 require-spec 都是在相对于包围的子模块中使用。 要访问包围模块的其他子模块,使用模块路径的形式 (submod ".." name ...) 。

syntax

(proc-doc/names id contract arg-specs (desc-expr ...))

 
arg-specs = ((arg-id ...) ((arg-id default-expr) ...))
  | (arg-id ...)
     
contract = (-> arg ... result)
  | (->* (mandatory ...) (optional ...) result)
  | (case-> (-> arg ... result) ...)
     
mandatory = contract-expr
  | keyword contract-expr
     
optional = contract-expr
  | keyword contract-expr
一个 provide 子表单,像使用 contract-out 一样, 用 contract 描述的合同输出 id 。

arg-spec 指定了参数的名称和缺省值,这些参数通常不作为契约的一部分来写。 它们与契约表达式相结合,通过 defproc 生成文档中的绑定描述。 (arg-id default-expr) 对指定了可选参数的名称和默认值。 如果契约支持可选参数,那么必须使用第一个 arg-spec 形式,否则必须使用第二个。

The desc-expr 是一串文档时间表达式, 它产生描述导出的绑定的散文–也就是生成的 defproc 的最后一部分, 所以描述可以使用 racket 指向 arg-id 。

在生成文档时,包围库的正常 require 被有效地转换为 for-label require , 因此 contract 中的标识符被链接到它们相应的文档。 同样地,任何在包围库的运行时阶段可用的绑定都可以使 racket 形式在文档散文中被引用。

syntax

(proc-doc id contract maybe-defs (desc-expr ...))

 
contract = (-> result)
  | (->i (arg ...) (opt ...) maybe-pre [id res])
  | (->i (arg ...) (opt ...) maybe-pre (values [id res] ...))
  | (->i (arg ...) (opt ...) #:rest rest [id result-expr])
  | (->d (arg ...) () maybe-pre (values [id result] ...))
  | (->d (arg ...) () maybe-pre [id result])
  | (->d (arg ...) () #:rest id rest [id result])
     
maybe-pre = 
  | #:pre (pre-id ...) condition
     
maybe-defs = 
  | (default-expr default-expr ...)
像 proc-doc/names, 但支持嵌入参数标识符的契约形式。 目前只支持 ->i 和 ->d 形式的一个子集。

如果可选的参数序列, (opt ...) 是空的,那么 maybe-arg-desc 必须不存在。 如果它是非空的,那么它必须有和可选参数一样多的默认表达式。

syntax

(thing-doc id contract-expr (desc-expr ...))

像 proc-doc 一样,但用于导出一个任意值。

syntax

(parameter-doc id (parameter/c contract-expr) arg-id (desc-expr ...))

像 proc-doc 一样,但用于导出一个参数。

syntax

(struct*-doc struct-name
             ([field-name contract-expr-datum] ...)
             maybe-omit-constructor
             maybe-mutable maybe-non-opaque maybe-constructor
             (desc-expr ...))
 
maybe-omit-constructor = 
  | #:omit-constructor
像 proc-doc 一样,但用于使用 struct 的结构声明。

maybe-mutable, maybe-non-opaque 和 maybe-constructor 选项与 defstruct 一样。

syntax

(struct-doc struct-name
            ([field-name contract-expr-datum] ...)
            maybe-omit-constructor
            maybe-mutable maybe-non-opaque maybe-constructor
            (desc-expr ...))
与 struct*-doc 类似,但适用于使用 define-struct 的结构声明。

syntax

(form-doc options form-datum
  maybe-grammar maybe-contracts
  (desc-expr ...))
 
options = maybe-kind maybe-link maybe-id maybe-literals
     
maybe-kind = 
  | #:kind kind-string-expr
     
maybe-link = 
  | #:link-target? link-target?-expr
     
maybe-id = 
  | #:id id
  | #:id [id id-expr]
     
maybe-literals = 
  | #:literals (literal-id ...)
     
maybe-grammar = 
  | #:grammar ([nonterm-id clause-datum ...+] ...)
     
maybe-contracts = 
  | 
#:contracts ([subform-datum contract-expr-datum]
             ...)
与 proc-doc 类似,但适用于语法形式的输出。 如果提供了 #:id ,那么 id 就是导出的标识符, 否则导出的标识符将从 form-datum 中提取。

关于 options, form-datum, maybe-grammar 和 maybe-contracts 的信息见 defform 。

Added in version 1.6 of package scribble-lib.

syntax

(class*-doc id super (intf-id ...) pre-flow)

像 proc-doc 一样,但对于使用 class* 的类声明。

id, super 和 intf-id 表达式的含义与 defclass 相同。

Added in version 1.30 of package scribble-lib.

syntax

(class-doc id super pre-flow)

像 class*-doc 一样,但适用于使用 class 的类声明,省略了 interface-expr 。

id 和 super 表达式的含义与 defclass 中相同。

Added in version 1.30 of package scribble-lib.

syntax

(begin-for-doc form ...)

与 begin-for-syntax 类似,但用于文档时间而不是扩展时间。 form 可以参考 for-doc 所 require 的绑定。

例如, begin-for-doc 中的定义可以被 proc-doc/names 中的 desc-expr 所引用。

syntax

(generate-delayed-documents)

导致文档信息被记录为一个宏, 在使用 include-extracted 或 provide-extracted 的模块中被展开 (连同任何 for-doc 导入),而不是在声明信息的模块(子模块)中。

以这种方式延迟文档的生成,允许 (for-doc (for-label ....)) 导入,否则会产生循环的模块依赖关系。

为了避免跨模块的累积 for-doc 导入问题, generate-delayed-documents 声明应该出现在任何 for-doc 导入之前。

syntax

(require/doc require-spec ...)

(require (for-doc require-spec ...)) 的传统速记。

syntax

(provide/doc spec ...)

一个替代 (provide spec ...) 的遗留问题。

4.5.2 从源代码中提取文档

 (require scribble/extract) package: scribble-lib

syntax

(include-extracted module-path)

扩展到从 module-path 中提取的一连串文档形式, 预计是一个使用 scribble/srcdoc 的模块(因此该模块有一个 srcdoc 子模块)。

syntax

(provide-extracted module-path)

与 include-extracted 类似,但文档被打包并作为 exported 导出,而不是留在内联。

当来自单一来源的文档要在多个文档位置中分割和排版时, 请将这种形式与 include-previously-extracted 结合使用。 provide-extracted 形式提取一次文档, 然后 include-previously-extracted 形式根据需要提取特定绑定的文档。

syntax

(include-previously-extracted module-path regexp)

与 include-extracted 类似, 但 module-path 指的是使用 provide-extracted 的模块,而不是指包含其自身文档的源代码。 include-previously-extracted 形式扩展到所有标识符的文档形式,其字符串形式与 regexp 匹配。
