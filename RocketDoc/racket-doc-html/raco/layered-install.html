6.17 分层安装

典型 的Racket 配置包括两层:一个 installation 层和一个 user 层。 其目的是 installation 层对系统的所有用户都是只读的, 而 user 层允许每个用户安装额外的包来扩展 installation 层。 installation 层的目的不仅是只读,而且在用户开始在自己的层中安装后不会改变。

在一个 Racket 本身正在开发的环境中, installation 层会发生变化。 在这种情况下,如果 user 层被使用,必须注意在修改安装层时不要给用户层造成冲突,否则用户层必须偶尔被修复。

默认情况下, raco setup 在运行时都会更新这两个层; 如果用户对安装层没有写入权限,没有参数的 raco setup 几乎肯定会报告权限错误。 raco setup 的动作可以通过提供 --avoid-main 参数限制在 user 层, 或者通过使用 --no-user 参数将 raco setup 限制在 installation 层。 当 raco pkg 执行设置动作时, 它有效地根据软件包的范围提供了其中的一个(并且 raco pkg 拒绝同时在两个范围/层上操作)。

user 层总是既针对用户又针对版本。 更确切地说,它是特定于用户和一个安装的名字的,其中安装的名字通常是它的版本号。 然而,一个安装的名称可以通过 "config.rktd" 中的 'installation 设置来改变(见 安装配置和搜索路径)。 设置一个安装名称可以改变软件包和可执行文件所在的目录 (find-system-path 'addon-dir) 。 (find-system-path 'addon-dir) 的结果本身可以通过 "config.rktd" 中的 'addon-dir 来改变。

通过在 "config.rktd" 中设置搜索路径, installation 层和 user 配置层可以被泛化为多个层。 这些搜索路径基本上将最接近 user 的层视为 installation 层, 可能会被 raco setup 和 raco pkg 调整,但搜索路径可以与现有的(不变的)实现连锁, 就像 user 与 installation 连锁一样。 要建立一个新的层,创建新的 "config.rktd" ,就像底层的 "config.rktd", 但

  • 'lib-dir, 'share-dir, 'links-file, 'pkgs-dir, 'bin-dir, 'gui-bin-dir, 'apps-dir, 'doc-dir 和 'man-dir 中的每一个都是一个新的目录或文件;并且

  • 相应的搜索列表 'lib-search-dirs, 'share-search-dirs, 'links-search-files, 'pkgs-search-dirs, 'bin-search-dirs, 'gui-bin-search-dirs, (不需要搜索 'apps-dir), 'doc-search-dirs 'man-search-dirs 都将旧目录或文件加入 #f 之后的搜索列表; 注意,每个搜索列表的默认是 (list #f).

raco setup 没有类似于 --avoid-main 的参数来避免修改嵌套层; 相反,嵌套层应该是完全设置好的,所以 raco setup 不需要改变它们。 当 raco setup 要将一个可执行文件安装到配置为 'bin-dir 的目录中时, 它会查询 'bin-search-dirs 列表,检查该可执行文件是否已经安装在这些目录中, 如果是,它将避免在新层中创建一个副本。 同样的搜索列表检查也适用于本地库、共享文件和手册,但还需要检查要安装的文件是否与已经安装的文件相匹配。

"config.rktd" 的默认路径在 racket 可执行文件中是硬性规定的。 在某些情况下,最内层的配置指向另一层是有意义的,也许是因为文件系统提供了一个转接。 例如,在 Unix 中,Racket 安装在 "/usr" 中, 可能会合理地将 installation 层的目录配置为 "/usr/local" , 并将 "/usr" 目录包含在搜索列表中。

要使用带有新的 "config.rktd" 的 racket , 你可以给 racket 提供 --config 或 --G 标志, 或者设置 PLTCONFIGDIR 环境变量指向包含 "config.rktd" 的目录。 另外,你可以创建一个 tethered 层,创建像 racket 这样的替换可执行文件,硬连接到该层的配置目录。
