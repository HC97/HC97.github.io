6.1 运行 raco setup

在没有命令行参数的情况下, raco setup 会找到当前所有的集合— 见 Libraries and Collections—并编译每个集合中的库。 (名为 ".git" 或 ".svn" 的目录不被视为集合。)

要将 raco setup 限制在一组集合中,请提供集合名称作为参数。 例如, raco setup scribblings/raco 将只编译和渲染 raco 的文档, 它是在 "scribblings/raco" 集合中实现的。

集合中可选的 "info.rkt" 可以具体说明如何编译集合的文件, 以及在设置集合时要采取的其他行动,比如创建可执行文件或构建文档。 更多信息请参见 用 "info.rkt" 文件控制 raco setup 。

raco setup 命令接受以下命令行标志:

  • 限制到指定的集合或 PLaneT 包:

    • --only — 将设置限制在指定的集合和 PLaneT 包上,即使没有指定。 如果任何集合被指定为命令行参数,或者通过 -l, --pkgs 或 -P 标志,这种模式是默认的。

    • -l ‹collection› ... — 将设置动作限制在指定的 ‹collection› 上 (也就是说,与提供没有标志的 ‹collections› 相同,但没有可能将 ‹collection› 解释为一个标志)。

    • --pkgs ‹pkg› ... — 将设置动作限制在指定的 ‹pkg› 内(或部分在)的集合上。

    • -P ‹owner› ‹package-name› ‹maj› ‹min› — 将设置动作限制在指定的 PLaneT 包上,以及任何其他指定的 PLaneT 包或集合。

    • --doc-index — 构建实现文档索引的集合(当文档构建被启用时),以及指定的集合。

    • --tidy — 删除不存在的集合的元数据缓存信息和文档,或者在删除后清理的文档, 即使设置动作被限制在指定的集合中。 尽管整理不限于指定的集合,但可以用 --avoid-main 或 --no-user 来限制它。

  • 限制于特定的任务:

    • --clean 或 -c — 删除现有的 ".zo" 文件,从而确保从源文件进行干净的构建。 被删除的文件的确切集合可以由 "info.rkt" 控制;更多信息见 clean 。 除非同时指定了 --no-info-domain 或 -d ,否则 "info.rkt" 缓存将被清除。 除非还指定了 --no-docs 或 -D ,否则文档索引数据库会被重置。

    • --fast-clean — 类似于 --clean, 但不强制从源码启动 raco setup (这意味着 --fast-clean 不能清除影响 raco setup 本身的损坏)。

    • --no-zo 或 -n — 避免将源文件编译成 ".zo" 文件。

    • --trust-zos — 修复 ".zo" 文件的时间戳,假定它们已经是最新的了 (除非 PLT_COMPILED_FILE_CHECK 环境变量被设置为 exists, 在这种情况下,时间戳被忽略)。

    • --recompile-only — 不允许从源代码重新编译模块,强加给每个 ".zo" 文件都是最新的, 只需要调整时间戳,或者可以从现有的 ".zo" 中重新编译成独立于机器的格式(当编译成独立于机器的格式时)。

    • --sync-docs-only — 将文档同步或移动到合适的地方以 "构建 "它,但不运行或渲染文档源。

    • --no-launcher 或 -x — 避免创建可执行文件或安装 man (如 "info.rkt" 中指定的;见 用 "info.rkt" 文件控制 raco setup) 。

    • --no-foreign-libs 或 -F — 避免安装外国库(如 "info.rkt"; 见 用 "info.rkt" 文件控制 raco setup) 。

    • --only-foreign-libs — 禁用除安装外国库以外的操作;相当于 -nxiIdD , 只是 --only-foreign-libs 不拒绝(多余的)指定那些单独的标志。

    • --no-install 或 -i — 避免运行预安装动作(如 "info.rkt" 文件中指定的; 见 用 "info.rkt" 文件控制 raco setup) 。

    • --no-post-install 或 -I — 避免运行安装后的操作(如 "info.rkt" 文件中指定的; 见 用 "info.rkt" 文件控制 raco setup) 。

    • --no-info-domain 或 -d — 避免从 "info.rkt" 文件中建立元数据信息的缓存。 这个缓存被其他工具所需要。例如, raco 本身使用这个缓存来定位插件工具。

    • --no-docs 或 -D — 避免建立文档。

    • --doc-pdf ‹dir› — 除了建立 HTML 文档外,将文档渲染成 PDF,并将文件放在 ‹dir› 中。

    • --no-pkg-deps 或 -K — 避免检查库之间的依赖关系是否被包级的依赖声明正确反映, 模块是否被多个包所声明,以及包的版本依赖是否被满足。 更多信息请参见 软件包的依赖性检查 。

    • --check-pkg-deps — 检查软件包的依赖性(除非明确禁用), 即使是在向 raco setup 提供了特定的集合,甚至是没有依赖性声明的软件包。 参见 软件包的依赖性检查 以获得更多信息。

    • --fix-pkg-deps — 试图通过调整软件包 "info.rkt" 文件来纠正依赖关系的不匹配(这只对作为链接安装的软件包有意义)。 更多信息请参见 软件包的依赖性检查 。

    • --unused-pkg-deps — 试图报告那些已声明但未使用的依赖关系。 请注意,有些软件包的依赖关系可能是故意不使用的(例如,为了方便而声明强制安装其他软件包), 也请注意,软件包的依赖关系被报告为未使用,只是因为相关模块的编译被抑制了。 更多信息请参见 软件包的依赖性检查 。

  • 限制用户与安装的设置:

    • --no-user 或 -U — 避免任何针对用户(而不是针对安装)的设置操作。

    • --no-planet — 避免任何针对 PLaneT 的安装操作;这个标志是由 --no-user 暗示的。

    • --avoid-main — 避免任何影响安装的设置操作,而不是针对用户的操作。

    • --force-user-docs — 在构建文档时,创建一个用户专用的文档入口点,即使它的内容与主安装相同。

  • 选择并行性和其他构建模式:

    • --jobs ‹n›, --workers ‹n›, 或 -j ‹n› — 使用最多 ‹n› 个并行进程。 默认情况下, raco setup 使用 (processor-count) jobs,这通常会使用机器的所有处理核心。

    • --places — 使用 Racket places 进行并行作业;如果 Racket places 并行运行,这种模式是默认的。

    • --processes — 使用单独的进程进行并行作业;如果 Racket places 不能并行运行,这种模式是默认的。

    • --verbose 或 -v — 更多关于 raco setup 动作的粗略输出。

    • --make-verbose 或 -m — 更多关于依赖性检查的粗略输出。

    • --compiler-verbose 或 -r — 更多关于依赖性检查和编译的粗略输出。

    • --mode ‹mode› — 使用默认编译器以外的 ".zo" 编译器, 并将产生的 ".zo" 文件放在以 ‹mode› 命名的子目录中(通常的地方)。 通过使用 ‹mode› 作为集合名称,在该集合中找到一个 "zo-compile.rkt" 模块, 并提取其 zo-compile ,就可以获得该编译器。 zo-compile 输出应该是一个类似于 compile 的函数; 见 "errortrace" 集合的例子。

    • --fail-fast — 一旦发现任何错误,就试图中断。

    • --error-out ‹file› — 通过写入 ‹file› 并成功退出来处理可存活的错误, 这有利于与 --error-in 相结合的多个 raco setup 调用的连锁。 如果没有错误,并且 ‹file› 已经存在,它将被删除。

    • --error-in ‹file› — 将 ‹file› 的存在视为 “errors were reported by a previous process” 的错误。 通常情况下, ‹file› 是由以前运行的 raco setup 使用 --error-out 创建的。 在通过 --error-in 创建文件之前,会检测到 --error-out 的文件, 所以同一个文件可以用来连接一系列的 raco setup 步骤。

    • --pause 或 -p — 在报告任何错误时暂停用户输入(这样用户就有时间检查输出, 否则在 raco setup 过程结束时可能会消失)。

  • 解压 ".plt" 档案:

    • -A ‹archive› ... — 安装每个 ‹archive›; 见 安装 ".plt" 档案.

    • --force — 与 -A 一起使用,将存档的版本不匹配视为单纯的警告。

    • --all-users 或 -a — 与 -A 一起使用,将归档文件安装到安装中,而不是用户特定的位置。

  • 引导:

    • --boot ‹module-file› ‹build-dir› — 用于直接运行 setup ,而不是通过 raco setup, 以 raco setup 通常加载自己的方式加载 ‹module-file› , 自动检测是否需要从源代码开始并重建编译文件,甚至是编译管理器本身。 ‹build-dir› 路径被安装为 current-compiled-file-roots 中的唯一路径, 所以所有的编译文件都在那里。

    • --chain ‹module-file› ‹build-dir› — 类似于 --boot, 但是将 ‹build-dir› 添加到 current-compiled-file-roots 的开头,而不是替换当前值, 这意味着已经在正常位置(包括编译管理器本身)构建的库将被使用而不是重建。 这种模式对交叉编译有意义。

在构建 racket 时,可以通过设置 PLT_SETUP_OPTIONS makefile 变量, 向 make install 运行的 raco setup 提供标志。 例如,下面的命令行在安装过程中使用一个单一的进程来构建集合:

  make install PLT_SETUP_OPTIONS="-j 1"

运行 raco setup 对 PLT_COMPILED_FILE_CHECK 环境变量的敏感性与 raco make 相同。 具体来说,如果 PLT_COMPILED_FILE_CHECK 被设置为 exists , 那么如果文件没有被重新编译, raco make 就不会尝试更新已编译文件的时间戳。

一些额外的环境变量对性能调试很有用:

  • PLT_SETUP_DMS_ARGS 在每个集合被编译后触发对 dump-memory-stats 的调用, 环境变量的值被 read 解析以获得 dump-memory-stats 的参数列表。

  • PLT_SETUP_LIMIT_CACHE (设置为任意值)避免在不同的集合中缓存编译文件信息, 这对于寻找内存泄露时减少噪音很有用。

  • PLT_SETUP_NO_FORCE_GC (设置为任意值) 抑制了对 collect-garbage 的调用, 对于非并行构建来说,在每个集合被编译和每个文件被运行或渲染之后,默认会对 collect-garbage 进行调用。

  • PLT_SETUP_SHOW_TIMESTAMPS (设置为任意值) 在 raco setup 打印的每条状态信息的  @  后面加上当前进程时间。
