6.14 跨平台配置的 API

请参阅 the raco cross documentation 以了解有关 raco cross 的信息, 这是一个由 "raco-cross" 包提供的工具,作为 Racket 交叉编译的方便接口。 这里记录的底层 API 支持 raco cross 和其他工具。

 (require setup/cross-system) package: base
setup/cross-system 库提供了查询目标平台的系统属性的函数,在交叉安装模式下,目标平台可能与当前平台不同。

Racket 的安装包括一个 "system.rktd" 文件,位于 (find-lib-dir) 报告的目录中。 当该文件中的信息与运行中的 Racket 的信息不一致时, 那么 setup/cross-system 模块就会推断Racket是在跨安装模式下运行的。

例如,如果不同平台的 Racket BC 就地安装在 ‹cross-dir›, 那么运行 Racket BC 的方式是

  racket -C -G ‹cross-dir›/etc -X ‹cross-dir›/collects -l- raco pkg

使用当前平台的 racket 可执行文件运行 raco pkg , 但使用 ‹cross-dir› 的集合和其他配置信息,以及修改 ‹cross-dir› 的包。 只要不需要运行特定平台的库来执行请求的 raco pkg 动作(例如,在安装已建好的包时), 或者只要当前平台的安装已经包括了这些库,这就可以工作。

对于 Racket CS 来说,交叉编译更为复杂,因为 Racket CS 的 ".zo" 文件是特定平台的:

  • 需要一个目标安装 ‹cross-dir› ,它包括对主机平台的交叉编译支持, 作为安装 "‹cross-dir›/lib" 目录下的插件。 该安装可能是通过在主机平台上从源码编译创建的。 只有 Racket CS 可以使用 CS 交叉编译插件。

    当在交叉模式下运行 racket 时, 使用 --cross-compiler 标志来指定目标机器和 "‹cross-dir›/lib" 目录的路径。

  • 需要一个带有参数 "‹absolute-zo-dir›:" 的标志组合 -MCR , 以使主机平台(使用 : 前的目录)和目标平台(当 : 后的路径为空时,使用正常的编译文件子目录) 都能创建 ".zo" 文件。

    ‹absolute-zo-dir› 可以是任何绝对路径。 一般来说,它应该在 raco pkg 等命令之前以交叉模式运行 raco setup 来填入。

例如,Racket CS 的 raco pkg 例子是

  racket --cross-compiler ‹target-machine› ‹cross-dir›/lib \

    -MCR ‹absolute-zo-dir›: \

    -G ‹cross-dir›/etc -X ‹cross-dir›/collects -l- raco pkg

提供给 --cross-compiler 的 ‹target-machine› 应该与 "‹cross-dir›/lib/systemd.rktd" 中的 target-machine 条目相同。

-C 标志是 --cross, -M 是 --compile-any 的缩写, -R 是 --compiled 的缩写, -G 是 --config 的缩写, -X 是 --collects 的缩写, 而 -MCR 是 -M -C -R 的缩写。

Added in version 6.3 of package base.

procedure

(cross-system-type [mode])

 → (or/c symbol? string? bytes? exact-positive-integer? vector?)
  mode : 
(or/c 'os 'word 'gc 'vm 'link 'machine
      'so-suffix 'so-mode 'fs-change)
 = 'os
与 system-type 类似,但在交叉安装模式下是针对目标平台而不是当前平台。 当不在交叉安装模式下,结果与 system-type 相同。

参见 system-type 的 'cross 。

procedure

(cross-system-library-subpath [mode]) → path-for-some-system?

  mode : (or/c 'cgc '3m 'cs #f) = (system-type 'gc)
与 system-library-subpath 类似,但在交叉安装模式下是针对目标平台而不是当前平台。 在非交叉安装模式下,其结果与 system-library-subpath 的结果相同。

在交叉安装模式下,目标平台可能与当前平台有不同的路径约定, 所以结果是 path-for-some-system? 而不是 path? 。

procedure

(cross-installation?) → boolean?

如果检测到交叉安装模式,返回 #t ,否则返回 #f 。
