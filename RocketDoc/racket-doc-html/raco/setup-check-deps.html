6.5 软件包的依赖性检查

当 raco setup 在没有参数的情况下运行时, 除非指定了 --check-pkg-deps , 否则如果为 raco setup 指定了任何集合,依赖性检查将被禁用。 在建立所有的集合和文档后, raco setup 会检查软件包的依赖性。 具体来说,它检查已编译的文件和文档,以检查跨包边界的引用是否反映在每个包级 "info.rkt" 文件的依赖声明中(见 包的元数据)。

raco setup 中的依赖性检查是作为软件包开发者的辅助工具,帮助他们正确声明依赖关系。 raco setup 过程本身并不依赖于软件包的依赖性声明。 同样,一个缺少依赖性声明的软件包可能会为其他用户成功安装,只要他们碰巧已经安装了依赖性。 一个缺失的依赖关系会给那些没有安装依赖关系的其他人带来麻烦。

实际上,每一个包都依赖于 "base" 包,其中包括 Racket 的最小变体中的集合。 声明对 "base" 的依赖似乎是不必要的,因为它的集合总是被安装。 然而,在未来的 Racket 版本中,最小的集合可能会改变,新的最小集合会有一个包名,比如 "base2" 。 声明对 "base" 的依赖性可以确保向前的兼容性,如果缺少声明, raco setup 会抱怨。

为了适应包开发的早期阶段,对于没有依赖性声明的包来说,缺失的依赖性不会被当作一个错误。

6.5.1 声明构建时的依赖关系

构建时依赖关系是指在软件包被转换为 binary package 时不存在的一种依赖关系。 (参见 源码、二进制和内置包)。 例如, "tests" 和 "scribblings" 目录在二进制包中被默认剥离, 所以来自这些名称的目录的跨包引用被视为构建依赖。 同样地, test 和 doc 子模块也被剥离, 所以在这些子模块中的引用会产生构建依赖关系。

在软件包的 "info.rkt" 文件中,只在构建期的依赖关系可以被列为 build-deps 而不是 deps 。 同时,列在 deps 中的依赖关系被视为运行时和构建时的依赖关系。 使用 build-deps 而不是在 deps 中列出所有的依赖项的好处是, 软件包的二进制版本可以在较少的依赖项下安装。

6.5.2 依赖性检查的工作方式

依赖性检查使用 ".zo" 文件、相关的 ".dep" 文件(见 相关性文件)和文档索引。 动态引用,例如通过 dynamic-require, 对于依赖检查器来说是不可见的; 只有通过 require, define-runtime-module-path-index 和其他与 raco make 合作的形式的依赖才是依赖检查的对象。

依赖性检查对一个依赖性是否只作为构建时的依赖性而被需要很敏感。 如果 raco setup 检测到一个缺失的依赖可以被添加为构建时依赖,它将建议添加, 但 raco setup 不会建议将一个普通的依赖转换为构建时依赖(因为每个普通的依赖也算作构建时依赖)。
