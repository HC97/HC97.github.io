8 raco demod: 去模块化程序

raco demodularize 命令(通常与简称 raco demod) 一起使用)接收一个 Racket 模块, 并将其所有的依赖关系扁平化为一个单一的编译模块。 一个文件 "‹name›.rkt" 被分解成 "‹name›_rkt_merged.zo" 。

解构后的 ".zo" 文件可以作为参数传递给 racket 命令行程序来运行, 也可以用 raco exe 将其变成可执行文件。

由分解器生成的大的单一模块可能会触发编译器的大小限制,从而阻碍整个模块的优化。 设置 PLT_CS_COMPILE_LIMIT 环境变量来提高限制, 并在 'linklet 主题下检查 'info 日志 (例如,将 PLTSTDERR 设置为 info@linklet ),以获得有关编译被限制为较小函数的信息。

raco demod 命令接受这些标志:

  • -o ‹file› — 将扁平化的模块写入 ‹file› , 而不是 "‹name›_‹ext›_merged.zo" 的输入文件 "‹name›.‹ext›" 。

  • -e ‹path› 或 --exclude ‹path› — 将 ‹path› 中的模块排除在扁平化之外,以及它的所有依赖项。

  • -M or --compile-any — 将模块扁平化为与机器无关的形式, 而不是根据当前平台和Racket虚拟机重新编译扁平化的模块; 用 -M 生成的输出比特定机器的形式加载得更慢, 但 raco decompile 可以显示扁平化的模块,其格式更接近于源代码。

  • -r 或 --recompile — 在扁平化后将模块(重新)编译成与机器相关的形式; 这种模式是默认的,除了在 BC 上,扁平化可以在字节码文件方面发挥作用。

  • --work ‹dir› — 使用 ‹dir› 将编译好的模块以中间形式缓存起来, 以便进行扁平化处理;在多次使用 raco demod 时使用– --work 和相同的 ‹dir› 可以大大加快脱模速度, 而且由于缓存是基于 raco make 的, 所以即使输入文件不同或者自上次使用缓存后需要扁平化的模块发生了变化, 它也能发挥作用。

  • -g 或 --garbage-collect — 基于定义的右侧没有副作用的假设,积极地修剪没有引用的定义; 由于这个假设没有被检查,这种转换可能不会保留输入模块的行为。

Changed in version 1.10 of package compiler-lib: 增加了 -M/--compile-any, --work, 以及对 Racket CS 的支持。
