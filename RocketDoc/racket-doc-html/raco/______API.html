1.10 原始编译的 API

 (require compiler/compiler) package: base
compiler/compiler 库为编译成字节码提供了 raco make 的功能,但通过 Racket API。

1.10.1 比特码编译

procedure

((compile-zos expr    
  [#:module? module?    
  #:verbose? verbose?])    
  racket-files    
  dest-dir) → void?
  expr : any/c
  module? : any/c = #f
  verbose? : any/c = #f
  racket-files : (listof path-string?)
  dest-dir : (or/c path-string? false/c (one-of/c 'auto))
只提供 expr ,就会返回一个用表达式 expr 初始化的编译器,如下所述。

编译器接收一个 Racket 文件列表,并将每个文件编译成字节码, 将产生的字节码放置在由 dest-dir 指定的目录中的 ".zo" 文件中。 如果 dest-dir 是 #f ,每个字节码的结果将被放置在与其源文件相同的目录中。 如果 dest-dir 是 'auto, 每个字节码文件将被放置在相对于源文件的 "compiled" 子目录中; 如果有必要,将创建该目录。

如果 expr 不是 #f,那么将创建一个命名空间来编译随后提供的文件, 并且 expr 将被评估以初始化创建的命名空间。 例如, expr 可能加载一组宏。 此外,后来编译的每个表达式的扩展时间部分在编译前在命名空间中被评估,这样在编译后来的表达式时,其效果是可见的。

如果 expr 是 #f, 那么就不创建编译命名空间(使用当前的命名空间), 并且假定文件中的表达式是独立编译的(所以不需要评估表达式的扩展时部分来编译)。

通常,在编译 module 文件时, expr 是 #f , 而在编译带有顶层定义和表达式的文件时,它是 (void) 。

如果 module? 是 #t, 那么给定的文件将作为模块被读取和编译(所以对当前名字空间的顶层环境没有依赖性)。

如果 verbose? 是 #t,那么每个给定文件的输出文件将通过当前输出端口报告。

procedure

(compile-collection-zos 
  collection ...+ 
  [#:skip-path skip-path 
  #:skip-paths skip-paths 
  #:skip-doc-sources? skip-docs? 
  #:managed-compile-zo managed-compile-zo]) 
 → void?
  collection : string?
  skip-path : (or/c path-string? #f) = #f
  skip-paths : (listof path-string?) = null
  skip-docs? : any/c = #f
  managed-compile-zo : (path-string? . -> . void?)
   = (make-caching-managed-compile-zo)
通过对每个源文件使用 managed-compile-zo ,将指定集合的文件编译为 ".zo" 文件。 这些 ".zo" 文件被放置在集合的 "compiled" 目录中。

默认情况下,一个集合中所有扩展名为 ".rkt", ".ss" 或 ".scm" 的文件都被编译, 子目录中的所有此类文件也被编译;如 get-module-suffixes 中所述,此类后缀的集合是可扩展的, 并且 compile-collection-zos 可以识别来自 'libs 组的后缀。 然而,任何文件或目录的路径以 skip-path 或 skip-paths 的某个元素开始,都会被跳过。 (“Starts with” 是指在 (simplify-path p #f) 之后的简化完整路径 p 字节串形式以 (simplify-path skip-path #f) 的字节串形式开始; 而不是说每个 skip-path 通常应该是一个完整的路径。)

集合编译器读取集合的 "info.rkt" 文件(见 "info.rkt" File Format)以获得编译集合的进一步指示。以下字段被使用:

  • name : 集合的名称,作为一个字符串,只用于状态和错误报告。

  • compile-omit-paths : 路径和 regexp values 的列表或 'all 。 在一个列表中,路径被视为一个不应该被编译的文件或一个目录,其文件不应该被编译, 其 "info.rkt" 文件应该被 raco setup 忽略; 路径是相对于集合(即包含 "info.rkt" 文件的目录)而言的, 并且可以参考子集合中的文件和目录,这些文件和目录是由子目录代表的。 列表中的 regexp 与相对于集合的文件和目录路径相匹配 (因此,例如,以 ^ 开始的 regexp 只匹配直接集合中的路径,而不是子集合中的路径), 以排除这些文件和目录的编译和 raco setup 。 值 'all 相当于指定集合中的所有文件和目录(在编译时有效地忽略集合)。 自动省略的文件和目录是 "compiled", "doc", 以及那些名字以 . 开头的文件。

    其他文件所需要的文件总是在编译需要的文件的过程中被编译–即使所需要的文件与这个字段一起被列出, 或者这个字段的值是 'all 。

  • compile-omit-files : 除了 compile-omit-paths 的内容之外, 还有一个不被编译的文件名(没有目录路径)的列表。不要使用这个字段;它是为了向后兼容。

  • scribblings : 一个列表,每个列表以文档源的路径开始。 更多信息见 用 "info.rkt" 文件控制 raco setup。 除非 skip-docs? 是一个 true 值,否则这些源文件(以及它们所需要的文件)的编译方式与其他模块文件相同。

  • compile-include-files : 要编译的文件名(不含目录路径)的列表, 此外还有根据文件的扩展名、在 scribblings 中、或被其他编译文件 require 的文件来编译。

  • module-suffixes 和 doc-module-suffixes : 通过 get-module-suffixes 来间接使用。

Changed in version 6.3 of package base: 增加了对 compile-include-files 的支持。
Changed in version 7.8.0.8: 更改了 skip-path 的 “starts with” ,以包括精确匹配。
Changed in version 8.1.0.5: 在 compile-omit-paths 中增加了对 regexps 的支持。

procedure

(compile-directory-zos 
  path 
  info 
  [#:verbose verbose? 
  #:skip-path skip-path 
  #:skip-paths skip-paths 
  #:skip-doc-sources? skip-docs? 
  #:managed-compile-zo managed-compile-zo]) 
 → void?
  path : path-string?
  info : procedure?
  verbose? : any/c = #f
  skip-path : (or/c path-string? #f) = #f
  skip-paths : (listof path-string?) = null
  skip-docs? : any/c = #f
  managed-compile-zo : (path-string? . -> . void?)
   = (make-caching-managed-compile-zo)
像 compile-collection-zos 一样,但是编译给定的目录而不是一个集合。 info 函数的行为类似于 get-info 的结果,以提供 "info.rkt" 字段, 而不是使用目录中的 "info.rkt" 文件(如果有的话)。

Changed in version 7.8.0.8 of package base: 改变了对 info 的处理, 将 info 用于 'compile-omit-paths , 忽略父目录和子目录中的任何 "info.rkt" 文件。

1.10.2 识别模块的后缀

 (require compiler/module-suffix) package: base
compiler/module-suffix 库提供了识别文件后缀的功能, 这些后缀对应于 Racket 模块,用于编译目录中的文件,为目录中的文件运行测试,等等。 后缀集总是包括 ".rkt", ".ss" 和 ".scm", 但它可以通过集合中的 "info.rkt" 配置进行全面扩展。

Added in version 6.3 of package base.

procedure

(get-module-suffixes [#:group group    
  #:mode mode    
  #:namespace namespace]) → (listof bytes?)
  group : (or/c 'all 'libs 'docs) = 'all
  mode : (or/c 'preferred 'all-available 'no-planet 'no-user)
   = 'preferred
  namespace : (or/c #f namespace?) = #f
检查已安装的集合的 "info.rkt" 文件(见 "info.rkt" File Format) 以产生一个应该被识别为 Racket 模块的文件后缀列表。 每个后缀被报告为一个字节字符串,不包括后缀前面的 . 。

mode 和 namespace 参数被传播到 find-relevant-directories , 以确定哪些集合目录可能配置后缀集。 因此,只有在运行 raco setup (或触发 raco setup 的软件包安装或更新)时,才能可靠地找到后缀注册。

group 参数决定了结果是包括所有注册的后缀,还是只包括那些注册为一般库的后缀,或者只包括那些注册为文档的后缀。 一般库后缀的集合总是包括 ".rkt", ".ss" 和 ".scm" 。 文档后缀的集合总是包括 ".scrbl" 。

"info.rkt" 文件中的下列字段扩展了后缀集:

  • module-suffixes : 对应于一般图书馆模块后缀的字节字符串的列表 (不包括必须出现在后缀前的 . )。 非列表或列表中的非字节字符串元素被忽略。

  • doc-module-suffixes : 和 module-suffixes 一样的字节字符串列表, 但用于文档模块。

procedure

(get-module-suffix-regexp [#:group group    
  #:mode mode    
  #:namespace namespace]) → byte-regexp?
  group : (or/c 'all 'libs 'docs) = 'all
  mode : (or/c 'preferred 'all-available 'no-planet 'no-user)
   = 'preferred
  namespace : (or/c #f namespace?) = #f
返回一个符合 get-module-suffixes 所报告的以后缀结尾的路径的 regexp value 。 该模式包括一个后缀的子模式,但不包括前导的 .

1.10.3 加载编译器支持

编译器单元通过 dynamic-require 和 get-info 按需加载某些工具。 如果编译过程中使用的名字空间与加载编译器时使用的名字空间不同, 或者设置了其他与加载有关的参数,那么可以使用以下参数来恢复 dynamic-require 的设置。

parameter

(current-compiler-dynamic-require-wrapper)

 → ((-> any) . -> . any)
(current-compiler-dynamic-require-wrapper proc) → void?
  proc : ((-> any) . -> . any)
一个参数,其值是一个需要应用 thunk 的过程。 默认的包装器在调用 thunk 前设置当前的名字空间(通过 parameterize) , 使用 compiler/compiler 库最初实例化的名字空间。

1.10.4 编译器的选项

 (require compiler/option) package: base
compiler/option 模块提供了控制编译器行为的选项(以参数的形式)。

更多的选项是由 dynext/compile 和 dynext/link 库定义的, 它们控制着实际的 C 编译器和链接器,通过 C 语言进行编译。

parameter

(somewhat-verbose) → boolean?

(somewhat-verbose on?) → void?
  on? : any/c
参数的 #t 值会使编译器打印它所编译和产生的文件。 默认值是 #f 。

parameter

(verbose) → boolean?

(verbose on?) → void?
  on? : any/c
参数的 #t 值使编译器打印有关其操作的粗略信息。 默认值是 #f 。

parameter

(compile-subcollections) → (one-of/c #t #f)

(compile-subcollections cols) → void?
  cols : (one-of/c #t #f)
一个参数,用于指定子集是否由 compile-collection-zos 进行编译。默认值是 #t 。

1.10.5 作为一个单元的编译器
1.10.5.1 签名

 (require compiler/sig) package: compiler-lib

signature

compiler^ : signature

包括由 compiler/compiler 输出的所有名称。

signature

compiler:option^ : signature

包括由 compiler/option 导出的所有名称。

1.10.5.2 主编译器单元

 (require compiler/compiler-unit) package: compiler-lib

value

compiler@ : unit?

以单元形式提供 compiler/compiler 的输出,其中 C-compiler 操作是单元的导入,尽管它们没有被使用。

该单元导入 compiler:option^, dynext:compile^, dynext:link^ 和 dynext:file^ 。 它输出 compiler^ 。

1.10.5.3 选项单元

 (require compiler/option-unit) package: compiler-lib

value

compiler:option@ : unit?

以单元形式提供 compiler/option 的输出。 它不导入任何签名,并导出 compiler:option^ 。
