3 raco distribute: 共享独立的可执行文件

raco distribute 命令将由 raco exe 创建的独立可执行文件与运行它所需的所有共享库以及通过 define-runtime-path 声明的任何运行时文件结合起来。 由此产生的软件包可以被移动到运行相同操作系统的其他机器上。

在 Windows 和 Mac OS 上,本地库往往被包含在 raco distribute 的输出中。 在 Unix 平台上,本地库往往不包括在内,所以系统库将在主机上使用。 区别在于Racket安装本身是包括捆绑的本地库还是依赖系统安装的库。 在 Racket 的 "lib" 目录中为系统安装的库添加符号链接会导致该库被包含在由 raco distribute 创建的分发目录中;另见 define-runtime-path 。

在 raco distribute 命令之后,提供一个目录来包含分发的组合文件。 每个命令行参数都是一个可执行文件,要包含在分发文件中,因此可以将多个可执行文件打包在一起。 例如,在 Windows 上、

  raco distribute greetings hello.exe goodbye.exe

创建一个目录 "greetings" (如果该目录不存在), 并将可执行文件 "hello.exe" 和 "goodbye.exe" 复制到 "greetings" 中。 如果需要包含 DLL,它还会在 "greetings" 中创建一个 "lib" 子目录, 在这种情况下, 它将复制的 "hello.exe" 和 "goodbye.exe" 调整为使用 "lib" 中的DLLs。

需要的支持文件的数量部分取决于创建发行版的可执行文件的方式。 向 raco exe 提供 --embed-dlls 或 --orig-exe 可以减少对支持文件的需求, 但代价是如果发行版包含多个可执行文件,则会使其变得更大。

分发目录中的文件布局与平台有关:

  • 在Windows上,可执行文件直接放在发行目录中,而 DLL 和其他运行时文件则放在 "lib" 子目录中。

  • 在 Mac OS 上,GUI 可执行文件被放入分发目录,其他可执行文件被放入 "bin" 子目录, 而框架(即共享库)与其他运行时文件一起被放入 "lib" 子目录。 作为一种特殊情况,如果发行版只有一个 --gui-exe 可执行文件, 那么 "lib" 目录就隐藏在应用程序捆绑包中。

  • 在 Unix 上,可执行文件进入 "bin" 子目录, 共享库(如果有的话)与其他运行时文件一起进入 "lib" 子目录, 被包装的可执行文件被放在 "lib/plt" 子目录下,并有特定的版本名称。 这种布局与 Unix 安装惯例一致; 共享库和封装的可执行文件的特定版本名称意味着发行版可以安全地解压到目标机器上的一个标准位置, 而不会与现有的Racket安装或由 raco exe 创建的其他可执行文件相冲突。

一个发行版也有一个 "collects" 目录,作为打包的可执行文件的主要库集合目录。 默认情况下,该目录是空的。 使用 raco distribute 的 ++collects-copy 标志来提供一个目录, 其内容将被复制到发行版的 "collects" 目录中。 ++collects-copy 标志可以多次使用以提供多个目录。

当多个可执行文件一起发布时,用 raco exe 分别创建可执行文件会产生多个可执行文件所使用的基于集合的库的副本。 为了共享库的代码,可以使用 raco exe 的 --collects-dest 标志为库的副本指定一个目标目录, 并为每个可执行文件指定相同的目录(这样所有可执行文件使用的库集就被集中起来了)。 最后,在用 raco distribute 打包时,使用 ++collects-copy 标志,将复制的库包含在发行版中。

3.1 分发可执行文件的 API

 (require compiler/distribute) package: base
compiler/distribute 库提供了一个函数来执行与 raco distribute 相同的工作。

procedure

(assemble-distribution dest-dir    
  exec-files    
  [#:executables? executables?    
  #:relative-base relative-base    
  #:collects-path path    
  #:copy-collects dirs]) → void?
  dest-dir : path-string?
  exec-files : (listof path-string?)
  executables? : any/c = #t
  relative-base : (or/c path-string? #f) = #f
  path : (or/c false/c (and/c path-string? relative-path?)) = #f
  dirs : (listof path-string?) = null
将 exec-files 中的可执行文件复制到目录 dest-dir 中, 同时复制 DLL、框架、共享库和/或可执行文件在不同机器上运行所需的运行时文件。 如果 executables? 是 #f, 那么 exec-files 将被视为普通的数据文件, 而不是可执行文件,并且它们将被就地修改。

可执行文件和支持文件在 dest-dir 中的排列取决于平台。 一般来说, assemble-distribution 试图做正确的事情, 但 relative-base 的非 #f 值指定了在运行时到达相对于可执行文件的装配内容的路径。 当 executables? 是 #f 时,那么默认的访问路径是 dest-dir, 并保留其相对性。

如果给出 #:collects-path 参数,它将覆盖打包后的可执行文件的主 "collects" 目录的默认位置。 它应该是相对于 dest-dir 目录的(通常是在它里面)。

#:copy-collects 参数中每个目录的内容都被复制到打包的可执行文件的主 "collects" 目录中。

Changed in version 6.3 of package base: 增加了 #:executables? 和 #:relative-base 参数。

3.2 用于捆绑发行版的 API

 (require compiler/bundle-dist) package: compiler-lib
compiler/bundle-dist 库提供了一个函数, 用于将一个目录(通常由 assemble-distribution 组装)打包成一个发行文件。 在 Windows 上,结果是一个 ".zip" 档案; 在 Mac OS 上,它是一个 ".dmg" 磁盘镜像;在 Unix 上,它是一个 ".tgz" 档案。

procedure

(bundle-directory dist-file dir [for-exe?]) → void?

  dist-file : file-path?
  dir : file-path?
  for-exe? : any/c = #f
将 dir 打包成 dist-file 。 如果 dist-file 没有扩展名, 会自动添加一个文件扩展名(使用 bundle-put-file-extension+style+filters 的第一个结果)。

创建的归档文件包含一个与 dir 同名的目录,但在 Mac OS 上, 当 for-exe? 为真且 dir 包含一个单一的文件或目录时, 在这种情况下,创建的磁盘镜像只包含该文件或目录。 for-exe? 的默认值是 #f 。

如果 dist-file 存在,则归档创建失败。

procedure

(bundle-put-file-extension+style+filters)

 → 
(or/c string? false/c)
(listof (one-of/c 'packages 'enter-packages))
(listof (list/c string? string?))
返回三个合适的值,分别作为 put-file 的 extension, style 和 filters 参数,以选择一个分发文件名。
