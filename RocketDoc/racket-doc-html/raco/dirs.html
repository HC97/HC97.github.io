6.8 寻找安装目录的API

 (require setup/dirs) package: base
setup/dirs 库提供了几个定位安装目录的程序。 其中许多路径可以通过 configuration directory (见 安装配置和搜索路径)进行配置。

在跨平台构建模式下(见 跨平台配置的 API), setup/dirs 提供的函数通常报告目标系统的路径,而不是当前系统的路径。 例外的是 get-lib-search-dirs 和 find-dll-dir,它们报告当前系统的路径, 而 get-cross-lib-search-dirs 和 find-cross-dll-dir 则报告目标系统的路径。

procedure

(find-collects-dir) → (or/c path? #f)

返回到安装的主 "collects" 目录的路径,如果没有找到,则返回 #f 。 一个 #f 的结果可能只出现在没有库的独立可执行文件中。

procedure

(find-user-collects-dir) → path?

返回用户特定的 "collects" 目录的路径;返回的路径所指示的目录可能存在也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(get-collects-search-dirs) → (listof path?)

返回与 (current-library-collection-paths) 相同的结果, 这意味着这个结果对 use-user-specific-search-paths 参数的值不敏感。

procedure

(get-main-collects-search-dirs) → (listof path?)

返回一个安装 "collects" 目录的路径列表,通常包括 find-collects-dir 的结果。 这些目录通常包括在 (current-library-collection-paths) 的结果中, 但是 PLTCOLLECTS 的设置或参数的改变可能导致它们被省略。 (current-library-collection-paths) 中的任何其他路径都被视为用户特定的。 返回的路径所指示的目录可能存在,也可能不存在。

主库的搜索路径可以通过 "config.rktd" 中的 'collects-search-dirs 来配置(见 安装配置和搜索路径)。

procedure

(find-config-dir) → (or/c path? #f)

返回到安装的 "etc" 目录的路径, 该目录包含配置和软件包信息–包括一些其他目录的配置(见 安装配置和搜索路径)。 一个 #f 的结果表示没有配置目录可用。

procedure

(find-links-file) → (or/c path? #f)

返回一个指向安装的 collection links file 的路径。 返回的路径所指示的文件可能存在,也可能不存在。 一个 #f 的结果表示没有链接文件可用。

参见 安装配置和搜索路径 中的 'links-file 。

procedure

(find-user-links-file [vers]) → path?

  vers : string? = (get-installation-name)
返回用户的 collection links file 的路径。 返回的路径所指示的文件可能存在,也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 vers 。

procedure

(get-links-search-files) → (listof path?)

返回一个安装 collection links files 的路径列表,以便依次搜索。 (通常情况下,结果包括 (find-links-file) 的结果, 这是由 raco link 或 links 安装新的安装范围的链接的地方)。 find-user-links-file 的结果 不 会被添加到返回的列表中。 返回的路径所指示的文件可能存在,也可能不存在。

参见 安装配置和搜索路径 中的 'links-search-files 。

procedure

(find-pkgs-dir) → (or/c path? #f)

返回一个包含有安装范围的软件包的目录的路径;返回的路径所指示的目录可能存在也可能不存在。 一个 #f 的结果表示没有软件包安装目录可用。

参见 安装配置和搜索路径 中的 'pkgs-dir 。

procedure

(find-user-pkgs-dir [vers]) → path?

  vers : string? = (get-installation-name)
返回一个包含具有用户特定范围的软件包的目录的路径,用于安装名称 vers; 返回的路径所指示的目录可能存在也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 vers 。

procedure

(get-pkgs-search-dirs) → (listof path?)

返回一个包含安装范围内软件包的目录的路径列表。 (通常情况下,结果包括 (find-pkgs-dir) 的结果,这是 raco pkg install 安装新软件包的地方。) find-user-pkgs-dir 的结果 不 会被添加到返回的列表中。 返回的路径所指示的目录可能存在,也可能不存在。

参见 安装配置和搜索路径 中的 'pkgs-search-dirs 。

procedure

(find-doc-dir) → (or/c path? #f)

返回一个通往安装的 "doc" 目录的路径。 如果没有这样的目录,结果是 #f 。

参见 安装配置和搜索路径 中的 'doc-dir 。

procedure

(find-user-doc-dir) → path?

返回一个到用户特定的 "doc" 目录的路径。返回的路径所指示的目录可能存在,也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(get-doc-search-dirs) → (listof path?)

返回一个搜索文档的路径列表,不包括存储在单个集合中的文档。 除非另有配置,否则结果包括 (find-doc-dir) 和 (find-user-doc-dir) 的任何非 #f 结果–但后者仅在 use-user-specific-search-paths 参数的值为 #t 时才包括。

参见 安装配置和搜索路径 中的 'doc-search-dirs 。

procedure

(get-doc-extra-search-dirs) → (listof path?)

和 get-doc-search-dirs 一样, 但避免将 (find-doc-dir) 和 (find-user-doc-dir) 添加到底层的 'doc-search-dirs 配置。

Added in version 8.1.0.6 of package base.

procedure

(find-lib-dir) → (or/c path? #f)

返回到安装的 "lib" 目录的路径,该目录包含库和其他构建信息。 如果没有这样的目录,结果是 #f 。

参见 安装配置和搜索路径 中的 'lib-dir 。

procedure

(find-user-lib-dir) → path?

返回一个用户特定的 "lib" 目录的路径;返回的路径所指示的目录可能存在也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(get-lib-search-dirs) → (listof path?)

返回一个搜索国外库的路径列表。

除非另有配置,而且除了跨平台构建模式外, 结果包括 (find-lib-dir) 和 (find-user-lib-dir) 的任何非 #f 结果– 但后者仅在 use-user-specific-search-paths 参数的值为 #t 时才包括。

在跨平台构建模式下(见 跨平台配置的 API), get-lib-search-dirs 报告的是适合当前系统的结果,而不是目标系统。 另请参见 get-cross-lib-search-dirs 。

参见 安装配置和搜索路径 中的 'lib-search-dirs 。

Changed in version 6.1.1.4 of package base: 从默认情况下要明确包括的路径中删除了 (find-dll-dir) 。
Changed in version 6.9.0.1: 改变了跨平台构建模式下的行为。

procedure

(get-cross-lib-search-dirs) → (listof path?)

与 get-lib-search-dirs 类似,但在跨平台构建模式下, 报告目标系统的目录(包括 (find-lib-dir) 的任何非 #f 结果等),而不是当前系统。

Added in version 6.9.0.1 of package base.

procedure

(get-cross-lib-extra-search-dirs) → (listof path?)

与 get-cross-lib-search-dirs 类似, 但避免将 (find-lib-dir) 和 (find-user-lib-dir) 添加到底层的 'lib-search-dirs 配置。

Added in version 8.1.0.6 of package base.

procedure

(find-dll-dir) → (or/c path? #f)

返回一个包含用于当前可执行文件的 DLL 的目录的路径(例如,Windows 上的 "libracket.dll" )。 如果没有这样的目录,或者没有特定的目录(即,除了平台的正常搜索路径之外),结果是 #f 。

在跨平台构建模式下(见 跨平台配置的 API), find-dll-dir 报告一个适合当前系统的结果,而不是目标系统。 另请参见 find-cross-dll-dir 。

Changed in version 6.9.0.1 of package base: 改变了跨平台构建模式下的行为。

procedure

(find-cross-dll-dir) → (or/c path? #f)

与 find-dll-dir 类似,但在跨平台构建模式下,报告的是目标系统的目录,而不是当前系统。

Added in version 6.9.0.1 of package base.

procedure

(find-share-dir) → (or/c path? #f)

返回安装的 "share" 目录的路径, 其中包含已安装的软件包和其他与平台无关的文件。 如果没有这样的目录,结果是 #f 。

参见 安装配置和搜索路径 中的 'share-dir 。

procedure

(find-user-share-dir) → path?

返回一个用户特定的 "share" 目录的路径;返回的路径所指示的目录可能存在也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(get-share-search-dirs) → (listof path?)

返回一个搜索文件的路径列表,这些文件通常在一个 "share" 目录下。

除非另有配置,否则结果包括 (find-share-dir) 和 (find-user-share-dir) 的任何非 #f 结果–但后者仅在 use-user-specific-search-paths 参数的值为 #t 时才包括。

参见 安装配置和搜索路径 中的 'share-search-dirs 。

Added in version 8.1.0.6 of package base.

procedure

(get-share-extra-search-dirs) → (listof path?)

与 get-share-search-dirs 类似,但避免在底层的 'share-search-dirs 配置中加入 (find-share-dir) 和 (find-user-share-dir) 。

Added in version 8.1.0.6 of package base.

procedure

(find-include-dir) → (or/c path? #f)

返回安装的 "include" 目录的路径,该目录包含用于构建 Racket 扩展和嵌入程序的 ".h" 文件。 如果没有这样的目录,结果是 #f 。

参见 安装配置和搜索路径 中的 'include-dir 。

procedure

(find-user-include-dir) → path?

返回一个用户特定的 "include" 目录的路径;返回的路径所指示的目录可能存在也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(get-include-search-dirs) → (listof path?)

返回一个搜索 ".h" 文件的路径列表。 除非另有配置,否则结果包括 (find-include-dir) 和 (find-user-include-dir) 的任何非 #f 结果–但后者仅在 use-user-specific-search-paths 参数值为 #t 时才会包括。

参见 安装配置和搜索路径 中的 'include-search-dirs 。

procedure

(find-console-bin-dir) → (or/c path? #f)

返回到安装的可执行目录的路径,即独立的 Racket 可执行文件所在的目录。 如果没有这样的目录,结果是 #f 。

参见 安装配置和搜索路径 中的 'bin-dir 。

procedure

(find-gui-bin-dir) → (or/c path? #f)

返回安装的可执行目录的路径,即独立的 GRacket 可执行文件所在的位置。如果没有这样的目录,结果是 #f 。

参见 安装配置和搜索路径 中的 'gui-bin-dir 。

procedure

(find-user-console-bin-dir) → path?

返回用户的可执行目录的路径;返回的路径所指示的目录可能存在,也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(find-user-gui-bin-dir) → path?

返回用户图形程序的可执行目录的路径;返回的路径指示的目录可能存在也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(get-console-bin-search-dirs) → (listof path?)

类似于 get-share-search-dirs, 是用于搜索可执行文件的路径,如 racket 。

参见 安装配置和搜索路径 中的 'bin-search-dirs 。

Added in version 8.1.0.6 of package base.

procedure

(get-console-bin-extra-search-dirs) → (listof path?)

类似于 get-share-extra-search-dirs 的底层 'bin-search-dirs 配置。

Added in version 8.1.0.6 of package base.

procedure

(get-gui-bin-search-dirs) → (listof path?)

类似于 get-share-search-dirs, 但用于搜索可执行文件的路径,如 gracket 。

参见 安装配置和搜索路径 中的 'gui-bin-search-dirs 。

Added in version 8.1.0.6 of package base.

procedure

(get-gui-bin-extra-search-dirs) → (listof path?)

类似于 get-share-extra-search-dirs 的基础 'gui-bin-search-dirs 配置。

Added in version 8.1.0.6 of package base.

procedure

(find-apps-dir) → (or/c path? #f)

返回安装目录中 ".desktop" 文件的路径(对于 Unix)。 如果没有这样的目录存在,结果是 #f 。

参见 安装配置和搜索路径 中的 'apps-dir 。

procedure

(find-user-apps-dir) → path?

返回用户的 ".desktop" 文件目录的路径(对于 Unix);返回的路径所指示的目录可能存在,也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(find-man-dir) → (or/c path? #f)

返回一个指向安装的手册目录的路径。如果没有这样的目录存在,结果是 #f 。 参见 安装配置和搜索路径 中的 'man-dir 。

procedure

(find-user-man-dir) → path?

返回到用户的手册目录的路径;返回的路径所指示的目录可能存在,也可能不存在。

用户特定的路径至少取决于 (find-system-path 'addon-dir) 和 (get-installation-name) 。

procedure

(get-man-search-dirs) → (listof path?)

类似于 get-share-search-dirs, 但用于搜索手册的路径。

参见 安装配置和搜索路径 中的 'man-search-dirs 。

Added in version 8.1.0.6 of package base.

procedure

(get-man-extra-search-dirs) → (listof path?)

类似于 get-share-extra-search-dirs 的基础 'man-search-dirs 配置。

Added in version 8.1.0.6 of package base.

procedure

(get-doc-search-url) → string?

返回一个字符串,由文档系统使用,并加上版本和搜索键查询,用于远程文档链接。

参见 安装配置和搜索路径 中的 'doc-search-url 。

procedure

(get-doc-open-url) → (or/c string? #f)

返回 #f 或一个根 URL 的字符串,作为打开本地文件获取文档的替代方案。 非 #f 配置意味着 DrRacket 通过指定的 URL 而不是从本地安装的文档进行关键词搜索。

参见 安装配置和搜索路径 中的 'doc-open-url 。

Added in version 6.0.1.6 of package base.

procedure

(get-installation-name config) → string?

  config : (read-installation-configuration-table)
返回当前安装的名称,通常是 (version), 但是如果 (use-user-specific-search-paths) 是 #t 的话, 安装名称可以通过 config 中的 'installation-name 值和用户特定的目录状态组合来设置。

用户特定的结果取决于在 (find-system-path 'addon-dir) 中是否存在一个目录 "other-version" 。 如果该目录存在,并且没有安装配置名称的目录存在,那么 "other-version" 将被用作安装名称。 因此,通过创建 "other-version" 目录,用户可以选择在不同的安装/版本中共享软件包和集合, 同时通过创建一个带有该安装名称的目录来选择退出某个特定的安装/版本。

Changed in version 8.4.0.3 of package base: 增加了 config 参数和对用户特定安装名称的支持。

procedure

(get-build-stamp) → (or/c #f string?)

返回一个标识安装版本的字符串,它可以用来增加Racket版本号, 以更具体地标识该版本。 如果是发行版,通常会产生一个空字符串。如果没有构建标识,结果是 #f 。

procedure

(get-absolute-installation?) → boolean?

如果此安装使用绝对路径名来引用可执行文件和库,则返回 #t 否则返回 #f 。

procedure

(find-addon-tethered-console-bin-dir) → (or/c #f path?)

procedure

(find-addon-tethered-gui-bin-dir) → (or/c #f path?)

procedure

(find-addon-tethered-apps-dir) → (or/c #f path?)

返回一个用户特定目录的路径,以保存每个已安装的可执行文件和 ".desktop" 文件(Unix)的额外副本, 其中额外的副本由 raco setup 创建, 并与 (find-system-path 'addon-dir) 和 (find-config-dir) 的特定结果绑定。

与其他目录不同,这些目录是通过 (find-config-dir) 目录下的 "config.rktd" 来配置的 (见 安装配置和搜索路径), 这些路径是通过 (build-path (find-system-path 'addon-dir) "etc") 中的 "config.rktd" 中的 'addon-tethered-console-bin-dir, 'addon-tethered-gui-bin-dir 和 'addon-tethered-apps-dir 条目配置的。 如果没有配置,相应的函数 find-addon-tethered-console-bin-dir, find-addon-tethered-gui-bin-dir 或 find-addon-tethered-apps-dir 的结果是 #f 而不是路径。

更多信息请看 拴住的安装 。

Added in version 6.5.0.2 of package base.
Changed in version 8.3.0.11: 增加了 find-addon-tethered-apps-dir 。
