1 调度服务器

Web Server 只是调度服务器的一个配置。

1.1 调度服务器的签名

 (require web-server/private/dispatch-server-sig)
  package: web-server-lib
web-server/private/dispatch-server-sig 库提供了 dispatch-server^, dispatch-server-connect^ 和 dispatch-server-config*^ 。

signature

dispatch-server^ : signature

dispatch-server^ 签名是 web-server^ 的一个别名。

procedure

(serve [#:confirmation-channel confirmation-ach]) → (-> any)

  confirmation-ach : 
(or/c #f (async-channel/c
          (or/c exn? port-number?)))
 = #f
运行服务器。 如果提供了确认通道,如果在启动服务器时发生异常,将发送一个异常,如果服务器成功启动,则发送端口号。

调用返回的过程将关闭服务器。

procedure

(serve-ports ip op) → any

  ip : input-port?
  op : output-port?
异步地服务于由端口 ip 和 op 代表的单个连接。

signature

dispatch-server-connect^ : signature

dispatch-server-connect^ 签名抽象了调度服务器所使用的连接端口的转换(例如,实现 SSL)。

input-port? output-port?
  ip : input-port?
  op : output-port?
根据需要转换连接端口。

连接端口通常是 TCP 端口,但链接到调度器的 tcp^ 的另一种实现可以提供不同种类的端口。

dispatch-server-config*^ : signature

Added in version 1.6 of package web-server-lib.

value

procedure

(read-request c p port-addresses) → 
any/c boolean?
  c : connection?
  p : listen-port-number?
  port-addresses : (input-port? . -> . (values string? string?))
定义服务器从连接中读取请求的方式,以传递给 dispatch 。 port-addresses 参数应该是一个类似 tcp-addresses 的过程。

read-request 的第一个结果通常是一个 request 值, 但这在 dispatch-server 层面不是一个要求。 第二个结果是 #true ,如果连接 c 在处理完这个请求后应该被关闭, 或者 #false ,如果连接可以被重新使用。

Web 服务器使用不透明的 safety limits 值,由谓词 safety-limits? 识别, 以封装策略,防止行为不端或恶意的客户端和 servlets。 使用 make-safety-limits 构建 safety limits 值, 它提供了合理安全的默认策略,应该适用于大多数应用程序。 更多细节请参见 compatability note 和 make-unlimited-safety-limits 。

make-safety-limits 的参数使用如下:

  • max-concurrent 参数限制了开放到服务器的并发连接的数量。 一旦达到限制,新的连接将在 TCP 层面上排队(见 max-waiting),直到现有的连接完成或超时。

  • max-waiting 参数被传递给 tcp-listen ,以指定等待接受的客户端连接的最大数量。 当 max-waiting 的客户端在等待接受时,不能有新的客户端连接。

  • request-read-timeout 限制了标准的 read-request 实现(例如来自 serve 或 web-server@)在关闭连接之前等待来自客户端的请求数据的时间,单位是秒。 如果你需要支持通过慢速连接上传大文件,你可能需要调整这个值。

  • max-request-line-length 限制了 HTTP 请求的第一行( “request line”)的长度(以字节为单位), 它指定了请求方法、路径和协议版本。 第一行长度超过 max-request-line-length 的请求会被标准的 read-request 实现 (例如,来自 serve 或 web-server@)拒绝。 如果你有很长的URL,请增加这个长度,但也请看 is-url-too-big? 。

  • max-request-headers 和 max-request-header-length 参数分别限制了每个 HTTP 请求允许的头的数量和单个请求头的长度(字节)。 超过这些限制的请求会被标准的 read-request 实现 (例如,来自 serve 或 web-server@) 拒绝。

  • max-request-body-length 限制了HTTP请求体的大小, 以字节为单位,但它不适用于多部分(文件上传)请求: 见下文 max-form-data-files 和相关限制。 长于 max-request-body-length 的请求会被标准的 read-request 实现 (例如,来自 serve 或 web-server@) 拒绝。

  • max-form-data-files, max-form-data-fields, max-form-data-file-length, max-form-data-field-length, max-form-data-parts, form-data-file-memory-threshold 和 max-form-data-header-length 参数控制标准 read-request 实现 (例如,来自 serve 或 web-server@ ) 对 multipart/form-data (文件上传)请求的处理。

    每个请求的文件和非文件 “fields” 的数量分别由 max-form-data-files 和 max-form-data-fields 限制, 而 max-form-data-file-length 和 max-form-data-field-length 限制了单个文件或非文件字段的长度(字节)。 此外,包括文件和字段的 “parts” 总数不得超过 max-form-data-parts 。 超过这些限制的请求将被拒绝。

    超过 request-file-memory-threshold (字节)的文件会作为临时文件自动卸载到磁盘,以避免内存耗尽。

    max-form-data-header-length 参数限制了单个部分(文件或字段)的头的长度。 由于这种头文件已经受到 RFC 7578 §4.8. 的严格限制, 应该很少需要增加这个限制,但这样做可能会使文件或字段名特别长。

  • response-timeout 和 response-send-timeout 参数限制了单个请求处理程序(如 dispatch) 被允许运行的时间。

    response-timeout 指定了处理程序在读取请求后, 在写入第一个字节的响应数据之前,允许运行的最大时间(秒)。 如果在这个时间限制内没有写入数据,连接将被杀死。

    response-send-timeout 指定了服务器等待一个响应数据块的最长时间(秒)。 每次向客户端发送一大块数据时,这个超时都会被重置。 如果你的应用程序使用流式响应或长时间的轮询,要么调整这个值,要么确保你的请求处理程序定期发送数据, 比如一个no-op,以避免触及这个限制。

兼容性说明: safety limits 类型在未来可能会被扩展,以提供额外的保护。 用 make-safety-limits 创建 safety limits 值将允许应用程序利用任何新添加的限制的合理默认值。 然而,添加新的限制确实有可能破坏一些现有的应用程序: 作为一种选择, make-unlimited-safety-limits 构造函数使用默认值,避免强加任何没有明确指定的限制。 (在大多数情况下,这意味着默认值为 +inf.0 )。 当然,使用 make-unlimited-safety-limits 的应用程序可能仍然容易受到 make-safety-limits 的值所保护的威胁。

safety limits 类型是在 web-server-lib 包的 1.6 版本中引入的。 该库以前的版本只支持 max-waiting 限制和(在某些情况下) initial-connection-timeout 限制,这与 request-read-timeout 类似, 但有 some problems 。 这些限制是通过 dispatch-server-config^, web-config^, 和 serve 等函数的可选参数指定的: 如果没有明确提供数值, 默认行为最接近于使用 (make-unlimited-safety-limits #:request-read-timeout 60) 。

然而,1.6 版采用了 (make-safety-limits) 作为默认值,因为大多数应用程序会从使用合理的保护措施中受益。 当从这个库的早期版本移植时,如果你认为你的应用程序可能特别耗费资源, 你可能宁愿使用 make-unlimited-safety-limits ,同时确定对你的应用程序有效的限制。

1.4 线程和托管人

调度服务器在一个专门的线程中运行。每次启动连接时,都会启动一个新的线程来处理它。 连接线程是在一个专门的托管人内部创建的,该托管人是服务器托管人的一个子节点。 当服务器被用来提供 Servlet 时,每个 Servlet 也会收到一个新的托管人, 这个托管人是服务器托管人的一个子节点,而 不 是连接托管人。
