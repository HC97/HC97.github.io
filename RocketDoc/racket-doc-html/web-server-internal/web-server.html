3 启动服务器

 (require web-server/web-server) package: web-server-lib
本模块提供了启动调度服务器的功能。

procedure

(serve #:dispatch dispatch 
  [#:confirmation-channel confirmation-channel 
  #:connection-close? connection-close? 
  #:dispatch-server-connect@ dispatch-server-connect@ 
  #:tcp@ tcp@ 
  #:port port 
  #:listen-ip listen-ip

  #:initial-connection-timeout request-read-timeout 
  #:safety-limits safety-limits]) 
 → (-> any)
  dispatch : dispatcher/c
  confirmation-channel : 
(or/c #f (async-channel/c
          (or/c exn? port-number?)))
   = #f
  connection-close? : boolean? = #f
  dispatch-server-connect@ : 
(unit/c (import)
        (export dispatch-server-connect^))
   = raw:dispatch-server-connect@
  tcp@ : (unit/c (import) (export tcp^)) = raw:tcp@

  listen-ip : (or/c string? #f) = #f

  request-read-timeout : timeout/c = 60

 #:request-read-timeout request-read-timeout)
构建一个适当的 dispatch-server-config*^, 调用 dispatch-server@, 并调用其 serve 功能。

如果 connection-close? 是 #t, 那么每个连接在一次请求后就会被关闭。 否则,客户端根据其使用的 HTTP 版本来决定。
dispatch-server-connect@ 参数支持原始连接的转换; 例如, make-ssl-connect@ 通过将原始 TCP 端口转换为 SSL 端口来产生一个服务 SSL 的单元; 参见 我如何设置服务器使用 HTTPS? 。
tcp@ 参数支持用其他类型的连接替换 TCP 连接(以前推荐用于支持 SSL)。 请注意,服务器希望 tcp@ 的 tcp-accept 操作是有效的原子操作; 当 tcp-accept 正在进行时,新的连接不会被接受。
safety-limits 参数提供了一个 safety limits 值,指定了在读取和处理请求时要使用的策略。 在构建的 dispatch-server-config*^ 中, 它被直接用作 safety-limits 值, 同时也被 read-request 实现所使用。
为了向后兼容,支持 max-waiting 和 request-read-timeout 参数。 如果给出了 safety-limits 参数, max-waiting 和 request-read-timeout 参数将被忽略; 否则,它们将被传递给 make-safety-limits 以构建 safety limits 值。 如果既没有给出 max-waiting, request-read-timeout, 也没有给出 safety-limits , 那么默认的 safety limits 值就等同于 (make-safety-limits) 。
下面是一个简单的 Web 服务器的例子,它从一个给定的路径提供文件:
(define (start-file-server base)
  (serve
   #:dispatch
   (files:make
    #:url->path (make-url->path base)
    #:path->mime-type
    (lambda (path)
      #"application/octet-stream"))
   #:port 8080))

Changed in version 1.6 of package web-server-lib: 增加了 safety-limits 参数, 并改为使用 dispatch-server-config*^ 而不是 dispatch-server-config^: 见 compatability note 。 纠正了 max-waiting 和 request-read-timeout 参数的文档契约。

3.1 简单的单一 Servlet 服务器

 (require web-server/servlet-dispatch)
  package: web-server-lib
这些函数优化了单个 servlet 和交互式开发的调度器的构造和服务器的启动。

procedure

(dispatch/servlet 
  start 
  [#:regexp regexp 
  #:stateless? stateless? 
  #:stuffer stuffer 
  #:manager manager 
  #:current-directory servlet-current-directory 
  #:responders-servlet-loading responders-servlet-loading 
  #:responders-servlet responders-servlet]) 
 → dispatcher/c
  start : (request? . -> . response?)
  regexp : regexp? = #rx""
  stateless? : boolean? = #f
  stuffer : (stuffer/c serializable? bytes?) = default-stuffer
  manager : manager?
   = (make-threshold-LRU-manager #f (* 1024 1024 64))
  servlet-current-directory : path-string? = (current-directory)
  responders-servlet-loading : (url? any/c . -> . can-be-response?)
   = servlet-loading-responder
  responders-servlet : (url? any/c . -> . can-be-response?)
   = servlet-error-responder
serve/servlet 启动一个服务器并使用一个特定的调度序列。 对于一些应用来说,这钉子钉得太紧了,但用户却很矛盾,因为这个接口太方便了。 对于这些用户来说, dispatch/servlet 做了 serve/servlet 中最难的部分, 并且仅仅为 start servlet 构造了一个调度器。

该调度器对符合 regexp 的请求做出响应。 servlet 执行的当前目录是 servlet-current-directory 。

如果 stateless? 为真,那么servlet将作为无状态的

#lang web-server

模块运行, stuffer 被用作 stuffer 。

servlet 以 manager 作为其延续管理器被加载。 (默认的管理器将内存量限制在 64MB,并处理内存压力,在 make-threshold-LRU-manager 文档中讨论过)。
servlet在 (current-namespace) 中运行。
如果一个 servlet 加载失败,就会使用 responders-servlet-loading 。 如果一个 servlet 在运行过程中出错,就会使用 responders-servlet 。

procedure

(serve/launch/wait make-dispatcher    
  [#:connection-close? connection-close?    
  #:launch-path launch-path    
  #:banner? banner?    
  #:listen-ip listen-ip    
  #:port port    
  #:ssl-cert ssl-cert    
  #:ssl-key ssl-key    
  #:max-waiting max-waiting    
  #:safety-limits safety-limits]) → any
  make-dispatcher : (semaphore? . -> . dispatcher/c)
  connection-close? : boolean? = #f
  launch-path : (or/c #f string?) = #f
  banner? : boolean? = #f
  listen-ip : (or/c #f string?) = "127.0.0.1"
  port : number? = 8000
  ssl-cert : (or/c #f path-string?) = #f
  ssl-key : (or/c #f path-string?) = #f
  max-waiting : exact-nonnegative-integer? = 511
  safety-limits : safety-limits?
   = (make-safety-limits #:max-waiting max-waiting)
serve/servlet 的另一个有趣的部分是它能够启动一个服务器并立即向它发起一个浏览器。 这是由 serve/launch/wait 提供的。

它使用 make-dispatcher 的结果作为调度器来启动服务器。 make-dispatcher 参数被调用时带有一个信号,如果发布,将导致服务器退出。
如果 launch-path 不是假的,那么就会启动一个浏览器,将该路径附加到服务器本身的 URL 上。
如果 banner? 为真,那么就会打印一个横幅,告知用户服务器的 URL。
服务器在 listen-ip 和 port 端口上监听。 如果 listen-ip 是 #f, 那么服务器接受对所有监听机器地址的连接。 否则,服务器只接受与给定字符串相关的接口的连接。 例如,提供 "127.0.0.1" (默认值)作为 listen-ip , 会创建一个只接受来自本地机器的 "127.0.0.1" (回环接口)连接的服务器。
如果 ssl-key 和 ssl-cert 不是假的,那么服务器就以 HTTPS 模式运行, ssl-cert 和 ssl-key 为证书和私钥的路径。
如果 connection-close? 是 #t, 那么每个连接在一次请求后就会关闭。 否则,客户端会根据它使用的 HTTP 版本来决定。
safety-limits 参数提供了一个 safety limits 值,指定了在读取和处理请求时要使用的策略。
为了向后兼容,支持 max-waiting 参数。 如果给出了一个 safety-limits 参数, 那么 max-waiting 参数将被忽略; 否则,它将被传递给 make-safety-limits 来构建 safety limits 值。 如果既没有给出 max-waiting 也没有给出 safety-limits , 默认的 safety limits 值等同于 (make-safety-limits) 。

Changed in version 1.6 of package web-server-lib: 增加了 safety-limits 参数: 见 compatability note 。
