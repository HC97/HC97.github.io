OpenSSL: 安全通信

 (require openssl) package: base

openssl 库为 OpenSSL 库和 Racket 端口系统提供了胶水。 它提供的功能几乎与 Racket 中的标准 TCP 子系统相同, 另外还有一个通用的 ports->ssl-ports 接口。

要使用这个库,你需要在你的机器上安装 OpenSSL,但在许多平台上,必要的库已经包含在操作系统或 Racket 发行版中。 特别是:

  • 对于 Windows, openssl 依赖于 "libeay32.dll" 和 "ssleay32.dll", 它们包含在 Windows 的 Racket 发行版中。

  • 对于 Mac OS, openssl 依赖于 "libssl.dylib" 和 "libcrypto.dylib" 。 尽管这些库是由 Mac OS 10.2 及以后的版本提供的,但它们的使用已被废弃, 所以 Mac OS 的 Racket 发行版包括了较新的版本。

  • 对于 Unix, openssl 依赖于 "libssl.so" 和 "libcrypto.so", 它们必须安装在标准库位置或 LD_LIBRARY_PATH 列的目录中。 这些库被包含在许多操作系统的发行版中。

value

ssl-available? : boolean?

一个布尔值,报告系统 OpenSSL 库是否被成功加载。 当此值为 #f(库未加载)时,调用 ssl-connect 等将引发一个异常。

value

ssl-load-fail-reason : (or/c #f string?)

要么是 #f (当 ssl-available? 为 #t), 要么是一个错误字符串(当 ssl-available? 为 #f 时)。

1 类似 TCP 的客户端程序

使用 ssl-connect 或 ssl-connect/enable-break 来通过 TCP 创建一个 SSL 连接。 要创建一个安全连接,需要提供 ssl-secure-client-context 的结果, 或者用 ssl-make-client-context 创建一个客户端上下文, 并使用 上下文程序 中描述的功能对其进行配置。

procedure

(ssl-connect hostname    
  port-no    
  [client-protocol    
  #:alpn alpn-protocols]) → 
input-port? output-port?
  hostname : string?
  port-no : (integer-in 1 65535)
  client-protocol : (or/c ssl-client-context? ssl-protocol-symbol/c)
   = 'auto
  alpn-protocols : (listof bytes?) = null
连接到由 hostname 给出的主机,在由 port-no 给出的端口。这个连接将使用 SSL 进行加密。 返回值与 tcp-connect 相同:一个输入端口和一个输出端口。

默认的 'auto 协议是 insecure 。 使用 'secure 可以获得安全连接。详见 ssl-secure-client-context 。

可选的 client-protocol 参数决定了使用哪种加密协议,是否检查服务器的证书,等等。 该参数可以是由 ssl-make-client-context 创建的客户端上下文,也可以是指定要使用的协议的符号; 详情见 ssl-make-client-context ,包括协议符号的含义。

关闭产生的输出端口并不向服务器发送关闭消息。也请参见 ports->ssl-ports 。

如果启用了主机名验证(见 ssl-set-verify-hostname!), 对等体的证书将根据 hostname 进行检查。

如果 alpn-protocols 不是空的,客户端会尝试使用 ALPN 来协商应用级协议。 协议应按优先顺序列出,每个协议必须是一个长度在 1 到 255 之间(含)的字节字符串。 另见 ssl-get-alpn-selected 。

Changed in version 8.0.0.13: 增加了 #:alpn 参数。

procedure

(ssl-secure-client-context) → ssl-client-context?

返回一个客户端上下文,该客户端使用 (ssl-default-verify-sources) 的默认验证源验证证书, 验证主机名,并避免使用弱密码。 其结果基本上等同于以下内容:

(let ([ctx (ssl-make-client-context 'auto)])
  ; Load default verification sources (root certificates)
  (ssl-load-default-verify-sources! ctx)
  ; Require certificate verification
  (ssl-set-verify! ctx #t)
  ; Require hostname verification
  (ssl-set-verify-hostname! ctx #t)
  ; No weak cipher suites
  (ssl-set-ciphers! ctx "DEFAULT:!aNULL:!eNULL:!LOW:!EXPORT:!SSLv2")
  ; Seal context so further changes cannot weaken it
  (ssl-seal-context! ctx)
  ctx)

上下文被缓存了,所以对 ssl-secure-client-context 的不同调用会返回相同的上下文, 除非 (ssl-default-verify-sources) 已经改变。

注意, (ssl-secure-client-context) 返回的是一个密封的上下文,所以不可能向它添加私钥和证书链。 如果需要客户端证书,请使用 ssl-make-client-context 代替。

procedure

(ssl-make-client-context 
  [protocol 
  #:private-key private-key 
  #:certificate-chain certificate-chain]) 
 → ssl-client-context?
  protocol : ssl-protocol-symbol/c = 'auto
  private-key : 
(or/c (list/c 'pem path-string?)
      (list/c 'der path-string?)
      #f)
 = #f
  certificate-chain : (or/c path-string? #f) = #f
创建一个提供给 ssl-connect 的上下文。 该上下文是 insecure ,除非提供了 'secure 或采取了额外的步骤; 详见 ssl-secure-client-context 。

客户端上下文标识了一个通信协议(由 protocol 选择),并且还持有证书信息(即客户的身份、其信任的证书机构等)。 关于证书的更多信息,请参见下面的 上下文程序 部分。

该 protocol 应该是以下之一:

  • 'secure : 相当于 (ssl-secure-client-context) 。

  • 'auto : 自动协商本库认为足够安全的协议版本–目前是 TLS 1.0 及以上版本,但可能会有变化。

  • 'tls12 : 仅限 TLS 协议 1.2 版本。

  • 'tls13 : 只有 TLS 协议的 1.3 版本。

以下 protocol 符号已被废弃,但仍被支持:

  • 'sslv2-or-v3 : 自动的 'auto 。 请注意,尽管有这个名字,但 SSL 2.0 和 3.0 都不被认为是足够安全的, 所以这个 protocol 不再允许它们中的任何一个。

  • 'sslv2 : SSL 协议 2.0 版本。 被 RFC 6176 (2011)废止。 请注意,许多平台已经取消了对 SSL 2.0 的支持。

  • 'sslv3 : SSL 协议3.0版本。 已被 RFC 7568 (2015)废除。

  • 'tls : 只有 TLS 协议 1.0 版本。 被 RFC 8996 (2021)废弃。

  • 'tls11 : 只有 TLS 协议的 1.1 版本。 被 RFC 8996 (2021)弃用。

不是所有的服务器都支持所有的协议版本。 'secure 和 'auto 选项在合理的安全水平上提供广泛的兼容性。 请注意,连接的安全性不仅仅取决于协议版本; 详情见 ssl-secure-client-context 。 参见 supported-client-protocols 和 supported-server-protocols 。

如果提供了 private-key 和 certificate-chain , 它们将分别用 ssl-load-private-key! 和 ssl-load-certificate-chain! 加载到上下文中。 客户端证书很少用于 HTTPS,但它们偶尔会被用于其他类型的服务器。

2 类似 TCP 的服务器程序

procedure

(ssl-listen port-no    
  [queue-k    
  reuse?    
  hostname-or-#f    
  server-protocol]) → ssl-listener?
  port-no : listen-port-number?
  queue-k : exact-nonnegative-integer? = 5
  reuse? : any/c = #f
  hostname-or-#f : (or/c string? #f) = #f
  server-protocol : (or/c ssl-server-context?  ssl-protocol-symbol/c)
   = 'auto
和 tcp-listen 一样,但结果是一个 SSL 监听器。 额外的可选的 server-protocol 和 ssl-connect 一样, 除了一个上下文必须是一个服务器上下文而不是一个客户端上下文, 而且 'secure 只是 'auto 的一个别名。

调用 ssl-load-certificate-chain! 和 ssl-load-private-key! 以避免在接受连接时出现 no shared cipher 的错误。 在测试时,"openssl" 集合中的文件 "test.pem" 是两个调用的合适参数。 然而,由于 "test.pem" 是公开的,这样的测试配置显然不能提供安全。

一个 SSL 监听器是一个可同步的值(见 sync)。 当底层的 TCP 监听器准备好时,它就准备好了–以它自己为值。 然而,在这一点上,用 ssl-accept 接受一个连接可能不会立即完成,因为需要进一步的通信来建立连接。

Changed in version 6.3.0.12 of package base: 增加了 'secure 。

3 SSL 封装接口

procedure

(ports->ssl-ports input-port 
  output-port 
  [#:mode mode 
  #:context context 
  #:encrypt protocol 
  #:close-original? close-original? 
  #:shutdown-on-close? shutdown-on-close? 
  #:error/ssl error 
  #:hostname hostname 
  #:alpn alpn-protocols]) 
 → 
input-port? output-port?
  input-port : input-port?
  output-port : output-port?
  mode : (or/c 'connect 'accept) = 'accept
  context : (or/c ssl-client-context? ssl-server-context?)
   = 
((if (eq? mode 'accept)
     ssl-make-server-context
     ssl-make-client-context)
 protocol)
  protocol : ssl-protocol-symbol/c = 'auto
  close-original? : boolean? = #f
  shutdown-on-close? : boolean? = #f
  error : procedure? = error
  hostname : (or/c string? #f) = #f
  alpn-protocols : (listof bytes?) = null
返回两个值–一个输入端口和一个输出端口–通过给定的输入和输出端口实现 SSL 协议。 (给定的端口应该连接到另一个运行 SSL 协议的进程中)。

mode 参数可以是 'connect 或 'accept 。 模式决定了 SSL 协议是如何在端口上初始化的,是作为客户端还是作为服务器。 和 ssl-listen 一样,在 'accept 模式下, 提供一个已经被 ssl-load-certificate-chain! 和 ssl-load-private-key! 初始化的 context ,以避免出现 no shared cipher 的错误。

context 参数应该是 'connect 模式的客户机上下文或 'accept 模式的服务器上下文。 如果没有提供,将使用 protocol 参数所指定的协议创建一个上下文。

如果没有提供 protocol 参数,则默认为 'auto 。 参见 ssl-make-client-context 以了解更多细节(包括所有选项和协议符号的含义)。 如果提供了 context 参数,这个参数将被忽略。

如果 close-original? 为 true,那么当两个 SSL 端口都关闭时,给定的输入和输出端口会自动关闭。

如果 shutdown-on-close? 为真,那么当输出 SSL 端口被关闭时,它会向 SSL 连接的另一端发送一个关闭消息。 当关闭被启用时,如果给定的输出端口变得不可写(例如,因为给定端口的另一端已经被另一个进程关闭), 关闭输出端口可能会失败。

error 参数是一个错误程序,用于引发通信错误。 默认是 error, 它引发 exn:fail; 相反, ssl-accept 和 ssl-connect 使用一个错误函数, 引发 exn:fail:network 。

关于对 SSL 连接的读写限制(即一次一个方向),也请参见 ssl-connect 。

如果启用了主机名验证(见 ssl-set-verify-hostname!), 将根据 hostname 检查对等方的证书。

如果 alpn-protocols 不是空的,并且 mode 是 'connect, 那么客户端就会尝试使用 ALPN; 另见 ssl-connect 和 ssl-get-alpn-selected 。 如果 alpn-protocols 不是空的,并且 mode 是 'accept, 则会产生一个异常 (exn:fail) ; 使用 ssl-set-server-alpn! 来为服务器上下文设置 ALPN 协议。

Changed in version 8.0.0.13 of package base: 增加了 #:alpn 参数。

4 上下文程序

procedure

(ssl-load-verify-source! context    
  src    
  [#:try? try?]) → void?
  context : (or/c ssl-client-context? ssl-server-context?)
  src : 
(or/c path-string?
      (list/c 'directory path-string?)
      (list/c 'win32-store string?)
      (list/c 'macosx-keychain (or/c #f path-string?)))
  try? : any/c = #f
将验证源从 src 加载到 context 中。 目前,只加载证书;这些证书用于验证连接对等体的证书。 多次调用此过程以加载多组可信的证书。

支持下列类型的验证源:

  • 如果 src 是一个路径或字符串,它将被视为一个包含根证书的 PEM 文件。该文件会被立即加载。

  • 如果 src 是 (list 'directory dir), 那么 dir 应该包含带有散列符号链接的 PEM 文件(参见 openssl c_rehash 工具)。 目录内容不会被立即加载;相反,只有在证书需要验证时才会被搜索。

  • 如果 src 是 (list 'win32-store store), 则立即加载名为 store 的存储空间中的证书。仅在 Windows 上支持。

  • 如果 src 是 (list 'macosx-keychain #f), 则立即加载 Mac OS 信任锚(根)证书 (如 SecTrustCopyAnchorCertificates) 所返回)中的证书。仅在 Mac OS 上支持。

  • 如果 src 是 (list 'macosx-keychain path), 则立即加载存储在 path 上的钥匙串中的证书。仅在 Mac OS 上支持。

如果 try? 为 #f ，且加载 src 失败(例如,因为文件或目录不存在), 则会产生一个异常。 如果 try? 是一个真值,那么加载失败将被忽略。

你可以使用 "openssl" 集合中的 "test.pem" 文件进行测试。 由于 "test.pem" 是公开的,这样的测试配置显然不能提供安全保障。

Changed in version 8.4.0.5 of package base: 增加了 (list 'macosx-keychain #f) 变量。

parameter

(ssl-default-verify-sources)

 → 
(let ([source/c (or/c path-string?
                      (list/c 'directory path-string?)
                      (list/c 'win32-store string?)

procedure

(ssl-set-server-name-identification-callback! context    
  callback) → void?
  context : ssl-server-context?
  callback : (string? . -> . (or/c ssl-server-context? #f))
为 SSL 服务器上下文提供了一个程序,它可以用来在每个连接的基础上切换到其他上下文。 该程序被赋予客户端试图连接的主机名,以作为其决定的基础。

客户端通过 TLS 的 Server Name Identification 扩展来发送这一信息,该扩展是为了允许安全服务器的 virtual hosting 。

建议使用的方法是准备适当的服务器上下文,定义一个可以在它们之间调度的单一回调, 然后在密封它们之前将其应用于所有的上下文。 一个最小的例子:

(define ctx-a (ssl-make-server-context 'tls))
(define ctx-b (ssl-make-server-context 'tls))
...
(ssl-load-certificate-chain! ctx-a "cert-a.pem")
(ssl-load-certificate-chain! ctx-b "cert-b.pem")
...
(ssl-load-private-key! ctx-a "key-a.pem")
(ssl-load-private-key! ctx-b "key-b.pem")
...
(define (callback hostname)
  (cond [(equal? hostname "a") ctx-a]
        [(equal? hostname "b") ctx-b]
        ...
        [else #f]))
(ssl-set-server-name-identification-callback! ctx-a callback)
(ssl-set-server-name-identification-callback! ctx-b callback)
...
(ssl-seal-context! ctx-a)
(ssl-seal-context! ctx-b)
...
(ssl-listen 443 5 #t #f ctx-a)

如果回调返回 #f,连接尝试将继续,使用原来的服务器上下文。

procedure

(ssl-set-server-alpn! context    
  alpn-protocols    
  [allow-no-match?]) → void?
  context : ssl-server-context?
  alpn-protocols : (listof bytes?)
  allow-no-match? : boolean? = #t
设置服务器上下文所支持的 ALPN 协议。这些协议是按优先级排列的,最优先的在前。 也就是说,当客户端连接时,服务器会选择其 alpn-protocols 中第一个被客户端支持的协议。 如果客户端不使用 ALPN,那么连接被接受,并且不选择任何协议。 如果客户端使用 ALPN,但与服务器没有共同的协议,那么如果 allow-no-match? 为真, 则接受连接,不选择协议; 如果 allow-no-match? 为假,则拒绝连接。

Added in version 8.4.0.5 of package base.

procedure

(ssl-set-keylogger! context logger) → void?

  context : (or/c ssl-server-context? ssl-client-context?)
  logger : (or/c #f logger?)
指示 context 在产生或收到TLS密钥材料时向 logger 记录一条消息。 该消息被记录下来,其级别被设置为 'debug, 其主题被设置为 'openssl-keylogger, 其相关数据是代表密钥材料的字节字符串。 当 logger 为 #f 时,上下文被指示停止记录这些信息。

警告: 如果 logger 有任何祖先,那么这些信息也可能被他们获得,这取决于 logger 的传播设置。

调试是这个功能的典型用例。 上下文的所有者可以使用它将关键材料写入文件,供 Wireshark 等工具使用。 在下面的例子中,任何能够访问 "keylogfile.txt" 的人都能够解密通过 ctx 进行的连接:

(define out
  (open-output-file
   #:exists 'append
   "keylogfile.txt"))
(define logger
  (make-logger))
(void
 (thread
  (lambda ()
    (define receiver
      (make-log-receiver logger 'debug 'openssl-keylogger))
    (let loop ()
      (match-define (vector _ _ key-data _)
        (sync receiver))
      (write-bytes key-data out)
      (newline out)
      (flush-output out)
      (loop)))))
 
(define ctx (ssl-make-client-context 'auto))
(ssl-set-keylogger! ctx logger)

5 对等体验证

procedure

(ssl-set-verify! clp on?) → void?

  clp : 
(or/c ssl-client-context? ssl-server-context?
      ssl-listener? ssl-port?)
  on? : any/c
当 on? 为 #t 时,要求对等 SSL 连接的证书验证。 如果 clp 是一个 SSL 端口,那么连接将立即被重新协商,如果证书验证失败,将立即引发一个异常。 如果 clp 是一个上下文或监听器,证书验证会发生在使用该上下文或监听器的每个后续连接上。

启用验证还需要至少用 ssl-load-verify-source! 指定受信任的证书机构。

验证证书并不足以防止主动对手的攻击,如 man-in-the-middle attacks 。 另请参见 ssl-set-verify-hostname! 。
