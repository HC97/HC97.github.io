15.4 受信任和不信任代码的代码检查员

Code inspectors 提供了确定哪些模块被信任使用 module->namespace 这样的函数或 ffi/unsafe 等不安全模块的机制。 当一个模块被声明时, current-code-inspector 的值与该模块声明相关。 当一个模块被实例化时(即,当声明的主体被实际执行时),会创建一个子检查器来保护模块的出口。 访问模块的 protected 出口需要一个比模块实例化检查器更强的代码检查器(即在检查器层次结构中更高); 注意,模块的声明检查器总是比其实例化检查器更强,所以用相同代码检查器声明的模块可以访问对方的出口。

为了区分可信代码和不可信代码,首先加载可信代码, 然后将 current-code-inspector 设置为 (make-inspector (current-code-inspector)) 的结果, 以安装一个较弱的检查器,最后在较弱的检查器到位的情况下加载不可信代码。 当任何不受信任的代码被运行时,较弱的检查器应该留在原地。 如果有必要,可信代码可以在可信代码的动态范围内暂时恢复原来的检查器(只要它不回调到不可信代码)。

模块中的语法对象常量,如模板中的字面标识符,保留其源模块的检查器。 这样,一个受信任模块的宏可以在一个不受信任的模块中使用, 宏扩展中的 protected 标识符仍然有效,即使它们最终出现在一个不受信任的模块中。 为了防止通过从扩展的代码中提取标识符来滥用标识符, 像 local-expand 这样的函数是 protected , 而像 expand 这样的函数如果没有给一个足够强大的检查器,就会返回 tainted 。

不幸的是,来自 ".zo" 文件的编译代码本质上是不可信的, 因为它可以通过 compile 以外的方式进行合成。 当编译后的代码被写入 ".zo" 文件时,编译后的代码中的语法对象常量会失去它们的检查器。 当代码被加载时,编译后的代码中的所有语法对象常量都会获得包围模块的声明时检查器。
