2 Connections

本节介绍了创建连接的功能以及管理连接的管理功能。

2.1 基础连接

有五种基础连接,它们被分为两组: wire-based connections 和 FFI-based connections 。 PostgreSQL、MySQL 和 Cassandra 连接是基于线的,而 SQLite 和 ODBC 连接是基于 FFI 的。 请参见 基于 FFI 的连接和并发性 。

基本连接是使用以下函数建立的。

procedure

(postgresql-connect 
  #:user user 
  #:database database 
  [#:server server 
  #:port port 
  #:socket socket 
  #:password password 
  #:allow-cleartext-password? allow-cleartext-password? 
  #:ssl ssl 
  #:ssl-context ssl-context 
  #:notice-handler notice-handler 
  #:notification-handler notification-handler]) 
 → connection?
  user : string?
  database : string?
  server : string? = "localhost"
  port : exact-positive-integer? = 5432
  socket : (or/c path-string? 'guess #f) = #f
  password : (or/c string? #f) = #f
  allow-cleartext-password? : (or/c boolean? 'local) = 'local
  ssl : (or/c 'yes 'optional 'no) = 'no
  ssl-context : ssl-client-context? = (ssl-make-client-context)
  notice-handler : 
(or/c 'output 'error output-port?
      (-> string? string? any))
 = void
  notification-handler : 
(or/c 'output 'error output-port?
      (-> string? any)
      (-> string? string? any))
   = void
创建一个到 PostgreSQL 服务器的连接。 只有 database 和 user 参数是必须的。

默认情况下,连接是通过 TCP 与 5432 端口的 "localhost" 建立的。 要建立一个不同的 TCP 连接,请提供一个或两个 server 和 port 参数。

要通过本地套接字进行连接,请在 socket 参数中指定套接字路径。 如果你还提供了 TCP 参数中的任何一个,你就不能提供 socket 参数。 关于套接字路径的说明,请参见 用于 Postgre SQL 和 My SQL 服务器的本地套接字 。 提供一个 'guess 的 socket 参数与提供 (postgresql-guess-socket-path) 是一样的。

如果服务器要求密码验证, password 参数必须存在,否则会产生一个异常。 如果服务器不要求密码认证, password 参数会被忽略,可以省略。 一个连接通常只发送密码散列(使用 md5 或 scram-sha-256 认证方法)。 如果服务器要求以明文(未加密)方式发送密码,连接将被终止, 除非 allow-cleartext-password? 是 #t, 或者 allow-cleartext-password? 是 'local , 并且连接是到 "localhost" 或一个本地套接字。

如果 ssl 参数是 'yes 或 'optional, 该连接将尝试协商一个 SSL 连接。 如果服务器拒绝 SSL,如果 ssl 被设置为 'yes ,连接会引发一个异常; 如果 ssl 被设置为 'optional ,则继续使用未加密的连接。 一些服务器使用 SSL 证书来验证客户, 见 ssl-load-certificate-chain! 和 ssl-load-private-key! 。 SSL 只能用于 TCP 连接,不能用于本地套接字。 请参阅 安全地建立数据库连接 。

notice-handler 在收到来自服务器的异步通知信息时被调用。 一个常见的例子是为一个表的主键自动创建索引的通知。 notice-handler 函数需要两个字符串参数:条件的 SQLSTATE 和一个消息。 notification-handler 在响应事件通知时被调用(见 LISTEN 和 NOTIFY 语句); 它的参数是通道的名称和有效载荷,作为字符串。 如果处理程序只接受一个参数,那么它将被调用而不包含有效载荷。 在 PostgreSQL 9.0 中加入了在通知中包含有效载荷的能力。 可以提供一个输出端口,而不是一个处理程序,在这种情况下,一个消息将被打印到指定的端口。 最后,符号 'output 导致消息被打印到当前的输出端口, 而 'error 导致消息被打印到当前的错误端口。

如果不能建立连接,将引发一个异常。

Examples:
> (postgresql-connect #:server "db.mysite.com"
                      #:port 5432
                      #:database "webappdb"
                      #:user "webapp"
                      #:password "ultra5ecret")

(object:connection% ...)

> (postgresql-connect #:user "me"
                      #:database "me"
                      #:password "icecream")

(object:connection% ...)

> (postgresql-connect ; Typical socket path
                      #:socket "/var/run/postgresql/.s.PGSQL.5432"
                      #:user "me"
                      #:database "me")

(object:connection% ...)

> (postgresql-connect #:socket 'guess ; or (postgresql-guess-socket-path)
                      #:user "me"
                      #:database "me")

(object:connection% ...)

Changed in version 1.2 of package db-lib: 增加了对 SCRAM-SHA-256 认证的支持。
Changed in version 1.7: 增加了对 SCRAM-SHA-256-PLUS 认证的支持。

(postgresql-guess-socket-path) → path-string?

试图根据常规位置猜测套接字的路径。 这个函数返回文件系统中存在的第一个这样的路径。 它不检查该路径是否是一个套接字文件,也不检查该路径是否连接到 PostgreSQL 服务器。

procedure

(sqlite3-connect #:database database 
  [#:mode mode 
  #:busy-retry-limit busy-retry-limit 
  #:busy-retry-delay busy-retry-delay 
  #:use-place use-place]) 
 → connection?
  database : (or/c path-string? 'memory 'temporary)
  mode : (or/c 'read-only 'read/write 'create) = 'read/write
  busy-retry-limit : (or/c exact-nonnegative-integer? +inf.0)
   = 10
  busy-retry-delay : (and/c rational? (not/c negative?)) = 0.1
  use-place : (or/c boolean? 'os-thread 'place) = #f
如果 database 是一个字符串或路径,在 database 命名的文件中打开 SQLite 数据库。 如果 database 是 'temporary, 将创建一个基于磁盘的私有数据库。 如果 database 是 'memory, 将创建一个基于内存的私有数据库。

如果 mode 是 'read-only, 数据库将以只读模式打开。 如果 mode 是 'read/write (默认),数据库将被打开用于读写(如果文件系统权限允许)。 'create 模式与 'read/write 模式类似, 只是如果给定的文件不存在,就会作为一个新的数据库来创建。

SQLite 使用 coarse-grained locking, 当无法获得锁时,许多内部操作会以 SQLITE_BUSY 条件失败。 当一个内部操作因为数据库繁忙而失败时,连接会休眠 busy-retry-delay 秒, 然后重试操作,最多重试次数为 busy-retry-limit 。 如果 busy-retry-limit 为 0, 则该操作只尝试一次。 如果在 busy-retry-limit 重试之后,操作仍然没有成功,就会产生一个异常。

如果 use-place 是 'os-thread, 那么查询将在一个 separate OS thread 中执行。 如果 use-place 是 'place, 那么实际连接是在一个单独的数据库连接 place 中创建的,并返回一个代理。 如果 use-place 是 #t, 那么如果有的话,它就像 'os-thread ,否则就像 'place 。 更多信息请参见 基于 FFI 的连接和并发性 。

如果不能建立连接,将引发一个异常。

Examples:
> (sqlite3-connect #:database "/path/to/my.db")

(object:connection% ...)

> (sqlite3-connect #:database "relpath/to/my.db"
                   #:mode 'create)

(object:connection% ...)

报告是否找到了 SQLite 本地库,在这种情况下, sqlite3-connect 会工作,否则会引发异常。

  notice-handler : 
(or/c output-port? 'output 'error
      (-> string? string? any))
 = void

  quirks : (listof symbol?) = null
  use-place : (or/c boolean? 'place 'os-thread) = #f
创建一个与名为 dsn 的 ODBC 数据源的连接。 user 和 password 参数是可选的, 因为这些信息可能被纳入数据源的定义中,或者与数据源的驱动程序无关。 notice-handler 参数的行为与 postgresql-connect 中的行为相同。

如果 strict-parameter-types? 为 true,那么连接将尝试确定并执行查询参数的特定类型。 参见 ODBC 类型 以了解更多细节。

character-mode 参数控制字符数据的处理;支持以下值:

  • 'wchar (默认值) – 使用 SQL_C_WCHAR , 并将数据视为 UTF-16(或者当驱动管理器为 iODBC 时为 UTF-32/UCS-4)。

  • 'utf-8 – 使用 SQL_C_CHAR 并将数据作为 UTF-8 处理

  • 'latin-1 – 使用 SQL_C_CHAR 并将数据作为 Latin-1 处理。 不在 Latin-1 中的字符会被替换成 #\? 。

quirks 参数代表一个标志列表,用于修改连接的行为。 目前支持以下怪癖:

  • 'no-c-bigint — 不要使用 SQL_C_BIGINT 来绑定参数或获取字段值。

  • 'no-c-numeric — 不要使用 SQL_C_NUMERIC 来获取 NUMERIC/DECIMAL 值。

关于具体的 ODBC 驱动程序的说明和连接选项的建议,请参见 ODBC 状态 。

use-place 参数的解释与 sqlite3-connect 相同。

如果不能建立连接,会产生一个异常。

2.2 连接池

创建一个数据库连接可能是一个昂贵的操作;它可能涉及到进程创建和 SSL 协商等步骤。 connection pool 有助于通过重复使用连接来减少连接成本。

procedure

(connection-pool connect 
  [#:max-connections max-connections 
  #:max-idle-connections max-idle-connections]) 
 → connection-pool?
  connect : (-> connection?)
  max-connections : (or/c (integer-in 1 10000) +inf.0) = +inf.0
  max-idle-connections : (or/c (integer-in 1 10000) +inf.0) = 10
创建一个 connection pool 。 该池由最多 max-connections 组成, 分为租赁连接和最多 max-idle-connections 的空闲连接。 该池在需要时使用 connect 来创建新的连接。 connect 函数的调用与创建连接池时的 current-custodian 值相同, 并且每次调用时必须返回一个新的连接。

Examples:
> (define pool
    (connection-pool
     (lambda () (displayln "connecting!") (sqlite3-connect ....))
     #:max-idle-connections 1))
> (define c1 (connection-pool-lease pool))

connecting!

> (define c2 (connection-pool-lease pool))

connecting!

> (disconnect c1)
> (define c3 (connection-pool-lease pool)) ; reuses actual conn. from c1

参见 virtual-connection , 该机制消除了明确调用 connection-pool-lease 和 disconnect 。

procedure

(connection-pool? x) → boolean?

  x : any/c
如果 x 是一个连接池,返回 #t , #f 。

procedure

(connection-pool-lease pool [release]) → connection?

  pool : connection-pool?
  release : (or/c evt? custodian?) = (current-thread)
从连接 pool 中获取一个连接,如果池中有空闲的连接,则使用一个现有的空闲连接。 如果没有空闲的连接,并且连接池中的连接数少于其允许的最大连接数,那么将创建一个新的连接;否则将引发一个异常。

在获得的连接上调用 disconnect 会导致该连接被释放到连接池中。 如果 release 变得可用,如果它是一个可同步的事件,或者如果 release 被关闭, 如果它是一个监管者,那么连接也会被释放。 release 的默认值是当前线程,所以当请求连接的线程终止时,产生的连接会被释放。

当一个连接被释放时,如果 pool 的空闲连接限制不会被超过,它将被保留为一个空闲连接;否则,它将被断开连接。 在这两种情况下,如果该连接是在一个事务中,该事务将被回滚。

2.3 虚拟连接

virtual connection 按需创建实际连接,并在不再需要时自动释放它们。

procedure

(virtual-connection connect) → connection?

  connect : (or/c (-> connection?) connection-pool?)
创建一个 virtual connection ,使用 connect 函数按需创建实际连接, 如果 connect 是一个 connection pool, 则通过调用 (connection-pool-lease connect) 。 如果 connect 是一个函数,它被调用时的 current-custodian 值与创建虚拟连接时相同。

一个虚拟连接封装了线程与实际连接的映射。 当一个查询函数以虚拟连接被调用时,当前线程的相关实际连接被用来执行查询。 如果没有与当前线程相关的实际连接,则通过调用 connect 获得一个实际连接。 实际连接在其相关线程死亡时被断开。

虚拟连接在 Web servlets(见 数据库和 Web Servlets)等情况下特别有用, 在这些情况下,每个请求都在一个新的线程中处理。 可以定义一个单一的全局虚拟连接,将每个 Servlet 请求从明确地打开和关闭自己的连接中解放出来。 特别是,一个由 connection pool 支持的 virtual connection 结合了便利和效率:

(define the-connection
  (virtual-connection (connection-pool (lambda () ....))))

所产生的虚拟连接为每个 servlet 请求线程从连接池中租赁一个连接,并在线程终止时(也就是请求被处理后)释放该连接。

当给定一个由 virtual-connection 产生的连接时, connected? 表明是否有一个与当前线程相关的实际连接。 同样, disconnect 会导致当前与线程相关的实际连接(如果有的话)被断开, 但如果查询函数被执行,该连接将被重新创建。

Examples:
> (define c
    (virtual-connection
     (lambda ()
       (printf "connecting!\n")
       (postgresql-connect ....))))
> (connected? c)

#f

> (query-value c "select 1")

connecting!

1

> (connected? c)

#t

> (void (thread (lambda () (displayln (query-value c "select 2")))))

connecting!

2

> (disconnect c)
> (connected? c)

#f

> (query-value c "select 3")

connecting!

3

由 virtual-connection 产生的连接不能被用于 prepare 函数。 然而,它们仍然可以被用来执行以字符串表示的参数化查询,或者通过 virtual-statement 进行封装。

Examples:
> (prepare c "select 2 + $1")

prepare: cannot prepare statement with virtual connection

> (query-value c "select 2 + $1" 2)

4

> (define pst (virtual-statement "select 2 + $1"))
> (query-value c pst 3)

5

2.4 杀死安全连接

procedure

(kill-safe-connection c) → connection?

  c : connection?
为连接 c 创建一个代理。 所有通过代理进行的查询都是杀戮安全的; 也就是说,如果一个线程在调用查询函数(如 query )时被杀死,连接将不会被锁定或损坏。 (连接通常是线程安全的,但不是杀戮安全的)。

注意:一个杀戮安全的连接,其底层连接使用端口与数据库服务器进行通信,不会受到监管者关闭其端口的保护。

2.5 数据源名称

DSN(数据源名称)是一个与 DSN 文件中的连接规范相关的符号。它们受 ODBC 的 DSN 启发,但与之不同。

struct

(struct data-source (connector args extensions)
    #:mutable)
  connector : (or/c 'postgresql 'mysql 'sqlite3 'odbc 'odbc-driver)
  args : list?
  extensions : (listof (list/c symbol? any/c))
代表一个数据源。 connector 字段决定了使用哪个连接函数来创建连接。 args 字段是传递给连接函数的部分参数列表; 当 dsn-connect 被调用时,可以添加额外的参数。 extensions 字段包含关于一个连接的额外信息;例如,这个库的测试框架使用它来存储 SQL 方言标志。

数据源也可以用 postgresql-data-source 等辅助函数来创建。

procedure

(dsn-connect dsn    
  [#:dsn-file dsn-file]    
  arg ...    
  #:<kw> kw-arg ...) → connection?
  dsn : (or/c symbol? data-source?)
  dsn-file : path-string? = (current-dsn-file)
  arg : any/c
  kw-arg : any/c
使用与 dsn-file 中 dsn 相关的连接信息建立一个连接。 给定的 arg 和 kw-arg 被添加到 dsn 指定的参数中, 形成提供给 connect 函数的完整参数。

如果 dsn-file 不存在,或者它不包含 dsn 的条目,将产生一个异常。 如果 dsn 是一个 data-source, 那么 dsn-file 被忽略。

Examples:
> (put-dsn 'pg
           (postgresql-data-source #:user "me"
                                   #:database "mydb"
                                   #:password "icecream"))
> (dsn-connect 'pg)

(object:connection% ...)

> (dsn-connect 'pg #:notice-handler (lambda (code msg) ....))

(object:connection% ...)

parameter

(current-dsn-file) → path-string?

(current-dsn-file x) → void?
  x : path-string?
保留默认 DSN 文件位置的参数。 初始值是位于 (find-system-path 'prefs-dir) 内的文件。

procedure

(get-dsn dsn [default #:dsn-file dsn-file])

 → (or/c data-source? any/c)
  dsn : symbol?
  default : any/c = #f
  dsn-file : path-string? = (current-dsn-file)
返回与 dsn-file 中的 dsn 相关的 data-source 。

如果 dsn-file 不存在,会产生一个异常。 如果 dsn-file 中没有 dsn 条目, 如果它是一个函数,将调用 default ,否则将返回。

procedure

(put-dsn dsn ds [#:dsn-file dsn-file]) → void?

  dsn : symbol?
  ds : (or/c data-source? #f)
  dsn-file : path-string? = (current-dsn-file)
将 dsn 与 dsn-file 中给定的数据源 ds 联系起来, 如果存在的话,将取代以前的联系。

2.7 系统专用模块

db 模块导出本手册中列出的所有函数,除了 实用工具 中描述的那些。 数据库系统特定的连接模块被懒散地加载,以避免对国外库的不必要的依赖。

下面的模块提供了本手册中描述的绑定的子集。

 (require db/base) package: db-lib

提供所有的通用连接操作(在 管理连接 和 查询 中描述的那些)和 SQL 数据支持 (SQL 类型转换) 。

 (require db/postgresql) package: db-lib

只提供 postgresql-connect 和 postgresql-guess-socket-path 。

 (require db/mysql) package: db-lib

只提供 mysql-connect 和 mysql-guess-socket-path 。

 (require db/cassandra) package: db-lib

只提供 cassandra-connect 和 cassandra-consistency 。

 (require db/sqlite3) package: db-lib

提供 sqlite3-connect 和 sqlite3-available? 。 当无法找到 SQLite 本地库时, sqlite3-connect 会引发一个异常。

 (require db/odbc) package: db-lib

只提供 odbc-connect, odbc-driver-connect, odbc-data-sources 和 odbc-drivers 。 与 db 不同的是,该模块在需要时立即尝试加载 ODBC 本地库,如果找不到它,则引发异常。
