2 有状态的小程序

 (require web-server/servlet) package: web-server-lib

一个有状态的 servlet 应该 provide 以下输出:

value

interface-version : (one-of/c 'v2)

这表明该 servlet 是一个版本二的 servlet。

value

manager : manager?

该 servlet 的延续 manager 。参见 延续管理器 的选项。

procedure

(start initial-request) → can-be-response?

  initial-request : request?
当这个 servlet 的一个实例被启动时,这个函数被调用。参数是启动该实例的 HTTP 请求。

一个版本 2 模块的例子:

#lang racket
(require web-server/http
         web-server/managers/none)
(provide interface-version manager start)
 
(define interface-version 'v2)
(define manager
  (create-none-manager
   (lambda (req)
     (response/xexpr
      `(html (head (title "No Continuations Here!"))
             (body (h1 "No Continuations Here!")))))))
(define (start req)
  (response/xexpr
   `(html (head (title "Hello World!"))
          (body (h1 "Hi Mom!")))))

这些 servlet 有广泛的 API 可用: net/url, web-server/http, web-server/http/bindings, web-server/servlet/servlet-structs, web-server/servlet/web, web-server/servlet/web-cells 和 web-server/dispatch 。 其中一些将在下面的小节中记录。

2.2 响应

 (require web-server/servlet/servlet-structs)
  package: web-server-lib
Servlets 通过返回 HTTP 响应与 Web 服务器通信。 为了适应轻量级程序(和向后兼容), Web 服务器提供了一个从应用程序特定的响应格式和内部 HTTP 响应格式 response 的转接。

procedure

(can-be-response? x) → boolean?

  x : any/c

procedure

(any->response x) → (or/c false/c response?)

  x : any/c

procedure

(set-any->response! new-any->response) → void

  new-any->response : (-> any/c (or/c false/c response?))
any->response 将任何值强制转换成响应,如果强制不可能,则返回 #f 。 any->response 保证任何 response? 输入必须总是准确返回(即 eq? 。) 默认情况下总是返回 #f, 表示不可能有胁迫行为。

can-be-response? 如果 x 是一个响应或者可以通过调用 any->response 变成一个响应,则返回 #t 。
any->response 的用户应该通过使用 can-be-response? 作为契约来保护自己。 如果他们这样做,他们可以安全地忽略 any->response 的 #f 返回情况。
set-any->response! 用提供的参数替换全局的 any->response 。 这个函数应该为 eq? 输入返回相同的值,以确保 can-be-response? 是任何准确的谓词。 同样地,这个函数应该是便宜的,可以在同一个输入上多次调用,因为它将被用于契约检查以及传输前的强制执行。 你可能想使用一个基于弱 eq? 的哈希表来缓存结果,以达到这个目的。 (参见 make-weak-hasheq 。)

2.3 网络交互

 (require web-server/servlet/web) package: web-server-lib
web-server/servlet/web 库提供了 servlet 开发者感兴趣的主要功能。

procedure

(send/back response) → void?

  response : can-be-response?
向客户端发送 response 。 没有捕捉到延续,所以 servlet 就完成了。

例如:
(send/back
 (response/xexpr
  `(html
    (body
     (h1 "The sum is: "
         ,(+ first-number
             second-number))))))

procedure

(send/suspend make-response) → request?

  make-response : (string? . -> . can-be-response?)
捕获当前的延续, 用 (current-servlet-continuation-expiration-handler) 存储它作为过期处理程序, 并将它绑定到一个 URL 上。用这个 URL 调用 make-response , 预计会生成一个 can-be-response?, 并发送给客户端。 如果调用 continuation URL,则调用捕获的 continuation,并从这个调用 send/suspend 返回请求。

例如:
(send/suspend
 (lambda (k-url)
   (response/xexpr
    `(html (head (title "Enter a number"))
           (body
            (form ([action ,k-url])
                  "Enter a number: "
                  (input ([name "number"]))
                  (input ([type "submit"]))))))))

当这个表单被浏览器提交时,请求将被发送到由 send/suspend 生成的 URL。 因此,请求将从 send/suspend 中 "返回" 到这个调用的继续中。

procedure

(send/suspend/url make-response) → request?

  make-response : (url? . -> . can-be-response?)
与 send/suspend 类似,但有一个 URL 结构。

procedure

(send/suspend/dispatch make-response) → any

  make-response : (((request? . -> . any) . -> . string?) . -> . can-be-response?)
用一个函数(通常被命名为 embed/url)调用 make-response , 当用 request? 到 any/c 的过程调用时,将产生一个 URL, 当调用时将用 request? 对象调用该函数并将结果返回给 send/suspend/dispatch 的调用者。 因此,如果你把 embed/url 传递给身份函数, send/suspend/dispatch 就会演变成 send/suspend:

(define (send/suspend response-generator)
  (send/suspend/dispatch
   (lambda (embed/url)
     (response-generator (embed/url (lambda (x) x))))))

当一个页面有多个 "逻辑" 延续时,使用 send/suspend/dispatch 。 例如,我们可以对一个数字进行加法或减法:
(define (count-dot-com i)
  (count-dot-com
   (send/suspend/dispatch
    (lambda (embed/url)
      (response/xexpr
       `(html
         (head (title "Count!"))
         (body
          (h2 (a ([href
                   ,(embed/url
                     (lambda (req)
                       (sub1 i)))])
                 "-"))
          (h1 ,(number->string i))
          (h2 (a ([href
                   ,(embed/url
                     (lambda (req)
                       (add1 i)))])
                 "+")))))))))
注意,在这个例子中,处理程序的结果被返回到 send/suspend/dispatch 的延续中。 然而,在你的应用程序中, send/suspend/dispatch 的返回值是不相关的, 这是非常常见的,你可以把它看作是 "嵌入" 无价值的回调。 下面是这个风格的相同例子:
(define (count-dot-com i)
  (send/suspend/dispatch
   (lambda (embed/url)
     (response/xexpr
      `(html
        (head (title "Count!"))
        (body
         (h2 (a ([href
                  ,(embed/url
                    (lambda (req)
                      (count-dot-com (sub1 i))))])
                "-"))
         (h1 ,(number->string i))
         (h2 (a ([href
                  ,(embed/url
                    (lambda (req)
                      (count-dot-com (add1 i))))])
                "+"))))))))

procedure

(send/suspend/url/dispatch make-response) → any

  make-response : (((request? . -> . any) . -> . url?) . -> . can-be-response?)
像 send/suspend/dispatch, 但有一个 URL 结构。

procedure

(send/forward make-response) → request?

  make-response : (string? . -> . can-be-response?)
调用 clear-continuation-table!, 然后是 send/suspend 。

如果用户在你的应用程序中逻辑上可以 "向前",但不能 "向后",则使用这种方式。

procedure

(send/finish response) → void?

  response : can-be-response?
调用 clear-continuation-table!, 然后 send/back 。

如果用户真正 "完成" 了你的应用程序,就使用这个方法。 例如,它可能被用来显示退出后的页面:
(send/finish
 (response/xexpr
  `(html (head (title "Logged out"))
         (body (p "Thank you for using the services "
                  "of the Add Two Numbers, Inc.")))))

procedure

(redirect/get [#:headers hs]) → request?

  hs : (listof header?) = empty
用 redirect-to 调用 send/suspend ,将 hs 作为头信息, see-other 作为 redirection status 。

这实现了 Post-Redirect-Get 模式。 用它来防止 Refresh 按钮的重复效果,例如向数据库添加项目。

Changed in version 1.3 of package web-server-lib: 使用 see-other 而不是 temporarily 。

procedure

(redirect/get/forget [#:headers hs]) → request?

  hs : (listof header?) = empty
像 redirect/get, 但使用 send/forward 而不是 send/suspend 。

Changed in version 1.3 of package web-server-lib: 使用 see-other 而不是 temporarily, 和 redirect/get 一样。

value

current-servlet-continuation-expiration-handler

 : 
(parameter/c (or/c false/c
                   (request? . -> . can-be-response?)))
保存过期处理程序,当在此上下文中捕获的续集过期时使用,然后查找。

例如:
(parameterize
    ([current-servlet-continuation-expiration-handler
      (lambda (req)
        (response/xexpr
         `(html (head (title "Custom Expiration!")))))])
  (send/suspend
   ....))

procedure

(clear-continuation-table!) → void?

调用 Servlet 管理器的 clear-continuation-table! 函数。 通常情况下,这将删除所有先前捕获的延续。

procedure

(with-errors-to-browser send/finish-or-back    
  thunk) → any
  send/finish-or-back : (can-be-response? . -> . request?)
  thunk : (-> any)
调用 thunk 的异常处理程序,生成一个 HTML 错误页面并调用 send/finish-or-back 。

例如:
(with-errors-to-browser
 send/back
 (lambda ()
   (/ 1 (get-number (request-number)))))

procedure

(adjust-timeout! t) → void?

  t : number?
调用 servlet 管理器的 adjust-timeout! 函数。

Warning: 这是一个废弃的功能,在未来的版本中会被删除。

procedure

(continuation-url? u)

 → (or/c false/c (list/c number? number? number?))
  u : url?
检查 u 是否是一个指向续集的 URL,如果是,则返回实例 ID、续集 ID 和 nonce。

value

servlet-prompt : continuation-prompt-tag?

用于 Web 交互式续集捕获的标签。

2.4 网络单元

 (require web-server/servlet/web-cells)
  package: web-server-lib
web-server/servlet/web-cells 库提供了 Web 单元的接口。
一个 Web 单元是一种相对于 frame tree 定义的状态。 帧树是用户浏览会话的一个镜像。 每次调用一个续集时,都会创建一个新的框架(称为 current frame) , 作为捕获续集时当前框架的一个子框架。
如果你想让一个效果被封装在与正在生成的 HTTP 响应相联系的所有交互中(在传递意义上),你应该使用Web单元。 关于其语义的更多信息,请参考论文 "Interaction-Safe State for the Web" 。

procedure

(web-cell? v) → boolean?

  v : any/c
确定 v 是否是一个 web-cell。

procedure

(make-web-cell v) → web-cell?

  v : any/c
创建一个默认值为 v 的 web-cell。

procedure

(web-cell-ref wc) → any/c

  wc : web-cell?
查阅最近的框架中发现的 wc 的值。

procedure

(web-cell-shadow wc v) → void

  wc : web-cell?
  v : any/c
将 wc 绑定到当前帧中的 v 上,影射当前帧中与 wc 的任何其他绑定关系。

下面是一个扩展的例子,演示了 Web 单元如何允许创建可重用的 Web 抽象, 而不需要将程序全局性地转变为延续或存储传递风格。
#lang web-server/insta
 
(define (start initial-request)
  (define counter1 (make-counter))
  (define counter2 (make-counter))
  (define include1 (include-counter counter1))
  (define include2 (include-counter counter2))
  (send/suspend/dispatch
   (lambda (embed/url)
     (response/xexpr
      `(html
        (body (h2 "Double Counters")
              (div (h3 "First")
                   ,(include1 embed/url))
              (div (h3 "Second")
                   ,(include2 embed/url))))))))
 
(define (make-counter)
 (make-web-cell 0))
 
(define (include-counter a-counter)
 (call-with-current-continuation
  (λ (k)
    (let loop ()
      (k
       (lambda (embed/url)
         `(div (h3 ,(number->string (web-cell-ref a-counter)))
               (a ([href
                    ,(embed/url
                      (lambda _
 
                        (define last (web-cell-ref a-counter))
 
                        (web-cell-shadow a-counter (add1 last))
 
                        (loop)))])
                  "+"))))))
  servlet-prompt))

2.5 延续管理器

由于 Racket servlets 在服务器上存储它们的延续,所以它们在服务器上占用了内存。 此外,垃圾收集不能用来释放这些内存,因为系统外有根:用户的浏览器、书签、大脑和笔记本。 因此,如果要控制内存的使用,必须使用一些其他策略。 这个功能是可以通过 manager 接口插入的。

2.5.1 一般

 (require web-server/managers/manager)
  package: web-server-lib
该模块定义了管理器的接口。 管理器的用户和实现者都需要它。

struct

(struct manager (create-instance
    adjust-timeout!
    clear-continuations!
    continuation-store!
    continuation-lookup
    continuation-peek)
    #:extra-constructor-name make-manager)
  create-instance : ((-> void) . -> . number?)
  adjust-timeout! : (number? number? . -> . void)
  clear-continuations! : (number? . -> . void)
  continuation-store! : 
(number? any/c
         (or/c false/c
               (request? . -> . can-be-response?))
         . -> . (list/c number? number?))
  continuation-lookup : (number? number? number? . -> . any/c)
  continuation-peek : (number? number? number? . -> . any/c)
create-instance 被调用来初始化一个实例,以保持一个 servlet 会话的连续性。 当实例过期时,它被传递一个函数来调用。它运行该实例的 ID。

adjust-timeout! 是一个即将被淘汰的函数,它接收一个实例id和一个数字。 它是基于超时的管理器所特有的,将被删除。
clear-continuations! 终止一个实例的所有延续。
continuation-store! 给定一个实例 id、一个 continuation 值, 以及一个函数,以便在查询 continuation 并已过期的情况下将其纳入抛出的异常中。 返回的两个数字是一个 continuation-id 和一个 nonce。
continuation-lookup 可以找到与给定的 instance-id、 continuation-id 和 nonce triple 相关的 continuation 值。
continuation-peek 与 continuation-lookup 相同, 但它的使用不能影响对所访问的实例或延续的资源管理策略决定。 它旨在被调试器和基准所使用。

struct

(struct exn:fail:servlet-manager:no-instance exn:fail
  (expiration-handler)
    #:extra-constructor-name
    make-exn:fail:servlet-manager:no-instance)
  expiration-handler : 
(or/c false/c
      (request? . -> . can-be-response?))
当查询到不存在的实例时,管理器应该抛出这个异常。

struct

(struct exn:fail:servlet-manager:no-continuation exn:fail
  (expiration-handler)
    #:extra-constructor-name
    make-exn:fail:servlet-manager:no-continuation)
  expiration-handler : 
(or/c false/c
      (request? . -> . can-be-response?))
当查询到一个不存在的延续时,管理器应该抛出这个异常。

2.5.2 没有延续性

 (require web-server/managers/none) package: web-server-lib
本模块定义了一个管理器构造函数:

procedure

(create-none-manager instance-expiration-handler) → manager?

  instance-expiration-handler : 
(or/c false/c
      (request? . -> . can-be-response?))
这个管理器实际上并不存储任何延续性或实例数据。 如果你知道你的 servlet 不使用 continuation capturing 函数, 并希望服务器不为每个实例分配元数据结构,你可以使用它。

如果你 确实 使用了延续捕获函数,那么延续就不会被存储。 如果访问了 URL, instance-expiration-handler 会随着请求被调用。
如果你正在考虑使用这个管理器,也要考虑使用网络语言。(参见 无状态 Servlet 。)

2.5.3 超时

 (require web-server/managers/timeouts)
  package: web-server-lib
本模块定义了一个管理器构造函数:

procedure

(create-timeout-manager instance-exp-handler    
  instance-timeout    
  continuation-timeout) → manager?
  instance-exp-handler : 
(or/c false/c
      (request? . -> . can-be-response?))
  instance-timeout : number?
  continuation-timeout : number?
由这个管理器管理的实例将在最后一次被访问后的 instance-timeout 秒内过期。 如果一个过期的实例被查询到,就会抛出 exn:fail:servlet-manager:no-instance 异常, instance-exp-handler 作为过期处理程序。

由该管理器管理的延续将在最后一次被访问后的 continuation-timeout 秒内过期。 如果查询到一个过期的延续,则抛出 exn:fail:servlet-manager:no-continuation 异常, 并将 instance-exp-handler 作为过期处理程序, 如果没有过期处理程序传递给 continuation-store! 。
adjust-timeout! 对应于负责 Servlet 实例的定时器上的 reset-timer! 。
这个管理器被认为是......有问题的......在 Web Server 的大规模部署中。

2.5.4 LRU

 (require web-server/managers/lru) package: web-server-lib
这个模块定义了一个管理器构造函数:

procedure

(create-LRU-manager instance-expiration-handler    
  check-interval    
  collect-interval    
  collect?    
  [#:initial-count initial-count    
  #:inform-p inform-p]) → manager?
  instance-expiration-handler : 
(or/c false/c
      (request? . -> . can-be-response?))
  check-interval : integer?
  collect-interval : integer?
  collect? : (-> boolean?)
  initial-count : integer? = 1
  inform-p : (integer? . -> . void) = (lambda _ (void))
由该管理器管理的实例在实例解锁后,如果没有与之相关的延续,则会过期。 如果查询到一个过期的实例,就会抛出 exn:fail:servlet-manager:no-instance 异常, 并将 instance-exp-handler 作为过期处理程序。

由该管理器管理的延续最初被赋予 initial-count 的 "Life Count" 。 如果查询到一个过期的延续,就会抛出 exn:fail:servlet-manager:no-continuation 异常, 并将 instance-exp-handler 作为过期处理程序, 如果没有过期处理程序传递给 continuation-store! 。
每个 check-interval 秒都会调用 collect? 来确定是否应该运行收集程序。 每一个 collect-interval 秒,收集例程就会运行。
每次收集例程运行时,每个延续的 "生命值" 都被减去 1 。 如果一个续集的计数达到 0, 它就会过期。 如果有任何续集过期,就会调用 inform-p 函数,并显示过期的续集数量。
这个管理器的推荐用法被编为以下函数:

procedure

(make-threshold-LRU-manager instance-expiration-handler 
  memory-threshold) 
 → manager?
  instance-expiration-handler : 
(or/c false/c
      (request? . -> . can-be-response?))
  memory-threshold : number?
这将创建一个具有以下行为的 LRU 管理器: 内存限制被设置为 memory-threshold 字节。 继续以 24 个生命值开始。 生命点以每 10 分钟一个的速度被扣除,或者在超过内存限制时以每 5 秒一个的速度被扣除。 因此,一个延续的最大生命时间是 4 小时,最小是 2 分钟。

如果服务器上的负载激增–如内存使用量所示–服务器将迅速过期,直到内存恢复到可控状态。 如果负载保持在较低水平,它仍然会有效地过掉旧的延续。
