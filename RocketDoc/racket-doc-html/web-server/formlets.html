6 Formlets: 功能性表单抽象

 (require web-server/formlets) package: web-server-lib

Web Server 提供了一种叫做 formlet 的 Web 表单抽象。

Formlet 起源于 Links 研究小组在其论文 The Essence of Form Abstraction 中的工作。

6.1 Formlet 的基本用法

假设我们想创建一个在 HTML 表单中输入日期的抽象。下面这个 formlet 就体现了这个想法:

(define date-formlet
  (formlet
   (div "Month:" ,{input-int . => . month}
        "Day:" ,{input-int . => . day})
   (list month day)))

formlet 语法的第一部分是一个 X-expression 的模板,是 formlet 的渲染。 它可以包含像 ,(=> formlet name) 这样的元素, 其中 formlet 是一个 formlet 表达式, name 是 formlet 语法的第二部分中绑定的标识符。

这个 formlet 被显示(用 formlet-display) 为下面的 X-expression 森林(列表):

(list
 '(div "Month:" (input ([name "input_0"]))
       "Day:" (input ([name "input_1"]))))

date-formlet 不仅捕获了表单的渲染,而且还捕获了请求处理逻辑。 如果我们用 formlet-process 向它发送一个 HTTP 请求,并将 "input_0" 绑定为 "10" ， "input_1" 绑定为 "3", 那么它就会返回:

(list 10 3)

这是 formlet 语法的第二部分,其中 month 被替换为 "input_0" 所代表的整数, day 被替换为 "input_1" 所代表的整数。

formlet 的真正威力在于它们可以被嵌入到彼此之间。 例如,假设我们想结合两个日期表单来捕获一个旅行行程。下面的 formlet 可以完成这个工作:

(define travel-formlet
  (formlet
   (div
    "Name:" ,{input-string . => . name}
    (div
     "Arrive:" ,{date-formlet . => . arrive}
     "Depart:" ,{date-formlet . => . depart})
   (list name arrive depart))))

(注意 date-formlet 被嵌入了两次。)这被渲染成:

(list
 '(div
   "Name:"
   (input ([name "input_0"]))
   (div
    "Arrive:"
    (div "Month:" (input ([name "input_1"]))
         "Day:" (input ([name "input_2"])))
    "Depart:"
    (div "Month:" (input ([name "input_3"]))
         "Day:" (input ([name "input_4"]))))))

注意到 formlet-display 已经为每个输入元素自动生成了唯一的名字。 当我们把这些名字的绑定传递给 formlet-process 时,会返回以下列表:

(list "Jay"
      (list 10 3)
      (list 10 6))

在所有这些例子中,我们使用了 input-int 和 input-string formlets 。 任何具有 formlet 契约的值都可以在这些位置使用。 例如, (to-string (required (text-input))) 也可以被使用。 手册的其余部分给出了 formlet 的用法、扩展和现有的 formlet 组合器的细节。

6.2 静态句法速记

 (require web-server/formlets/syntax)
  package: web-server-lib
大多数用户会希望使用语法速记来创建 formlet 。

syntax

(formlet rendering-xexpr yields-expr)

用指定的 rendering-xexpr 构造一个 formlet ,处理结果是对 yields-expr 表达式的评估。 rendering-xexpr 形式是一个准引号的语法 X-expression, 有三个特殊的注意事项:

,{=> formlet-expr name} 嵌入 formlet-expr 给出的 formlet ; 处理这个 name 的结果可以在 yields-expr 中作为 name 使用。
,{=> formlet-expr (values name ...)} 嵌入 formlet-expr 给定的 formlet ; 处理这个 formlet 的结果可在 yields-expr 中以 name ... 获得。
(#%# xexpr ...) 渲染了一个 X-expression forest 。

这些表单 不得 出现在 unquote 或 unquote-splicing 里面嵌套。 例如,这是非法的:
(formlet (div ,@(for/list ([i (in-range 10)])
                  `(p ,((text-input) . => . name))))
         name)

syntax

#%#

只允许出现在 formlet 和 formlet* 里面。

6.3 动态句法速记

 (require web-server/formlets/dyn-syntax)
  package: web-server-lib
formlet 的语法对某些应用来说限制性太大, 因为它迫使 rendering 在 syntactically 成为一个 X-expression 。 你可能会发现你想使用一个更 "动态" 的缩写。

syntax

(formlet* rendering-expr yields-expr)

构建一个 formlet ,其中 rendering-expr 被评估(有注意事项)以构建渲染, 处理结果是对 yields-expr 表达式的评估。 rendering-expr 应该评估为一个 "X-expression" , 它可以嵌入以下形式的结果,这些结果只在 formlet* 中具有意义:

{=>* formlet-expr name} 嵌入 formlet-expr 给出的 formlet ; 处理这个formlet 的结果可以在 yields-expr 中作为 name 使用。
{=>* formlet-expr (values name ...)} 嵌入 formlet-expr 给出的 formlet ; 处理这个 formlet 的结果在 yields-expr 中以 name ... 出现。
(#%# xexpr-expr ...) 渲染一个 X-expression 森林。
这些表单中的每一个都评估为一个不透明的值, rendering-expr 不能以任何方式对其进行操作, 但是如果它作为 "X-expression" 的一部分被返回给 formlet* ,它将被渲染,formlet 的处理阶段将被执行,等等。

因为这些表单 可以 出现在 rendering-expr 的任何地方,它们可以被重复。 因此,formlet 可能会被渲染(并被处理)多次。 因此,在 yields-expr 中,formlet 的结果名称被绑定到结果列表上, 而不是像 formlet 中那样的单一结果。 结果列表是根据 rendering-expr 的结果中的 formlet 的顺序来排序的。例如,在
(formlet* `(div ,@(for/list ([i (in-range 1 10)])
                    `(p ,(number->string i)
                        ,((text-input) . =>* . name))))
          name)
name 被绑定到一个字符串列表,而不是一个单一的字符串, 其中第一个元素是在网页上输入的字符串 1 旁边的字符串。

在这个例子中,很明显,这就是所期望的行为。 然而,有时 formlet 的结果值可能会令人吃惊。例如,在
(formlet* `(div (p ,((text-input) . =>* . name)))
          name)
name 被绑定到一个字符串列表, 因为 formlet* 不能从语法上确定其结果被绑定到 name 的 formlet 是否被多次使用。

syntax

=>*

只允许在 formlet* 内部使用。

6.4 功能性使用方法

 (require web-server/formlets/lib) package: web-server-lib
语法速记法用下面的库来缩写 formlet 的构造。 这些组合器可以直接用于构造低级别的 formlet,比如那些新的 INPUT 元素类型。 关于使用这些组合器的低级 formlet 的例子,请参考 预定义的 Formlets 。

value

xexpr-forest/c : contract?

相当于 (listof xexpr/c)

procedure

(formlet/c content ...) → contract?

  content : contract?
相当于 (-> integer? (values xexpr-forest/c (-> (listof binding?) (values (coerce-contract 'formlet/c content) ...)) integer?)).

一个 formlet 的内部表示是一个从初始输入数字到 X-expression 森林渲染的函数, 一个处理函数,以及下一个允许的输入数字。
(实际上, formlet/c 是一个宏,它避免了在编译时已知处理函数的范围合同数时使用 dynamic->* 。)

Changed in version 1.3 of package web-server-lib: 修正了对多个返回值的支持。

value

formlet*/c : contract?

类似于 formlet/c 创建的合约,但使用 any 来避免检查处理函数的结果(甚至是指定结果的数量)。

Changed in version 1.3 of package web-server-lib: 修正了对多个返回值的支持。

procedure

(pure value) → (formlet/c any/c)

  value : any/c
构建一个没有渲染的 formlet ,并且总是在处理阶段返回 value 。

procedure

(cross f g) → (formlet/c any/c ...)

  f : (formlet/c procedure?)
  g : (formlet/c any/c ...)
构造一个渲染等于 formlet f 和 g 的渲染的连接的 formlet ; 一个将 g 的处理结果应用于 f 的处理结果的处理阶段。

procedure

(cross* f g ...) → (formlet/c any/c)

  f : (formlet/c (() () #:rest (listof any/c) . ->* . any/c))
  g : (formlet/c any/c)
相当于将 cross 提升到许多参数。

procedure

(xml-forest r) → (formlet/c procedure?)

  r : xexpr-forest/c
构造一个 formlet ,用渲染 r 和身份程序作为处理步骤。

procedure

(xml r) → (formlet/c procedure?)

  r : xexpr/c
等同于 (xml-forest (list r)) 。

procedure

(text r) → (formlet/c procedure?)

  r : string?
等同于 (xml r) 。

procedure

(tag-xexpr tag attrs inner) → (formlet/c any/c)

  tag : symbol?
  attrs : (listof (list/c symbol? string?))
  inner : (formlet/c any/c)
构造一个具有渲染 (list (list* tag attrs inner-rendering)) 的 formlet , 其中 inner-rendering 是 inner 的渲染,处理阶段与 inner 相同。

procedure

(formlet-display f) → xexpr-forest/c

  f : (formlet/c any/c)
渲染 f 。

procedure

(formlet-process f r) → 
any/c ...
  f : (formlet/c any/c ...)
  r : request?
在 r 中的绑定物上运行 f 的处理阶段。

6.7 Formlet 和无状态 Servlet

 (require web-server/formlets/stateless)
 (require web-server/formlets/unsafe)
  package: web-server-lib

在使用 formlet 和无状态的 #lang web-server 时,有一些额外的考虑。

首先,如果在你的 servlet 中捕获的连续体在不可序列化的数据结构上关闭,那么它们就不能被序列化。 有一些普遍适用的方法可以避免让数据结构成为闭合的一部分: 例如,如果你将所有的 formlet 定义为模块级变量,它们将永远不会成为闭合的一部分,也不需要被序列化。 然而,有时动态地创建 formlet 是很有用的。 为了支持这一点,上面记录的所有组合器产生的 formlet 都是可序列化的(只要它们只包含可序列化的值)。 这并不保证第三方的 formlet 是真的。

formlet 和序列化的一个潜在隐患是 pure 。 注意 (serialize (pure +)) 会 fail, 因为 + 是不可序列化的。 为了避免这种情况,你可以写 (pure (λ args (apply + args))) (在 #lang web-server 中,匿名程序是可序列化的, 或者使用 web-server/lang/serial-lambda)。

其次,无状态的 #lang web-server 是基于与有状态的 servlets 不同的 web 交互基元, 所以 web-server/formlets 的 send/formlet 版本将无法工作。 相反,库 web-server/formlets/stateless 提供了与 web-server/formlets 相同的 API, 但有一个用于无状态 Servlets 的 send/formlet 版本。 (使用 web-server/formlets/stateless 还可以提供 web-server/formlets/lib 的所有绑定, 而 web-server/formlets 只提供其中的一部分)。 另外,你可以直接使用低级的 formlet-process 和 formlet-display 程序。

另一个问题涉及到在 formlet 的处理阶段捕获连续。 回想一下, #lang web-server 中的可序列化的连续体只能从已转换的上下文中捕获。 契约系统没有被转换,所以这个库的契约阻止在 formlet 的处理阶段捕获连续。 在大多数情况下,最好的解决办法是避免在 formlet 的处理阶段使用捕获连续的操作。 相反,让处理阶段返回一个值,并根据这个值在 formlet 之外的代码中与用户交互。 另外,你也可以使用普遍适用的方法从未转换的上下文中捕获延续,例如 web-server/lang/native 。 然而,如果这些方法都不能令人满意, 库 web-server/formlets/unsafe 提供了与 web-server/formlets/stateless 相同的API, 但 没有强制执行契约。 顾名思义,使用 web-server/formlets/unsafe 可能会产生难以捉摸的错误信息和其他没有契约的编程所带来的令人不快的影响:我们已经警告过你了。

Changed in version 1.3 of package web-server-lib: 添加了 web-server/formlets/stateless 和 web-server/formlets/unsafe , 并改变了 web-server/formlets 的组合器,以产生可序列化的 formlets。
