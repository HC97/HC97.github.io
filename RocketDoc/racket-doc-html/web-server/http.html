4 HTTP: 超文本传输协议

 (require web-server/http) package: web-server-lib

Web Server 实现了许多由该模块提供的 HTTP 库。

4.1 请求

 (require web-server/http/request-structs)
  package: web-server-lib

struct

(struct header (field value)
    #:extra-constructor-name make-header)
  field : bytes?
  value : bytes?
代表一个 field 为 value 的头文件。

procedure

(headers-assq id heads) → (or/c false/c header?)

  id : bytes?
  heads : (listof header?)
返回头的字段等于 heads 或 #f 中的 id 。

procedure

(headers-assq* id heads) → (or/c false/c header?)

  id : bytes?
  heads : (listof header?)
返回头,其字段的大小写与 heads 或 #f 的 id 。

你几乎 总是 想使用这个,而不是 headers-assq ,因为网络浏览器可能会发送任意大小写的头信息。

struct

(struct binding (id)
    #:extra-constructor-name make-binding)
  id : bytes?
代表 id 的一个绑定。

struct

(struct binding:form binding (value)
    #:extra-constructor-name make-binding:form)
  value : bytes?
代表 id 与 value 的绑定形式。

struct

(struct binding:file binding (filename headers content)
    #:extra-constructor-name make-binding:file)
  filename : bytes?
  headers : (listof header?)
  content : bytes?
文件 filename 与 id id 和内容 content 的绑定, 其中 headers 是文件所在的 MIME 信封中的附加头信息。 例如, #"Content-Type" 头可能被一些浏览器包含。

也请参见 binding:file/port-in, 这是一个替代文件上传的接口,可以明显提高内存效率。

Changed in version 1.6 of package web-server-lib: 扩展到支持基于端口的表示:见 binding:file/port-in 。

procedure

(binding:file/port-in binding) → input-port?

  binding : binding:file/port?

procedure

(binding:file/port? v) → boolean?

  v : any/c

procedure

(make-binding:file/port id    
  filename    
  headers    
  content-in) → binding:file/port?
  id : bytes?
  filename : bytes?
  headers : (listof header?)
  content-in : input-port?
网络服务器可以避免将上传的文件存储在内存中。 特别是,一个 safety limits 值可以指示这个库在文件大于某个阈值时将其卸载到磁盘。 即使是不写入磁盘的文件上传,Web 服务器最初也会将内容放在一个输入端口中, 而不是一个字节串,这样在内容被读取后就不需要保留存储。

基于端口的接口是暴露给程序员的,它可以比基于字节串的接口明显地提高内存效率。 然而,基于端口的接口是有状态的:使用它的程序员要负责管理输入端口的状态。 请继续阅读细节。
为了保持兼容性,基于端口的接口使用 binding:file 的一个私有的、不透明的子类型。 这个扩展类型的实例被谓词 binding:file/port? 识别, 并使用 make-binding:file/port 创建, 这就像 make-binding:file, 但把文件内容作为一个输入端口而不是一个字节字符串。 只有被 binding:file/port? 识别的 binding:file 实例才支持 binding:file/port-in 。 web 服务器在读取 request 结构时使用 make-binding:file/port , 这是大多数程序遇到 binding:file 实例的主要方式: 然而,反序列化的实例(见下文)和使用 make-binding:file 手动构造的实例不支持基于端口的 API。

了解 binding:file-content 如何与基于端口的实例一起工作是很重要的。 第一次 binding:file-content 在基于端口的实例 v 上被调用, 它消耗端口的剩余内容,就像 (port->bytes (binding:file/port-in v)) 一样, 为以后调用 binding:file-content 记忆结果,并关闭输入端口。 这种行为意味着:

  • 一个给定的输入字节既可以存储在 binding:file-content 字段中, 也可以直接从输入端口读取,但决不是两者都有;而且

  • 如果在 binding:file-content 第一次被调用时,输入端口已经被直接关闭, binding:file-content 将引发一个异常。

间接访问 binding:file-content 字段,比如通过使用 match, 与直接调用 binding:file-content 的行为相同。 特别是,在一个 binding:file 实例上调用 serialize 时, 隐含地使用了 binding:file-content, 而反序列化的实例则有效地使用 make-binding:file 来构造。

Added in version 1.6 of package web-server-lib.

procedure

(bindings-assq id binds) → (or/c false/c binding?)

  id : bytes?
  binds : (listof binding?)
返回 binds 或 #f 中 id 等于 id 的绑定。

procedure

(bindings-assq-all id binds) → (listof binding?)

  id : bytes?
  binds : (listof binding?)
与 bindings-assq 类似,但返回与 id 匹配的所有绑定的列表。

struct

(struct request (method
    uri
    headers/raw
    bindings/raw-promise
    post-data/raw
    host-ip
    host-port
    client-ip)
    #:extra-constructor-name make-request)
  method : bytes?
  uri : url?
  headers/raw : (listof header?)
  bindings/raw-promise : (promise/c (listof binding?))
  post-data/raw : (or/c #f bytes?)
  host-ip : string?
  host-port : number?
  client-ip : string?
一个 HTTP method 请求到 uri , 从 client-ip 到 host-ip:host-port 的服务器, 带有 headers/raw 头部, bindings/raw GET 和 POST 查询以及 post-data/raw POST 数据。

你 不太可能需要构建 一个请求结构。

Changed in version 1.6 of package web-server-lib: 固定回答 #f 为 serializable?, 因为所有 request 实例都包含不可序列化的部分。

procedure

(request-bindings/raw r) → (listof binding?)

  r : request?
强使 (request-bindings/raw-promise r) 。

这里是一个典型的例子,你会在许多应用程序中发现:
(define (get-number req)
  (match
    (bindings-assq
      #"number"
      (request-bindings/raw req))
    [(? binding:form? b)
     (string->number
      (bytes->string/utf-8
       (binding:form-value b)))]
    [_
     (get-number (request-number))]))

4.2 绑定

 (require web-server/http/bindings) package: web-server-lib
这些函数虽然方便,但可能会给你的应用程序带来微妙的错误。 例如:它们不区分大小写,可能会带来错误;如果提交的数据不是 UTF-8 格式,那么转换为字符串就会失败; 如果攻击者提交的表单字段是一个文件,而它不是,那么 request-bindings 将持有一个 bytes? 对象, 你的程序会出错;对于文件上传,你会失去文件名。 因此,我们建议不要使用它们,但提供它们是为了与旧代码兼容。

procedure

(request-bindings req)

 → 
(listof (or/c (cons/c symbol? string?)
              (cons/c symbol? bytes?)))
  req : request?
翻译 req 的 request-bindings/raw , 将 bytes? 解释为 string? , 除了 binding:file 绑定的情况,其内容以字节形式返回。 Id 随后被翻译成小写符号。

procedure

(request-headers req) → (listof (cons/c symbol? string?))

  req : request?
通过将 bytes? 解释为 string? 来翻译 req 的 request-headers/raw 。 然后 Id 被翻译成小写的符号。

procedure

(extract-binding/single id binds) → string?

  id : symbol?
  binds : (listof (cons/c symbol? string?))
如果正好有一个绑定,返回与 a-list binds 中 id 相关的单个绑定。 否则会引发 exn:fail .

procedure

(extract-bindings id binds) → (listof string?)

  id : symbol?
  binds : (listof (cons/c symbol? string?))
返回 a-list binds 中 id 的所有绑定的列表。

procedure

(exists-binding? id binds) → boolean?

  id : symbol?
  binds : (listof (cons/c symbol? string))
如果 binds 包含了 id 的绑定,则返回 #t 。 否则,返回 #f .

这里是一个典型的例子,你会在许多应用程序中发现它:
(define (get-number req)
  (string->number
   (extract-binding/single
    'number
    (request-bindings req))))

4.3 回应

 (require web-server/http/response-structs)
  package: web-server-lib

struct

(struct response (code message seconds mime headers output))

  code : response-code/c
  message : bytes?
  seconds : real?
  mime : (or/c #f bytes?)
  headers : (listof header?)
  output : (output-port? . -> . any)

value

response-code/c : flat-contract? = (integer-in 100 999)

一个 HTTP 响应,其中 output 通过写到输出端口来产生正文。 code 是响应代码, message 是消息, seconds 是生成时间, mime 是文件的 MIME 类型, headers 是头文件。

如果 headers 不包括 Date,Last-Modified 或 Server 头, 那么服务器会自动添加它们,其中 Date 是基于 current-seconds, Last-Modified 是基于 seconds, Server 是 Racket 。
如果 headers 信息不包括 Content-Type , 并且 mime 不是 #f, 那么 mime 将作为 Content-Type 头信息添加。
如果服务器需要确保连接将被关闭,它总是会替换你的 Connection 头。 (通常是在 HTTP/1.0 客户端。)
服务器将总是把它产生的头信息放在 response 结构中的头信息之前, 并保证所提供的头信息按照给定的顺序出现在输出中。 (如果同一个头的多次出现在客户端有不同的解释,例如 Set-Cookie ,这就很重要)。

例子:
(response
 301 #"OK"
 (current-seconds) TEXT/HTML-MIME-TYPE
 empty
 (λ (op) (write-bytes #"<html><body>Hello, World!</body></html>" op)))
(response
 301 #"Moved Permanently"
 (current-seconds) TEXT/HTML-MIME-TYPE
 (list (make-header #"Location"
                    #"http://racket-lang.org/download"))
 (λ (op) (write-bytes #"Moved" op)))
(response
 304 #"Not Modified"
 (current-seconds) #f
 (list (make-header #"Location"
                    #"http://racket-lang.org/download"))
 void)

Changed in version 1.2 of package web-server-lib: 削弱了 output 的契约,允许 any 结果(而不是要求 void?)。
Changed in version 1.3: 增加了 response-code/c , 并使 code 和 seconds 的契约更强(而不是接受 number?)。

4.4 放置 Cookies

 (require web-server/http/cookie) package: web-server-lib
本模块提供创建 cookie 的函数和设置 cookie 的响应。

procedure

(make-cookie name    
  value    
  [#:comment comment    
  #:domain domain    
  #:max-age max-age    
  #:path path    
  #:expires expires    
  #:secure? secure?    
  #:http-only? http-only?    
  #:extension extension]) → cookie?
  name : cookie-name?
  value : cookie-value?
  comment : any/c = #f
  domain : (or/c domain-value? #f) = #f
  max-age : (or/c (and/c integer? positive?) #f) = #f
  path : (or/c path/extension-value? #f) = #f
  expires : (or/c date? string? #f) = #f
  secure? : any/c = #f
  http-only? : any/c = #f
  extension : (or/c path/extension-value? #f) = #f
构建一个具有适当字段的 cookie 。

这是一个围绕 net/cookies/server 的 make-cookie 的包装器,用于向后兼容。 comment 参数被忽略。 如果 expires 是一个字符串,它应该符合 RFC 7231, Section 7.1.1.2, 在这种情况下,它将被转换为一个 date? 。 如果转换失败,将引发 exn:fail:contract? 。

Changed in version 1.3 of package web-server-lib: 通过 net/cookies/server 增加了对 RFC 6265 的支持。 对字符串值的参数执行更强的契约。 允许 expires 是一个 date? , 允许 secure 是 any/c (而不是 boolean?) 。 禁止使用 0 作为 max-age 。 支持 http-only? 和 extension 参数。忽略 comment 。

procedure

(cookie->header c) → header?

  c : cookie?
构建一个设置 cookie 的头。

例子:
(define time-cookie
  (make-cookie "time" (number->string (current-seconds))))
(define id-cookie
  (make-cookie "id" "joseph" #:secure? #t))
 
(redirect-to
 "http://localhost/logged-in"
 see-other
 #:headers
 (map cookie->header
      (list time-cookie id-cookie)))
 
(send/suspend
 (lambda (k-url)
   (response/xexpr
    #:cookies (list time-cookie id-cookie)
    `(html (head (title "Cookie Example"))
           (body (h1 "You're cookie'd!"))))))

4.5 认证的 Cookies

 (require web-server/http/id-cookie)
  package: web-server-lib
Cookie 对于存储用户浏览器的信息很有用,特别是对于存储认证、会话等的识别信息很有用。 然而,在使用 cookie 作为认证器时存在固有的困难,因为 cookie 数据完全由用户控制,因此不能被信任。
该模块提供了创建和验证具有内在时间戳的认证 cookies 的功能。 它基于 MIT Cookie Eaters 提出的算法: 如果你在 authored-seconds 时间存储 data , 那么用户将收到 digest&authored-seconds&data, 其中 digest 是 authored-seconds 和 data 的 HMAC-SHA1 摘要, 使用一个任意的秘密密钥。 当你收到一个 cookie 时,它将重新验证这个 digest, 并检查 cookie 的 authored-seconds 是否在一个超时周期之后,然后才将 cookie 数据返回给程序。
该接口将秘密密钥表示为一个字节字符串。 为了安全起见,这应该使用加密质量的随机性来创建。 一个方便的纯粹基于 Racket 的选项是 make-secret-salt/file, 它是用 crypto-random-bytes 实现的。 你也可以使用 OpenSSL 或 /dev/random 之类的东西来生成随机字节: this FAQ 列出了几个选项。

procedure

(make-id-cookie name    
  value    
  #:key secret-salt    
  [#:path path    
  #:expires expires    
  #:max-age max-age    
  #:domain domain    
  #:secure? secure?    
  #:http-only? http-only?    
  #:extension extension]) → cookie?
  name : cookie-name?
  value : cookie-value?
  secret-salt : bytes?
  path : (or/c path/extension-value? #f) = #f

4.7 重定向

 (require web-server/http/redirect) package: web-server-lib

procedure

(redirect-to uri [status #:headers headers]) → response?

  uri : non-empty-string?
  status : redirection-status? = temporarily
  headers : (listof header?) = '()

procedure

(redirection-status? v) → boolean?

  v : any/c

value

temporarily : redirection-status?

value

temporarily/same-method : redirection-status?

value

see-other : redirection-status?

value

permanently : redirection-status?

函数 redirect-to 生成一个 HTTP 响应,将浏览器重定向到 uri, 同时在响应中包括 headers 。 status 参数是一个 redirection status 值,它决定了要使用的 HTTP 重定向的具体类型。

默认的 redirection status, temporarily 是为了向后兼容而保留的: 新的代码通常应该使用 temporarily/same-method 或 see-other 来代替。 temporarily redirection status 对应于 302 Found 。 不幸的是,对于 GET 和(实际上,除了一些非常老的浏览器之外) POST 以外的方法, 浏览器并没有一致地实现这个状态。
temporarily/same-method redirection status 使用 307 Temporary Redirect 。 这将浏览器重定向到使用与原始请求相同的 HTTP 方法的 uri 。
see-other redirection status 对应的是 303 See Other 。 它最常被用来实现 Post-Redirect-Get 模式: 作为对使用 POST 或其他有副作用的 HTTP 方法的请求的响应, 它使浏览器对 uri 执行 GET 或 HEAD 请求, 从而给出对原始 POST 请求的响应。 这可以防止 Back 和 Refresh 按钮产生重复的效果,例如进行购买或向数据库添加物品。 Web服务器提供 redirect/get ,以增加 Post-Redirect-Get 的便利性。

permanently redirection status 使用 HTTP 状态 301 Moved Permanently 。 它和 temporarily 一样,除了顾名思义,它标志着移动是永久性的, 例如,搜索引擎应该使用 uri 而不是原始请求的 URI。 不幸的是, permanently 也和 temporarily 一样, 浏览器对 GET 和 HEAD 以外的方法的实现不一致: 特别是 RFC 7231 允许, 由于历史原因,用户代理 可以 将后续请求的请求方法从 POST 改为 GET 。 当必须确保对 uri 的请求使用相同的方法时,有一些可能的替代方法:

  • RFC 7231 建议使用 307 Temporary Redirect, 即 temporarily/same-method 。 这有一个缺点,即搜索引擎和其他人不会更新对旧 URI 的引用。

  • RFC 7538 规定了一个新的 HTTP 状态, 308 Permanent Redirect, 它禁止改变请求方法,类似于 307 Temporary Redirect 。 然而,RFC 还强调了这种状态的一些重要的 deployment considerations 。 特别是,旧的浏览器–包括截至本文写作时仍在 relatively common use 的一些浏览器–不理解这种状态,会退回到 300 Multiple Choices 的语义,这通常是不可取的。

  • 应用程序可以注意到原始请求的方法,并对 GET 和 HEAD 请求使用 permanently , 或对其他方法使用其他替代方法之一。

例子: (redirect-to "http://www.add-three-numbers.com" permanently)

Changed in version 1.3 of package web-server-lib: 添加了 temporarily/same-method 。
