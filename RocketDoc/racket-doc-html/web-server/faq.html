10 故障排除和提示

10.1 为什么我的 Servlet 在更新 Racket 后出现了 can-be-response? 违约的情况?

在 5.0.2 之后,Web 服务器有一个向后不兼容的变化,使得 X 表达式和字节列表不能直接从 servlet 中返回。 请阅读 "PLTHOME/collects/web-server/compat/0/README" 来了解如何向前移植你的 servlets。 不要担心。这很容易。

10.2 为什么当我改变磁盘上的文件时,我的模板在服务器上没有更新?

模板被编译到你的应用程序中,所以当你改变它们时, 在文件系统中的改变和已经在运行的 Web 服务器进程中加载的编译字节码之间没有任何联系。 更多的讨论,请看 为什么当我改变磁盘上的文件时,我的有状态 Servlet 在服务器上没有更新?.

10.3 为什么模板被编译成了程序?

因为模板可以包括任意的 Racket 代码、宏等,并引用任意的标识符， include-template 实际上只是一个模糊的 require 。

10.4 为什么当我改变磁盘上的文件时,我的有状态 Servlet 在服务器上没有更新?

如果你使用的是 serve/servlet, 它启动了一个 Web 服务器,直接引用了一个与磁盘上某个文件没有联系的闭包。

如果你使用的是命令行工具,或配置文件,那么默认情况下, 服务器使用 make-cached-url->servlet 来从磁盘上加载 servlet 。 当它加载这些小程序时,它们会被缓存起来,并且在未来的请求中不会提到磁盘。 这确保了每个 servlet 都有一个命名空间,这样不同的实例可以共享资源,如数据库连接,并通过存储进行通信。 服务器的默认配置(指加载配置文件时使用的调度器序列)为 localhost 提供了一个特殊的 URL, 可以重置缓存: "/conf/refresh-servlets" 。

如果你想让服务器重新加载你修改过的 servlet 代码,那么 GET 这个 URL,服务器会在下一次请求中重新加载 servlet 。 然而,你可能会对下一次请求中发生的事情感到惊讶。 更多讨论,请参见 在刷新我的有状态 Servlet 后,旧的捕获的延续没有变化,或者旧的全局效果没有了。为什么? 。

10.5 在刷新我的有状态 Servlet 后,旧的捕获的延续没有变化,或者旧的全局效果没有了。为什么?

你的 Servlet 的每次加载都是在一个新的命名空间中进行的。 当你刷新时,一个没有旧效果的新命名空间被创建。 旧的捕获的延续指的是原来的命名空间,永远不会更新。 一般来说,不可能将一个延续从一个命名空间移植到另一个命名空间,因为代码可能任意不同。

10.6 无状态 Servlet 与有状态 Servlet 在刷新方面有什么不同?

延续是用哈希值进行序列化的,它可以确保任何源代码的修改都会使所有旧的延续不兼容,原因与本地延续自然是一样的。

然而,这个哈希值只能防止单个源文件的变化。因此,如果你将你的应用程序模块化, 那么只有那些引用已修改的源文件的连续体才会不兼容。 例如,如果你把所有的模板放在一个单一的模块中,那么它就可以改变而不使旧的连续体失效。

10.7 在 Web 服务器的安全方面有什么特别的考虑?

最大的问题是,对连续体的天真使用会让连续体颠覆认证机制。 通常情况下,执行一个续集所需的是它的 URL。因此,URL 必须像续集中的信息一样受到保护。

考虑一下,如果你从一个私有的延续性 URL 链接到一个公共网站:新的 HTTP 请求中的 Referrer 字段将包含私有的 URL 。 此外,如果你的 HTTP 流量是透明的,那么这些 URL 就很容易被盗用。

对此的一个解决方案是使用一个特殊的 cookie 作为验证器。 这样一来,如果一个 URL 逃脱了,它将无法被使用,除非 cookie 存在。 关于如何做好这一点的建议,请参阅麻省理工学院 Cookie Eaters 的 Dos and Don’ts of Client Authentication on the Web 。

注意:URL 可以通过这种方式共享,这可能被认为是一个很好的功能,因为授权很容易通过 URL 内置于一个应用程序。

10.8 我的浏览器显示我的页面很奇怪:我的 CSS 被忽略了,有些部分不见了,等等。

大多数 Web Server 开发人员使用 X-expression 来表示他们页面的 HTML。 然而, X-expression 只代表 XML,而 HTML 不完全是 XML 。 这是一个常见的问题来源。

例如,XML 允许在每个标签上使用 "空标签速记",例如 <img src='...' />, 而 HTML 偶尔需要一个结束标签, 例如 TEXTAREA 。 同样,XML 允许在每个标签上有一个结束标签,例如 <img src='...'></img>, 而 HTML 偶尔会禁止一个结束标签, 例如 IMG 。 (当然,浏览器不一定按照指定的方式实现 HTML 解析,并且可能或多或少地对类似 XML 的 HTML 宽容, 因此您的测试浏览器可能不会将这些表单视为有问题。)

由于 Web Server 使用 xml 来格式化 X-expression , 它继承了 xml 的一般默认渲染行为,特别是它对 "空标签速记" 的使用。 xml 的默认是总是使用 html-empty-tags 中的标签速记,而不是其他。 这个列表应该包含 W3C 的批准列表。 你可以用 empty-tag-shorthand 参数来改变它。

你也可以改变你的 X-expression ,以便强制使用结束标签。 例如, '(textarea [(name "text")]) 显示为 <textarea name="text" />, 而 '(textarea [(name "text")] "") 显示为 <textarea name="text"></textarea>, 因为 X-expression 中的字符串内容。 在这种情况下,不管 empty-tag-shorthand 的值是多少,结束标签将始终存在。 不可能强迫另一种可能性;也就是说,永远不包括结束标记。

你可能认为 Web Server 可以做得更好,宣传它所提供的内容在默认情况下更像 XML。 不幸的是,浏览器对这种 advertisement 的支持是 lacking 。 你可以使用 response/xexpr 来轻松定制你的应用程序的 MIME 类型和响应头。

最后,你可能会发现 Web 浏览器检查器,如 Safari 检查器、Firebug 和 Google Chrome 的错误控制台,是识别违规标签的有用工具。

10.9 我如何 "动态" 地使用模板?

一个常见的功能要求是将一个模板动态地包含在另一个模板中。 希望大家能明白, include-template 可以包含在一个模板中,以包含一个 static 子模板。比如说、

(include-template "posts.html")

可以出现在 "blog.html" 模板中。 但你很快就会发现,当 expr 在语法上不是一个路径时, (include-template expr) 会失败,例如:
....
(include-template (if logged-in?
                      "user-info.html"
                      "auth.html"))
....

有什么解决办法呢? 模板系统已经允许你对模板进行参数化处理,因此特定的组件来自于包含范围。 没有理由这些值不能是其他模板的结果。 在前面的例子中,假设包括器是
(define (main-page logged-in?)
  (include-template "site.html"))

我们可以把它改成
(define (main-page logged-in?)
  (define user-content
    (if logged-in?
        (include-template "user-info.html")
        (include-template "auth.html")))
  (include-template "site.html"))

和 "site.html" 改为:
....
user-content
....

这允许你做同样的事情,但更安全、更有效: 更安全是因为没有办法包含没有被程序员命名的模板,更有效是因为所有的模板都和其他的代码一起被编译(和优化)。

如果你坚持要用动态主义,还有 eval 。
