6 Utilities

类型化的 Racket 提供了一些额外的实用函数来促进类型化编程。

procedure

(assert v) → A

  v : (U #f A)
(assert v p?) → B
  v : A
  p? : (A -> Any : B)
验证参数是否满足约束条件。 如果没有提供谓词,则简单地检查该值是否为 #f 。

也请看 cast 形式。

Examples:
> (define: x : (U #f String) (number->string 7))
> x

- : (U False String)

"7"

> (assert x)

- : String

"7"

> (define: y : (U String Symbol) "hello")
> y

- : (U String Symbol)

"hello"

> (assert y string?)

- : String

"hello"

> (assert y boolean?)

Assertion #<procedure:boolean?> failed on "hello"

syntax

(with-asserts ([id maybe-pred] ...) body ...+)

 
maybe-pred = 
  | predicate
用断言来保护主体。如果任何断言失败,程序就会出错。 这些断言的行为类似于 assert 。

procedure

(defined? v) → boolean?

  v : any/c
一个用于确定 v 是 不是 #<undefined> 的谓词。

procedure

(index? v) → boolean?

  v : any/c
一个用于 Index 类型的谓词。

syntax

(typecheck-fail orig-stx maybe-msg maybe-id)

 
maybe-msg = 
  | msg-string
     
maybe-id = 
  | #:covered-id id
明确地产生一个类型错误,带有源位置或 orig-stx 。 如果 msg-string 存在,它必须是一个字面字符串,它被用作错误信息,否则将使用错误信息 "Incomplete case coverage" 。 如果 id 存在并且有 T 类型,那么错误信息中会加上 "missing coverage of T" 。

Examples:
> (define-syntax (cond* stx)
    (syntax-case stx ()
      [(_ x clause ...)
       #`(cond clause ... [else (typecheck-fail #,stx "incomplete coverage"
                                                #:covered-id x)])]))
> (define: (f [x  : (U String Integer)]) : Boolean
    (cond* x
           [(string? x) #t]
           [(exact-nonnegative-integer? x) #f]))

eval:10:0: Type Checker: incomplete coverage; missing

coverage of Negative-Integer

  in: #f

syntax

(assert-typecheck-fail body-expr)

(assert-typecheck-fail body-expr #:result result-expr)
如果 body-expr 不产生类型错误,则明确地产生类型错误。 如果提供了 result-expr ,它将是表达式的求值结果,否则将返回 (void) 。 如果有一个预期的类型,那么在检查 body-expr 时,这个类型会被传播为预期的类型。

Added in version 1.7 of package typed-racket-lib.

6.1 忽略类型信息

在某些情况下,让类型检查器忘记特定表达式的类型信息是有用的。 任何具有 'typed-racket:ignore-type-information 语法属性为真值的形状 (#%expression sub) 的表达式将具有 Any 类型, 并且类型检查器不会学习关于该表达式的任何信息以用于细化其他类型。

Added in version 1.7 of package typed-racket-lib.

表达式 sub 仍然必须进行类型检查,但可以有任何单值类型。

这与 (ann sub Any) 类似,但不同的是类型检查器是否可以用它来精炼其他类型, 并且可以在不依赖 Typed Racket 的情况下使用。

6.2 非类型化实用程序

 (require typed/untyped-utils) package: typed-racket-lib

这些实用程序有助于将类型化的代码与非类型化的代码连接起来, 特别是使用不能转换为契约的类型的类型库,或者在类型化和非类型化上下文中必须以不同方式扩展的输出语法转换程序。

Changed in version 1.14 of package typed-racket-lib: 该模块从 typed-racket-more 转移到 typed-racket-lib 。

syntax

(require/untyped-contract maybe-begin module [name subtype] ...)

 
maybe-begin = 
  | (begin expr ...)
使用这种形式来导入类型化的标识符,其类型不能被转换为契约,但有可以被转换为契约的 subtypes 。

例如,假设我们定义并提供了 typed Racket 函数
(: negate (case-> (-> Index Fixnum)
                  (-> Integer Integer)))
(define (negate x) (- x))
试图在一个未类型化的模块中使用 negate 将引发一个错误,因为不能仅仅通过 arity 来区分这些情况。

如果 negate 的定义模块是 "my-numerics.rkt", 它可以被导入并在未类型化的代码中这样使用:
(require/untyped-contract
 "my-numerics.rkt"
 [negate  (-> Integer Integer)])
类型 (-> Integer Integer) 被转换为用于 negate 的契约。

require/untyped-contract 形式扩展为一个子模块,语言为 typed/racket/base 。 在 subtype 表达式中使用的标识符必须在 Typed Racket 的基础类型环境中 (例如 Integer 和 Listof),或者由 maybe-begin 形式的表达式定义,它被拼接到子模块中。 例如, math/matrix 模块以这种方式导入和再导出 matrix-expt, 它有一个 case-> 类型,用于非类型化使用:
(provide matrix-expt)
 
(require/untyped-contract
 (begin (require "private/matrix/matrix-types.rkt"))
 "private/matrix/matrix-expt.rkt"
 [matrix-expt  ((Matrix Number) Integer -> (Matrix Number))])
(require "private/matrix/matrix-types.rkt") 表达式导入 Matrix 类型。

如果一个标识符的 name 使用 require/untyped-contract 导入, 重新导出,并导入到类型化的代码中,它有其原始类型,而不是 subtype 。 换句话说, subtype 只用于为 name 生成契约,而不是缩小其类型。

由于宏扩展器的限制, require/untyped-contract 目前不能在类型化代码中使用。

syntax

(define-typed/untyped-identifier name typed-name untyped-name)

syntax

(define-typed/untyped-identifier name deep-name untyped-name shallow-name optional-name)

定义一个标识符 name ,根据上下文扩展为以下标识符之一。 当提供两个标识符时, name 在类型化的上下文中扩展为 typed-name , 在未类型化的上下文中扩展为 untyped-name (更确切地说,在其他地方)。 当提供四个标识符时, name 在深类型语境中扩展为 deep-name , 在未类型语境中扩展为 untyped-name ,在浅类型语境中扩展为 shallow-name , 在可选类型语境中扩展为 optional-name 。 深层、浅层和可选语义 解释了这些不同的语境。

假设我们定义并提供一个具有这种类型的Racket函数:

(: my-filter (All (a) (-> (-> Any Any : a) (Listof Any) (Listof a))))

这个类型不能被转换为契约,因为它接受了一个谓词。 更糟的是, require/untyped-contract 也无济于事,因为 (All (a) (-> (-> Any Any) (Listof Any) (Listof a))) 不是一个子类型。

在这种情况下,我们仍然可以使用以下方法为未类型化的代码提供 my-filter
(provide my-filter)
 
(define-typed/untyped-identifier my-filter
  typed:my-filter
  untyped:my-filter)
其中 typed:my-filter 是原始的 my-filter, 但使用 prefix-in 导入, 而 untyped:my-filter 是它的类型 (All (a) (-> (-> Any Any) (Listof Any) (Listof a))) 的 Typed Racket 实现或未 typed Racket 实现。

如果可能的话,请避免这样做。只在一个类型没有可以转换为契约的子类型的情况下使用; 也就是说,不能使用 require/untyped-contract 的情况。

procedure

(syntax-local-typed-context?) → boolean?

如果在类型的上下文中扩展代码时被调用,返回 #t ;否则返回 #f 。

这是个核选项,提供这个选项是因为它有时但很少有用。避免。
