12 不安全的 Typed Racket 操作

 (require typed/racket/unsafe) package: typed-racket-lib

警告: 本节中记录的操作是 unsafe,也就是说,它们可以绕过类型系统的不变性。 除非使用 #:no-optimize 语言选项,否则可能会导致不可预测的行为,甚至可能使 Typed Racket 崩溃。

syntax

(unsafe-require/typed m rt-clause ...)

这种形式要求从模块 m 中导入标识符,其导入规范与 require/typed 相同。

与 require/typed 不同的是,这种形式是不安全的, 不会生成对应于指定类型的契约,以检查值是否真的与它们的类型相符。

Examples:
> (require typed/racket/unsafe)
; import with a bad type
> (unsafe-require/typed racket/base [values (-> String Integer)])
; unchecked call, the result type is wrong
> (values "foo")

- : Integer

"foo"

Added in version 1.3 of package typed-racket-lib.
Changed in version 1.6: 增加了对结构类型变量的支持

syntax

(unsafe-provide provide-spec ...)

这种形式从一个模块中声明出口,其语法与 provide 形式相同。

与 provide 不同,这种形式是不安全的,Typed Racket 不会生成任何与指定类型相对应的契约。 这意味着在其他模块中使用出口可能会规避类型系统的不变性。 特别是,一个类型化模块可能不安全地提供从另一个类型化模块导入的标识符。

此外,将一个用 unsafe-provide 导出的标识符导入另一个类型化模块, 然后用 provide 重新导出,将不会导致合约的产生。

在其他类型化模块中使用被提供的标识符不受 unsafe-provide 的影响, 在这些情况下,它的行为与 provide 相同。 此外,其他 使用 在 unsafe-provide 中的绑定的类型化模块仍然会像往常一样生成合约。

Examples:
> (module t typed/racket/base
    (require typed/racket/unsafe)
    (: f (-> Integer Integer))
    (define (f x) (add1 x))
    ; unsafe export, does not install checks
    (unsafe-provide f))
> (module u racket/base
    (require 't)
    ; bad call that's unchecked
    (f "foo"))
> (require 'u)

add1: contract violation

  expected: number?

  given: "foo"

Added in version 1.3 of package typed-racket-lib.
Changed in version 1.8: 添加了对重新提供的类型变量的支持

syntax

(unsafe-require/typed/provide m rt-clause ...)

与 require/typed/provide 类似,除了这种形式是不安全的, 并且不会生成与指定类型相对应的合约来检查值是否真正符合其类型。
