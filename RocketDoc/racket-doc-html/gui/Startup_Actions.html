12 启动动作

racket/gui/base 模块在每个操作系统进程中只能被实例化一次, 因为它在 Racket 运行时系统中设置了钩子来协调 Racket 线程调度和 GUI 事件。 试图第二次实例化它将导致一个异常。 此外,在 Mac OS 上,racket/gui/base 的唯一实例化必须在进程的原始 place 。

实例化 racket/gui/base 会设置两个参数:

  • executable-yield-handler — 可执行的产量处理程序被设置为在链到先前安装的处理程序之前评估 (yield initial-eventspace)。 因此,Racket 进程通常会等待所有顶层窗口关闭,所有回调被调用,所有定时器在初始事件空间中停止,然后进程退出。

  • current-get-interaction-input-port — 交互端口处理程序被设置为包裹先前安装的处理程序的结果, 以便在输入端口读取时阻塞时屈服于 GUI 事件。 只有当当前线程是某个事件空间的处理程序线程时,才会触发默认处理程序的这一扩展行为, 在这种情况下, current-eventspace 在调用 yield 前被设置为事件空间。 因此,GUI 事件通常可以在 read-eval-print-loop (如由普通的 Racket 可执行文件运行)在输入时被阻塞时被处理。

racket/gui/base 被实例化的线程也成为初始事件空间的 handler thread 。 另见 事件空间和线程 。
