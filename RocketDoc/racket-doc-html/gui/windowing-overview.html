1 窗口

窗口工具箱提供了 GUI 程序的基本构件,包括框架(顶级窗口)、模态对话框、菜单、按钮、复选框、文本字段和单选按钮–都是类。

参见 类和对象 ， 了解 Racket 中的类和接口的介绍。

1.1 创建窗口

要创建一个新的顶层窗口,需要实例化 frame% 类:

; Make a frame by instantiating the frame% class
(define frame (new frame% [label "Example"]))
 
; Show the frame by calling its show method
(send frame show #t)

内置的类提供了各种机制来处理 GUI 事件。 例如,在实例化 button% 类时,提供一个事件回调程序,当用户点击按钮时,就会被调用。 下面的例子程序创建了一个带有文本信息和按钮的框架;当用户点击按钮时,信息会改变:

; Make a frame by instantiating the frame% class
(define frame (new frame% [label "Example"]))
 
; Make a static text message in the frame
(define msg (new message% [parent frame]
                          [label "No events so far..."]))
 
; Make a button in the frame
(new button% [parent frame]
             [label "Click Me"]
             ; Callback procedure for a button click:
             [callback (lambda (button event)
                         (send msg set-label "Button click"))])
 
; Show the frame by calling its show method
(send frame show #t)

程序员从不直接实现 GUI 的事件循环。 相反,窗口系统会自动从一个内部队列中提取每个事件,并将该事件派发到一个适当的窗口。 派遣调用窗口的回调程序或调用窗口的一个方法。 在上面的程序中,只要用户点击 Click Me ,窗口系统就会自动调用按钮的回调过程。

如果一个窗口接收多种事件,这些事件会被派发到窗口类的方法中,而不是派发到回调过程中。 例如,一个画布接收更新事件、鼠标事件、键盘事件和尺寸事件;要处理这些事件,可以从内置的 canvas% 类派生出一个新的类,并覆盖事件处理方法。 下面的表达式用一个可以处理鼠标和键盘事件的画布来扩展上面创建的框架:

; Derive a new canvas (a drawing window) class to handle events
(define my-canvas%
  (class canvas% ; The base class is canvas%
    ; Define overriding method to handle mouse events
    (define/override (on-event event)
      (send msg set-label "Canvas mouse"))
    ; Define overriding method to handle keyboard events
    (define/override (on-char event)
      (send msg set-label "Canvas keyboard"))
    ; Call the superclass init, passing on all init args
    (super-new)))
 
; Make a canvas that handles events in the frame
(new my-canvas% [parent frame])

运行上述代码后,手动调整框架的大小以看到新的画布。 将光标移到画布上,用一个代表运动事件的对象调用画布的 on-event 方法。 点击画布就会调用 on-event 。 当画布拥有键盘焦点时,在键盘上打字会调用画布的 on-char 方法。

视窗系统按顺序分配 GUI 事件;也就是说,在调用一个事件处理回调或方法后,视窗系统会等待处理程序返回,然后再分配下一个事件。 为了说明事件的顺序性,再次扩展框架,添加一个 Pause 按钮:

(new button% [parent frame]
             [label "Pause"]
             [callback (lambda (button event) (sleep 5))])

在用户点击 Pause 后,整个框架在五秒钟内没有反应; 窗口系统不能分派更多的事件,直到对 sleep 的调用返回。 关于事件调度的更多信息,请参见 事件调度和事件空间 。

除了调度事件外,GUI 类还处理窗口的图形布局。 我们的例子框架展示了一个简单的布局;框架的元素从上到下排成一排。 一般来说,程序员通过将每个 GUI 元素分配给一个父 container 来指定窗口的布局。 一个垂直的容器,如框架,将其子元素排列成一列,而一个水平的容器将其子元素排列成一排。 一个容器可以是另一个容器的子代;例如,为了在我们的框架中并排放置两个按钮,为新按钮创建一个水平面板:

(define panel (new horizontal-panel% [parent frame]))
(new button% [parent panel]
             [label "Left"]
             [callback (lambda (button event)
                         (send msg set-label "Left click"))])
(new button% [parent panel]
             [label "Right"]
             [callback (lambda (button event)
                         (send msg set-label "Right click"))])

关于窗口布局和容器的更多信息,请参见 几何图形管理 。

1.2 在画布上绘图

画布的内容是由它的 on-paint 方法决定的, 默认的 on-paint 会调用画布创建时提供的 paint-callback 函数。 on-paint 方法不接收任何参数, 并使用画布的 get-dc 方法来获得 drawing context (DC) ; 默认的 on-paint 方法将画布和这个 DC 传递给 paint-callback 函数。 racket/draw 工具箱在 DC 上的绘制操作会反映在屏幕上的画布内容中。

例如,下面的程序创建了一个画布,显示大而友好的字母:

(define frame (new frame%
                   [label "Example"]
                   [width 300]
                   [height 300]))
(new canvas% [parent frame]
             [paint-callback
              (lambda (canvas dc)
                (send dc set-scale 3 3)
                (send dc set-text-foreground "blue")
                (send dc draw-text "Don't Panic!" 0 0))])
(send frame show #t)

画布的背景颜色可以通过 set-canvas-background 方法来设置。 要使画布透明(这样它就会采用其父辈的颜色和纹理作为其初始内容), 在创建画布时在 style 参数中提供 'transparent 。

参见 Racket 绘图工具箱 中的 概述 , 以了解使用 racket/draw 库绘图的概况。 关于画布绘制的更多高级信息,请看 画布中的动画 。

1.3 核心窗体类

窗口工具箱中的基本图形元素是一个 area 。 下面的类实现了视窗工具箱中不同类型的区域:

  • Containers — 可以包含其他区域的区域:

    • frame% — frame 是一个顶层窗口,用户可以移动和调整大小。

    • dialog% — dialog 是一个模式化的顶层窗口; 当一个对话框被显示时,其他顶层窗口被禁用,直到对话框被驳回。

    • panel% — panel 是容器中的一个子容器。 工具箱提供了 panel% 的三个子类: vertical-panel%, horizontal-panel% 和 tab-panel% 。

    • pane% — pane 是一个轻量级的面板。 它没有图形表示或事件处理能力。 pane% 类有三个子类: vertical-pane%, horizontal-pane% 和 grow-box-spacer-pane% 。

  • Containees — 必须包含在其他区域中的区域:

    • panel% — 一个面板是一个被包含者,也是一个容器。

    • pane% — 一个窗格是一个被包含的对象,也是一个容器。

    • canvas% — canvas 是一个子窗口,用于在屏幕上绘图。

    • editor-canvas% — editor canvas 是一个子窗口,用于显示文本编辑器或粘贴板编辑器。 editor-canvas% 类与编辑器类一起被记录在 编辑器 。

    • Controls — 用户可以操作的对象:

      • message% — message 是一个静态的文本字段或位图,没有用户互动。

      • button% — button 是一个可点击的控件。

      • check-box% — check box 是一个可点击的控件; 用户点击该控件来设置或删除其复选标记。

      • radio-box% — radio box 是一个相互排斥的 radio buttons 的集合; 当用户点击一个单选按钮时,它被选中,而该单选框之前被选中的单选按钮被取消选择。

      • choice% — choice item 是一个弹出式的文本选择菜单;用户在控件中选择一个项目。

      • list-box% — list box 是一个可滚动的文本选择列表; 用户选择列表中的一个或多个项目(取决于列表框的样式)。

      • text-field% — text field 是一个用于简单文本输入的盒子。

      • combo-field% — combo field 结合了一个文本字段和一个弹出的选择菜单。

      • slider% — slider 是一个可拖动的控件,可以在一个固定的范围内选择一个整数值。

      • gauge% — gauge 是一个只输出的控件(用户不能改变数值), 用于报告一个固定范围内的整数值。

正如上面列出的建议,某些 areas, 称为 containers, 管理某些其他的区域,称为 containees 。 有些区域,如面板,既是 containers 和 containees 。

大多数区域是 windows, 但有些是 non-windows 。 一个 window ,如 panel, 有一个图形表示,接收键盘和鼠标事件,并且可以被禁用或隐藏。 相反,一个 non-window, 如 pane, 只对几何图形管理有用; non-window 不接收鼠标事件,也不能被禁用或隐藏。

每个 area 都是 area<%> 接口的一个实例。 每个 container 也是 area-container<%> 接口的一个实例, 而每个 containee 是 subarea<%> 的一个实例。 Windows 是 window<%> 的实例。 area-container<%>, subarea<%> 和 window<%> 接口是 area<%> 的子接口。

菜单栏、菜单和菜单项是图形元素,但不是区域(也就是说,它们没有区域所共有的所有属性,如可调整的图形大小)。 相反,菜单类形成了一个单独的容器-容器的层次结构:

  • 菜单项容器

    • menu-bar% — menu bar 是与一个框架相关的菜单的顶级集合。

    • menu% — menu 包含一组菜单项。 该菜单可以出现在菜单栏、弹出式菜单中,或者作为另一个菜单的子菜单。

    • popup-menu% — popup menu 是一个顶层菜单,动态地显示在画布或编辑器画布中。

  • 菜单项

    • separator-menu-item% — separator 是菜单或弹出式菜单中不可选择的行。

    • menu-item% — plain menu item 是一个菜单中可选择的文本项。 当该项目被选中时,其回调过程被调用。

    • checkable-menu-item% — checkable menu item 是菜单中的一个文本项; 用户选择一个可检查的菜单项来切换该项目旁边的一个检查标记。

    • menu% — 菜单是一个菜单项,也是一个菜单项容器。

1.4 几何图形管理

窗口工具箱的几何管理使得设计窗口变得容易,尽管 GUI 元素的图形表示方式不同,但在所有平台上看起来都是正确的。 几何管理是基于容器的;每个容器根据简单的约束条件来安排它的子代,比如一个框架的当前尺寸和一个按钮的自然尺寸。

内置的容器类包括水平面板(和窗格)和垂直面板(和窗格),前者将其子代排列成一排,后者将其子代排列成一列。 通过嵌套水平和垂直容器,程序员可以实现大多数的布局。 例如,要构建一个形状为

   ------------------------------------------------------

  |              -------------------------------------   |

  |  Your name: |                                     |  |

  |              -------------------------------------   |

  |                    --------     ----                 |

  |                   ( Cancel )   ( OK )                |

  |                    --------     ----                 |

   ------------------------------------------------------

with the following program:

; Create a dialog
(define dialog (instantiate dialog% ("Example")))
 
; Add a text field to the dialog
(new text-field% [parent dialog] [label "Your name"])
 
; Add a horizontal panel to the dialog, with centering for buttons
(define panel (new horizontal-panel% [parent dialog]
                                     [alignment '(center center)]))
 
; Add Cancel and Ok buttons to the horizontal panel
(new button% [parent panel] [label "Cancel"])
(new button% [parent panel] [label "Ok"])
(when (system-position-ok-before-cancel?)
  (send panel change-children reverse))
 
; Show the dialog
(send dialog show #t)

每个容器使用每个子项的自然尺寸来排列它的子项,这通常取决于子项的实例化参数,如按钮上的标签或单选框中的选择数。 在上面的例子中,对话框水平拉伸以匹配文本字段的最小宽度,它垂直拉伸以匹配字段和按钮的总高度。 然后,对话框将水平面板拉长,以填满对话框的下半部分。 最后,水平面板使用按钮的最小宽度的总和来使它们在水平方向上居中。

正如这个例子所演示的,一个可拉伸的容器会增长以填充其环境,并在其可拉伸的子代中分配额外的空间。 默认情况下,面板可以向两个方向拉伸,而按钮则不能向任何方向拉伸。 程序员可以改变一个单独的 GUI 元素是否是可伸展的。

下面的小节详细描述了容器系统,首先在 容器 中讨论了容器的属性, 然后在 容器 中描述了容器的属性。 除了内置的垂直和水平容器外,程序员还可以定义新的容器类型,在最后一小节 定义新类型的容器 中讨论。

1.4.1 容器

每一个 containee, 或者说孩子,都有以下属性:

  • graphical minimum width 和 graphical minimum height;

  • requested minimum width 和 requested minimum height;

  • 水平和垂直 stretchability (开或关);以及

  • 水平和垂直 margins 。

container 根据每个 containee 的这四个属性来安排其子代。 一个 containee 的父容器是在创建 containee 时指定的。 一个窗口的 containee 可以在其父级容器中 hidden 或 deleted , 其父级容器可以通过 reparent 来改变。

正如 get-graphical-min-size 所报告的那样, 某个特定被包含体的 graphical minimum size 取决于平台、 被包含体的标签(对于控件)以及创建被包含体时指定的样式属性。 例如,一个按钮的最小图形尺寸可以确保标签的整个文本是可见的。 控件(如按钮)的最小图形尺寸不能改变;它在创建时是固定的。 (当一个控件的标签被改变时,它的最小尺寸 不 会被重新计算)。 面板或窗格的图形最小尺寸取决于其子对象的总最小尺寸和它们的排列方式。

要为一个被包含者选择尺寸,其父级容器会考虑被包含者的 requested minimum size , 而不是其图形最小尺寸(假设请求的最小尺寸大于图形最小尺寸)。 与图形最小尺寸不同, 程序员可以使用 min-width 和 min-height 方法随时改变被包含体的请求最小尺寸。

除非被包含物是可拉伸的(在特定的方向),否则它总是收缩到其最小尺寸(在相应的方向)。 否则,被包含体将被拉伸以填满容器中的所有可用空间。每个容器开始时都有一个默认的伸展性。 例如,按钮最初是不可拉伸的,而单行文本字段最初在水平方向上是可拉伸的。 程序员可以在任何时候使用 stretchable-width 和 stretchable-height 方法来改变一个容器的伸展性。

margin 是指被包含体周围的空间。 每个被包含体的边距都与它的最小尺寸无关,但从容器的角度来看,边距实际上增加了被包含体的最小尺寸。 例如,如果一个按钮的垂直边距为 2,那么容器必须分配足够的空间,除了为按钮的最小高度分配的空间外, 还要在按钮上方和下方留出两个像素的空间。 程序员可以用 horiz-margin 和 vert-margin 来调整一个容器的边距。 控件的默认边距为 2 ,其他类型的被包含物的默认边距为 0 。

在实践中,控件的 requested minimum size 和 margin 很少被改变, 尽管它们经常被改变为 canvas 。 Stretchability 通常为任何类型的被包含物进行调整,这取决于程序员所期望的视觉效果。

1.4.2 容器

一个容器有以下属性:

  • 一个(未删除的)子容器的列表;

  • 一个要求的最小宽度和一个要求的最小高度;

  • 子代之间使用的间距;

  • 围绕全部子集使用的边距;

  • 水平和垂直伸展性(开或关);以及

  • 用于定位剩余空间的对齐设置。

这些属性被计入容器对其自身尺寸的计算和对其子代的安排。 对于一个同时也是被容纳者的容器(如面板)来说,该容器所要求的最小尺寸和可伸缩性与它的被容纳者方面相同。

在创建被包含体时,会指定被包含体的父容器。 一个被包含的窗口可以在它的父容器中被 hidden 或 deleted , 它的父容器也可以通过 reparent 来改变(但是一个非窗口的被包含者不能被 hidden, deleted 或 reparent):

  • 一个 hidden 子节点对用户来说是不可见的,但是在一个容器中仍然为每个隐藏的子节点分配空间。 要隐藏或显示一个孩子,请调用孩子的 show 方法。

  • 一个 deleted 子代会被隐藏, 并 在容器安排其他子代时被忽略, 所以容器中不会为被删除的子代保留空间。 要使一个子代被删除或不被删除, 请调用容器的 delete-child 或 add-child 方法(它调用子代的 show 方法)。

  • 要 reparent 一个窗口的被包含者,使用 reparent 方法。 窗口在其新的父代中保留其 hidden 或 deleted 的状态。

当一个子窗口被创建时,它最初被显示并且不被删除。 当一个被删除的子窗口不存在对该子窗口的外部引用时,该子窗口会被垃圾回收。 容器可以通过它的 get-children 方法得到非删除的子代(无论是否隐藏)的列表。

容器的非删除列表中的子节点的顺序是很重要的。例如,一个垂直面板将其列表中的第一个孩子放在面板的顶部,以此类推。 当一个新的孩子被创建时,它被放在其容器的孩子列表的最后。 容器列表的顺序可以通过 change-children 方法动态地改变。 ( change-children 方法也可以用来激活或停用子代。)

容器的 graphical minimum size , 正如 get-graphical-min-size 所报告的那样, 是通过结合其子代的最小尺寸(根据容器的布局策略,将它们相加或取其最大值)以及容器的间距和边距来计算的。 程序员可以使用 min-width 和 min-height 方法指定一个更大的最小值; 当一个容器的计算的最小值大于程序员指定的最小值时,那么程序员指定的最小值将被忽略。

容器的间距决定了容器中相邻的子节点之间的空间量,此外还有子节点的边距所要求的空间。 容器的边距决定了在子集周围增加的空间量;它有效地减少了容器内可以放置子集的区域。 程序员可以通过 border 和 spacing 方法动态地调整容器的边界和间距。 对于所有的容器类型,默认的边框和间距是 0 。

因为面板或窗格既是一个容器,也是一个包含物,所以除了它的边框边距之外,它还有一个包含物边距。 对于面板来说,这些边距并不是多余的,因为面板可以有一个图形边框;边框是在面板的容器边距内绘制的,但在面板的边框边距之外。

对于一个顶层窗口容器,如框架或对话框,容器的可伸展性决定了用户是否可以将窗口的大小调整到大于其最小尺寸。 因此,用户不能调整一个不可拉伸的框架的大小。 对于其他类型的容器(即面板和窗格),容器的可伸展性是它作为其他容器的被包含者的可伸展性。 所有类型的容器最初都是双向可拉伸的–除了 grow-box-spacer-pane% 的实例, 它的目的是作为一个轻量级的间隔类而不是一个有用的容器类–但是程序员可以在任何时候通过 stretchable-width 和 stretchable-height 方法改变一个区域的可拉伸性。

容器的对齐规范决定了当容器有剩余空间时它如何定位它的子代。 (一个容器只有在它的子代没有可伸展的方向时,才能在特定的方向上有剩余的空间。) 例如,当容器的水平对齐方式是 'left 时,子代在容器中是左对齐的,剩余的空间被累积到右边。 当容器的水平对齐方式是 'center 时,每个孩子在容器中都是水平居中的。 容器的对齐方式可以用 set-alignment 方法来改变。

1.4.3 定义新类型的容器

尽管嵌套的水平和垂直容器可以表达大多数的布局模式,但程序员可以用一个显式的布局程序来定义一种新的容器类型。 程序员通过从 panel% 或 pane% 派生出一个类并重写 container-size 和 place-children 方法来定义一个新类型的容器。 container-size 方法接收每个子节点的尺寸规格列表并返回两个值:容器的最小宽度和高度。 place-children 方法接收容器的尺寸和每个子节点的尺寸规格列表, 并返回一个尺寸和位置的列表(与原始列表并行)。

一个输入的尺寸规格是一个包含四个值的列表:

  • 子项的最小宽度;

  • 子项的最小高度;

  • 子项的水平伸展性(#t 表示可伸展, #f 表示不可伸展); 以及

  • 子项的垂直伸展性。

对于 place-children, 输出位置和尺寸规格是一个包含四个值的列表:

  • 子项的新水平位置(相对于父代);

  • 子项的新的垂直位置;

  • 子项的新的实际宽度;

  • 子项的新的实际高度。

输入和输出的宽度和高度都包括子代的边距。 每个孩子的返回位置都会自动增加,以考虑到孩子在放置控件时的边距。

1.5 鼠标和键盘事件

每当用户移动鼠标,点击或释放鼠标按钮,或按下键盘上的一个键,就会为某个窗口产生一个事件。 接收事件的窗口取决于图形显示的当前状态:

  • 鼠标事件的接收窗口通常是鼠标移动或点击时光标下的窗口。 如果鼠标在一个子窗口上,子窗口接收事件,而不是其父窗口。

    当用户在一个窗口中点击时,该窗口会 "抓住 "鼠标, 因此 all 的鼠标事件都会进入该窗口,直到鼠标按钮被释放(不管光标的位置如何)。 因此,用户可以点击一个滚动条的拇指并拖动它,而不需要将光标严格保持在滚动条控件内。

    通常为每个鼠标按键下降事件生成一个鼠标按键释放事件,但按键释放事件可能会被丢弃。 例如,一个模式化的对话框可能会出现并接管鼠标。 更普遍的是,任何种类的鼠标事件原则上都可能被丢弃,所以要避免依赖精确的鼠标事件序列的算法。 例如,一个鼠标跟踪处理程序在收到除拖动事件以外的其他事件时,应该重置跟踪状态。

  • 键盘事件的接收窗口是在事件发生时拥有 keyboard focus 的窗口。 在任何时候都只有一个窗口拥有焦点,焦点所有权通常由一个窗口以某种方式显示。 例如,一个文本字段控件通过显示一个闪烁的光标来显示焦点所有权。

    在一个顶层窗口中,只有某些类型的子窗口可以拥有焦点,这取决于平台的惯例。 此外,最初拥有焦点的子窗口是特定于平台的。 用户可以通过各种方式移动焦点,通常是通过点击目标窗口。 一个程序可以使用 focus 方法将焦点移动到一个子窗口或设置初始焦点。

    'wheel-up 或 'wheel-down 的事件可能会被发送到一个非键盘焦点的窗口, 这取决于操作系统如何处理滚轮事件。

    按键事件可以对应于实际按键或自动按键的重复。 多次按键事件而没有间隔的按键释放事件,通常表示一个自动按键。 然而,像任何输入事件一样,按键释放事件有时会被放弃(例如,由于模态对话框的出现)。

控件,如按钮和列表框,自动处理键盘和鼠标事件,最终调用创建控件时提供的回调程序。 一个画布将鼠标和键盘事件分别传播到它的 on-event 和 on-char 方法。

鼠标和键盘事件会以一种特殊的方式传递给它的窗口。 接收窗口的每个祖先都有机会通过 on-subwindow-event 和 on-subwindow-char 方法拦截该事件。 更多信息请参见方法描述。

顶层窗口的默认 on-subwindow-char 方法拦截键盘事件,以检测菜单快捷键事件和焦点导航事件。 详情见 on-subwindow-char in frame% 和 on-subwindow-char in dialog% 。 某些操作系统特定的按键组合在低级别的情况下被捕获,并且不能被覆盖。 例如,在 Windows 和 Unix 中,按下并释放 Alt 键总是将键盘焦点移至菜单栏。 同样,在 Windows 上,Alt-Tab 会切换到一个不同的应用程序。 (Alt-Space 在 Windows 上调用系统菜单, 但这个快捷键是由 on-system-menu-char 实现的, 它被 on-subwindow-char in frame% 和 on-subwindow-char in dialog% 调用)。

1.6 事件调度和事件空间

图形用户界面是一个固有的多线程系统:一个线程是管理屏幕上窗口的程序,另一个线程是用户移动鼠标和在键盘上打字。 GUI 程序通常使用一个 event queue 将这个多线程系统转化为一个顺序的系统,至少从程序员的角度来看是这样。 每个用户的动作都是一个一个地处理的,在前一个动作完全处理完之前,忽略其他的用户动作。 从多线程进程到单线程进程的转换大大简化了 GUI 程序的实现。

尽管纯顺序的事件队列提供了编程上的便利,但在某些情况下需要与用户进行不那么僵硬的对话:

  • Nested event handling: 在处理一个事件的过程中,可能需要从用户那里获得进一步的信息。 通常,这样的信息是通过模式对话框获得的;无论以何种方式获得输入,在最初的事件被完全处理之前,必须接收和处理更多的用户事件。 为了允许进一步处理事件,原始事件的处理程序必须明确地 yield 于系统。 屈服会使事件以嵌套的方式被处理,而不是以纯粹的顺序方式。

  • Asynchronous event handling: 一个应用程序可能由代表与用户独立对话的窗口组成。 例如,一个绘图程序可能支持多个绘图窗口,一个窗口中特别耗时的任务(例如,图像上的特殊过滤效果)不应妨碍用户在不同窗口中工作。 这样的应用程序需要对每个单独的窗口进行顺序的事件处理,但在不同的窗口中需要异步(可能是并行)的事件处理。 换句话说,应用程序需要为每个窗口提供一个单独的事件队列,并为每个事件队列提供一个单独的事件处理线程。

一个 eventspace 是处理 GUI 事件的一个上下文。 每个事件空间都维护它自己的事件队列,单个事件空间中的事件由指定的 handler thread 按顺序分派。 在这个处理线程中运行的事件处理程序可以通过调用 yield 向系统屈服, 在这种情况下,其他的事件处理程序可以在同一个处理线程中以嵌套(但是单线程)的方式调用。 来自不同事件空间的事件是由不同的处理程序线程异步分派的。

当 一个框架或对话框在没有父级的情况下被创建时, 它与 创建和设置事件空间 中描述的 current eventspace 相关。 顶层窗口及其下级的事件总是在窗口的事件空间中被分派。 每个对话框都是模态的;一个对话框的 show 方法隐含地调用 yield 来处理对话框显示时的事件。 (关于线程和模态对话框的信息,请参见 事件空间和线程 。 此外,当一个模态对话框被显示时,系统会禁用对话框事件空间中其他顶层窗口的按键和鼠标按压/释放事件, 但其他事件空间中的窗口不受模态对话框的影响。 (当显示模态对话框时,鼠标运动、输入和离开事件仍然传递给所有窗口)。

1.6.1 事件类型和优先级

除了与用户和窗口操作相对应的事件,如按钮点击、按键和更新,系统还派发两种内部事件: timer events 和 explicitly queued events 。

Timer events 是由 timer% 的实例创建的。 当一个定时器被启动,然后过期时,定时器会排队调用定时器的 notify 方法。 像顶层窗口一样,每个定时器在创建时都与一个特定的事件空间( 创建和设置事件空间 中描述的 current eventspace )相关联,并且定时器在其事件空间中排队等待事件。

Explicitly queued events 是用 queue-callback 创建的,它接受一个回调过程来处理事件。 在调用 queue-callback 时,事件被排在当前的事件空间中, 其优先级是高还是低,由 queue-callback 的第二个参数(可选)指定。

一个事件空间的事件队列实际上是一个优先级队列,事件根据它们的种类进行排序,从最高优先级(首先派发)到最低优先级(最后派发):

  • 用 queue-callback 安装的高优先级事件具有最高优先级。

  • 通过 timer% 的定时器事件具有第二高的优先级。

  • 窗口刷新事件具有第三高的优先级。

  • 输入事件,如鼠标点击或按键,具有第二低的优先级。

  • 用 queue-callback 安装的低优先级事件的优先级最低。

尽管程序员不能直接控制事件被派发的顺序, 但程序员可以通过 event-dispatch-handler 参数设置 event dispatch handler 来控制派发的时间。 这个参数和其他事件空间程序在 事件空间 中有更详细的描述。

1.6.2 事件空间和线程

当一个新的事件空间被创建时,一个相应的 handler thread 线程将为该事件空间创建。 初始事件空间不会创建一个新的处理程序线程, 而是使用 racket/gui/base 实例化的线程作为初始事件空间的处理程序线程; 另见 启动动作 。

当系统为一个事件空间分派事件时,它总是在事件空间的处理程序线程中进行。 一个处理程序可以创建新的线程,无限期地运行,但只要处理程序线程在运行处理程序,就不能为相应的事件空间分派新的事件。

当一个处理程序线程显示一个对话框时,只要对话框被显示, 对话框的 show 方法就会隐含地调用 yield 。 当一个非处理程序线程显示一个对话框时,非处理程序线程简单地阻塞,直到对话框被驳回。 在非处理程序线程中调用没有参数的 yield 没有任何效果。 在非处理程序线程中调用带有信号的 yield 等同于调用 semaphore-wait 。

来自 racket/gui/base 的窗口函数和方法可以在任何线程中被调用, 但要注意在线程之间或与处理程序线程之间产生竞赛条件:

  • 尽管图形对象是线程安全的,但如果图形元素在多个线程中被操作,回调或其他事件处理程序可能会看到对象状态的变化。

  • 编辑器类有特定的线程约束。 参见 编辑器和线程 。

因为在非处理程序线程中操作 GUI 元素很容易产生混乱的竞赛条件(而回调可能在处理程序线程中运行), 所以最好在处理程序线程中执行所有 GUI 设置和操作。 queue-callback 函数可以帮助你从任何其他线程中安排处理程序线程的工作。 当已经在处理程序线程中运行时,使用 yield 来等待非 GUI 事件,同时允许 GUI 事件继续进行。

1.6.3 创建和设置事件空间

每当一个框架、对话框或计时器被创建时,它就会与 current eventspace 相关联, 这由 current-eventspace 参数 (参见 参数) 决定。

make-eventspace 过程创建一个新的事件空间。 下面的例子在事件空间中创建了一个新的事件空间和一个新的框架 ( parameterize 的语法形式临时设置了一个参数值):

(let ([new-es (make-eventspace)])
  (parameterize ([current-eventspace new-es])
    (new frame% [label "Example"])))

当一个事件空间被创建时,它被置于 current custodian 的管理之下。 当监护人关闭一个事件空间时, 与该事件空间相关的所有框架和对话框都被销毁 (不需要调用 can-close? 或 on-close in top-level-window<%>), 事件空间中的所有计时器都被停止,所有排队的回调都被删除。 试图在一个关闭的事件空间中创建一个新的窗口、计时器或明确的队列事件会引发 exn:misc 异常。

事件空间是一个 synchronizable event (不要与 GUI 事件相混淆), 因此它可以与 sync 一起使用。 作为一个可同步的事件,当一个框架是可见的,一个计时器是激活的,一个回调是排队的, 或者一个 menu-bar% 是用一个 'root 父级创建的,事件空间就处于一个阻塞状态。 请注意,事件空间的阻塞状态与一个事件是否准备好被分派无关。 还要注意的是,在事件被处理、定时器被触发或回调被运行时,事件空间不一定处于阻塞状态, 如果一个事件空间的 handler thread 已经终止,它可能会被留在阻塞状态中。

1.6.4 连续性和事件调度

每当系统分派一个事件时,对处理程序的调用就会被包裹上一个 continuation prompt (见 call-with-continuation-prompt),该提示限定了延续中止(例如当一个异常被引发时)和处理程序捕获的延续。 划定的继续提示被安装在对 event dispatch handler 的调用之外, 所以任何捕获的继续包括对 event dispatch handler 的调用。

例如,如果一个按钮回调引发了一个异常,那么由默认的异常处理程序执行的中止会返回到事件调度点, 而不是终止程序或逃过一个封闭的 (yield) 。 如果 with-handlers 封装了一个导致按钮回调引发异常的 (yield) , 那么这个异常可以被 with-handlers 捕获。

类似地,如果一个按钮回调捕获了一个续集(使用默认的续集提示标签), 那么应用续集只重新安装处理程序在返回之前要做的工作;调用按钮回调的调度机制不包括在续集中。 因此,在按钮回调过程中捕获的续集在同一回调之外可能是有用的。

1.6.5 记录

GUI 系统记录了事件被处理的时间以及它们被处理的时间。 每个涉及到回调到 Racket 代码的事件都有两个事件被记录下来,它们都使用 gui-event 结构：

(struct gui-event (start end name) #:prefab)

start 字段是事件处理开始时 (current-inexact-milliseconds) 的结果。 end 字段是事件处理开始时日志信息的 #f , 以及事件结束时日志信息的 (current-inexact-milliseconds) 结果。 name 字段是处理该事件的函数的名称; 如果是基于 queue-callback 的事件,它是传递给 queue-callback 的 thunk 的名称。

1.7 画布中的动画

画布的内容是缓冲的,所以如果画布必须重新绘制, 通常不需要再次调用 on-paint方法或 paint-callback 函数。 为了进一步减少闪烁,在调用 on-paint 方法或 paint-callback 函数时, 窗口系统会避免将画布内容的缓冲区冲到屏幕上。

画布内容可以通过使用画布的 get-dc 方法的结果随时更新,而且画布是线程安全的。 对画布内容的更改会定期刷新到屏幕上(不一定是在事件处理边界), 但 flush 方法会立即刷新到屏幕上–只要刷新没有被暂停。 suspend-flush 和 resume-flush 方法可以暂停和恢复自动和显式刷新, 尽管在某些平台上,自动刷新在少数情况下是强制的。

对于大多数动画来说, suspend-flush, resume-flush 和 flush 可以用来避免动画的闪烁和对额外的绘图缓冲区的需要。 在动画制作过程中,用 suspend-flush 和 resume-flush 来包围每个动画帧的构建,以确保部分绘制的帧不会被刷新到屏幕上。 如果 suspend-flush 很快就会发生, 那么就使用 flush 来确保画布内容在准备好的时候被刷新, 因为如果经常暂停刷新,刷新到屏幕的过程就会被饿死。 方法 refresh-now in canvas% 方便地封装了这个顺序。

1.8 屏幕分辨率和文本缩放

在 Mac OS 上,屏幕尺寸是以绘图单位来向用户描述的。 Retina 显示器为每个绘图单位提供了两个像素,而绘图单位则统一用于窗口大小、子窗口位置和画布绘制。 用于字体大小的 "点" 相当于一个绘图单位。

在 Windows 和 Unix 系统中,屏幕尺寸是以像素为单位向用户描述的,而用户可以独立选择一个比例来应用于文本和其他项目。 典型的文本比例是 125%、150% 和 200%。 racket/gui 库对所有的 GUI 元素使用这个比例,包括屏幕、窗口、按钮和画布绘制。 例如,如果比例是 200%,那么 get-display-size 报告的屏幕尺寸将是每个维度上像素数量的一半。 请注意,四舍五入效应可能导致报告的窗口尺寸与刚刚设置的窗口尺寸不同。 字体大小的 "点" 相当于 (/ 96 72) 个绘图单位。

在 Unix 上,如果 PLT_DISPLAY_BACKING_SCALE 环境变量被设置为一个正的实数, 那么它就会覆盖掉某些系统设置的 racket/gui 缩放比例。 在 GTK+ 3 中(见 平台依赖性), 环境变量覆盖了全系统的文本缩放; 在 GTK+ 2 中,环境变量覆盖了文本和控件的缩放。 然而,菜单、使用默认标签字体的控制标签和非标签控制部分不会使用通过 PLT_DISPLAY_BACKING_SCALE 指定的比例。

Changed in version 1.14 of package gui-lib: 增加了对 Unix 系统中缩放的支持。
