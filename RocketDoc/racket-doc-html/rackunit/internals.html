5 RackUnit 内部和扩展 API

本节介绍了 RackUnit 用于定制检查和测试行为以及创建新型测试运行器的设施。

5.1 自定义检查评估

检查的语义由参数 current-check-around 和 current-check-handler 决定。 其他测试形式,如 test-begin 和 test-suite 会改变这些参数的值。

parameter

(current-check-handler) → (-> any/c any)

(current-check-handler handler) → void?
  handler : (-> any/c any)
包含处理检查失败引起的异常的函数的参数。 默认值是一个程序,它将以用户友好的格式显示异常数据。

parameter

(current-check-around) → (-> (-> any) any)

(current-check-around check) → void?
  check : (-> (-> any) any)
包含处理检查执行的函数的参数。 默认值是将 thunk 的评估包裹在一个 with-handlers 的调用中,如果有异常发生, 则调用 current-check-handler ,然后(当没有异常发生时)丢弃结果,返回 (void) 。

5.2 自定义测试评估

就像检查一样,有几个参数可以控制复合测试形式的语义。

parameter

(current-test-name) → (or/c string? false/c)

(current-test-name name) → void?
  name : (or/c string? false/c)
这个参数存储了当前测试用例的名称。 值为 #f 表示一个没有名字的测试用例,比如由 test-begin 构建的测试用例。

parameter

(current-test-case-around) → (-> (-> any) any)

(current-test-case-around handler) → void?
  handler : (-> (-> any) any)
这个参数处理测试用例的评估。 该参数的值是一个函数,它被传递给 thunk(一个没有参数的函数)。 该函数,当应用时,评估测试案例中的表达式。 current-test-case-around 参数的默认值是在捕获异常的情况下评估 thunk, 并打印出适当的信息表明测试案例失败。

procedure

(test-suite-test-case-around thunk) → any

  thunk : (-> any)
current-test-case-around 参数在 test-suite 的范围内被参数化为这个值。 这个函数创建一个测试用例结构,而不是立即评估 thunk。

procedure

(test-suite-check-around thunk) → any/c

  thunk : (-> any/c)
current-check-around 参数在 test-suite 的范围内被设定为这个值。 这个函数创建一个测试用例结构,而不是立即评估一个检查。

5.3 程序化运行测试和检查结果

RackUnit 提供了一个运行测试的 API,可以从中创建自定义 UI。

5.3.1 结果类型

struct

(struct exn:test exn:fail ()
    #:extra-constructor-name make-exn:test)
RackUnit 异常的基本结构。 你不应该捕捉这种类型的实例,只应该捕捉下面记录的子类型。

struct

(struct exn:test:check exn:test (stack)
    #:extra-constructor-name make-exn:test:check)
  stack : (listof check-info)
exn:test:check 是在检查失败时引发的,它包含失败时检查信息栈的内容。

struct

(struct test-result (test-case-name)
    #:extra-constructor-name make-test-result)
  test-case-name : (or/c string #f)
test-result 是以给定的名字运行测试的结果(用 #f 表示没有名字)。

struct

(struct test-failure test-result (result)
    #:extra-constructor-name make-test-failure)
  result : any
表示测试失败的测试结果的子类型。

struct

(struct test-error test-result (result)
    #:extra-constructor-name make-test-error)
  result : exn
表示测试错误的测试结果的子类型。

struct

(struct test-success test-result (result)
    #:extra-constructor-name make-test-success)
  result : any
表示测试成功的测试结果的子类型。

5.3.2 运行测试的函数

procedure

(run-test-case name action) → test-result

  name : (or/c string #f)
  action : (-> any)
运行给定的测试用例,返回一个代表成功、失败或错误的结果。

procedure

(run-test test)

 → (flat-murec-contract ([R (listof (or/c test-result? R))]) R)
  test : (or/c test-case? test-suite?)
运行给定的测试(测试用例或测试套件),返回一棵树(列表)的结果

例如:

(run-test
   (test-suite
    "Dummy"
    (test-case "Dummy" (check-equal? 1 2))))

procedure

(fold-test-results result-fn    
  seed    
  test    
  #:run run    
  #:fdown fdown    
  #:fup fup) → 'a
  result-fn : ('b 'c ... 'a . -> . 'a)
  seed : 'a
  test : (or/c test-case? test-suite?)
  run : (string (() -> any) . -> . 'b 'c ...)
  fdown : (string 'a . -> . 'a)
  fup : (string 'a . -> . 'a)
将 result-fn 从左到右深度优先折叠在 run 的结果上。 默认情况下, run 是 run-test-case ， fdown 和 fup 只是返回种子,所以 result-fn 是在测试结果上折叠的。

这个函数对于在测试结果上编写自定义折叠(以及由此产生的 UI)是非常有用的,而不需要你去处理所有预期的设置和拆除。 例如, fold-test-results 将为你运行测试套件的前后动作。 然而,它仍然足够灵活,通过其关键字参数,几乎可以做任何 foldts-test-suite 可以做的事情。 因此,它应该优先于 foldts-test-suite 使用。

result-fn 参数是一个从 run 结果(默认为 test-result) 和种子到一个新种子的函数。

seed 参数是任何值。

test 参数是一个测试用例或测试套件。

run 参数是一个从测试用例名称(字符串)和动作(thunk)到任何值的函数。 由 run 产生的值被送入 result-fn 。

fdown 参数是一个从测试套件名称(字符串)和种子到一个新种子的函数。

fup 参数是一个从测试套件名称(字符串)和种子到新种子的函数。

例子:

下面的代码计算了成功的数量:

(define (count-successes test)
  (fold-test-results
   (lambda (result seed)
     (if (test-success? result)
         (add1 seed)
         seed))
   0
   test))

下面的代码返回符号 'burp ,而不是运行测试案例。 注意 result-fn 是如何接收 run 的值的。

(define (burp test)
  (fold-test-results
   (lambda (result seed) (cons result seed))
   null
   test
   #:run (lambda (name action) 'burp)))

procedure

(foldts-test-suite fdown fup fhere seed test) → 'a

  fdown : (test-suite string thunk thunk 'a -> 'a)
  fup : (test-suite string thunk thunk 'a 'a -> 'a)
  fhere : (test-case string thunk 'a -> 'a)
  seed : 'a
  test : (or/c test-case? test-suite?)
foldts-test-suite 函数是一个漂亮的树形折叠(由 Oleg Kiselyov 创建), 它以一种有用的方式折叠测试 (fold-test-results 并不那么有用,因为你不能指定测试用例周围的动作)。

fdown 参数是一个测试套件的函数,测试套件名称,之前的动作,之后的动作,以及种子。 当一个测试套件在树上遇到时,它会被运行(预排序)。

fup 参数是测试套件、测试套件名称、行动前、行动后、当前级别的种子和子代返回的种子的函数。 它是在树上运行的(后序)。

fhere 参数是测试用例、测试用例名称、测试用例动作和种子的一个函数。 (注意,在不久的将来,这可能会改变为只有测试用例。 这种变化是为了让 fhere 辨别测试用例的子类型,这反过来又会让测试用例,例如,被忽略)。

例子:

下面是 foldts-test-suite 对 fold-test-results 的实现:

(define (fold-test-results suite-fn case-fn seed test)
  (foldts-test-suite
   (lambda (suite name before after seed)
     (before)
     (suite-fn name seed))
   (lambda (suite name before after seed kid-seed)
     (after)
     kid-seed)
   (lambda (case name action seed)
     (case-fn
       (run-test-case name action)
       seed))
   seed
   test))

如果你习惯于折叠,你可能会有点惊讶,你传递给 foldts-test-suite 的函数既接收它们操作的结构, 又接收该结构的内容。 这的确是不寻常的。 这样做是为了允许测试案例和测试套件的子类型以自定义的方式运行。 例如,你可以定义被忽略(不运行)的测试用例的子类型,或记录其执行时间,等等。 要做到这一点,运行测试用例的函数需要知道测试用例的类型,因此有必要提供这些信息。
