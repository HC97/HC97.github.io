4.4 评估和实例

 (require scribble/example) package: scribble-lib
scribble/example 库提供了在文档构建时评估代码并将结果纳入文档的实用程序, 特别是显示定义的程序和语法的使用示例。

Added in version 1.16 of package scribble-lib.

syntax

(examples option ... datum ...)

 
option = #:eval eval-expr
  | #:once
  | #:escape escape-id
  | #:label label-expr
  | #:hidden
  | #:result-only
  | #:no-inset
  | #:no-prompt
  | #:preserve-source-locations
  | #:no-result
  | #:lang language-name
类似于 racketinput, 只是每个输入 datum 的结果都显示在下一行。 结果是通过使用 eval-expr 产生的评估器对 datum 的 quote 形式进行评估而确定的。

每个关键字选项最多可以提供一次:

  • #:eval eval-expr — 指定一个评估器, 其中 eval-expr 必须产生 #f 或通过 make-evaluator 或 make-module-evaluator 产生一个沙盒评估器, sandbox-output 和 sandbox-error-output 参数设置为 'string 。 如果没有提供 eval-expr 或者是 #f, 就会用 make-base-eval 创建一个评估器。 也请参见 make-eval-factory 。

  • #:once — 指定评估器在评估完所有 datum 后用 close-eval 关闭。 如果没有指定 #:eval ,则假定 #:once 选项。

  • #:escape escape-id — 指定一个转义标识符,如 racketblock 。

  • #:label label-expr — 为例子指定一个标签,默认为 “Example:” 或 “Examples:” (取决于 datum 的数量)。 label-expr 的 #f 值会抑制标签的出现。

  • #:hidden — 指定 datum 和结果不应该被排版, 而应该被评估为一个副作用,并且禁用 eval:error 。 通常情况下,这个选项与 #:eval 结合起来配置一个评估器。

  • #:result-only — 指定 datum 结果应该被排版, 而不是 datum 本身,并且意味着 #:label #f 。

  • #:no-result — 暗示 #:no-prompt 和 #:label #f, 指定不应排版任何结果,并且禁用 eval:error 。

  • #:no-inset — 指定例子应该被排版,没有缩进,也就是说,像 racketinput0 而不是 racketinput 。

  • #:no-prompt — 指定每个例子应该在排版时没有前导提示,也就是说, 像 racketblock 而不是 racketinput 。 可以通过使用 eval:no-prompt 来省略特定 datum 的提示。

  • #:preserve-source-locations — 指定每个 datum 的原始源位置应该在评估时被保留下来。 保留源位置对于记录依赖于源位置的形式很有用,比如Redex的排版宏。

  • #:lang — 暗示 #:no-result 在排版的 datum 序列前加上 #lang 行,使用 language-name 作为模块的语言。

datum 中的某些模式被特别处理:

  • 形式为 (code:line code-datum (code:comment comment-datum ...)) 的 datum 将被视为 code-datum 进行评估。

  • 形式为 (code:line code-datum ...) 的 datum 会评估每个 code-datum,但只有最后的结果被使用。

  • 其他使用 code:comment, code:contract 和 code:blank 的情况在评估前会从每个 datum 中剥离出来。

  • 形式为 (eval:error eval-datum) 的 datum 与 eval-datum 一样被处理,但 eval-datum 被期望引发一个异常,并且一个错误被显示为评估的结果。

  • 形式为(eval:alts show-datum eval-datum) 的 datum 在排版时被视为 show-datum ,在评估时被视为 eval-datum 。

  • 形式为 (eval:check eval-datum expect-datum) 的 datum 被视为 eval-datum, 但 check-datum 也被评估,如果它们 equal? ,就会产生一个错误。

  • 形式为 (eval:result content-expr out-expr err-expr) 的 datum 不涉及沙盒评估;相反, content-expr 的 content 结果被用作结果的排版形式, out-expr 被视为表达式打印的输出, err-expr 是表达式打印的错误输出。 out-expr 和/或 err-expr 可以被省略,在这种情况下它们默认为空字符串。

    通常, eval:result 被用于 eval:alts 组合的第二部分。 否则, content-expr 被排版为输入形式(这对例子的读者很少有意义)。

  • 形式为 (eval:results content-list-expr out-expr err-expr) 的 datum 被视为 eval:result 形式, 除了 content-list-expr 应该为多个评价结果产生一个 content 列表。 与 eval:result 一样, out-expr 和 err-expr 是可选的。

  • 形式为 (eval:no-prompt eval-datum ...) 的 datum 与 (code:line eval-datum ...) 的处理方式相同, 但在组前不显示提示,并且在 eval-datum 和其结果前后添加一个空行。

一个 datum 不能是一个关键词。 要指定一个关键字的 datum ,请用 code:line 包裹它。

当评估一个 datum 产生错误时(并且 datum 没有 eval:error 包装), examples 会引发一个异常。

如果沙盒中的 current-print 的值从其默认值被改变, 或者沙盒中的 print-as-expression 被设置为 #f, 那么每个评估结果通过对值应用 (current-print) 而被格式化到一个端口; 输出端口被设置为支持 write-special 意义上的 special 的管道, 写入端口的非字符值被作为 content 使用。 否则,当默认的 current-print 到位时,结果值将使用 to-element/no-color 进行排版。

作为一个例子,

#lang scribble/manual
@(require racket/sandbox
          scribble/example)
@(define my-evaluator
   (parameterize ([sandbox-output 'string]
                  [sandbox-error-output 'string]
                  [sandbox-memory-limit 50])
     (make-evaluator 'typed/racket/base)))
 
@examples[#:eval my-evaluator
          (: my-sqr (Real -> Real))
          (define (my-sqr x)
            (* x x))
          (my-sqr 42)]

使用一个评估器,其语言是 typed/racket/base 。

procedure

(make-base-eval [#:pretty-print? pretty-print? 
  #:lang lang] 
  input-program ...) 
 → (any/c . -> . any)
  pretty-print? : any/c = #t
  lang : 
(or/c module-path?
      (list/c 'special symbol?)
      (cons/c 'begin list?))
 = '(begin)
  input-program : any/c
使用 (make-evaluator 'racket/base #:lang lang input-program ...) 创建一个评估器, 将沙盒参数设置为禁用限制,将输出设置为 'string, 并且不增加额外的安全防护。

如果 pretty-print? 为真,沙盒的打印机会被设置为 pretty-print-handler 。 在这种情况下,在 convertible? 意义上的可转换的值会使用 write-special 打印, 除了在 serializable? 意义上的可序列化的值被序列化用于从沙盒内部到外部的传输 (这可以避免将代码和支持从沙盒环境拉到文档渲染环境)。

Changed in version 1.6 of package scribble-lib: 改变了对可序列化的可转换值的处理。

procedure

(make-base-eval-factory mod-paths 
  [#:pretty-print? pretty-print? 
  #:lang lang]) 
 → (-> (any/c . -> . any))
  mod-paths : (listof module-path?)
  pretty-print? : any/c = #t
  lang : 
(or/c module-path?
      (list/c 'special symbol?)
      (cons/c 'begin list?))
 = '(begin)
产生一个类似于 make-base-eval 的函数, 只是 mod-paths 中的每个模块都被附加到评估器的命名空间。 这些模块被加载和实例化一次(当返回的像 make-base-eval 函数第一次被调用时), 然后被附加到每个被创建的评估器上。

procedure

(make-eval-factory mod-paths 
  [#:pretty-print? pretty-print? 
  #:lang lang]) 
 → (-> (any/c . -> . any))
  mod-paths : (listof module-path?)
  pretty-print? : any/c = #t
  lang : 
(or/c module-path?
      (list/c 'special symbol?)
      (cons/c 'begin list?))
 = '(begin)
像 make-base-eval-factory 一样, 但 mod-paths 中的每个模块也需要进入每个生成的评估器的顶级环境。

procedure

(make-log-based-eval log-file mode) → (-> any/c any)

  log-file : path-string?
  mode : (or/c 'record 'replay)
创建一个评估器(像 make-base-eval) ,使用一个日志文件来记录或重放评估结果。

如果 mode 是 'record, 评估器将每个交互记录到 log-file , 如果 log-file 已经存在,则将其替换。 每个交互的结果必须是 serializable 。

如果 mode 是 'replay , 评价器使用 log-file 的内容,而不是实际进行评价。 对于每个交互,它把要评估的术语与 log-file 中记录的下一个交互进行比较。 如果术语匹配,则返回存储的结果;如果不匹配,则评价器引发一个错误,表明它与 log-file 不同步。

当嵌入的例子依赖于外部功能时,使用 make-log-based-eval 来记录库, 而这些外部功能可能并不是在所有机器上都存在或适当配置的。

Added in version 1.12 of package scribble-lib.

procedure

(close-eval eval) → (one-of/c "")

  eval : (any/c . -> . any)
关闭一个由 make-base-eval 产生的评估器。 当垃圾回收无法回收一个评估器时,请使用 close-eval (例如,因为它被定义在模块主体中)。

parameter

(scribble-eval-handler)

 → ((any/c . -> . any) boolean? any/c . -> . any)
(scribble-eval-handler handler) → void?
  handler : ((any/c . -> . any) boolean? any/c . -> . any)
一个参数,作为评估的钩子。要使用的评估器被作为参数值的第一个参数提供。 第二个参数是 #t ,如果捕获的是异常(显示异常结果),否则是 #f 。第三个参数是要评估的形式。

parameter

(scribble-exn->string) → (-> (or/c exn? any/c) string?)

(scribble-exn->string handler) → void?
  handler : (-> (or/c exn? any/c) string?)
一个控制异常如何被 interaction 呈现的参数。默认值是
(λ (e)
  (if (exn? e)
      (exn-message e)
      (format "uncaught exception: ~s" e)))

4.4.1 传统的评估

 (require scribble/eval) package: scribble-lib
scribble/eval 库为 scribble/example 的功能提供了一个旧的接口。 应该使用 scribble/example 库来代替。

除了下面列出的形式之外, scribble/eval 还重新输出了 scribble/example 的几个函数: make-base-eval make-base-eval-factory, make-eval-factory, make-log-based-eval, close-eval 和 scribble-eval-handler 。

syntax

(interaction maybe-options datum ...)

 
maybe-options = maybe-eval
  | maybe-escape
  | maybe-no-errors
     
maybe-eval = 
  | #:eval eval-expr
     
maybe-escape = 
  | #:escape escape-id
     
maybe-no-errors = 
  | #:no-errors? no-errors?-expr
就像 scribble/example 中的 examples ,除了

  • “Examples:” 标签总是被抑制,

  • 在评估 datum 时产生的异常总是被显示为错误,除非 #:no-errors? 被指定为真值;以及

  • #:once 选项从不隐式使用。

Changed in version 1.14 of package scribble-lib: 增加了 #:no-errors?, eval:no-prompt 和 eval:error, 并修改了 code:line 以支持多个 datum 。

syntax

(interaction0 maybe-options datum ...)

像 interaction 一样,但没有通过 nested 来嵌入代码。

使用 #:no-indent 的 examples 来代替。

syntax

(interaction/no-prompt maybe-eval maybe-escape datum)

类似于 interaction ,但不对每个 datum 进行提示。

使用 #:no-prompt 的 examples 来代替。

syntax

(interaction-eval maybe-eval datum)

像 interaction 一样,评估 quote 形式的 datum,但返回空字符串,不捕获异常(所以 eval:error 没有作用)。

使用带有 #:hidden 的 examples 来代替。

syntax

(interaction-eval-show maybe-eval datum)

像 interaction-eval 一样,但产生一个代表评价结果的打印形式的元素。

使用 #:result-only 的 examples 来代替。

syntax

(racketblock+eval maybe-eval maybe-escape datum ...)

结合了 racketblock 和 interaction-eval 。

使用 #:no-result 的 examples 来代替。

syntax

(racketblock0+eval maybe-eval maybe-escape datum ...)

结合 racketblock0 和 interaction-eval 。

使用 #:no-result 和 #:no-indent 的 examples 来代替。

syntax

(racketmod+eval maybe-eval maybe-escape name datum ...)

结合 racketmod 和 interaction-eval 。

使用 #:lang 的 examples 来代替。

syntax

(def+int maybe-options defn-datum expr-datum ...)

类似于 interaction, 除了 defn-datum 的排版与 racketblock 一样(即没有提示), 并且在 expr-datum 前插入一行空格。

syntax

(defs+int maybe-options (defn-datum ...) expr-datum ...)

像 def+int 一样,但用于多个前导定义。

在定义上使用带有 eval:no-prompt 包装器的 examples ,而不是。

syntax

(examples maybe-options datum ...)

像 interaction 一样,但前面有一个 “Examples:” 的标签。

使用 scribble/example 中的 examples ,而不是。

syntax

(examples* label-expr maybe-options datum ...)

像 examples 一样,但使用 label-expr 的结果来代替默认的 “Examples:” 标签。

使用来自 scribble/example 的 examples ,并使用 #:label 选项,代替。

syntax

(defexamples maybe-options datum ...)

像 examples 一样, 但在 datum 中使用 define 或 define-struct 的每个定义都是无提示排版, 并且在它后面有一行空格。

在定义上使用带有 eval:no-prompt 包装器的 examples ,而不是。

syntax

(defexamples* label-expr maybe-options datum ...)

像 defexamples 一样,但使用 label-expr 的结果来代替默认的 “Examples:” 标签。

使用带有 #:label 选项和 eval:no-prompt 包装器的 examples 来代替定义。

procedure

(as-examples b) → block?

  b : block?
(as-examples label b) → block?
  label : (or/c block? content?)
  b : block?
为 b 添加一个 “examples” 标签,使用默认标签或给定的 label 。

syntax

(with-eval-preserve-source-locations expr ...)

默认情况下,本模块提供的评估形式,如 interaction 和 examples, 会丢弃它们评估的表达式的源位置。 在 with-eval-preserve-source-locations 表单中,源位置被保留下来。 这对记录依赖源位置的表单很有用,比如 Redex 的排版宏。

使用带有 #:preserve-source-locations 选项的 examples 来代替。
