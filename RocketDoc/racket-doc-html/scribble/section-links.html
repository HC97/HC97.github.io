4.2.7 链接

也见 链接 。

syntax

(racketlink id #:style style-expr pre-content ...)

(racketlink id pre-content ...)
 
  id : identifier?
  pre-content : pre-content?
一个元素, decode 后的 pre-content 被超链接到 id 的定义上。

syntax

(schemelink id pre-content ...)

racketlink 的兼容性别名。

procedure

(link url    
  pre-content ...    
  [#:underline? underline?    
  #:style style]) → element?
  url : string?
  pre-content : any/c
  underline? : any/c = #t
  style : (or/c style? string? symbol? #f)
   = (if underline? #f "plainlink")
hyperlink 的别名,用于向后兼容。

procedure

(other-manual module-path    
  [#:underline? underline?]) → element?
  module-path : module-path?
  underline? : any/c = #t
other-doc 的别名,用于后向兼容。

procedure

(deftech pre-content    
  ...    
  [#:key key    
  #:normalize? normalize?    
  #:style? style?]) → element?
  pre-content : pre-content?
  key : (or/c string? #f) = #f
  normalize? : any/c = #t
  style? : any/c = #t
为 decode 后的 pre-content 产生一个元素, 同时也定义了一个可以在其他地方用 tech 引用的术语。

当 key 为 #f 时, decode 后的 pre-content 的 content->string 结果被用作引用的 key。 如果 normalize? 为 true,那么键的字符串将被规范化,如下所示:

  • 该字符串是大小写折叠的。

  • 后面的 "ies "被替换成 "y"。

  • 后面的 "s "被删除。

  • 连续的连字符和空白处都被一个空格所取代。

这些规范化步骤有助于支持与定义的形式略有不同的自然语言引用。 例如,"香蕉" 的定义可以用 "banana" 来引用。

如果 style? 为真,那么 defterm 就会被用在 pre-content 上。

procedure

(tech pre-content    
  ...    
  [#:key key    
  #:normalize? normalize?    
  #:doc module-path    
  #:tag-prefixes prefixes    
  #:indirect? indirect?]) → element?
  pre-content : pre-content?
  key : (or/c string? #f) = #f
  normalize? : any/c = #t
  module-path : (or/c module-path? #f) = #f
  prefixes : (or/c (listof string?) #f) = #f
  indirect? : any/c = #f
为 decode 后的 pre-content 产生一个元素,并将其超链接到由 deftech 建立的键的定义上。 如果 key 为 false,则解码后的内容被转换为一个字符串(使用 content->string) 作为 key; 在这两种情况下,如果 normalize? 为 true,则 key 被规范化,与 deftech 的方式相同。 #:doc and #:tag-prefixes 参数支持跨文档和特定章节的引用,就像在 secref 中。 例如:

(tech #:doc '(lib "scribblings/reference/reference.scrbl")  "blame object")

在 Racket 参考 中创建一个指向 blame object 的链接。 如果 indirect? 不是 #f, 链接在HTML中的解析可能会延迟; link-element 见 'indirect-link 。

在默认的样式文件中,由 tech 创建的超链接比大多数超链接要安静一些: HTML 输出中的下划线是灰色的,而不是蓝色的,而且只有当鼠标移到术语上时,术语和下划线才会变成蓝色。

在某些情况下,将一个术语的自然语言用法和适当的链接结合起来,可能需要一些创造性,即使对该术语进行了规范化处理。 例如,如果 "bind "被定义了,但一个句子使用了术语 "binding", 后者可以用 @tech{bind}ing 链接到前者。

Changed in version 1.46 of package scribble-lib: 增加了 #:indirect? 参数。

procedure

(techlink pre-content    
  ...    
  [#:key key    
  #:normalize? normalize?    
  #:doc module-path    
  #:tag-prefixes prefixes    
  #:indirect? indirect?]) → element?
  pre-content : pre-content?
  key : (or/c string? #f) = #f
  normalize? : any/c = #t
  module-path : (or/c module-path? #f) = #f
  prefixes : (or/c (listof string?) #f) = #f
  indirect? : any/c = #f
和 tech 一样,但链接并不安静。 例如,在 HTML 输出中,即使鼠标不在链接上,也会出现超链接下划线。

Changed in version 1.46 of package scribble-lib: 增加了 #:indirect? 参数。
