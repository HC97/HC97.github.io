4.2.1 编排代码

codeblock 和 code 形式(见 #lang-Specified 代码) 对代码进行逐字排版, 根据 DrRacket 使用的语法着色解析器,为代码添加一层颜色。 被解析为标识符的输入被进一步赋予词法上下文,并通过 for-label 导入进行超链接。

racketblock 和 racket 形式(见 Racket 代码) 对 S 表达式代码进行了大致的逐字排版, 但大致是用 syntax 来引用源术语。 被引用的 S 表达式中的标识符通过 for-label 导入进行超链接。

这两种不同的代码排版方法—codeblock 和 code 与 racketblock 和 racket—有不同的优势和劣势:

  • codeblock 和 code 形式适用于非 S 表达式的语法, 它们给作者提供了对输出的更多控制(例如,字面数字 2/4 不被规范化为 1/2)。 codeblock 和 code 还不支持转义到 Scribble 元素模式,而且它们不能根据转义中元素的宽度来调整间距。

  • racketblock 和 racket 形式更有效,允许转义到 Scribble 元素模式。 然而, racketblock 和 racket 形式与 S 表达式的语法相联系, 而且它们是基于一种倾向于规范化源术语的语法表示法(例如,字面数字 2/4 被规范化为 1/2)。

4.2.1.1 #lang-Specified 代码

syntax

(codeblock option ... str-expr ...+)

 
option = #:keep-lang-line? keep-expr
  | #:indent indent-expr
  | #:expand expand-expr
  | #:context context-expr
  | #:line-numbers line-number-expr
  | #:line-number-sep line-number-sep-expr
 
  keep-expr : any/c
  indent-expr : exact-nonnegative-integer?
  expand-expr : (or/c #f (syntax? . -> . syntax?))
  context-expr : (or/c #f syntax?)
  line-number-expr : (or/c #f exact-nonnegative-integer?)
  line-number-sep-expr : exact-nonnegative-integer?
将 str-expr 产生的字符串形成的代码解析为 Racket 模块(大致如此), 并产生一个 block ,通过 nested 'code-inset 的样式将代码嵌入排版。 也请看 typeset-code 。

str-expr 通常应该以 #lang 开始,以确定模块的读者语法, 但产生的 "模块" 不需要扩展或编译,除非是 expand-expr 需要。 如果 expand-expr 被省略或产生 false,那么由 str-expr 形成的输入将被读取, 直到遇到文件结束,否则将从输入中读取单一形式。

当 keep-expr 产生一个真值(默认值)时,输入中的第一行(通常是 #lang) 在排版输出中被保留,否则第一行被删除。 排版代码的缩进量由 indent-expr 指定,默认为 0 。

当 expand-expr 产生 #f 时(这是默认的), 排版代码中的标识符会根据 context-expr 提供的语法对象的词法环境中的 for-label 绑定来着色和链接。 默认的 context-expr 与第一个 str-expr 具有相同的词法环境。 当 line-number-expr 为真时,行号从 line-number-expr开始启用, line-number-sep-expr 控制行号和代码之间的分隔(空格,默认为1)。

当 expand-expr 产生一个程序时,它被用来对解析的程序进行宏扩展,语法着色是基于解析的程序。

例如、

@codeblock|{
  #lang scribble/manual
  @codeblock{
    #lang scribble/manual
    @title{Hello}
  }
}|

产生的排版结果是

#lang scribble/manual
@codeblock{
  #lang scribble/manual
  @title{Hello}
}

syntax

(codeblock0 option ... str-expr ...+)

像 codeblock, 但没有 'code-inset nested 包装器。

syntax

(code option ... str-expr ...+)

 
option = #:lang lang-line-expr
  | #:expand expand-expr
  | #:context context-expr
 
  lang-line-expr : (or/c #f string?)
  expand-expr : (or/c #f (syntax? . -> . syntax?))
  context-expr : (or/c #f syntax?)
像 codeblock, 但产生的是 content 而不是 block 。 字符串内容中不应该出现 #lang 行; 相反,如果需要,应该提供 #:lang (作为一个没有 "#lang " 的字符串), 当提供 #lang 行时,总是从输出中剥离。 另外, str-expr 中的每个换行符都会和周围的所有空白一起被折叠成一个空格。

例如、

This is @code[#:lang "at-exp racket"]|{@bold{Hi}}|'s result:
@bold{Hi}.

产生的排版结果是

This is @bold{Hi}’s result: Hi.

procedure

(typeset-code [#:context context 
  #:expand expand 
  #:indent indent 
  #:keep-lang-line? keep? 
  #:line-numbers line-numbers 
  #:line-number-sep line-number-sep 
  #:block? return-block?] 
  strs ...) 
 → (if return-block? block? element?)
  context : (or/c #f syntax?) = #f
  expand : (or/c #f (syntax? . -> . syntax?)) = #f
  indent : exact-nonnegative-integer? = 2
  keep? : any/c = #t
  line-numbers : (or/c #f exact-nonnegative-integer?) = #f
  line-number-sep : exact-nonnegative-integer? = 1
  return-block? : any/c = #t
  strs : string?
一个基于函数的 codeblock 版本,允许你计算 strs 参数。

与 codeblock 不同的是,默认的 context 参数 (#f) 意味着上下文不被触及, return-block? 参数决定了结果结构。 其他参数的处理方式与 codeblock 相同。

4.2.1.2 Racket 代码

syntax

(racketblock maybe-escape datum ...)

 
maybe-escape = 
  | #:escape escape-id
通过 nested 将 datum 序列排版为 Racket 代码嵌套表,样式为 'code-inset 。 datum 的来源位置决定了生成的布局。比如说、

(racketblock
 (define (loop x)
   (loop (not x))))

产生的输出是

(define (loop x)
  (loop (not x)))

(loop (not x)) 缩进到 define 下,因为这是使用 racketblock 的识别方式。 来源位置的跨度信息被用来保留 #true 与 #t 以及 #false 与 #f; 跨度信息还被启发式地用来在一个不精确的数字的开头加上 #i , 如果它的打印形式比来源短两个字符的话;语法对象属性被用来保留方括号和大括号与小括号; 否则,使用语法对象往往会使 S 表达式元素的形式正常化,比如把 2/4 渲染成 1/2 。 当源位置信息不可用时,例如它被字节码编译的宏丢失时,间距的插入方式与 racket 形式相同(在单行内)。

参见 quote-syntax/keep-srcloc ,用于在模板中保留源位置信息的宏。

在上面的例子中, define 被排版为一个关键词(黑色),并作为超链接指向参考手册中 define 的定义, 因为这个文档是使用 define 的 for-label 绑定(在源文件中)建立的,与参考手册中的定义相匹配。 同样, not 也是参考手册中其定义的超链接。

像其他通过 define-code 定义的形式一样, racketblock 扩展了被绑定为 element transformers 的标识符。

一个 #:escape 子句指定了一个标识符来转义到一个产生 element 的表达式。 默认情况下,转义标识符是 unsyntax 。比如说

(racketblock
  (+ 1 #,(elem (racket x) (subscript "2"))))

产生

(+ 1 x2)

默认为 unsyntax 的 escape-id 可以通过 free-identifier=? 识别,所以绑定可以隐藏转义行为:

(racketblock
  (let ([unsyntax #f])
    (racketblock
      #'(+ 1 #,x))))

RACKETBLOCK 形式的默认转义是 UNSYNTAX 而不是 unsyntax 。

其他一些转义是通过符号来识别的:

  • (code:line datum ...) 排版为 datum 序列 (即没有 code:line 包装)。

  • (code:comment content) 和 content 一样排版,但颜色是注释,前缀为分号。 一个典型的 content 使用 unsyntax Racket-typesetting 模式中逃脱出来, 产生一个字符串,使用 elem 产生一个元素,或者使用 t 产生一个段落:

      (code:comment @#,elem{this is a comment})

    (注意, @#,foo{...} 读作 #,(foo "...")。)

  • (code:comment2 content) 和 code:comment 类似,但使用两个分号作为评论的前缀。

  • (code:comment# content) 和 code:comment 一样,但使用 #; 来作为注释的前缀。

  • (code:contract datum ...) 的排版方式与 datum 的序列一样 (包括其着色),但用分号作为前缀。

  • (code:contract# datum ...) 和 code:contract 一样,但用 #; 来作为合同的前缀。

  • code:blank 排版为一个空白处。

  • (code:hilite datum) 与 datum 相似,但有一个背景高亮。

  • (code:quote datum) 像 (quote datum) 的排版方式相同, 但没有将 quote 渲染为 ' 。

  • _id 排版为 id, 但着色为变量(像 racketvarfont); 这个转义只适用于 _id 没有 for-label 绑定, 并且没有通过 defform 特别着色为子表格非终端,通过 defproc 着色为变量,等等。

另请参见 scribble/comment-reader 。

Changed in version 1.9 of package scribble-lib: 添加了启发式方法,用于在不精确的数字上添加 #i 。

和 racketblock 一样, 但使用了默认的表达式转义 UNSYNTAX 而不是 unsyntax 。

4.2.1.3 保存注释

 #reader scribble/comment-reader package: scribble-lib

作为一个阅读模块, scribble/comment-reader 读取一个包含 ;-based 注释行的 S 表达式, 它用 code:comment 包装注释,以便与 racketblock 这样的表单一起使用。 更确切地说, scribble/comment-reader 扩展了当前的阅读器,以调整对 ; 的解析。

例如,在一个导入了 scribble/manual 的 Scribble 文档中、

  @#reader scribble/comment-reader

   (racketblock

    ;; This is not a pipe

    (make-pipe)

   )

产生了

; This is not a pipe
(make-pipe)

上面需要初始的 @ 来转变为 S 表达式模式, 这样 #reader 就会被识别为一个阅读器声明,而不是字面文本。 另外,这个例子使用了 (racketblock ....) 而不是 @racketblock[....], 因为 @-reader 会在给 scribble/comment-reader 转换注释的机会之前将注释丢在 racketblock 里。

scribble/comment-reader 的实现使用 unsyntax 来排版注释。 当把 scribble/comment-reader 和例如 RACKETBLOCK 一起使用时, unsyntax 不会转义,因为 RACKETBLOCK 使用 UNSYNTAX 为它的转义形式。 你可以用 #:escape-id 为 scribble/comment-reader 声明一个转义标识符。比如说

  @#reader scribble/comment-reader #:escape-id UNSYNTAX

   (RACKETBLOCK

     (define-syntax (m stx)

       (syntax-case stx ()

         [(_ x)

          ;; Well this was silly

          #`(#,x)]))

   )

产生

(define-syntax (m stx)
  (syntax-case stx ()
    [(_ x)
     ; Well this was silly
     #`(#,x)]))
4.2.1.4 代码字体和样式

syntax

(racketmodname datum maybe-indirect)

(racketmodname (unsyntax expr) maybe-indirect)
 
maybe-indirect = 
  | #:indirect
像 racket 一样,但是排版为模块路径, 并且没有对标识符的特殊处理(比如 code:blank 或以 _)开头的标识符)。 如果 datum 是一个标识符,或者 expr 产生一个符号, 那么它将被超链接到由 defmodule 创建的模块路径的定义上。

如果 #:indirect 被指定,那么超链接将被赋予 'indirect-link style property, 这使得超链接在 HTML 中的解析可能会被延迟;参见 'indirect-link 中的 link-element 。

Changed in version 1.21 of package scribble-lib: 禁用 racket 风格的标识符的特殊处理。

syntax

(racketmodlink datum pre-content-expr ...)

像 racketmodname 一样,但把要链接的模块路径和要链接的内容分开。 datum 模块路径总是被链接,即使它不是一个标识符。

procedure

(litchar str ...) → element?

  str : string?
将 str 作为字面文本的代表进行排版。 当你必须谈论文本流中的单个字符时,比如在记录一个阅读器扩展时,就使用这个。

procedure

(racketfont pre-content ...) → element?

  pre-content : pre-content?
与 (tt pre-content ...) 相同, 它对 pre-content 参数中的直接字符串和符号应用 'tt 风格。 要注意的是, pre-content 会像往常一样被 decode , 这使得 racketfont 成为排版字面代码的不良选择。

procedure

(racketplainfont pre-content ...) → element?

  pre-content : pre-content?
将 'tt 样式应用于 pre-content 。 注意 pre-content 会像往常一样被 decode , 使 racketplainfont 成为直接排版字面代码的糟糕选择,但对实现代码格式化功能很有用。

Added in version 1.6 of package scribble-lib.

procedure

(racketvalfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样, 但作为一个值被着色。

procedure

(racketresultfont [#:decode? decode?]    
  pre-content ...) → element?
  decode? : boolean? = #t
  pre-content : pre-content?
像 racketplainfont 一样,但作为REPL结果着色。 当 decode? 为 #f 时,与 racketplainfont 不同, racketresultfont 避免对其参数进行 decode 。

procedure

(racketidfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont,一样,但作为一个标识符被着色。

procedure

(racketvarfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但作为一个变量(即一个参数或被记录的过程中的子表格)着色。

procedure

(racketkeywordfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但作为一个语法形式的名称着色。

procedure

(racketparenfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样，但颜色像圆括号。

procedure

(racketoptionalfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但颜色是可选的。

Added in version 1.36 of package scribble-lib.

procedure

(racketmetafont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但用颜色表示元语法,如反引号或取消引号。

procedure

(racketcommentfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但是用颜色表示为注释。

procedure

(racketerror pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但作为错误信息的文本而着色。

procedure

(racketmodfont pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但作为模块名称着色。

procedure

(racketoutput pre-content ...) → element?

  pre-content : pre-content?
像 racketplainfont 一样,但作为输出的颜色。

procedure

(procedure pre-content ...) → element?

  pre-content : pre-content?
将 decode 后的 pre-content 作为 REPL 结果中的过程名称进行排版 (例如,用打字机字体,带有 #<procedure: 前缀和 > 后缀)。

(var datum)

将 datum 排版为一个标识符,它是被记录的过程中的一个参数或子形式。 通常情况下, defproc 和 defform 安排 racket 在过程的描述中自动格式化这些标识符, 但如果由于某种原因不能工作,则使用 var 。

syntax

(svar datum)

像 var 一样,但是对于表格定义中的子表格非终端。
