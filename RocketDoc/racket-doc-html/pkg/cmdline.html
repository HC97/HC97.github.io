3 使用 raco pkg

raco pkg 命令通过子命令提供软件包管理工具。

3.1 raco pkg install

raco pkg install ‹option› ... ‹pkg-source› ... — 安装指定的 package sources (消除完全重复的 ‹pkg-source›)。 如果给定的 ‹pkg-source› 是 auto-installed (为了满足其他软件包的依赖性), 那么它会被提升为明确安装。

如果没有提供 ‹pkg-source› ,并且没有提供 --clone 标志,那么当前目录将作为一个链接被安装。 更多细节请参见下面的 --link 标志。

如果没有提供 ‹pkg-source› ,并且提供了 --clone 标志, 那么克隆目录的名字将被用作唯一的 ‹pkg-source› 参数。 更多细节见下面的 --clone 标志。

The install 子命令接受以下 ‹option›:

  • --type ‹type› 或 -t ‹type› — 指定软件包源的解释, 其中 ‹type› 是 file, dir, file-url, dir-url, git, git-url, github 或 name 。 git 和 git-url 的区别在于,对于 git 类型, http 或 https URL 会添加一个 .git 后缀,但对于 git-url 类型则不会。 类型通常是为每个 ‹pkg-source› 推断出来的。

  • --name ‹pkg› 或 -n ‹pkg› — 指定软件包的名称, 这只有在提供单一 ‹pkg-source› 时才有意义。 通常情况下,这个名字是为每个 ‹pkg-source› 推断出来的。

  • --checksum ‹checksum› — 指定软件包的校验值,通常只有在提供单个 ‹pkg-source› 时才有意义。 使用 ‹checksum› 取决于 ‹pkg-source›: 对于 Git 或 GitHub 源, ‹checksum› 选择一个校验和; 对于作为源的 package name 、文件路径或远程 URL, ‹checksum› 指定一个预期的校验和; 对于作为源的目录路径(包括没有 ".CHECKSUM" 文件的远程目录 URL), ‹checksum› 指派一个校验和。

  • --deps ‹behavior› — 选择依赖关系的行为,其中 ‹behavior› 是以下之一

    • fail — 如果依赖关系未被卸载或版本要求未被满足,则取消安装。 这种行为是非 interactive mode 的默认值。

    • force — 尽管缺少依赖关系或版本要求,仍然安装软件包。强制安装可能会使软件包内容处于不一致的状态。 即使没有指定 --ignore-implies , 通过 implies 或 update-implies (见 包的元数据) 暗示的软件包也不会被更新。

    • search-ask — 通过配置的 package catalogs 寻找依赖关系(当卸载时)或更新(当版本要求未满足时), 但会询问用户是否应该安装或更新软件包。 这种行为在 interactive mode 下是默认的。

    • search-auto — 和 search-ask 一样,但不询问安装或更新依赖关系的权限。

    除非指定 --ignore-implies ,否则当一个软件包被更新或安装时, 它通过 implies 或 update-implies (见 包的元数据) 暗示的任何软件包都会自动更新, 与 fail, search-ask 或 search-auto 无关,但暗示的依赖关系不会因 force 行为被更新。

  • --auto — 简称为 --deps search-auto 。

  • --update-deps — 与 search-ask 或 search-auto 依赖关系行为一起, 检查已经安装的依赖关系是否有更新 (即使没有被版本要求、implies 或 update-implies) 强制更新), 当有更新时询问或自动更新软件包。

    除非指定了 --ignore-implies 或 --deps force , 否则当一个软件包被更新或安装时, 它所暗示的 implies 或 update-implies (见 包的元数据) 会自动更新, 与通过 --update-deps 要求的行为无关。

  • --ignore-implies — 禁止对已安装或已更新软件包的 implies 或 update-implies (见 包的元数据) 中所列的依赖关系进行特殊处理。

  • --link — 暗示 --type dir 并将现有的目录链接为已安装的软件包,而不是将目录的内容复制到安装中。 除非指定了 --copy ,或者目录名称是由目录报告的,而不是直接指定的,否则目录 package sources 默认被视为链接。

    软件包在安装时被识别为 single-collection package 或 multi-collection package , 即使 "info.rkt" 中的 collection 定义被改变, 该分类也不会改变(即,必须删除软件包并重新安装,该改变才能生效)。

  • --static-link — 暗示了 --link, 并且还表示对于每个实现 multi-collection package 的给定目录,给定目录的子目录将不会改变。

  • --copy — 禁用对目录 package sources 作为链接的默认处理,而是像对待其他源一样对待它们: 包的内容被复制到安装。

  • --clone ‹dir› — 一个 Git 或 GitHub package source 被克隆为 ‹dir› , 并作为包的实现进行本地链接。 参见 将 Git 签出链接为一个克隆文件 。

    如果没有提供 ‹pkg-source› , 那么 ‹dir› 的最后一个路径元素将被用作软件包名称并作为 ‹pkg-source› 参数。

    多个 ‹pkg-source› 参数只有在它们都指定同一个 Git 仓库(但进入仓库的路径不同)时才有意义。 --clone 标志意味着 --type ， 即每个 ‹pkg-source› 必须是一个 Git 或 GitHub 的 package source 或 package name, 其中 package name 必须被 package catalog 映射到 Git 或 GitHub 的 package source 。

  • --source — 在安装前剥离软件包的构建元素,并暗示 --copy 。参见 源码、二进制和内置包 。

  • --binary — 在安装前剥离软件包的源码元素,并暗示 --copy 。 参见 源码、二进制和内置包 。

  • --binary-lib — 在安装前剥离软件包的源代码和文档元素,并暗示 --copy 。 参见 源码、二进制和内置包 。

  • --scope ‹scope› — 选择安装 package scope ,其中 ‹scope› 是下列之一

    • installation — 为 Racket 安装的所有用户安装软件包,而不是针对特定用户。

    • user — 为当前用户和当前安装的名称/版本安装软件包。

    默认的软件包范围通常是 user, 但可以用 raco pkg config --set default-scope ‹scope› 进行配置。 默认的安装名称通常是 Racket 版本,但可以用 raco pkg config --set name ‹name› 来配置。

  • -i 或 --installation — --scope installation 的缩写。

  • -u 或 --user — --scope user 的缩写。

  • --scope-dir ‹dir› — 选择 ‹dir› 作为 package scope 。

  • --catalog ‹catalog› — 使用 ‹catalog› 来代替当前配置的 package catalogs 。 这个标志可以被多次提供。目录会按照所提供的顺序进行尝试。

  • --skip-installed — 忽略任何名称与已经安装的软件包相对应的 ‹pkg-source› , 除了推广 auto-installed 。

  • --pkgs — 在没有提供 ‹pkg-source› 的情况下,禁用当前目录的默认安装。

  • --all-platforms — 考虑独立于当前平台的软件包依赖(而不是过滤对当前平台以外的依赖)。

  • --force — 忽略模块冲突,包括由于在多个范围内安装一个软件包而产生的冲突。 强制安装可能会使软件包内容处于不一致的状态。

  • --ignore-checksums — 忽略验证软件包 checksums 的错误(不安全)。

  • --strict-doc-conflicts — 当文档目录名称与其他软件包冲突时,拒绝在用户 package scope 内安装。 "冲突 "的文档名称通常允许在用户 package scope, 中安装,但严格检查确保在其他 package scopes 中安装成功。

  • --no-cache — 禁止使用下载缓存。

  • --multi-clone ‹mode› — 指定处理来自同一个 Git 仓库但以不同的克隆链接模式或不同的克隆目录安装的包。 (参见 使用 Git 开发包 )。 有以下几种模式可供选择:

    • convert — 将非克隆链接的包(无论是新安装的还是以前安装的)转换为克隆链接的包, 假设被克隆链接的包都使用同一个克隆目录。 如果克隆链接的软件包目前使用不同的克隆目录,则安装失败。

    • ask — 在软件包可以转换的情况下,询问用户是否要转换或允许不同的克隆链接模式或克隆目录。 如果转换不是一个选项,则安装失败。 这种克隆处理模式在 interactive mode 下是默认的。

    • fail — 报告一个错误并取消安装(在非 interactive mode 下的默认值)。

    • force — 允许软件包有不同的克隆链接模式或克隆目录。

  • --pull ‹mode› — 指定在安装或更新时提交合并到克隆链接包的方式(见 使用 Git 开发包)。 有以下几种模式可供选择:

    • ff-only — 使用 --ff-only 合并提交,如果快进式合并失败,则安装失败。

    • try — 和 ff-only 一样,但如果快进失败,仓库签出会保持原样,并继续安装。

    • rebase — 使用 git rebase 而不是 git merge 来合并提交。

  • --dry-run — 阻止对当前安装的更改。 所有的安装和更新工作都被分阶段检查,但最后的安装步骤被跳过。

  • --no-setup — 安装后不运行 raco setup 。 如果环境变量 PLT_PKG_NOSETUP 被设置为任何非空值,也会出现这种行为。

  • --no-docs 或 -D — 安装后不在设置过程中呈现文档。这个标志对 --no-setup 没有影响。

  • --recompile-only — 限制 raco setup 最多只能从独立于机器的形式重新编译一个模块, 如果需要从源代码编译,则报告错误。 这种行为作为安装已建软件包时的理智检查是很有用的(以确保它们是正确构建的),但如果报告编译错误,那将是在软件包安装后。

  • --jobs ‹n› 或 -j ‹n› — 安装和运行 raco setup ,有 ‹n› 个并行作业。

  • --batch — 禁用 interactive mode, 抑制对用户的潜在提示(例如,关于软件包依赖性或克隆共享)。

  • --no-trash — 避免将更新或删除的软件包移到垃圾文件夹中。

  • --fail-fast — 一旦遇到任何错误,就立即中断 raco setup 。

3.2 raco pkg update

raco pkg update ‹option› ... ‹pkg-source› ... — 检查指定的软件包名称是否有 package updates ,或用给定的来源替换现有的软件包安装。 如果一个更新或替换不能被安装(例如,它与另一个已安装的软件包冲突),那么这个命令就会失败, 而不会安装任何 ‹pkg-source› (或它们的依赖)。

对 ‹pkg-source› 的处理取决于它的解析方式:

  • 当 ‹pkg-source› 解析为一个 package name 时,那么这个被命名的包一定已经安装了,并且会被检查是否有更新。 --lookup 和 --clone 标记改变了对 ‹pkg-source› 的这种解释。

  • 如果 ‹pkg-source› 解析为一个目录 package source, 并且如果命名的软件包是作为一个 Git repository clone 的链接安装的,那么这个 clone 将被检查更新。 --link, --static-link 和 --copy 标志改变了对 ‹pkg-source› 的这种解释。

  • 否则, ‹pkg-source› 会指定一个 package source ,以替换当前安装的指定软件包。

如果没有指定 ‹pkg-source›, --all 或 -a 标志、或 --clone 标志, 并且如果当前目录在一个软件包内,那么包围的软件包将被更新。 如果没有指定 ‹pkg-source› ,但提供了 --clone , 那么克隆目录的名字将被用作唯一的 ‹pkg-source› 参数。

如果没有指定 package scope ,那么将从给定的 ‹pkg-source› 中推断出范围。

update 子命令接受以下 ‹option›:

  • --all 或 -a — 更新所有的软件包,如果在参数列表中没有给出软件包。

  • --lookup — 导致一个作为 package name 的 ‹pkg-source› 被用作在目录中查找的替代物, 而不是可能有更新的当前来源的已安装软件包的名称。 (如果命名的软件包是通过软件包名称安装的,那么实际上没有区别,除非使用不同的目录)。

    默认情况下,如果 ‹pkg-source› 指的是一个当前被链接为 Git repository clone 的软件包, 那么用一个目录指定的来源来替换安装,其效果就是删除克隆链接。 然而, --lookup 标志可以与 --clone 标志结合使用(假设目录将软件包映射到 Git 仓库),这样得到的安装是一个链接的仓库克隆。

  • --type ‹type› 或 -t ‹type› — 与 raco pkg install 相同。

  • --name ‹pkg› 或 -n ‹pkg› — 与 raco pkg install 相同。

  • --checksum ‹checksum› — 与 raco pkg install 相同。

  • --deps ‹behavior› — 与 raco pkg install 相同。

  • --auto — Shorthand for --deps search-auto 加上 --update-deps 的简写。

  • --update-deps — 与 raco pkg install 相同,但仅在 raco pkg update 时由 --auto 隐含。

  • --ignore-implies — 与 raco pkg install 相同。

  • --link — 与 raco pkg install 相同,但目录包源只有在不对应链接或 Git 仓库克隆的情况下才会被默认为链接。

  • --static-link — 与 raco pkg install 相同。

  • --copy — 与 raco pkg install 相同。

  • --clone ‹dir› — 与 raco pkg install 相同, 只是作为 package name 的 ‹pkg-source› 被视为已安装包的名称(除非 --lookup 被指定)。 在这种情况下,该软件包必须是当前从 Git 或 GitHub 源安装的 — 可能是按照目录的指示–并且该源被用于克隆(取代现有软件包的安装)。

    要将一个克隆链接的软件包转换为正常的安装,可以使用 raco pkg update 的 --lookup 标志, 或者使用一个不是软件包名称的 package source 来替换。

  • --unclone — --lookup 的别名,它(没有 --clone) 的效果是用正常的软件包安装替换到克隆的版本库的链接。

  • --binary — 与 raco pkg install 相同。

  • --source — 与 raco pkg install相同。

  • --scope ‹scope› — 选择一个 package scope, 与 raco pkg install 相同。

  • -i or --installation — --scope installation 安装的速记。

  • -u or --user — --scope user 用户的简写。

  • --scope-dir ‹dir› — 选择 ‹dir› 作为 package scope ,与 raco pkg install 相同。

  • --catalog ‹catalog› — 与 raco pkg install 相同。

  • --skip-uninstalled — 忽略任何与已安装软件包不对应的 ‹pkg-source› 。

  • --all-platforms — 与 raco pkg install 相同

  • --force — 与 raco pkg install 相同

  • --ignore-checksums — 与 raco pkg install 相同

  • --strict-doc-conflicts — 与 raco pkg install 相同

  • --no-cache — 与 raco pkg install 相同

  • --multi-clone ‹mode› — 与 raco pkg install 相同, 只是当指定了 --lookup 并且没有指定 --clone 时, 就会从 clone 链接转换为非 clone 链接–但只针对即时命令行参数所暗示的共享差异与现有软件包安装进行对比。

  • --pull ‹mode› — 与 raco pkg install 相同

  • --dry-run — 与 raco pkg install 相同

  • --no-setup — 与 raco pkg install 相同

  • --no-docs or -D — 与 raco pkg install 相同

  • --recompile-only — 与 raco pkg install 相同

  • --jobs ‹n› 或 -j ‹n› — 与 raco pkg install 相同

  • --batch — 与 raco pkg install 相同

  • --no-trash — 与 raco pkg install 相同。

3.3 raco pkg remove

raco pkg remove ‹option› ... ‹pkg› ... — 试图删除指定的软件包。 默认情况下,如果一个软件包是另一个没有列出的软件包的依赖关系,这个命令就会失败,不会删除任何 ‹pkg› 。

如果没有指定 package scope ,范围将从给定的 ‹pkg› 中推断出来。

remove 子命令接受以下 ‹option›:

  • --demote — “Removes” 明确安装的软件包,将其降级为 auto-installed (保留自动安装的软件包)。 与 --auto 结合,删除没有依赖关系的软件包。

  • --force — 移除软件包时忽略依赖关系。

  • --auto — 除了删除每个 ‹pkg› 之外,还删除 auto-installed 的软件包 (即通过 search-auto 或 search-ask 依赖行为安装的,或通过 --demote 降级的), 这些软件包不再需要任何明确安装的软件包。

  • --scope ‹scope› — 选择一个 package scope ,与 raco pkg install 相同。

  • -i 或 --installation — --scope installation 的速记。

  • -u 或 --user — --scope user 的速记。

  • --scope-dir ‹dir› — 选择 ‹dir› 作为 package scope, 与 raco pkg install 相同。

  • --dry-run — 与 raco pkg install 相同。

  • --no-setup — 与 raco pkg install 相同。

  • --no-docs 或 -D — 与 raco pkg install 相同。

  • --recompile-only — 与 raco pkg install 相同。

  • --jobs ‹n› 或 -j ‹n› — 与 raco pkg install 相同。

  • --batch — 与 raco pkg install 相同。

  • --no-trash — 与 raco pkg install 相同。

Changed in version 6.1.1.5: 增加了 --batch 标志。
Changed in version 6.1.1.6: 增加了 --no-trash 标志。
Changed in version 6.4.0.14: 增加了 --dry-run 标志。
Changed in version 7.2.0.8: 增加了 --recompile-only 标志。
Changed in version 7.4.0.4: 增加了 --no-docs, -D 标志。

3.4 raco pkg new

raco pkg new ‹pkg› — 用新软件包的存根填充一个目录,其中 ‹pkg› 是新软件包的名字。 如果 ‹pkg› 已经作为一个目录存在于当前目录中,则不会创建新包。

Added in version 6.1.1.5.

3.5 raco pkg show

raco pkg show ‹option› ... ‹pkg› ... — 打印当前安装的软件包的信息。

如果指定了 ‹pkg› ,那么就只显示这些软件包。 默认情况下,所有 package scopes 的包都会被显示出来,但只显示没有标记为 auto-installed 的包。 如果一个软件包被明确指定,即使它被标记为自动安装,也会被显示。 除非指定了 -l 或 --long ,否则输出被大致限制在80列或由 COLUMNS 环境变量指定的列数。 除非指定了 --full-checksum ,否则校验和会被缩减为8个字符。

show 子命令接受下列 ‹option›:

  • -a 或 --all — 在列表中包括 auto-installed 软件包。

  • -l 或 --long — 显示完整的列,而不是缩略到一个宽度,并对一些列使用更常规的(但不太容易被人读懂的)格式。

  • --rx — 将 ‹pkg› 视为显示特定软件包的正则表达式。

  • --full-checksum — 打印完整的而不是简略的校验值。

  • -d 或 --dir — 在输出中增加一列,显示软件包的安装目录。

  • --scope ‹scope› — 只显示 ‹scope› 中的软件包,它是下列之一

    • installation — 只显示安装范围的软件包。

    • user — 只显示当前安装的名称/版本或用 --version 或 -v 指定的名称/版本的用户特定软件包。

    默认是显示所有 package scopes 的软件包。

  • -i 或 --installation — --scope installation 的速记。

  • -u 或 --user — --scope user 的速记。

  • --scope-dir ‹dir› — 只显示安装在 ‹dir› 的软件包。

  • --version ‹vers› 或 -v ‹vers› — 只显示安装名称/版本 ‹vers› 的用户特定软件包。

Changed in version 6.1.1.5: 增加了 -l/--long 和 COLUMNS 支持。
Changed in version 6.1.1.6: 增加了明确的 ‹pkg› 以及 --rx 和 --full-sha 。

3.6 raco pkg migrate

raco pkg migrate ‹option› ... ‹from-version› — 为 ‹from-version› 安装先前在 user package scope 中安装的软件包, 其中 ‹from-version› 是一个安装名称/版本。

migrate 子命令接受以下 ‹option›:

  • --deps ‹behavior› — 与 raco pkg install 相同,除了 search-auto 是默认的。

  • --source — 与 raco pkg install 相同。

  • --binary — 与 raco pkg install 相同。

  • --binary-lib — 与 raco pkg install 相同。

  • --scope ‹scope› — 与 raco pkg install 相同。

  • -i 或 --installation — --scope installation 的速记。

  • -u 或 --user — --scope user 的速记。

  • --scope-dir ‹dir› — 选择 ‹dir› 作为 package scope 。

  • --catalog ‹catalog› — 与 raco pkg install 相同。

  • --all-platforms — 与 raco pkg install 相同。

  • --force — 与 raco pkg install 相同。

  • --ignore-checksums — 与 raco pkg install 相同。

  • --strict-doc-conflicts — 与 raco pkg install 相同。

  • --no-cache — 与 raco pkg install 相同。

  • --dry-run — 与 raco pkg install 相同。

  • --no-setup — 与 raco pkg install 相同。

  • --no-docs 或 -D — 与 raco pkg install 相同。

  • --recompile-only — 与 raco pkg install 相同。

  • --jobs ‹n› 或 -j ‹n› — 与 raco pkg install 相同。

Changed in version 6.4.0.14: 增加了 --dry-run 标志。
Changed in version 7.2.0.8: 增加了 --recompile-only 标志。
Changed in version 7.4.0.4: 增加了 --no-docs, -D 标志。
Changed in version 7.6.0.14: 允许多个 --catalog 标志。

3.7 raco pkg create

raco pkg create ‹option› ... ‹directory-or-package› — 将一个软件包捆绑到一个存档中。对于直接从 Git 仓库或其他非存档格式提供的软件包,不需要捆绑。 create 子命令可以从目录(默认)或已安装的软件包中创建一个归档。 它也可以调整归档的内容,使其只包括源码、只包括编译过的字节码和渲染过的文档, 或者两者都包括–但软件包通常是以源码形式提供的,并由自动服务转换为二进制形式,而不是由软件包作者提供。

create 子命令接受以下 ‹option›:

  • --from-dir — 将 ‹directory-or-package› 视为一个目录路径;这是默认模式。

  • --from-install — 将 ‹directory-or-package› 视为一个已安装软件包的名称(而不是目录)。

  • --format ‹format› — 指定存档格式。 允许的 ‹format› 有: zip (默认), tgz 和 plt 。 如果 --manifest 不存在,必须指定这个选项。

  • --manifest — 为一个目录创建一个清单文件,而不是一个存档。

  • --as-is — 按原样捆绑软件包目录的所有内容,不对源代码、编译文件或版本库元素进行过滤。

  • --source — 只捆绑软件包目录下的源文件;见 源码、二进制和内置包 。

  • --binary — 在软件包目录中捆绑编译过的字节码和渲染过的文档;见 源码、二进制和内置包 。

  • --binary-lib — 在软件包目录下只捆绑已编译的字节码;见 源码、二进制和内置包 。

  • --built — 将编译好的源码、字节码和渲染好的文档捆绑在软件包目录下,过滤存储库元素;见 源码、二进制和内置包 。

  • --dest ‹dest-dir› — 将生成的包写到 ‹dest-dir› 。

3.8 raco pkg config

raco pkg config ‹option› ... [‹key›] ‹val› ... — 查看和修改软件包管理器的配置。 如果没有提供 ‹key› ,就会显示所有认可的键的值。 只有在使用 --set 时才允许 ‹val› 参数,在这种情况下, ‹val› 被用作 ‹key› 的新值。

config 子命令接受以下 ‹option›:

  • --set — 设置一个选项,而不是打印它。

  • --scope ‹scope› — 选择一个 package scope, 与 raco pkg install 相同。 在 installation 范围内设置的配置值会作为 user 范围内的默认值。

  • -i 或 --installation — --scope installation 的速记。

  • -u 或 --user — --scope user 的速记。

  • --scope-dir ‹dir› — 与 raco pkg install 相同。

有效的 ‹key› 和对应的 ‹val› 是:

  • name — 安装名称的字符串,它被 user package scope 使用,默认为Racket版本。

  • catalogs — 一个 package catalogs 的 URLs 列表。 一个空字符串 ‹val› 会被默认配置的目录序列所取代。 如果 ‹val› 不是以字母开头,后面是 :// ,则被视为相对于配置目录的路径(由 find-config-dir 报告)。

  • default-scope — installation 或 user 。 这个键在 user 范围的值(可能默认为 installation 范围)是 raco pkg 命令的默认 package scope , 对于这些命令,范围不是从一组给定的软件包名称推断出来的(即使是 raco pkg config, 这也是一致的,但有可能造成混淆)。

  • download-cache-dir — 保存下载软件包副本的目录,用于在再次请求相同的 URL 和校验码组合时避免重新下载。 默认的缓存目录是针对用户的(但不是针对 Racket 版本或安装名称的)。

  • download-cache-max-files — 在下载缓存目录中保存文件的数量限制。

  • download-cache-max-bytes — 对保存在下载缓存目录中的文件的总大小的限制。

  • doc-open-url — 用来代替本地文件系统路径查看(或至少是搜索)文档的 URL; 一个空字符串,即默认值,禁用URL,以便使用本地文件系统。 这个键只能在 installation 范围内设置。

  • git-checkout-credentials — 一个以格式说明开始的列表(目前只支持 'basic ), 后面是 ‹username›:‹password› 形式的 git 凭证, 在使用 HTTP 或 HTTPS 协议下载 git 源的包时,会尝试使用这些凭证。 这些凭证目前被 unencrypted 地存储在文件系统中。

  • trash-max-packages — 当软件包被删除或更新时,保留在垃圾文件夹中的软件包实现的数量限制。

  • trash-max-seconds — 自软件包被删除或更新后,其实现被保留在垃圾文件夹中的时间限制。 只有当另一个软件包可能被添加到垃圾箱中或使用 raco pkg empty-trash 时,软件包的实现才会从垃圾箱中删除。

  • network-retries — 重试因连接错误而失败的网络通信的次数。

Changed in version 6.1.1.6: 添加了 trash-max-packages 和 trash-max-seconds 。
Changed in version 6.3: 添加了 network-retries 。
Changed in version 6.6.0.5: 添加了 git-checkout-credentials 。

3.9 raco pkg catalog-show

raco pkg catalog-show ‹option› ... ‹package-name› ... — 查询 package catalogs (不一定要安装),并显示该软件包的目录信息,比如它的源 URL 和校验值。

catalog-show 子命令接受以下 ‹option›:

  • --all — 显示所有可用软件包的信息。当使用这个标志时,不要提供 ‹package-name› 。

  • --only-names — 只显示软件包名称。这个选项主要是和 --all 一起使用, 但是当提供 ‹package-name› 时,会查询目录以确保该包是可用的。

  • --modules — 显示一个软件包所实现的模块。

  • --catalog ‹catalog› — 查询 ‹catalog› ,而不是当前配置的 package catalogs。 这个标志可以被多次提供。目录会按照所提供的顺序被尝试。

  • --version ‹version› 或 -v ‹version› — 查询目录以获得特定于 ‹version› 的结果, 而不是安装的Racket版本。

Changed in version 7.6.0.14: 允许多个 --catalog 标志。

3.10 raco pkg catalog-copy

raco pkg catalog-copy ‹option› ... ‹src-catalog› ... ‹dest-catalog› — 将 ‹src-catalog› 命名的 package catalog 中的信息复制到本地数据库或目录 ‹dest-catalog›, 该目录可以作为新的 package catalog 使用。

‹src-catalog› 可以是远程的,也可以是本地的, 而 ‹dest-catalog› 必须是本地的(即,从路径推断出的目录路径或 SQLite 数据库路径)。 如果 ‹src-catalog› 或 ‹dest-catalog› 不是以 URL 方案开始的,它将被视为一个文件系统路径。 来自多个 ‹src-catalog› 的信息被合并,早期 ‹src-catalog› 的信息优先于后期 ‹src-catalog› 的信息。

catalog-copy 子命令接受下列 ‹option›:

  • --from-config — -将当前配置的 package catalogs 添加到 ‹src-catalog› 列表的最后。

  • --force — 如果 ‹dest-catalog› 已经存在,则替换它。

  • --merge — 如果 ‹dest-catalog›已经存在,则将其添加到该目录。 默认情况下, ‹dest-catalog› 中已有的信息优先于新信息。

  • --override — 改变合并方式,使新信息优先于 ‹dest-catalog› 中已有的信息。

  • --relative — 在可能的情况下,以相对路径的形式将包的来源写到 ‹dest-catalog› 。

  • --version ‹version› 或 -v ‹version› — 复制特定于 ‹version› 的目录结果 (对于有区别的目录),而不是安装的Racket版本。

3.11 raco pkg catalog-archive

raco pkg catalog-archive ‹option› ... ‹dest-dir› ‹src-catalog› ... — 将由 ‹src-catalog› 命名的 package catalog 中的信息复制到 ‹dest-dir› 中的 "catalog" 目录, 同时将所有软件包源复制到 ‹dest-dir› 中的 "pkgs" 目录。

包源被下载并根据需要重新打包,所以所有的包都以 ".zip" 档案的形式写到 "pkgs" 目录中。 这种转换可能会改变每个归档软件包的校验和。

catalog-archive 子命令接受以下 ‹option›:

  • --from-config — 将当前配置的 package catalogs 添加到 ‹src-catalog› 列表的最后。

  • --state ‹state-database› — 为了实现增量更新,读取和写入数据库 ‹state-database›, 其后缀必须是 ".sqlite", 作为 ‹dest-dir› 的当前状态。

  • --relative — 以相对路径的形式将软件包源写入 ‹dest-catalog› 。

  • --version ‹version› 或 -v ‹version› — 复制特定于 ‹version› 的目录结果(对于有区别的目录),而不是安装的Racket版本。

  • --pkg-fail ‹mode› — 决定对单个软件包的故障处理,例如当 ‹src-catalog› 包含一个坏的软件包源时。 有以下 ‹mode› 可供选择:

    • fail (默认) — 归档停止并失败;

    • skip — 软件包被跳过并从归档目录中省略;或者

    • continue — 和 skip 一样,但如果有软件包被跳过, raco pkg catalog-archive 会以 5 的状态码退出。

  • --include ‹pkg› — 可以多次指定。 如果 --include 至少被指定一次,那么归档和生成的目录只包括用 --include 指定的 ‹pkg› , 如果 --include-deps 被指定,再加上每个 ‹pkg› 的依赖关系,再加上通过 --exclude 排除的软件包。

  • --include-deps — 修改 --includes ‹pkg› 标志以暗示 ‹pkg› 的所有依赖项。

  • --include-deps-platform ‹sys› ‹subpath› — 修改 --include-deps , 只暗示与平台 ‹sys› 相匹配的依赖关系,这应该是 (system-type) 的可能结果, 而 ‹subpath› 应该是 (system-library-subpath #f) 的可能结果。

  • --exclude ‹pkg› — 可以多次指定。将 ‹pkg› 从存档和生成的目录中的软件包集合中移除。 如果 --include 被用于相同的 ‹pkg›, 那么 --exclude 会优先。 如果 --include 与 --include-deps 一起用于 ‹pkg› 或依赖于 ‹pkg› 的软件包, 那么 --exclude 会停止考虑 ‹pkg› 的依赖关系(但不一定排除依赖关系,因为它们可能是被包含的软件包的依赖关系)。

  • --fast-file-copy — 当本地文件系统上有软件包文件时,直接从 ‹src-catalog› 中复制,而不是解压和重新打包。

Added in version 6.0.17.
Changed in version 7.7.0.1: 添加了 --include, --include-deps, --include-deps-platform, --exclude 和 --fast-file-copy 。

3.12 raco pkg archive

raco pkg archive ‹option› ... ‹dest-dir› ‹pkg› ... — 将由 ‹pkgs› 命名的已安装软件包的信息复制到 ‹dest-dir› 的 "catalog" 目录下, 同时将所有软件包的源代码复制到 ‹dest-dir› 的 "pkgs" 目录下。

包源被复制并根据需要重新打包,所以所有的包都以 ".zip" 档案的形式写到 "pkgs" 目录。 这种转换可能会改变每个归档软件包的校验和。

archive 子命令接受以下 ‹option›:

  • --include-deps — 在结果目录中包括指定软件包的依赖关系。

  • --exclude ‹pkg› — 从结果目录中省略指定的 ‹pkg› 。 如果指定了 --include-deps ,这也会使 ‹pkg› 的依赖关系被省略。这个标志可以被多次提供。

  • --relative — 以相对路径的形式将软件包源写入 ‹dest-catalog› 。

Added in version 6.1.0.8.

3.13 raco pkg empty-trash

raco pkg empty-trash ‹option› ... — 移除或列出先前被移除或更新的软件包实现,这些实现目前在指定的 package scope 的垃圾目录中。 trash-max-packages 和 trash-max-seconds 配置键(见 raco pkg config) 控制了有多少包被保存在垃圾箱目录中, 以及保存多长时间。

empty-trash 子命令接受以下 ‹option›:

  • --scope ‹scope› — 选择一个 package scope, 与 raco pkg install 相同。

  • -i or --installation — --scope installation 的速记。

  • -u or --user — --scope user 的速记。

  • --scope-dir ‹dir› — 与 raco pkg install 相同。

  • --list 或 -l — 显示垃圾目录路径及其内容,而不是删除当前内容。

Added in version 6.1.1.6.
