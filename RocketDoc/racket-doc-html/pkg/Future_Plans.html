11 Future Plans

11.1 短期计划

本节列出了软件包管理器的一些短期计划。 这些都很重要,但并没有阻止它的发布。 当这些完成后,软件包管理器将被视为脱离了测试阶段。

  • 官方目录服务器将把软件包分为三类: ring-0, ring-1 和 ring-2 。 这些类别的定义是:

    • ring-2 — 没有限制。

    • ring-1 — 不得与 ring-1 或 ring-0 中的任何软件包冲突。

    • ring-0 — 必须不与 ring-1 或 ring-0 中的任何包冲突。必须有文档和测试。 作者必须对针对 Racket 的变化而进行的退步修复做出反应,等等。

    这些类别将由 PLT 来策划。

    我们的目标是让所有的软件包都在 ring-1 中, ring-2 作为一个临时的地方, 而策展人将与有冲突的软件包的作者一起工作,以确定模块应该如何重命名以保持统一。

    然而,在策展工作完成之前,每个软件包都会根据其冲突情况被自动放置在 ring-2 或 ring-1 中, 并优先考虑旧的软件包。 (例如,如果一个新的软件包 B 与一个旧的软件包 A 冲突,那么 A 将被放在 ring-1 ,而 B 将被放在 ring-2) 。 然而,在策展过程中,不一定是老的包有优先权。 (例如， tic-tac-toe 可能不应该提供 "data/matrix.rkt", 但它可以被剥离成另一个同时被 tic-tac-toe 和 factory-optimize 使用的包)。

    相比之下, ring-0 类别将是一个特殊类别,作者可以申请加入。 入选需要进行代码审计,并意味着 PLT 的 "批准印章"。 在未来,这个类别的软件包将有更多的好处,比如在 DrDr 上进行自动回归测试,在发布期间进行测试,提供二进制文件,以及在安装期间做广告。

    PLaneT 的兼容包也将被自动纳入 ring-1 类别。

  • 为了减轻外部链接的成本和无法理解孤立的代码,我们将创建异常打印机,在配置的 package catalogs 上搜索模块的提供者。 例如,如果一个模块需要 "data/matrix.rkt", 而该模块不可用,那么将查阅目录以发现哪些包提供该模块。 只有 ring-1 或 ring-0 中的包会被返回。 (这个类别的限制确保了要安装的软件包是唯一的)。

    用户可以配置他们的系统,使其自动安装具有适当类别的软件包 (例如,一些用户可能希望自动安装 ring-0 的软件包, 而不是 ring-1 的软件包,而另一些用户可能不希望安装任何软件包)。

    这个功能将在所有的 package catalogs 中通用,所以用户可以用不同的策略维护他们自己的类别定义。

11.2 长期计划

本节列出了软件包管理器的一些长期计划。 其中许多需要大量的跨 Racket 集成。

  • 官方的目录服务器是赤裸裸的。 可以想象,它可以做得更多:跟踪更多的统计数据,实现关于包和许可证的 "社会" 互动,等等。 这其中有些是很容易和明显的,但社区的需求并不明确。

  • 如果能用 Racket 提供的公钥对官方 package catalog 中的信息进行加密, 并允许其他目录实施类似的安全方案,那就更好了。

  • ring-0 类别的软件包应该在 DrDr 上进行测试。 这就需要一种方法来向 DrDr 传达关于它们应该如何运行的信息。 这一点目前是通过 "meta/props" 脚本来完成的,用于核心中的东西。 我们应该把这个脚本推广到 "meta/props.d" 目录中,这样软件包就可以把 DrDr 的元数据安装到其中。

  • 我们希望这个软件包系统能够鼓励对 Racket 的各个部分进行更多的渐进式改进。 特别是,如果能对不同的数据结构有一个非常彻底的 "data" 收集,那将是非常好的。 然而,我们对 Scribble 的现有设置将迫使每个新的数据结构有一个不同的顶级文档手册, 而不是扩展现有 "data" 集合的文档。 类似的问题也会存在于 "net" 和 "file" 集合中。 我们应该设计一种方法,在 Scribble 中拥有这样的 "文档插件",并支持代码库中其他地方的类似 "插件" 系统。

  • 用户界面可以改进。 例如,如果 DrRacket 能够定期轮询软件包的更新,并且在第一次启动时能够显示可用的流行软件包,那就更好了。
