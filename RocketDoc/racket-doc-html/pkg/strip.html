5 源码、二进制和内置包

一个包,尤其是资源库格式的包,通常以源码形式提供模块的实现和文档。 这样的 source packages 可以适用于多个 Racket 版本,模块被编译成字节码,并且在安装包的时候会呈现出文档。

binary package 只提供编译过的字节码和渲染过的文档,而不是包和文档源。 由于编译的字节码是特定于 Racket 的某个版本,所以 binary package 也是特定于 Racket 的某个版本。 二进制包的好处是,它可以有较少的依赖性(例如,在 Scribble 上实现文档或在其他文档被引用的包上),并且可以更快地安装。 二进制包的缺点是它是特定版本的,源代码可能不太容易被其他程序员立即访问。

binary library package 和 binary package 一样,但它进一步省略了文档。

一个 built package 结合了源代码和编译元素。 built package 可以比源码包安装得更快,只要它是为合适的 Racket 版本安装的, 但源码仍然可以作为备份供其他程序员参考或为不同的 Racket 版本重新构建。

程序员通常只提供 source packages, 而包目录服务可以将每个源码包转换为 binary package, binary library package 或 built package 。 另外,程序员可以通过使用 raco pkg create 子命令和 --binary --binary-lib 或 --built 来创建 binary packages, binary library packages 或 built packages 。 为了方便起见, raco pkg create 子命令也可以从已安装的软件包或版本库签出中创建任何类型的软件包, 丢弃版本库元素(如 ".git" 目录)和编译的代码。 请注意, raco pkg create 默认是按原样捆绑一个软件包目录,没有过滤或注释。

尽管一个包可以被特别注释为 source package,binary package, binary library package 或 built package (见 包的元数据 中的 package-content-state), 不同种类的包主要只是基于包的内容的约定。 所有形式的包都可以在安装中混合使用,而且一个包可以从任何形式更新到任何其他形式。 此外, raco pkg install 和 raco pkg update 支持 --source, --binary, --binary-lib 标志, 以便在安装时转换为包;在这种情况下,包的现有注释会被检查以验证它是否与所要求的转换一致。

从一个目录或包的安装中创建一个 source package, binary package, binary library package 或 built package ,会删去下列文件和目录:

  • 以 ".svn" 命名的目录/文件;

  • 名字以 ".git" 开头的目录/文件;

  • 名字以 "~" 结尾的目录/文件;以及

  • 名字以 "#" 开头和结尾的目录/文件。

然而,上述任何情况都可以通过在软件包或任何子目录的 "info.rkt" 中定义 source-keep-files (用于 source package 和 built package 捆绑), binary-keep-files (用于 binary package, binary library package 和 built package 捆绑) 或 binary-lib-keep-files (用于 binary library package 捆绑)来抑制。 一个 source-keep-files, binary-keep-files 或 binary-lib-keep-files 定义应该把名字绑定到相对于 "info.rkt" 文件的路径列表上。

创建一个 source package 会删去以下额外的文件和目录:

  • 名为 "compiled" 的目录/文件;

  • 名为 "doc" 的目录/文件;

  • 以 "synced.rktd" 命名的目录/文件,它可以作为标记出现在渲染的文档中;

  • 在 "info.rkt" 文件的 source-omit-files 定义中命名的目录/文件。

上述的任何删除都可以通过 source-keep-files 来抑制,即使是在一个被删除的目录中的文件和目录。

创建一个 binary package 会删除以下额外的文件和目录:

  • 名称以 ".rkt" 或 ".ss" 结尾的目录/文件, 其中有相应的编译字节码文件(在 "compiled" 子目录中),不包括 "info.rkt";

  • 名称以 ".scrbl", "_scrbl.zo" 或 ".dep" 结尾的目录/文件;

  • 以 ".css" 或 ".js" 结尾的目录/文件,紧邻名为 "doc" 的目录;

  • 以 "tests" 或 "scribblings" 命名的目录/文件 (但见下文关于 "doc" 和 "info.rkt" 的例外);

  • 在 "info.rkt" 文件的 binary-omit-files 定义中命名的目录/文件。

上述任何删除都可以通过 binary-keep-files 来抑制–即使是在一个被删除的目录中的文件和目录。 此外,当 "doc" 或 "info.rkt" 目录/文件位于 "scribblings" 目录内而不是 "tests" 目录内时,它将被保留。 创建一个 binary package 会进一步调整以下文件(当它们没有被修剪时):

  • 对于任何名称以 ".zo" 结尾的文件, 名为 test, doc 或 srcdoc 的子模块都会被删除;

  • 对于每个引用 "local-redirect.js" 脚本的 ".html" 文件,删除指向该脚本的路径;

  • 每个 "info.rkt" 被调整如下:增加一个 assume-virtual-sources 定义, 任何 copy-foreign-libs 定义被改为 move-foreign-libs, 任何 copy-shared-files 定义被改为 move-shared-files, 任何 copy-man-pages 定义被改为 move-man-pages, 任何 build-deps 定义被删除,任何 update-implies 定义被删除, 以及增加一个 package-content-state 改为 (list 'binary (version)); 并且

  • 路径中的每个集合都得到一个 "info.rkt" ,如果它还没有一个的话, 这样 assume-virtual-sources 就可以被定义。

与 binary package 相比,创建 binary library package 会删除以下额外的文件和目录,并进行额外的调整:

  • 名为 "doc" 的目录/文件被删除;

  • 在 "info.rkt" 文件的 binary-lib-omit-files 定义中命名的目录/文件被删除;以及

  • 每个 "info.rkt" 文件都被调整为删除任何 scribblings 的定义, 并且 package-content-state 被调整为 (list 'binary-lib (version)) 。

除了通过 binary-lib-keep-files 的抑制外, 还可以通过 binary-keep-files 的抑制来抑制任何的删除。

创建一个 built package 会删除任何对于 source package 和 binary package 来说会被删除的文件或目录, 它执行 binary package 的 ".html" 文件更新, 并且包的 "info.rkt" 文件(如果它不存在就添加) 被调整以定义 package-content-state 为 (list 'built (version)) 。

最后,创建一个 binary package, binary library package 或 built package 时, 会 “unmoves” 那些在 "info.rkt" 文件中通过 move-foreign-libs, move-shared-files 或 move-man-pages 定义安装的文件,如果它们不存在于引用的位置, 但存在于一个用户特定的目标目录(即,分别 find-user-lib-dir, find-user-share-dir 或 find-user-man-dir 报告的目录),则会检索它们。 在 Mac OS 上,当 move-foreign-libs 的未移动文件是一个 Mach-O 文件, 其中包括对( (get-lib-search-dirs) 报告的某个目录中的另一个库的引用, 那么该引用将被改变为 @loader_path/ 引用。 在 Unix 上,当 move-foreign-libs 的未移动文件是一个 ELF 文件, 其 RPATH 配置为 $ORIGIN: 后面是由 (find-lib-dir) 报告的主安装库目录的路径时, 那么其 RPATH 值将被设置为 $ORIGIN 。

 (require pkg/strip) package: base
pkg/strip 模块提供了对将一个包式目录复制到指定目的地的支持, 其文件/目录的省略和更新与 raco pkg create 相同。

procedure

(generate-stripped-directory mode    
  src-dir    
  dest-dir) → void?
  mode : (or/c 'source 'binary 'binary-lib 'built)
  src-dir : path-string?
  dest-dir : path-string?
将 src-dir 复制到 dest-dir , 其文件/目录的遗漏和更新与 source package, binary package, binary library package 或 built package 时进行相应的文件/目录省略和更新,如 mode 所指示。 给定的 src-dir 和 dest-dir 必须都已经存在。 如果 src-dir 和 dest-dir 相同,那么 src-dir 将被直接修改,这可能涉及到删除文件。

注意, generate-stripped-directory 并不编译或渲染在 src-dir 中发现的源文件。 要在剥离源代码目录之前进行预编译或渲染,请使用 raco setup 或 raco make 。

Changed in version 7.2.0.10 of package base: 增加了对 src-dir 和 dest-dir 为同一路径的支持。
