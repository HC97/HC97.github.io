1 概述 (CS)

Racket CS 运行时系统是由 Chez Scheme 内核的一个包装器实现的。 该包装器实现了与操作系统的额外连接(例如,用于 I/O 和网络),并提供了进入 Racket 层评估器的入口。

1.1 “S” 与 “Racket” 的区别

在 Racket CS 的 C API 中,以 S 开头的名字来自 Chez Scheme 层, 而以 racket_ 开头的名字则来自 Racket 包装器。

1.2 Racket CS 的内存管理

在任何时候,只要使用 racket_... 函数来运行 Racket 代码, {Racket} 值就可以被移动或垃圾回收。 不要在这种调用中保留对任何 Racket 值的引用。 这一要求与 Racket 的 BC 实现形成对比,后者为 C 代码提供了一种与内存管理器更直接合作的方式。

除非另有说明,以 S 开头的 API 函数不会收集或移动对象,因此在这种调用中对 Racket 值的引用是安全的。

Slock_object 函数可以阻止一个对象被移动或被垃圾回收,但应该少用。 垃圾收集可以通过调用 Chez Scheme 函数 disable-interrupts 完全禁用, 然后通过平衡调用 enable-interrupts 重新启用; 通过 racket_primitive 访问这些函数并通过 Scall0 调用它们。 请注意,禁用中断的同时也禁用了 Racket 线程的上下文切换和中断的信号处理。 假设中断开始时是启用的,调用 disable-interrupts 可能会在进一步的收集被禁用之前触发垃圾收集。

1.3 Racket CS 和 Places

每个 Racket place 对应一个 Chez Scheme 线程,它也对应一个操作系统实现的线程。 Chez Scheme 线程共享一个全局分配空间,所以 GC 管理的对象可以安全地从一个地方传递到另一个地方。 不过要注意的是,Chez Scheme 线程是不安全的;任何在不同地方安全分享数值的同步都必须明确实现。 地方的 Racket 级函数只有在两个地方都能安全使用的情况下才会跨地方共享值。

在一个 embedding application 中, 最初调用 racket_boot 的操作系统线程是原始地方的操作系统线程。

1.4 Racket CS 和线程

Racket 为 Racket 程序实现了线程,没有来自操作系统或 Chez Scheme 的线程的援助, 因此从 C 代码的角度看,Racket 线程是合作的。 独立的 Racket 使用一些私有的操作系统实现的线程来完成后台任务, 但这些操作系统实现的线程从未被 Racket 的 API 暴露过。

Racket 可以与额外的操作系统实现的线程共存,但在调用 S 函数时必须注意, 额外的操作系统或 Chez Scheme 线程不能调用任何 racket_ 函数。 其他操作系统线程要调用 S 函数, 必须先用 Sactivate_thread 将该线程激活为 Chez Scheme 线程。

1.5 Racket CS 整数

C 类型 iptr 被 Racket CS 头文件定义为一个整数类型,其大小足以容纳一个指针值。 换句话说,它是 intptr_t 的一个别名。 uptr 类型是无符号的变体。
