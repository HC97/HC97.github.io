2 Loading Foreign Libraries

FFI 通常用于从 shared objects (又称 shared　libraries 或 dynamically loaded libraries)中提取函数和其他对象。 ffi-lib 函数加载一个共享对象。

procedure

(ffi-lib? v) → boolean?

  v : any/c
如果 v 是一个 foreign-library value　,返回 #t ,否则返回　#f 。

procedure

(ffi-lib path    
  [version    
  #:get-lib-dirs get-lib-dirs    
  #:fail fail    
  #:global? global?    
  #:custodian custodian]) → any
  path : (or/c path-string? #f)
  version : (or/c string? (listof (or/c string? #f)) #f) = #f
  get-lib-dirs : (-> (listof path?)) = get-lib-search-dirs
  fail : (or/c #f (-> any)) = #f
  global? : any/c = (eq? 'global (system-type 'so-mode))
  custodian : (or/c 'place custodian? #f) = #f
返回一个 foreign-library value 或 fail　的结果。 通常,

  • path 是一个没有版本或后缀的路径(即,没有 ".dll", ".so" 或 ".dylib");而

  • version 是一个依次尝试的版本列表, #f (即无版本)是该列表的最后一个元素; 例如, '("2" #f) 表示版本2,并退回到无版本库。

    当 (system-type 'so-suffix) 报告的库后缀是 ".dylib"　时, 则在 path 中的 "." 之后和 ".dylib" 后缀之前添加一个版本。 当库的后缀是 ".dll"　时, 则在 path 中的 "-" 后和 ".dll" 后缀前添加一个版本。 对于任何其他的后缀,版本号被加在后缀加 "."　之后。

一个字符串或 #f version 等同于一个只包含字符串或 #f　的列表, 而一个空字符串(本身或在一个列表中)等同于 #f　。

小心依赖无版本的库名。 在一些平台上,无版本的库名只由开发包提供。 同时,其他平台可能需要一个无版本的回退。 一个版本字符串的列表,后面跟着 #f ,通常是最好的 version　。

假设 path 不是 #f, 来自　ffi-lib 的结果代表了通过以下搜索过程找到的库:

  • 如果 path 不是绝对路径,则在 get-lib-dirs　报告的每个目录中寻找; 默认列表是 (get-lib-search-dirs)　的结果。 在每个目录中,用 version　中的第一个版本尝试 path ,如果 path 还没有以后缀结尾,则添加一个合适的后缀,然后尝试 version　中的第二个版本,等等。 (如果 version 是一个空的列表,在这一步就不尝试任何路径)。

  • 再次尝试相同的文件名,但不将路径转换为绝对路径,这允许操作系统使用自己的搜索路径。 (如果 version 是一个空列表,在这一步中不尝试任何路径。)

  • 试试没有添加任何版本或后缀的 path ,也没有转换为绝对路径。

  • 再次尝试经过版本调整的文件名,但要相对于当前目录。 (如果 version 是一个空列表,在这一步中不尝试路径。)

  • 尝试没有添加任何版本或后缀的 path ,但转换为相对于当前目录的绝对路径。

如果没有一个路径成功,而且 fail 是一个函数,那么　fail 在尾部位置被调用。 如果 fail 是　#f, 那么在尝试上面第二条中的第一个路径或(如果 version 是一个空列表)上面第三条中的第一个路径时就会报告错误。 一个库文件可能存在,但由于某种原因无法加载; 最终的错误信息将不幸地命名为第二或第三项中的回退路径,因为一些操作系统没有办法确定为什么一个给定的库路径会失败。

如果 path 是 #f, 那么产生的 foreign-library 值代表当前进程中加载的所有库, 包括之前用 ffi-lib　打开的库。 特别是,使用 #f 来访问由运行时系统导出的 C 级功能(如 Inside: Racket C API 中所述)。 当 path 是 #f　时, version 参数被忽略。

如果 path 不是 #f, global? 为真,并且操作系统支持以 "全局" 模式打开一个库, 这样该库的符号就会被用来解析以后加载的库的引用,那么就会使用全局模式来打开该库。 否则,库将以 "本地" 模式打开,库的符号不会被用于未来的解析。 这种本地与全局的选择并不影响库的符号是否可以通过 (ffi-lib #f)　获得。

如果 custodian 是 'place 或一个监护人,当一个监护人被关闭时, 库就会被卸载–如果 custodian 是　'place　,则是给定的监护人或该地方的主要监护人。 当一个库被卸载时,所有对该库的引用都变得无效。 为　custodian 提供 'place 与通过　ffi/unsafe/alloc 进行的终结一致, 但不会,例如,在 DrRacket 中点击 Run 按钮时卸载库。 为 custodian 提供 (current-custodian) 往往可以急切地卸载库, 但需要更加小心,以确保库的引用在库被卸载后不会被访问。

如果 custodian 是 #f, 那么在这个过程中,加载的库会与 Racket(或 DrRacket)关联。 再次用 ffi-lib　加载,不会强制重新加载相应的库。

当 ffi-lib 返回一个先前在当前位置加载的库的引用时,它会增加已加载库的引用计数,而不是重新加载该库。 卸载一个库的引用会减少引用计数,并且只有在引用计数为零时才会要求在操作系统层面上卸载。

ffi-lib 过程在 'ffi-lib　主题上记录 (见 记录) 。 特别是,在失败时,它记录了根据上述规则尝试的路径,但由于操作系统的库搜索路径,它不能报告尝试的路径。

Changed in version 6.1.0.5 of package base: 改变了版本号与 ".dll" 后缀的添加方式,将其放在后缀之前,而不是之后。
Changed in version 7.3.0.3: 增加了日志记录。
Changed in version 7.4.0.7: 增加了 #:custodian 参数。

procedure

(get-ffi-obj objname lib type [failure-thunk]) → any

  objname : (or/c string? bytes? symbol?)
  lib : (or/c ffi-lib? path-string? #f)
  type : ctype?
  failure-thunk : (or/c (-> any) #f) = #f
在　lib 库中寻找 objname 。 如果 lib 不是一个 foreign-library value　,则通过调用 ffi-lib　转换为外国库。 如果 objname　在 lib　中被找到,它将被转换为 Racket 的 type　。 类型的描述见 C 类型　;尤其是 get-ffi-obj 过程最常被用于用 _fun　创建的函数类型。

请记住, get-ffi-obj 是一个不安全的过程;详见　概述 。

如果没有找到名称,并且提供了 failure-thunk ,它将被用来产生一个返回值。 例如,如果一个名字没有找到,可以提供一个 failure-thunk 来报告一个特定的错误:

(define foo
  (get-ffi-obj "foo" foolib (_fun _int -> _int)
    (lambda ()
      (error 'foolib
             "installed foolib does not provide \"foo\""))))

默认情况下(也是当 failure-thunk 被提供为 #f　时)是引发一个异常。

procedure

(set-ffi-obj! objname lib type new) → void?

  objname : (or/c string? bytes? symbol?)
  lib : (or/c ffi-lib? path-string? #f)
  type : ctype?
  new : any/c
在 lib 中寻找 objname ,与　get-ffi-obj　类似, 但它会将给定的 new 值存储到库中,将其转换为 C 值。 这可用于设置库的自定义变量,这些变量是其接口的一部分,包括 Racket 的回调。

procedure

(make-c-parameter objname    
  lib    
  type    
  [failure-thunk]) → 
(and/c (-> any)
       (any/c -> void?))
  objname : (or/c string? bytes? symbol?)
  lib : (or/c ffi-lib? path-string? #f)
  type : ctype?
  failure-thunk : (or/c (-> any) #f) = #f
返回一个类似于参数的过程,可以引用指定的外来值,或者设置它。参数的处理和　get-ffi-obj　一样。

类参数函数在 Racket 代码和库代码通过库值进行交互的情况下非常有用。 尽管　make-c-parameter 可以在任何时候使用,但不建议将其用于外国函数, 因为每次通过参数的引用都会在实际调用前构建低级接口。

Changed in version 8.4.0.5 of package base: 增加了 failure-thunk 参数。

syntax

(define-c id lib-expr type-expr)

定义 id 的行为类似于 Racket 绑定, 但 id 实际上是通过　make-c-parameter　创建的类似参数的过程重定向的。 id 既用于 Racket 绑定,也用于外文名称。

procedure

(ffi-obj-ref objname lib [failure-thunk]) → any

  objname : (or/c string? bytes? symbol?)
  lib : (or/c ffi-lib? path-string? #f)
  failure-thunk : (or/c (-> any) #f) = #f
返回一个指定的外文名称的指针,如果没有找到该名称,则调用　failure-thunk ， 如果　failure-thunk 是 #f　,则引发一个异常。

通常情况下,应该使用 get-ffi-obj 来代替。
