4 指针函数

procedure

(cpointer? v) → boolean?

  v : any/c
如果 v 是一个 C 语言的指针或者一个可以作为指针使用的值, 则返回 #t : #f (用作 NULL 指针), 字节字符串(用作内存块), 或者具有 prop:cpointer structure type property 的结构实例。 对于其他值返回 #f 。

procedure

(ptr-equal? cptr1 cptr2) → boolean?

  cptr1 : cpointer?
  cptr2 : cpointer?
比较两个指针的值。 两个不同的 Racket 指针对象可以包含同一个指针。

如果这两个值都是没有用 #f 表示的指针,一个字节字符串,一个回调, 一个基于 _fpointer 的指针,或者一个具有 prop:cpointer 属性的结构, 那么 ptr-equal? 的比较就和使用 equal? 。

procedure

(ptr-add cptr offset [type]) → cpointer?

  cptr : cpointer?
  offset : exact-integer?
  type : ctype? = _byte
返回一个与 cptr 相似的 cpointer,被 ctype 的 offset 实例所抵消。

产生的 cpointer 将基指针和偏移量分开。 在对指针进行任何操作之前,例如将指针提供给外国函数,这两部分在最后一刻被合并。 特别是,指针和偏移量在所有导致外国函数调用的分配之后才结合起来; 如果被调用的函数本身没有调用任何可能触发垃圾收集的东西,它可以安全地使用偏移到 GCable 对象中间的指针。

procedure

(offset-ptr? cptr) → boolean?

  cptr : cpointer?
对于有偏移量的 cpointers,例如使用 ptr-add 创建的指针,是一个谓词。 即使这样的偏移量刚好是 0,也会返回 #t 。 对于其他 cpointers 和非 cpointers 返回 #f 。

procedure

(ptr-offset cptr) → exact-integer?

  cptr : cpointer?
返回一个有偏移量的指针的偏移量。 结果的偏移量总是以字节为单位。

procedure

(cpointer-gcable? cptr) → boolean?

  cptr : cpointer?
如果 cptr 被视为对内存的引用,并被垃圾收集器(假定)管理,则返回 #t ,否则返回 #f 。

对于一个基于 _gcpointer 作为结果类型的指针, cpointer-gcable? 将返回 #t 。 对于一个基于 _pointer 作为结果类型的指针, cpointer-gcable? 将返回 #f 。

4.1 指针解除引用

procedure

(set-ptr-offset! cptr offset [ctype]) → void?

  cptr : cpointer?
  offset : exact-integer?
  ctype : ctype? = _byte
设置一个偏移指针的偏移分量。 参数的使用方式与 ptr-add 相同。 如果 cptr 没有偏移量,会引发 exn:fail:contract 异常。

procedure

(ptr-add! cptr offset [ctype]) → void?

  cptr : cpointer?
  offset : exact-integer?
  ctype : ctype? = _byte
像 ptr-add, 但破坏性地修改一个指针中包含的偏移量。 同样的操作可以使用 ptr-offset 和 set-ptr-offset! 进行。

procedure

(ptr-ref cptr type [offset]) → any

  cptr : cpointer?
  type : ctype?
  offset : exact-nonnegative-integer? = 0
(ptr-ref cptr type abs-tag offset) → any
  cptr : cpointer?
  type : ctype?
  abs-tag : (one-of/c 'abs)
  offset : exact-nonnegative-integer?
(ptr-set! cptr type val) → void?
  cptr : cpointer?
  type : ctype?
  val : any/c
(ptr-set! cptr type offset val) → void?
  cptr : cpointer?
  type : ctype?
  offset : exact-nonnegative-integer?
  val : any/c
(ptr-set! cptr type abs-tag offset val) → void?
  cptr : cpointer?
  type : ctype?
  abs-tag : (one-of/c 'abs)
  offset : exact-nonnegative-integer?
  val : any/c
ptr-ref 过程返回由 cptr 引用的对象,使用给定的 type 。 ptr-set! 过程在 cptr 指向的内存中存储 val , 使用给定 type 进行转换。

在每种情况下, offset 默认为 0 (这是唯一应该用于 ffi-obj 对象的值, 见 未导出的原始函数)。 如果一个 offset 的索引不是 0, 值就会被读取或存储在那个位置, 把指针看作是一个 type 的向量 — 所以实际地址是指针加上 type 的大小乘以 offset 。 此外,可以使用 'abs 标志来使用 offset 作为字节计数,而不是指定 type 的增量。

注意 ptr-ref 和 ptr-set! 过程不保留任何关于如何使用指针的元信息。 程序员有责任只在适当的时候使用这个工具。 例如,在一个小 endian 机器上:

> (define block (malloc _int 5))
> (ptr-set! block _int 0 196353)
> (map (lambda (i) (ptr-ref block _byte i)) '(0 1 2 3))
(1 255 2 0)

此外, ptr-ref 和 ptr-set! 不能检测到偏移量超出对象的内存边界; 越界访问很容易导致分段故障或内存损坏。

procedure

(memmove cptr    
  [offset]    
  src-cptr    
  [src-offset]    
  count    
  [type]) → void?
  cptr : cpointer?
  offset : exact-integer? = 0
  src-cptr : cpointer?
  src-offset : exact-integer? = 0
  count : exact-nonnegative-integer?
  type : ctype? = _byte
从 src-cptr 复制到 cptr 。 目的指针可以被一个可选的 offset 所偏移,它在 type 实例中。 源指针可以通过 src-offset 进行类似的偏移。 从源码复制到目的码的字节数由 count 决定,当提供时,它在 type 实例中。

procedure

(memcpy cptr    
  [offset]    
  src-cptr    
  [src-offset]    
  count    
  [type]) → void?
  cptr : cpointer?
  offset : exact-integer? = 0
  src-cptr : cpointer?
  src-offset : exact-integer? = 0
  count : exact-nonnegative-integer?
  type : ctype? = _byte
与 memmove 类似,但如果目标和源重叠,结果就不确定。

procedure

(memset cptr [offset] byte count [type]) → void?

  cptr : cpointer?
  offset : exact-integer? = 0
  byte : byte?
  count : exact-nonnegative-integer?
  type : ctype? = _byte
类似于 memmove, 但目的地被统一填充为 byte (即,0 到 255 之间的精确整数,包括在内)。 当有 type 参数时,其结果是调用没有 type 参数的 memset, count 乘以与 type 相关的大小。

procedure

(cpointer-tag cptr) → any

  cptr : cpointer?
返回 Racket 对象,该对象是给定 cptr 指针的标签。

procedure

(set-cpointer-tag! cptr tag) → void?

  cptr : cpointer?
  tag : any/c
设置给定 cptr 的标签。 tag 可以是任何任意的值;其他指针操作会忽略它。 当一个 cpointer 值被打印时,如果它是一个符号、一个字节串、一个字符串,它的标签会被显示。 此外,如果标签是一个在其 car 中持有其中一个的对,则显示 car (以便标签可以包含其他信息)。

4.2 内存管理

关于 Racket 的 C 级内存管理的一般信息,请参见 Inside: Racket C API 。

procedure

(malloc bytes-or-type    
  [type-or-bytes    
  cptr    
  mode    
  fail-mode]) → cpointer?
  bytes-or-type : 
(or/c (and/c exact-nonnegative-integer? fixnum?)
      ctype?)
  type-or-bytes : 
(or/c (and/c exact-nonnegative-integer? fixnum?)
      ctype?)
   = absent
  cptr : cpointer? = absent
  mode : 
(one-of/c 'raw 'atomic 'nonatomic 'tagged
          'atomic-interior 'interior
          'stubborn 'uncollectable 'eternal)
 = absent
  fail-mode : (one-of/c 'failok) = absent
使用指定的分配方式分配一个指定大小的内存块。 其结果是一个指向所分配内存的 C 语言指针,如果要求的大小为 0,则为 #f 。 虽然上面没有反映,但四个参数可以以任何顺序出现,因为它们都是不同类型的 Racket 对象;至少需要一个大小规格:

  • 如果给定的是 C 类型的 bytes-or-type ,它的大小将被用来决定块的分配大小。

  • 如果给出了一个整数 bytes-or-type ,它将指定所需的字节大小。

  • 如果同时给出 bytes-or-type 和 type-or-bytes , 那么分配的大小是一个值的向量(C 类型的大小和整数的乘法)。

  • 如果给了一个 cptr 指针,它的内容将被复制到新块中。

  • 可以给出一个符号 mode 参数,它指定了要使用的分配函数。它应该是以下的一个:

    • 'raw — 分配垃圾收集器空间之外的、不被垃圾收集器追踪的内存 (即,被视为不持有指向可收集内存的指针)。 这个内存必须用 free 来释放。该内存的初始内容是未指定的。

    • 'atomic — 分配可以被垃圾收集器回收的内存,但不被垃圾收集器追踪。 该内存的初始内容未被指定。

      对于 BC Racket 的实现,这种分配模式对应于 C API 中的 scheme_malloc_atomic 。

    • 'nonatomic — 分配可以被垃圾收集器回收的内存, 被垃圾收集器视为只持有指针,并且最初填充为零。 该内存允许包含对垃圾收集器管理的对象的引用和垃圾收集器空间之外的地址的混合。

      对于 BC Racket 的实现,这种分配模式对应于 C API 中的 scheme_malloc 。

    • 'atomic-interior — 和 'atomic 一样,但是只要被分配的对象被保留,垃圾收集器就不会移动。

      这种分配模式更好的名字是 'atomic-immobile, 但由于历史原因,它是 'atomic-interior 。

      对于 BC Racket 的实现,一个引用可以指向对象的内部,而不是它的起始地址。 这种分配模式对应于 C API 中的 scheme_malloc_atomic_allow_interior 。

    • 'interior — 就像 'nonatomic 一样,但只要被分配的对象被保留,垃圾收集器就不会移动。

      这种分配模式更好的名字是 'nonatomic-immobile, 但由于历史原因,它是 'interior 。

      对于 BC Racket 的实现,一个引用可以指向对象的内部,而不是它的起始地址。 这种分配模式对应于 C API 中的 scheme_malloc_allow_interior 。

    • 'tagged — 分配的内存必须以一个 short 值开始, 该短值在垃圾收集器中注册为一个标签。

      这种模式只支持 BC Racket 的实现,它对应于 C API 中的 scheme_malloc_tagged 。

    • 'stubborn — 与 'nonatomic 类似, 但支持在对对象进行了所有修改后,通过 end-stubborn-change 向 GC 发出提示。

      这种模式只支持 BC Racket 的实现,它对应于 C API 中的 scheme_malloc_stubborn 。

    • 'eternal — 和 'raw 一样,除了分配的内存不能被释放。

      这种模式只支持 CGC Racket 变体,它对应于 C API 中的 scheme_malloc_uncollectable 。

    • 'uncollectable — 分配的内存永远不会被收集, 不能被释放,并且可能包含指向可收集内存的指针。

      这种模式只支持 CGC Racket 变体,它对应于 C API 中的 scheme_malloc_uncollectable 。

  • 如果给了一个额外的 'failok 标志, 那么可以努力检测分配失败并引发 exn:fail:out-of-memory 而不是崩溃。

如果没有指定模式,那么当类型是基于 _gcpointer 或 _scheme 的类型时, 就使用 'nonatomic 分配,否则就使用 'atomic 分配。

Changed in version 6.4.0.10 of package base: 增加了 'tagged 分配模式。
Changed in version 8.0.0.13: 修改了 CS 以支持 'interior 分配模式。
Changed in version 8.1.0.6: 修改了 CS,删除了对使用 'nonatomic 和 'interior 分配模式分配的内存的限制。

procedure

(free cptr) → void

  cptr : cpointer?
对于 'raw 分配的指针,以及外国库分配的、我们应该释放的指针,使用了操作系统的 free 函数。 请注意,作为 finalizer (见下文)程序钩子的一部分,这很有用(例如,在 Racket 指针对象上,当指针对象被收集时释放内存,但要注意别名)。

用 malloc 和 'raw 以外的模式分配的内存不能被 free , 因为这些模式分配的内存是由垃圾收集器管理的。

procedure

(end-stubborn-change cptr) → void?

  cptr : cpointer?
在给定的顽固分配的指针上使用 scheme_end_stubborn_change 。

procedure

(malloc-immobile-cell v) → cpointer?

  v : any/c
分配足够大的内存来容纳一个任意的(可收集的) Racket 值,但其本身不是可收集的或由内存管理器移动的。 单元用 v 初始化; 使用 _scheme 类型的 ptr-ref 和 ptr-set! 来获取或设置单元的值。 该单元必须用 free-immobile-cell 明确释放。

procedure

(free-immobile-cell cptr) → void?

  cptr : cpointer?
释放一个由 malloc-immobile-cell 创建的非移动单元。

procedure

(register-finalizer obj finalizer) → void?

  obj : any/c
  finalizer : (any/c . -> . any)
用给定的 obj 注册一个 finalizer 程序 finalizer-proc ,obj 可以是任何 Racket(GC-able) 对象。 finalizer 被注册在一个 “late” will executor 身上, 这个遗嘱执行者只有在该值的所有弱引用(比如在一个 weak box) 中) 被清除后才会为该值准备好遗嘱,这意味着该值是不可触及的, 并且没有正常的 will executor 为该值准备好遗嘱。 当 obj 的遗嘱在 “late” 遗嘱执行者中准备好时,终结者就会被调用, 这意味着安全代码无法触及该值(甚至从遗嘱中,甚至从它本身)。

finalizer 在一个负责触发 register-finalizer 的遗嘱执行器的线程中被调用。 给定的 finalizer 程序一般不应该依赖触发线程的环境, 而且它不能使用任何参数或调用任何参数函数,除了依赖默认的记录器和/或调用 current-logger 是允许的。

终结器主要用于 cpointer 对象(用于释放不受 GC 控制的未使用的内存),但它也可以用于任何 Racket 对象–即使是与外来代码无关的对象。 然而,请注意,终结器是为代表指针的 Racket 对象注册的。 如果你打算释放一个指针对象,那么你必须注意不要为两个指向同一地址的 cpointer 注册终结器。 同时,要注意不要让最终处理程序成为一个保持对象的闭包。 最后,要注意当一个地方退出时,终结器并不保证会被运行; 参见 ffi/unsafe/alloc 和 register-finalizer-and-custodian-shutdown 获取更完整的解决方案。

作为 register-finalizer 的一个例子,假设你正在处理一个返回 C 指针的外国函数, 你应该释放这个指针,但你主要想使用 16 字节偏移的内存。这里是一个创建合适类型的尝试:

(define _pointer-at-sixteen/free
  (make-ctype _pointer
              #f ; i.e., just _pointer as an argument type
              (lambda (x)
                (let ([p (ptr-add x 16)])
                  (register-finalizer x free)
                  p))))

上面的代码是错误的 :finalizer 是为 x 注册的,在新的指针 p 被创建后就不再需要它了。 把这个例子改成为 p 注册 finalizer 可以纠正这个问题, 但此时 free 被调用 p 而不是 x 。 在修复这个问题的过程中,我们可能会小心翼翼地记录一条信息以备调试:

(define _pointer-at-sixteen/free
  (make-ctype _pointer
              #f
              (lambda (x)
                (let ([p (ptr-add x 16)])
                  (register-finalizer p
                    (lambda (ignored)
                      (log-debug (format "Releasing ~s\n" p))
                      (free x)))
                  p))))

现在,我们从未看到任何记录的事件。问题在于 finalizer 是一个保持对 p 引用的闭包。 与其引用被终结的值,不如使用终结器的输入参数;简单地将上面的 ignored 改为 p ,就可以解决这个问题。 (删除调试信息也可以避免这个问题,因为最终化程序就不会在 p 上关闭)。

procedure

(make-late-weak-box v) → weak-box?

  v : any/c

procedure

(make-late-weak-hasheq v) → (and/c hash? hash-eq? hash-weak?)

  v : any/c
就像 make-weak-box 和 make-weak-hasheq 一样, 但是 "迟到的" 弱引用比 make-weak-box 或 make-weak-hasheq 的结果中的引用持续时间更长。 具体来说,如果一个值无法到达,但尚未被 register-finalizer 安装的 finalizer 处理, 那么 "迟到的" 弱引用将保持完整。 迟到的 "弱引用 "是为了给这些终结者使用的。

procedure

(make-sized-byte-string cptr length) → bytes?

  cptr : cpointer?
  length : exact-nonnegative-integer?
在 Racket 的 BC 实现中,返回一个由给定指针和给定长度组成的字节字符串;不进行复制。 在 CS 实现中,会引发 exn:fail:unsupported 异常。

请注意,Racket 字节串的表示方法通常要求在字节串的末尾有一个 nul 结束符(在 length 字节之后),但是有些函数可以使用没有这种结束符的字节串表示方法,特别是 bytes-copy 。

如果 cptr 是一个由 ptr-add 创建的偏移量指针,那么偏移量将立即被添加到该指针。 因此,这个函数不能与 ptr-add 一起使用来创建一个 Racket 字节串的子串, 因为偏移量指针将指向一个可收集对象的中间(这是不允许的)。

procedure

(void/reference-sink v ...) → void?

  v : any/c
返回 #<void>, 但与调用 void 函数不同的是,编译器可能会优化掉该调用, 并以 #<void> 的结果代替, 调用 void/reference-sink 确保参数被认为是可被垃圾收集器到达的,直到调用返回。

Added in version 6.10.1.2 of package base.

4.3 指针结构属性

value

prop:cpointer : struct-type-property?

一个 structure type property ,使一个结构类型的实例作为 C 语言的指针值工作。 该属性值必须是一个精确的非负整数,表示结构中的一个不可变的字段(反过来,它必须被初始化为一个 C 指针值), 一个接收结构实例并返回一个 C 指针值的过程,或者一个C指针值。

prop:cpointer 属性允许结构实例被透明地用作 C 指针值, 或者允许 C 指针值被一个可能有额外值或属性的结构透明地包裹起来。
