5.15 Objective-C FFI

 (require ffi/unsafe/objc) package: base
ffi/unsafe/objc 库建立在 ffi/unsafe 之上,支持与 Objective-C 的交互。

该库分两层支持 Objective-C 交互。 上层提供了发送消息和衍生子类的语法形式。 下层是对 Objective-C runtime library 函数的一个薄的包装。 即使是上层也是不安全的,与普通的 Racket 库相比也是相对低级的,因为参数和结果类型必须用 FFI C 类型来声明(见 (see 类型构造器) 。

5.15.1 FFI 类型和常量

value

_id : ctype?

一个 Objective-C 对象的类型,一个不透明的指针。

value

_Class : ctype?

一个 Objective-C 类的类型,它也是一个 _id 。

value

_Protocol : ctype?

一个 Objective-C 协议的类型,也是一个 _id 。

value

_SEL : ctype?

一个 Objective-C 选择器的类型,一个不透明的指针。

value

_BOOL : ctype?

Objective-C 的布尔类型。 Racket 值以通常的方式转换为 C 语言: #f 是假的,其他任何值都是真的。 C 语言的值被转换为 Racket 的布尔型。

value

YES : boolean?

#t 的同义词

value

NO : boolean?

#f 的同义词

5.15.2 语法形式和程序

syntax

(tell result-type obj-expr method-id)

(tell result-type obj-expr arg ...)
 
result-type = 
  | #:type ctype-expr
     
arg = method-id arg-expr
  | method-id #:type ctype-expr arg-expr
向由 obj-expr 产生的 Objective-C 对象发送一个消息。 当结果或参数的类型被省略时,该类型被假定为 _id, 否则它必须被指定为 FFI C 类型 (see 类型构造器) 。

如果只提供了一个没有参数的 method-id , 那么 method-id 不能以 : 结尾; 否则,每个 method-id 必须以 : 结尾。

Examples:
> (tell NSString alloc)

#<cpointer:id>

> (tell (tell NSString alloc)
        initWithUTF8String: #:type _string "Hello")

#<cpointer:id>

syntax

(tellv obj-expr method-id)

(tellv obj-expr arg ...)
像 tell 一样,但是有一个结果类型 _void 。

syntax

(import-class class-id ...)

将每个 class-id 定义为用 class-id 的字符串形式注册的类(FFI 类型为 _Class 的值)。 注册的类是通过 objc_lookUpClass 获得的。

Example:
> (import-class NSString)

通过 import-class 访问的类通常是作为加载一个外国库的副作用来声明的。 例如,如果你想在 Mac OS 上导入 NSString 类,首先必须加载 "Foundation" 框架。 请注意,如果你在 DrRacket 或 require racket/gui/base 的模块中使用 import-class ,那么 "Foundation" 将已经被加载到 Racket 进程中。 为了避免依赖其他库来加载 "Foundation", 请用 ffi-lib 明确加载它::

> (ffi-lib
   "/System/Library/Frameworks/Foundation.framework/Foundation")
> (import-class NSString)

syntax

(import-protocol protocol-id ...)

将每个 protocol-id 定义为用 protocol-id 的字符串形式注册的协议 (FFI 类型为 _Protocol 的值)。 注册的类通过 objc_getProtocol 获得。

Example:
> (import-protocol NSCoding)

syntax

(define-objc-class class-id superclass-expr
  maybe-mixins
  maybe-protocols
  [field-id ...]
  method ...)
 
maybe-mixins = 
  | #:mixins (mixin-expr ...)
     
maybe-protocols = 
  | #:protocols (protocol-expr ...)
     
method = (mode maybe-async result-ctype-expr (method-id) body ...+)
  | (mode maybe-async result-ctype-expr (arg ...+) body ...+)
     
mode = +
  | -
  | +a
  | -a
     
maybe-async = 
  | #:async-apply async-apply-expr
     
arg = method-id [ctype-expr arg-id]
将 class-id 定义为一个新的、注册的 Objective-C 类(FFI 类型为 _Class)。 superclass-expr 应该产生一个 Objective-C 类或 #f 的超类。 一个可选的 #:mixins 子句可以指定用 define-objc-mixin 定义的 mixins。 一个可选的 #:protocols 子句可以指定由该类实现的 Objective-C 协议, 其中一个 protocol-expr 的 #f 结果被忽略了。

每个 field-id 是一个持有 Racket 值的实例字段,当对象被分配时,它被初始化为 #f 。 这些 field-id 可以在方法 body 中被直接引用和 set! 。 在对象外部,它们可以通过 get-ivar 和 set-ivar! 来引用和设置。

每个 method 都会向类添加或重写一个方法(当 mode 为 - 或 -a 时),以便在实例上调用, 或者向元类添加一个方法(当 mode 是 + 或 +a 时),以便在类本身上调用。 所有的结果和参数类型必须使用 FFI 的 C 类型来声明(参见 (see 类型构造器)。 当 mode 是 +a 或 -a 时,该方法以原子模式被调用(见 _cprocedure) 。 一个可选的 #:async-apply 规范决定了该方法在被外国线程调用时的工作方式, 与 _cprocedure 的方式相同。

如果一个 method 被声明为只有一个 method-id 而没有参数, 那么 method-id 就不能以 : 结尾。 否则,每个 method-id 必须以 : 结尾。

如果特殊方法 dealloc 被声明为模式 -, 那么它必须不调用超类方法, 因为一个 (super-tell dealloc) 被自动添加到方法的末尾。 此外,在 (super-tell dealloc) 之前,实例中的每个 field-id 的空间都会被取消分配。

Example:
> (define-objc-class MyView NSView
    [bm] ; <- one field
    (- _racket (swapBitwmap: [_racket new-bm])
       (begin0 bm (set! bm new-bm)))
    (- _void (drawRect: [_NSRect exposed-rect])
       (super-tell drawRect: exposed-rect)
       (draw-bitmap-region bm exposed-rect))
    (- _void (dealloc)
       (when bm (done-with-bm bm))))

Changed in version 6.90.0.26 of package base: 改变了 #:protocols 的处理方式, 忽略了 #f 的表达结果。

syntax

(define-objc-mixin (class-id superclass-id)
  maybe-mixins
  maybe-protocols
  [field-id ...]
  method ...)
与 define-objc-class 类似,但定义了一个mixin, 通过 define-objc-class 或 define-objc-mixin 与其他方法定义相结合。 指定的 field-id 不会被 mixin 添加,但必须是为添加方法的类所声明的 field-id 的一个子集。

syntax

self

当在 define-objc-class 或 define-objc-mixin 方法的主体中使用时,指的是其方法被调用的对象。 这种形式不能在 define-objc-class 或 define-objc-mixin 方法之外使用。

syntax

(super-tell result-type method-id)

(super-tell result-type arg ...)
当在 define-objc-class 或 define-objc-mixin 方法的主体中使用时,调用一个超类方法。 result-type 和 arg 子形式的语法与 tell 中的相同。 这种形式不能在 define-objc-class 或 define-objc-mixin 方法之外使用。

syntax

(get-ivar obj-expr field-id)

提取用 define-objc-class 创建的类中一个字段的 Racket 值。

syntax

(set-ivar! obj-expr field-id value-expr)

设置用 define-objc-class 创建的类中某个字段的 Racket 值。

syntax

(selector method-id)

返回一个选择器(FFI 类型为 _SEL) ,用于选择 method-id 的字符串形式。

Example:
> (tellv button setAction: #:type _SEL (selector terminate:))

(objc-is-a? obj cls) → boolean?

  cls : _Class
检查 obj 是 Objective-C 类 cls 的一个实例还是一个子类。

检查 subcls 是否是 cls 或一个子类。

(objc-get-class obj) → _Class

提取 obj 的类。

procedure

(objc-set-class! obj cls) → void?

  obj : _id
  cls : _Class
将 obj 的类改变为 cls 。 对象的现有表示必须与新的类兼容。

Added in version 6.3 of package base.

procedure

(objc-get-superclass cls) → _Class

  cls : _Class
返回 cls 的超类。

Added in version 6.3 of package base.

procedure

(objc-dispose-class cls) → void?

  cls : _Class
销毁 cls ,它必须没有现有的实例或子类。

Added in version 6.3 of package base.

procedure

(objc-block function-type? proc #:keep keep) → cpointer?

  function-type? : ctype
  proc : procedure?
  keep : (box/c list?)
将Racket函数 proc 包装成一个 Objective-C 块。 该过程必须接受一个初始指针参数,它是该块的 "自我" 参数, 并且该额外参数必须包含在给定的 function-type 中。

为实现该块而分配的额外记录被添加到 keep 中的列表中, 该列表也可能通过 _fun 的 #:keep 选项包含在 function-type 中。 在 keep 中注册的指针必须被保留,只要该块仍在使用。

Added in version 6.3 of package base.

syntax

(with-blocking-tell form ...+)

导致任何在 form 中的 tell, tellv 或 super-tell 表达式在 #:blocking? 参数的意义上成为 _cprocedure 的阻塞。 否则, (with-blocking-tell form ...+) 等同于 (let () form ...+) 。

Added in version 7.0.0.19 of package base.

5.15.4 遗留库

 (require ffi/objc) package: base
ffi/objc 库是 ffi/unsafe/objc 的一个废弃的入口点。 它只直接输出安全操作,而不安全的操作则使用 objc-unsafe! 导入, 类似于 scheme/foreign 。

syntax

(objc-unsafe!)

使得 ffi/unsafe/objc 的不安全绑定在导入模块中可用。
