8 注意事项和限制

本节介绍了程序员在将程序移植到 Typed Racket 时经常会遇到的限制和类型系统的微妙方面。

8.1 Integer 类型和 integer?

在 Typed Racket 中, Integer 类型对应于返回 #t 的 exact-integer? 谓词的值, 而 不是 integer? 谓词。 特别是,对 integer? 返回 #t 的值可能是 inexact number (例如, 1.0)。

当把一个程序移植到 Typed Racket 时, 你可能需要用相应的精确函数(如 exact-round 和 exact-floor)来替换 round 和 floor 等函数的使用。

在其他情况下,可能需要使用 assert 或 cast 。

8.2 多态函数的类型推断

Typed Racket 的本地类型推理算法目前还不能推断出用于高阶参数的多态函数的类型,而这些高阶参数本身就是多态的。

例如,下面的程序导致了一个类型错误,证明了这个限制:

> (map cons '(a b c d) '(1 2 3 4))

eval:2:0: Type Checker: Polymorphic function `map' could not

be applied to arguments:

Domains: (-> a b ... b c) (Listof a) (Listof b) ... b

         (-> a c) (Pairof a (Listof a))

Arguments: (All (a b) (case-> (-> a (Listof a) (Listof a))

(-> a b (Pairof a b)))) (List 'a 'b 'c 'd) (List One

Positive-Byte Positive-Byte Positive-Byte)

  in: 4

问题是, cons 的类型也是多态的:

> cons

- : (All (a b) (case-> (-> a (Listof a) (Listof a)) (-> a b (Pairof a b))))

#<procedure:cons>

为了使这个表达式进行类型检查, 可以使用 inst 形式将多态参数(例如 cons)在一个特定的类型上实例化:

> (map (inst cons Symbol Integer) '(a b c d) '(1 2 3 4))

- : (Listof (Pairof Symbol Integer))

'((a . 1) (b . 2) (c . 3) (d . 4))

8.3 类型-非类型交互和契约生成

当一个类型化模块 require 来自非类型化模块的绑定时(反之亦然),有一些类型不能被转换为相应的契约。

这可能是因为合约系统还不支持某个类型,或者因为 Typed Racket 的合约生成器还没有更新,或者因为合约太难生成。 在其中的一些情况下,这个限制将在未来的版本中被修复。

下面说明了一个不能被转换为契约的例子类型:

> (require/typed racket/base
    [object-name (case-> (-> Struct-Type-Property Symbol)
                         (-> Regexp (U String Bytes)))])

eval:5:0: Type Checker: Error in macro expansion -- Type

(case-> (-> Struct-Type-Property Symbol) (-> Regexp (U Bytes

String))) could not be converted to a contract: function

type has two cases of arity 1

  in: (case-> (-> Struct-Type-Property Symbol) (-> Regexp (U

String Bytes)))

这个函数类型按情况来说是一个有效的类型,但是相应的契约很难生成,因为对结果的检查取决于对域的检查。 在未来,这可能会得到依赖契约的支持。

一个更近似的类型将对这种情况起作用,但在使用地点会损失类型的精度:

> (require/typed racket/base
    [object-name (-> (U Struct-Type-Property Regexp)
                     (U String Bytes Symbol))])
> (object-name #rx"a regexp")

- : (U Bytes String Symbol)

"a regexp"

使用 define-predicate 也涉及契约的生成,因此有些类型不能为它们生成谓词。 下面说明了一个不能为其生成谓词的类型:

> (define-predicate p? (All (A) (Listof A)))

eval:8:0: Type Checker: Error in macro expansion -- Type

(All (A) (Listof A)) could not be converted to a predicate:

cannot generate contract for non-function polymorphic type

  in: (All (A) (Listof A))

8.4 不支持的特性

不能为非函数多态类型生成契约

8.5 类型泛化

与其说是一个注意事项,不如说是一个可能产生意想不到的后果的特性。 为了使使用不变类型构造函数(如 Boxof)的编程更容易, Typed Racket对作为不变类型构造函数参数的类型进行了泛化。比如说:

> 0

- : Integer [more precisely: Zero]

0

> (define b (box 0))
> b

- : (Boxof Integer)

'#&0

0 的类型是 Zero, 这意味着 b “应该” 有类型 (Boxof Zero) 。 另一方面,这种类型并不是特别有用,因为它只允许 0 被存储在盒子里。 最有可能的是,我们的意图是拥有一个更通用类型的盒子(比如 Integer) ,并以 0 来初始化它。 类型泛化正是这样做的。

然而,在某些情况下,类型泛化会导致意外的结果:

> (box (ann 1 Fixnum))

- : (Boxof Integer)

'#&1

这段代码的意图可能是创建一个 Fixnum 的盒子,但是 Typed Racket 还是将它泛化了。 要创建一个 Fixnum 的盒子,盒子本身应该有一个类型注释:

> (ann (box 1) (Boxof Fixnum))

- : (Boxof Fixnum)

'#&1

> ((inst box Fixnum) 1)

- : (Boxof Fixnum)

'#&1

8.6 宏和编译时计算

Typed Racket 将在给定模块的运行时阶段对所有表达式进行类型检查,并将防止在运行时发生的错误。 然而,编译时的表达式–包括发生在宏里面的计算–是不被检查的。

具体来说,这意味着例如在 begin-for-syntax 块内的表达式不被检查:

> (begin-for-syntax (+ 1 "foo"))

+: contract violation

  expected: number?

  given: "foo"

同样,在 Typed Racket 中定义的宏里面的表达式也不进行类型检查。 另一方面,宏的扩展总是要进行类型检查的:

(define-syntax (example-1 stx)
  (+ 1 "foo")
  #'1)

 

(define-syntax (example-2 stx)
  #'(+ 1 "foo"))

 

> (example-1)

+: contract violation

  expected: number?

  given: "foo"

> (example-2)

eval:17:0: Type Checker: type mismatch

  expected: Number

  given: String

  in: (quote "foo")

请注意,在 Typed Racket 中定义的函数如果在编译时被其他类型化模块或未类型化模块使用,将被进行类型检查, 然后按照 Typed-Untyped 交互 中的描述用契约进行保护。

此外,在类型化 Racket 模块中定义的宏不能在普通的 Racket 模块中使用,因为这种使用可以规避类型系统的保护。

8.8 模式匹配和出现类型

由于 Typed Racket 在宏展开 后 会对代码进行类型检查, 所以某些形式–比如 match—对于 Typed Racket 来说很难完全推理。 特别是,在一个 match 子句中,标识符的类型通常 不 会被更新,以反映之前的模式匹配失败的事实。 例如,在下面的函数中,类型检查器不知道如果执行到最后一个子句,那么 string? 谓词已经在 x 的值上匹配失败了, 所以最后一个子句中的 (abs x) 不能进行类型检查:

> (: size (-> (U String Integer) Integer))
> (define (size x)
    (match x
      [(? string?) (string-length x)]
      [_ (abs x)]))

eval:21:0: Type Checker: type mismatch

  expected: Integer

  given: (U Integer String)

  in: x

因为它们是更简单的形式,类似的 cond 和 if 表达式可以成功地进行类型检查:

> (: size (-> (U String Integer) Integer))
> (define (size x)
      (cond
        [(string? x) (string-length x)]
        [else (abs x)]))

一个解决方法是不依赖一个需要知道以前的模式未能匹配以进行类型检查的全面的 "else" 子句:

> (: size (-> (U String Integer) Integer))
> (define (size x)
      (match x
        [(? string?) (string-length x)]
        [(? exact-integer?) (abs x)]))

然而,重要的是要注意,如果没有提供 catch-all 失败子句, match 总是 插入一个 catch-all 失败子句! 这意味着类型检查器不会通知程序员匹配子句的覆盖率不足, 因为隐含的(即宏插入的)失败子句 将 覆盖程序员在模式匹配中未能预见的任何情况,例如:

> (: size (-> (U String Integer) Integer))
> (define (size x)
      (match x
        [(? string?) (string-length x)]))
> (size 42)

match: no matching clause for 42

涉及省略号的模式 ... 的重复可能会产生一个 for 循环,需要对变量进行注释以进行类型检查。 下面这段(刻意模糊的)代码在没有对匹配模式变量 c 进行类型注释的情况下不会进行类型检查。

(: do-nothing (-> (Listof Integer) (Listof Integer)))
(define (do-nothing lst)
  (match lst
    [(list (? number? #{c : (Listof Integer)}) ...)   c]))

8.9 is-a? 和出现类型

类型化的Racket不使用 is-a? 谓词来细化对象类型,因为目标对象可能是在未类型化的代码中创建的, 而 is-a? 并不检查字段和方法的类型。

例如,下面的代码定义了一个类类型 Pizza%, 一个子类类型 Sauce-Pizza%, 以及一个函数 get-sauce (这个函数包含一个类型错误)。 get-sauce 函数使用 is-a? 来测试其参数的类型; 如果测试成功,该函数期望参数有一个名为 topping 的字段,该字段包含一个 Sauce 类型的值。

#lang typed/racket
 
(define-type Pizza%
  (Class (field [topping Any])))
 
(define-type Sauce
  (U 'tomato 'bbq 'no-sauce))
 
(define-type Sauce-Pizza%
  (Class #:implements Pizza% (field [topping Sauce])))
 
(define sauce-pizza% : Sauce-Pizza%
  (class object%
    (super-new)
    (field [topping 'tomato])))
 
(define (get-sauce [pizza : (Instance Pizza%)]) : Sauce
  (cond
    [(is-a? pizza sauce-pizza%)
     (get-field topping pizza)] ; type error
    [else
     'bbq]))

类型错误信息解释说 (get-field topping pizza) 可以返回任何类型的值, 即使 pizza 是 sauce-pizza% 类的一个实例。 特别是, pizza 可能是一个未定型的子类的实例,它将其 topping 设置为整数 0:

; #lang racket
(define evil-pizza%
  (class sauce-pizza%
    (inherit-field topping)
    (super-new)
    (set! topping 0)))

要想按预期进行下行转换,请在 is-a? 测试后添加一个 cast 。 下面是一个完整的例子,它通过了类型检查器并引发了一个运行时错误, 以防止类型化的 get-sauce 函数返回一个非 Sauce 值。

Examples:
> (module pizza typed/racket
    (provide get-sauce sauce-pizza%)
  
    (define-type Pizza%
      (Class (field [topping Any])))
  
    (define-type Sauce
      (U 'tomato 'bbq 'no-sauce))
  
    (define-type Sauce-Pizza%
      (Class #:implements Pizza% (field [topping Sauce])))
  
    (define sauce-pizza% : Sauce-Pizza%
      (class object%
        (super-new)
        (field [topping 'tomato])))
  
    (define (get-sauce [pizza : (Instance Pizza%)]) : Sauce
      (cond
        [(is-a? pizza sauce-pizza%)
         (define p+ (cast pizza (Instance Sauce-Pizza%)))
         (get-field topping p+)]
        [else
         'no-sauce])))
> (require 'pizza)
> (define evil-pizza%
    (class sauce-pizza%
      (inherit-field topping)
      (super-new)
      (set! topping 0)))
> (get-sauce (new evil-pizza%))

sauce-pizza%: broke its own contract

  promised: (or/c (quote no-sauce) (quote bbq) (quote

tomato))

  produced: 0

  in: the topping field in

      (recursive-contract g40 #:impersonator)

  contract from: pizza

  blaming: pizza

   (assuming the contract is correct)
