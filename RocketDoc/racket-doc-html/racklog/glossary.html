12 Racklog 原语词汇表

12.1 Racket 谓词

procedure

(logic-var? x) → boolean?

  x : any/c
标识逻辑变量。

procedure

(atomic-struct? x) → boolean?

  x : any/c
标识 (current-inspector) 不能检查的结构。

procedure

(atom? x) → boolean?

  x : any/c
标识可能出现在 Racklog 程序中的原子值。 相当于合同 (or/c boolean? number? string? bytes? char? symbol? regexp? pregexp? byte-regexp? byte-pregexp? keyword? null? procedure? void? set? atomic-struct?) 。

procedure

(compound-struct? x) → boolean?

  x : any/c
识别 (current-inspector) 可以检查的结构。

procedure

(compound? x) → boolean?

  x : any/c
识别可能出现在 Racklog 程序中的复合值。 相当于合同 (or/c pair? vector? mpair? box? hash? compound-struct?) 。

procedure

(unifiable? x) → boolean?

  x : any/c
识别可能出现在 Racklog 程序中的值。 本质上是一个 atom?, logic-var? 或包含 unifiable? 的 compound? 。

procedure

(answer-value? x) → boolean?

  x : any/c
识别可能出现在 answer? 中的值。 基本上是不包含 logic-var? 的 unifiable? 。

procedure

(answer? x) → boolean?

  x : any/c
识别由 %more 和 %which 返回的答案。 相当于合同 (or/c false/c (listof (cons/c symbol? answer-value?))) 。

value

goal/c : contract?

目标契约。

12.2 用户界面

syntax

(%which (V ...) G ...)

 
  V : identifier?
  G : goal/c
返回一个满足所有 G, ... 的变量 V, ...的 answer?， 如果 G, ..., 不能被满足,返回 #f 。 调用 thunk %more 会产生更多的实例,如果有的话。

procedure

(%more) → answer?

thunk %more 会产生更多最近的 %which-form 中的变量的实例, 以满足该 %which-form 的目标。 如果找不到更多的解决方案, %more 会返回 #f 。

syntax

(%find-all (V ...) G ...)

 
  V : identifier?
  G : goal/c
就像 (list (%which (V ...) G ...) (%more) ...) , 有多少个 (%more) 就有多少个答案。 (如果有无限多的答案,这不会终止。)

12.3 关系

syntax

(%rel (V ...) clause ...)

 
clause = [(E ...) G ...]
 
  V : identifier?
  E : expression?
  G : goal/c
返回一个谓词函数。每个子句 C 表示,如果所有的目标 G, ... 都能反过来被证明是成功的, 那么通过将谓词对象应用于任何与 (E ...) 相匹配的目标就被认为是成功的。 变量 V, ..., 是 clause, .... 的局部逻辑变量。

procedure

(%empty-rel E ...) → goal/c

  E : unifiable?
目标 (%empty-rel E ...) 总是失败。 value %empty-rel 被用作谓词的起始值, 以后可以用 %assert! 和 %assert-after! 加强。

syntax

(%assert! Pname (V ...) clause ...)

 
  Pname : identifier?
  V : identifier?
将子句 clauses, ..., 添加到谓词的 end ,即 Racket 变量 Pname 的值。 变量 V, ..., 是 clause, .... 的局部逻辑变量。

syntax

(%assert-after! Pname (V ...) clause ...)

 
  Pname : identifier?
  V : identifier?
就像 %assert! 一样,但是将新的子句添加到现有谓词的 front 。

12.4 Racklog 变量

procedure

(_) → logic-var?

一个产生新逻辑变量的 thunk。 可以用在我们想要一个逻辑变量但不想给它命名的情况下。 (与此相反,%let 为其创建的逻辑变量引入了新的词法名称。)

syntax

(%let (V ...) expr ...)

 
  V : identifier?
引入 V, ..., 作为词法范围内的逻辑变量,用于 expr, ...

12.5 剪切

syntax

(%cut-delimiter . any)

引入了一个切点。参见 剪切 (!) 。

syntax

!

切割目标,见 剪切 (!) 。

只能在 %cut-delimiter 或 %rel 内语法上使用。

12.6 Racklog 操作符

value

%fail : goal/c

目标 %fail 总是失败。

value

%true : goal/c

目标 %true 成功。重试时失败。

procedure

(%repeat) → goal/c

目标 (%repeat) 总是成功的(即使重试)。 对失败驱动的循环有用。

syntax

(%and G ...)

 
  G : goal/c
目标 (%and G ...) 成功,如果所有的目标 G, ..., 都成功。

syntax

(%or G ...)

 
  G : goal/c
目标 (%or G ...) 如果 G, ... 中的一个,按这个顺序尝试,就会成功。

procedure

(%not G) → goal/c

  G : goal/c
如果 G 失败,目标 (%not G) 就成功。

procedure

(%if-then-else G1 G2 G3) → goal/c

  G1 : goal/c
  G2 : goal/c
  G3 : goal/c
目标 (%if-then-else G1 G2 G3) 首先尝试 G1 : 如果它成功了,就尝试 G2; 如果不成功,就尝试 G3 。

12.7 统一

procedure

(%= E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 能与 E2 统一,则目标 (%= E1 E2) 成功。 任何产生的逻辑变量的绑定都被保留。

procedure

(%/= E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
%/= 是 %= 的否定。 如果 E1 不能与 E2 统一,则目标 (%/= E1 E2) 成功。

procedure

(%== E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 与 E2 identical,则目标 (%== E1 E2) 成功。 它们在结构上应该是相等的。 如果包含逻辑变量,它们应该在同一位置有相同的变量。 与 %=- 调用不同,这个目标不会绑定任何逻辑变量。

procedure

(%/== E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
%/== 是 %== 的否定。 如果 E1 和 E2 不完全相同,则目标 (%/== E1 E2) 成功。

syntax

(%is E1 E2)

目标l (%is E1 E2) 将 E2 作为 Racket 表达式进行评估的结果与 E1 统一。 E2 可能包含逻辑变量,这些变量会被自动取消引用。 如果 E2 包含非绑定的逻辑变量,则失败。

parameter

(use-occurs-check?) → boolean?

(use-occurs-check? on?) → void?
  on? : boolean?
如果这是错误的(默认),Racklog 的统一将不使用发生检查。 如果它是 true,则发生检查被启用。

12.8 数字谓词

procedure

(%< E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 和 E2 被绑定为数字且 E1 小于 E2, 则目标 (%< E1 E2) 成功。

procedure

(%<= E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 和 E2 都是数字,并且 E1 小于或等于 E2, 则目标 (%<= E1 E2) 成功。

procedure

(%=/= E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 和 E2 都是数字,且 E1 不等于 E2, 则目标 (%=/= E1 E2) 成功。

procedure

(%=:= E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 和 E2 都是数字,并且 E1 等于 E2, 则目标 (%=:= E1 E2) 成功。

procedure

(%> E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 和 E2 都是数字,并且 E1 大于 E2, 则目标 (%> E1 E2) 成功。

procedure

(%>= E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 和 E2 都是数字,并且 E1 大于或等于 E2, 则目标 (%>= E1 E2) 成功。

12.9 列表谓词

procedure

(%append E1 E2 E3) → goal/c

  E1 : unifiable?
  E2 : unifiable?
  E3 : unifiable?
如果 E3 可以与通过追加 E1 和 E2 得到的列表统一, 则目标 (%append E1 E2 E3) 成功。

procedure

(%member E1 E2) → goal/c

  E1 : unifiable?
  E2 : unifiable?
如果 E1 是 E2 中列表的一个成员,则目标 (%member E1 E2) 成功。

12.10 集合谓词

procedure

(%set-of E1 G E2) → goal/c

  E1 : unifiable?
  G : goal/c
  E2 : unifiable?
目标 (%set-of E1 G E2) 与 E2 统一, 即目标 G 成功的 E1 的所有实例的 set 。

procedure

(%set-of-1 E1 G E2) → goal/c

  E1 : unifiable?
  G : goal/c
  E2 : unifiable?
与 %set-of 类似,但是如果集合是空的,则失败。

procedure

(%bag-of E1 G E2) → goal/c

  E1 : unifiable?
  G : goal/c
  E2 : unifiable?
目标 (%bag-of E1 G E2) 与 E2 统一了目标 G 成功的 E1 的所有实例的 bag (多集)。

procedure

(%bag-of-1 E1 G E2) → goal/c

  E1 : unifiable?
  G : goal/c
  E2 : unifiable?
与 %bag-of 类似,但如果包是空的,则失败。

syntax

(%free-vars (V ...) G)

 
  V : identifier?
  G : goal/c
将目标 G 中的变量 V, ... 的出现识别为自由。 它被用来避免在调用集合谓词 (%bag-of, %set-of, 等等)时的存在性量化。

12.11 Racklog 谓词

procedure

(%compound E) → goal/c

  E : unifiable?
如果 E 是一个复合值,目标 (%compound E) 就会成功。

procedure

(%constant E) → goal/c

  E : unifiable?
如果 E 是一个原子值,目标 (%constant E) 就会成功。

procedure

(%var E) → goal/c

  E : unifiable?
如果 E 没有完全实例化,即它至少有一个未绑定的变量,则目标 (%var E) 成功。

procedure

(%nonvar E) → goal/c

  E : unifiable?
%nonvar 是 %var 的否定值。 如果 E 是完全实例化的,即其中没有未绑定的变量,那么目标 (%nonvar E) 就会成功。

12.12 高阶谓词

procedure

(%apply P L) → goal/c

  P : unifiable?
  L : unifiable?
如果 L 是一个有元素 E,...的列表, 如果 P 是一个接受与 E一样多的参数的谓词,并且如果目标 (P E ...) 成功, 则目标 (%apply P L) 成功。

如果 L 没有被充分实例化以确定其长度,则该目标将失败。

例如,目标

(%apply %= (list 1 X))

相当于

(%= 1 X)

如果 X 能与 1 统一,则成功。

procedure

(%andmap P L ...+) → goal/c

  P : unifiable?
  L : unifiable?
如果所有的值 L,......,都是等长的列表,目标 (%andmap P L ...) 就会成功, 如果目标 (P E ...) 从每个列表 L,......中依次取出的每组值 E, ... 就会成功。

作为一个例子,特别是目标 (%andmap %<= '(1 2 3) '(4 5 6)) 相当于
(%and (%<= 1 4)
      (%<= 2 5)
      (%<= 3 6))

12.13 Racklog 变量操作

procedure

(%freeze S F) → goal/c

  S : unifiable?
  F : unifiable?
目标 (%freeze S F) 将 S 中的结构与 F 统一为一个新的冻结版本。 冻结意味着所有未绑定的变量都被保留下来。 此后, F 可以被用作 bound 对象,而不用担心其变量会被统一绑定。

procedure

(%melt F S) → goal/c

  F : unifiable?
  S : unifiable?
目标 (%melt F S) 将 S 与 F 中冻结结构的解冻(原始)形式统一起来。

procedure

(%melt-new F S) → goal/c

  F : unifiable?
  S : unifiable?
目标 (%melt-new F S) 将 S 与 F 中冻结结构的解冻 copy 统一起来。 这意味着新的逻辑变量被用于 F 中的非绑定逻辑变量。

procedure

(%copy F S) → goal/c

  F : unifiable?
  S : unifiable?
目标 (%copy F S) 将 S 与 F 中冻结结构的副本统一起来。
