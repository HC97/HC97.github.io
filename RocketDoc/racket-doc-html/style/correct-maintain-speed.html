1 生命的基本事实

偏爱读者而不是作家。 — Yaron Minsky, JaneStreet, 2010 at NEU/CCS

这种排序偶尔也是错误的。 例如,我们几乎可以在所有的时候都避免 IEEE 浮点数。 为了精确起见,Racket 的 sqrt 函数可以返回一个与 IEEE 浮点数结果接近的有理数。 然而,我们不做这种傻事,因为我们已经决定在这种情况下重视速度而不是精度。 努力写出正确的;可维护的;和快速的代码。 这些形容词的排序很关键:正确比可维护更重要;可维护比快速更重要;而快速是很重要的,因为没有人愿意和慢的程序一起生活。

就 Racket 代码库而言,本节将解释这三点。 本指南的其余部分将阐述一些建议,这些建议应有助于你对 Racket 代码库做出正确、可维护和快速的贡献。

1.1 正确性和测试

我有错误报告,所以我存在。 – Matthias, watching Matthew, Robby, Shriram 和其他人创建的原始代码库

决定我们性格的是我们选择与 bug 斗争的方式,而不是它们的存在与否。 – Robby, in response

PLT 的目标是发布好的代码,并尽可能快地消除错误。所有的软件都有错误;完全正确是一个完美主义的目标。 如果错误是未知的,那么软件就没有被使用。 然而,我们的目标是在一个功能发布之前确保一些基本的正确性,并确保同样的错误不再被引入。

我们通过大型的测试套件来确保这种基本的正确性。我们的测试套件包含所有级别的测试。 除了单元测试,你会发现测试套件使用了 "随机测试" 策略和工具,还有一些使用了模糊测试, 还有一些是端到端的 "系统级" 测试,DrRacket 还附带了一个探索其功能的自动 GUI 播放器。

关于 Racket 代码库中的测试细节,请参见 测试 。

1.2 维护

如果我们希望创建可维护的代码,我们必须确保我们的代码是可理解的。 当你能理解它的外部目的时,当你能从它的外部目的中猜到它的组织时,当组织和代码达到一致的风格标准时, 当偶尔的复杂部分带有内部文档时,代码就是可理解的。

发布的代码必须有文档。 反之,对代码的外部行为的改变必须引起对其文档的同步改变。 这里的 "同时" 是指这两个改变是在同一个 "推送" 到代码库中,而不一定是在同一个 "提交" 中。 更多关于 Git 操作的信息请参见 礼仪: 分支机构和承诺 。

关于记录代码的风格规则,请参考 style guide in the Scribble manual 。 理想情况下,文档分为两部分,可能位于同一个文档中:"指南" 部分,解释目的并建议使用案例,以及传统的 "参考" 部分,介绍细枝末节的内容。 HtDP/2e 教学包的文档就是一个例子,这两部分被放在一起。 还可以考虑在你的 "参考资料" 部分增加每个功能和结构的例子。 最后,确保你有所有正确的 for-label require 并利用其他有用的交叉引用。

说了这么多,像 Racket 这样的系统的制作偶尔也需要实验。 不过,一旦我们理解了这些新的功能,就必须抛弃实验中的 "失败分支",把成功的部分变成可维护的包。 你甚至可以考虑最终将你的代码转换为 Typed Racket。

如果不遵守基本的风格元素,代码的理解就变得不可能。 本文档的其余部分主要是关于这些风格元素的,包括一些关于最小内部文档的建议。

1.3 速度

使代码快速是一项无止境的任务。 让代码 reasonably fast 是我们的目标。

与正确性一样,性能需要一些 "测试"。 至少要在一些合理的现实的输入和一些较大的输入上锻炼你的代码。在测试套件中添加一个文件,定期运行大型输入。 例如,Universe 显示器的常规测试套件处理一个 50 x 50 的显示窗口; 其中一个压力测试检查 Universe 事件处理程序和绘图例程是否能应付笔记本电脑大小的显示器,甚至是 30 英寸的显示器。 或者,如果你为一个队列数据结构编写一个库,常规测试套件确保它能正确处理小队列(包括空队列)的 enqueue 和 dequeue; 同一个库的压力测试套件将在各种队列大小上运行队列操作,包括非常大的队列,例如几万个元素。

压力测试通常没有预期的输出,所以它们永远不会通过。 编写压力测试的做法暴露了实现的缺陷,或者提供了比较数据,以便在两个 API 之间进行选择。 仅仅是编写和保留它们,就可以提醒我们,事情可能会变坏,我们可以通过一些其他的门来检测性能下降的情况。 最重要的是,压力测试可能会发现,你的代码并没有实现预期的 O(.) 运行时间的算法。 仅仅发现这一点就很有用。 如果你想不出改进的办法,就把外部库中的弱点记录下来,然后继续前进。

当你继续阅读时,请记住,我们不是完美主义者。 我们生产合理的软件。
