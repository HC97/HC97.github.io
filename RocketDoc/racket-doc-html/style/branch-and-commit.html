8 礼仪: 分支机构和承诺

本节是专门为提交 Racket 代码库的 Rackete 们准备的。

与错误数据库一起工作需要一个关键的工作流程规则。

在代码库中工作需要对仓库的行动制定风格规则。 目前我们使用的是 Git,下面是一些关于在这种情况下如何行动的规则。

8.1 错误修复工作流程

只有在你能消除自己的代码是错误的源头之后,才重新分配错误报告。 实现这一目标的最好方法是创建一个新的例子,在不涉及你的代码的情况下重新创建问题。 当你有这样一个代码片段时,将代码重新分配给负责明显有缺陷的组件的人,并提交代码片段作为理由的一部分。

8.2 提交

新特性的提交: 按照你的意愿提交新特性、它的测试和它的文档,但请将它们一起推送。 但是,不要提交不运行的状态。 (在 Git 中,这意味着 "提交" 而不仅仅是 "推送")。

错误修复提交: 当你修复一个 bug 时,请确保一次性提交(1)代码 delta,(2)新的测试用例,以及(3)修改后的文档(如果适用)。 如果单个提交的创建过于复杂,或者你希望剔除其中一个提交的因素,请一次性推送所有部分。 这样一来,代码库始终处于代码、测试和文档同步的状态,提交信息的读者也可以完全评估修改。

样式变更提交: 将对文件风格的修改与对其行为的修改(新功能、bug)分开提交。

编写有意义的提交信息。 第一行(比如 72 个字符)应该提供一个简洁的提交摘要。 如果信息必须更长,可以在文本编辑器中编辑信息的其余部分,并在摘要行和信息的其余部分之间留出一行空白,像这样:

  some quick description

  

  more blah blah blah, with more

  details about the actual change

空行的好处是 git log 和其他工具可以正确显示提交信息。 如果你喜欢用 -m 命令行标志而不是编辑器,你可以连续使用几个。

错误报告修复的消息应该包含 “Close PR NNNNN” ,这样错误报告就会自动关闭。

为了避免 "合并提交",用 git --rebase pull 更新你的版本库。

8.3 请不要提交 "炸弹"。

有时,你会发现你花了大量的时间来处理别人的代码。 为了避免潜在的痛苦合并,请(1)在创建分支时通知作者,(2)设置邮件钩子,让 git 向你和原作者发送提交消息。 此外,你应该在实际的代码库中测试你的修改。 在某些情况下,推迟这种测试是可以接受的,例如,当你在很长一段时间内不知道性能影响是否允许提交到 PLT 仓库时。
