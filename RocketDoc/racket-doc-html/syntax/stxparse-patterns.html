1.5 语法模式

下表给出了 syntax/parse 设施所使用的 syntax patterns 的语法。 有四种主要的语法模式:

  • single-term patterns, 缩写为 S-pattern

  • head patterns, 缩写为 H-pattern

  • ellipsis-head patterns, 缩写为 EH-pattern

  • action patterns, 缩写为 A-pattern

第五种, list patterns (缩写为 L-pattern), 只是 single-term patterns 的一个语法上的限制性子集。

当本手册中的特殊形式提到 syntax-pattern 时(例如,对 syntax-parse 特殊形式的描述), 它特指 single-term pattern 。

  S-pattern = pvar-id
  | pvar-id:syntax-class-id
  | pvar-id:literal-id
  | literal-id
  | (~vars- id)
  | (~vars+ id syntax-class-id maybe-role)
  | (~vars+ id (syntax-class-id arg ...) maybe-role)
  | (~literal literal-id maybe-phase)
  | atomic-datum
  | (~datum datum)
  | (H-pattern . S-pattern)
  | (A-pattern . S-pattern)
  | (EH-pattern ... . S-pattern)
  | (H-pattern ...+ . S-pattern)
  | (~ands proper-S/A-pattern ...+)
  | (~or*s S-pattern ...+)
  | (~not S-pattern)
  | #(pattern-part ...)
  | #s(prefab-struct-key pattern-part ...)
  | #&S-pattern
  | (~rest S-pattern)
  | (~describes maybe-opaque maybe-role expr S-pattern)
  | (~commits S-pattern)
  | (~delimit-cuts S-pattern)
  | (~posts S-pattern)
  | A-pattern
     
  L-pattern = ()
  | (A-pattern . L-pattern)
  | (H-pattern . L-pattern)
  | (EH-pattern ... . L-pattern)
  | (H-pattern ...+ . L-pattern)
  | (~rest L-pattern)
     
  H-pattern = pvar-id:splicing-syntax-class-id
  | (~varh id splicing-syntax-class-id maybe-role)
  | 
(~varh id (splicing-syntax-class-id arg ...)
       maybe-role)
  | (~seq . L-pattern)
  | (~andh proper-H/A-pattern ...+)
  | (~or*h H-pattern ...+)
  | (~optionalh H-pattern maybe-optional-option)
  | (~describeh maybe-opaque maybe-role expr H-pattern)
  | (~commith H-pattern)
  | (~delimit-cuth H-pattern)
  | (~posth H-patter)
  | (~peek H-pattern)
  | (~peek-not H-pattern)
  | proper-S-pattern
     
  EH-pattern = (~alt EH-pattern ...)
  | (~once H-pattern once-option ...)
  | (~optionaleh H-pattern optional-option ...)
  | (~between H min-number max-number between-option)
  | H-pattern
     
  A-pattern = ~!
  | (~bind [attr-arity-decl expr] ...)
  | (~fail maybe-fail-condition maybe-message-expr)
  | (~parse S-pattern stx-expr)
  | (~anda A-pattern ...+)
  | (~posta A-pattern)
  | (~do defn-or-expr ...)
  | (~undo defn-or-expr ...)
     
  proper-S-pattern = S-pattern 这不是一个 A-pattern
     
  proper-H-pattern = H-pattern 这不是一个 S-pattern

以下模式关键字可以在多个模式变体中使用:

syntax

~var

~vars-, ~vars+ 或 ~varh 中的一个。

syntax

~and

~ands, ~andh 或 ~anda 中的一个:

  • ~anda 如果所有的连接词都是 action patterns

  • ~andh 如果任何一个联结词是 proper head pattern

  • ~ands 否则

syntax

~or*

~or*s 或 ~or*h 中的一个:

  • ~or*h 如果任何一个不连接点是一个 proper head pattern

  • ~or*s 否则

syntax

~or

表现为像 ~or*s, ~or*h 或 ~alt:

  • 如果该模式直接出现在省略号 (...) 之前, 或紧接在另一个 ~alt 模式中,则像 ~alt 。

  • 像 ~or*h 一样,如果任何一个不连贯体是一个 proper head pattern

  • 否则就像 ~or*s

对 ~or 的上下文敏感的解释是一个设计错误,也是一个常见的混淆来源。 使用 ~alt 和 ~or* 代替。

syntax

~describe

~describes 可 ~describeh 中的一个:

  • ~describeh 如果该子模式是一个 proper head pattern

  • ~describes 否则

syntax

~commit

~commits 或 ~commith 中的一个:

  • ~commith 如果子模式是一个 proper head pattern

  • ~commits 否则

syntax

~delimit-cut

~delimit-cuts 或 ~delimit-cuth 中的一个:

  • ~delimit-cuth 如果子模式是一个 proper head pattern

  • ~delimit-cuts 否则

syntax

~post

~posts, ~posth 或 ~posta 中的一个:

  • ~posta 如果子模式是一个 action pattern

  • ~posth 如果子模式是一个 proper head pattern

  • ~posts 否则

syntax

~optional

~optionalh 或 ~optionaleh 中的一个:

  • ~optionaleh 如果它是一个 ~alt 模式的直接不连接点

  • ~optionalh 否则

1.5.1 单词模式

single-term pattern (缩写为 S-pattern) 是描述单一术语的模式。 这些模式与 syntax-rules 和 syntax-case 中使用的传统模式一样, 但有额外的变体,使其更具表现力。

“Single-term” 并不意味着 “atomic”; single-term pattern 可以有复杂的结构,它可以匹配有许多部分的术语。 例如, (17 ...) 是一个 single-term pattern , 它可以匹配任何重复 17 个数字的适当列表的术语。

一个 proper single-term pattern 是指不是 action pattern 。

list patterns (代表 “list pattern”) 是具有限制性结构的 single-term patterns , 保证它们只匹配适当列表的术语。

以下是 single-term pattern 的变体:

id

一个标识符可以是一个 pattern variable, 一个 annotated pattern variable ,或者一个 literal:

  • 如果 id 是 literals 列表中一个条目的 “pattern” 名称, 它就是一个 literal 模式,其行为类似于 (~literal id) 。

    Examples:
    > (syntax-parse #'(define x 12)
        #:literals (define)
        [(define var:id body:expr) 'ok])

    'ok

    > (syntax-parse #'(lambda x 12)
        #:literals (define)
        [(define var:id body:expr) 'ok])

    lambda: expected the identifier `define'

      at: lambda

      in: (lambda x 12)

    > (syntax-parse #'(define x 12)
        #:literals ([def define])
        [(def var:id body:expr) 'ok])

    'ok

    > (syntax-parse #'(lambda x 12)
        #:literals ([def define])
        [(def var:id body:expr) 'ok])

    lambda: expected the identifier `define'

      at: lambda

      in: (lambda x 12)

  • 如果 id 的形式是 pvar-id:syntax-class-id (也就是两个用冒号连接的名字),它就是一个 annotated pattern variable, 该模式等同于 (~var pvar-id syntax-class-id) 。

    Examples:
    > (syntax-parse #'a
        [var:id (syntax-e #'var)])

    'a

    > (syntax-parse #'12
        [var:id (syntax-e #'var)])

    ?: expected identifier

      at: 12

      in: 12

    > (define-syntax-class two
        #:attributes (x y)
        (pattern (x y)))
    > (syntax-parse #'(a b)
        [t:two (syntax->datum #'(t t.x t.y))])

    '((a b) a b)

    > (syntax-parse #'(a b)
        [t
         #:declare t two
         (syntax->datum #'(t t.x t.y))])

    '((a b) a b)

    注意 :syntax-class-id 形式的 id 是合法的; 见关于 ~vars+ 形式的零长度的 pvar-id 的讨论。

  • 如果 id 的形式是 pvar-id:literal-id, 其中 literal-id 在 literals 列表中, 那么它就等同于 (~and (~var pvar-id) literal-id) 。

    Examples:
    > (require (only-in racket/base [define def]))
    > (syntax-parse #'(def x 7)
        #:literals (define)
        [(d:define var:id body:expr) #'d])

    #<syntax:eval:11:0 def>

  • 否则, id 是一个 pattern variable, 该模式等同于 (~var id) 。

(~var pvar-id)

一个 pattern variable 。 如果 pvar-id 没有语法类(根据 #:convention), 模式变量可以匹配任何东西。 除非模式变量是通配符 (_), 在这种情况下不发生绑定,否则模式变量将被绑定到匹配的子项上。

如果 pvar-id 确实有一个相关的语法类, 那么它的行为就像一个插入了隐含语法类的 annotated pattern variable 。

(~var pvar-id syntax-class-use maybe-role)
 
syntax-class-use = syntax-class-id
  | (syntax-class-id arg ...)
     
maybe-role = 
  | #:role role-expr
 
  role-expr : (or/c string? #f)

一个 annotated pattern variable 。 该模式只与 syntax-class-id 所接受的术语相匹配(如果存在的话,由 arg 参数化)。

除了绑定 pvar-id 之外,注解的模式变量还绑定了来自语法类的 nested attributes 。 嵌套属性的名称是由 pvar-id. 的前缀构成的。 (即 pvar-id 后面跟一个 “dot” 字符)到语法类的属性名称中。

如果 pvar-id 是 _, 则没有属性被绑定。 如果 pvar-id 是零长度的标识符 (||), 那么 pvar-id 不被绑定,但是 syntax-class-use 的嵌套属性被绑定,没有前缀。

如果 role-expr 被给出并且被评估为一个字符串,那么在错误信息中它将与语法类的描述相结合。

Examples:
> (syntax-parse #'a
    [(~var var id) (syntax-e #'var)])

'a

> (syntax-parse #'12
    [(~var var id) (syntax-e #'var)])

?: expected identifier

  at: 12

  in: 12

> (define-syntax-class two
    #:attributes (x y)
    (pattern (x y)))
> (syntax-parse #'(a b)
    [(~var t two) (syntax->datum #'(t t.x t.y))])

'((a b) a b)

> (define-syntax-class (nat-less-than n)
    (pattern x:nat #:when (< (syntax-e #'x) n)))
> (syntax-parse #'(1 2 3 4 5)
    [((~var small (nat-less-than 4)) ... large:nat ...)
     (list #'(small ...) #'(large ...))])

'(#<syntax:eval:17:0 (1 2 3)> #<syntax:eval:17:0 (4 5)>)

> (syntax-parse #'(m a b 3)
    [(_ (~var x id #:role "variable") ...) 'ok])

m: expected identifier for variable

  at: 3

  in: (m a b 3)

(~literal literal-id maybe-phase)
 
maybe-phase = 
  | #:phase phase-expr

一个 literal 标识符模式。 匹配任何标识符 free-identifier=? 到 literal-id 。

Examples:
> (syntax-parse #'(define x 12)
    [((~literal define) var:id body:expr) 'ok])

'ok

> (syntax-parse #'(lambda x 12)
    [((~literal define) var:id body:expr) 'ok])

lambda: expected the identifier `define'

  at: lambda

  in: (lambda x 12)

如果给定了 phase-expr, 标识符将在 phase-expr 给出的阶段进行比较, 否则将在 (syntax-local-phase-level) 进行比较。

atomic-datum

数字、字符串、布尔运算、关键字和空列表作为字面符号进行匹配。

Examples:
> (syntax-parse #'(a #:foo bar)
    [(x #:foo y) (syntax->datum #'y)])

'bar

> (syntax-parse #'(a foo bar)
    [(x #:foo y) (syntax->datum #'y)])

a: expected the literal #:foo

  at: foo

  in: (a foo bar)

(~datum datum)

匹配其S表达式内容(通过 syntax->datum 获得)与给定的 datum equal? 的语法。

Examples:
> (syntax-parse #'(a #:foo bar)
    [(x (~datum #:foo) y) (syntax->datum #'y)])

'bar

> (syntax-parse #'(a foo bar)
    [(x (~datum #:foo) y) (syntax->datum #'y)])

a: expected the literal #:foo

  at: foo

  in: (a foo bar)

~datum 形式对于识别符号很有用,与 ~literal 形式相反,后者通过绑定来识别它们。

Examples:
> (define-syntax (is-define? stx)
    (syntax-parse stx
      [(is-define? id)
       (syntax-parse #'id
         [(~literal define) #''yes]
         [(~datum   define) #''not-really]
         [_                 #''not-even-close])]))
> (is-define? define)

'yes

> (let ([define 42])
    (is-define? define))

'not-really

> (is-define? something-else)

'not-even-close

(H-pattern . S-pattern)

匹配任何可以被分解成与 H-pattern 匹配的列表前缀和与 S-pattern 匹配的后缀的术语。

注意,该模式可以匹配甚至不是不恰当的列表的术语; 如果头部模式可以匹配零长度的头部,那么整个模式可以匹配尾部模式接受的任何内容。

第一个模式可以是一个 single-term pattern, 在这种情况下, 整个模式匹配任何第一元素与第一个模式相匹配、其余元素与第二个模式相匹配的对。

更多信息见 head patterns 。

(A-pattern . S-pattern)

执行 A-pattern 所指定的动作,然后匹配任何与 S-pattern 相匹配的术语。

实事求是地说,我们可以把一个 action pattern 扔到任何列表模式中。 因此, (x y z) 是一个匹配三个术语的列表的模式, 而 (x y ~! z) 是一个匹配三个术语的列表的模式,在第二个术语之后进行 cut 。 换句话说, action patterns “don’t take up space.”

更多信息请参见 action patterns 。

(EH-pattern ... . S-pattern)

匹配任何可以分解成一个列表头的术语, 该列表头匹配 EH-pattern 的一些重复的替代品(受其重复限制), 后面是一个列表尾匹配 S-pattern 。

换句话说,整个模式要么与第二个模式(不需要是一个列表)相匹配, 要么是一个头部与第一个模式的一个备选方案相匹配、尾部与整个序列模式递归匹配的术语。

更多信息见 ellipsis-head patterns 。

(H-pattern ...+ . S-pattern)

和省略号 (...) 模式一样,但要求头部模式至少有一次出现。

也就是说,以下模式是等价的:

  • (H ...+ . S)

  • ((~between H 1 +inf.0) ... . S)

Examples:
> (syntax-parse #'(1 2 3)
    [(n:nat ...+) 'ok])

'ok

> (syntax-parse #'()
    [(n:nat ...+) 'ok]
    [_ 'none])

'none

(~and S/A-pattern ...)

匹配任何符合所有子模式的术语。

子模式可以包含 single-term patterns 和 action patterns 的混合,但必须至少包含一个 single-term pattern 。

在子模式中绑定的属性对后续的子模式是可用的。整个模式绑定了所有子模式的属性。

~and-patterns 的一个用途是保留整个术语(包括其词汇上下文、源位置等),同时也检查其结构。 语法类对于同样的目的是有用的,但 ~and 可以更轻巧。

Examples:
> (define-syntax (import stx)
    (raise-syntax-error #f "illegal use of import" stx))
> (define (check-imports stx) ....)
> (syntax-parse #'(m (import one two))
    #:literals (import)
    [(_ (~and import-clause (import i ...)))
     (let ([bad (check-imports
                 (syntax->list #'(i ...)))])
       (when bad
         (raise-syntax-error
          #f "bad import" #'import-clause bad))
       'ok)])

'ok

(~or* S-pattern ...)

匹配任何与所含模式之一相匹配的术语。 备选方案将按顺序进行尝试。

整个模式绑定了 所有 子模式的属性。 没有被 "所选 "子模式绑定的属性,其值为 #f 。 同一个属性可能被多个子模式绑定,如果它被所有的子模式绑定,那么如果整个模式匹配,它肯定会有一个值。

Examples:
> (syntax-parse #'a
    [(~or* x:id y:nat) (values (attribute x) (attribute y))])

#<syntax:eval:34:0 a>

#f

> (syntax-parse #'(a 1)
    [(~or* (x:id y:nat) (x:id)) (values #'x (attribute y))])

#<syntax:eval:35:0 a>

#<syntax:eval:35:0 1>

> (syntax-parse #'(b)
    [(~or* (x:id y:nat) (x:id)) (values #'x (attribute y))])

#<syntax:eval:36:0 b>

#f

(~not S-pattern)

匹配任何与子模式不匹配的术语。 没有一个子模式的属性被绑定在 ~not-pattern 之外。

Example:
> (syntax-parse #'(x y z => u v)
    #:literals (=>)
    [((~and before (~not =>)) ... => after ...)
     (list #'(before ...) #'(after ...))])

'(#<syntax:eval:37:0 (x y z)> #<syntax:eval:37:0 (u v)>)

#(pattern-part ...)

匹配一个是向量的术语,其元素在被视为列表时, 与对应于 (pattern-part ...) 的 single-term pattern 相匹配。

Examples:
> (syntax-parse #'#(1 2 3)
    [#(x y z) (syntax->datum #'z)])

3

> (syntax-parse #'#(1 2 3)
    [#(x y ...) (syntax->datum #'(y ...))])

'(2 3)

> (syntax-parse #'#(1 2 3)
    [#(x ~rest y) (syntax->datum #'y)])

'(2 3)

#s(prefab-struct-key pattern-part ...)

匹配一个预制结构的术语,该预制结构的键正好是给定的键,并且其字段序列, 当被视为一个列表时,与 (pattern-part ...) 对应的 single-term pattern 相匹配。

Examples:
> (syntax-parse #'#s(point 1 2 3)
    [#s(point x y z) 'ok])

'ok

> (syntax-parse #'#s(point 1 2 3)
    [#s(point x y ...) (syntax->datum #'(y ...))])

'(2 3)

> (syntax-parse #'#s(point 1 2 3)
    [#s(point x ~rest y) (syntax->datum #'y)])

'(2 3)

#&S-pattern

匹配的术语是一个盒子,其内容与内部的 single-term pattern 相匹配。

Example:
> (syntax-parse #'#&5
    [#&n:nat 'ok])

'ok

(~rest S-pattern)

与 S-pattern 一样匹配。 ~rest 模式形式在读者不允许不适当的("点")列表的位置上是有用的,如向量和结构模式(见上文)。

Examples:
> (syntax-parse #'(1 2 3)
    [(x ~rest y) (syntax->datum #'y)])

'(2 3)

> (syntax-parse #'#(1 2 3)
    [#(x ~rest y) (syntax->datum #'y)])

'(2 3)

(~describe maybe-role maybe-opaque expr S-pattern)
 
maybe-opaque = 
  | #:opaque
     
maybe-role = 
  | #:role role-expr
 
  expr : (or/c string? #f)
  role-expr : (or/c string? #f)

~describe 模式形式用描述来注释模式,这是一个字符串表达式,在所有先前属性绑定的范围内被评估。 如果解析内部模式失败,那么描述被用来合成错误信息。 一个 ~describe 模式不会影响回溯。

如果给出 #:opaque ,来自 S-pattern 内部的失败信息将被丢弃,错误将仅以给出的描述来报告。

如果给出 role-expr 并产生一个字符串,其值将与错误信息中的描述相结合。

Examples:
> (syntax-parse #'(m 1)
    [(_ (~describe "id pair" (x:id y:id))) 'ok])

m: expected id pair

  at: 1

  in: (m 1)

> (syntax-parse #'(m (a 2))
    [(_ (~describe "id pair" (x:id y:id))) 'ok])

m: expected identifier

  at: 2

  in: (m (a 2))

  parsing context:

   while parsing id pair

    term: (a 2)

    location: eval:48:0

> (syntax-parse #'(m (a 2))
    [(_ (~describe #:opaque "id pair" (x:id y:id))) 'ok])

m: expected id pair

  at: (a 2)

  in: (m (a 2))

> (syntax-parse #'(m 1)
    [(_ (~describe #:role "formals" "id pair" (x y))) 'ok])

m: expected id pair for formals

  at: 1

  in: (m 1)

(~commit S-pattern)

~commit 模式形式以两种方式影响回溯:

  • 如果模式成功,那么所有在子模式内创建的选择点都会被丢弃, 而在 ~commit 模式 之后 的失败只回溯到 ~commit 模式 之前 的选择点, 而不会回溯到其 内部 的选择点。

  • 在一个 ~commit 模式内的切割 (~!) 只消除在 ~commit 模式内创造的选择点。 在这个意义上,它的作用就像 ~delimit-cut 。

(~delimit-cut S-pattern)

~delimit-cut 模式会以如下方式影响回溯:

  • 在 ~delimit-cut 模式中的切割 (~!) 只消除在 ~delimit-cut 模式中创建的选择点。

(~post S-pattern)

将子模式内的失败标记为发生在 "后序检查" 中;也就是说,它们被认为比正常的失败取得了更大的进展。

A-pattern

当没有歧义时,一个 action pattern 被认为是一个 single-term pattern ;它可以匹配任何术语。

1.5.2 标题模式

head pattern (缩写为 H-pattern) 是一个描述出现在某个列表(可能是一个不恰当的列表)头部的一些术语的模式。 head pattern 的用处来自于能够匹配不同长度的头部,例如像关键字参数这样的可选形式。

一个 proper head pattern 是一个不是 single-term pattern 的 head pattern 。

下面是 head pattern 的变体:

pvar-id:splicing-syntax-class-id

相当于 (~var pvar-id splicing-syntax-class-id) 。

(~var pvar-id splicing-syntax-class-use maybe-role)
 
splicing-syntax-class-use = splicing-syntax-class-id
  | (splicing-syntax-class-id arg ...)
     
maybe-role = 
  | #:role role-expr
 
  role-expr : (or/c string? #f)

带有 splicing syntax class 注释的模式变量。 类似于普通的 annotated pattern variable, 除了匹配一个头部模式。

(~seq . L-pattern)

匹配一个术语序列,其元素如果放在一个列表中,将匹配 L-pattern 。

Example:
> (syntax-parse #'(1 2 3 4)
    [((~seq 1 2 3) 4) 'ok])

'ok

关于 ~seq 的更多有趣的例子,请参见 ellipsis-head patterns 。

(~and H-pattern ...)

(~describe expr H-pattern)

与 single-term pattern 版本一样, ~describes, 但匹配一个头部模式。

(~commit H-pattern)

和 single-term pattern 版本一样, ~commits, 但是匹配一个头部模式。

(~delimit-cut H-pattern)

就像 single-term pattern 版本, ~delimit-cuts, 但是匹配一个头部模式。

(~post H-pattern)

像 single-term pattern 版本, ~posts, 但是匹配头部模式。

(~peek H-pattern)

匹配 H-pattern ,但随后重设匹配位置,所以 ~peek 模式不消耗输入。 用来在一个序列中向前看。

Examples:
> (define-splicing-syntax-class nf-id ; non-final id
    (pattern (~seq x:id (~peek another:id))))
> (syntax-parse #'(a b c 1 2 3)
    [(n:nf-id ... rest ...)
     (printf "nf-ids are ~s\n" (syntax->datum #'(n.x ...)))
     (printf "rest is ~s\n" (syntax->datum #'(rest ...)))])

nf-ids are (a b)

rest is (c 1 2 3)

(~peek-not H-pattern)

与 ~peek 类似,但如果子模式失败则成功,如果子模式成功则失败。 成功时, ~peek-not 重设匹配位置,因此该模式不消耗输入。 用来在一个序列中向前看。 在 ~peek-not-pattern 之外,没有任何子模式的属性被绑定。

Examples:
> (define-splicing-syntax-class final ; final term
    (pattern (~seq x (~peek-not _))))
> (syntax-parse #'(a b c)
    [((~alt f:final other) ...)
     (printf "finals are ~s\n" (syntax->datum #'(f.x ...)))
     (printf "others are ~s\n" (syntax->datum #'(other ...)))])

finals are (c)

others are (a b)

S-pattern

匹配一个元素的序列,该元素必须是一个术语匹配的 S-pattern 。

1.5.3 省略号-头部模式

ellipsis-head pattern (简称 EH-pattern) 是描述一些术语数量的模式, 就像 head pattern 一样,但也对它在重复中出现的次数进行了限制。 它们对匹配很有用,例如,关键词参数可能以任何顺序出现。 多个选项通过 ~alt 组合在一起。

Examples:
> (define parser1
    (syntax-parser
     [((~alt (~once (~seq #:a x) #:name "#:a keyword")
             (~optional (~seq #:b y) #:name "#:b keyword")
             (~seq #:c z)) ...)
      'ok]))
> (parser1 #'(#:a 1))

'ok

> (parser1 #'(#:b 2 #:c 3 #:c 25 #:a 'hi))

'ok

> (parser1 #'(#:a 1 #:a 2))

?: too many occurrences of #:a keyword

  at: ()

  within: (#:a 1 #:a 2)

  in: (#:a 1 #:a 2)

该模式要求 #:a 关键字和参数正好出现一次, #:b 关键字和参数最多出现一次, 以及任意数量的 #:c 关键字和参数。 这些 "片段" 可以以任何顺序出现。

下面是 ellipsis-head pattern 模式的变体:

(~alt EH-pattern ...)

如果任何内部的 EH-pattern 备选方案匹配,则匹配。

(~once H-pattern once-option ...)
 
once-option = #:name name-expr
  | #:too-few too-few-message-expr
  | #:too-many too-many-message-expr
 
  name-expr : (or/c string? #f)
  too-few-message-expr : (or/c string? #f)
  too-many-message-expr : (or/c string? #f)

如果内部的 H-pattern 匹配,则匹配。该模式在整个重复序列的匹配中必须完全匹配一次。

如果该模式在重复序列中没有被匹配,那么省略号模式就会失败, 消息是 too-few-message-expr 或 "missing required occurrence of name-expr" 。

如果该模式在重复序列中被选择了一次以上,那么省略号模式就会失败, 信息是 too-many-message-expr 或 "too many occurrences of name-expr" 。

(~optional H-pattern optional-option ...)
 
optional-option = #:name name-expr
  | #:too-many too-many-message-expr
  | #:defaults ([attr-id expr] ...)
 
  name-expr : (or/c string? #f)
  too-many-message-expr : (or/c string? #f)

如果内部 H-pattern 匹配,则匹配。 该模式在整个重复的匹配中最多可使用一次。

如果该模式在重复序列中被匹配超过一次,那么省略号模式就会失败, 信息是 too-many-message-expr 或 "too many occurrences of name-expr" 。

如果给了 #:defaults 选项,那么如果子模式在序列中完全不匹配,就会使用以下属性绑定。 默认属性必须是子模式的属性的子集。

(~between H-pattern min-number max-number between-option ...)
 
between-option = #:name name-expr
  | #:too-few too-few-message-expr
  | #:too-many too-many-message-expr
 
  name-expr : (or/c string? #f)
  too-few-message-expr : (or/c string? #f)
  too-many-message-expr : (or/c string? #f)

如果内部的 H-pattern 匹配,则进行匹配。 该模式必须在整个重复中至少匹配 min-number 和 max-number 次。

如果该模式被匹配的次数太少,那么省略号模式就会失败,如果提供了name-expr , 就会出现 too-few-message-expr 或 "too few occurrences of name-expr" 的信息。

如果该模式被选择的次数过多,那么省略号模式就会失败,信息要么是 too-many-message-expr , 要么是 "too many occurrences of name-expr", 当 name-expr 被提供时。

1.5.4 动作模式

一个 action pattern (缩写为 A-pattern) 并不描述任何语法; 相反,它有一个效果,如属性的绑定或匹配过程的修改。

~!

cut 操作符,写成 ~!, 消除了回溯选择点,并将解析提交给它正在探索的模式的当前分支。

切割模式的常见机会来自于识别基于关键词的特殊形式。 考虑一下下面的表达:

> (syntax-parse #'(define-values a 123)
    #:literals (define-values define-syntaxes)
    [(define-values (x:id ...) e) 'define-values]
    [(define-syntaxes (x:id ...) e) 'define-syntaxes]
    [e 'expression])

'expression

考虑到这个错误的术语 (define-values a 123), syntax-parse 尝试了第一个子句,未能与模式 (x:id ...) 匹配 a , 然后回溯到第二个子句,最终到第三个子句,产生了值 'expression 。 但是这个子句并不是一个表达式;它是对 define-values 的不规范使用。 syntax-parse 表达式的正确写法如下:

> (syntax-parse #'(define-values a 123)
    #:literals (define-values define-syntaxes)
    [(define-values ~! (x:id ...) e) 'define-values]
    [(define-syntaxes ~! (x:id ...) e) 'define-syntaxes]
    [e 'expression])

define-values: bad syntax

  in: (define-values a 123)

现在,给定同一个术语, syntax-parse 会尝试第一个子句, 由于关键字 define-values 是匹配的,所以 cut-pattern 会提交给当前的模式, 消除了第二和第三子句的选择点。 因此,当子句不能匹配时, syntax-parse 表达式会引发一个错误。

一个 ~! 模式的效果是由最近的封闭的 ~delimit-cut 或 ~commit 模式限定的。 如果没有封闭的 ~describe 模式,但切割发生在一个语法类定义中,那么只有语法类定义中的选择点被丢弃。 除非有 ~delimit-cut 或 ~commit 模式, 否则 ~! 模式不允许出现在 ~not 模式中。

(~bind [attr-arity-decl expr] ...)
 
attr-arity-decl = attr-name-id
  | (attr-name-id depth)

评估 expr 并将其与给定的 attr-id 绑定为属性。

(~fail maybe-fail-condition maybe-message-expr)
 
maybe-fail-condition = 
  | #:when condition-expr
  | #:unless condition-expr
     
maybe-message-expr = 
  | message-expr
 
  message-expr : (or/c string? #f)

如果条件不存在,或者如果 #:when 条件求值为一个真值, 或者如果 #:unless 条件求值为 #f, 那么该模式就会以给定的消息失败。 如果信息被省略,则使用默认值 #f 代表 “no message.”

失败模式可以与切割模式一起使用,以识别特定的不合格术语,并以自定义的失败信息来处理它们。

(~parse S-pattern stx-expr)

评估 stx-expr 并与 S-pattern 匹配。 如果 stx-expr 没有产生一个语法对象,该值将被隐含地转换为一个语法对象, 除非该转换会产生 3D syntax, 在这种情况下,将引发一个异常。

(~and A-pattern ...+)

执行每个 A-pattern 的动作。

(~do defn-or-expr ...)

接受一连串的定义和表达式,这些定义和表达式可能是混合的,并在所有先前的属性绑定的范围内对它们进行评估。 被定义绑定的名字在随后的模式和子句的表达式中都有范围。

目前还没有办法使用 ~do 模式来绑定属性。在 ~do 块中用定义影射属性绑定是一个错误。

Example:
> (syntax-parse #'(1 2 3)
    [(a b (~do (printf "a was ~s\n" #'a)) c:id) 'ok])

a was #<syntax:eval:71:0 1>

?: expected identifier

  at: 3

  in: (1 2 3)

(~undo defn-or-expr ...)

在最初匹配时没有影响,但是如果回溯到 ~undo 模式 之前 的一个点, defn-or-expr 被执行。 它们在所有先前的属性绑定的范围内被评估。

使用与 ~undo 配对的 ~do 来执行副作用,如果包围模式后来被丢弃了,则解除它们。

Examples:
> (define total 0)
> (define-syntax-class nat/add
    (pattern (~and n:nat
                   (~do (printf "adding ~s\n" (syntax-e #'n))
                        (set! total (+ total (syntax-e #'n))))
                   (~undo (printf "subtracting ~s\n" (syntax-e #'n))
                          (set! total (- total (syntax-e #'n)))))))
> (syntax-parse #'(1 2 3)
    [(x:nat/add ...) 'ok])

adding 1

adding 2

adding 3

'ok

> total

6

> (set! total 0)
> (syntax-parse #'(1 2 3 bad)
    [(x:nat/add ...) 'ok]
    [_ 'something-else])

adding 1

adding 2

adding 3

subtracting 3

subtracting 2

subtracting 1

'something-else

> total

0

(~post A-pattern)

就像 single-term pattern 版本, ~posts, 但只包含 action patterns 。

1.5.5 模式扩展器

syntax patterns 的语法是可以通过使用 pattern expanders 来扩展的, 它允许通过将新的模式形式改写成现有的模式形式来定义。

为了避免歧义, pattern expander 的名称通常以 ~ 字符开头。

procedure

(pattern-expander proc) → pattern-expander?

  proc : (-> syntax? syntax?)
返回一个使用 proc 来转换模式的 pattern expander 。

Example:
> (define-syntax ~maybe
    (pattern-expander
     (lambda (stx)
       (syntax-case stx ()
         [(~maybe pat ...)
          #'(~optional (~seq pat ...))]))))

value

prop:pattern-expander

 : (struct-type-property/c (-> pattern-expander? (-> syntax? syntax?)))
一个 structure type property , 用于识别作为 pattern expanders 的结构类型,就像 pattern-expander 所创建的那样。

(begin-for-syntax
  (struct thing (proc pattern-expander)
    #:property prop:procedure (struct-field-index proc)
    #:property prop:pattern-expander (λ (this) (thing-pattern-expander this))))
(define-syntax ~maybe
  (thing
   (lambda (stx) .... macro behavior ....)
   (lambda (stx) .... pattern-expander behavior ....)))

procedure

(pattern-expander? v) → boolean?

  v : any/c
如果 v 是一个 pattern expander,返回 #t ,否则返回 #f 。

procedure

(syntax-local-syntax-parse-pattern-introduce stx) → syntax?

  stx : syntax?
仅用于向后兼容;等同于语法本地引入 syntax-local-introduce 。

Changed in version 6.90.0.29 of package base: 与 syntax-local-introduce 等同。
