4 模块处理帮助程序

4.1 读取模块源代码

 (require syntax/modread) package: base

procedure

(with-module-reading-parameterization thunk) → any

  thunk : (-> any)
调用 thunk ,并将所有读者参数重置为默认值。

procedure

(check-module-form stx 
  expected-module-sym 
  source-v) 
 → (or/c syntax? false/c)
  stx : (or/c syntax? eof-object?)
  expected-module-sym : symbol?
  source-v : (or/c string? false/c)
检查 stx 以检查评估它是否会声明一个模块– 至少如果 module 在顶层与 Racket 的 module 绑定。 语法对象 stx 可以包含一个编译的表达式。 另外, stx 可以是一个文件结尾,理由是 read-syntax 可以产生一个文件结尾。

expected-module-sym 参数目前被忽略。 在以前的版本中,模块形式 stx 必须声明一个名称与 expected-module-sym 匹配的模块。

如果 stx 可以在适当的顶层声明一个模块, 那么 check-module-form 过程就会返回一个语法对象, 该对象肯定会在任何顶层声明一个模块(如果需要的话,给主导 module 添加明确的上下文)。 否则,如果 source-v 不是 #f, 就会使用消息中的源的 write 形式引发一个合适的异常; 如果 source-v 是 #f,就会返回 #f 。

如果 stx 是 eof 或者 eof 被包装成一个语法对象,那么将引发一个错误或者返回 #f 。

4.2 获取模块编译后的代码

 (require syntax/modcode) package: base

procedure

(get-module-code path    
  [#:submodule-path submodule-path    
  #:sub-path compiled-subdir0    
  compiled-subdir    
  #:roots roots    
  #:compile compile-proc0    
  compile-proc    
  #:extension-handler ext-proc0    
  ext-proc    
  #:notify notify-proc    
  #:source-reader read-syntax-proc    
  #:rkt-try-ss? rkt-try-ss?    
  #:choose choose-proc]) → any
  path : path-string?
  submodule-path : (listof symbol?) = '()
  compiled-subdir0 : 
(or/c (and/c path-string? relative-path?)
      (listof (and/c path-string? relative-path?)))
   = (use-compiled-file-paths)
  compiled-subdir : 
(or/c (and/c path-string? relative-path?)
      (listof (and/c path-string? relative-path?)))
   = compiled-subdir0
  roots : (listof (or/c path-string? 'same))
   = (current-compiled-file-roots)
  compile-proc0 : (any/c . -> . any) = compile
  compile-proc : (any/c . -> . any) = compile-proc0
  ext-proc0 : (or/c false/c (path? boolean? . -> . any)) = #f
  ext-proc : (or/c false/c (path? boolean? . -> . any))
   = ext-proc0
  notify-proc : (any/c . -> . any) = void
  read-syntax-proc : (any/c input-port? . -> . (or/c syntax? eof-object?))
   = read-syntax
  rkt-try-ss? : boolean? = #t
  choose-proc : 
(or/c (-> path? path? path?
          (or/c 'src 'zo 'so #f))
      #f)
 = #f
返回由 path 和 submodule-path 指定的模块声明的编译表达式, 其中 submodule-path 对于根模块是空的,对于子模块是一个列表。

roots, compiled-subdir, choose-proc 和 rkt-try-ss? 以及 submodule-path 参数决定了哪个文件被查询以找到编译的代码。 如果提供了默认值,那么这个函数使用与 current-load/use-compiled 的默认值相同的逻辑。更详细地讲:

  • 如果 submodule-path 不是空列表, 那么编译后的代码将永远不会被定位在 dynamic extension 中; 相反,将使用原始源文件或 "zo" 文件。

  • rkt-try-ss? 参数默认为 #t 。 如果它不是 #f ,那么如果 path 以 ".rkt" 结尾, 那么也将尝试以 ".ss" 结尾的相应文件。

  • 在调用 choose-proc 参数时, 会用到原始的源文件(该文件可能已经改变了结尾, 例如 rkt-try-ss? 参数)和另外两个以 "zo" 和 "so" 结尾的路径 (但它们不一定是 get-module-code 使用的路径)。 如果 choose-proc 返回 'src, 那么就不使用已编译的文件。 如果它返回任何其他结果,则该结果被忽略。 在这个函数的前几个版本中, choose-proc 提供了对使用哪个文件的更多控制,但现在已经没有了; 为了向后兼容,保留了当前的接口。

  • compiled-subdir 参数默认为 (use-compiled-file-paths); 它指定了要搜索模块的编译版本的子目录。 如果 compiled-subdir 是一个列表,那么第一个包含有适当名称的文件的目录将被作为编译文件。

  • roots 列表以与 current-compiled-file-roots 参数相同的方式指定一个编译文件搜索路径; 它默认为 current-compiled-file-roots 的当前值。

compile-proc 参数的默认值为 compile 。 如果没有已经编译过的版本,这个程序用于编译模块的源代码。 如果 submodule-path 不是 '(), 那么 compile-proc 必须返回一个已编译的模块形式。

ext-proc 参数的默认值是 #f 。 如果它不是 #f, 它必须是一个有两个参数的过程,当需要使用本地代码版本的 path 时,会被调用。 在这种情况下, ext-proc 的参数是 dynamic extension 的路径, 以及一个布尔值,表示该扩展是否是 _loader 文件 (#t) 或不是 (#f)。

如果一个 dynamic extension 是首选,或者是唯一存在的文件,当 ext-proc 为 #f 时,它将被提供给 ext-proc , 或者当 ext-proc 为 #f 时,将引发一个异常(报告一个扩展文件不能被使用)。

如果提供了 notify-proc ,它就会为所选择的文件(源文件、 ".zo" 或 dynamic extension) 被调用。

如果提供了 read-syntax-proc ,它被用来从源文件(但不是从字节码文件)读取模块。

Changed in version 6.90.0.7 of package base: 使用 (default-compiled-sub-path) 作为 compiled-subdir 的默认值。
Changed in version 8.6.0.12: 普及 #:sub-path 参数, 将 #:sub-path 的默认值改为 (use-compiled-file-paths), 并减少对 #:choose 参数的关注。

产生两个值。第一个是 path 指定的模块的最新源文件或编译文件的路径; 这个结果是 get-module-code 将读取的文件的路径,以产生一个编译的模块表达。 第二个值是 'src, 'zo 或 'so, 取决于第一个值是代表 Racket 源文件、编译的字节码文件还是本地库文件。

compiled-subdir, roots, choose-proc 和 rkt-try-ss? 等参数的解释与 get-module-code 相同。

submodule? 参数表示所需的模块是否是 path 指定的模块的一个子模块。 当 submodule? 为真时,结果路径永远不会指向一个 dynamic extension , 结果符号永远不会是 'so, 因为本地库不能提供子模块。

Changed in version 6.90.0.7 of package base: 使用 (default-compiled-sub-path) 作为 compiled-subdir 的默认值。
Changed in version 8.6.0.12: 普及 #:sub-path 参数, 将 #:sub-path 的默认值改为 (use-compiled-file-paths), 并减少对 #:choose 参数的关注。

procedure

(default-compiled-sub-path) → path-string?

这个函数曾经为 get-module-code 和 get-module-path 的 #:sub-path 参数提供了默认值, 但是这个库已经不再使用了。

4.3 将模块路径解析为文件路径

 (require syntax/modresolve) package: base

procedure

(resolve-module-path module-path-v 
  [rel-to-path-v]) 
 → 
(or/c path? symbol?
      (cons/c 'submod (cons/c (or/c path? symbol?) (listof symbol?))))
  module-path-v : module-path?
  rel-to-path-v : (or/c #f path-string? (-> any)) = #f
将一个模块路径解析为文件名路径。 如果模块路径是一个路径字符串(假定是一个文件), 则相对于 rel-to-path-v 进行解析, 如果是 thunk,则相对于调用 thunk 的目录结果进行解析,否则相对于当前目录进行解析。

当 module-path-v 使用基于集合的路径指向一个模块时, 解析会调用当前的 module name resolver, 但不会加载该模块,即使它没有被 declare 。 注意,在共享模块注册表的命名空间中的并发解析会在加载模块时产生竞争条件; 另见 namespace-call-with-registry-lock 。

procedure

(resolve-module-path-index module-path-index 
  [rel-to-path-v]) 
 → 
(or/c path? symbol?
      (cons/c 'submod (cons/c (or/c path? symbol?) (listof symbol?))))
  module-path-index : module-path-index?
  rel-to-path-v : (or/c #f path-string? (-> any)) = #f
与 resolve-module-path 类似, 但输入是一个 module path index; 在这种情况下, rel-to-path-v 基准被使用,其中模块路径索引包含 “self” 索引。 如果 module-path-index 依赖于 “self” 模块的路径索引, 那么会产生一个异常,除非 rel-to-path-v 是一个路径字符串。

参见 module-path-index-resolve 。

Examples:
> (resolve-module-path-index
   (module-path-index-join 'racket #f))

#<path:/home/ice/racket-8.9-2023-07-14/collects/racket/main.rkt>

> (resolve-module-path-index
   (module-path-index-join "apple.rkt" #f))

#<path:/home/ice/racket-8.9-2023-07-14/share/pkgs/racket-doc/syntax/apple.rkt>

> (resolve-module-path-index
   (module-path-index-join '(submod "." test) #f)
   (string->path "banana.rkt"))

'(submod #<path:banana.rkt> test)

4.4 简化模块路径

 (require syntax/modcollapse) package: base

procedure

(collapse-module-path module-path-v    
  rel-to-module-path-v) → module-path?
  module-path-v : module-path?
  rel-to-module-path-v : 
(or/c module-path?
      (-> module-path?))
通过将 module-path-v 和 rel-to-module-path-v 结合起来, 返回一个 "简化的" 模块路径,后者必须具有以下形式之一:一个 '(lib ....) )或符号模块路径; 一个 '(file ....) 模块路径;一个 '(planet ....) 模块路径; 一个 path;一个 '(quote symbol); 一个 '(submod base symbol ...) 模块路径, 其中允许 base ;或者一个 thunk 来生成其中的内容。

如果 module-path-v 包含一个结果所需要的路径元素, 或者 rel-to-module-path-v 是一个结果所需要的非字符串路径,那么结果就可以是一个路径。 同样地,结果可以是 'submod 包裹的路径。 否则,结果是一个模块路径(在 module-path? 的意义上),它不是一个普通的文件系统路径。

当结果是 'lib 或 'planet 模块路径时,它被规范化了, 因此同等的模块路径由 equal? 的结果表示。 当结果是一个 'submod 模块路径时,它只包含基本模块路径之后的符号, 在 'lib 或 'planet 的情况下,基本模块被规范化了。

Examples:
> (collapse-module-path "m.rkt"  '(lib "n/main.rkt"))

'(lib "n/m.rkt")

> (collapse-module-path '(submod "." x)  '(lib "n/main.rkt"))

'(submod (lib "n/main.rkt") x)

> (collapse-module-path '(submod "." x)  '(submod (lib "n/main.rkt") y))

'(submod (lib "n/main.rkt") y x)

procedure

(collapse-module-path-index module-path-index 
  rel-to-module-path-v) 
 → module-path?
  module-path-index : module-path-index?
  rel-to-module-path-v : 
(or/c module-path?
      (-> module-path?))
(collapse-module-path-index module-path-index)
 → (or/c module-path? #f)
  module-path-index : module-path-index?
像 collapse-module-path 一样,当给出两个参数时, 但输入是一个 module path index; 在这种情况下, rel-to-module-path-v 基准被使用, 其中模块路径索引包含 “self” 索引(见 module-path-index-split)。

当给定一个参数时,如果给定的模块路径索引是相对的, collapse-module-path-index 返回一个相对的模块路径, 但如果它的参数是 “self” 模块路径索引,则返回 #f 。 得到的模块路径不一定是标准化的。

Changed in version 6.1.1.8 of package base: 增加了用于折叠相对模块路径索引的单参数变量。
Changed in version 6.9.0.5: 增加了对 "self" 模块路径索引作为唯一参数的支持, 这意味着扩展结果契约以包括 #f 。

4.5 检查模块和模块的依赖关系

 (require syntax/moddep) package: base

procedure

(show-import-tree module-path-v    
  [#:dag? dag?    
  #:path-to path-to-module-path-v    
  #:show show]) → void?
  module-path-v : module-path?

4.6 包裹模块主体表达式

 (require syntax/wrap-modbeg) package: base

Added in version 6.0.0.1 of package base.

procedure

(make-wrapping-module-begin wrap-form 
  [module-begin-form]) 
 → (syntax? . -> . syntax?)
  wrap-form : syntax?
  module-begin-form : syntax? = #'#%plain-module-begin
提供了 for-syntax 。

构造一个适合作为 #%module-begin 的替代品的函数, 特别是用来替代 #%module-begin 的面,即包装每个顶层表达式以打印该表达式的结果。

该函数接收一个语法对象,并返回一个使用 module-begin-form 的语法对象。 假设 module-begin-form 类似于 #%plain-module-begin, 每个顶层表达式 expr 将被包装成 (wrap-form expr), 而顶层声明(如 define-values 和 require 形式) 将保持原样。 表达式在宏扩展和 begin 拼接后被检测出来,扩展与声明处理照常交错进行。
