4.8 正则表达式

正则表达式 in Racket 指南 introduces regular expressions.

Regular expressions 被指定为字符串或字节字符串,使用与 Unix 工具 egrep 或 Perl 相同的模式语言。 一个字符串指定的模式会产生一个字符重合表达式匹配器,而一个字节字符串模式会产生一个字节重合表达式匹配器。 如果字符 regexp 与字节串或输入端口一起使用,它匹配字符流的 UTF-8 编码(参见 编码和地域) ; 如果字节 regexp 与字符串一起使用,它匹配该字符串的 UTF-8 编码中的字节。

以字符串或字节串形式表示的正则表达式可以被编译成 regexp value , 与字符串或字节串形式相比, regexp-match 等函数可以更有效地使用。 regexp 和 byte-regexp 程序使用与 egrep 最兼容的正则表达式语法将字符串或字节串(分别)转换成 regexp 值。 pregexp 和 byte-pregexp 过程使用与 Perl 更加兼容的正则表达式的略微不同的语法, 产生一个 regexp 值。

如果两个 regexp values 具有相同的来源,使用相同的模式语言, 并且都是字符 regexps 或都是字节 regexps,那么它们就是 equal? 。

一个字面的或打印的 regexp value 以 #rx 或 #px 开头。 See 阅读正则表达式 for information on reading regular expressions and 打印正则表达式 for information on printing regular expressions. 默认读取器生成的 Regexp 值在 read-syntax 模式下被 interned 。

在 Racket 的 BC 变体上, 一个 regexp value 的内部大小被限制为 32KB; 这个限制大致相当于一个有 32000 个字面字符或 5000 个操作符的源字符串。

4.8.2 额外的句法限制

除了匹配一个语法,正则表达式还必须满足两个语法限制:

  • 在 ‹atom›? 以外的 ‹repeat› 中, ‹atom› 不能匹配空序列。

  • 在一个 (?<=‹regexp›) 或 (?<!‹regexp›) 中, ‹regexp› 必须只匹配一个有界的序列。

这些约束由以下类型系统进行语法检查。 一个类型 [n, m] 对应于在 n 和 m 个字符之间匹配的表达式。 在 (‹Regexp›) 的规则中, N 指的是这样的数字, 即开口括号是收集匹配报告的第 N 个开口括号。 非 emptiness 被推断为一个回文模式, \‹N› ,因此回文可以用于重复模式; 在回文之间相互依赖的情况下,推断选择最大非 emptiness 的 fixpoint。 对于反向推理来说,有限性是不被推断的(也就是说,一个反向推理被假定为与一个任意大的序列相匹配)。 没有任何语法约束禁止在其引用的组内进行反向引用, 尽管这种自我引用可能会产生一个没有可能匹配的模式(如 (.\1)的情况, 尽管 (^.|\1){2} 匹配以相同两个字符开始的输入)。

4.8.4 Regexp 匹配

procedure

(regexp-match pattern 
  input 
  [start-pos 
  end-pos 
  output-port 
  input-prefix]) 
 → 
(if (and (or (string? pattern) (regexp? pattern))
         (or (string? input) (path? input)))
    (or/c #f (cons/c string? (listof (or/c string? #f))))
    (or/c #f (cons/c bytes?  (listof (or/c bytes?  #f)))))
  pattern : (or/c string? bytes? regexp? byte-regexp?)
  input : (or/c string? bytes? path? input-port?)
  start-pos : exact-nonnegative-integer? = 0
  end-pos : (or/c exact-nonnegative-integer? #f) = #f
  output-port : (or/c output-port? #f) = #f
  input-prefix : bytes? = #""
试图将 pattern (一个字符串、字节字符串、 regexp value 或字节 regexp value) 与 input 的一部分匹配一次。 匹配器会找到一个匹配的、最接近 input 开始的部分 start-pos 之后)。

如果 input 是一个路径,如果 pattern 是一个字节字符串或一个基于字节的 regexp, 则用 path->bytes 将其转换为一个字节字符串。 否则, input 将被转换为一个字符串,使用 path->string 。

可选的 start-pos 和 end-pos 参数选择 input 的一部分进行匹配; 默认是整个字符串或直到文件结束的流。 当 input 是一个字符串时, start-pos 是一个字符位置; 当 input 是一个字节字符串时, start-pos 是一个字节位置; 当 input 是一个输入端口时, start-pos 是开始匹配前要跳过的字节数。 end-pos 参数可以是 #f ,它对应于字符串的结尾或流中的文件结尾; 否则,它是一个字符或字节的位置,和 start-pos 样。 如果 input 是一个输入端口, 并且在跳过 start-pos 字节之前到达了文件的末端,那么匹配就会失败。

在 pattern, start-of-string ^ 指的是 start-pos 之后 input 的第一个位置,假设 input-prefix 是 #"" 。 输入端 $ 指的是 end-pos 位置或者(在输入端口的情况下)文件的末端,以先到者为准。

input-prefix 指定了在 input 之前有效的字节,用于 ^ 和其他后视匹配。 例如, #"" 前缀意味着 ^ 在数据流的开头匹配,而 #"\n" input-prefix 意味着行首 ^ 可以匹配输入的开头,而文件首 ^ 不能。

如果匹配失败,会返回 #f 。 如果匹配成功,会返回一个包含字符串或字节串的列表,可能还有 #f 。 只有当 input 是字符串,并且 pattern 不是字节重合表达式时,该列表才包含字符串。 否则,该列表包含字节字符串(如果 input 是一个字符串,则为UTF-8编码的子串)。

结果列表中的第一个[字节]字符串是 input 中与 pattern 匹配的部分。 如果 input 的两个部分可以匹配 pattern ,那么找到最早开始的匹配。

如果 pattern 包含括号内的子表达式(但如果开头的括号后面是 ? ,则不返回),则在列表中返回其他[字节]字符串。 子表达式的匹配是按照 pattern 中括号开头的顺序来提供的。 当子表达式出现在一个 | “or” 模式的分支中, 在一个 * “zero or more” 模式中, 或者在其他没有子表达式匹配的情况下整个模式可以成功的地方, 那么如果子表达式对最终的匹配没有贡献,将返回一个 #f 。 当一个子表达式出现在 * “zero or more” 模式或其他多重匹配位置时,则在列表中返回与该子表达式相关的最右边的匹配。

如果提供了可选的 output-port 作为输出端口, 那么 input 中从开始(不是 start-pos) 开始的部分,在匹配之前被写到端口上。 如果没有找到匹配项,那么直到 end-pos 的所有 input 都会被写到端口上。 当 input 是一个输入端口时,这个功能是最有用的。

当匹配一个输入端口时,匹配失败会读到 end-pos 字节 (或文件末尾),即使 pattern 是以字符串开头的 ^ 开始的;也请看 regexp-try-match 。 在成功的情况下,所有直到并包括匹配的字节最终都会从端口中读出, 但匹配的过程是先从端口中偷看字节 (使用 peek-bytes-avail!), 然后在确定了匹配结果之后,再 (re‑)读取匹配的字节来丢弃它们。 不匹配的字节可以在确定匹配结果之前被读取和丢弃。 匹配器在阻塞模式下仅在确定匹配的必要范围内偷看,但如果立即可用(即不阻塞), 它可以偷看额外的字节以填充内部缓冲区。 pattern 中的贪婪的重复操作,比如 * 或 +, 倾向于强制读取端口的全部内容 (直到 end-pos) 来确定匹配。

如果输入端口被另一个线程同时读取,或者该端口是一个具有不一致的读取和偷看程序的自定义端口 (参见 Custom Ports), 那么被偷看并用于匹配的字节可能与读取并在匹配完成后丢弃的字节不同; 匹配器只检查偷看的字节。 为了避免这种交错,请使用 regexp-match-peek (带有 progress-evt 参数) 和 port-commit-peeked 。

Examples:
> (regexp-match #rx"x." "12x4x6")

'("x4")

> (regexp-match #rx"y." "12x4x6")

#f

> (regexp-match #rx"x." "12x4x6" 3)

'("x6")

> (regexp-match #rx"x." "12x4x6" 3 4)

#f

> (regexp-match #rx#"x." "12x4x6")

'(#"x4")

> (regexp-match #rx"x." "12x4x6" 0 #f (current-output-port))

12

'("x4")

> (regexp-match #rx"(-[0-9]*)+" "a-12--345b")

'("-12--345" "-345")

> (regexp-match* #rx"x." "12x4x6")

> (regexp-match* #rx"x(.)" "12x4x6" #:match-select cadr)

> (regexp-match* #rx"x(.)" "12x4x6" #:match-select values)

> (regexp-match* #rx"x(.)" "12x4x6" #:match-select cadr #:gap-select? #t)

> (regexp-match* #rx"x(.)" "12x4x6" #:match-select #f #:gap-select? #t)

procedure

(regexp-replace pattern 
  input 
  insert 
  [input-prefix])

使用 pattern 对 input 进行匹配,然后返回一个字符串或字节串, 其中 input 的匹配部分被替换为 insert 。 如果 pattern 没有匹配到 input 的任何部分,则返回未修改的 input 。

insert 参数可以是一个(字节)字符串,或一个返回(字节)字符串的函数。 在后一种情况下,该函数被应用于 regexp-match 将返回的值列表 (即,第一个参数是完整的匹配,然后每个括号内的子表达式有一个参数)以获得一个替换的(字节)字符串。

如果 pattern 是一个字符串或字符 regexp,并且 input 是一个字符串,那么 insert 必须是一个字符串或一个接受字符串的过程,并且结果是一个字符串。 如果 pattern 是一个字节字符串或字节重合表达式,或者 input 是一个字节字符串, 那么 insert 作为一个字符串被转换为一个字节字符串, insert 作为一个过程被调用,结果是一个字节字符串。

如果 insert 包含 &, 那么 & 在被替换到匹配的位置之前,会被替换成 input 的匹配部分。 如果 insert 包含 \‹n› 的整数 ‹n› , 那么它将被替换为 input 中的 ‹n› 匹配子表达式。 & 和 \0 是别名。 如果第 ‹n› 个子表达式没有在匹配中使用, 或者如果 ‹n› 大于 pattern 中的子表达式数量, 那么 \‹n› 将被替换成空字符串。

要替换一个字面意义上的 & 或 \, 在 insert 中分别使用 \& 和 \\ 。 insert 中的一个 \$ 相当于一个空序列; 这可以用来终止 \ 后面的数字 ‹n› 。 如果在 insert 中的一个 \ 后面是除了数字 &, \ 或 $ 以外的任何东西,那么这个 \ 本身就被视为 \0 。

请注意,前面几段中描述的 \ 是一个字符或字节的 insert 。 要把这样的 insert 物写成 Racket 字符串字头,需要在 \ 之前加一个转义的 \ 。 例如,Racket 常量"\\1" 就是 \1 。

Examples:
> (regexp-replace #rx"mi" "mi casa" "su")

"su casa"

> (regexp-replace #rx"mi" "mi casa" string-upcase)

"MI casa"

> (regexp-replace #rx"([Mm])i ([a-zA-Z]*)" "Mi Casa" "\\1y \\2")

"My Casa"

> (regexp-replace #rx"([Mm])i ([a-zA-Z]*)" "mi cerveza Mi Mi Mi"
                  "\\1y \\2")

"my cerveza Mi Mi Mi"

> (regexp-replace #rx"x" "12x4x6" "\\\\")

"12\\4x6"

> (display (regexp-replace #rx"x" "12x4x6" "\\\\"))

12\4x6
