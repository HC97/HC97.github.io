18.6 调试

Racket 的内置调试支持仅限于与异常一起打印的上下文(即 "堆栈跟踪")信息。 在某些情况下,对于 Racket 的 BC 实现,禁用 JIT 编译器会影响上下文信息。 对于 Racket 的 CS 实现,设置 PLT_CS_DEBUG 环境变量会使编译记录表达式级别的上下文信息, 而不仅仅是函数级信息。

errortrace 库支持更加一致(独立于编译器)和精确的上下文信息。 racket/trace 库提供简单的跟踪支持。 最后, DrRacket 编程环境提供了更多的调试支持。

18.6.1 追踪

 (require racket/trace) package: base
The bindings documented in this section are provided by the racket/trace library, not racket/base or racket.

racket/trace 库模仿了Chez Scheme中的追踪功能。

syntax

(trace id ...)

每个 id 必须与 trace 表达式环境中的一个过程绑定,并且不能从其他模块导入。 每个 id 都被 set! 为一个新的过程,该过程追踪过程的调用, 并通过 current-trace-notify 打印调用的参数和结果来返回。 如果有多个返回值,每个值都会显示在单独的一行。

当被追踪的过程互相调用时,嵌套的调用会通过打印嵌套前缀来显示。 如果嵌套深度增长到 10 或更多,会打印一个数字来显示实际的嵌套深度。

trace 形式可以用在一个已经被追踪的标识符上。 在这种情况下,假设变量的值没有被改变, trace 没有任何作用。 如果该变量已经被改变为一个不同的过程,那么将安装一个新的跟踪。

追踪尊重尾部调用以保护循环,但是它的效果可能通过延续标记可见。 当对一个被追踪的过程的调用发生在相对于前一个被追踪的调用的尾部位置时, 那么该调用的尾部性将被保留(并且该调用的结果不会因为尾部调用而被打印出来,因为同样的结果将被打印在一个封闭的调用中)。 但是,否则,相对于对过程的调用而言,被追踪的过程的主体不会在尾部位置被评估。

trace 表达式的结果是 #<void> 。

trace 也可以用来调试 syntax transformers 。 直接用 trace 来做这个是很麻烦的;参考 trace-define-syntax 来做这个更简单的方法。

Examples:
> (require (for-syntax racket/trace))
> (begin-for-syntax
    (define let
      (syntax-rules ()
        [(_ ([x v]) e) ((lambda (x) e) v)]))
    (trace let))
> (define-syntax let let)
> (let ([x 120]) x)

>(_let #<syntax:eval:9:0 (let ((x 120)) x)>)

<#<syntax:eval:9:0 ((lambda (x) x) 120)>

syntax

(trace-define-syntax id expr)

(trace-define-syntax (head args) body ...+)
trace-define-syntax 形式是先定义一个语法转换器然后追踪它的简写。 这种形式支持所有 define-syntax 形式。

例如:

Examples:
> (trace-define-syntax fact
    (syntax-rules ()
      [(_ x) 120]))
> (fact 5)

>(fact #<syntax:eval:15:0 (fact 5)>)

<#<syntax:eval:15:0 120>

120

默认情况下, trace 在追踪一个语法转换器时,会打印出语法对象。 如果你不需要看,例如源信息,这可能会导致过多的输出。 为了通过打印语法对象作为基准点获得更可读的输出,我们可以修改 current-trace-print-args 和 current-trace-print-results 。 参见 current-trace-print-args 的例子。

syntax

(trace-lambda [#:name id] args expr)

trace-lambda 形式可以追踪一个匿名函数。 该表格将尝试使用 syntax-local-infer-name 来推断一个名称, 或者使用可选的 #:name 参数来指定一个名称。 如果没有给出名字,并且不能推断出名字,就会产生语法错误。

Example:
> ((trace-lambda (x) 120) 5)

>(eval:16:0 5)

<120

120

(trace-let id ([arg expr] ...+) body ...+)

syntax

(untrace id ...)

撤销对每个 id 的 trace 效果,将每个 id 重新 set! 为未被跟踪的过程,但前提是 id 的当前值是被跟踪的过程。 如果一个 id 的当前值不是由 trace 安装的过程,那么这个变量就不会被改变。

untrace 表达式的结果是 #<void> 。

parameter

(current-trace-notify) → (string? . -> . any)

(current-trace-notify proc) → void?
  proc : (string? . -> . any)
一个 parameter ,决定了跟踪输出的显示方式。 给予 proc 的字符串是一个跟踪;它不以换行结束,但可能包含内部换行。 每个调用或结果都会用 pretty-print 转换为字符串。 该参数的默认值是将给定的字符串以换行方式打印到 (current-output-port) 。

procedure

(trace-call id proc #:<kw> kw-arg ...) → any/c

  id : symbol?
  proc : procedure?
  kw-arg : any/c
用 args 中提供的参数调用 proc ,也可能使用关键字参数。 在调用过程中还会打印出跟踪信息,如上面 trace 的文档中所述,使用 id 作为 proc 的名称。

parameter

(current-trace-print-args) → 
(-> symbol?
    list?
    (listof keyword?)
    list?
    number?
    void?)
(current-trace-print-args trace-print-args) → void?
  trace-print-args : 
(-> symbol?
    list?
    (listof keyword?)
    list?
    number?
    void?)
这个参数的值被调用来打印出一个被追踪的调用的参数。 它接收函数的名称,函数的普通参数,它的关键字,关键字的值,以及一个表示调用深度的数字。

修改这个和 current-trace-print-results 对于在追踪语法变换器时获得更多的可读性或额外的输出是有用的。 例如,我们可以使用 debug-scopes 将作用域信息添加到跟踪中, (见 debug-scopes 的例子),或者删除源位置信息,只显示语法对象的形状。

在这个例子中,我们更新了打印机 current-trace-print-args 和 current-trace-print-results , 通过存储当前打印机 (ctpa 和 ctpr) 使用 syntax->datum 将语法对象转换为datum,然后将转换后的参数和结果传递给前一台打印机。追踪时,语法参数的显示将没有源位置信息,缩短了输出。

Examples:
> (require (for-syntax racket/trace))
> (begin-for-syntax
    (current-trace-print-args
      (let ([ctpa (current-trace-print-args)])
        (lambda (s l kw l2 n)
          (ctpa s (map syntax->datum l) kw l2 n))))
    (current-trace-print-results
      (let ([ctpr (current-trace-print-results)])
        (lambda (s r n)
         (ctpr s (map syntax->datum r) n)))))
> (trace-define-syntax fact
    (syntax-rules ()
      [(_ x) 120]))
> (fact 5)

>(fact '(fact 5))

<120

120

我们在修改这些参数时必须小心,特别是当转换对被追踪标识符的参数/结果进行假设或改变其类型时。 对 current-trace-print-args 和 current-trace-print-results 的修改是一个强制性的更新, 会影响所有被追踪的标识符。 这个例子假设 all traced functions 的参数和结果都是语法对象,只有当你只追踪语法变换器时才是这样的。 如果按原样使用,上述代码在追踪函数和语法变换器时可能会导致类型错误。 最好是只在参数或结果实际上是语法对象时才使用 syntax->datum , 例如,通过定义 maybe-syntax->datum 如下。

Examples:
> (require (for-syntax racket/trace))
> (begin-for-syntax
    (define (maybe-syntax->datum syn?)
      (if (syntax? syn?)
          (syntax->datum syn?)
          syn?))
    (current-trace-print-args
      (let ([ctpa (current-trace-print-args)])
        (lambda (s l kw l2 n)
          (ctpa s (map maybe-syntax->datum l) kw l2 n))))
    (current-trace-print-results
      (let ([ctpr (current-trace-print-results)])
        (lambda (s l n)
         (ctpr s (map maybe-syntax->datum l) n))))
  
  (trace-define (precompute-fact syn n) (datum->syntax syn (apply * (build-list n add1)))))
> (trace-define (run-time-fact n) (apply * (build-list n add1)))
> (require (for-syntax syntax/parse))
> (trace-define-syntax (fact syn)
    (syntax-parse syn
      [(_ x:nat) (precompute-fact syn (syntax->datum #'x))]
      [(_ x) #'(run-time-fact x)]))
> (fact 5)

>(fact '(fact 5))

>(precompute-fact '(fact 5) 5)

<120

120

> (fact (+ 2 3))

>(fact '(fact (+ 2 3)))

<'(run-time-fact (+ 2 3))

>(run-time-fact 5)

<120

120

parameter

(current-trace-print-results) → 
(-> symbol?
    list?
    number?
    any)
(current-trace-print-results trace-print-results) → void?
  trace-print-results : 
(-> symbol?
    list?
    number?
    any)
这个参数的值被调用来打印出一个被追踪的调用的结果。 它接收函数的名称,函数的结果,以及一个表示调用深度的数字。

parameter

(current-prefix-in) → string?

(current-prefix-in prefix) → void?
  prefix : string?
这个字符串被 current-trace-print-args 的默认值使用,表示当前行显示对被追踪函数的调用。

它的默认值是 ">" 。

parameter

(current-prefix-out) → string?

(current-prefix-out prefix) → void?
  prefix : string?
这个字符串用于 current-trace-print-results 的默认值,表示当前行显示的是一个被追踪的调用的结果。

默认值为 "<" 。
