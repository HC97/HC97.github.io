14.4 Module Names and Loading

14.4.1 解决模块名称问题

syntax/modresolve 库为解析和处理模块名称提供了额外的操作。

一个已声明的模块的名字由一个 resolved module path 表示, 它封装了一个符号或一个完整的文件系统路径(见 Paths)。 符号通常指的是一个预定义的模块或通过反射性评估(例如, eval)声明的模块。 一个文件系统路径通常指的是通过 require 或其他形式按需加载的模块声明。

module path 是一个符合 require 的 module-path 语法的数据。 一个模块路径是相对于另一个模块的。

procedure

(resolved-module-path? v) → boolean?

  v : any/c
如果 v 是一个 resolved module path ,则返回 #t ,否则返回 #f 。

procedure

(make-resolved-module-path path) → resolved-module-path?

  path : 
(or/c symbol?
      (and/c path? complete-path?)
      (cons/c (or/c symbol?
                    (and/c path? complete-path?))
              (non-empty-listof symbol?)))
返回一个封装了 path 的 resolved module path , 其中一个列表 path 对应于一个 submodule 路径。 如果 path 是一个路径或以 path 开头, 通常应该对路径进行 cleanse (见 cleanse-path) 和简化(见 simplify-path, 包括咨询文件系统)。

一个 resolved module path 是内部的。 也就是说,如果两个 resolved module path 值封装的路径是 equal?, 那么 resolved module path 值就是 eq? 。

procedure

(resolved-module-path-name module-path)

 → 
(or/c symbol?
      (and/c path? complete-path?)
      (cons/c (or/c symbol?
                    (and/c path? complete-path?))
              (non-empty-listof symbol?)))
  module-path : resolved-module-path?
返回被 resolved module path 所封装的路径或符号。 一个列表结果对应于一个 submodule 路径。

procedure

(module-path? v) → boolean?

  v : any/c
如果 v 对应的数据与 require 的 module-path 的语法相符, 则返回 #t ,否则返回 #f 。 注意,一个路径(在 path? 的意义上)是一个模块路径。

 → 
(case->
 (resolved-module-path? (or/c #f namespace?) . -> . any)
 (module-path?
  (or/c #f resolved-module-path?)
  (or/c #f syntax?)
  boolean?
  . -> .
  resolved-module-path?))
(current-module-name-resolver proc) → void?
  proc : 
(case->
 (resolved-module-path? (or/c #f namespace?) . -> . any)
 (module-path?
  (or/c #f resolved-module-path?)
  (or/c #f syntax?)
  boolean?
  . -> .
  resolved-module-path?))
一个决定当前 module name resolver 的 parameter , 它管理着从其他类型的模块引用到 resolved module path 的转换。 例如,当扩展器遇到 (require module-path) 时, module-path 不是一个标识符,那么扩展器将 'module-path 传递给模块名解析器以获得一个符号或解析的模块路径。 当这样的 require 出现在一个模块中时, module path resolver 也被赋予包围模块的名称, 这样相对引用可以被转换为绝对的符号或 resolved module path 。

默认的 module name resolver 使用 collection-file-path 来将 lib 和 符号速记模块路径转换为文件系统路径。 collection-file-path 函数反过来使用 current-library-collection-links 和 current-library-collection-paths 参数。

一个 module name resolver 需要两个和四个参数:

  • 当给定两个参数时,第一个参数是现在在当前命名空间中声明的模块的名称,第二个参数是可选的命名空间,该声明是从那里复制的。 在这种情况下,模块名称解析器的结果被忽略。

    当前模块名称解析器被 namespace-attach-module 或 namespace-attach-module-declaration 用两个参数调用, 以通知解析器一个模块声明被附加到当前命名空间(并且在未来不应该被加载到该命名空间的 module registry)。 对模块声明的评估也会以两个参数调用当前模块名称解析器,其中第一个参数是声明的模块,第二个参数是 #f。 没有其他 Racket 操作会用两个参数调用模块名称解析器,但其他工具(如 DrRacket)可能会以这种模式调用该解析器, 以避免多余的模块加载。

  • 当给出四个参数时,第一个是模块路径,相当于 require 的引号 module-path 。 第二个是源模块的名字,如果有的话,路径是相对的; 如果第二个参数是 #f, 模块路径是相对的 (or (current-load-relative-directory) (current-directory))。 第三个参数是一个 syntax object ,如果它不是 #f 的话,可以用来报告错误。 如果最后一个参数是 #t, 那么模块声明应该被加载(如果还没有), 否则模块路径应该被简单地解析为一个名字。结果是解析出来的名字。

对于第二种情况,标准的模块名称解析器为每个 module registry 保留了一个包含加载模块名称的表格。 如果被解析的模块路径不在表中,并且 #f 没有作为第四个参数提供给 module name resolver, 那么该名称就会被放入表中,并且相应的文件会被加载, 加载时采用 load/use-compiled 的变体,将预期的模块名称传递给 compiled-load handler 。

在加载文件时,默认的 module name resolver 将 current-module-declare-name 参数设置为已解析的模块名称(而 compiled-load handler 序设置为 current-module-declare-source)。 另外,默认的 module name resolver 在一个私有的 continuation mark 中记录正在加载的模块, 并且它检查这样的标记是否已经存在;如果这样的延续标记确实存在于当前的延续中, 那么 exn:fail exception is raised 将被引发,并给出一个关于依赖循环的消息。

默认的模块名称解析器与默认的 compiled-load handler 合作: 在模块附加通知中,由 compiled-load handler 为源名称空间的 module registry 记录的字节码文件信息被转移到目标名称空间的 module registry 。

默认的模块名称解析器还维护着一个小型的、特定于 module registry 的缓存, 将 lib 和符号模块路径映射到它们的解析上。 在检查诸如 current-library-collection-links 和 current-library-collection-paths 等参数之前, 会参考这个缓存,所以即使这些参数值发生变化,结果也可能 "坚持" 下来。 只有当模块名称解析器的第四个参数为真(表示应该加载一个模块),并且只有在加载成功的情况下,才会将一个条目添加到缓存中。

最后,默认的模块名称解析器可能会特别处理一个 submod 路径。 如果作为 submod 形式的第一个元素的模块路径指的是不存在的集合, 那么默认的模块名称解析器不是引发一个异常,而是为产生的 resolved module path 合成一个未加约束的符号模块名称。 这种对子模块路径的特殊处理与 compiled-load handler 对不存在的子模块的特殊处理是一致的, 因此 module-declared? 可以更容易地用于检查子模块的存在。

当解析语法对象中的 syntax objects 时(见 语法对象), 模块加载被抑制(即 #f 被作为第四个参数提供给模块名称解析器)。 当 syntax object 被操作时,当前的名字空间可能与语法对象的原始名字空间不一致, 模块不一定要在当前的名字空间加载。

由于历史原因,目前默认的模块名称解析器除了接受两个和四个参数外,还接受三个参数。 三个参数与四个参数的处理方式相同,第四个参数为 #t, 只是也会记录一个错误。 对三个参数的支持将在未来的版本中删除。

current-module-name-resolver 绑定在 protect-out 的意义上被提供为 protected 。

Changed in version 6.0.1.12 of package base: 当调用三个参数时,为默认模块名称解析器增加了错误记录。
Changed in version 7.0.0.17: 增加了默认模块名称解析器对不存在集合的 submod 形式的特殊处理。
Changed in version 8.2.0.4: 修改了绑定为 protected 。

 → (or/c resolved-module-path? #f)

  name : (or/c resolved-module-path? #f)
一个确定模块名称的 parameter ,在评估 module 声明时使用(当参数值不是 #f 时)。 在这种情况下, module 声明中的 id 被忽略,而参数的值被用作声明的模块名称。

在声明 submodules 时, current-module-declare-name 决定了子模块的根模块所使用的名称, 而其相对于根模块的子模块路径不受影响。

parameter

               (module-path? (syntax->datum s)))))
(current-module-path-for-load path) → void?

(or/c #f module-path?
      (and/c syntax?

               (module-path? (syntax->datum s)))))
一个确定模块路径的 parameter , 用于 exn:fail:syntax:missing-module 和 exn:fail:filesystem:missing-module , 由默认 load handler 引发。该参数通常由 module name resolver 设置。

14.4.2 编译的模块和引用

在扩展 module 声明时,扩展器会解析导入的模块路径,以便在必要时加载 module 声明并确定导入的绑定关系,但模块声明的编译形式保留了原始的模块路径。 因此,一个已编译的模块可以被移到另一个文件系统中,在那里模块名称解析器可以解析已编译代码中的模块间引用。

当一个模块的引用从编译后的形式(见 module-compiled-imports) 或从宏扩展的语法对象(见 语法对象内容)中提取时,模块的引用以 module path index的形式报告。 module path index是一个半内联的(对同一相对模块的多个引用倾向于使用相同的 module path index 值, 但并不总是如此)不透明的值,它编码了一个模块路径(见 module-path?)和一个 resolved module path 或它相对的另一个 module path index 。

一个同时使用 #f 作为路径和基础 module path index 的 module path index 代表 "自我"–即作为 module path index来源的模块声明– 这样的 module path index 可以在编译时被用作 module path index 链的根。 例如,当提取一个标识符在模块中的绑定信息时,如果该标识符被同一模块中的定义所绑定, 则使用 module path index 报告该标识符的源模块。 如果标识符被定义在一个通过模块路径(而不是字面模块名称)导入的模块中, 那么标识符的源模块将使用一个 module path index 来报告, 其中包含 require 模块路径和 "自我 " module path index 。 一个 "自我 " module path index 有一个子模块路径,当它所指的模块是一个 submodule 。

一个 module path index 有状态。当它被 resolved 为一个 resolved module path 时, 那么 resolved module path 与 module path index 一起被存储。 特别是,当一个模块被加载时,它的根 module path index 被解析为与该模块的声明时间名称相匹配。 然而,这个被解析的路径被遗忘在模块贡献给其他模块的编译和 marshaled 形式的标识符中。 解析后的名称的瞬时性允许模块代码以与编译时不同的解析后的名称被加载。

当两个 module path index 值具有 equal? 的路径和基础值时, 它们是 equal? (即使它们具有不同的 resolved 值)。

procedure

(module-path-index? v) → boolean?

  v : any/c
如果 v 是一个 module path index,返回 #t ,否则返回 #f 。

procedure

(module-path-index-resolve mpi    
  [load?    
  src-stx]) → resolved-module-path?
  mpi : module-path-index?
  load? : any/c = #f
  src-stx : (or/c syntax? #f) = #f
返回被解析的模块名称的 resolved module path ,如果之前没有计算过, 则计算被解析的名称(并将其存储在 mpi 中)。

解析 module path index 使用当前 module name resolver (见 current-module-name-resolver)。 根据 mpi 封装的模块路径的种类,计算出的解析名称可能取决于 current-load-relative-directory 或 current-directory 的值。 load? 参数被作为最后一个参数传播给 module name resolver, 而 src-stx 参数被作为下一个参数传播。

请注意,在共享模块注册表的命名空间中的并发解析会在加载模块时产生竞赛条件。 另请参见 namespace-call-with-registry-lock 。

如果 mpi 表示一个 "自己"(见上文)的模块路径,该路径不是由扩展器创建的,已经被解析, 那么 module-path-index-resolve 会引发 exn:fail:contract,而不调用模块名称解析器。

参见 resolve-module-path-index 。

Changed in version 6.90.0.16 of package base: 增加了 load? 的可选参数。
Changed in version 8.2: 增加了 src-stx 的可选参数。

                 (or/c module-path-index?

对于列表中的每个元素,前面的符号是出口的名称。

第二部分– module path index 值等的列表–描述了导出标识符的起源。 如果原点列表为 null, 那么导出的标识符就在模块中定义。 如果导出的标识符被重新导出,那么origin列表提供了关于被重新导出的导入的信息。 如果绑定物被多次从(可能)不同的来源导入,那么原产地列表就有一个以上的元素。

最后一个部分,一个符号,只有在 verbosity 为 'defined-names 时才会被包含。 在这种情况下,包含的符号是其定义模块中的定义的名称(可能与导出的名称不同)。

对于每个起源, module path index 本身意味着绑定被导入, phase level 转移为 0 (即没有 for-meta, for-syntax 等的普通 require ) 到默认 binding space (即没有 for-space), 并且导入的标识符与重新导出的名称相同。 用列表表示的原点明确指出了导入、导入标识符所绑定的 phase level 加 binding space (关于表示方法的更多信息,请参见 phase+space? )、 导入模块中绑定的导入符号名称以及导出模块中标识符的 phase level 加 binding space 。

Example:
> (module-compiled-exports
   (compile
    '(module banana racket/base
       (require (only-in racket/math pi)
                (for-syntax racket/base))
       (provide pi
                (rename-out [peel wrapper])
                bush
                cond
                (for-syntax compile-time))
       (define peel pi)
       (define bush (* 2 pi))
       (begin-for-syntax
         (define compile-time (current-seconds)))))
   'defined-names)

'((0

   (bush () bush)

   (pi (#<module-path-index:racket/math>) pi)

   (wrapper () peel))

  (1 (compile-time () compile-time)))

'((0 (cond (#<module-path-index:racket/base>) cond)))

Changed in version 8.2.0.3: 将结果泛化为相位空间组合。

14.4.3 动态模块访问

procedure

(dynamic-require mod provided [fail-thunk]) → (or/c void? any/c)

  mod : 
(or/c module-path?
      resolved-module-path?
      module-path-index?)
  provided : (or/c symbol? #f 0 void?)
  fail-thunk : (-> any) = (lambda () ....)

因为 dynamic-require 是一个过程, 用与 require 表达式相同的方式给 mod 提供一个普通的S表达式可能不会给你带来预期的结果。 你所需要的是一个可以评估为S表达式的东西;使用 quote 是一种方法。

动态 instantiates 当前命名空间的注册表中由 mod 指定的模块,如果它还没有 instantiate 的话,在命名空间的 base phase 。 当前 module name resolver 可以加载模块声明来解析 mod (见 current-module-name-resolver); 路径是相对于 current-load-relative-directory 和/或 current-directory 解析的。 注意,在共享 module registry 的命名空间中, 并发的 dynamic-require 可能会产生竞争条件; 也可参见 namespace-call-with-registry-lock 。

如果 provided 是 #f, 那么结果是 #<void>, 模块不会被 visit (见 模块扩展、阶段和访问) ， 甚至不会在 base phase 以上的阶段中 available (用于按需 visits) 。

Examples:
> (module a racket/base (displayln "hello"))
> (dynamic-require ''a #f)

hello

双引号 ''a 评估为 root-module-path 'a (见 require 的语法)。 用 'a 表示 mod 是行不通的,因为它的值是 root-module-path a, 而这个例子不是安装在集合中的模块。 使用 a 也行不通,因为 a 是一个未定义的变量。

在另一个模块中声明 (module a ....) ,而不是在 read-eval-print 循环中声明,将创建一个子模块。 在这种情况下, (dynamic-require ''a #f) 将不能访问该模块,因为 ''a 并不指向一个子模块。

当 provided 是一个符号时,将返回给定名称的模块出口的值, 并且仍然不会 visit 该模块或使其在更高阶段 available 。

Examples:
> (module b racket/base
    (provide dessert)
    (define dessert "gulab jamun"))
> (dynamic-require ''b 'dessert)

"gulab jamun"

如果 provided 模块导出为语法,那么绑定的使用将被扩展并在模块所依附的新命名空间中评估, 这意味着模块在新命名空间中被 visit 。 扩展的语法必须返回一个单一的值。

Examples:
> (module c racket/base
    (require (for-syntax racket/base))
    (provide dessert2)
    (define dessert "nanaimo bar")
    (define-syntax dessert2
      (make-rename-transformer #'dessert)))
> (dynamic-require ''c 'dessert2)

"nanaimo bar"

如果模块没有这样的导出变量或语法,那么会调用 fail-thunk ; 默认的 fail-thunk 会引发 exn:fail:contract 。 如果由 provided 命名的变量被导出保护(见 Code Inspectors), 那么会引发 exn:fail:contract exception is raised 。

如果 provided 是 0, 那么模块会被 instantiate , 但不会被 visit ,这与 provided 是 #f 的情况相同。 然而,在 0 的情况下,该模块在更高的阶段是 available 。

如果 provided 是 #<void>, 那么模块被 visit 但不被 instantiate (见 模块扩展、阶段和访问),结果是 #<void> 。

下面给出了更多使用不同 module-path 语法表达的例子:

Example:
> (dynamic-require 'racket/base #f)

Example:
> (dynamic-require (list 'lib "racket/base") #f)

Examples:
> (module a racket/base
    (module b racket/base
      (provide inner-dessert)
      (define inner-dessert "tiramisu")))
> (dynamic-require '(submod 'a b) 'inner-dessert)

"tiramisu"

上述例子中的最后一行可以写成

Example:
> (dynamic-require ((lambda () (list 'submod ''a 'b))) 'inner-dessert)

"tiramisu"

这是等价的。
