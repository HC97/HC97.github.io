在底层操作系统中创建一个新的进程来异步执行 command , 为新进程提供环境变量 current-environment-variables 。 参见 racket/system 中的 system 和 process 。

在 Unix 和 Mac OS 上,子进程的创建与启动 command 所指示的程序是分开的。 特别是,如果 command 指的是一个不存在或不可执行的文件, 那么错误将在子进程中报告(通过标准错误和非 0 退出代码),而不是在创建进程中。

command 参数是一个程序可执行的路径, arg 是该程序的命令行参数。 参见 find-executable-path ,根据 PATH 环境变量定位可执行程序。 在 Unix 和 Mac OS 上,命令行参数是以字节字符串的形式传递的, 而字符串 arg 则使用当前地区的编码进行转换(见 编码和地域)。 在 Windows 上,命令行参数以字符串形式传递,字节字符串使用UTF-8转换。

在 Windows 上,第一个 arg 可以用 'exact 代替,这将触发一个 Windows 特有的行为: 唯一的 arg 被完全用作子进程的命令行。 否则,在 Windows 上,一个命令行字符串是由 command 和 arg 构建的, 这样一个典型的 Windows 控制台应用程序可以将其解析为一个参数数组。 如果在非 Windows 平台上提供 'exact ,就会引发 exn:fail:contract exception is raised 。

关于 Windows 命令行约定的信息,请在 http://msdn.microsoft.com/ 搜索 “command line parsing” 。

当作为端口提供时, stdout 用于启动进程的标准输出, stdin 用于进程的标准输入, 而 stderr 用于进程的标准错误。 所有提供的端口必须是文件流端口。 任何一个端口都可以是 #f ,在这种情况下,一个系统管道被创建并由 subprocess 返回。 stderr 参数可以是 'stdout, 在这种情况下, 作为标准输出的同一文件流端口或系统管道也被用于标准错误。 对于每个提供的端口或 'stdout ,都不会创建管道,相应的返回值是 #f 。 如果 stdout 或 stderr 是一个端口,并且 port-waiting-peer? 返回 true, 那么 subprocess 将等待该端口准备好进行写入,然后再继续创建 subprocess。

如果 group 是 'new,那么新进程将作为一个新的操作系统级进程组被创建。 在这种情况下, subprocess-kill 试图终止该组内的所有进程,其中可能包括子进程创建的其他进程。 请注意,创建一个组可能会干扰交互式 shell 中的作业控制,因为作业控制是基于进程组的。 详见 subprocess-kill 。 如果 group 是一个子进程,那么该子进程必须是用 'new 创建的,新的子进程将被添加到组中; 添加到组中只在 Unix 和 Mac OS 上成功, 而且只在 subprocess-kill 会产生作用的相同情况下(即子进程不知道已经终止), 否则将无声地失败。

subprocess 过程返回四个值:

  • 一个代表所创建进程的 subprocess 值;

  • 一个从进程的标准输出管道输入的端口,如果 stdout 是一个端口,则为 #f ;

  • 一个从进程的标准输入中输送过来的输出端口,如果 stdin 是一个端口,则为 #f ;

  • 从进程的标准错误中输送的输入端口, 或者如果 stderr 是一个端口或 'stdout,则为 #f 。

重要的是： 所有从 subprocess 返回的端口必须明确关闭, 通常用 close-input-port 或 close-output-port 。

用于与子进程通信的 file-stream port 通常是一个容量有限的管道。 小心在子进程做同样事情的时候,通过向子进程串行化写和读,造成死锁, 这样两个进程在写的时候都会阻塞,因为另一端必须先读以在管道中腾出空间。 还要注意等待子进程完成而不读其输出,因为子进程在试图将输出写入一个完整的管道时可能会被阻塞。

返回的端口是 file-stream ports (见 文件端口), 它们被置于当前监护人的管理之下(见 Custodians) 。 当一个低级错误阻止了进程的生成或为进程通信创建操作系统管道时,会引发 exn:fail exception is raised 。

The current-subprocess-custodian-mode 参数决定子进程本身是否在当前的 custodian 处注册,以便监护人关闭时为子进程调用 subprocess-kill 。

current-subprocess-keep-file-descriptors 参数决定了当前进程中的文件描述符和句柄如何与子进程共享。 由 stdin, stdout 和 stderr 代表的文件描述符(在 Unix 和 Mac OS 上)或句柄(在 Windows 上)总是与子进程共享。 默认参数值为 'inherited, 在 Windows 中继承的句柄也被共享,而在 Unix 和 Mac OS 中没有其他文件描述符被共享。 参数值 'all 在 Windows 上等同于 'inherited , 但在 Unix 和 Mac OS 上,当前进程的所有文件描述符都与子进程共享–除了文件描述符 0、1 和 2, 当相应的 stdin,stdout 和 stderr 参数为 #f 时,由新创建的管道取代。 参数值 '() 与 Unix 和 Mac OS 上的 'inherited 相同,但它防止在 Windows 上共享可继承句柄。 (未来的扩展可能支持共享特定的文件描述符或句柄的列表)。

一个子进程可以作为一个 synchronizable event 使用(见 事件)。 当 subprocess-wait 不会阻塞时,一个子进程值是 ready for synchronization ; the synchronization result of a subprocess value is the subprocess value itself 。

Example:

(define-values (sp out in err)
  (subprocess #f #f #f "/bin/ls" "-l"))
(printf "stdout:\n~a" (port->string out))
(printf "stderr:\n~a" (port->string err))
(close-input-port out)
(close-output-port in)
(close-input-port err)
(subprocess-wait sp)

Changed in version 7.4.0.5: 增加了等待没有读者的 fifo 作为 stdout 和/或 stderr 。
Changed in version 8.3.0.4: 增加了对 current-subprocess-custodian-mode 的支持。

(or/c 'running
      exact-nonnegative-integer?)
  subproc : subprocess?
如果 subproc 所代表的进程仍在运行,则返回 'running ,否则返回其退出代码。 退出代码是一个精确的整数, 0 通常表示成功。 如果进程由于故障或信号而终止,则退出代码为非零。

procedure

(subprocess-kill subproc force?) → void?

  subproc : subprocess?
  force? : any/c
终止 subproc 所代表的子进程。 准确的动作取决于 force? 是否为真, 该进程是否通过设置 subprocess-group-enabled 参数为真值而在自己的组中创建,以及当前的平台:

  • force? 为真,不是一个组,所有平台: 如果进程仍在运行,则终止该进程。

  • force? 是假的,不是一个组,在 Unix 或 Mac OS 上: 向进程发送一个中断信号,而不是杀戮信号。

  • force? 是假的,不是一个组,在 Windows 上: 不采取任何行动。

  • force? 是 true,是一个组,在 Unix 或 Mac OS 上: 终止该组中的所有进程, 但前提是 subprocess-status 从未产生过该子进程的非 'running 结果, 而且只有在 subprocess-wait 和 sync 等函数没有检测到该子进程完成的情况下。 否则,不采取任何行动(因为已知直接进程已经终止,而该组的继续存在是未知的)。

  • force? 是真实的,一个组,在 Windows 上: 如果该进程仍在运行,则终止该进程。

  • force? 为 false,一个组,在 Unix 或 Mac OS 上: 与 force? 为 #t 时相同, 但当该组被发送一个信号时,它是一个中断信号,而不是一个杀死信号。

  • force? 是假的,一个组,在 Windows 上: 该组中的所有进程都收到一个 CTRL-BREAK 信号(与直接子进程是否终止无关)。

如果在终止过程中发生错误,会引发 exn:fail exception is raised 。

parameter

(current-subprocess-custodian-mode)

 → (or/c #f 'kill 'interrupt)
(current-subprocess-custodian-mode mode) → void?
  mode : (or/c #f 'kill 'interrupt)
一个 parameter ,用于确定一个子进程(由 subprocess 或像 process 这样的包装器创建) 是否在当前的 custodian 处注册。 如果参数值是 #f, 那么子进程就不会在监管者那里注册–尽管任何创建的端口都会被注册。 如果参数值是 'kill 或 'interrupt, 那么子进程将通过 subprocess-kill 关闭, 其中 'kill 为 force? 参数提供一个 #t 值, 'interrupt 为 #f 值。 关闭可能发生在为子进程创建的端口被关闭之前或之后。

托管人触发的关闭受到主机系统中进程处理细节的限制。 例如, process 和 system 可能会创建一个中间的 shell 进程来运行一个程序, 在这种情况下,基于监护人的终止会关闭 shell 进程,可能不会关闭 shell 启动的进程。 另见 subprocess-kill 。 进程组(见 subprocess-group-enabled) 可以解决一些限制,但不是所有的限制。

procedure

(shell-execute verb    
  target    
  parameters    
  dir    
  show-mode) → #f
  verb : (or/c string? #f)
  target : string?
  parameters : string?
  dir : path-string?
  show-mode : symbol?
在 Windows 中,在 target 上 Performs 由 verb 指定的动作。 对于 Windows 以外的平台,会引发 exn:fail:unsupported exception is raised 。

比如说

(shell-execute #f "http://racket-lang.org" ""
               (current-directory) 'sw_shownormal)

在一个浏览器窗口中打开 Racket 主页。

verb 可以是 #f, 在这种情况下,操作系统会使用一个默认的动词。 常见的动词包括 "open", "edit", "find", "explore" 和 "print" 。

target 是动作的目标,通常是一个文件名路径。 该文件可以是可执行的,也可以是具有公认扩展名的文件,可以由安装的应用程序处理。

parameters 参数被传递给系统以执行该动作。 例如,在打开一个可执行文件的情况下, parameters 被用作命令行(在可执行文件名之后)。

dir 在执行动作时被用作当前目录。

show-mode 设置受操作影响的窗口的显示模式。 它必须是以下符号之一;每个符号的含义描述取自 Windows API 文档。

  • 'sw_hide 或 'SW_HIDE — 隐藏窗口并激活另一个窗口。

  • 'sw_maximize 或 'SW_MAXIMIZE — 使窗口最大化。

  • 'sw_minimize 或 'SW_MINIMIZE — 将窗口最小化,并激活Z型顺序中的下一个顶级窗口。

  • 'sw_restore 或 'SW_RESTORE — 激活并显示该窗口。如果窗口被最小化或最大化,Windows 将其恢复到原来的大小和位置。

  • 'sw_show 或 'SW_SHOW — 激活窗口并以其当前的大小和位置显示它。

  • 'sw_showdefault 或 'SW_SHOWDEFAULT — 使用一个默认值。

  • 'sw_showmaximized 或 'SW_SHOWMAXIMIZED — 激活窗口并以最大化的窗口显示。

  • 'sw_showminimized 或 'SW_SHOWMINIMIZED — 激活窗口并将其显示为最小化窗口。

  • 'sw_showminnoactive 或 'SW_SHOWMINNOACTIVE — 将窗口显示为最小化的窗口。活动窗口保持活动状态。

  • 'sw_showna 或 'SW_SHOWNA — 以当前状态显示该窗口。活动的窗口保持活动状态。

  • 'sw_shownoactivate 或 'SW_SHOWNOACTIVATE — 以最近的大小和位置显示窗口。活动的窗口保持活动状态。

  • 'sw_shownormal 或 'SW_SHOWNORMAL — 激活并显示一个窗口。 如果窗口被最小化或最大化,Windows 将其恢复到原来的大小和位置。

如果该操作失败,会引发 exn:fail exception is raised 。如果操作成功,结果是 #f 。

在 Racket 的未来版本中,如果操作系统确实返回了一个进程句柄, 那么结果可能是一个子进程值(但如果返回一个子进程值,其进程 ID 将是 0 ,而不是真正的进程 ID)。

另见 current-subprocess-custodian-mode 和 subprocess-group-enabled, 它们影响用于实现 system 的子进程。

    • 'exit-code 返回子进程的整数退出代码,如果它仍在运行,则返回 #f 。

    • 'wait 阻断当前线程的执行,直到子进程完成。

    • 'interrupt 在 Unix and Mac OS 中向子进程发送一个中断信号,在 Windows 中不采取任何行动。 其结果是 #<void> 。

      在 Unix 和 Mac OS 上,如果 command 运行一个程序, 那么 sh 通常会以这样的方式运行该程序,在同一进程中取代 sh 。 然而,为了可靠和精确地控制进程的创建,请使用 process* 。

    • 'kill 终止子进程并返回 #<void> 。 注意,由 process 创建的直接进程是一个 shell 进程,它可能运行另一个程序; 终止 shell 进程可能不会终止 shell 启动的进程,特别是在 Windows 上。

重要提示： 从 process 返回的所有三个端口必须用 close-input-port 或 close-output-port 明确关闭。

如果 set-pwd? 为真,那么 PWD 的设置方式与 system 相同。

另请参见 current-subprocess-custodian-mode 和 subprocess-group-enabled, 它们影响用于实现 process 的子进程。 特别是， 'interrupt 和 'kill 进程控制信息是通过 subprocess-kill 实现的, 因此它们可以影响一个进程组而不是一个单一的进程。
