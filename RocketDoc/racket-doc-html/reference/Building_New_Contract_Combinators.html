这些函数分别建立了简单的高阶契约、 chaperone contracts, 和 flat contracts 。 它们都接受相同的三个可选参数:一个名称、一个一阶谓词和一个追责投影。 关于 make-flat-contract, 也请看 flat-contract-with-explanation 。

name 参数是任何要用 display 渲染的值,用来描述违反合同的情况。 简单高阶合约的默认名称是 anonymous-contract, chaperone contracts 的默认名称是 anonymous-chaperone-contract, flat contracts 的默认名称是 anonymous-flat-contract.

一阶谓词 first-order 被用来确定合约适用于哪些值。 这个测试被 contract-first-order-passes? 使用,并且间接地被 or/c 和 first-or/c 使用,并且间接地被 or/c 和 first-or/c 使用, 以确定在有多个高阶合约可供选择时,用哪个高阶合约来包装一个值。 默认值接受任何值,但它必须与投影参数的行为相匹配(具体方法见下文)。 谓词应该受到 (contract-first-order-okay-to-give-up?) 的值的影响(更多解释见它的文档)。

late-neg-proj 参数定义了通过 late neg projection 应用合同的行为。 如果它被提供,这个参数接受一个缺少一方的 blame object (参见 blame-missing-party?)。 然后,它必须返回一个函数,该函数接受得到契约的值和缺失的责任方的名字,按顺序排列。 结果必须是这个值(也许用一个 chaperone 或 impersonator 适当地包装一下,以执行合同), 或者用 raise-blame-error 发出违反合同的信号。默认是 #f 。

collapsible-late-neg-proj 参数代替了支持折叠的合同的 late-neg-proj 参数。 如果它被提供,这个参数接受一个缺少一方的 blame object 。它必须返回两个值。 第一个值必须是一个函数,它接受得到合同的值和缺失的责任方的名字,按顺序排列。 第二个值应该是合同的 collapsible 表示。

投影 proj 和 val-first-proj 是定义应用契约的行为的旧机制。 proj 参数是一个由两个参数组成的curried函数: 第一个应用接受一个责备对象,第二个接受一个值,以保护与合同。 投射必须产生该值,并对其进行适当的包装以执行契约的任何高阶方面, 或者使用 raise-blame-error 发出违反契约的信号。 当一阶测试失败时,默认的投射会产生一个错误,否则会产生不变的值 。 val-first-proj 和 late-neg-proj 一样,只是多了一层currying。

late-neg-proj, proj, val-first-proj 或 first-order 中至少有一个必须是非 #f 。

投射参数 (late-neg-proj, proj 和 val-first-proj) 必须与 first-order 参数同步。 特别是,如果 first-order 参数对某个值返回 #f ,那么投影必须对这个值发出责备错误, 如果 first-order 参数对某个值返回 #t , 那么投影必须对这个值不发出任何责备信号,除非以后有更高阶的交互。 换句话说,对于 flat contracts, first-order 和 projection 参数必须检查相同的谓词。 为了方便起见,默认的投影使用 first-order 参数, 当它返回 #f 时发出错误信号,否则永远不会发出错误信号。

chaperone contracts 的投影必须产生一个与原始的、未收缩的值相比能通过 chaperone-of? 的值。 当 first-order 合约失败时, flat contracts 的投影必须失败,否则必须产生不变的输入值。 应用 flat contract 可能会导致谓词的应用,或者投影的应用,或者两者都是;因此,这两者必须是一致的。 独立投影的存在只是为了提供更具体的错误信息。 大多数 flat contracts 不需要提供明确的投射。

stronger 参数用于实现 contract-stronger? 。 第一个参数总是合同本身,第二个参数是作为第二个参数传递给 contract-stronger? 的东西。 如果没有提供 stronger 参数,那么对于 flat contracts 和 chaperone contracts, 将使用一个默认的将其参数与 equal? 进行比较的参数。 对于用 make-contract 构建的 impersonator contracts , 如果没有提供 stronger 参数, contract-stronger? 返回 #f 。

同样, equivalent 参数用于实现 contract-equivalent?。 如果没有提供这个参数或者提供了 #false , 那么 equal? 将被用于陪衬和平面契约,否则将使用 (λ (x y) #f) 。

is-list-contract? 参数被 list-contract? 谓词使用, 以确定这是否是一个只接受 list? 值的合约。

syntax

(contract-pos/neg-doubling e1 e2)

一些合同组合器需要为子合同建立具有常规和 blame-swap 版本的投射, 以便检查访问和突变(例如, vector/c 和 vectorof)。 在这种组合器被深深地嵌套在彼此之间的情况下,有可能会出现嵌套投影被建立的指数级爆炸。 为了避免这种爆炸,可以用 contract-pos/neg-doubling 来包装对组合器的受责部分的每个调用。 它返回三个值。 第一个是一个布尔值,表示如何解释其他两个结果。 如果布尔值是 #t, 那么另外两个结果就是 e1 和 e2 的值,我们在嵌套中不会太深。 如果布尔值是 #f,那么我们已经超过了一个阈值,现在评估 e1 和 e2 还不安全, 因为我们有可能遇到指数级的减速。 在这种情况下,最后两个结果是thunks,当被调用时,计算 e1 和 e2 的值。

作为一个例子, vectorof 使用 contract-pos/neg-doubling 来包装它对其子包的投射的受责部分的两次调用。 当它收到 #f 作为第一个布尔值时,它不会立即调用 thunks,而是等待,直到它附着在陪护向量上的互斥过程被调用。 然后调用它们(并缓存结果)。 这样就把投影的构建推迟到实际需要的时候,避免了指数级的膨胀。

Added in version 6.90.0.27 of package base.

8.7.1 责备对象

本节描述了 blame objects 和对它们的操作。

procedure

(blame? v) → boolean?

  v : any/c
这个谓词可以识别 blame objects 。

procedure

(raise-blame-error b    
  #:missing-party missing-party    
  v    
  fmt    
  v-fmt ...) → none/c
  b : blame?
  missing-party : #f
  v : any/c
  fmt : 
(or/c string?
      (listof (or/c string?
                    'given 'given:
                    'expected 'expected:)))
  v-fmt : any/c
标志着一个合同的违反。 第一个参数 b 记录了当前的责备信息,包括正方和反方,合同的名称,价值的名称,以及合同应用的来源位置。 #:missing-party 参数提供了其中一个责备方。 当 b 对象在创建时没有提供消极方时,它应该是非 #f 。 参见 blame-add-missing-party 和 make-contract 的 late-neg-proj 参数的描述。

第二个位置参数, v, 是未能满足合同的值。

其余的参数是一个格式字符串, fmt, 和它的参数, v-fmt ...,指定一个特定于精确违反的错误信息。

如果 fmt 是一个列表,那么这些元素被连接在一起(加上空格,除非字符串的末端已经有了空格), 首先用它们的对应字符串替换符号,或者用 "produced" 替换 'given , 用 "promised" 替换 'expected , 这取决于 b 参数是否被交换了(见 blame-swap)。

如果 fmt 包含符号 'given: 或 'expected:, 它们会像 'given 和 'expected 一样被替换, 但替换的内容会以字符串 "\n  " 为前缀,以符合 错误信息约定 的错误信息指南。

procedure

(blame-add-context blame    
  context    
  [#:important important    
  #:swap? swap?]) → blame?
  blame : blame?
  context : (or/c string? #f)
  important : (or/c string? #f) = #f
  swap? : boolean? = #f
在责备错误信息中添加一些上下文信息,说明合同的哪一部分失败了(并且被 raise-blame-error 渲染)。

context 参数描述了合同部分的一个层次,典型的形式是 "the 1st argument of" (在函数合同的情况下)或 "a conjunct of" (在 and/c 合同的情况下)。

例如,考虑这个合同的违反:
> (define/contract f
    (list/c (-> integer? integer?))
    (list (λ (x) x)))
> ((car f) #f)

f: contract violation

  expected: integer?

  given: #f

  in: the 1st argument of

      the 1st element of

      (list/c (-> integer? integer?))

  contract from: (definition f)

它显示被违反的合同部分是 integer? 的第一次出现, 因为 -> 和 list/c 组合器各自在内部调用了 blame-add-context 来增加错误信息中 "in "之后的两行。

swap? 参数具有调用 blame-swap 的效果, 同时增加了上下文层,但没有创建一个额外的 blame object 。

传递 #f 作为 context 字符串参数已经没有意义了。 为了向后兼容, 当 context 是 #f 时, blame-add-context 返回 b .

8.7.2 作为结构的合约

属性 prop:contract 允许任意的结构充当合同。 属性 prop:chaperone-contract 允许任意结构充当 chaperone contracts; prop:chaperone-contract 继承了 prop:contract, 所以 chaperone contract 合同结构也可以充当一般合同。 属性 prop:flat-contract 允许任意结构充当 flat contracts; prop:flat-contract 同时继承了 prop:chaperone-contract 和 prop:procedure, 所以 flat contract 结构也可以充当 chaperone contracts 、一般合约和谓词程序。

contract property 指定了一个结构作为契约使用时的行为。它以七个属性的形式被指定:

  • get-name 产生一个描述,作为合同违约的一部分来 write ;

  • get-first-order, 产生一个一阶谓词,被 contract-first-order-passes? 使用;

  • late-neg-proj, 产生一个定义合同行为的责任追踪投影( get-projection 和 val-first-proj 参数也指定了投影,但使用了不同的签名。它们在这里是为了向后兼容);

  • collapsible-late-neg-proj, 类似于 late-neg-proj , 产生一个定义了契约行为的追责投影,这个函数额外指定了契约的可折叠 collapsible 行为;

  • stronger, 一个决定这个契约(在第一个参数中传递)是否比其他一些契约(在第二个参数中传递)更强的谓词, 其默认值总是返回 #f;

  • equivalent, 一个决定这个合同(在第一个参数中传递)是否等同于其他一些合同(在第二个参数中传递)的谓词; 对于flat和chaperone合同,默认是 equal? 而对于 impersonator 合同则返回 #f;

  • generate, 返回一个生成与合同相匹配的随机值的 thunk (使用 contract-random-generate-fail 表示失败)或 #f 表示不支持该合同的随机生成。;

  • exercise, 它返回一个行使与合同相匹配的值的函数(例如,如果它是一个函数合同,它可以调用该函数) 和一个合同的列表,其值将由这个过程产生。;

  • 和 is-list-contract?, 它被 flat-contract? 用来确定这个合同是否只接受 list? 。

late-neg-proj, collapsible-late-neg-proj, get-projection, val-first-proj 或 get-first-order 中的至少一个必须是非 #f 。

flat contract property 属性指定了一个结构作为 flat contract 使用时的行为。 它通过 build-flat-contract-property 来指定,并接受与 build-contract-property 类似的参数。 不同之处在于:

  • 投影访问器被期望不以高阶方式包裹其参数,类似于 make-flat-contract 中对投影的约束;

  • 省略了 #:exercise 关键字参数,因为它与扁平合同无关。

8.7.3 检查语法中的义务信息

DrRacket中的 Check Syntax 根据合同组合器在程序的扩展形式中留下的 syntax-property 来显示合同的义务信息。 这些属性表明合同在源文件中出现的位置,以及合同的正负位置出现的位置。

要使 Check Syntax 为你的新合同组合器显示义务信息,请使用以下属性(下面是一些辅助宏和函数):

  • 'racket/contract:contract : (vector/c symbol? (listof syntax?) (listof syntax?))
    这个属性应该被附加到实现契约组合器的转化器的结果上。它向 Check Syntax 发出信号,这是一个契约的开始。

    向量中的第一个元素应该是一个唯一的(在 eq? 的意义上)值, Check Syntax 可以使用一个标签来匹配这个契约和它的子件(由下面两个语法属性指定)。

    矢量的第二和第三元素是来自合约中各部分的语法对象,Check Syntax 会给它们着色。 第一个列表应该包含由提供合同实现的各方(通常是模块)负责的子部分。 第二个列表应该包含由客户负责的子部分。

    例如,在 (->* () #:pre #t any/c #:post #t) 中, ->* 和 #:post 应该在第一个列表中, #:pre 在第二个列表中。

  • 'racket/contract:negative-position : symbol?
    这个属性应该被附加到合同组合器的子表达式上,这些子表达式应该是其他合同。 该属性的值应该是键(来自 'racket/contract:contract 属性的向量的第一个元素),表示这是哪个合同。

    当表达式的值是由客户负责的合同时,应该使用这个属性。

  • 'racket/contract:positive-position : symbol?
    这种形式和 'racket/contract:negative-position 一样, 只是当表达式的值是原始方应该负责的合同时应该使用。

  • 'racket/contract:contract-on-boundary : symbol?
    这个属性的存在告诉 Check Syntax 它应该从这个点开始着色。 它希望这个表达式是一个合同(因此有 'racket/contract:contract 属性); 这个属性表示这个合同是在一个(模块)边界上。

    (该属性的值不被使用)。

  • 'racket/contract:internal-contract : symbol?
    和 'racket/contract:contract-on-boundary 一样,这个属性的存在会触发着色, 但这是为了在包含合同的一方(模块)(不管这个模块是否输出与合同匹配的东西)可以被指责为违反了合同时使用。 这在 ->i 合同中会发挥作用,因为合同本身可以通过依赖关系访问合同中的值。

8.7.4 建立新组合器的实用程序

procedure

(contract-stronger? c1 c2) → boolean?

  c1 : contract?
  c2 : contract?
如果合同 c1 接受的值比 c2 少或与之相同,则返回 #t 。

Chaperone contracts 和 flat contracts 如果相同 (即 c1 equal? c2) ,则被认为总是比对方强。

这个函数是保守的,所以当 c1 事实上接受较少的值时,它可能会返回 #f 。

Examples:
> (contract-stronger? integer? integer?)

#t

> (contract-stronger? (between/c 25 75) (between/c 0 100))

#t

> (contract-stronger? (between/c 0 100) (between/c 25 75))

#f

> (contract-stronger? (between/c -10 0) (between/c 0 10))

#f

> (contract-stronger? (λ (x) (and (real? x) (<= x 0)))
                      (λ (x) (and (real? x) (<= x 100))))

#f

procedure

(contract-equivalent? c1 c2) → boolean?

  c1 : contract?
  c2 : contract?
如果合同 c1 接受的价值集与 c2 相同,则返回 #t 。

Chaperone contracts 和 flat contracts 如果相同 (即 c1 equal? c2) ,则被认为总是相互等同的。

这个函数是保守的,所以当 c1 事实上接受与 c2 相同的值集时,它可能返回 #f 。

Examples:
> (contract-equivalent? integer? integer?)

#t

> (contract-equivalent? (non-empty-listof integer?)
                        (cons/c integer? (listof integer?)))

#t

> (contract-equivalent? (λ (x) (and (real? x) (and (number? x) (>= (sqr x) 0))))
                        (λ (x) (and (real? x) (real? x))))

#f

Added in version 6.90.0.30 of package base.

procedure

(contract-first-order-passes? contract v) → boolean?

  contract : contract?
  v : any/c
返回一个布尔值,表明 contract 的一阶测试是否通过了 v 。

如果它返回 #f, 合同对该值保证不成立;如果它返回 #t, 合同可能成立也可能不成立。 如果合同是一个一阶合同, #t 的结果保证合同成立。

参见 contract-first-order-okay-to-give-up? 和 contract-first-order-try-less-hard 。
