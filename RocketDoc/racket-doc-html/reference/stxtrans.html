12.4 句法变换器

procedure

(set!-transformer? v) → boolean?

  v : any/c
如果 v 是由 make-set!-transformer 创建的值或者是具有 prop:set!-transformer 属性的结构类型的实例,则返回 #t ,否则返回 #f 。

procedure

(make-set!-transformer proc) → set!-transformer?

  proc : (syntax? . -> . syntax?)
创建一个与 assignment transformer 配合的 set! 。 如果 make-set!-transformer 的结果作为 transformer 绑定在 id 上, 那么当 id 在表达式位置上使用时, 或者当它作为 set! 赋值的目标作为 (set! id expr) 时, proc 将作为转化器应用。 当标识符作为 set! 目标出现时,整个 set! 表达式被提供给转化器。

Example:
> (let ([x 1]
        [y 2])
    (let-syntax ([x (make-set!-transformer
                      (lambda (stx)
                        (syntax-case stx (set!)
                          ; Redirect mutation of x to y
                          [(set! id v) #'(set! y v)]
                          ; Normal use of x really gets x
                          [id (identifier? #'id)  #'x])))])
      (begin
        (set! x 3)
        (list x y))))

'(1 3)

procedure

(set!-transformer-procedure transformer)

 → (syntax? . -> . syntax?)
  transformer : set!-transformer?
返回创建 transformer 时传递给 make-set!-transformer 的过程, 或者由 transformer 的 prop:set!-transformer 属性标识的过程。

value

prop:set!-transformer : struct-type-property?

一个 structure type property 用于识别像 make-set!-transformer 所创建的 assignment transformers 那样的结构类型。

该属性值必须是一个精确的整数或一个或两个参数的过程。 在前一种情况下,整数指定了结构中的一个字段,该字段应该包含一个过程; 整数必须在 0 (包括)和结构类型中的非自动字段数量之间(不包括,不计算超类型字段), 并且指定的字段还必须被指定为不可变的。

如果属性值是一个参数的过程,那么该过程作为 syntax transformer 和用于 set! 转换。 如果属性值是一个有两个参数的过程,那么第一个参数是结构,其类型有 prop:set!-transformer 属性, 第二个参数是一个语法对象,作为 syntax transformer 和用于 set! 转换; set!-transformer-procedure 应用于结构产生一个新的函数,它只接受语法对象并通过属性调用相关的过程。 最后,如果属性值是一个整数,则从结构实例中提取目标标识符;如果字段值不是一个参数的过程, 则使用一个总是调用 raise-syntax-error 的过程来代替。

如果一个值同时具有 prop:set!-transformer 和 prop:rename-transformer 属性,那么后者优先。 如果一个结构类型有 prop:set!-transformer 和 prop:procedure 属性,那么在宏扩展时,前者优先。

procedure

(rename-transformer? v) → boolean?

  v : any/c
如果 v 是由 make-rename-transformer 创建的值或者是具有 prop:rename-transformer 属性的结构类型的实例,则返回 #t ,否则返回 #f 。

Examples:
> (rename-transformer? (make-rename-transformer #'values))

#t

> (rename-transformer? 'not-a-rename-transformer)

#f

procedure

(make-rename-transformer id-stx) → rename-transformer?

  id-stx : syntax?
创建一个 rename transformer ,当作为 transformer 绑定时, 作为一个变换器,插入标识符 id-stx 来代替绑定变换器的任何标识符, 包括在非应用位置,在 set! 表达式中。

这样的转化器可以手工编写,但是由 make-rename-transformer 创建的转化器在 id 绑定到 rename 转化器时,会触发与解析器和其他语法形式的特殊合作:

  • 只要 id-stx 没有 'not-free-identifier=? syntax property 的真值, 解析器就会在 id 和 id-stx 之间安装 free-identifier=? 和 identifier-binding 的等效关系。

  • 只要 id-stx 的 'not-free-identifier=? syntax property 没有一个真值, 并且只要 id-stx 有一个绑定, provide 的 id 就会提供 id-stx 所指示的绑定,而不是 id 。

  • 如果 provide 输出 id, 它使用符号值的 'nominal-id 属性 id-stx 来指定由 identifier-binding 报告的绑定的 "名义源标识符"。

  • 如果 id-stx 的 'not-provide-all-defined syntax property 为真值, 那么 id (或其目标)就不会被 all-defined-out 导出。

  • syntax-local-value 函数可以识别重名变换器的绑定,并咨询它们的目标。

Examples:
> (define-syntax my-or (make-rename-transformer #'or))
> (my-or #f #t)

#t

> (free-identifier=? #'my-or #'or)

#t

procedure

(rename-transformer-target transformer) → identifier?

  transformer : rename-transformer?
返回传递给 make-rename-transformer 以创建 transformer 的标识符, 或者如 transformer 上的 prop:rename-transformer 属性所指示。

Example:
> (rename-transformer-target (make-rename-transformer #'or))

#<syntax:eval:8:0 or>

value

prop:rename-transformer : struct-type-property?

一个 structure type property ,用于识别作为 rename transformers 的结构类型, 如 make-rename-transformer 所创建的结构类型。

该属性值必须是一个精确的整数,一个标识符 syntax object ,或者一个需要一个参数的过程。 在前一种情况下,整数指定了结构中应该包含标识符的字段; 整数必须在 0 (包括)和结构类型中非自动字段的数量之间(不包括,不计算超类型字段), 并且指定的字段也必须被指定为不可变的。

如果属性值是一个标识符,该标识符将作为重命名的目标,就像 make-rename-transformer 的第一个参数。 如果属性值是一个整数,则从结构实例中提取目标标识符;如果字段值不是一个标识符, 则使用带有空上下文的标识符 ? 。

如果属性值是一个需要一个参数的过程,那么该过程被调用以获得重命名转换器将作为目标标识符的标识符。 返回的标识符可能应该具有 'not-free-identifier=? 语法属性。 如果该过程返回任何不是标识符的值,就会引发 exn:fail:contract 。

Examples:
; Example of a procedure argument for prop:rename-transformer
> (define-syntax slv-1 'first-transformer-binding)
> (define-syntax slv-2 'second-transformer-binding)
> (begin-for-syntax
    (struct slv-cooperator (redirect-to-first?)
      #:property prop:rename-transformer
      (λ (inst)
        (if (slv-cooperator-redirect-to-first? inst)
            #'slv-1
            #'slv-2))))
> (define-syntax (slv-lookup stx)
    (syntax-case stx ()
      [(_ id)
       #`'#,(syntax-local-value #'id)]))
> (define-syntax slv-inst-1 (slv-cooperator #t))
> (define-syntax slv-inst-2 (slv-cooperator #f))
> (slv-lookup slv-inst-1)

'first-transformer-binding

> (slv-lookup slv-inst-2)

'second-transformer-binding

procedure

(local-expand stx    
  context-v    
  stop-ids    
  [intdef-ctx]) → syntax?
  stx : any/c
  context-v : (or/c 'expression 'top-level 'module 'module-begin list?)
  stop-ids : (or/c (listof identifier?) empty #f)
  intdef-ctx : 
(or/c internal-definition-context?
      #f
      (listof internal-definition-context?))
 = #f
在当前被展开的表达式的词法上下文中展开 stx 。 context-v 参数作为 syntax-local-context 的结果用于即时展开; 列表表示 internal-definition context ,关于列表形式的更多信息见下文。 如果 stx 还不是一个 syntax object, 那么在扩展之前会用 (datum->syntax #f stx) 强制执行。

stop-ids 参数控制 local-expand 对 stx 的扩展程度:

  • 如果 stop-ids 是一个空列表,那么 stx 将被递归扩展(即扩展到子表达式)。 结果保证是一个完全展开的形式,它可以包括 完全展开的程序 中列出的绑定, 以及任何表达位置的 #%expression 。

  • 如果 stop-ids 是一个只包含 module* 的列表, 那么扩展过程就像 stop-ids 是一个空列表一样, 除了扩展不会递归到用 module* 定义的 submodules (在结果中未被扩展)。

  • 如果 stop-ids 是任何其他列表,那么 begin, quote, set!, #%plain-lambda, case-lambda, let-values, letrec-values, if, begin0, with-continuation-mark, letrec-syntaxes+values, #%plain-app, #%expression, #%top 和 #%variable-reference 被隐式加入 stop-ids 。 扩张是递归进行的,当扩张器遇到 stop-ids 中的任何形式时就会停止,其结果是部分扩张的形式。

    当扩展器通常会隐含地引入一个 #%app, #%datum 或 #%top 标识符, 如 扩展步骤 所述,它检查是否有一个与要引入的标识符相同的 binding 出现在 stop-ids 中。 如果是这样,标识符就 不 会被引入; 扩展的结果是裸露的应用程序、字面数据表达式或未绑定的标识符,而不是被包裹在各自的显式形式中。

    当 #%plain-module-begin 不在 stop-ids 中时, #%plain-module-begin 转化器会检测并扩展子形式(比如 define-values) , 而不管标识符在 stop-ids 中是否存在。

    扩展不会替换局部变量引用中的作用域以匹配绑定标识符。

  • 如果 stop-ids 是 #f 而不是一个列表,那么只有当 stx 的最外层形式是一个宏时, stx 才会被扩展(也就是说,扩展 不 会进行到子表达式,也不会替换局部变量引用中的作用域以匹配绑定标识符)。 #%app, #%datum 和 #%top 标识符从未被引入。

与 stop-ids 无关,当 local-expand 遇到一个有本地绑定但在当前扩展上下文中没有绑定的标识符时, 该变量将保持原样(而不是触发一个 "超出上下文" 的语法错误)。

当 context-v 是 'module-begin 时,并且扩展的结果是 #%plain-module-begin 形式, 那么一个 'submodule syntax property 被添加到每个被包围的 module 形式(但不是 module* 形式),其方式与 module 扩展相同。

如果 intdef-ctx 参数是一个内部定义上下文,那么在调用 local-expand 的动态范围内, 它的 bindings 和来自所有 parent internal-definition contexts 的 bindings 被添加到 local binding context 中。 此外,除非在创建内部定义上下文时为 syntax-local-make-definition-context 的 add-scope? 参数提供了 #f , 否则其 inside-edge scope (而 不 是任何 parent internal-definition contexts 的范围) 将被添加到 stx 在其扩展前和扩展结果的 lexical information 中 (因为扩展可能引入绑定或对内部定义绑定的引用)。

为了向后兼容,当 intdef-ctx 是一个列表时, 来自所有提供的内部定义上下文的所有 bindings 和它们的父级被添加到 local binding context 中, 来自每个 add-scope? 不是 #f 的上下文的 inside-edge scope 也以同样的方式添加。

扩充记录用于从定义绑定中移除的 use-site scopes 。 当 intdef-ctx 参数是一个内部定义上下文时,使用现场作用域被记录在该上下文中。 当 intdef-ctx 是 #f 或(为了向后兼容)一个列表时,使用站点作用域被记录在当前的扩展上下文中。

对于一个特定的 internal-definition context, 生成一个唯一的值,并将其放入 context-v 的列表中。 为了允许 define 形式的 liberal expansion ,生成的值应该是一个结构的实例, prop:liberal-define-context 的值为真。 如果内部定义上下文是自成一体的,那么用于 context-v 的列表应该只包含生成的值; 如果内部定义上下文是为了拼接到一个紧邻的上下文中,那么当 syntax-local-context 产生一个列表时, 就把生成的值 cons 到这个列表上。

当表达式通过内部定义上下文 intdef-ctx 的 local-expand 扩展时, 当扩展后的表达式被并入整体形式 new-stx 时, 通常应将 internal-definition-context-track 应用于 intdef-ctx 和 new-stx 以向外部工具提供扩展历史。

这个过程必须在扩展器应用 syntax transformer 的动态范围内或在模块被 visit时被调用(见 syntax-transforming?), 否则会引发 exn:fail:contract exception is raised 。

Examples:
> (define-syntax-rule (do-print x ...)
    (printf x ...))
> (define-syntax-rule (hello x)
    (do-print "hello ~a" x))
> (define-syntax (show stx)
    (syntax-case stx ()
      [(_ x)
       (let ([partly (local-expand #'(hello x)
                                   'expression
                                   (list #'do-print))]
             [fully (local-expand #'(hello x)
                                  'expression
                                  #f)])
         (printf "partly expanded: ~s\n" (syntax->datum partly))
         (printf "fully expanded: ~s\n" (syntax->datum fully))
         fully)]))
> (show 1)

partly expanded: (do-print "hello ~a" 1)

fully expanded: (printf "hello ~a" 1)

hello 1

这个过程的绑定被提供为 protect-out 意义上的 protected 。

Changed in version 6.0.90.27: 放松了对 intdef-ctx 参数的约束,允许使用空列表。
Changed in version 8.2.0.4: 将绑定改为 protected 。

procedure

(syntax-local-expand-expression stx 
  [opaque-only?]) 
 → 
(if opaque-only? #f syntax?) syntax?
  stx : any/c
  opaque-only? : any/c = #f
像 local-expand 一样,给出 'expression 和一个空的停止列表,但有两个结果: 一个是完全展开的表达式的语法对象,另一个是内容不透明的语法对象。

后者可以用来代替前者(也许是在由宏转换器产生的更大的表达式中),当宏扩展器遇到不透明对象时, 它会替换完全扩展的表达式,而不重新扩展它;如果扩展的上下文包括原始扩展时不存在的 scopes , 就会引发 exn:fail:syntax exception is raised ,在这种情况下,重新扩展可能产生不同的结果。 因此,一致地使用 syntax-local-expand-expression 和 opaque 对象可以避免在局部扩展被嵌套时产生二次扩展时间。

如果 opaque-only? 为真,那么第一个结果是 #f ,而不是扩展后的表达式。 只获得第二个不透明的结果,在某些扩展环境中可能更有效率。

与 local-expand 不同, syntax-local-expand-expression 通常产生一个不包含 #%expression 形式的扩展表达式。 然而,如果 syntax-local-expand-expression 是在一个由封闭的 local-expand 调用触发的扩展中使用, 那么 syntax-local-expand-expression 的结果可以包含 #%expression 形式。

这个过程必须在扩展器应用 syntax transformer 的动态范围内或在模块被 visit时被调用(见 syntax-transforming?), 否则会引发 exn:fail:contract exception is raised 。
这个过程的绑定被提供为 protect-out 意义上的 protected 。

Changed in version 6.90.0.13 of package base: 增加了 opaque-only? 参数。
Changed in version 8.2.0.4: 修改了绑定为 protected 。

procedure

(syntax-local-make-definition-context [parent-ctx 
  add-scope?]) 
 → internal-definition-context?
  parent-ctx : (or/c internal-definition-context? #f) = #f
  add-scope? : any/c = #t
创建一个不透明的 internal-definition context 值,以便与 local-expand 和其他函数一起使用。 一个转化器应该为每一组要展开的内部定义创建一个上下文。

在扩展词法上下文应该包括定义的形式之前, 转换者应该使用 internal-definition-context-add-scopes 来将上下文的作用域应用于语法。 调用诸如 local-expand 之类的程序来扩展形式,应该提供 internal-definition context 值作为参数。

在发现内部 define-values 或 define-syntaxes 形式后, 使用 syntax-local-bind-syntaxes 来为上下文添加 bindings 关系。

一个 internal-definition context 在内部创建了一个 outside-edge scope 和一个 inside-edge scope 来表示该上下文。 inside-edge scope 被添加到任何在上下文中被展开的形式或作为上下文中(部分)展开的结果出现的形式。 为了向后兼容,为 add-scope? 提供 #f 将禁用这种行为。

如果 parent-ctx 不是 #f, 那么 parent-ctx 将成为新的内部定义上下文的 parent internal-definition context 。 每当新上下文的 bindings 被添加到 local binding context (例如,通过向 local-expand, syntax-local-bind-syntaxes 或 syntax-local-value 提供上下文),那么来自 parent-ctx 的绑定也会被添加。 如果 parent-ctx 也是用 parent internal-definition context 创建的, 那么来自其父级的 bindings 也会被添加,依次类推。 请注意,父上下文的 scopes 不 会被隐式添加,只有 bindings, 即使子上下文的 inside-edge scope 也会被隐式添加。 如果要添加父定义上下文的 scopes ,必须明确提供父上下文。

此外,如果创建的定义上下文打算被拼接到周围的定义上下文中, 那么周围的上下文应该始终被提供给 parent-ctx 参数, 以确保必要的 use-site scopes 被添加到上下文中扩展的宏中。 否则,嵌套定义的扩展可能与周围上下文中定义的扩展不一致。

internal-definition context 还跟踪在定义上下文的扩展过程中创建的 use-site scopes , 这样它们就可以从上下文中创建的绑定中删除, 在 syntax-local-identifier-as-binding 和 internal-definition-context-splice-binding-identifier 。

与新定义上下文相关的作用域只有在 quote-syntax 应用的动态范围内或在正在扩展的模块中的 syntax transformer 可能是嵌套的)创建时, 才会从 begin-for-syntax 形式中修剪。

这个过程必须在扩展器应用 syntax transformer 的动态范围内或在模块被 visit时被调用(见 syntax-transforming?), 否则会引发 exn:fail:contract exception is raised 。

12.4.2 provide 变形器

 (require racket/provide-transform) package: base
The bindings documented in this section are provided by the racket/provide-transform library, not racket/base or racket.

一个 transformer 绑定的值是一个带有 prop:provide-transformer 属性的结构, 它实现了 provide 的派生 provide-spec ,作为一个 provide transformer 。 在模块中的所有其他声明和表达式被扩展后, provide transformer 被作为模块扩展的最后阶段的一部分来应用。

其值为带有 prop:provide-pre-transformer 属性的结构的 transformer , 实现了作为 provide pre-transformer 的 provide 的派生 provide-spec 。 provide pre-transformer 被作为模块扩展的第一阶段的一部分来应用。 因为它是在第一阶段使用的,所以 provide pre-transformer 可以使用诸如 syntax-local-lift-expression 这样的函数来引入包围模块中的表达式和定义。

一个标识符可以有一个与值绑定的 transformer , 它既可以作为 provide transformer 又可以作为 provide pre-transformer 。 provide pre-transformer 的结果 不 会自动再扩展,所以在这种情况下, provide pre-transformer 可以有效地扩展到自己。

变换器被调用,它的语法对象代表它在 provide 形式中作为 provide-spec 使用, 它的符号列表代表由包围的 provide-spec 指定的输出模式。 provide transformer 的结果必须是一个 export , 而 provide pre-transformer 的结果是一个语法对象,将在模块扩展的最后阶段作为 provide-spec 使用。

如果一个派生形式包含一个作为 provide-spec 的子形式, 那么它可以调用 expand-export 或 pre-expand-export 来转换子 provide-spec 子形式。

另请参见 define-provide-syntax, 它支持宏式 provide transformers 。

procedure

(expand-export provide-spec modes) → (listof export?)

  provide-spec : syntax?
  modes : (listof phase+space?)
将给定的 provide-spec 扩展到一个出口列表。 modes 列表控制了子 provide-specs 的扩展; 例如,除非 modes 列表另有规定, 否则标识符指的是包围 provide 形式的 phase level 中的一个绑定。 通常情况下, modes 要么是空的,要么包含一个元素。 将模式泛化为相位空间的组合。

Changed in version 8.2.0.3 of package base: 将 modes 泛化为相位空间的组合。

procedure

(pre-expand-export provide-spec modes) → syntax?

  provide-spec : syntax?
  modes : (listof phase+space?)
在 provide pre-transformers 的层面上扩展给定的 provide-spec 。 参数 modes 与 expand-export 的参数相同。

Changed in version 8.2.0.3 of package base: 将 modes 泛化到相位空间组合中。

procedure

(make-provide-transformer proc) → provide-transformer?

  proc : 
(syntax? (listof phase+space?)
 . -> . (listof export?))
(make-provide-transformer proc pre-proc)
 → (and/c provide-transformer? provide-pre-transformer?)
  proc : 
(syntax? (listof phase+space?)
 . -> . (listof export?))
  pre-proc : 
(syntax? (listof phase+space?)
 . -> . syntax?)
使用给定的程序作为转化器,创建一个 provide transformer (即具有 prop:provide-transformer 属性的结构)。 如果提供了一个 pre-proc ,那么结果也是一个 provide pre-transformer 。 通常与 expand-export 和/或 pre-expand-export 结合使用。

procedure

(make-provide-pre-transformer pre-proc)

 → provide-pre-transformer?
  pre-proc : 
(syntax? (listof phase+space?)
 . -> . syntax?)
与 make-provide-transformer 类似,但只针对一个 provide pre-transformer 的值。 通常与 pre-expand-export 结合使用。

Examples:
> (module m racket
    (require
      (for-syntax racket/provide-transform syntax/parse syntax/stx))
  
    (define-syntax wrapped-out
      (make-provide-pre-transformer
       (lambda (stx modes)
         (syntax-parse stx
           [(_ f ...)
            #:with (wrapped-f ...)
                   (stx-map
                    syntax-local-lift-expression
                    #'((lambda args
                         (printf "applying ~a, args: ~a\n" 'f args)
                         (apply f args)) ...))
            (pre-expand-export
             #'(rename-out [wrapped-f f] ...) modes)]))))
  
    (provide (wrapped-out + -)))
> (require 'm)
> (- 1 (+ 2 3))

applying +, args: (2 3)

applying -, args: (1 5)

-4

value

prop:provide-transformer : struct-type-property?

一个用于识别 provide transformers 的属性。 该属性的值必须是一个过程,该过程接收结构并返回一个转化器过程; 返回的转化器过程接收一个语法对象和模式列表并返回一个输出列表。

value

prop:provide-pre-transformer : struct-type-property?

一个用于识别 provide pre-transformers 的属性。 该属性值必须是一个接受结构并返回转化器过程的过程; 返回的转化器过程接受一个语法对象和模式列表并返回一个语法对象。

procedure

(provide-transformer? v) → boolean?

  v : any/c
如果 v 有 prop:provide-transformer 属性,返回 #t ,否则返回 #f 。

procedure

(provide-pre-transformer? v) → boolean?

  v : any/c
如果 v 有 prop:provide-pre-transformer 属性,则返回 #t ,否则返回 #f 。

struct

(struct export (local-id out-sym mode protect? orig-stx)
    #:extra-constructor-name make-export)
  local-id : identifier?
  out-sym : symbol?
  mode : phase+space?
  protect? : any/c
  orig-stx : syntax?
一个代表单一出口标识符的结构:

  • local-id — 绑定在导出模块中的标识符。

  • out-sym — 绑定的外部名称。

  • mode — 出口的 phase level 和 binding space (影响它的导入方式)。

  • protect? — 表示标识符是否应该被保护(见 Code Inspectors)。

  • orig-stx — 导出源的 syntax object ,用于报告错误。

Changed in version 8.2.0.3 of package base: 将 mode 泛化为相位空间组合。
