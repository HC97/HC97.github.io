14.2 评价和编译

反射和动态求值 in Racket 指南 introduces dynamic evaluation.

Racket 通过 eval 和相关函数提供了对评价的程序化控制。 关于 Racket 编译器相关的语言外设施的信息,请参见 Controlling and Inspecting Compilation 。

parameter

(current-eval) → (any/c . -> . any)

(current-eval proc) → void?
  proc : (any/c . -> . any)
一个 parameter ,决定了当前的 evaluation handler。 评价处理程序是一个过程,它接收一个顶层表单并对其进行评价,返回结果值。 evaluation handler 被 eval, eval-syntax, 默认 load handler 和 read-eval-print-loop 调用,以评价一个顶层表单。 该处理程序应该在尾部位置评估其参数。

提供给处理程序的 top-level-form 可以是一个语法对象、 一个编译过的表单、一个被包装成语法对象的编译过的表单,或者一个任意数据。

默认处理程序使用 datum->syntax 将任意数据集转换为语法对象, 然后以与 eval 相同的方式丰富其 lexical information 。 (如果 top-level-form 是一个语法对象,那么它的 lexical information 就不会被丰富。) 默认的评估处理程序部分扩展表单,将顶层 begin 表单的主体拼接到顶层(见 expand-to-top-form), 然后在继续扩展、编译和评估后来的表单之前,单独编译和评估每个拼接的表单。

procedure

(eval top-level-form) → any

  top-level-form : any/c
(eval top-level-form namespace) → any
  top-level-form : any/c
  namespace : namespace?

See also 命名空间 in Racket 指南.

调用当前的 evaluation handler 来评估 top-level-form 。 相对于 eval 的调用,评估处理程序是在尾部位置被调用的。 评价处理程序使用 current namespace ;在 eval 的双参数情况下, 对评价处理程序的调用被 parameterize , 以将 current-namespace 设置为 namespace 。

如果 top-level-form 是一个语法对象,其基准不是一个编译过的形式, 那么它的 lexical information 在被发送到 evaluation handler 之前就被丰富了:

  • 如果 top-level-form 是一个 car 是一个符号或标识符的对, 并且如果将 namespace-syntax-introduce 应用到 (datum->syntax-converted) 标识符, 则在对应于 namespace 的 base phase 的 phase level 生成绑定到 module 的标识符, 则只有该标识符被充实。

  • 对于任何其他 top-level-form, namespace-syntax-introduce 被应用于整个句法对象。

对于 read-eval-print-loop 和 load 风格的交互式评价,在将其传递给 eval 之前, 用 #%top-interaction 包裹每个表达式,该表达式通常被绑定到 #%top-interaction 。

procedure

(eval-syntax stx) → any

  stx : syntax?
(eval-syntax stx namespace) → any
  stx : syntax?
  namespace : namespace?
和 eval 一样,除了 stx 必须是一个语法对象, 而且它的词法上下文在被传递给 evaluation handler 。

(current-load)

 →

             symbol?

       any)
一个 parameter ,决定了当前从文件中加载顶层表单的 load handler 。 load, load-relative, load/cd 和默认的 compiled-load handler 都会调用该 load handler 。

一个加载程序需要两个参数:一个路径(见 Paths) 和一个期望的模块名称。 预期的模块名称是一个符号或一个列表,当调用是为了加载一个响应 require 的模块声明(在这种情况下,文件应该包含一个模块声明),或者对于任何其他的加载来说是 #f 。

当从一个以包含子模块的编译模块开始的流中加载模块时,加载处理程序应该只加载请求的模块, 其中作为加载处理程序的符号表示根模块,列表表示一个子模块,其相对于根模块的路径由列表的 cdr 给出。 列表以 #f 开头,当一个子模块 只有 在可以独立加载的情况下才应该被加载(即,从编译的形式-从未从源代码); 如果子模块不能独立加载,加载处理程序应该返回而不从文件中加载。 当预期的模块名称是一个以符号开始的列表时,根模块和任何其他子模块可以从给定的文件中加载, 该文件可能来自源文件,如果没有找到预期的子模块,加载程序仍然不应该抱怨。 当从一个不存在的源文件加载模块时,无论是否请求子模块,加载处理程序都可能引发一个异常。

默认的加载处理程序以 read-syntax 从文件中读取表格, 并启用文件端口的行数计算,除非路径有一个 ".zo" 的后缀。 它还将每次读取的 parameterize 设置为 read-accept-compiled, read-accept-reader 和 read-accept-lang 为 #t 。 此外,如果 load-on-demand-enabled 是 #t, 那么在 read-syntax 调用过程中, read-on-demand-source 被设置为 cleanse 、绝对的 path 形式。 在读取单个表单后,该表单被传递给当前的 evaluation handler, 将评估包裹在一个延续提示中(见 call-with-continuation-prompt) , 默认的延续提示标签与处理程序一起将中止传播到 load 调用的延续中。

如果加载处理程序的第二个参数是一个符号,那么:

  • 从文件中 read-syntax 被额外地 parameterize ,如下(以提供一致的模块源的读取):

    (current-readtable #f)
    (read-case-sensitive #t)
    (read-square-bracket-as-paren #t)
    (read-curly-brace-as-paren #t)
    (read-accept-box #t)
    (read-accept-compiled #t)
    (read-accept-bar-quote #t)
    (read-accept-graph #t)
    (read-syntax-accept-graph #f)
    (read-decimal-as-inexact #t)
    (read-accept-dot #t)
    (read-accept-infix-dot #t)
    (read-accept-quasiquote #t)
    (read-accept-reader #t)
    (read-accept-lang #t)

  • 如果读取的结果不是一个 module 的形式, 或者如果第二个 read-syntax 没有产生文件结束,那么 exn:fail exception is raised 将被引发, 而不评估从文件中读取的形式。 (在以前的版本中,模块声明被检查为与作为加载处理程序第二个参数的名称相匹配,但现在不再进行这种检查)。

  • 初始 module 标识符的 lexical information 用 module 的绑定来充实, 因此该表格对应于独立于当前命名空间绑定的模块声明。

如果加载处理程序的第二个参数是 #f , 那么从文件中读取的每个表达式都会被 #%top-interaction 包裹, 通常会被绑定到 #%top-interaction, 然后再传递给 evaluation handler 。

默认 load handler 的返回值是加载文件中最后一个表单的值,如果文件中没有表单,则返回 #<void> 。 如果给定的路径是一个相对路径,那么它将使用 current-directory 的值来解析。

procedure

(load file) → any

  file : path-string?

See also 命名空间 in Racket 指南.

一个 parameter ,用于确定当前的 compiled-load handler ,以便从一个可能有编译形式的文件中加载。 compiled-load handler 由 load/use-compiled 调用。

compiled-load handler 的协议与 load handler (见 current-load), 只是 compiled-load handler 应该自己设置 current-load-relative-directory 。 此外,默认的 compiled-load handler 做了以下事情:

  • 当给定的路径以 ".rkt" 结尾时,不存在 ".rkt" 文件, 并且当处理程序的第二个参数不是 #f , 默认的 compiled-load handler 检查 ".ss" 文件。

  • 默认的 compiled-load handler 检查是否有机会从 ".zo" (字节码)文件中加载, 当 (system-type 'vm) 返回 'racket 时, 检查 ".so" (本地 Unix), ".dll" (本地 Windows) 或 ".dylib" (本地 Mac OS) 文件。

  • 当默认的 compiled-load handler 需要从给定的路径加载时, 给定的路径不存在,当处理程序的第二个参数不是 #f 时, 默认的 compiled-load handler 会返回,不会引发异常。

只要给定的路径 file 以任何扩展名(例如, ".rkt" 或 ".scrbl")结尾,就会发生对已编译文件的检查, 并且检查由 current-compiled-file-roots 和 use-compiled-file-paths 参数指示的相对于 file 的子目录,其中前者提供编译文件的 "根",后者则提供子目录。 参见 compiler/compilation-path. “root” 可以是一个绝对路径,在这种情况下, file 的目录将与 reroot-path 结合起来,根作为第二个参数; 如果 "根 "是一个相对路径,那么相对路径将作为 file 目录的后缀。 根目录被依次尝试,而子目录在每个根目录中被依次检查。 如果一个 ".zo" 版本的文件(其名称由传递 file 和 #".zo" 到 path-add-extension) 形成)直接存在于指定的子目录中,则会被加载, 或者当 (system-type 'vm) 返回 'racket 时, 如果该文件存在于 use-compiled-file-paths 目录的 "native" 子目录中, 甚至是由 system-library-subpath 命名的更深的子目录中, 则会加载一个 ".so"/".dll"/".dylib" 版本。 只有当一个编译文件根据 (use-compiled-file-check) 检查出来时才会被加载; 在默认参数值为 'modify-seconds 时, 只有当一个编译文件的修改日期不超过 file 的日期时才会被使用。 如果当 (system-type 'vm) 返回 'racket 时, ".zo" 和 ".so"/".dll"/".dylib" 文件都可用, 那么将使用 ".so"/".dll"/".dylib" 文件。 如果文件以 ".rkt" 结尾,不存在这样的 file ,处理程序的第二个参数是一个符号, 并且存在一个 ".ss" 文件,那么 ".zo" 和 ".so"/".dll"/".dylib" 文件只使用基于 file 的名称, 其后缀被 ".ss" 替换。

当 ".zo", ".so", ".dll" 或 ".dylib" 文件被加载时, 当前 load-relative 目录被设置为原始 file 的目录。 如果要加载的文件的后缀是 ".ss" ,而被请求的文件的后缀是 ".rkt", 那么 current-module-declare-source 参数被设置为加载文件的完整路径, 否则 current-module-declare-source 参数被设置为 #f 。

如果加载的是原始 file 或者加载的是 ".zo" 变体, 则调用 load handler 来加载该文件。如果任何其他类型的文件被加载, 就会调用 extension-load handler 。

当默认的 compiled-load handler 从字节码(即 ".zo") 文件加载模块时, 处理程序在当前命名空间的 module registry 中记录字节码文件路径。 更具体地说,处理程序记录了加载模块的顶层模块的路径,如果加载的模块是一个子模块,它就是一个封闭的模块。 此后,通过默认的 compiled-load handler 对同一顶层模块的加载使用记录的文件, 与 compiled-load handler 选择的文件无关(例如,即使 use-compiled-file-paths 参数值改变)。 当一个模块声明被附加到一个新的命名空间时,默认的 module name resolver 传输字节码文件信息。 该协议支持从字节码文件中独立但一致的加载子模块。

parameter

(use-compiled-file-check) → (or/c 'modify-seconds 'exists)

(use-compiled-file-check check) → void?
  check : (or/c 'modify-seconds 'exists)
一个 parameter ,它决定了如何将编译的文件与它的源文件进行检查,以便能够使用编译的文件。 默认情况下,文件检查模式是 'modify-seconds, 当文件系统的修改日期至少与源文件的修改日期一样新时,就使用编译文件。 'exists 模式使编译后的文件被用来代替其源文件,只要编译后的文件存在。

如果 PLT_COMPILED_FILE_CHECK 环境变量被设置为 modify-seconds 或 exists ,那么当 Racket 启动时,环境变量的值将配置该参数。

Added in version 6.6.0.3 of package base.

procedure

(read-eval-print-loop) → any

使用当前的输入、输出和错误端口启动一个新的 REPL 。 REPL 用 #%top-interaction 包装每个要评估的表达式, 它通常与 #%top-interaction 绑定, 并且它用默认的continuation prompt标签和提示处理程序 (见 call-with-continuation-prompt)包装每个评估的continuation提示。 REPL 还用默认标签的提示来包装读取和打印操作,其处理程序忽略中止参数并继续循环。 read-eval-print-loop 过程不返回,直到 eof 被读取,这时它返回 #<void> 。

read-eval-print-loop 过程可以通过 current-prompt-read, current-eval 和 current-print 参数进行配置。

parameter

(current-prompt-read) → (-> any)

(current-prompt-read proc) → void?
  proc : (-> any)
一个 parameter 决定了一个 prompt read handler, 这是一个不需要参数的过程,显示一个提示字符串,并返回一个顶层表格来评估。 提示阅读处理程序由 read-eval-print-loop 调用,在打印提示后, 处理程序通常应该调用 read interaction handler (由 current-read-interaction 参数决定),其端口由 interaction port handler 产生(由 current-get-interaction-input-port 参数决定)。

默认的提示读取处理程序打印 >  并返回结果为

(let ([in ((current-get-interaction-input-port))])
  ((current-read-interaction) (object-name in) in))

如果输入和输出端口都是终端(在 terminal-port? 的意义上), 并且如果输出端口似乎在计算行数(因为 port-next-location 返回非 #f 的行和列),那么在返回读取结果之前,输出端口的行被递增, 其列被通过 set-port-next-location! 重置为 0 。

parameter

(current-get-interaction-input-port) → (-> input-port?)

(current-get-interaction-input-port proc) → void?
  proc : (-> input-port?)
一个决定 interaction port handler 的 parameter , 它返回一个用于 read-eval-print-loop 输入的端口。

默认的交互端口处理程序返回当前的输入端口。 此外,如果该端口是初始的当前输入端口,则初始的当前输出和错误端口会被刷新。

parameter

(current-compile)

 → (any/c boolean? . -> . compiled-expression?)
(current-compile proc) → void?
  proc : (any/c boolean? . -> . compiled-expression?)
一个决定当前 compilation handler 的 parameter 。 compilation handler 是一个过程,它接收一个顶层表格并返回一个编译过的表格; 关于编译的更多信息,请参见 编译 。

compilation handler 被 compile 调用, 并间接地被默认的 evaluation handler 和默认的 load handler 调用。

处理程序的第二个参数是 #t ,如果编译后的表单只用于即时评估,或者是 #f , 如果编译后的表单可以被保存起来供以后使用;默认的编译处理程序是针对即时评估的特殊情况而优化的。

当一个编译过的表格被写入一个输出端口时,写入的表格以 #~ 开头。 更多信息请参见 打印编译后的代码 。

出于内部测试的目的,当 PLT_VALIDATE_COMPILE 环境变量被设置时, 默认的编译处理程序会在自己的编译结果上立即运行字节码验证器(而不是只在加载编译的字节码时依赖验证)。

在 protect-out 的意义上, current-compile 绑定是作为 protected 的方式提供的。

Changed in version 8.2.0.4 of package base: 将绑定改为 protected 。

parameter

(compile-enforce-module-constants) → boolean?

(compile-enforce-module-constants on?) → void?
  on? : any/c
一个决定模块声明如何被编译的 parameter 。

当常量被强制执行时,当模块的宏扩展体不包含对模块内定义的特定变量的 set! 赋值时, 那么在定义被评估时,该变量被标记为常量。 之后,该变量的值不能通过 module->namespace 进行赋值或取消定义,也不能通过重新声明模块来定义它。

强制执行常量允许编译器内联一些变量值,并允许本地代码即时编译器生成跳过某些运行时检查的代码。

parameter

(compile-allow-set!-undefined) → boolean?

(compile-allow-set!-undefined allow?) → void?
  allow? : any/c
一个 parameter ,决定了 set! 表达式在突变全局变量时的编译方式。 如果这个参数的值是一个真值,那么全局变量的 set! 表达式在编译时就会使全局变量被设置,即使它以前没有被定义过。 否则,如果在执行 set! 时全局变量没有被定义, 那么全局变量的 set! 表达式将被编译为引发 exn:fail:contract:variable 。 注意,这个参数是在 compiled 表达式时使用,而不是在 evaluated 表达式时使用。

parameter

(current-compile-target-machine)

 → (or/c #f (and/c symbol? compile-target-machine?))
(current-compile-target-machine target) → void?
  target : (or/c #f (and/c symbol? compile-target-machine?))
一个决定新编译的表达式的平台和/或虚拟机目标的 parameter 。

如果目标是 #f, 编译后的表达式会以独立于机器的格式写入(通常是 ".zo" 文件)。 与机器无关的编译代码适用于任何平台和任何 Racket 虚拟机。 当独立于机器的编译表达式被读回时,它需要为当前的平台和虚拟机进行进一步的编译, 这可能会比读一个为平台和虚拟机完全编译的格式要慢得多。

默认是 #f 以外的内容,除非通过独立的 Racket (或 GRacket)的 -M/--compile-any 命令行标志或通过 PLT_COMPILE_ANY 环境变量(设置为任何值)启用机器独立模式。

Added in version 7.1.0.6 of package base.
