10.6 中断

break 是一个异步异常,通常是通过用户控制的外部源,或通过 break-thread 来触发的。 例如,用户可以在终端输入 Ctl-C 来触发中断。 在某些平台上,Racket 进程可能会收到 SIGINT, SIGHUP 或 SIGTERM; 后两者对应于挂起和终止中断,分别由 exn:break:hang-up 和 exn:break:terminate 反映。 多个中断可以合并为一个异常,不同类型的多个中断可以合并为一个 "最强 "的中断, 其中终止中断比挂起中断强,而挂起中断比中断强。

只有当中断被启用时,中断异常才能在线程中发生。 当检测到并启用断点时, exn:break (或 exn:break:hang-up 或 exn:break:terminate)异常会在之后的某个时间在线程中被引发; 如果在调用 break-thread 时断点被禁用,那么断点会被暂停,直到断点再次为线程被启用。 当一个线程有一个暂停的中断时,其他的中断会被忽略。

通过类似于参数化的 break-enabled 过程和 parameterize-break 形式启用断点, 这与 parameterize 相类似。 break-enabled 过程并不代表与 parameterize 一起使用的参数, 因为改变线程的 break-enabled 状态需要对 break 进行明确的检查, 而这种检查与 parameterize 表达式主体的尾部评估不兼容。

某些程序,例如 semaphore-wait/enable-break,在执行一个阻塞动作时暂时启用中断。 如果一个线程启用了中断,并且如果该线程触发了中断,但尚未以 exn:break 异常的形式传递, 那么在该线程的中断被禁用之前,该中断保证会被传递。 exn:break 异常的时间不以任何其他方式保证。

在调用 with-handlers 谓词或处理程序、异常处理程序、错误显示处理程序、错误转义处理程序、 错误值转换处理程序或 dynamic-wind 的 pre-thunk 或 post-thunk 之前,该调用被 parameterize-break 以禁止中断。 此外,在与异常有关的处理程序之间的转换过程中,在 dynamic-wind 的 pre-thunk 和 post-thunk 之间的转换过程中,以及在继续跳转的其他转换过程中,中断被禁用。 例如,如果在调用一个续集时禁用断点,并且如果在目标续集中也禁用断点, 那么从调用开始到目标续集执行为止,断点将一直被禁用, 除非相关的 dynamic-wind pre-thunk 或 post-thunk 明确启用断点。

如果一个被阻塞在嵌套线程上的线程触发了断裂(见 call-in-nested-thread), 并且如果断裂在被阻塞的线程中被启用,那么断裂会被隐式地处理,将其转移到嵌套线程中。

当断点被启用时,它们可以在执行中的任何一点发生,这使得某些执行任务变得微妙。 例如,假设在执行以下代码时启用了断点,

(with-handlers ([exn:break? (lambda (x) (void))])
  (semaphore-wait s))

那么就 不 是说 #<void> 的结果意味着信号被递减或者收到了断裂,完全是这种情况。 这两种情况 都 有可能发生:中断可能发生在 semaphore 被成功递减之后, 但在 semaphore-wait 返回 #<void> 结果之前。 一个中断异常永远不会损坏 semaphore 或任何其他的内置结构, 但是许多内置程序(包括 semaphore-wait) 包含的内部子表达式可以被一个中断所打断。

一般来说,只使用 semaphore-wait 是不可能实现的,它可以保证信号被递减或产生一个异常,但不能同时发生。 因此,Racket提供了 semaphore-wait/enable-break (参见 Semaphores), 它确实允许实现这样一种排他性的保证:

(parameterize-break #f
  (with-handlers ([exn:break? (lambda (x) (void))])
    (semaphore-wait/enable-break s)))

在上面的表达式中,中断可以在任何时候发生,直到中断被禁用,在这种情况下,中断异常被传播到包围的异常处理程序。 否则,中断只能在 semaphore-wait/enable-break 中发生, 这保证了如果中断异常被引发,semaphore 不会被递减。

为了允许在阻塞端口操作上有类似的实现模式,Racket 提供了 read-bytes-avail!/enable-break, write-bytes-avail/enable-break 和其他程序。

procedure

(break-enabled) → boolean?

(break-enabled on?) → void?
  on? : any/c
获取或设置当前线程的中断启用状态。 如果没有提供 on? ,如果当前启用了断点,结果是 #t ,否则是 #f 。 如果 on? 提供的是 #f, 则断点被禁用,如果 on? 是一个真值,则断点被启用。

syntax

(parameterize-break boolean-expr body ...+)

评估 boolean-expr ,以确定在依次评估 body 时,最初是否启用 break。 parameterize-break 表达式的结果是最后一个 expr 的结果。

与 parameterize 一样,分配了一个新的 thread cell 被分配来保存续集的中断启用状态, 续集内对 break-enabled 的调用访问或修改新单元。 与参数不同,通过 break-enabled 对 break 设置的突变不会被新线程继承(即线程单元不会被 preserved) 。

procedure

(current-break-parameterization) → break-parameterization?

类似于 (current-parameterization) (见 参数); 它返回一个持有当前续集的break-enable状态的 break 参数化(实际上是一个线程单元)。

procedure

(call-with-break-parameterization break-param    
  thunk) → any
  break-param : break-parameterization?
  thunk : (-> any)
类似于 (call-with-parameterization parameterization thunk) (见 参数), 在一个中断启用状态为 break-param 的延续中调用 thunk 。 相对于 call-with-break-parameterization 的调用, thunk 不 会在尾部位置被调用。

procedure

(break-parameterization? v) → boolean?

  v : any/c
如果 v 是由 current-break-parameterization 产生的 break 参数化, 返回 #t ,否则返回 #f 。

Added in version 6.1.1.8 of package base.
