15.1.1 Manipulating Paths

procedure

(path? v) → boolean?

  v : any/c
如果 v 是当前平台的路径值(不是字符串,也不是不同平台的路径),返回 #t ,否则返回 #f 。

procedure

(path-string? v) → boolean?

  v : any/c
如果 v 是一个 path or string ,则返回 #t : 要么是当前平台的路径,要么是一个没有空字符的非空字符串。否则返回 #f 。

procedure

(path-for-some-system? v) → boolean?

  v : any/c
如果 v 是某个平台的路径值(不是字符串),则返回 #t ,否则返回 #f 。

procedure

(string->path str) → path?

  str : string?
在 Unix and Mac OS 上产生一个字节串编码为 (string->bytes/locale str (char->integer #\?)) 的路径, 在 Windows 上产生一个字节串编码为 (string->bytes/utf-8 str) 的路径。

请注意,当前的 locale 可能不会对每个字符串进行编码, 在这种情况下, string->path 可能会对不同的 str 产生相同的路径。 也请参见 string->path-element, 当一个字符串代表一个 path element 时, 应该用它来代替 string->path 。 关于字符串和字节串如何编码路径的信息,见 Unix 的路径表示法 和 Windows的路径表示法 。

也请参见 string->some-system-path, 关于字符串如何编码路径的信息, 请参见 Unix 的路径表示法 和 Windows的路径表示法 。

Changed in version 6.1.1.1 of package base: 改变了 Windows 的转换,始终使用 UTF-8。

procedure

(bytes->path bstr [type]) → path?

  bstr : bytes?
  type : (or/c 'unix 'windows) = (system-path-convention-type)
产生一个字节字符串编码为 bstr 的路径(针对某个平台),其中 bstr 不能包含一个 nul 字节。 可选的 type 指定了路径所使用的惯例。

对于从字面意义上转换相对 path elements ,可以使用 bytes->path-element, 它为单个元素应用合适的编码。

关于字节字符串如何编码路径的信息,见 Unix 的路径表示法 和 Windows的路径表示法 。

procedure

(path->string path) → string?

  path : path?
通过在 Unix and Mac OS 上使用当前的 locale 对 path 的字节串编码进行解码, 以及在 Windows 上使用 UTF-8,产生一个代表 path 的字符串。 在前一种情况下,在编码失败的情况下, ? 被用于结果字符串,如果编码结果是空字符串,那么结果就是 "?" 。

结果字符串适合于向用户显示、字符串排序比较等, 但不适合于通过 string->path 重新创建路径(可能已修改),因为解码和重新编码路径的字节字符串可能会丢失信息。

此外,对于基于单个 path elements (如无路径的文件名)的显示和排序, 请使用 path-element->string,以避免使用特殊编码来表示一些相对路径。 关于 Windows 路径转换的具体信息,请参见 Windows 路径 。

另请参见 some-system-path->string 。

Changed in version 6.1.1.1 of package base: 改变了 Windows 的转换,始终使用 UTF-8。

procedure

(path->bytes path) → bytes?

  path : path-for-some-system?
产生了 path 的字节串表示。 在这种转换中没有任何信息丢失,所以 (bytes->path (path->bytes path) (path-convention-type path)) 总是产生一个与 path equal? 的路径。 path 参数可以是任何平台的路径。

字节值的转换对于 marshaling 和 unmarshaling 路径很有用,但是操作字节形式的路径一般是错误的。 特别是,对于 Windows 的路径来说,字节串可能是以 \\?\REL 编码开始的。 使用 split-path 和 path-element->bytes 来操作单个 path elements, 而不是 path->bytes 。

关于字节字符串如何编码路径的信息,见 Unix 的路径表示法 和 Windows的路径表示法 。

procedure

(string->path-element str 
  [false-on-non-element?]) 
 → (or/c (and/c path? path-element?) #f)
  str : string?
  false-on-non-element? : any/c = #f
与 string->path 类似,只是 str 对应于路径中的单个相对元素,并根据需要对其进行编码以转换为路径。 关于路径转换的更多信息,请参见 Unix and Mac OS 路径 和 Windows 路径 。

如果 str 不对应于任何 path element(例如,它是一个绝对路径,或者它可以被分割), 或者如果它对应于 Unix and Mac OS 上的上层目录或同层目录指标,那么要么返回 #f , 要么引发 exn:fail:contract exception is raised 。 只有当 false-on-non-element? 为真时才会返回 #f 。

和 path->string 一样,在向路径进行特定的本地化转换时,str 中的信息可能会丢失。

Changed in version 8.1.0.6 of package base: 增加了 false-on-non-element? 参数。

procedure

(bytes->path-element bstr 
  [type 
  false-on-non-element?]) 
 → (or/c path-element? #f)
  bstr : bytes?
  type : (or/c 'unix 'windows) = (system-path-convention-type)
  false-on-non-element? : any/c = #f
和 bytes->path 一样,只是 bstr 对应于路径中的一个相对元素。 在转换、对 bstr 的限制以及对 false-on-non-element? 的处理方面, bytes->path-element 与 string->path-element 类似。

当需要对 path elements 进行 ASCII 级操作时, bytes->path-element 程序通常是基于另一路径重建路径的最佳选择 (其中另一路径是用 split-path 和 path-element->bytes 解构的)。

Changed in version 8.1.0.6 of package base: 增加了 false-on-non-element? 参数。

procedure

(path-element->string path) → string?

  path : path-element?
和 path->string 一样,只是尾部的路径分隔符被移除(和 split-path)。 在Windows中,任何 \\?\REL 编码的前缀也会被删除;更多信息见 Windows 路径 。

path 参数必须是 split-path 应用于 path 的第一个结果是 'relative , 第二个结果是 path,否则会引发 exn:fail:contract exception is raised 。

path-element->string 过程通常是向用户展示无路径文件或目录名称的最佳选择。

procedure

(path-element->bytes path) → bytes?

  path : path-element?
和 path->bytes 一样,除了任何编码前缀被移除,等等,和 path-element->string 一样。

对于任何合理的地区, path 的打印形式中连续的 ASCII 字符被映射为连续的字节值, 与每个字符的编码点值相匹配,领先或落后的 ASCII 字符被分别映射为领先或落后的字节。 path 参数可以是任何平台的路径。

path-element->bytes 过程通常是正确的选择(与 split-path 结合), 用于提取路径的内容,在 ASCII 级别上对其进行操作 (然后用 bytes->path-element 和 build-path 重新组合结果)。

procedure

(path<? a-path b-path ...) → boolean?

  a-path : path?
  b-path : path?
如果参数被排序,则返回 #t ,其中每对路径的比较与使用 path->bytes 和 bytes<?

Changed in version 7.0.0.13 of package base: Allow one argument, in addition to allowing two or more.

procedure

(path-convention-type path) → (or/c 'unix 'windows)

  path : path-for-some-system?
接受一个路径值(不是字符串)并返回其约定类型。

procedure

(system-path-convention-type) → (or/c 'unix 'windows)

返回当前平台的路径约定类型: 'unix 用于 Unix and Mac OS, 'windows 用于 Windows 。

procedure

(build-path base sub ...) → path-for-some-system?

  base : (or/c path-string? path-for-some-system? 'up 'same)
  sub : 
(or/c (and/c (or/c path-string? path-for-some-system?)
             (not/c complete-path?))
      (or/c 'up 'same))
给出一个基本路径和任意数量的子路径扩展,创建一个路径。 如果 base 路径是一个绝对路径,则结果是一个绝对路径,否则结果是一个相对路径。

base 和每个 sub 必须是相对路径、符号 'up (表示相对父目录)或符号 'same (表示相对当前目录)。 对于Windows路径,如果 base 是一个驱动器规格(带或不带尾部斜线), 第一个 sub 可以是一个绝对(无驱动)路径。 对于所有平台,最后一个 sub 可以是一个文件名。

base 和 sub 参数可以是任何平台的路径。 结果路径的平台是由 base 和 sub 参数推断出来的,其中字符串参数意味着当前平台的路径。 如果不同的参数用于不同的平台,就会引发 exn:fail:contract exception is raised 。 如果没有参数暗示平台(即所有参数都是 'up 或 'same),则生成的路径是当前平台的。

每个 sub 和 base 点可以选择以目录分隔符结束。 如果最后一个 sub 以分隔符结尾,它将被包含在生成的路径中。

如果 base 或 sub 是一个非法的路径字符串(因为它是空的或者包含一个nul字符), 就会引发 exn:fail:contract exception is raised 。

build-path 过程建立了一个路径,而 without 检查路径的有效性或访问文件系统。

关于构建路径的更多信息,请参见 Unix and Mac OS 路径 和 Windows 路径 。

下面的例子假设 Unix 的当前目录是 "/home/joeuser" ,Windows的例子是 "C:\Joe’s Files" 。

(define p1 (build-path (current-directory) "src" "racket"))
 ; Unix: p1 is "/home/joeuser/src/racket"
 ; Windows: p1 is "C:\\Joe's Files\\src\\racket"
(define p2 (build-path 'up 'up "docs" "Racket"))
 ; Unix: p2 is "../../docs/Racket"
 ; Windows: p2 is "..\\..\\docs\\Racket"
(build-path p2 p1)
 ; Unix and Windows: raises exn:fail:contract; p1 is absolute
(build-path p1 p2)
 ; Unix: is "/home/joeuser/src/racket/../../docs/Racket"
 ; Windows: is "C:\\Joe's Files\\src\\racket\\..\\..\\docs\\Racket"

procedure

(build-path/convention-type type    
  base    
  sub ...) → path-for-some-system?
  type : (or/c 'unix 'windows)
  base : (or/c path-string? path-for-some-system? 'up 'same)
  sub : 
(or/c (and/c (or/c path-string? path-for-some-system?)
             (not/c complete-path?))
      (or/c 'up 'same))
和 build-path 一样,只是明确指定了一个路径约定类型。

请注意,就像 build-path 一样,无论是 base 还是 sub 的任何字符串参数, 在与其他参数结合之前都会被隐含地转换为当前平台的路径。 由于这个原因,你不能使用这个函数从当前平台以外的字符串中建立路径; 在这种尝试中, type 与推断的字符串的约定类型不匹配,并且会引发 exn:fail:contract exception is raised 。 (要为外国平台创建路径,请参见 bytes->path 。)

build-path/convention-type 相对于 build-path 的作用仅限于子路径包含 'same 或 'up 元素的情况。

procedure

(absolute-path? path) → boolean?

  path : (or/c path? string? path-for-some-system?)
如果 path 是一个绝对路径,返回 #t ,否则返回 #f 。 path 参数可以是任何平台的路径。 如果 path 不是一个合法的路径字符串(例如,它包含一个 nul 字符),则返回 #f 。 这个过程不访问文件系统。

procedure

(relative-path? path) → boolean?

  path : (or/c path? string? path-for-some-system?)
如果 path 是一个相对路径,返回 #t ,否则返回 #f 。 path 参数可以是任何平台的路径。 如果 path 不是一个合法的路径字符串(例如,它包含一个 nul 字符),则返回 #f 。 这个过程不访问文件系统。

procedure

(complete-path? path) → boolean?

  path : (or/c path? string? path-for-some-system?)
如果 path 是一个 complete 确定的路径(不 是相对于一个目录或驱动器), 则返回 #t ,否则返回 #f 。 path 参数可以是任何平台的路径。 注意,对于 Windows 的路径,绝对路径可以省略驱动器的说明,在这种情况下,路径既不是相对的也不是完整的。 如果 path 不是一个合法的路径字符串(例如,它包含一个 nul 字符),就会返回 #f 。

这个过程不访问文件系统。

procedure

(path->complete-path path [base]) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
  base : (or/c path-string? path-for-some-system?)
   = (current-directory)
将 path 作为一个完整的路径返回。 如果 path 已经是一个完整的路径,它将作为结果返回。 否则, path 将被解析为完整路径的 base。 如果 base 不是一个完整的路径,会引发 exn:fail:contract exception is raised 。

path 和 base 参数可以是任何平台的路径;如果它们是不同平台的, 就会引发 exn:fail:contract exception is raised 。

这个过程不访问文件系统。

procedure

(path->directory-path path) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
如果 path 在语法上指的是一个目录,并且以分隔符结尾,则返回 path , 否则它返回 path 的扩展版本,指定一个目录并以分隔符结尾。 例如,在 Unix and Mac OS 上,路径 "x/y/" 在语法上是指一个目录并以分隔符结尾,但 "x/y" 将被扩展为 "x/y/" ， "x/.." 将被扩展为 "x/../" 。 path 参数可以是任何平台的路径,其结果将是同一平台的。

这个过程不访问文件系统。

procedure

(resolve-path path) → path?

  path : path-string?
清理 path 并返回一个与 path 引用相同文件或目录的路径。 如果 path 是另一个路径的软链接,那么返回引用的路径(这可能是相对于拥有 path 的目录而言的相对路径), 否则返回 path (经过清理)。

在 Windows 上,链接的路径应该在语法上进行简化,这样,一个向上的目录指示器会删除前面的路径元素, 而不考虑前面的元素本身是否指向一个链接。 对于相对路径的链接,应该对路径进行特别解析;更多信息请参见 Windows 路径 。

Changed in version 6.0.1.12 of package base: 增加了对 Windows 上链接的支持。

procedure

(cleanse-path path) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
清理 path (如本章开头所述),不需要咨询文件系统。

Example:
> (let ([p (string->some-system-path "tiny//dancer" 'unix)])
    (cleanse-path p))

#<path:tiny/dancer>

procedure

(expand-user-path path) → path?

  path : path-string?
Cleanse path 此外,在 Unix and Mac OS 上,前面的 ~ 被视为用户的主目录并被扩展; 用户名跟在 ~ 后面(在一个 / 或路径的末端之前), 其中 ~ 本身表示当前用户的主目录。

procedure

(simplify-path path [use-filesystem?]) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
  use-filesystem? : boolean? = #t
消除多余的路径分隔符(除了单个尾部分隔符)、上目录 .. 和 path 中的同目录 . 指示符, 并将 Windows 路径中的 / 分隔符改为 \ 分隔符, 从而使结果访问与 path 相同的文件或目录(如果它存在)。

一般来说,路径名被尽可能地规范化–如果 use-filesystem? 为 #f, 则不需要咨询文件系统,并且(在 Windows 下)不需要改变路径中字母的大小写。 如果 path 在语法上指的是一个目录,那么结果将以一个目录分隔符结束。

当 path 被简化,而不仅仅是将斜线转换为反斜线,并且 use-filesystem? 为 true (默认)时, 会返回一个完整的路径。 如果 path 是相对的,它将被解析为与当前目录有关。 在 Unix and Mac OS 上,在考虑到软链接的情况下,上层目录的指示符被移除(这样得到的路径指的是与之前相同的目录); 在 Windows 上,上层目录的指示符是通过删除前面的 path element 来移除。

当 use-filesystem? 是 #f 时,通过删除前面的 path element 来删除上层目录指示符, 结果可能是一个相对路径,上层目录指示符保留在路径的开头;当上层目录指示符指向根目录的父目录时,将被放弃。 同样,如果消除了上层目录指示器,只留下同层目录指示器, 那么结果可以与 (build-path 'same) 相同(但有尾部分隔符)。

use-filesystem? 为 #f 时, path 参数可以是任何平台的路径,而结果是同一平台的路径。

当 use-filesystem? 为 true 时,文件系统可能被访问,但源路径或简化路径可能是一个不存在的路径。 如果 path 由于链接的循环而不能被简化, 就会引发 exn:fail:filesystem exception is raised 但是如果循环没有抑制简化,那么成功简化的路径仍然可能涉及链接的循环)。

关于简化路径的更多信息,见 Unix and Mac OS 路径 和 Windows 路径 。

Example:
> (let ([p (string->some-system-path "tiny//in/my/head/../../../dancer" 'unix)])
    (simplify-path p #f))

#<path:tiny/dancer>

procedure

(normal-case-path path) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
返回带有 "规范化" 大小写字符的 path 。 对于 Unix and Mac OS 的路径,这个程序总是返回输入的路径,因为这些平台的文件系统可能对大小写敏感。 对于 Windows 的路径,如果 path 不是以 \\?\ 开头,则根据当前的地区设置,返回的字符串只使用小写字母。 此外,对于 Windows 路径,如果路径不是以 \\?\ 开头,所有的 / 都被转换为 \ , 并且删除尾部的空格和 . 。

path 参数可以是任何平台的路径,但要注意,在当前平台上对路径的本地敏感解码和转换可能与路径的平台不同。

这个过程不访问文件系统。

procedure

(split-path path) → 
(or/c path-for-some-system? 'relative #f)
(or/c path-for-some-system? 'up 'same)
boolean?
  path : (or/c path-string? path-for-some-system?)
将 path 解构为一个较小的路径和一个直接的目录或文件名。有三个值被返回:

  • base 是

    • path,

    • 如果 path 是一个直接的相对目录或文件名,则为 'relative , 或者

    • #f ,如果 path 是一个根目录。

  • name 是

    • 目录名的 path,

    • filename,

    • 如果 path 的最后部分指定了前一个路径的父目录(例如,Unix 上的 .. ), 则为 'up ,或者

    • 如果 path 的最后部分指定了与前面的路径相同的目录(例如,Unix 上的 . ), 则为 'same 。

  • must-be-dir? 是 #t ,如果 path 明确指定了一个目录(例如,有一个尾部分隔符), 否则就是 #f 。 注意, must-be-dir? 并不指定 name 是否真的是一个目录, 而是指定 path 是否在语法上指定了一个目录。

与 path 相比,多余的分隔符(如果有的话)在结果 base 和 name 中被删除。 如果 base 是 #f ,那么 name 就不能是 'up 或 'same 。 path 参数可以是任何平台的路径,以及同一平台的结果路径。

这个过程不访问文件系统。

关于分割路径的更多信息,请参见 Unix and Mac OS 路径 和 Windows 路径 。

procedure

(explode-path path)

 → (listof (or/c path-for-some-system? 'up 'same))
  path : (or/c path-string? path-for-some-system?)
返回构成 path 的 path elements 的列表。 如果 path 是按照 simple-form-path 的含义简化的, 那么结果总是一个路径的列表,列表的第一个元素是根。

explode-path 函数计算其结果的时间与 path 的长度成正比 (与使用 split-path 的循环不同,它必须分配中间路径)。

procedure

(path-replace-extension path ext) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
  ext : (or/c string? bytes?)
返回一个与 path 相同的路径,只是路径中最后一个元素的扩展名(包括扩展名分隔符)被改为 ext 。 如果 path 的最后一个元素没有扩展名,那么 ext 将被添加到路径中。

扩展名被定义为不在路径元素开始的 . ,后面是任何数量的非 . 字符/字节, 只要 path element 不是目录指示器,如 ".." 。

path 参数可以是任何平台的路径,其结果也是同一平台的。 如果 path 代表一个根,会引发 exn:fail:contract exception is raised 。 给定的 ext 通常以 . 开头,但不要求以扩展分隔符开头。

Examples:
> (path-replace-extension "x/y.ss" #".rkt")

#<path:x/y.rkt>

> (path-replace-extension "x/y.ss" #"")

#<path:x/y>

> (path-replace-extension "x/y" #".rkt")

#<path:x/y.rkt>

> (path-replace-extension "x/y.tar.gz" #".rkt")

#<path:x/y.tar.rkt>

> (path-replace-extension "x/.racketrc" #".rkt")

#<path:x/.racketrc.rkt>

Added in version 6.5.0.3 of package base.

procedure

(path-add-extension path ext [sep]) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
  ext : (or/c string? bytes?)
  sep : (or/c string? bytes?) = #"_"

#<path:x/.racketrc.rkt>

procedure

(reroot-path path root-path) → path-for-some-system?

  path : (or/c path-string? path-for-some-system?)
  root-path : (or/c path-string? path-for-some-system?)
在 path 的完整形式的基础上,产生一个扩展了 root-path 的路径。

如果 path 还没有 complete ,是否通过 path->complete-path 完成, 在这种情况下, path 必须是当前平台的路径。 path 参数也会通过 normal-case-path 进行 cleanse 和大小写规范化。 然后将路径追加到 root-path 中;在 Windows 路径中,根字母驱动器成为字母路径元素, 而根 UNC 路径则以 "UNC" 作为路径元素的前缀,机器和卷名成为路径元素。

Examples:
> (reroot-path (bytes->path #"/home/caprica/baltar" 'unix)
               (bytes->path #"/earth" 'unix))

#<path:/earth/home/caprica/baltar>

> (reroot-path (bytes->path #"c:\\usr\\adama" 'windows)
               (bytes->path #"\\\\earth\\africa\\" 'windows))

#<windows-path:\\earth\africa\c\usr\adama>

> (reroot-path (bytes->path #"\\\\galactica\\cac\\adama" 'windows)
               (bytes->path #"s:\\earth\\africa\\" 'windows))

#<windows-path:s:\earth\africa\UNC\galactica\cac\adama>
