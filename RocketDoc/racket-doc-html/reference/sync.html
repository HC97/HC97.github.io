11.2.1 事件

synchronizable event (简称 event) 与 sync 过程一起工作,协调线程之间的同步。 某些类型的对象可以作为事件,包括端口和线程。其他类型的对象只是为了作为事件使用而存在。 Racket的事件系统是基于 Concurrent ML [Reppy99] 。

在任何时候,一个事件要么是 ready for synchronization ,要么不是; 根据事件的种类和它被其他线程使用的方式,一个事件可以在任何时候从未准备好切换到准备好(或返回)。 如果一个线程在准备好的时候对一个事件进行同步,那么该事件就会产生一个特定的 synchronization result.

同步一个事件可能会影响该事件的状态。 例如,当同步一个信号灯时,那么这个信号灯的内部计数会被递减,就像 semaphore-wait 一样。 然而,对于大多数类型的事件(如端口),同步化并不修改事件的状态。

作为 synchronizable events 的 Racket 值包括 asynchronous channels, channels, custodian boxes, log receivers, place channels, ports, semaphores, subprocesses, TCP listeners, threads 和 will executors。 库可以定义新的可同步事件,特别是通过 prop:evt 。

procedure

(evt? v) → boolean?

  v : any/c
如果 v 是一个 synchronizable event,返回 #t ,否则返回 #f 。

Examples:
> (evt? never-evt)

#t

> (evt? (make-channel))

#t

> (evt? 5)

#f

procedure

(sync evt ...) → any

  evt : evt?
只要没有一个 synchronizable events evt 准备好,就会被阻塞,如上文定义。

当至少有一个 evt 准备好了,它的 synchronization result (通常是 evt 本身) 被返回。 如果多个 evt 都准备好了,那么将伪随机地选择其中一个 evt 作为结果; current-evt-pseudo-random-generator 参数设置了控制这一选择的随机数发生器。

Examples:
> (define ch (make-channel))
> (thread (λ () (displayln (sync ch))))

#<thread>

> (channel-put ch 'hellooooo)

hellooooo

Changed in version 6.1.0.3 of package base: 允许0个参数,而不是1个或更多。

procedure

(sync/timeout timeout evt ...) → any

  timeout : (or/c #f (and/c real? (not/c negative?)) (-> any))
  evt : evt?
如果 timeout 是 #f ,就像 sync 。 如果 timeout 是一个实数,那么如果 timeout 秒没有成功的同步,那么结果就是 #f。 如果 timeout 是一个过程,那么如果轮询 evt 时没有发现准备好的事件,它将在尾部位置被调用。

timeout 的零值相当于 (lambda () #f) 。 在任何一种情况下,在返回 #f 或调用 timeout 之前,每个 evt 都至少被检查一次。

另一种超时机制见 alarm-evt 。

Examples:
; times out before waking up
> (sync/timeout
   0.5
   (thread (λ () (sleep 1) (displayln "woke up!"))))

#f

> (sync/timeout
   (λ () (displayln "no ready events"))
   never-evt)

no ready events

Changed in version 6.1.0.3 of package base: 允许1个参数而不是2个或更多。

procedure

(sync/enable-break evt ...) → any

  evt : evt?
像 sync 一样,但是在等待 evt 的时候启用 breaking (见 中断) 。 如果在调用 sync/enable-break 时禁用 break,那么要么所有的 evt 保持未选择状态, 要么引发 exn:break ,但不会同时发生。

procedure

(sync/timeout/enable-break timeout evt ...) → any

  timeout : (or/c #f (and/c real? (not/c negative?)) (-> any))
  evt : evt?
与 sync/enable-break 类似,但与 sync/timeout 一样有一个超时。

procedure

(choice-evt evt ...) → evt?

  evt : evt?
创建并返回一个结合了 evt 的单一事件。 将结果提供给 sync 与将每个 evt 提供给同一个调用是一样的。

也就是说,当提供给 choice-evt 的一个或多个 evt ready for synchronization 时, choice-evt 返回的事件就 ready for synchronization 。 如果选择事件被选中,它的一个准备好的 evt 会被伪随机地选择, 而 synchronization result 是被选择的 evt 的 synchronization result 。

Examples:
> (define ch1 (make-channel))
> (define ch2 (make-channel))
> (define either-channel (choice-evt ch1 ch2))
> (thread (λ () (displayln (sync either-channel))))

#<thread>

> (channel-put
   (if (> (random) 0.5) ch1 ch2)
   'tuturuu)

tuturuu

procedure

(wrap-evt evt wrap) → evt?

  evt : evt?
  wrap : (any/c ... . -> . any)
创建一个事件,当 evt 是 ready for synchronization 时, 该事件是 ready for synchronization , 但其 synchronization result 是由对 evt 的 synchronization result 应用 wrap 决定的。 wrap 所接受的参数数量必须与 evt 的同步结果的数值数量相匹配。

对 wrap 的调用是 parameterize-break ,以便最初禁用断裂。

Examples:
> (define ch (make-channel))
> (define evt (wrap-evt ch (λ (v) (format "you've got mail: ~a" v))))
> (thread (λ () (displayln (sync evt))))

#<thread>

> (channel-put ch "Dear Alice ...")

you've got mail: Dear Alice ...

procedure

(handle-evt evt handle) → handle-evt?

  evt : evt?
  handle : (any/c ... . -> . any)
和 wrap-evt 一样,只是当 handle 是在相对于同步请求的 tail position 被调用的, 并且没有明确地禁用中断, 当它没有被 wrap-evt, chaperone-evt 或另一个 handle-evt 包裹时。

Examples:
> (define msg-ch (make-channel))
> (define exit-ch (make-channel))
> (thread
   (λ ()
     (let loop ([val 0])
       (printf "val = ~a~n" val)
       (sync (handle-evt
              msg-ch
              (λ (val) (loop val)))
             (handle-evt
              exit-ch
              (λ (val) (displayln val)))))))

val = 0

#<thread>

> (channel-put msg-ch 5)

val = 5

> (channel-put msg-ch 7)

val = 7

> (channel-put exit-ch 'done)

done

procedure

(guard-evt maker) → evt?

  maker : (-> (or/c evt? any/c))
创建一个表现为事件的值,但它实际上是一个事件制造者。

一个由 guard-evt 返回的事件 guard , 当 guard 与 sync 一起使用时(或者当它是与 sync 一起使用的选择事件的一部分时,等等), 会产生一个事件,其中产生的事件是调用 maker 的结果。 对于一个给定的对 sync 的调用, maker 过程最多可以被 sync 调用一次, 但是如果在考虑 guard 之前就选择了一个准备好的事件, 那么 maker 就可能不会被调用。

如果 maker 返回一个非事件, 那么 maker 的结果将被替换为一个 ready for synchronization 的事件, 其 synchronization result 是 guard 。

procedure

(nack-guard-evt maker) → evt?

  maker : (evt? . -> . (or/c evt? any/c))
与 guard-evt 类似,但当 maker 被调用时,它被赋予一个 NACK("否定的确认")事件。 在开始调用 maker 后,如果 maker 的事件最终没有被选为准备好的事件, 那么提供给 maker 的 NACK 事件就成为 ready for synchronization ,其值为 #<void> 。

当事件被放弃时,NACK 事件就变成了 ready for synchronization , 这时要么选择了其他事件,要么同步线程死亡,要么控制权从对 sync 的调用中逃脱(即使 nack-guard 的 maker 还没有返回一个值)。 如果 maker 返回的事件被选中,那么 NACK 事件永远不会成为 ready for synchronization 。

procedure

(poll-guard-evt maker) → evt?

  maker : (boolean? . -> . (or/c evt? any/c))
与 guard-evt 类似,但是当 maker 被调用时,它被提供了一个布尔值, 表明该事件是用于轮询, #t, 还是用于阻塞性同步,#f 。

如果 #t 被提供给 maker ,如果中断被禁用,如果轮询线程没有被终止, 如果轮询产生的事件产生了一个 synchronization result, 那么该事件肯定会被选择为其结果。

procedure

(replace-evt evt maker) → evt?

  evt : evt?
  maker : (any/c ... . -> . (or/c evt? any/c))
像 guard-evt 一样, 但是只有在 evt 成为 ready for synchronization 之后才会调用 maker 之后才会调用 maker , 并且 evt 的 synchronization result 会传递给 maker 。

对 evt 进行同步的尝试与对来自 replace-evt 的结果 guard 进行同步的尝试同时进行; 尽管有这种并发性,如果 maker 被调用,它将在正在对 guard 进行同步的线程中被调用。 同步可以同时为 evt 和另一个与 guard 同步的事件成功; 同步的单选保证只适用于 maker 的结果和与 guard 同步的其他事件。

如果 maker 返回一个非事件, 那么 maker 的结果将被替换为一个 ready for synchronization 的事件, 其 synchronization result 是 guard 。

Added in version 6.1.0.3 of package base.

value

always-evt : evt?

一个总是 ready for synchronization 的常量事件, 它的 synchronization result 是它自己。

Example:
> (sync always-evt)

#<always-evt>

value

never-evt : evt?

一个从未 ready for synchronization 的常量事件。

Example:
> (sync/timeout 0.1 never-evt)

#f

procedure

(system-idle-evt) → evt?

当系统在其他方面处于空闲状态时,返回一个 ready for synchronization 的事件: 如果结果事件被 never-evt 取代,系统中就没有线程可以运行了。 换句话说,所有的线程都必须在超时未过的事件上被暂停或被阻塞。system-idle 事件的 synchronization result 是 #<void> 。 system-idle-evt 程序的结果始终是同一个事件。

Examples:
> (define th (thread (λ () (let loop () (loop)))))
> (sync/timeout 0.1 (system-idle-evt))

#f

> (kill-thread th)
> (sync (system-idle-evt))

procedure

(alarm-evt msecs [monotonic?]) → evt?

  msecs : real?
  monotonic? : any/c = #f
返回一个 synchronizable event ,当 (milliseconds) 会返回一个小于 msecs 的值时, 它还没有 ready for synchronization ,而当 (milliseconds) 返回的值大于 msecs 时, 它是 ready for synchronization 。 当 monotonic? 为 #f 时, milliseconds 的值是 current-inexact-milliseconds , 否则就是 current-inexact-monotonic-milliseconds 。 The synchronization result of a alarm event is the alarm event itself.

Examples:
> (define alarm (alarm-evt (+ (current-inexact-milliseconds) 100)))
> (sync alarm)

#<alarm-evt>

Changed in version 8.3.0.9 of package base: 增加了 monotonic? 参数。

procedure

(handle-evt? evt) → boolean?

  evt : evt?
如果 evt 是由 handle-evt 创建的, 或者由 choice-evt 应用于另一个事件, 并且 handle-evt? 产生 #t ,则返回 #t 。 对于任何其他事件, handle-evt? 产生 #f 。

Examples:
> (handle-evt? never-evt)

#f

> (handle-evt? (handle-evt always-evt values))

#t

value

prop:evt : struct-type-property?

一个 structure type property ,用于识别其实例可以作为 synchronizable events 的结构类型。 该属性值可以是以下任何一种:

  • 一个事件 evt: 在这种情况下,使用结构作为事件等同于使用 evt 。

  • 一个由一个参数组成的过程 proc : 在这种情况下,该结构类似于由 guard-evt 生成的事件, 只是可能的 guard 过程 proc 接收结构作为参数,而不是没有参数; 另外, proc 的非事件结果被替换为已经 ready for synchronization 的事件, 其 synchronization result 就是该结构。

  • 一个精确的非负整数,介于 0 (包括)和结构类型中的非自动字段数量之间(不包括,不计算超类型字段): 这个整数标识了结构中的一个字段,并且该字段必须被指定为不可变的。 如果该字段包含一个对象或一个参数的事件生成过程,那么该事件或过程的使用方法同上。 否则,该结构作为一个事件,永远不会准备好。

对于与外国库一起工作,一个 prop:evt 值也可以是 unsafe-poller 的结果, 尽管这种可能性在 prop:evt 的安全契约中被省略。

具有 prop:input-port 或 prop:output-port 属性的结构类型的实例由于是端口, 所以也是 synchronizable events 。 如果结构类型有多个 prop:evt, prop:input-port 和 prop:output-port, 那么 prop:evt 值(如果有的话)在确定实例作为事件的行为时具有优先权, 而 prop:input-port 属性在同步时比 prop:output-port 优先。

Examples:
> (struct wt (base val)
    #:property prop:evt (struct-field-index base))
> (define sema (make-semaphore))
> (sync/timeout 0 (wt sema #f))

#f

> (semaphore-post sema)
> (sync/timeout 0 (wt sema #f))

#<semaphore>

> (semaphore-post sema)
> (sync/timeout 0 (wt (lambda (self) (wt-val self)) sema))

#<semaphore>

> (semaphore-post sema)
> (define my-wt (wt (lambda (self)
                      (wrap-evt
                       (wt-val self)
                       (lambda (x) self)))
                    sema))
> (sync/timeout 0 my-wt)

#<wt>

> (sync/timeout 0 my-wt)

#f
