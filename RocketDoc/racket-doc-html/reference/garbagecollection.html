16.4 垃圾收集

在 Racket 启动前设置 PLTDISABLEGC 环境变量(为任何值),以禁用 garbage collection 。 将 PLT_INCREMENTAL_GC 环境变量设置为以 1, y 或 Y 开头的值, 以在任何时候都请求增量模式,但在有周期性任务的程序中调用 (collect-garbage 'incremental) 通常是请求增量模式的更好机制。 将 PLT_INCREMENTAL_GC 环境变量设置为以 0, n 或 N 开头的值, 可以禁用增量模式请求。

每次垃圾收集都会在 'debug 级别记录一条消息(见 记录) ,主题为 'GC 。 在 Racket 的 CS 和 3m 实现中, "主要的" 收集也会在 'debug 级别以 'GC:major 为主题进行记录。 在 Racket 的 CS 和 3m 实现中, 消息的数据部分是一个 gc-info prefab 结构类型的实例, 有如下 10 个字段,但 Racket 的未来版本可能会使用有额外字段的 gc-info prefab 结构:

(struct gc-info (mode pre-amount pre-admin-amount code-amount
                      post-amount post-admin-amount
                      start-process-time end-process-time
                      start-time end-time)
  #:prefab)

  • mode 字段是一个符号 'major, 'minor 或 'incremental; 'major 表示检查所有内存的集合, 'minor 表示主要检查最近的分配的集合， 'incremental 表示对下一个主要集合执行额外的工作的小集合。

    Changed in version 6.3.0.7 of package base: 将第一个字段从布尔值 (#t 表示 'major, #f 表示 'minor) 改为模式符号。

  • pre-amount 字段报告了 garbage collection 开始时地方-本地的内存使用(即不计算子地方的内存使用),单位是字节。 通过 make-phantom-bytes 注册的额外字节也包括在内。

  • pre-admin-amount 是一个更大的数字,它包括垃圾收集器开销的内存使用, 例如映射了但目前没有使用的内存页上的空间。

  • code-amount 字段报告了生成的本地代码的额外内存使用情况 (在垃圾收集之前和之后是一样的,因为它是通过最终化释放的)。

  • post-amount 和 post-admin-amount 字段对应于 pre-amount 和 pre-admin-amount, 但在垃圾回收之后。 在典型的配置中, post-amount 和 pre-amount 之间的差异有助于 post-admin-amount, 因为回收的页面倾向于保留,期望它们会被再次需要(但如果多次收集后不需要这些页面,这些页面会被释放)。

  • start-process-time 和 end-process-time 字段报告了垃圾收集开始和结束时的处理器时间(从 current-process-milliseconds 的角度来看)。 这两个时间之间的差异是收集所消耗的处理器时间。

  • start-time 和 end-time 字段报告了垃圾收集开始和结束时的实际时间 (从 current-inexact-milliseconds 意义上讲)。 这两个时间之间的差异是垃圾收集所消耗的实际时间。

记录的消息文本的格式可能会有变化。 目前,在表示 place 和收集模式的前缀之后,文本的格式为

‹used›(‹admin›)[‹code›]; free ‹reclaimed›(‹adjust›) ‹elapsed› @ ‹timestamp›

‹used›

 

在垃圾收集之前使用的可收集内存

‹admin›

 

用来管理可收集内存的额外内存

‹code›

 

用于生成机器代码的额外内存

‹reclaimed›

 

通过垃圾收集回收的可收集内存

‹adjust›

 

管理内存的变化减去 ‹reclaimed› 后的负值

‹elapsed›

 

用于执行垃圾收集的处理器时间

‹timestamp›

 

从启动垃圾收集开始的处理器时间

Changed in version 6.3.0.7 of package base: 添加了 PLT_INCREMENTAL_GC 。
Changed in version 7.6.0.9: 为 'GC:major 主题添加了主要收集日志。

procedure

(collect-garbage [request]) → void?

  request : (or/c 'major 'minor 'incremental) = 'major
要求立即进行 garbage collection 或要求垃圾收集模式,取决于 request:

  • 'major — 强制进行 "major" 收集,它检查所有的内存。 一些实际上无法到达的数据可能仍然没有被收集,因为收集器不能证明它是无法到达的。

    这种模式的 collect-garbage 程序提供了对收集时间的一些控制,但是即使这个程序从未被调用, 垃圾显然也会被收集,除非通过设置 PLTDISABLEGC 来禁止垃圾收集。

  • 'minor — 请求一个 "minor" 收集,它主要是检查最近的分配。 如果不支持次要收集(例如,当 (system-type 'gc) 返回 'cgc) 时, 或者如果下一次收集必须是主要收集,则不执行收集。 更一般地说,由 (collect-garbage 'minor) 触发的次要收集不会比其他情况下的主要收集更早发生。

  • 'incremental — 不要求立即收集,但要求额外的努力来避免主要的收集, 即使每个次要的收集需要更多的工作来递增执行主要收集的工作。 这种增量模式的请求在下一次主要收集时失效。

    增量模式的意图是大大减少由于主要收集而导致的暂停时间,但增量模式可能意味着更长的次要收集时间和更高的内存使用。 目前,增量模式只对 CS 和 3m Racket 实现有意义,对其他 Racket 实现没有影响。

    如果 PLT_INCREMENTAL_GC 环境变量的值在启动时以 0, n 或 N 开头, 那么增量模式的请求会被忽略。

Changed in version 6.3 of package base: 添加了 request 参数。
Changed in version 6.3.0.2: 添加了 'incremental 模式。
Changed in version 7.7.0.4: 为Racket CS 增加了 'incremental 效果。

procedure

(current-memory-use [mode]) → exact-nonnegative-integer?

  mode : (or/c #f 'cumulative custodian?) = #f
返回关于内存使用的信息:

  • 如果 mode 是 #f (默认),结果是对任何托管人可达到的字节数的估计。

  • 如果 mode 是 'cumulative, 返回自启动以来分配的字节总数的估计值,包括后来被垃圾回收的字节。

  • 如果 mode 是一个监护人,则返回一个估计值,即从 mode 出发的可达数据所占用的内存字节数。 这个估计值是由最后一次垃圾回收计算出来的,如果没有发生,可以是 0 (或者在给定的监护人创建后没有发生)。 current-memory-use 函数本身并 不 执行收集; 在调用之前做一次收集通常会减少结果(如果还没有发生收集,则从 0 增加)。

    当 Racket 在编译时不支持内存核算时,对于任何一个单独的托管人来说, 估算结果与 mode 为 #f (即所有内存)时相同。 参见 custodian-memory-accounting-available? 。

参见 vector-set-performance-stats! 。

Changed in version 6.6.0.3 of package base: 增加了 'cumulative 模式。

procedure

(dump-memory-stats v ...) → any

  v : any/c
将内存使用情况的信息转储到低级错误端口或控制台。

v 参数的各种组合可以控制转储的信息。 可用的信息取决于你的 Racket 构建;检查特定构建的转储的结尾,看它是否提供了额外的信息;否则,所有 v 都被忽略了。
