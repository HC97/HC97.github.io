10.4 Continuations

延续 in Racket 指南 introduces continuations.

参见 子表达式的评价和续篇 和 提示、限定的延续和障碍物 ,了解关于连续的一般信息。 Racket 对提示和可组合连续的支持 [Flatt07] 与 Sitaram 的 % 和 fcontrol 操作符 [Sitaram93] 非常相似。

Racket 在以下情况下的评估周围安装了一个 continuation barrier , 防止完全延续性跳转到受屏障保护的评估上下文:

  • 应用异常处理程序、错误转义处理程序或错误显示处理程序(见 异常情况);

  • 应用一个宏转换器(见 句法变换器) ，评估一个编译时表达式, 或应用一个模块名称解析器(见 解决模块名称问题);

  • 应用自定义端口程序(见 Custom Ports),事件保护程序(见 事件), 或参数保护程序(见 参数);

  • 应用一个安全保护程序(见 Security Guards);

  • 应用一个意志过程(见 遗嘱和执行者); 或

  • 从独立的 Racket 命令行评估或加载代码(见 运行 Racket 或 GRacket)。

此外,Racket 的扩展可以在额外的上下文中安装障碍。 最后, call-with-continuation-barrier 在应用程序和当前的 continuation 之间应用一个 thunk barrier。

procedure

(call-with-continuation-prompt proc    
  [prompt-tag    
  handler]    
  arg ...) → any
  proc : procedure?
  prompt-tag : continuation-prompt-tag?
   = (default-continuation-prompt-tag)
  handler : (or/c procedure? #f) = #f
  arg : any/c
将 proc 应用于给定的 arg ,并将当前的 continuation 扩展为一个提示。 提示由 prompt-tag 标记, 它必须是 default-continuation-prompt-tag (默认)或 make-continuation-prompt-tag 的结果。 对 call-with-continuation-prompt 的调用返回 proc 的结果。

handler 参数指定了一个处理程序, 当安装的提示符是带有 prompt-tag 的 abort-current-continuation 调用的目标时,将在相对于 call-with-continuation-prompt 调用的尾部位置调用; abort-current-continuation 的其余参数被提供给处理程序。 如果 handler 是 #f , 默认处理程序接受一个 abort-thunk 参数并调用 (call-with-continuation-prompt abort-thunk prompt-tag #f); 也就是说,默认处理程序重新安装提示符并继续执行指定的thunk。

procedure

(abort-current-continuation prompt-tag    
  v ...) → any
  prompt-tag : any/c
  v : any/c
将当前延续重置为当前延续中由 prompt-tag 标记的最近的提示; 如果不存在这样的提示,将引发 exn:fail:contract:continuation exception is raised 。 v 被作为参数传递给目标提示符的处理过程。

提供给中止的 v 的协议是特定于 prompt-tag 的。 当 abort-current-continuation 与 (default-continuation-prompt-tag)一起使用时, 一般来说,应该提供一个适合用于默认提示处理程序的 thunk。 同样,当 call-with-continuation-prompt 与 (default-continuation-prompt-tag)一起使用时, 相关的处理程序一般应该接受一个 thunk 参数。

每个 thread 的延续都是从使用默认处理程序的 (default-continuation-prompt-tag) 的提示开始的, 该处理程序接受单个 thunk 来应用(带有完整的提示)。

procedure

(make-continuation-prompt-tag) → continuation-prompt-tag?

(make-continuation-prompt-tag name) → continuation-prompt-tag?
  name : symbol?
创建一个与任何其他值的结果(包括 make-continuation-prompt-tag 的先前或未来的结果) 不 equal? 的提示标签。 如果提供了可选的 name 参数,则指定用于打印的提示标签的名称或 object-name 。

Changed in version 7.9.0.13 of package base: name 参数给出了提示标签的名称。

procedure

(default-continuation-prompt-tag) → continuation-prompt-tag?

返回一个恒定的提示标签,每个线程的延续开始时都会为其安装一个提示;每个线程的初始提示的处理程序接受任意数量的值并返回。 default-continuation-prompt-tag 的结果是任何接受提示标签的过程的默认标签。

procedure

(call-with-current-continuation proc    
  [prompt-tag]) → any
  proc : (continuation? . -> . any)
  prompt-tag : continuation-prompt-tag?
   = (default-continuation-prompt-tag)
捕获当前的延续,直到由 prompt-tag 标记的最近的提示; 如果不存在这样的提示,就会引发 exn:fail:contract:continuation exception is raised 。 截断的续集只包括提示后安装的续集标记和 dynamic-wind 框架。

捕获的续集被传递给 proc, proc 在相对于 call-with-current-continuation 调用的尾部位置被调用。

如果 proc 的 continuation 参数被应用,那么它将删除当前 continuation 的部分, 直到由 prompt-tag 标记的最近的提示符 (不包括提示符;如果不存在这样的提示符,将引发 exn:fail:contract:continuation exception is raised), 或者直到当前和捕获的 continuation 共享的最近的 continuation 帧(如果有的话),以先到的为准。 在移除延续帧的同时, dynamic-wind post-thunk 也被执行。 最后,捕获的续集(未共享的部分)被附加到剩余的续集上,应用 dynamic-wind pre-thunk 。

提供给应用程序的参数成为恢复的续集的结果值。 特别是,如果提供了多个参数,那么该延续就会收到多个结果。

如果在应用时,用应用的延续替换当前的延续会引入 continuation barrier , 那么会引发 exn:fail:contract:continuation exception is raised 。

一个延续可以从捕获它的线程(见 线程) 中调用。

procedure

(call/cc proc [prompt-tag]) → any

  proc : (continuation? . -> . any)
  prompt-tag : continuation-prompt-tag?
   = (default-continuation-prompt-tag)
call/cc 绑定是 call-with-current-continuation 的一个别名。

procedure

(call-with-composable-continuation proc    
  [prompt-tag]) → any
  proc : (continuation? . -> . any)
  prompt-tag : continuation-prompt-tag?
   = (default-continuation-prompt-tag)
与 call-with-current-continuation 类似,但应用产生的 continuation 过程不会删除当前 continuation 的任何部分。 相反,应用总是用捕获的续集来扩展当前的续集(不安装除续集中捕获的提示外的任何提示)。

当调用 call-with-composable-continuation 时, 如果 continuation 中的障碍物出现在由 prompt-tag 标记的最近的提示之前, 就会引发 exn:fail:contract:continuation exception is raised (因为试图应用 continuation 总是会失败)。

procedure

(call-with-escape-continuation proc) → any

  proc : (continuation? . -> . any)
与 call-with-current-continuation 类似,但 proc 不在尾部位置调用, 而且提供给 proc 的继续过程只能在 call-with-escape-continuation 调用的动态范围内调用。

从 call-with-escape-continuation 中获得的续程实际上是一种提示。 提供 Escape 延续主要是为了向后兼容,因为它们早于 Racket 中的一般提示语。 在 Racket 的 BC 实现中, call-with-escape-continuation 的实现比 call-with-current-continuation 更有效, 所以 call-with-escape-continuation 有时可以取代 call-with-current-continuation , 以提高 Racket 旧版本的性能。

procedure

(call/ec proc) → any

  proc : (continuation? . -> . any)
call/ec 绑定是 call-with-escape-continuation 的一个别名。

procedure

(call-in-continuation k proc) → any

  k : continuation?
  proc : (-> any)
类似于应用续集 k,但不是向续集传递值, 而是以 k 作为调用的续集来调用 proc (所以 proc 的结果会返回给续集)。 如果 k 是一个可组合的续集,那么调用 proc 的续集就是用 k 扩展的当前续集。

Examples:
> (+ 1
     (call/cc (lambda (k)
                (call-in-continuation k (lambda () 4)))))

5

> (+ 1
     (call/cc (lambda (k)
                (let ([n 0])
                  (dynamic-wind
                   void
                   (lambda ()
                     ; n accessed after post thunk
                     (call-in-continuation k (lambda () n)))
                   (lambda ()
                     (set! n 4)))))))

5

> (+ 1
     (with-continuation-mark
      'n 4
       (call/cc (lambda (k)
                  (with-continuation-mark
                   'n 0
                   (call-in-continuation
                    k
                    (lambda ()
                      ; 'n mark accessed in continuation
                      (continuation-mark-set-first #f 'n))))))))

5

(let/cc k body ...+)

相当于 (call/cc (lambda (k) body ...)) 。

syntax

(let/ec k body ...+)

相当于 (call/ec (lambda (k) body ...)) 。

procedure

(call-with-continuation-barrier thunk) → any

  thunk : (-> any)
应用 thunk ,在应用程序和当前继续之间设置一个 continuation barrier 。 thunk 的结果是调用 call-with-continuation-barrier 的结果。

procedure

(continuation-prompt-available? prompt-tag    
  [cont]) → any
  prompt-tag : continuation-prompt-tag?
  cont : continuation? = (call/cc values)
如果 cont (必须是一个续集)包括一个由 prompt-tag 标记的提示, 则返回 #t ,否则返回 #f 。

procedure

(continuation? v) → boolean?

  v : any/c
如果 v 是由 call-with-current-continuation, call-with-composable-continuation 或 call-with-escape-continuation 产生的 continuation,返回 #t ,否则返回 #f 。

procedure

(continuation-prompt-tag? v) → boolean?

  v : any/c
如果 v 是由 default-continuation-prompt-tag 或 make-continuation-prompt-tag 产生的 continuation,返回 #t 。

procedure

(dynamic-wind pre-thunk    
  value-thunk    
  post-thunk) → any
  pre-thunk : (-> any)
  value-thunk : (-> any)
  post-thunk : (-> any)
依次应用其三个 thunk 参数。 dynamic-wind 表达式的值是 value-thunk 返回的值。 pre-thunk 程序在调用 value-thunk 之前被调用, post-thunk 在 value-thunk 返回之后被调用。 当控制跳入或跳出 value-thunk 程序时(无论是由于提示中止还是继续调用), dynamic-wind 的特殊属性就体现出来了: 每当控制跳入 value-thunk 程序时, pre-thunk 就被调用, 而每当控制跳出 value-thunk 时, post-thunk就被调用。 (对于跳入或跳出 pre-thunk 和 post-thunk 程序的情况,没有进行特殊处理)。

当 dynamic-wind 调用 pre-thunk 对 value-thunk 进行正常评估时, pre-thunk 应用程序的继续调用 value-thunk (使用 dynamic-wind 的特殊跳转处理), 然后调用 post-thunk 。 同样地, post-thunk 应用的延续将前面 value-thunk 应用的值返回给整个 dynamic-wind 应用的延续。

当 pre-thunk 被调用时,由于延续性跳转,调用 pre-thunk 的延续性

  • 跳转到嵌套更深的 pre-thunk (如果有的话),或者跳转到目标延续;然后

  • 继续与目的延续中的封闭式 dynamic-wind 调用相同 (即与原始 dynamic-wind 调用的延续相匹配, 直到限定捕获的封闭式提示)。

通常情况下,由于第一部分的跳转,这个续集的第二部分永远不会到达。 然而,第二部分是相关的,因为它使跳转能够逃避目的延续中的 dynamic-wind 调用的延续中包含的延续。 此外,它意味着 pre-thunk 的延续标记(见 Continuation Marks) 和参数化(见 参数) 与包围的 dynamic-wind 调用的参数化相对应。 然而, pre-thunk 的调用是 parameterize-break , 以禁止中断(参见 中断) 。

同样地,当 post-thunk 由于延续性跳转而被调用时, 调用 post-thunk 的延续性会跳转到一个嵌套较少的 post-thunk (如果有的话), 或者跳转到一个保护目标的 pre-thunk (如果有的话),或者跳转到目标延续性, 然后继续进行与原延续性中的 dynamic-wind 调用一样的跳转。 对于 pre-thunk, 延续标记和 dynamic-wind 调用的参数化都在 post-thunk 位置, 只是该调用被进一步 parameterize-break 以禁止中断。

在这两种情况下,跳转的目的地都是在每个 pre-thunk 或 post-thunk 完成后重新计算的。 当在 post-thunk 中捕获一个以提示符为界限的续集(见 提示、限定的延续和障碍物) 时, 它可能被划定和实例化,以至于当续集被应用时,跳跃的目的地与捕获续集时不同。 甚至可能没有合适的目的地,如果相关的提示或转义续集不在恢复后的续集中;在这种情况下, pre-thunk 或 post-thunk 的续集的第一步会引发一个异常。

Examples:
> (let ([v (let/ec out
             (dynamic-wind
              (lambda () (display "in "))
              (lambda ()
                (display "pre ")
                (display (call/cc out))
                #f)
              (lambda () (display "out "))))])
    (when v (v "post ")))

in pre out in post out

> (let/ec k0
    (let/ec k1
      (dynamic-wind
       void
       (lambda () (k0 'cancel))
       (lambda () (k1 'cancel-canceled)))))

'cancel-canceled

> (let* ([x (make-parameter 0)]
         [l null]
         [add (lambda (a b)
                (set! l (append l (list (cons a b)))))])
    (let ([k (parameterize ([x 5])
               (dynamic-wind
                   (lambda () (add 1 (x)))
                   (lambda () (parameterize ([x 6])
                                (let ([k+e (let/cc k (cons k void))])
                                  (add 2 (x))
                                  ((cdr k+e))
                                  (car k+e))))
                   (lambda () (add 3 (x)))))])
      (parameterize ([x 7])
        (let/cc esc
          (k (cons void esc)))))
    l)

'((1 . 5) (2 . 6) (3 . 5) (1 . 5) (2 . 6) (3 . 5))

10.4.1 Additional Control Operators

 (require racket/control) package: base
The bindings documented in this section are provided by the racket/control library, not racket/base or racket.

racket/control 库提供了来自高阶控制操作符研究文献的各种控制操作符,加上一些额外的方便形式。 这些控制操作符是以 call-with-continuation-prompt, call-with-composable-continuation 等方式实现的,它们一般都能合理地协同工作。 许多是多余的;例如, reset 和 prompt 是别名。

procedure

(call/prompt proc [prompt-tag handler] arg ...) → any

  proc : procedure?
  prompt-tag : continuation-prompt-tag?
   = (default-continuation-prompt-tag)
  handler : (or/c procedure? #f) = #f
  arg : any/c
call/prompt 绑定是 call-with-continuation-prompt 的一个别名。

procedure

(abort/cc prompt-tag v ...) → any

  prompt-tag : any/c
  v : any/c
abort/cc 绑定是 abort-current-continuation 的一个别名。

procedure

(call/comp proc [prompt-tag]) → any

  proc : (continuation? . -> . any)
  prompt-tag : continuation-prompt-tag?
   = (default-continuation-prompt-tag)
call/comp 绑定是 call-with-composable-continuation 的一个别名。

procedure

(abort v ...) → any

  v : any/c
返回 v 到一个使用缺省延续提示标签和缺省中止处理程序的提示。

也就是说, (abort v ...) 等同于

(abort-current-continuation
 (default-continuation-prompt-tag)
 (lambda () (values v ...)))

Example:
> (prompt
    (printf "start here\n")
    (printf "answer is ~a\n" (+ 2 (abort 3))))

start here

3

syntax

(% expr)

(% expr handler-expr)
(% expr handler-expr #:tag tag-expr)

procedure

(fcontrol v #:tag prompt-tag) → any

  v : any/c
  prompt-tag : (default-continuation-prompt-tag)

Sitaram 的运算符 [Sitaram93] .

基本的还原规则是:

(% val proc) => val
(% E[(fcontrol val)] proc) => (proc val (lambda (x) E[x]))
  ; where E has no %

当 handler-expr 被省略时, % 与 prompt 相同。 如果提供了 prompt-tag , % 会使用特定的提示标签,如 prompt-at 。

Examples:
> (% (+ 2 (fcontrol 5))
     (lambda (v k)
       (k v)))

7

> (% (+ 2 (fcontrol 5))
     (lambda (v k)
       v))

5

syntax

(prompt expr ...+)

syntax

(control id expr ...+)

最早的高阶控制的运算符之一 [Felleisen88a, Felleisen88, Sitaram90] 。

基本的还原规则是:
(prompt val) => val
(prompt E[(control k expr)]) => (prompt ((lambda (k) expr)
                                         (lambda (v) E[v])))
  ; where E has no prompt

Examples:
> (prompt
    (+ 2 (control k (k 5))))

7

> (prompt
    (+ 2 (control k 5)))

5

> (prompt
    (+ 2 (control k (+ 1 (control k1 (k1 6))))))

7

> (prompt
    (+ 2 (control k (+ 1 (control k1 (k 6))))))

8

> (prompt
    (+ 2 (control k (control k1 (control k2 (k2 6))))))

6

syntax

(prompt-at prompt-tag-expr expr ...+)

syntax

(control-at prompt-tag-expr id expr ...+)

像 prompt 和 control, 但使用特定的提示标签:

(prompt-at tag val) => val
(prompt-at tag E[(control-at tag k expr)]) => (prompt-at tag
                                               ((lambda (k) expr)
                                                (lambda (v) E[v])))
  ; where E has no prompt-at for tag

syntax

(reset expr ...+)

syntax

(shift id expr ...+)

Danvy 和 Filinski 的运算符 [Danvy90] 。

基本的还原规则是:

(reset val) => val
(reset E[(shift k expr)]) => (reset ((lambda (k) expr)
                                     (lambda (v) (reset E[v]))))
  ; where E has no reset

reset 和 prompt 类似,但使用指定的提示标签。

syntax

(reset-at prompt-tag-expr expr ...+)

syntax

(shift-at prompt-tag-expr identifier expr ...+)

与 reset 和 shift 类似,但使用指定的提示标签。

syntax

(prompt0 expr ...+)

syntax

(reset0 expr ...+)

syntax

(control0 id expr ...+)

syntax

(shift0 id expr ...+)

prompt 的泛化,等等。 [Shan04] 。

基本的还原规则是:

(prompt0 val) => val
(prompt0 E[(control0 k expr)]) => ((lambda (k) expr)
                                   (lambda (v) E[v]))
(reset0 val) => val
(reset0 E[(shift0 k expr)]) => ((lambda (k) expr)
                                (lambda (v) (reset0 E[v])))

reset0 和 prompt0 的形式是可以互换的。此外,以下的减法也适用:

(prompt E[(control0 k expr)]) => (prompt ((lambda (k) expr)
                                          (lambda (v) E[v])))
(reset E[(shift0 k expr)]) => (reset ((lambda (k) expr)
                                      (lambda (v) (reset0 E[v]))))
(prompt0 E[(control k expr)]) => (prompt0 ((lambda (k) expr)
                                           (lambda (v) E[v])))
(reset0 E[(shift k expr)]) => (reset0 ((lambda (k) expr)
                                       (lambda (v) (reset E[v]))))

也就是说, prompt/reset 和 control/shift 站点都必须同意类似 0 的行为, 否则适用非 0 的行为。
