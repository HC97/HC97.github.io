15.1.4 Windows 路径

一般来说,Windows 路径名由一个可选的驱动器指定符和一个特定的驱动器路径组成。 一个 Windows 路径可以是 absolute ,但仍然是相对于当前驱动器的;这种路径以 / 或 \ 分隔符开始,不是 UNC 路径或以 \\?\ 开始的路径。

一个以驱动器规格开始的路径是 complete 。 粗略的说,驱动器说明是一个拉丁字母,后面跟着一个冒号,一个 UNC 路径,形式是 \\‹machine›\‹volume›, 或者是一个 \\?\ 形式,后面跟着 REL\‹element› 以外的东西, 或者 RED\‹element› 。 (下面将进一步描述 \\?\ 路径的变体)。

Racket 在一个方面没有实现通常的 Windows 路径语法。 在 Racket 之外,一个路径名 "C:rant.txt" 可以是一个特定驱动器的相对路径。 也就是说,它在驱动器 "C:" 上命名了一个文件 "rant.txt" , 但该文件的完整路径是由驱动器 "C:" 的当前工作目录决定的。 Racket 不支持特定驱动器的工作目录(只支持所有驱动器的工作目录,如 current-directory 参数所反映)。 因此,Racket 隐含地将 "C:rant.txt" 这样的路径转换为 "C:\rant.txt" 。

  • Racket-specific: 每当一个路径以驱动器指定符 ‹letter› 开头 : 后面没有 / 或 \, 在 cleanse 路径的时候就会插入一个 \ 。

否则,Racket 遵循标准的 Windows 路径约定,但也增加了 \\?\REL 和 \\?\RED 约定, 以处理标准约定中无法表达的路径,以及处理 \\?\ 路径中过多 \ 的约定。

在下文中, ‹letter› 代表拉丁字母(大小写不重要), ‹machine› 代表任何不包括 \ 或 / 且不是 ? 的字符序列, ‹volume› 代表任何不包括 \ 或 / 的字符序列, ‹element› 代表任何不包括 \ 的字符序列。

  • 当元素是路径中的最后一个时,路径元素中的尾部空格和 . 会被忽略, 除非路径以 \\?\ 开头或者元素只由空格和 . 组成。

  • 以下是访问设备的特殊 "文件",它们存在于所有目录中,不考虑大小写, 并且在句号或冒号后有所有可能的结尾,但在以 \\?\ 开头的路径名中除外: "NUL", "CON", "PRN", "AUX", "COM1", "COM2", "COM3", "COM4", "COM5", "COM6", "COM7", "COM8", "COM9", "LPT1", "LPT2", "LPT3", "LPT4", "LPT5", "LPT6", "LPT7", "LPT8", "LPT9"。

  • 除了 \\?\ 路径, / 等同于 \ 。 除了 \\?\ 路径和UNC路径的开始,多个相邻的 / 和 \ 算作一个单一的 \ 。 在一个开始于 \\?\ 的路径中,元素之间可以用一个或两个 \ 来分隔。

  • 一个目录可以用或不用尾部的分隔符来访问。 在非 \\?\ 路径的情况下,尾部分隔符可以是任何数量的 / 和 \ ; 在 \\?\ 路径的情况下,尾部分隔符必须是一个单一的 \, 但两个 \ 可以跟在 \\?\‹letter›: 后面。

  • 除了 \\?\ 路径,单个 . 作为路径元素意味着 "当前目录", 而 .. 作为路径元素意味着 "父目录"。 紧跟在驱动器之后的上层目录路径元素(即 ..) 被忽略。

  • 一个以 \\‹machine›\‹volume› 开头的路径名(其中 / 可以代替任何 \) 是一个 UNC 路径, 开头的 \\‹machine›\‹volume› 算作驱动器指定符。

  • 通常情况下,一个路径元素不能包含 #\x 0 到 #\x 1F 范围内的字符,也不能包含以下任何字符:

    < > : " / \ | ? *

    除了 \ 之外,包含这些字符的路径元素可以用一个 \\?\ 路径来访问 (假设底层文件系统允许这些字符)。

  • 在一个以 \\?\‹letter›:\ 开头的路径名中, 只要路径不包含非驱动元素和以两个连续的 \ 结尾,并且只要路径不包含三个或更多的 \ 序列, \\?\‹letter›:\ 前缀就算作路径的驱动。 两个 \ 可以代替 ‹letter› 前的 \ 出现。 一个 / 不能用来代替一个 \ (但是 / 可以用在元素名中, 尽管结果通常不是命名一个实际的目录或文件)。

  • 在一个以 \\?\UNC\‹machine›\‹volume› 开头的路径名中, 只要路径不是以两个连续的 \ 结尾,并且只要路径不包含三个或更多的 \ 序列, \\?\UNC\‹machine›\‹volume› 前缀就算作路径的驱动器。 两个 \ 可以出现在 UNC 之前的 \、 UNC 之后的 \ 和/或 ‹machine› 之后的 \ 。 UNC 部分的字母可以是大写或小写,而且 / 不能用来代替 \ (但 / 可以用在元素名称中)。

  • Racket-specific: 以 \\?\REL\‹element› 或 \\?\REL\\‹element› 开头的路径是一个相对路径, 只要该路径不以两个连续的 \ 结束,并且该路径不包含三个或更多的 \ 序列。 这种 Racket 特有的路径形式支持带有通常不能在 Windows 路径中表达的元素的相对路径(例如,以空格结束的最后一个元素)。 REL 部分必须正好是三个大写字母,而且不能用 / 来代替 \ 。 如果路径以 \\?\REL\.. 开始,那么只要路径继续重复 \.., 每个元素都算作一个上行目录元素;必须用一个单一的 \ 来分隔上行目录元素。 只要用第二个 \ 来分隔元素,或者只要遇到一个非 .. 的元素, 剩下的元素都是字面的(从来没有上层目录元素)。 当一个 \\?\REL 路径值被转换为字符串时(或者当路径值被写入或显示时), 字符串不包含开始的 \\?\REL 或紧随其后的 \ ; 将路径值转换为字节字符串时保留了 \\?\REL 前缀。

  • Racket-specific: 一个以 \\?\RED\‹element› 或 \\?\RED\\‹element› 开头的路径是一个驱动器相关路径, 只要该路径不以两个连续的 \ 结束,并且该路径不包含三个或更多的 \ 序列。 这种 Racket 特有的路径形式支持驱动器相关路径(即给定一个驱动器的绝对路径),其中的元素通常不能用 Windows 路径来表达。 RED 部分必须正好是三个大写字母,而且 / 不能用来代替 \ 。 与 \\?\REL 路径不同, .. 元素总是一个字面路径元素。 当一个 \\?\RED 路径值被转换为字符串时(或当路径值被写入或显示时),字符串不包含开始的 \\?\RED , 它包含一个开始的 \; 将路径值转换为一个字节字符串时,保留 \\?\RED 前缀。

三个额外的 Racket 特定规则为 Windows 路径中的字符序列提供了意义,否则这些字符序列就不符合 Windows 路径的形式:

  • Racket-specific: 在形式为 \\?\‹any›\\ 的路径名中, ‹any› 是除 ‹letter›: 或 \‹letter›: 以外的任何非空字符序列,整个路径算作该路径的(不存在的)驱动器。

  • Racket-specific: 在一个形式为 \\?\‹any›\\\‹elements› 的路径名中, ‹any› 是任何非空的字符序列, ‹elements› 是任何不以一个 \ 开始、不以两个 \ 结束、不包含三个 \ 的序列, 那么 \\?\‹any›\\ 算作路径的(不存在的)驱动器。

  • Racket-specific: 在一个以 \\?\开头的路径名中,不符合前面的任何模式， \\?\ 算作该路径的(不存在的)驱动器。

在 Racket 之外,除了 \\?\ 路径,路径名作为文件路径时通常被限制在 259 个字符, 作为目录路径时被限制在 247 个字符。 Racket 内部将超过 247 个字符的路径名转换为 \\?\ 形式以避免限制; 在这种情况下,路径首先在语法上被简化(在 simplify-path 意义上)。 操作系统不能通过长于 32000 个字符左右的 \\?\ 路径访问文件。

在上述描述中提到 "字符" 的地方,用 "字节" 代替,以便将字节串解释为路径。 将 Windows 路径编码为字节,保留了 ASCII 字符,上面提到的所有特殊字符都是 ASCII,所以所有的规则都是一样的。

请注意, \ 路径分隔符在 Racket 字符串中是一个转义字符。 因此,路径 \\?\REL\..\\.. 作为一个字符串必须写成 "\\\\?\\REL\\..\\\\.." .

一个以目录分隔符结尾的路径在语法上指的是一个目录。 此外,如果一个路径的最后一个元素是一个同目录或上目录的指示器(没有被 \\?\ 形式引用), 或者它指向一个根,那么它在语法上就是指一个目录。

即使在支持符号链接的 Windows 变体上,路径中的上层目录 .. 指标也是按语法解决的,对链接不敏感。 例如,如果一个路径以 d\..\f 结尾,而 d 指的是一个符号链接, 其引用的目录与 d 的父目录不同,那么这个路径仍然指的是与 d 相同目录中的 f 。 一个相对路径链接被解析为前缀为 \\?\REL 的路径,除了整个路径允许有 .. 和 . 元素, 而且允许有任何数量的多余的 \ 分隔符。

Windows路径的 cleanse 方法如下: 在以 \\?\ 开头的路径中,多余的 \ 被删除,在 \\?\REL 中加入一个额外的 \ ,如果一个额外的\还没有出现的话,以便将上行目录指示器与字面路径元素分开, 如果一个额外的\还没有出现的话,同样在 \\?\RED 之后加入一个额外的 \ 。 对于其他路径,多个 / 和 \ 被转换为单个 / 或 \ (除了在共享文件夹名称的开头),如果驱动器说明中缺少冒号, 则在冒号后插入一个 \ 。

对于 (bytes->path-element bstr), bstr 中的 / 、冒号、尾部点、尾部空白和特殊设备名称(例如,"aux")通过使用 \\?\REL 前缀编码为路径元素的字面部分。 bstr 参数不能包含一个 \, 否则会引发 exn:fail:contract exception is raised 。

对于 (path-element->bytes path) 或 (path-element->string path) , 如果 byte-string 形式的 path 以 \\?\REL 开头,该前缀不包括在结果中。

对于 (build-path base-path sub-path ...), 从 base-path 的最后一个元素和除最后一个 sub-path 之外的所有元素中删除尾部的空格和句号(除非该元素只由空格和句号组成), 除了那些以 \\?\ 开头的元素。 如果 base-path 以 \\?\ 开头, 那么在每个非\\?\REL\ 和非 \\?\RED\ sub-path 后, 所有的 / 都被转换为 \, 多个连续的 \ 被转换为一个单一的 \, 添加的 . 元素被移除, 添加的 .. 元素与前面的元素一起被移除; 这些转换不会在结果的原始 base-path 部分或任何的 \\?\REL\ 或 \\?\RED\ 或 sub-path 上进行。 如果一个 \\?\REL\ 或 \\?\RED\ sub-path 被添加到一个非 \\?\ base-path 上, base-path (以及任何添加到 \\?\REL\ 或 \\?\RED\ sub-path 的部分)被简化并转换为一个 \\?\ 路径。 在其他情况下,在组合路径之前可以添加或删除一个 \ , 以避免改变路径的根本含义(例如,组合 //x 和 y 产生 /x/y, 因为 //x/y 将是一个 UNC 路径而不是一个驱动器相关的路径)。

对于 (simplify-path path use-filesystem?), path 被展开, 如果 path 不是以 \\?\ 开始,则删除尾部的空格和句号,如果驱动器规格中的冒号缺少, 则在其后面插入一个 / ,如果有元素且已经没有额外的 \ , 则在 \\?\ 后面插入一个 \ 作为根。 否则,如果 path 中没有指示器或多余的分隔符,那么就会返回 path 。

对于产生 base,name 和 must-be-dir? 的 (split-path path) 来说, 分割一个不以 \\?\ 开头的路径可以产生以 \\?\ 开头的部分。 例如,分割 C:/x /aux/ 两次会产生 \\?\REL\\x  和 \\?\REL\\aux; 在这些情况下需要 \\?\ 来保留 x 后面的空格,并避免提及 AUX 设备而不是 "aux" 文件。

15.1.4.1  Windows的路径表示法

在 Windows 上,路径本身是一个 UTF-16 编码单元的序列,该序列可以包括未配对的代用词。 这个序列通过 UTF-8 的扩展被编码为一个字节字符串,其中 UTF-16 编码单元序列中的非配对代用词被转换为非代用词值。 这些扩展编码在 Windows 上被实现为 bytes-open-converter 的 "platform-UTF-16" 和 "platform-UTF-8" 。

Racket 对 Windows 路径的内部表示是一个字节字符串, 因此 path->bytes 和 bytes->path 总是反相的。 当把路径转换为本地 UTF-16 编码单元序列时, #\tab 被用来代替平台 UTF-8 解码错误 (理由是 tab 通常不允许作为 Windows 路径中的一个字符,与 #\uFFFD 不同)。

一个 Windows 路径被转换为一个字符串,将 平台-UTF-8 编码视为 UTF-8 编码,用 #\uFFFD 代替解码错误。 同样,一个字符串通过 UTF-8 编码转换为路径(在这种情况下不可能出现错误)。
