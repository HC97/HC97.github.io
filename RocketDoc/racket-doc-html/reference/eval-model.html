1.1 评价模型

Racket 求值可以被看作是对表达式的简化以获得数值。例如,就像一个小学生简化了

  1 + 1 = 2

Racket 求值简化了

(+ 1 1) → 2

箭头 → 取代了更传统的 = ,以强调评估朝着更简单的表达式进行的特定方向。 特别是,一个 value ,如数字 2, 是一个表达式,它的求值不再简化。

1.1.1 子表达式的评价和续篇

有些简化需要一个以上的步骤。例比:

(- 4 (+ 1 1)) → (- 4 2) → 2

一个不是 value 的表达式总是可以被分成两部分: redex (“可简化表达式”), 这是单步简化中可以改变的部分(突出显示),以及 continuation,这是围绕 redex 的求值环境。 在 (- 4 (+ 1 1)) 中,红字是 (+ 1 1) ,延续是 (- 4 []), 其中 [] 代替 redex 的位置,因为它被简化。 也就是说, continuation 说的是在 redex 被减少到一个 value 之后如何 "继续"。

在某些表达式被求值之前,它们的一些或全部子表达式必须被求值。 例如,在应用 (- 4 (+ 1 1)) 中,在子表达式 (+ 1 1) 被还原之前, - 的应用不能被还原。 因此,每个句法形式的规范都指定了如何求值(一些)其子表达式,然后如何将结果结合起来以减少该形式。

一个表达式的 dynamic extent 是求值步骤的序列,在这个过程中,表达式包含 redex。

1.1.2 尾部位置

如果在 expr1 成为 redex 的时候, 它的 continuation 与包围的 expr2 的 continuation 相同, 那么表达式 expr1 相对于包围的表达式 expr2 来说就处于 尾部位置 。

例如, (+ 1 1) 表达式相对于 (- 4 (+ 1 1)) 来说 不 在 尾部位置 。 为了说明这一点,我们使用符号 C[expr] 来表示在某个 continuation C 中用 expr 代替 [] 而产生的表达式。

C[(- 4 (+ 1 1))] → C[(- 4 2)]

在这种情况下,还原 (+ 1 1) 的 continuation 是 C[(- 4 [])], 而不仅仅是 C 。上面第一段中规定的要求没有得到满足。

相比之下, (+ 1 1) 相对于 (if (zero? 0) (+ 1 1) 3) 处于 尾部位置 , 因为对于任何 continuation C。

C[(if (zero? 0) (+ 1 1) 3)] → C[(if #t (+ 1 1) 3)] → C[(+ 1 1)]

第一段中规定的要求得到了满足。这个缩减序列中的步骤是由 if 的定义驱动的, 它们不依赖于 continuation C 。 一个 if 形式的 “then” 分支相对于 if 形式总是处于 尾部位置 。由于 if 和 #f 的类似还原规则, if 形式的 "else "分支也在 尾部位置。

尾部位置 规范提供了关于计算的渐近空间消耗的保证。 一般来说, 尾部位置 的规范伴随着每个句法形式的描述,比如 if。

1.1.3 多个返回值

一个 Racket 表达式可以评估为 multiple values ,以便与一个过程可以接受多个参数的事实相对称。

大多数 continuations 程序期望有一定数量的结果 values, 尽管有些 continuations 程序可以接受任意数量。 事实上,大多数 continuations, 如 (+ [] 1), 期望有一个单一的 value。 continuation (let-values ([(x y) []]) expr) 期待两个结果 values; 第一个结果替换主体 expr 中的 x , 第二个结果替换 expr 中的 y 。 continuation (begin [] (+ 1 2)) 接受任何数量的结果 values ,因为它忽略了结果。

一般来说,一个语法形式的规范表明了它产生的 values 的数量和它期望从每个子表达式中得到的数量。 此外,一些程序(特别是 values) 产生多个 values , 一些程序(特别是 call-with-values) 在内部创建接受一定数量的 values 的连续。

1.1.4 顶层变量

给出

  x = 10

那么一个代数学生将 x + 1 简化如下:

  x + 1 = 10 + 1 = 11

Racket 的工作方式与此基本相同,在求值过程中, 一组 顶层变量 (参见 变量和位置)可供替换使用。例如,给定

(define x 10)

那么

(+ x 1) → (+ 10 1) → 11

在 Racket 中,创建定义的方式与使用定义的方式同样重要。 因此,Racket 的评估会同时跟踪定义和当前的表达式,并根据评估的形式(如 define)扩展定义集。

那么,每个评估步骤都会将当前的定义和程序集转化为新的定义和程序集。 在 define 被移入定义集之前,它的表达式(即它的右手边)必须被还原成一个 value。 (左手边不是一个表达式的位置,所以它不被评估)。

1.1.5 对象和强制更新

除了 set! 用于 top-level variables 的强制更新外,还有各种程序可以修改复合数据结构中的元素。 例如, vector-set! 修改了一个向量的内容。

为了解释这种对数据的修改,我们必须区分 values 和 objects, 前者是表达式的结果,后者是实际持有的数据。

有几类 objects 可以直接作为值,包括布尔运算、 (void) 和小的精确整数。 然而,更普遍的是,一个 value 是对存储在其他地方的一个 object 的引用。 例如,一个 value 可以指一个特定的向量,该向量目前在其第一个槽中保存着值 10 。 如果一个 object 通过一个 value 被修改, 那么这个修改通过所有引用该 object 的 values 可见。

top-level variable 和对象引用之间的区别是至关重要的。 top-level variable 不是一个 value,所以它必须被评估。 每次评估一个 variable 表达式时,都要从当前的定义集中提取该变量的值。 与此相反,一个对象引用是一个值,因此不需要进一步评估。 上面的评估步骤使用角括号 <o1> 表示对象引用,以区别于 variable 名。

一个对象的引用不可能直接出现在基于文本的源程序中。 然而,用 datum->syntax 创建的程序表示,可以嵌入对现有 objects 的直接引用。

1.1.6 垃圾回收

与垃圾收集有关的功能见( 内存管理 部分）。

在程序状态下

objects:

(define <o1> (vector 10 20))
(define <o2> (vector 0))

defined:

(define x <o1>)

evaluate:

(+ 1 x)

评价不能依赖于 <o2>, 因为它不是要评价的程序的一部分, 而且它没有被程序可以访问的任何定义所引用。 这个对象被说成是 reachable。 因此, object <o2> 可以通过 垃圾收集 从程序状态中删除。

一些特殊的复合数据类型持有对对象的 weak references 。 垃圾收集器在确定哪些 objects 在剩余的计算中是可到达的时候,会特别处理这种弱引用。 如果一个 object 只能 通过 weak reference到达,那么这个对象可以被回收, weak reference 被一个不同的 value (通常是 #f)所取代。

作为一个特例, fixnum 总是被认为是可以被垃圾收集器到达的。 许多其他的值由于它们的实现和使用方式,总是可以达到的： 一个在Latin-1范围内的 character 总是可及的, 因为 equal? Latin-1字符总是 eq?, 而且所有的Latin-1字符都被一个内部模块所引用。 同样, null,#t, #f, eof 和 #<void> 也总是可达的。 当 quote 表达式本身可达时,由 quote 产生的值仍然可达。

1.1.7 程序应用和局部变量

给出

  f(x) = x + 10

一个代数学生将 f(7) 简化如下:

  f(7) = 7 + 10 = 17

这种简化的关键步骤是将定义的函数 f 的主体, 用实际value 7 替换每个 x 。 .

Racket过程应用的工作方式与此基本相同。过程是一个 object, 所以评估 (f 7) 要从一个variable 的查找开始:

objects:

(define <p1> (lambda (x) (+ x 10)))

defined:

(define f <p1>)

evaluate:

(f 7)

→

objects:

(define <p1> (lambda (x) (+ x 10)))

defined:

(define f <p1>)

evaluate:

(<p1> 7)

然而,与代数不同的是,与过程参数变量相关的 value 可以在过程的主体中通过使用 set! 来改变,如例子中的 (lambda (x) (begin (set! x 3) x)) 。 由于与参数变量 x 相关的 value 应该是可以改变的, 所以我们不能在第一次应用过程时直接用这个值来代替 x 。

我们不使用 "参数变量 "这一术语来指代与函数一起声明的参数变量名。 这一选择避免了与 parameters的混淆。

取而代之的是,在每个应用程序中为每个 variable 创建一个新的 location 。 参数 value 被放置在这个 location 上, 并且程序主体中的每个 variable 实例都被替换成新的 location:

objects:

(define <p1> (lambda (x) (+ x 10)))

defined:

(define f <p1>)

evaluate:

(<p1> 7)

→

objects:

(define <p1> (lambda (x) (+ x 10)))

defined:

(define f <p1>)
(define xloc 7)

evaluate:

(+ xloc 10)

→

objects:

(define <p1> (lambda (x) (+ x 10)))

defined:

(define f <p1>)
(define xloc 7)

evaluate:

(+ 7 10)

→

objects:

(define <p1> (lambda (x) (+ x 10)))

defined:

(define f <p1>)
(define xloc 7)

evaluate:

17

一个 location 和一个 top-level variable 是一样的, 但是当一个 location 被生成时,它(在概念上)使用的是一个以前没有使用过的名字, 而且不能再次生成或直接访问。

以这种方式生成一个 location 意味着 set! 对 local variables (包括参数变量)的评估方式与对 top-level variables 的评估方式相同, 因为 local variable 在 set! 形式被评估时总是被替换成一个 location :

程序应用的 location 生成和置换步骤要求参数是一个 value 。 因此,在 ((lambda (x) (+ x 10)) (+ 1 2)) 中, (+ 1 2) 子表达式必须简化为 value 3, 然后 3 可以被放置到 x 的 location 。 换句话说,Racket 是一种 call-by-value 的语言。

对局部变量形式的评估,如 (let ([x (+ 1 2)]) expr), 与过程调用相同。 在 (+ 1 2) 产生一个 value 后,它被存储在一个新的 location ,取代 expr 中的每个 x 实例。

1.1.8 变量和位置

variable 是一个 value 的占位符, 初始程序中的表达式是指 variables 。 一个 top-level variable 既是一个 variable 又是一个 location 。 任何其他 variable 在运行时总是被一个 location 取代; 因此,表达式的评估只涉及 locations 。 一个单一的 local variable (即非顶层、非模块级的 variable),如一个参数变量, 在不同的应用中可以对应不同的 locations 。

例如,在程序中

(define y (+ (let ([x 5]) x) 6))

y 和 x 都是 variables 。 y variable 是一个 top-level variable , 而 x 是一个 local variable 。 当这段代码被评估时,为 x 创建了一个存放数值 5 的 location ,同时也为 y 创建了一个存放数值 11 的 location 。

在评估过程中,用一个 location 替换一个 variable 实现了 Racket 的 lexical scoping。 为了将 xloc 替换为 x ,所有的变量绑定必须使用不同的名字, 所以没有任何一个真正的 x 会被替换为不同的变量。 确保这种区别是宏扩展器的工作之一;见 语法模型。 例如,当一个参数变量 x 被 location xloc 替换时, 它将被替换到 整个 过程的主体中,包括任何嵌套的 lambda 形式。 因此,未来对该 variable 的引用总是访问同一个 location。

1.1.9 模块和模块级变量

模块的语法见 Modules: module, module*, ... 。

Racket 中的大多数定义都在 modules 中。 就评估而言,模块本质上是一个定义名称的前缀,因此不同的模块可以定义相同的名称。 也就是说,从评估的角度来看, 一个 module-level variable 就像一个 top-level variable 。

模块和顶层定义之间的一个区别是, 模块可以在不实例化其模块级定义的情况下被 declared。 对 require 的评估会将声明的模块 require instantiates (即触发 instantiation ,从而创建与模块级定义对应的变量。

例如,给定模块声明

(module m racket
  (define x 10))

(require 'm) 的评估创建了变量 x 并将 10 作为它的值。 这个 x 与 x 的任何顶层定义无关(就像它被赋予了一个独特的、模块专用的前缀)。

1.1.9.1 阶段

See also 通用阶段等级 in Racket 指南.

phases 的目的是为了解决执行时定义的名称与扩展时定义的名称之间的必要分离。

一个模块可以在多个 phases 进行 instantiate 。 阶段是一个整数,和模块名称一样,实际上是模块级定义名称的前缀。阶段 0 是执行时间阶段。

顶级 require 在 phase 0 instantiates 一个模块, 如果该模块在阶段 0 还没有 instantiate 。 顶级 (require (for-syntax ....)) 在 phase 1 instantiates 一个模块 (如果它在该阶段还没有 instantiate); for-syntax 在进一步的程序解析中也有不同的绑定效果,如 引入绑定 所述。

在模块中,一些定义已经按阶段进行了移位: begin-for-syntax 形式类似于 begin ,但它将表达式和定义按相对 phase +1 进行移位。 同样, define-for-syntax 形式类似于 define ,但将定义移动了+1。 因此,如果模块在阶段 1 instantiate ,则使用 begin-for-syntax 定义的变量将在阶段 2 创建,依此类推。 此外,这个相对阶段充当前缀的另一层, 因此用 define 定义的 x 和用 define-for-syntax 定义的 x 可以在一个模块中共存而不发生冲突。 begin-for-syntax 可以嵌套在 begin-for-syntax 形式中, 在这种情况下,内部定义和表达式处于相对 phase +2, 依此类推。 高级阶段主要与程序解析有关,而不是正常的求值。

如果在 phase n instantiate 的模块 require 另一个模块, 那么所 require 的模块将首先在阶段 n instantiate ,以此类推。 (模块 require 不能形成循环。) 如果在第 n 阶段 instantiate 的模块 require 另一个模块 M for-syntax, 那么 M 在第 phase n+1 available , 之后可能会在第 phase n+1 instantiate 它。 如果在阶段 n (对于 n>0) available 的模块 require 另一个模块 M for-template, 则 M 在 phase n-1 available ,以此类推。 phase 0 以上 available 模块的 Instantiation 将按需触发, 如 模块扩展、阶段和访问 所述。

模块 instantiations 之间的最后一个区别是, 在 phase 1 和更高阶段可能存在多个 instantiations 。 这些 instantiations 是通过解析模块形式创建的(参见 模块扩展、阶段和访问), 并且在概念上也是通过前缀来区分的。

顶层变量可以存在于多个阶段,这与模块内的方式相同。 例如, begin-for-syntax 中的 define 创建了一个 phase 1 的变量。 此外,像 make-base-namespace 和 eval 这样的反射性操作提供了对更高 phases 的顶层变量的访问,而相对于这些顶层变量的模块 instantiations (由 require 触发)则处于相应的更高 phase 。

1.1.9.2 单独的编译保证

当一个模块被编译时,其 phase 1 被实例化了。 这反过来又会触发其他许多模块在其他阶段(包括阶段 1)的反式实例化。 Racket 为这种实例化提供了一个非常强大的保证,称为 “单独编译保证”:

由于编译对 Racket 运行系统造成的模块的第1阶段实例化的任何 effects 都会被 discarded。

担保涉及 effects 。有两种不同的效果:内部和外部。

内部效应以突变为例。 突变是一个函数的动作,比如 set-box!, 它改变了盒子里包含的值。 被修改的盒子在 Racket 之外是无法观察到的,所以这个效果被称为 "内部"。 根据定义,内部效应在 Racket 程序之外是无法检测的。

外部效应以输入/输出(I/O)为例。 I/O 是像 tcp-connect 这样的函数的行为,它与操作系统通信,向运行 Racket 的机器外发送网络数据包。 这些数据包的传输在 Racket 外部是可以观察到的,特别是由接收计算机或中间的任何路由器。 外部效应的存在是可以在 Racket 程序之外检测到的,而且通常可以用物理过程来检测。

当一个效应不再可被检测到时,它就 discarded 。 例如,当一个盒子从 3 到 4 的突变不再能被察觉到它曾经被改变, 从而仍然包含 3 的时候,它就被抛弃了。 因为外部效应在 Racket 外部是可以观察到的,它们是不可逆的,不能被丢弃。

因此,"单独编译保证" 只涉及像突变这样的效应,因为它们完全是 "对 Racket 运行系统" 而不是 "对物理宇宙" 的效应。

Racket 程序可以做很多看起来是内部效应但实际上是外部效应的事情。 例如, bytes-set! 通常是一个内部效应,除非字节是由 make-shared-bytes 创建的, 它在其他进程可观察到的空间中分配。 因此,修改这些字节的效果是不可丢弃的,所以 bytes-set! 在这种情况下,有一个外部效果。

反之亦然:一些看起来是外部的东西实际上是内部的。 例如,如果一个 Racket 程序启动了多个线程,并使用突变在它们之间进行通信,那么这种突变纯粹是内部的, 因为 Racket 的线程完全是内部定义的(它们与操作系统的线程没有关系)。

此外,每当Racket程序调用一个 unsafe 的函数时,Racket 运行时系统对其效果不做任何承诺。 例如,所有的外来调用都使用 ffi/unsafe, 所以所有的外来调用都是不安全的,它们的效果不能被 Racket 抛弃。

最后,"单独编译保证" 只涉及编译过程中第 1 阶段的实例化,而不是一般意义上的所有第 1 阶段的实例化, 例如,当其第1阶段需要并通过反射机制用于效果时。

这一保证的实际结果是,由于效果永远不可见,没有任何模块可以检测到它所 require 的模块是否已经被编译。 因此,它永远不能把一个模块的编译改变为已经编译了另一个模块。 特别是,如果模块 A 是由模块 X 和 Y 的第一阶段部分共享的, 那么在 X 被编译时的任何内部效应在 Y 的编译过程中是不可见的, 无论 X 和 Y 是否在 Racket 运行系统的同一执行过程中被编译,也无论编译的顺序如何。

下面的一组模块证明了这一保证。 首先,我们定义了一个具有通过 box 观察效果能力的模块:

(module box racket/base
  (provide (all-defined-out))
  (define b (box 0)))

接下来,我们定义两个语法转换器,使用和变异这个盒子:

(module transformers racket/base
  (provide (all-defined-out))
  (require (for-syntax racket/base 'box))
  (define-syntax (sett stx)
    (set-box! b 2)
    #'(void))
  (define-syntax (gett stx)
    #`#,(unbox b)))

接下来,我们定义一个使用这些变换器的模块:

(module user racket/base
  (provide (all-defined-out))
  (require 'transformers)
  (sett)
  (define gott (gett)))

最后,我们定义了第二个模块,该模块使用这些变换器和 user 模块:

(module test racket/base
  (require 'box 'transformers 'user)
  (displayln gott)
  (displayln (gett))
 
  (sett)
  (displayln (gett))
 
  (displayln (unbox b)))

这个模块显示:

  • 2, 因为模块 user 中的 (gett) 扩展为 2。

  • 0, 因为编译 user 的效果被丢弃了。

  • 2, 因为 test 里面的 (sett) 的效果还没有被抛弃。

  • 0, 因为第一阶段的 sett 的影响与 (unbox b) 中 b 的 0 阶段使用无关。

此外,无论这些模块以何种顺序编译,也无论它们是同时编译还是分别编译,这种显示都不会改变。

相反,如果这些模块被改变为在文件系统的一个文件中存储 b 的值, 那么程序将只显示 2。

分开编译保证在论文 "可组合和可编译的宏" [Flatt02] 和 "Racket中的子模块" [Flatt13] 中有更详细的描述, 包括翔实的例子。 论文 “Advanced Macrology and the implementation of Typed Scheme” [Culpepper07] 也包含了一个扩展的例子,说明为什么它很重要,以及如何在它的存在下设计有效的语法扩展。

1.1.9.3 跨阶段的持久性模块

符合高度约束形式的模块声明–包括模块主体中的 (#%declare #:cross-phase-persistent) 形式–可以创建 cross-phase persistent 模块。 一个 cross-phase persistent 模块在所有阶段的实例化都共享该模块第一个实例化产生的变量。 此外,当 cross-phase persistent 模块的实例共享一个共同的模块声明时, 这些实例会在 module registries 中持久化。

Examples:
> (module cross '#%kernel
    (#%declare #:cross-phase-persistent)
    (#%provide x)
    (define-values (x) (gensym)))
> (module noncross '#%kernel
    (#%provide x)
    (define-values (x) (gensym)))
> (define ns (current-namespace))
> (define (same-instance? mod)
    (namespace-require mod)
    (define a
      (parameterize ([current-namespace (make-base-namespace)])
        (namespace-attach-module-declaration ns mod)
        (namespace-require mod)
        (namespace-variable-value 'x)))
    (define b
      (parameterize ([current-namespace (make-base-namespace)])
        (namespace-attach-module-declaration ns mod)
        (namespace-require mod)
        (namespace-variable-value 'x)))
    (eq? a b))
> (same-instance? ''noncross)

#f

> (same-instance? ''cross)

#t

cross-phase persistent 模块的意图是支持跨 phase 后可识别的值。 例如,当运行在第 1 阶段的宏转换器引发一个由 exn:fail:syntax 实例表示的语法错误时, 这个实例可以被第 0 阶段的异常处理程序识别, 该异常处理程序包装了对引发语法错误的 eval 或 expand 的调用, 因为 exn:fail:syntax 结构类型是由 cross-phase persistent 模块定义的。

一个 cross-phase persistent 模块只导入其他 cross-phase persistent 模块, 并且它只包含将变量与函数、结构类型和相关函数、或结构类型属性和相关函数绑定的定义。 一个 cross-phase persistent 模块从不包括语法字面(通过 quote-syntax) 或变量引用(通过 #%variable-reference)。 关于 cross-phase persistent 模块声明的语法规范,见( 跨阶段持久模块声明 )。

一个被记录的模块应该被认为是 cross-phase persistent , 除非它被指定为 cross-phase persistent (比如 racket/kernel)。

1.1.9.4 模块的重新声明

当使用一个已经声明过的模块名称来声明一个模块时,新声明的定义会取代并扩展旧的声明。 如果旧的声明中的变量在新的声明中没有对应的变量,那么旧的变量继续存在, 但是它的绑定不包括在模块主体的 lexical information 中。 如果一个新的变量定义在旧的声明中有对应的内容,它将有效地分配给旧的变量。

如果一个模块在重新声明之前在当前命名空间的 base phase 被 instantiate , 那么模块的重新声明会立即在该 phase 被 instantiate 。

如果当前 inspector 不管理一个模块的声明检查器(见 Code Inspectors), 那么该模块不能被重新声明。 同样地,一个 cross-phase persistent 模块也不能被重新声明。 即使重新声明成功,如果为重新声明而进行的实例化试图修改常量变量,那么对先前实例化的模块的实例化可能会失败(参见 compile-enforce-module-constants)。

1.1.9.5 子模块

一个顶层模块中的 module 中的 module 或 module* 形式声明了一个 submodule。 一个子模块的访问是相对于它所包围的模块而言的,通常使用 submod 路径。 子模块可以被嵌套到任何深度。

尽管一个子模块在词法上嵌套在一个模块中,但它不一定能直接访问其包围模块的绑定。 更具体地说,一个用 module 声明的子模块不能从它的包围模块中 require , 但包围模块可以 require 子模块。 相反,一个用 module* 声明的子模块在概念上跟随它的包围模块, 所以可以从它的包围模块中 require ,但包围模块不能 require 子模块。 除非一个子模块从它的包围模块导入,或者反过来,那么这两个模块的 visits 或 instantiations 是独立的,它们的实现甚至可以在不同时间从字节码源加载。

一个用 module 声明的子模块可以导入前面任何用 module 声明的子模块。 一个用 module* 声明的子模块可以导入任何前面用 module* 声明的模块和任何用 module 声明的子模块。

当一个子模块声明的形式是 (module* name #f ....), 那么所有包围模块的主体的绑定在子模块的主体中是可见的,并且子模块隐含地导入包围模块。 子模块可以 provide 它从其包围模块继承的任何绑定。

1.1.10 延续框架和标记

参见 Continuation Marks 关于延续标记的形式和函数。

每个延续 C 都可以被划分为 continuation frames C1, C2, ..., Cn ,这样 C = C1[C2[...[Cn]]], 并且没有框架 Ci 本身可以被划分为更小的延续。 评估步骤将帧添加到当前延续中,并从当前延续中移除帧,通常是一次一个。

每个帧在概念上都有一组 continuation marks 的注释。 一个标记由一个键和它的值组成。 键是一个任意的值,每一帧最多包括一个给定键的标记。 各种操作从延续中设置和提取标记,因此,标记可以用来在 dynamic extent 内附加信息。 例如,标记可以用来记录异常发生时呈现的 "堆栈跟踪" 的信息,或者用来实现动态范围。

1.1.11 提示、限定的延续和障碍物

参见 Continuations 关于延续和提示函数。

prompt 是一种特殊的延续帧, 它被注释为一个特定的 prompt tag (本质上是一个 continuation mark)。 各种操作允许捕获从红字位置到最近的带有特定提示标记的包围提示的延续帧; 这样的延续有时被称为 delimited continuation。 其他操作允许用捕获的续集扩展当前的续集(特别是 composable continuation)。 还有一些操作中止了对最近的、带有特定标签的括号的计算,或者用另一个括号替换对最近的括号的继续。 当捕获一个限定的延续时,与相关框架相关的标记也被捕获。

continuation barrier 是另一种延续框架,它禁止用另一个延续来替换当前的延续。 具体来说,只有当替换没有引入任何延续障碍时,一个延续才能被另一个延续替换。 因此,延续屏障可以防止 "下跳" 到一个受屏障保护的延续中。 某些操作会自动安装屏障;特别是当调用异常处理程序时,一个延续屏障禁止处理程序的延续捕获超过异常点的延续。

escape continuation 本质上是一个派生概念。 它结合了用于逃逸目的的提示和用于标记收集目的的继续。 顾名思义,逃逸延续只用于中止到捕获点。

1.1.12 线程

关于线程和同步函数,见 Concurrency and Parallelism 。

Racket 支持多 threads 的评估。 线程的运行是并发的,在这个意义上,一个线程可以在没有它的合作下抢占另一个线程, 但线程目前都在同一个处理器上运行(即同一个底层操作系统进程和线程)。

线程是由 thread 等函数明确地创建的。 就评估模型而言,评估中的每一步实际上处理的是多个并发的表达式,每个线程最多一个,而不是一个单一的表达式。 这些表达式都共享相同的对象和顶层变量,因此它们可以通过共享状态进行通信, 而且 sequential consistency [Lamport79] 也得到了保证(即,结果与跨线程的所有评估步骤所施加的一些全局顺序一致)。 大多数评估步骤涉及单个线程中的一个步骤,但某些同步原语要求多个线程在一个步骤中共同进展; 例如,通过一个 channel 一个值,在两个线程中同时进展。

除非另有说明,Racket 提供的所有恒定时间程序和操作都是线程安全的,因为它们是 atomic: 它们作为一个单一的评估步骤发生。 例如, set! 对一个变量的赋值是对所有线程的原子操作,所以没有线程可以看到一个 "半赋值" 的变量。 同样地, vector-set! 也是以原子方式分配给一个向量。 请注意,对 set! 表达式及其子表达式的评估不一定是原子的,因为对子表达式的评估涉及一个单独的评估步骤。 只有赋值动作本身(在对子表达式进行评估以获得一个值之后)是原子的。 同样地,一个过程的应用可能涉及多个非原子的步骤,即使过程本身执行了一个原子动作。

hash-set! 不是原子的,但表受到锁的保护;更多信息请参见 Hash Tables 。 Port 操作通常不是原子的,但它们是线程安全的,即一个线程从输入端口消耗的数据不会同时返回给另一个线程,而且像 port-commit-peeked 和 write-bytes-avail 这样的过程提供了特定的并发性保证。

除了所有线程之间共享的状态外,每个线程都有自己的私有状态,通过 thread cells 进行访问。 一个线程单元类似于一个普通的可变对象,但一个线程单元内的值的变化只有在同一线程中从该单元中提取值时才能看到。 一个线程单元可以 preserved; 当一个新的线程被创建时,创建线程中被保留的线程单元的值将作为被创建线程中的单元的初始值。 对于非保留的线程单元,新线程看到的初始值(在创建线程单元时指定)与所有其他线程相同。

Futures 和 places 提供了不同种类的并发性和并行性,而且它们对共享状态的保证较弱。 (Places 可以通过 make-shared-bytes 等函数共享状态)。 未来或地方的每个评估线程都被限制在与其他一些线程的行为一致,这些线程可能会检查任何共享数据,从未来或地方开始的任何点开始。 在两个未来或两个地方共享状态的情况下,对共享状态的每个读或写操作都对应于虚拟内存级别的读或写操作, 并且这些操作被限制在它们可能被线程观察或影响的顺序上。 然而,Racket 并不强制执行关于可能在虚拟内存级别或更低级别执行的重新排序的额外保证, 除非是明确指定这种保证的操作(例如, box-cas!)。

1.1.13 参数

参见 参数 ,了解参数形式和函数。

在 Racket 中, Parameters 基本上是一个派生的概念; 它们被定义在 continuation marks 和 thread cells 中。 然而,参数也是 "内置" 的,因为一些原始程序会咨询参数值。 例如,原始输出操作的默认输出流是由一个参数指定的。

参数是一种既针对线程又针对延续的设置。 在空的延续中,每个参数对应于一个 preserved thread cell; 相应的 parameter procedure 为当前线程访问和设置线程单元的值。

在非空延续中,一个参数的值是通过一个 parameterization 确定的, 该参数化通过一个延续标记(其键不能直接访问)与最近的包围延续框架相关联。 一个参数化将每个参数映射到一个保留的线程单元,线程单元和当前线程的组合产生了参数的值。 一个 parameter procedure 为其参数设置或访问相关的线程单元。

各种操作,如 parameterize 或 call-with-parameterization 的调用, 将参数化安装到当前续集的框架中。

1.1.14 异常

的形式、函数和类型参见 异常情况 。

Exceptions 在 Racket 中基本上是一个派生的概念;它们被定义在延续、提示和延续标记方面。 然而,异常也是 "内建" 的,这是因为原始形式和程序可以引发异常。

catch 异常的 exception handler 可以通过一个 continuation mark (其键不能直接访问)与一个延续框架相关联。 当一个异常被引发时,当前延续的标记决定了一个 exception handler 链,这些异常处理程序被查询以处理该异常。 未捕获异常的处理程序是通过一个内置 parameter 指定的。

exception handler 的一个潜在动作是中止当前的 continuation , 直到一个具有特 prompt tag 的封闭的 prompt 。 特别是对于未捕获的异常,默认的处理程序会中止到一个特定的标签,而这个标签总是存在的, 因为对于任何新的线程来说,提示被安装在延续的最外层框架中。

1.1.15 保管人

参见 Custodians 的监护人功能。

一个 custodian 管理一个线程、 file-stream ports, TCP 端口, TCP listeners, UDP sockets, byte converters 和 places 的集合。 每当一个线程等被创建,它就被置于 current custodian 的管理之下, 由 current-custodian parameter 决定。

托管人也管理来自 racket/gui/base 的事件空间。

除了根保管人,每个 custodian 本身都由一个 custodian 管理,这样保管人就形成了一个层次结构。 每个由下级托管人管理的对象也由托管人的所有者管理。

当一个 custodian 通过 custodian-shutdown-all 被关闭时, 它会强制并立即关闭它所管理的端口、TCP 连接等,以及终止(或暂停)其线程。 一个被关闭的托管人不能管理新的对象。 在当前的托管人被关闭后,如果一个过程被调用,试图创建一个被管理的资源(例如, open-input-file, thread),那么会引发 exn:fail:contract exception is raised 异常。

一个线程可以有多个管理托管人,用 thread/suspend-to-kill 创建的暂停线程可以有零个托管人。 额外的托管人通过 thread-resume 与一个线程相关联(参见 暂停、恢复和杀戮线程)。 当一个线程有多个监护人时,它不一定会被一个 custodian-shutdown-all 杀死。 相反,关闭的托管人会从线程的管理托管人集合中移除,当线程的管理集合变空时,线程就会被杀死。

一个托管人所管理的值是由托管人半弱化地持有的:对于一个托管人所管理的值, 可以执行一个 will ;此外,通过弱化 hash tables, ephemerons 或 weak box 的弱化引用可以在 Racket 的 BC 实现中被丢弃, 但在 CS 实现中不能。 对于所有的变体,一个托管人只弱引用它的下级托管人; 如果一个下级托管人没有被引用但有自己的下级,那么该托管人可能会被垃圾收集, 这时它的下级会立即从属于被收集的托管人的上级(所有者)托管人。

除了由托管人管理的其他实体外,用 make-custodian-box 创建的 custodian box 会强势保留放在盒子里的值,直到盒子的托管人被关闭。 然而,托管人只是弱保留盒子本身,所以如果没有其他的引用,盒子和它的内容可以被收集。

当Racket在编译时支持每个托管人的内存核算(参见 custodian-memory-accounting-available?), current-memory-use 过程可以报告一个托管人特定的结果。 这个结果决定了从托管人的管理值(尤其是它的线程), 包括它的子托管人的管理值中 reachable 对象占用了多少内存。 如果一个对象可以从两个托管人那里到达,而这两个托管人都不是另一个托管人的祖先, 那么一个对象就会被任意地计入一个或另一个托管人,而且在每次收集之后选择都会改变; 然而,可以从一个托管人和它的子托管人那里到达的对象会被可靠地计入托管人而不是子托管人, 除非托管人只能通过一个子托管人或一个子托管人的线程到达这些对象。 每个托管人核算的可达性不包括弱引用、对其他托管人管理的线程的引用、对其他托管人的引用, 或对其他托管人的托管箱的引用。
