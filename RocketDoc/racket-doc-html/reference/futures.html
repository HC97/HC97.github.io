11.4 Futures

前景并行 in Racket 指南 introduces futures.

 (require racket/future) package: base
The bindings documented in this section are provided by the racket/future and racket libraries, but not racket/base.

目前,Windows、Linux x86/x86_64 和 Mac OS x86/x86_64 默认启用对 future 的并行支持。 要启用对其他平台的支持,在构建 Racket 时使用 --enable-futures 和 configure 。

racket/future future 和 touch 函数提供了对硬件和操作系统所支持的并行性的访问。 与 thread 相比,future 为任意的计算提供了并发性, 但没有并行性, future 为有限的计算提供了并行性。 一个 future 并行地执行它的工作(假设对并行的支持是可用的), 直到它检测到试图执行一个对系统来说过于复杂而无法安全地并行运行的操作。 同样地,如果未来的工作以某种方式依赖于当前的延续,例如引发一个异常,那么它就会被暂停。 当 touch 应用于未来时,一个未来的暂停计算会被恢复。

未来的 "安全" 并行执行意味着系统提供的所有操作必须能够执行合同并产生文件规定的结果。 "安全" 并不排除对程序中可见的可变数据的并发访问。 例如,一个未来的计算可能使用 set! 来修改一个共享变量, 在这种情况下,对该变量的并发赋值可以在其他未来和线程中可见。 此外,关于效果的可见性和排序的保证是由操作系统和硬件决定的–它们很少支持, 例如,为基于 thread 的并发提供的顺序一致性的保证;另见 机器内存顺序。 同时,那些看起来明显安全的操作可能在内部有足够复杂的实现,以至于它们无法并行运行。 参见 Racket 指南》中的 前景并行 。

如果所有允许其创建线程运行的 custodians 都关闭了,那么一个期货永远不会并行运行。 然而,这种期货可以通过调用 touch 来执行。

11.4.1 创建和触摸期货

procedure

(future thunk) → future?

  thunk : (-> any)

procedure

(touch f) → any

  f : future?
future 过程返回一个封装了 thunk 的未来值。 touch 函数强制对给定的 future 中的 thunk 进行评估,并返回 thunk 产生的值。 在 touch 强制评估 thunk 后,产生的值被 future 保留,以代替 thunk , 对 future 的额外 touch 会返回这些值。

在调用 future 和 touch 之间,对于一个给定的 future, 给定的 thunk 可以与其他计算并行地运行,如上所述。

Example:
> (let ([f (future (lambda () (+ 1 2)))])
    (list (+ 3 4) (touch f)))

'(7 3)

procedure

(futures-enabled?) → boolean?

返回当前 Racket 配置中是否启用了对期货的并行支持。

procedure

(current-future) → (or/c #f future?)

返回未来的描述符,其 thunk 的执行是当前的 continuation;也就是说,如果返回一个未来描述符 f , (touch f) 将产生当前 continuation 的结果。 如果一个 future thunk 本身使用 touch 的执行可以被嵌套,在这种情况下,最直接执行的 future 的描述符被返回。 如果当前的续集没有返回到任何一个 future 的 touch ,结果是 #f 。

procedure

(future? v) → boolean?

  v : any/c
如果 v 是一个未来值,返回 #t ,否则返回 #f 。

procedure

(would-be-future thunk) → future?

  thunk : (-> any)
返回一个从未并行运行的 future,但在执行 future 的 thunk 时,会持续记录所有潜在的 "不安全" 操作(即干扰并行执行的操作)。

对于正常的未来,某些情况可能会阻止对不安全操作的记录。 例如,在执行调试级日志时,

(touch (future (lambda ()
                 (printf "hello1")
                 (printf "hello2")
                 (printf "hello3"))))

可能会记录三条信息,每次调用 printf 都有一条。 然而,如果在 future 有机会开始并行运行之前就进行 touch , future 的 thunk 就会以与任何普通 thunk 相同的方式进行评估,并且不会记录不安全操作。 用 would-be-future 替换 future ,可以确保记录对 printf 的所有三次调用。

procedure

(processor-count) → exact-positive-integer?

返回当前机器上可用的并行计算单元(例如,处理器或内核)的数量。

这与 racket/place 提供的绑定相同。

syntax

(for/async (for-clause ...) body-or-break ... body)

syntax

(for*/async (for-clause ...) body-or-break ... body)

就像 for 和 for* 一样, 但是 body 的每一次迭代都是在一个单独的 future 中执行的, 并且这些未来可以以任何顺序被 touch 。

11.4.2 Future Semaphores

procedure

(make-fsemaphore init) → fsemaphore?

  init : exact-nonnegative-integer?
创建并返回一个新的 future semaphore ,其计数器最初设置为 init 。

一个 future semaphore 类似于一个普通的 semaphore, 但是 future-semaphore 操作可以安全地并行进行(以同步并行计算)。 相反,对普通 semaphores 的操作不能安全地并行进行,因此它们会阻止计算的继续并行进行。

小心尝试使用 fsemaphore 来实现锁。 一个 future 可以与其他 future 并行运行,但是一个没有被 Racket 线程要求的 future 可以在任何时候被暂停, 比如在它获得锁之后和释放锁之前。 如果你必须在期货之间共享可变数据,无锁数据结构通常更适合。

procedure

(fsemaphore? v) → boolean?

  v : any/c
如果 v 是一个 future semaphore 值,返回 #t ,否则返回 #f 。

procedure

(fsemaphore-post fsema) → void?

  fsema : fsemaphore?
增加 future semaphore 的内部计数器并返回 #<void> 。

procedure

(fsemaphore-wait fsema) → void?

  fsema : fsemaphore?
锁定,直到 fsema 的内部计数器为非零。 当计数器为非零时,它被递减, fsemaphore-wait 返回 #<void> 。

procedure

(fsemaphore-try-wait? fsema) → boolean?

  fsema : fsemaphore?
像 fsemaphore-wait 一样,但是 fsemaphore-try-wait? 不会阻塞执行。 如果 fsema 的内部计数器为 0, fsemaphore-try-wait? 立即返回 #f ,而不递减计数器。 如果 fsema 的计数器是正数,那么它将被递减并返回 #t 。

procedure

(fsemaphore-count fsema) → exact-nonnegative-integer?

  fsema : fsemaphore?
返回 fsema 的当前内部计数器值。

11.4.3 Future 性能日志

Racket 追踪广泛地使用日志(见 记录) 来报告关于期货如何被评估的信息。 日志输出对于调试使用期货的程序的性能非常有用。

尽管可以直接查看文本日志输出(或通过 trace-futures 在代码中检索), 但使用 future-visualizer 提供的图形分析工具要容易得多。

未来事件是以 'future 为主题进行记录的。 除了它的字符串信息外,每个记录的未来事件都有一个数据值,它是 future-event prefab 结构的一个实例:

(struct future-event (future-id proc-id action time prim-name user-data)
  #:prefab)

future-id 字段是一个精确的整数,用于识别一个未来, 或者当 action 'missing 时,它是 #f 。 future-id 字段对于关联记录的事件特别有用。

proc-id 字段是一个精确的非负整数,用于标识一个并行进程。 进程 0 是 Racket 的主进程,除 future thunks 外的所有表达式都在此进行评估。

time 字段是一个不精确的数字,以与 current-inexact-milliseconds 相同的方式表示时间。

action 字段是一个符号:

  • 'create: 一个未来被创建。

  • 'complete: 一个未来的 thunk 评估成功,所以 touch 将立即为未来产生一个值。

  • 'start-work 和 'end-work: 一个特定的进程开始和结束了对一个特定未来的工作。

  • 'start-0-work: 和 'start-work 一样,但对于一个未来的 thunk,由于某种结构性原因, 不能在 0 以外的进程中启动(例如,thunk 需要太多的本地存储来启动)。

  • 'start-overflow-work: 和 'start-work 一样, 未来 thunk 的工作之前因为内部堆栈溢出而停止。

  • 'sync: 在未来 thunk 的评估中,对一个 "不安全" 的操作进行阻塞(0 以外的进程)或启动交接(0 进程); 该操作必须在 0 进程中运行。

  • 'block: 和 'sync 一样,但是对于必须延迟到未来被 touch 的评估部分, 因为该评估可能依赖于当前的 continuation。

  • 'touch (从不在进程0中): 像 'sync 或 'block 一样,但用于未来thunk中的一个 touch 操作。

  • 'overflow (never in process 0): 像 'sync 或 'block 一样,但用于进程在评估一个未来 thunk 时遇到内部堆栈溢出的情况。

  • 'result 或 'abort: 等待或处理 'sync, 'block 或 'touch ,分别以一个值或一个错误结束。

  • 'suspend (从不在进程 0中): 被 'sync, 'block 或 'touch 阻塞的进程放弃了对一个 future 的评估;其他的进程可能会在以后捡起这个 future。

  • 'touch-pause 和 'touch-resume (从不在进程 0中): 在 touch 中等待一个未来, 其 thunk 正在其他进程中被评估。

  • 'missing: 进程的一个或多个事件在被报告之前由于内部缓冲区的限制而丢失, time-id 字段报告了丢失事件的时间上限;这种事件很罕见。

假设没有 'missing 事件,那么 'start-work, 'start-0-work, 'start-overflow-work 总是与 'end-work 配对; 'sync, 'block 和 'touch 总是与 'result, 'abort 或 'suspend 配对; 'touch-pause 总是与 'touch-resume 配对。

在流程0中,一些事件对可以嵌套在其他事件对中: 'sync, 'block 或 'touch 与 'result 或 'abort 配对; 'touch-pause 与 'touch-resume 配对; 'start-work 与 'end-work 配对。

当处理一个不安全的操作时,进程 0 中的 'block 会产生。 这种类型的事件将在 unsafe-op-name 字段中包含一个符号,是操作的名称。 在所有其它情况下,这个字段包含 #f 。

除非事件发生在进程 0上,并且其 action 是 'block 或 'sync , 否则 prim-name 字段将始终是 #f 。 如果满足这些条件, prim-name 将包含要求未来与运行时线程同步的 Racket 基元的名称(用符号表示)。

根据 action 和 prim-name 字段, user-data 字段可以有很多不同的值:

  • 'touch on process 0: 包含被触摸的未来的整数 ID。

  • 'sync 和 prim-name 是 '|allocate memory|: 请求分配的大小(以字节为单位)。

  • 'sync 和 prim-name 是 'jit_on_demand: 运行时线程正在代表未来的 future-id 执行 JIT 编译。 该字段包含被 JIT 编译的函数的名称(作为一个符号)。

  • 'create: 一个新的未来被创建。该字段包含新创建的未来的整数 ID。
