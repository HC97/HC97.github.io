7.4 推断链接

syntax

(define-unit unit-id
  (import tagged-sig-spec ...)
  (export tagged-sig-spec ...)
  init-depends-decl
  unit-body-expr-or-defn
  ...)
将 unit-id 绑定到一个单元和关于该单元的静态信息。

评估一个由 define-unit 绑定的 unit-id 的引用会产生一个单元, 就像评估一个由 (define id (unit ...)) 绑定的 id 一样。 此外, unit-id 可以被用于 compound-unit/infer。 关于 tagged-sig-spec, init-depends-decl 和 unit-body-expr-or-defn 的信息，参见 unit 。

syntax

(compound-unit/infer
  (import tagged-infer-link-import ...)
  (export tagged-infer-link-export ...)
  (link infer-linkage-decl ...))
 
tagged-infer-link-import = tagged-sig-id
  | (link-id : tagged-sig-id)
     
tagged-infer-link-export = (tag id infer-link-export)
  | infer-link-export
     
infer-link-export = link-id
  | sig-id
     
infer-linkage-decl = 
((link-binding ...) unit-id
                    tagged-link-id ...)
  | unit-id
与 compound-unit 类似。在语法上, compound-unit 和 compound-unit/infer 的区别在于, 链接单元的 unit-expr 被替换为 unit-id , 其中 unit-id 被 define-unit (或我们在本节后面介绍的其他单元绑定形式之一)绑定。 此外,导入可以只命名一个 sig-id 而不本地绑定一个 link-id , 导出可以基于 sig-id 而不是 link-id, link 子句中的声明可以只是一个 unit-id 而没有指定导出或导入。

通过在 import 子句中添加必要的 sig-ids , 用 link-id 替换 export 子句中的 sig-id , 以及完成 link 子句的声明, compound-unit/infer 形式扩展为 compound-unit 。 这种完成是基于与每个 unit-id 相关的静态信息。 当被链接的单位所输出的所有签名彼此不同,并与所有输入的签名不同,以及所有输入的签名不同时,可以推断出链接和输出。 只有当两个签名没有通过 extends 共享祖先时,它们才是 distinct 。

link 声明的长形式可以用来解决含糊不清的问题,即给单元的一些出口命名,并为单元的一些进口提供特定的绑定关系。 如果剩下的部分可以被推断出来,长形式不需要命名一个单元的所有出口或提供一个单元的所有进口。

当一个单元声明初始化依赖时, compound-unit/infer 会检查 link 声明是否与这些依赖一致,如果不一致,它会报告一个语法错误。

像 compound-unit 一样, compound-unit/infer 形式产生一个(复合)单元, 而不静态绑定关于结果单元的导入和导出信息。 也就是说, compound-unit/infer 消耗静态信息,但它不产生静态信息。 另外两种形式, define-compound-unit 和 define-compound-unit/infer , 产生静态信息(前者不消耗静态信息)。

Changed in version 6.1.1.8 of package base: 增加了对 link 子句的静态检查,与声明的初始化依赖关系有关。

syntax

(define-compound-unit id
  (import link-binding ...)
  (export tagged-link-id ...)
  (link linkage-decl ...))
像 compound-unit 一样,但像 define-unit 一样绑定复合单元的静态信息, 包括传播来自链接单元的初始化-依赖信息(在剩余的 inports 上)。

syntax

(define-compound-unit/infer id
  (import link-binding ...)
  (export tagged-infer-link-export ...)
  (link infer-linkage-decl ...))
和 compound-unit/infer 一样,但像 define-compound-unit 一样绑定了复合单元的静态信息。

syntax

(define-unit-binding unit-id
  unit-expr
  (import tagged-sig-spec ...+)
  (export tagged-sig-spec ...+)
  init-depends-decl)
与 define-unit 类似,但单元的实现是由 unit-expr 产生的现有单元决定的。 由 unit-expr 产生的单元的进口和出口必须与声明的进口和出口一致, 否则在评估 define-unit-binding 形式时将引发 exn:fail:contract exception is raised 。

syntax

(invoke-unit/infer unit-spec)

 
unit-spec = unit-id
  | (link link-unit-id ...)
像 invoke-unit 一样,但使用与 unit-id 相关的静态信息来推断哪些导入必须从当前上下文中装配。 如果给定一个包含多个 link-unit-id 的链接形式, 那么这些单元首先通过 define-compound-unit/infer 被链接。

当从当前上下文组装导入时, unit-id 的 lexical information 被用来构造单元导入的签名的词法信息(即通常从签名引用中得到的词法信息)。 更多信息请参见 define-signature 。

syntax

(define-values/invoke-unit/infer
  unit-spec
  maybe-exports
  maybe-results-clause)
(define-values/invoke-unit/infer
  (export tagged-sig-spec ...)
  unit-spec)
 
unit-spec = unit-id
  | (link link-unit-id ...)
     
maybe-exports = 
  | (export tagged-sig-spec ...)
     
maybe-results-clause = 
  | (values result-id ...)
  | (values result-id ... . rest-results-id)
与 define-values/invoke-unit 类似, 但使用与 unit-id 相关的静态信息来推断哪些进口必须从当前上下文中装配, 如果没有 export 子句,哪些出口应该被定义所约束。 如果给定一个包含多个 link-unit-id 的链接形式, 那么这些单元首先通过 define-compound-unit/infer 链接。

与 invoke-unit/infer 类似, unit-id 的 lexical information 被用来构造单元推断进口和推断出口的签名的词法信息(即通常从签名引用中得到的词法信息)。 更多信息见 define-signature 。

如果提供了 maybe-results-clause , 单元体返回的值将以与 define-values/invoke-unit 相同的方式绑定。

为了向后兼容,允许在 unit-spec 之前出现 export 条款 (在这种情况下可以不提供 maybe-results-clause )。 新程序应该首先提供 unit-spec (这与 define-values/invoke-unit 相一致)。

Changed in version 8.8.0.7 of package base: 允许 unit-spec 出现在 maybe-exports 之前,以便与 define-values/invoke-unit 保持一致,并增加了 maybe-results-clause 。
