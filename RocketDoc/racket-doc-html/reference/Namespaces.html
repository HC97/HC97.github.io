14.1 命名空间

关于 namespace 模型的基本信息,请参见 命名空间 。

一个新的 namespace 是由 make-empty-namespace 和 make-base-namespace 这样的程序创建的,它们返回一个一流的命名空间值。 命名空间通过设置 current-namespace参数值来使用, 或者通过向诸如 eval 和 eval-syntax 这样的程序提供命名空间。

procedure

(namespace? v) → boolean?

  v : any/c
如果 v 是一个命名空间的值,返回 #t ,否则返回 #f .

procedure

(make-empty-namespace) → namespace?

创建一个新的 namespace ,这个名字空间是空的, 它的 module registry 只包含一些内部的、预定义的模块的映射,比如 '#%kernel 。 该命名空间的 base phase 与 current namespace 的 base phase 相同。 用 namespace-attach-module 将现有命名空间的模块附加到新的命名空间。

新命名空间与一个新的 root namespace 相关联, 该命名空间的 module registry 与返回的命名空间相同,并且 base phase 为 0。 如果新的 root namespace 和返回的命名空间都是 base phase 为 0, 那么新的根命名空间和返回的命名空间是一样的。

procedure

(make-base-empty-namespace) → namespace?

创建一个新的 namespace ,就像 make-empty-namespace 一样, 但是附加了 racket/base 。 该命名空间的 base phase 与 make-base-empty-namespace 函数创建时的 phase 相同。

procedure

(make-base-namespace) → namespace?

像 make-empty-namespace一样创建一个新的 namespace , 但是附加了 racket/base ,并且 require 进入顶层环境。 该命名空间的 base phase 与创建 make-base-namespace 函数的 phase 相同。

syntax

(define-namespace-anchor id)

将 id 绑定到一个命名空间锚点,可以与 namespace-anchor->empty-namespace 和 namespace-anchor->namespace 一起使用。

这种形式只能在 top-level context 或 module-context 中使用。

procedure

(namespace-anchor? v) → boolean?

  v : any/c
如果 v 是一个命名空间-锚值,则返回 #t ,否则返回 #f 。

procedure

(namespace-anchor->empty-namespace a) → namespace?

  a : namespace-anchor?
返回一个空的名字空间,该名字空间与锚的来源共享一个 module registry 和 root namespace ,其 base phase 是创建锚的 phase 。

如果锚点来自模块上下文中的 define-namespace-anchor 形式,那么源是包含模块实例化的命名空间。 如果锚点来自顶层内容中的 define-namespace-anchor 形式,那么源是锚点定义被评估的名字空间。

procedure

(namespace-anchor->namespace a) → namespace?

  a : namespace-anchor?
返回一个与锚点的源头相对应的名字空间。

如果锚点来自模块上下文中的 define-namespace-anchor 形式,那么结果是锚点阶段中的模块主体的名字空间。 该结果与通过 module->namespace 获得的命名空间是一样的, 而且如果模块还没有可用的话,也同样会使其 available 。

如果锚点是来自顶层内容中的 define-namespace-anchor 形式,那么结果就是锚点定义被评估的名字空间。

parameter

(current-namespace) → namespace?

(current-namespace n) → void?
  n : namespace?
一个确定 current namespace 的 parameter 。

procedure

(namespace-symbol->identifier sym) → identifier?

  sym : symbol?
类似于仅限于符号的 datum->syntax 。 结果标识符的 lexical information 对应于当前命名空间的顶层环境;标识符没有源位置或属性。

procedure

(namespace-base-phase [namespace]) → exact-integer?

  namespace : namespace? = (current-namespace)
返回 namespace 的 base phase 。

procedure

(namespace-module-identifier [where]) → identifier?

  where : (or/c namespace? exact-integer? #f)
   = (current-namespace)
返回一个标识符,如果它是一个命名空间,其绑定在 where 的 base phase 中的 module , 否则在 where phase level 中。

标识符的 lexical information 包括所有出现在完全扩展代码(见 完全展开的程序) 中的句法形式的绑定(在同一 phase level) ， 但使用 identifier-binding 的第二个元素报告的名称作为绑定; lexical information 还可能包括其他绑定。

procedure

(namespace-variable-value sym    
  [use-mapping?    
  failure-thunk    
  namespace]) → any
  sym : symbol?
  use-mapping? : any/c = #t
  failure-thunk : (or/c (-> any) #f) = #f
  namespace : namespace? = (current-namespace)
返回 namespace 中 sym 的值,使用 namespace 的 base phase 。 返回的值取决于 use-mapping?:

  • 如果 use-mapping? 为真(默认), 并且如果 sym 映射到顶层变量或导入的变量(见 命名空间), 那么结果与评估 sym 作为表达式的结果相同。 如果 sym 映射到语法或导入的语法, 那么会调用 failure-thunk 或引发 exn:fail:syntax exception is raised 。 如果 sym 映射到一个未定义的变量或一个未初始化的模块变量, 那么 failure-thunk 将被调用或引发 exn:fail:contract:variable exception is raised 。

  • 如果 use-mapping? 是 #f, 命名空间的语法和导入映射将被忽略。 相反,命名空间中名为 sym 的顶级变量的值会被返回。 如果该变量是未定义的,那么将调用 failure-thunk 或引发 exn:fail:contract:variable exception is raised 。

如果 failure-thunk 不是 #f, namespace-variable-value 将调用 failure-thunk 来产生返回值, 而不是引发 exn:fail:contract:variable or exn:fail:syntax 异常。

procedure

(namespace-set-variable-value! sym    
  v    
  [map?    
  namespace    
  as-constant?]) → void?
  sym : symbol?
  v : any/c
  map? : any/c = #f
  namespace : namespace? = (current-namespace)
  as-constant? : any/c = #f
在 base phase 设置 namespace 顶层环境中的 sym 值, 如果 sym 还没有定义,则定义它。

如果 map? 被提供为 true,那么命名空间的 identifier 映射也会在与 base phase 相对应的 phase level 中被调整(见 命名空间) , 以便 sym 映射到该变量。

如果 as-constant? 为真,那么在 v 被安装为值之后,变量将被变成常量(因此未来的赋值将被拒绝)。

Changed in version 6.90.0.14 of package base: 增加了 as-constant? 参数。

procedure

(namespace-undefine-variable! sym    
  [namespace]) → void?
  sym : symbol?
  namespace : namespace? = (current-namespace)
删除 sym 变量,如果有的话,在其 base phase 的 namespace 的顶层环境中。 命名空间的 identifier 映射(见 命名空间)不受影响。

procedure

(namespace-mapped-symbols [namespace]) → (listof symbol?)

  namespace : namespace? = (current-namespace)
返回对应于 namespace base phase 的 phase level 的 namespace 中被映射到变量、语法和导入的所有符号的列表。

procedure

(namespace-require quoted-raw-require-spec    
  [namespace]) → void?
  quoted-raw-require-spec : any/c
  namespace : namespace? = (current-namespace)
在 namespace 的顶层环境中执行与 quoted-raw-require-spec 相对应的导入, 就像顶层的 #%require 。 quoted-raw-require-spec 参数必须是对应于 #%require 的 raw-require-spec 的数据集,其中包括模块路径,或者它可以是一个 resolved module path 。

引述的 quoted-raw-require-spec 中的模块路径是相对于 current-load-relative-directory 或 current-directory (如果前者是 #f)解析的,即使当前命名空间对应于一个模块体。

Changed in version 6.90.0.16 of package base: 增加了 namespace 的可选参数。

procedure

(namespace-require/copy quoted-raw-require-spec    
  [namespace]) → void?
  quoted-raw-require-spec : any/c
  namespace : namespace? = (current-namespace)
与 namespace-require 类似,用于从模块导出的语法,但在命名空间的 base phase 导出的变量被区别对待: 导出的当前值被复制到 namespace 中的顶级变量。

Changed in version 6.90.0.16 of package base: 增加了 namespace 的可选参数。

procedure

(namespace-require/constant quoted-raw-require-spec    
  [namespace]) → void?
  quoted-raw-require-spec : any/c
  namespace : namespace? = (current-namespace)
和 namespace-require 一样,但对于 namespace base phase 的每个导出变量, 导出的值会被复制到相应的顶层变量中,并使之成为不可变的。 尽管设置了顶层变量,但相应的标识符被绑定为导入。

Changed in version 6.90.0.16 of package base: 增加了 namespace 的可选参数。

procedure

(namespace-require/expansion-time quoted-raw-require-spec 
  [namespace]) 
 → void?
  quoted-raw-require-spec : any/c
  namespace : namespace? = (current-namespace)
和 namespace-require 一样, 但只有模块的转化器部分相对于 namespace 的 base phase 被执行; 也就是说,模块只是被 visit ,而不是被 instantiate (见 模块扩展、阶段和访问)。 如果所需的模块之前没有被实例化,那么该模块的变量仍未被定义。

Changed in version 6.90.0.16 of package base: 增加了 namespace 的可选参数。

procedure

(namespace-attach-module src-namespace    
  modname    
  [dest-namespace]) → void?
  src-namespace : namespace?
  modname : (or module-path? resolved-module-path?)
  dest-namespace : namespace? = (current-namespace)
将 src-namespace 中由 modname 命名的实例化模块(在其 base phase) 附加到 dest-namespace 的 module registry 中。

除了 modname 之外,它所导入的每个模块(直接或间接)也被记录在当前名字空间的 module registry 中, 同一 phase 的实例也被附加到 dest-namespace (而在模块阶段的 visits 和更高或更低阶段的实例不被附加, 甚至也不为按需 visits 而 available )。 在 dest-namespace 中调用模块的检查器与在 src-namespace 中调用的检查器相同。

如果 modname 不是一个符号,则调用当前的模块名称解析器来解析路径,但不加载模块; modname 的解析形式被用作 dest-namespace 中的模块名称。

如果 modname 指的是一个子模块或一个有子模块的模块, 除非该模块是从字节码(即 ".zo" 文件)中独立于同一顶层模块中的子模块加载的, 那么该模块的顶层模块中所有子模块的声明也被附加到 dest-namespace 。

如果 modname 不是指 src-namespace 中的一个 instantiate 模块, 或者如果要附加的任何模块的名称在 dest-namespace 中已经有了不同的声明或同 phase 的实例, 那么就会引发 exn:fail:contract exception is raised 。

如果 src-namespace 和 dest-namespace 不具有相同的 base phase, 那么将引发 exn:fail:contract exception is raised 。

与 namespace-require 不同, namespace-attach-module 不对模块进行 instantiate , 而是将模块实例从源命名空间复制到目标命名空间。

Examples:
> (module food racket/base
    (provide apple)
    (define apple (list "pie")))
> (namespace-require ''food)
> (define ns (current-namespace))
> (parameterize ([current-namespace (make-base-namespace)])
    (namespace-require ''food))

require: unknown module

  module name: 'food

> (parameterize ([current-namespace (make-base-namespace)])
    (namespace-attach-module ns ''food)
    (namespace-require ''food)
    (eq? (eval 'apple) apple))

#t

> (parameterize ([current-namespace (make-base-namespace)])
    (namespace-attach-module-declaration ns ''food)
    (namespace-require ''food)
    (eq? (eval 'apple) apple))

#f

procedure

(namespace-attach-module-declaration src-namespace    
  modname    
  [dest-namespace]) → void?
  src-namespace : namespace?
  modname : module-path?
  dest-namespace : namespace? = (current-namespace)
与 namespace-attach-module 类似, 但 modname 指定的模块只需在 src-namespace 中声明(而不一定是 instantiate), 而模块只需在 dest-namespace 中声明。

procedure

(namespace-unprotect-module inspector    
  modname    
  [namespace]) → void?
  inspector : inspector?
  modname : module-path?
  namespace : namespace? = (current-namespace)
在 namespace 的 module registry 中改变 modname 所引用的模块实例的检查器, 这样它就被当前的代码检查器所控制。 给定的 inspector 必须当前控制着 namespace 的 module registry 中的模块的调用, 否则检查器不会被改变。参见 Code Inspectors 。

procedure

(namespace-module-registry namespace) → any

  namespace : namespace?
返回给定命名空间的 module registry 。这个值只对通过 eq? 识别有用。

procedure

(namespace-call-with-registry-lock namespace    
  thunk) → any
  namespace : namespace?
  thunk : (-> any)
调用 thunk ,同时为命名空间的 module registry 持有一个可重入锁。

命名空间函数不会自动使用注册表锁, 但它可以通过 namespace-call-with-registry-lock 在加载和实例化模块的线程中使用,以避免内部竞赛条件。 对 available 模块的按需 instantiation 也需要锁;见 模块扩展、阶段和访问 。

Added in version 8.1.0.5 of package base.

procedure

(module->namespace mod [src-namespace]) → namespace?

  mod : 
(or/c module-path?
      resolved-module-path?
      module-path-index?)
  src-namespace : namespace? = (current-namespace)
返回一个与 src-namespace 的 module registry 和 src-namespace 的 base phase 中的实例化模块主体相对应的名字空间, 使得该模块 available 在 src-namespace 的 base phase 按需 visits 。 返回的名字空间具有与 src-namespace 相同的 module registry 。 在返回的名字空间中修改一个绑定,会改变在需要该名字空间的模块的模块中看到的绑定。

顶层 require 表达式中的模块路径是根据命名空间的模块来解决的。 新的 provide 声明是不允许的。

如果当前的代码检查器没有控制 src-namespace 的 module registry 中的模块的调用, 则会引发 exn:fail:contract exception is raised ;另请参阅 Code Inspectors 。

如果在声明模块时 compile-enforce-module-constants 参数为 true, 则不能修改结果命名空间中的绑定,除非模块声明本身包括通过 set! 。

Changed in version 6.90.0.16 of package base: 增加了 src-namespace 可选参数。

procedure

(namespace-syntax-introduce stx [namespace]) → syntax?

  stx : syntax?
  namespace : namespace? = (current-namespace)
返回一个像 stx 一样的语法对象, 除了 namespace 的绑定被包含在 syntax object 的 lexical information 中(见 语法对象)。 额外的上下文被 syntax object 的 lexical information 中任何现有的 top-level bindings 所覆盖, 或者被 lexical information中任何现有或未来的 module bindings 所覆盖。

Changed in version 6.90.0.16 of package base: 增加了 namespace 的可选参数。

procedure

(module-provide-protected? module-path-index    
  sym) → boolean?
  module-path-index : (or/c symbol? module-path-index?)
  sym : symbol?
如果 module-path-index 的模块声明定义了 sym 并且不受保护地导出, 则返回 #f ,否则返回 #t (这可能意味着该符号对应的是一个未导出的定义,一个受保护的导出,或者一个在模块中根本没有定义的标识符)。

The module-path-index 参数可以是一个符号;关于模块路径索引的更多信息,见 编译的模块和引用。

通常, module-provide-protected? 的参数对应于由 identifier-binding 产生的列表中的前两个元素。

procedure

(variable-reference? v) → boolean?

  v : any/c
如果 v 是由 #%variable-reference 产生的 variable reference, 返回 #t ,否则返回 #f 。

procedure

(variable-reference-constant? varref) → boolean?

  varref : variable-reference?
如果 varref 所代表的变量将保留其当前值, 则返回 #t (也就是说, varref 指的是一个不能被 set! 或 define 进一步修改的变量),否则返回 #f 。

procedure

(variable-reference->empty-namespace varref) → namespace?

  varref : variable-reference?
返回一个空的名字空间,与 varref 实例化的名字空间共享模块声明和实例, 并且与 varref 具有相同的相位。

procedure

(variable-reference->namespace varref) → namespace?

  varref : variable-reference?
如果 varref 指的是一个 module-level variable, 那么结果是在被引用的变量的 phase 中为模块的主体提供一个命名空间; 结果与通过 module->namespace 获得的命名空间相同,如果模块还没有 available 的话,同样也会被提供。

如果 varref 指的是 top-level variable, 那么结果就是被引用变量所定义的名字空间。

procedure

(variable-reference->resolved-module-path varref)

 → (or/c resolved-module-path? #f)
  varref : variable-reference?
如果 varref 指的是一个 module-level variable, 那么结果是一个命名该模块的 resolved module path 。

如果 varref 指的是一个 top-level variable, 那么结果是 #f 。

procedure

(variable-reference->module-path-index varref)

 → (or/c module-path-index? #f)
  varref : variable-reference?
如果 varref 指的是一个 module-level variable, 那么结果是一个命名该模块的 module path index 。

如果 varref 指的是一个 top-level variable, 那么结果是 #f 。

procedure

(variable-reference->module-source varref)

 → (or/c symbol? (and/c path? complete-path?) #f)
  varref : variable-reference?
如果 varref 指的是一个 module-level variable, 那么结果是一个命名模块源码的路径或符号(通常,但不总是,与 resolved module path 相同)。 如果相关的模块是一个 submodule, 那么结果对应于包围的顶层模块的源代码。

如果 varref 指的是一个 top-level variable, 那么结果是 #f 。

procedure

(variable-reference->phase varref) → exact-nonnegative-integer?

  varref : variable-reference?
返回 varref 所引用的变量的 phase 。

procedure

(variable-reference->module-base-phase varref) → exact-integer?

  varref : variable-reference?
返回 varref 引用的变量的模块实例化 phase , 如果 varref 的变量不在一个模块内,则返回 0 。

对于一个有模块的变量,当该变量在模块内被绑定在 phase level n 时,其结果比 (variable-reference->phase varref) 的结果少 n 。

procedure

(variable-reference->module-declaration-inspector varref)

 → inspector?
  varref : variable-reference?
返回 varref 的模块的声明 inspector (见 Code Inspectors), 其中 varref 必须是指由 (#%variable-reference) 产生的匿名模块变量。

procedure

(variable-reference-from-unsafe? varref) → boolean?

  varref : variable-reference?
如果变量引用的模块本身(不一定是被引用的变量)是以不安全模式编译的, 则返回 #t ,否则返回 #f 。 Unsafe mode 可以通过 linklet 接口启用,或者通过 (#%declare #:unsafe) 为一个模块启用。

variable-reference-from-unsafe? 程序是用来作为

(variable-reference-from-unsafe? (#%variable-reference))

编译器可以将其优化为字面意义上的 #t 或 #f (因为包围的模块正在以 unsafe mode 进行编译)。

Added in version 6.12.0.4 of package base.
