8.6 将合约附加到值上

syntax

(contract-out unprotected-submodule contract-out-item ...)

 
unprotected-submodule = 
  | #:unprotected-submodule submodule-name
     
contract-out-item = 
(struct id/ignored ((id contract-expr) ...)
  struct-option)
  | (rename orig-id id contract-expr)
  | (id contract-expr)
  | #:∃ poly-variables
  | #:exists poly-variables
  | #:∀ poly-variables
  | #:forall poly-variables
     
poly-variables = id
  | (id ...)
     
id/ignored = id
  | (id ignored-id)
     
struct-option = 
  | #:omit-constructor
一个用于 provide 的 provide-spec (目前只适用于与 provide 形式相同的 phase level ; 例如, contract-out 不能嵌套在 for-syntax 中) 。 每个 id 是由模块提供的。 此外,模块的客户必须遵守 contract-expr 为每个出口指定的合同。

contract-out 形式将模块作为责备的单位。 定义所提供的变量的模块被期望满足契约的正(共变)位。 每个导入所提供的变量的模块必须服从合同的负数(反变量)位置。 在 contract-out 形式中的每个 contract-expr 都被有效地移到包围模块的末尾, 所以 contract-expr 可以引用在同一模块中后来定义的变量。

只有在模块外使用契约变量才会被检查。 在模块内部,不发生契约检查。

contract-out 的 rename 形式将第一个变量(内部名称)与第二个变量(外部名称)所指定的名称导出。

contract-out 的 struct 形式提供了一个结构类型的定义 id , 每个字段都有一个合同,决定了字段的内容。 然而,与 struct 定义不同,所有的字段(和它们的契约)都必须被列出。 子结构与其父结构共享的字段的契约只用于子结构构造函数的契约,而超级结构的选择器或突变器没有被提供。 输出的结构类型名称总是作为一个构造函数,即使原始结构类型名称不作为一个构造函数。 如果 #:omit-constructor 选项存在,则不提供构造函数。 id/ignored 的第二种形式,即同时拥有 id 和 ignored-id, 已被弃用,仅在语法中允许向后兼容,其中 ignored-id 被忽略。应该使用第一种形式来代替。

注意,如果结构是用 serializable-struct 或 define-serializable-struct 创建的, contract-out 不会保护通过 deserialize 创建的结构实例。 请考虑使用 struct-guard/c 来代替。

#:∃, #:exists, #:∀ 和 #:forall 条款定义了新的抽象契约。 变量在 contract-out 形式的其余部分被绑定到新的合约中, 这些合约隐藏了它们接受的值,并确保导出的函数被参数化处理。 关于子句如何隐藏数值的细节,见 new-∃/c 和 new-∀/c 。

如果 #:unprotected-submodule 出现,它后面的标识符会被用作 contract-out 生成的子模块的名字。 该子模块输出 contract-out 中的所有名字,但没有合同。

contract-out 的实现使用语法属性（syntax-property )来给它生成的代码附加属性, 这些属性记录了完全展开的程序中的合同的语法。 具体来说,符号 'provide/contract-original-contract 被绑定到两个元素的向量上,即导出的标识符和产生控制导出的合同的表达式的语法对象。

Examples:
> (module math-example racket/base
    (require racket/contract)
    ; Compute the reciprocal of a real number
    (define (recip x) (/ 1 x))
    (provide
     (contract-out
      [recip (-> (and/c real? (not/c zero?)) real?)])))
> (require 'math-example)
> (recip 3)

1/3

> (recip 1+2i)

  given: 1+2i

  in: an and/c case of

      the 1st argument of

      (-> (and/c real? (not/c zero?)) real?)

  contract from: math-example

  blaming: top-level

   (assuming the contract is correct)

syntax

(recontract-out id ...)

一个 provide-spec ,用于 provide (目前,就像 contract-out 一样,只是用于与 provide 形式相同的 phase level )。

它重新输出 id, 但与包含 recontract-out 的模块相关的正责, 而不是 id 的原始站点的位置。

当一个公共模块想从一个私有模块导出一个标识符, 但任何违反合同的行为都应该以公共模块而不是私有模块的方式报告时,这可能很有用。

Examples:
> (module private-implementation racket/base
    (require racket/contract)
    (define (recip x) (/ 1 x))
    (define (non-zero? x) (not (= x 0)))
    (provide/contract [recip (-> (and/c real? non-zero?)
                                 (between/c -1 1))]))
> (module public racket/base
    (require racket/contract
             'private-implementation)
    (provide (recontract-out recip)))
> (require 'public)
> (recip +nan.0)

recip: broke its own contract

  promised: (between/c -1 1)

  produced: +nan.0

  in: the range of

      (->

       (and/c real? non-zero?)

       (between/c -1 1))

  contract from: public

  blaming: public

   (assuming the contract is correct)

  at: eval:3:0

用 recip 代替 recontract-out 的使用,会导致违反合同,责备私人模块。

syntax

(provide/contract unprotected-submodule contract-out-item ...)

(provide (contract-out unprotected-submodule contract-out-item ...)) 的传统速记, 只是 provide/contract 中的 contract-expr 在 provide/contract 形式的位置被评估,而不是在包围模块的末端。

8.6.1 嵌套合同的边界

 (require racket/contract/region) package: base

syntax

(with-contract blame-id (wc-export ...) free-var-list ... body ...+)

(with-contract blame-id results-spec free-var-list ... body ...+)
 
wc-export = (id contract-expr)
     
result-spec = #:result contract-expr
  | #:results (contract-expr ...)
     
free-var-list = 
  | #:freevar id contract-expr
  | #:freevars ([id contract-expr] ...)
生成一个局部契约边界。

第一个 with-contract 表单不能出现在表达式位置。 所有在第一个 with-contract 的表单中定义的名字在外部是可见的, 但是那些在 wc-export 列表中列出的名字被相应的契约所保护。 如果其上下文允许的话,表单的 body 允许定义/表达交错。

第二个 with-contract 表单必须出现在表达式位置。 最后的 body 表达式应该返回与 result-spec 中列出的契约数量相同的值, 并且每个返回的值都与它各自的契约相联系。 body 形式的序列被当作 let 来处理。

blame-id 用于与导出的 id 配对的合同的正位置。 在 with-contract body 中被破坏的合约将使用 blame-id 作为它们的负面位置。

如果给出了 free-var-list ,那么在 body 中对自由变量的任何使用都将受到合同的保护, 这些合同的正向位置指责 with-contract 形式的上下文, 而负向位置指责 with-contract 形式的上下文。

syntax

(define/contract id contract-expr free-var-list init-value-expr)

(define/contract (head args) contract-expr free-var-list body ...+)
工作原理与 define 类似,只是合同中的 contract-expr 被附加到绑定的值上。 关于 head 和 args 的定义,见 define 。 关于 free-var-list 的定义,见 with-contract 。

Examples:
> (define/contract distance (>=/c 0) 43.52)
> (define/contract (furlongs->feet fr)
    (-> real? real?)
    (* 660 fr))
; a contract violation expected here:
> (furlongs->feet "not a furlong")

furlongs->feet: contract violation

  expected: real?

  given: "not a furlong"

  in: the 1st argument of

      (-> real? real?)

  contract from: (function furlongs->feet)

  blaming: top-level

   (assuming the contract is correct)

  at: eval:3:0

define/contract 形式将单个定义视为一个契约区域。 定义本身负责契约的正向(共变体)位置,对定义之外的 id 的引用必须满足契约的负向位置。 由于契约的边界是在定义和周围的环境之间, 对 define/contract 形式内的 id 的引用不被检查。

Examples:
; an unsual predicate that prints when called
> (define (printing-int? x)
    (displayln "I was called")
    (exact-integer? x))
> (define/contract (fact n)
    (-> printing-int? printing-int?)
    (if (zero? n)
        1
        (* n (fact (sub1 n)))))
> (fact 5) ; only prints twice, not for each recursive call

I was called

I was called

120

如果给出了一个自由变量列表,那么在 body 内部对自由变量的任何使用都将受到合约的保护, 这些合约在正向位置上指责 define/contract 形式的上下文, 在反向位置上指责 define/contract 形式。

Examples:
> (define (integer->binary-string n)
    (number->string n 2))
> (define/contract (numbers->strings lst)
    (-> (listof number?) (listof string?))
    #:freevar integer->binary-string (-> exact-integer? string?)
    ; mistake, lst might contain inexact numbers
    (map integer->binary-string lst))
> (numbers->strings '(4.0 3.3 5.8))

integer->binary-string: contract violation

  expected: exact-integer?

  given: 4.0

  in: the 1st argument of

      (-> exact-integer? string?)

  contract from: top-level

  blaming: (function numbers->strings)

   (assuming the contract is correct)

  at: eval:3:0

syntax

(invariant-assertion invariant-expr expr)

建立 expr 的不变性,由 invariant-expr 决定。

与合同的规范不同, invariant-assertion 并没有在双方之间建立一个边界。 相反,它只是将一个逻辑断言附加到值上。 因为表单使用合同机制来检查断言,周围的模块被视为违反断言的责任方。

这意味着,例如,在递归调用中检查断言,当定义的右手边使用一个不变式时:

Examples:
> (define furlongss->feets

     (-> (listof real?) (listof real?))

              (furlongss->feets (list "wha?"))

8.6.2 低级别的契约边界

syntax

(define-module-boundary-contract id
  orig-id
  contract-expr
  pos-blame-party
  source-loc
  name-for-blame
  context-limit)
 
pos-blame-party = 
  | #:pos-source pos-source-expr
     
source-loc = 
  | #:srcloc srcloc-expr
     
name-for-blame = 
  | #:name-for-blame blame-id
     
context-limit = 
  | #:context-limit limit-expr
将 id 定义为 orig-id ,但要有 contract-expr 。

标识符 id 被定义为一个宏转换器,它查询其使用的上下文, 以确定负责分配的名称(使用整个模块,其中引用出现的是负方)。

正方默认为包含使用 define-module-boundary-contract 的模块, 但可以通过 #:pos-source 关键字明确指定。

在责备错误信息中使用的源位置是指合同被放在值上的位置, 默认为使用 define-module-boundary-contract 的源位置, 但可以通过 #:srcloc 参数指定,在这种情况下, 它可以是 datum->syntax 的第三个参数可以是任何东西。

在错误信息中使用的名称将是 orig-id, 除非 #:name-for-blame 被提供, 在这种情况下,它后面的标识符将被用作错误信息中的名称。

如果提供了 #:context-limit ,它的行为与提供给 contract 时的行为相同。

Examples:
> (module server racket/base
    (require racket/contract/base)
    (define (f x) #f)
    (define-module-boundary-contract g f (-> integer? integer?))
    (provide g))
> (module client racket/base
    (require 'server)
    (define (clients-fault) (g #f))
    (define (servers-fault) (g 1))
    (provide servers-fault clients-fault))
> (require 'client)
> (clients-fault)

g: contract violation

  expected: integer?

  given: #f

  in: the 1st argument of

      (-> integer? integer?)

  contract from: 'server

  blaming: client

   (assuming the contract is correct)

  at: eval:2:0

> (servers-fault)

g: broke its own contract

  promised: integer?

  produced: #f

  in: the range of

      (-> integer? integer?)

  contract from: 'server

  blaming: (quote server)

   (assuming the contract is correct)

  at: eval:2:0

Changed in version 6.90.0.29: 增加 #:context-limit 参数。

syntax

(contract contract-expr to-protect-expr
          positive-blame-expr negative-blame-expr)
(contract contract-expr to-protect-expr
          positive-blame-expr negative-blame-expr
          #:context-limit limit-expr)
(contract contract-expr to-protect-expr
          positive-blame-expr negative-blame-expr
          value-name-expr source-location-expr)
将一个契约附加到一个值上的原始机制。 契约的目的是作为一些更高级别的 contract 指定形式的扩展目标。

contract 表达式把由 contract-expr 指定的契约添加到由 to-protect-expr 产生的值上。 contract 表达式的结果是 to-protect-expr 表达式的结果, 但是在 to-protect-expr 上强制执行了 contract-expr 指定的契约。

positive-blame-expr 和 negative-blame-expr 的值表示如何分配由 contract-expr 指定的契约的正负位置的责任。 它们可以是任何值,并且为了违反合同的错误信息,它们的格式是由 display 决定的。

如果指定, value-name-expr 表示在错误信息中使用的保护值的名称。 如果没有提供,或者 value-name-expr 产生 #f, 则不打印名称。 否则,它也会像 display 那样被格式化。 更准确地说, value-name-expr 最后会出现在责备记录的 blame-value 字段中, 它被用作错误信息的第一部分。

Examples:
> (contract integer? #f 'pos 'neg 'timothy #f)

timothy: broke its own contract

  promised: integer?

  produced: #f

  in: integer?

  contract from: pos

  blaming: pos

   (assuming the contract is correct)

> (contract integer? #f 'pos 'neg #f #f)

broke its own contract

  promised: integer?

  produced: #f

  in: integer?

  contract from: pos

  blaming: pos

   (assuming the contract is correct)

如果指定的话, source-location-expr 表示违反合同所报告的源位置。 该表达式必须产生一个 srcloc 结构、 syntax object, #f, 或者是 datum->syntax 的第三个参数所接受的格式的列表或向量。

如果提供了 #:context-limit ,下面的表达式必须评估为 #f 或一个自然数。 如果表达式的值是一个自然数,那么上下文信息的层数最多限制在这个数。 例如,如果该数字为 0, 则不记录上下文信息,错误信息也不包含以 in: 开头的部分。
