12.3 语法对象绑定

procedure

(bound-identifier=? a-id b-id [phase-level]) → boolean?

  a-id : syntax?
  b-id : syntax?
  phase-level : (or/c exact-integer? #f)
   = (syntax-local-phase-level)
如果标识符 a-id 能够绑定 b-id (或者反之),则返回 #t , 如果标识符在适当的表达式上下文中被替换为由 phase level 表示的 phase-level , 则返回 #f 。 phase-level 的 #f 值与 label phase level 相对应。

Examples:
> (define-syntax (check stx)
    (syntax-case stx ()
      [(_ x y)
       (if (bound-identifier=? #'x #'y)
           #'(let ([y 'wrong]) (let ([x 'binds]) y))
           #'(let ([y 'no-binds]) (let ([x 'wrong]) y)))]))
> (check a a)

'binds

> (check a b)

'no-binds

> (define-syntax-rule (check-a x) (check a x))
> (check-a a)

'no-binds

procedure

(free-identifier=? a-id    
  b-id    
  [a-phase-level    
  b-phase-level]) → boolean?
  a-id : identifier?
  b-id : identifier?
  a-phase-level : (or/c exact-integer? #f)
   = (syntax-local-phase-level)
  b-phase-level : (or/c exact-integer? #f) = a-phase-level
如果 a-id 和 b-id 分别在 a-phase-level 和 b-phase-level 指示的 phase levels 上访问相同的 local binding, module binding 或 top-level binding —也许是通过 rename transformers—则返回 #t 。 a-phase-level 或 b-phase-level 的 #f 值与 label phase level 相对应。

“Same module binding” 意味着标识符指的是相同的原始定义站点,而不一定是指相同的 require 或 provide 站点。 由于在 require 和 provide 中的重命名,或者由于转化器与 rename transformer 的绑定, 标识符可能会以syntax-e 返回不同的结果。

Examples:
> (define-syntax (check stx)
    (syntax-case stx ()
      [(_ x)
       (if (free-identifier=? #'car #'x)
           #'(list 'same: x)
           #'(list 'different: x))]))
> (check car)

'(same: #<procedure:car>)

> (check mcar)

'(different: #<procedure:mcar>)

> (let ([car list])
    (check car))

'(different: #<procedure:list>)

> (require (rename-in racket/base [car kar]))
> (check kar)

'(same: #<procedure:car>)

procedure

(check-duplicate-identifier ids) → (or/c identifier? #f)

  ids : (listof identifier?)
用 bound-identifier=? 比较 ids 中的每个标识符和列表中的每个其他标识符。 如果任何比较的结果是 #t, 则返回其中一个重复的标识符( ids 中第一个重复的),否则结果是 #f 。

procedure

(identifier-binding id-stx 
  [phase-level 
  top-level-symbol? 
  exact-scopes?]) 
 → 
(or/c 'lexical
      #f
      (list/c module-path-index?
              symbol?
              module-path-index?
              symbol?
              exact-nonnegative-integer?
              phase+space-shift?
              phase+space?)
      (list/c symbol?))
  id-stx : identifier?
  phase-level : (or/c exact-integer? #f)
   = (syntax-local-phase-level)
  top-level-symbol? : any/c = #f
  exact-scopes? : any/c = #f
返回三种(如果 top-level-symbol? 是 #f )或四种 (如果 top-level-symbol? 是true) 值中的一种, 取决于id-stx 在 phase-level 指示的 phase level 的绑定情况 (其中 phase-level 的 #f 值对应于 label phase level):

  • 如果 id-stx 有一个 local binding,那么结果就是 'lexical 。

  • 如果 id-stx 有 module binding ,结果是一个七项的列表: (list from-mod from-sym nominal-from-mod nominal-from-sym from-phase import-phase+space-shift nominal-export-phase) 。

    • from-mod 是一个模块路径索引(见 编译的模块和引用) ,表示定义的模块。 如果绑定指的是 id-stx 的封闭模块中的定义,它就是 "自我 "模块路径索引。

    • from-sym 是一个符号,表示标识符在源模块中定义位置的名称。 这可能与 syntax->datum 返回的本地名称不同,原因有几个:标识符在导入时被重命名,在导出时被重命名, 或者因为绑定位置是由宏调用生成的,所以被隐式重命名。 在最后一种情况下,它可能是一个不可读的符号,而且它可能与原始源定义中标识符的 syntax->datum 结果不同。

    • nominal-from-mod 是一个模块路径索引(见 编译的模块和引用) , 表示绑定的模块,因为它在 id-stx 周围的源中出现: 它表示 id-stx 的上下文中 require 一个模块来提供它的绑定, 或者它是与 from-mod 相同的 "自我",用于指称 id-stx 的包围模块中的定义的一个绑定。 它可以与 from-mod 不同,因为它在 nominal-from-mod 中重新输出了一些导入的标识符。 如果同一个标识符以多种方式被导入,则会选择一个任意的代表。

    • nominal-from-sym 是绑定的标识符的一个符号,它出现在 id-stx 周围的源中: 它是由 nominal-from-mod 输出的标识符的名字, 或者它是 id-stx 包围模块中定义的源标识符的符号。 由于重命名的 provide,它可以与 from-sym 不同, 即使 from-mod 和 nominal-from-mod 是相同的,或者由于一个定义是由宏扩展引入的。

    • from-phase 是一个精确的非负整数,代表起始阶段。 例如,如果定义是for-syntax,它就是 1 。

    • import-phase+space-shift 如果 nominal-from-mode 的绑定导入是来自定义或普通 require,则为 0 ; 如果是来自 for-syntax 的导入,则为 1 ; 如果是来自 for-space 的导入,则为与空格名称相结合的阶段,等等。

    • nominal-export-phase+space 是指从 nominal-from-mod 导入的绑定的 phase level 和 binding space , 或者是 id-stx 的包围模块中绑定的定义的阶段级别。

  • 如果 id-stx 有一个 top-level binding , 并且 top-level-symbol? 为真,那么结果是 (list top-sym) 。 当绑定定义是由宏调用生成时, top-sym 可能与 syntax->datum 返回的名称不同。

  • 如果 id-stx 有 top-level binding , 并且 top-level-symbol? 是 #f , 或者 id-stx unbound ,结果是 #f 。 一个未绑定的标识符通常和一个顶级绑定为变量的标识符的处理方式相同。

如果 id-stx 被绑定到一个 rename-transformer 上, 那么 identifier-binding 的结果就是变换器中的标识符, 所以 identifier-binding 与 free-identifier=? 一致。

如果 exact-scopes? 是一个真值,那么结果是 #f , 除非对 id-stx 的绑定完全具有 id-stx 的 scopes 。 确切范围的检查对于检测一个标识符是否已经被绑定在一个特定的定义上下文中很有用,例如。

procedure

(identifier-distinct-binding id-stx 
  wrt-id-stx 
  [phase-level]) 
 → 
(or/c 'lexical
      #f
      (list/c module-path-index?
              symbol?
              module-path-index?
              symbol?
              exact-nonnegative-integer?
              phase+space-shift?
              phase+space?)
      (list/c symbol?))
  id-stx : identifier?
  wrt-id-stx : identifier?
  phase-level : (or/c exact-integer? #f)
   = (syntax-local-phase-level)
就像 (identifier-binding id-stx phase-level), 但是如果 id-stx 的绑定范围是 wrt-id-stx 的范围的子集,那么结果就是 #f 。 也就是说,如果 id-stx 和 wrt-id-stx 有相同的符号名称, 只有当绑定的 id-stx 不同时适用于 wrt-id-stx 时才会返回。

Added in version 8.3.0.8 of package base.

procedure

(identifier-binding-symbol id-stx    
  [phase-level]) → symbol?
  id-stx : identifier?
  phase-level : (or/c exact-integer? #f)
   = (syntax-local-phase-level)
像 identifier-binding 一样,但产生一个与绑定相对应的符号。 符号结果对于任何 free-identifier=? 的标识符都是一样的, 但是对于不是 free-identifier=? 的标识符, 结果也可能是一样的(也就是说,不同的符号意味着不同的绑定,但是相同的符号并不意味着相同的绑定)。

当 identifier-binding 会产生一个列表, 那么这个列表的第二个元素就是 identifier-binding-symbol 产生的结果。

procedure

(identifier-binding-portal-syntax id-stx 
  [phase-level]) 
 → (or/c #f syntax?)
  id-stx : identifier?
  phase-level : (or/c exact-integer? #f)
   = (syntax-local-phase-level)
如果 id-stx 在 phase-level 被绑定到 portal syntax, 无论是通过 define-syntax 还是 #%require ,那么将返回 portal 语法内容。 绑定 id-stx 的模块必须被声明,但它不需要在相关阶段被实例化, 而且 identifier-binding-portal-syntax 不会实例化该模块。

Added in version 8.3.0.8 of package base.

procedure

(syntax-bound-symbols stx    
  [phase-level    
  exact-scopes?]) → (listof symbol?)
  stx : stx?
  phase-level : (or/c exact-integer? #f)
   = (syntax-local-phase-level)
  exact-scopes? : any/c = #f
返回一个所有 interned 符号的列表,其中 (identifier-binding (datum->syntax stx sym) phase-level #f exact-scopes?) 会产生一个非 #f 值。 这个过程需要的时间与 stx 上的作用域数量和结果列表的长度成正比。

Added in version 8.6.0.6 of package base.
Changed in version 8.6.0.9: 增加了 exact-scopes? 参数。

procedure

(syntax-bound-phases stx) → (listof (or/c exact-integer? #f))

  stx : stx?
返回一个列表,其中包括 (syntax-bound-symbols stx phase-level) 可能产生非空列表的所有 phase-level 。

Examples:
> (syntax-bound-phases #'anything)

'(2 1 0)

> (require (for-meta 8 racket/base))
> (syntax-bound-phases #'anything)

'(9 8 2 1 0)

Added in version 8.6.0.8 of package base.
