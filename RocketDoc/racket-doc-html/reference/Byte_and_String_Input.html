13.2 Byte and String Input

procedure

(read-char [in]) → (or/c char? eof-object?)

  in : input-port? = (current-input-port)
从 in 中读取一个字符–这可能涉及到读取几个字节来对它们进行 UTF-8 解码(见 端口); 为了执行解码,需要读取/peek 最小数量的字节。 如果在文件结束前没有可用的字节,那么将返回 eof 。

Examples:
> (let ([ip (open-input-string "S2")])
    (print (read-char ip))
    (newline)
    (print (read-char ip))
    (newline)
    (print (read-char ip)))

#\S

#\2

#<eof>

> (let ([ip (open-input-bytes #"\316\273")])
    ; The byte string contains UTF-8-encoded content:
    (print (read-char ip)))

#\λ

procedure

(read-byte [in]) → (or/c byte? eof-object?)

  in : input-port? = (current-input-port)
从 in 中读取一个字节。如果在文件结束前没有可用的字节,则返回 eof 。

Examples:
> (let ([ip (open-input-string "a")])
    ; The two values in the following list should be the same.
    (list (read-byte ip) (char->integer #\a)))

'(97 97)

> (let ([ip (open-input-string (string #\λ))])
    ; This string has a two byte-encoding.
    (list (read-byte ip) (read-byte ip) (read-byte ip)))

'(206 187 #<eof>)

procedure

(read-line [in mode]) → (or/c string? eof-object?)

  in : input-port? = (current-input-port)
  mode : (or/c 'linefeed 'return 'return-linefeed 'any 'any-one)
   = 'linefeed
返回一个包含来自 in 的下一行字节的字符串。

从 in 中读取字符,直到读到行分隔符或文件结束。 分行符不包括在结果字符串中(但它会从端口的数据流中被删除)。 如果在遇到文件结束前没有读到任何字符,则返回 eof 。

mode 参数决定了行分隔符。 它必须是下列符号之一:

  • 'linefeed 在换行字符上断行。

  • 'return 在回车字符上断行。

  • 'return-linefeed 在 return-linefeed 组合中分行。 如果一个回车符后面没有换行符,它将被包含在结果字符串中;同样,一个没有回车符的换行符将被包含在结果字符串中。

  • 'any 在任何一个回车符、换行符或回车-换行符组合上断行。 如果回车符后面是换行符,则这两个字符被视为一个组合。

  • 'any-one 在回车或换行字符上断行,不识别回车-换行的组合。

回车和换行字符是在以文本模式读取文件时自动执行的转换之后检测出来的。 例如,在 Windows 上以文本模式读取文件,会自动将回车-换行组合改为换行。 因此,当以文本模式打开一个文件时, 'linefeed 通常是合适的 read-line 模式。

Examples:
> (let ([ip (open-input-string "x\ny\n")])
    (read-line ip))

"x"

> (let ([ip (open-input-string "x\ny\n")])
    (read-line ip 'return))

"x\ny\n"

> (let ([ip (open-input-string "x\ry\r")])
    (read-line ip 'return))

"x"

> (let ([ip (open-input-string "x\r\ny\r\n")])
    (read-line ip 'return-linefeed))

"x"

> (let ([ip (open-input-string "x\r\ny\nz")])
    (list (read-line ip 'any) (read-line ip 'any)))

'("x" "y")

> (let ([ip (open-input-string "x\r\ny\nz")])
    (list (read-line ip 'any-one) (read-line ip 'any-one)))

'("x" "")

procedure

(read-bytes-line [in mode]) → (or/c bytes? eof-object?)

  in : input-port? = (current-input-port)
  mode : (or/c 'linefeed 'return 'return-linefeed 'any 'any-one)
   = 'linefeed
与 read-line 类似,但读取的是字节,并产生一个字节的字符串。

procedure

(read-string amt [in]) → (or/c string? eof-object?)

  amt : exact-nonnegative-integer?
  in : input-port? = (current-input-port)

要把整个端口读成一个字符串,使用 port->string 。

返回一个包含从 in 开始的下一个 amt 字符的字符串。

如果 amt 为 0, 则返回空字符串。 否则,如果在遇到文件结束前,可用的字符少于 amt ,那么返回的字符串将只包含文件结束前的那些字符; 也就是说,返回的字符串的长度将小于 amt 。 (在读取输入时,会分配一个 amt 大小的临时字符串,即使结果的大小小于 amt 字符)。 如果在文件结束前没有可用的字符,那么将返回 eof 。

如果在读取过程中发生错误,一些字符可能会丢失; 也就是说,如果 read-string 在遇到错误之前成功地读取了一些字符,这些字符会被丢弃。

Example:
> (let ([ip (open-input-string "supercalifragilisticexpialidocious")])
    (read-string 5 ip))

"super"

procedure

(read-bytes amt [in]) → (or/c bytes? eof-object?)

  amt : exact-nonnegative-integer?
  in : input-port? = (current-input-port)

要以字节形式读取整个端口,使用 port->bytes 。

像 read-string 一样,但是读取字节并产生一个字节字符串。

Example:
> (let ([ip (open-input-bytes
                    (bytes 6
                           115 101 99 114 101
                           116))])
    (define length (read-byte ip))
    (bytes->string/utf-8 (read-bytes length ip)))

"secret"

procedure

(read-string! str [in start-pos end-pos])

 → (or/c exact-nonnegative-integer? eof-object?)
  str : (and/c string? (not/c immutable?))
  in : input-port? = (current-input-port)
  start-pos : exact-nonnegative-integer? = 0
  end-pos : exact-nonnegative-integer? = (string-length str)
像 read-string 一样从 in 读取字符,但是把它们放到 str 中, 从索引 start-pos (包括) 开始到 end-pos (不包括)。 像 substring 一样,如果 start-pos 或 end-pos 超出了 str 的范围, 就会引发 exn:fail:contract exception is raised 。

如果 start-pos 和 end-pos 之间的差值是 0, 那么返回 0 ， str 不被修改。 如果在文件结束前没有可用的字节,那么返回 eof 。否则,返回值是读取的字符数。 如果读取了 m 个字符,并且 m<end-pos-start-pos, 那么 str 在 start-pos+m 到 end-pos 的索引处不会被修改。

Example:
> (let ([buffer (make-string 10 #\_)]
        [ip (open-input-string "cketRa")])
    (printf "~s\n" buffer)
    (read-string! buffer ip 2 6)
    (printf "~s\n" buffer)
    (read-string! buffer ip 0 2)
    (printf "~s\n" buffer))

"__________"

"__cket____"

"Racket____"

procedure

(read-bytes! bstr [in start-pos end-pos])

 → (or/c exact-nonnegative-integer? eof-object?)
  bstr : bytes?
  in : input-port? = (current-input-port)
  start-pos : exact-nonnegative-integer? = 0
  end-pos : exact-nonnegative-integer? = (bytes-length bstr)
像 read-string! 一样,但是读取字节,把它们放入一个字节串,并返回读取的字节数。

Example:
> (let ([buffer (make-bytes 10 (char->integer #\_))]
        [ip (open-input-string "cketRa")])
    (printf "~s\n" buffer)
    (read-bytes! buffer ip 2 6)
    (printf "~s\n" buffer)
    (read-bytes! buffer ip 0 2)
    (printf "~s\n" buffer))

#"__________"

#"__cket____"

#"Racket____"

procedure

(read-bytes-avail! bstr [in start-pos end-pos])

 → (or/c exact-nonnegative-integer? eof-object? procedure?)
  bstr : bytes?
  in : input-port? = (current-input-port)
  start-pos : exact-nonnegative-integer? = 0
  end-pos : exact-nonnegative-integer? = (bytes-length bstr)
和 read-bytes! 一样,但是在读完立即可用的字节后返回,没有阻塞,而且它可以返回一个 "特殊" 结果的程序。 read-bytes-avail! 过程只在没有字节(或特殊)可用时才阻塞。 与 read-bytes! 不同的是, read-bytes-avail! 从来不会丢弃字节; 如果 read-bytes-avail! 成功地读取了一些字节,然后遇到了一个错误, 它会抑制这个错误(大致上像对待一个文件结束一样)并返回所读取的字节。 (该错误将在未来的读取中被触发)。 如果在读取任何字节之前就遇到了错误,就会引发一个异常。

当 in 产生一个特殊的值时,如 Custom Ports 所述,其结果是一个有四个参数的过程。 这四个参数对应于特殊值在端口中的位置,如 Custom Ports 所述。 如果这个过程在有效参数的情况下被调用了一次以上, 就会引发 exn:fail:contract exception is raised 异常。 如果 read-bytes-avail! 返回一个产生特殊值的过程,那么它就不会在 bstr 中放置字符。 类似地, read-bytes-avail! 只将端口流中的特殊值之前的可用字节数放入 bstr 。

procedure

(peek-string amt skip-bytes-amt [in]) → (or/c string? eof-object?)

  amt : exact-nonnegative-integer?
  skip-bytes-amt : exact-nonnegative-integer?
  in : input-port? = (current-input-port)
与 read-string 类似,除了返回的字符是被 peek : 保留在端口中供将来的读取和 peeks 。 (更确切地说,未编码的字节是为将来的读取和偷看留下的)。 skip-bytes-amt 参数指出了在收集要返回的字符之前要跳过的输入流中的字节数 (而 不是 字符数); 因此, 总的来说, 下一个 skip-bytes-amt 字节加上 amt 字符会被检查。

对于大多数类型的端口来说,检查 skip-bytes-amt 字节和 amt 字符至少需要与端口相关的 skip-bytes-amt+amt 字节的内存开销,至少在字节/字符被读取之前是如此。 当偷看字符串端口 (参见 字符串端口)、管道端口 (参见 管道) 或具有特定偷看程序的自定义端口 (取决于偷看程序的实现方式;参见 Custom Ports) 时,不需要这种开销。

如果一个端口在数据流中产生了 eof , 那么试图跳过 eof 进行 peek 的尝试总是产生 eof ,直到 eof 被读取。

(peek-char-or-special [in

 → (or/c char? eof-object? any/c)
  in : input-port? = (current-input-port)
  skip-bytes-amt : exact-nonnegative-integer? = 0

和 peek-char 一样,但是如果输入端口在 skip-bytes-amt 字节位置之后返回一个非字节值, 那么结果就取决于 special-wrap:

  • 如果 special-wrap 是 #f ，那么就会返回特殊值(和 read-char-or-special一样)。

  • 如果 special-wrap 是一个程序,那么它将应用特殊值来产生结果(与 read-char-or-special一样)。

  • 如果 special-wrap 是 'special, 那么 'special 将代替 special 值返回–不需要调用输入口实现返回的 special 值过程。

procedure

(port-progress-evt [in]) → progress-evt?

  in : (and/c input-port? port-provides-progress-evts?)
   = (current-input-port)
返回一个 synchronizable event (见 事件) , 该事件在任何后续的从 in 读取或 in 关闭后成为 ready for synchronization 。 在事件成为准备好后,它将保持准备好。 The synchronization result of a progress event is the progress event itself.

procedure

(port-provides-progress-evts? in) → boolean

  in : input-port?
如果 port-progress-evt 能够返回 in 的事件,则返回 #t 。 所有内置的端口都支持进度事件,但用 make-input-port 创建的端口(见 Custom Ports) 可能不支持。

procedure

(port-commit-peeked amt progress evt [in]) → boolean?

  amt : exact-nonnegative-integer?
  progress : progress-evt?
  evt : evt?
  in : input-port? = (current-input-port)
试图 commit 读取先前 peek 的第一个 amt 字节、非字节的特殊值和来自 in 的 eof ,或者从in 偷看的第一个 eof 或特殊值。 流中的 eof 可以被提交,但是当端口耗尽时的 eof 不一定会提交,因为它不对应于流中的数据。

只有在 progress 没有先准备好的情况下(即没有其他进程先从 in 中读取), 并且只有在 evt 被 port-commit-peeked 中的 sync 选择时(在这种情况下,事件结果被忽略), 读取才会提交; evt 必须是 channel-put 事件、channel、semaphore、semaphore-peek 事件、 always 事件或者 never 事件。 暂停调用 port-commit-peeked 的线程可能会也可能不会阻止提交的进行。

如果数据被提交, port-commit-peeked 的结果是 #t ,否则是 #f 。

如果没有数据从 in 中被偷看,并且 progress 还没有准备好, 那么会引发 exn:fail:contract exception is raised 。 如果在 in 的数据流的当前开始位置,被偷看的项目少于 amt ,那么只有被偷看的项目被提交为读。 如果 in 的数据流当前开始于一个 eof 或一个非字节的特殊值, 那么只有 eof 或特殊值被提交为读。

如果 progress 不是应用于 in 的 port-progress-evt 的结果, 那么会引发 exn:fail:contract exception is raised 。

procedure

(byte-ready? [in]) → boolean?

  in : input-port? = (current-input-port)
如果 (read-byte in) 不会阻塞(至少在 byte-ready? 被调用时),则返回 #t 。 相当于 (and (sync/timeout 0 in) #t) 。

byte-ready? 和 char-ready? 函数适合于相对较少的应用, 因为端口的目的是支持并发的生产者和消费者之间的数据流;一个字节或字符在某个瞬间没有准备好, 并不一定意味着生产者已经完成了数据供应。 (另外,如果一个端口有多个消费者,那么在某个进程使用 byte-ready? 轮询端口和它从端口读取数据之间,数据可能会被消耗掉)。 如果你正在实现你自己的调度器,或者你知道端口的实现和使用特别受限制,那么使用 byte-ready? 是有意义的。

procedure

(char-ready? [in]) → boolean?

  in : input-port? = (current-input-port)
如果 (read-char in) 不会阻塞(至少在调用 char-ready? 的时候),返回 #t 。 根据流的初始字节数,可能需要多个字节来形成 UTF-8 编码。

参见 byte-ready? ,了解 byte-ready? 和 char-ready? 很少是正确的选择。

procedure

(progress-evt? v) → boolean?

  v : any/c
(progress-evt? evt in) → boolean?

  in : input-port?

有两个参数,如果 evt 是 in 的一个进度事件,则返回 #t ,否则返回 #f 。
