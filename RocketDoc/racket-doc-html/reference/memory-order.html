11.7 机器内存顺序

与 Racket threads 不同,futures 和 place 可以暴露出底层机器的内存模型,包括弱的内存排序。 例如,当一个期货写到一个可变向量中的多个槽时,在某些平台上, 另一个期货有可能以不同的顺序或根本不观察这些写,除非这些期货是明确同步的。 同样,共享的字节串或 fxvectors 可以在不同地方暴露机器的内存模型。

Racket 确保机器的内存模型不会以不安全的方式暴露出原始数据类型的实现。 例如,一个 future 不可能因为读取一个被另一个 future 突变的向量而看到一个部分构建的基元值。

box-cas!, vector-cas!, unsafe-box*-cas!, unsafe-vector*-cas! 和 unsafe-struct*-cas! 操作都提供了机器级的比较和设置,所以它们可以被机器的内存模型特别支持的方式使用。 (memory-order-acquire) 和 (memory-order-release) 操作也同样限制了机器级的存储和加载。 同步操作,如放置消息、 future touch 和 future semaphores 等,意味着合适的机器级获取和释放顺序。

procedure

(memory-order-acquire) → void?

procedure

(memory-order-release) → void?

这些操作在需要同步的平台上实现了一个机器级内存围栏。 memory-order-acquire 操作确保至少在机器层面上有一个 load–load 和 load–store 栅栏, memory-order-release 操作确保至少在机器级别上有一个 store–store 和 store–load 栅栏。

Added in version 7.7.0.11 of package base.
