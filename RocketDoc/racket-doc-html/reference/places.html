11.5 Places

现场（place）并行 in Racket 指南 introduces places.

 (require racket/place) package: base
 (require racket/place/dynamic)
The bindings documented in this section are provided by the racket/place, racket/place/dynamic, and racket libraries, but not racket/base.

Places 可以开发并行程序,利用机器的多处理器、核心或硬件线程。

目前,在所有支持 Racket CS (Racket 的默认实现)的平台上都启用了地方的并行支持。 3m 的实现在 Windows、Linux x86/x86_64 和 Mac OS x86/x86_64 上也默认支持地方的并行执行。 要启用 3m 对其他平台的支持,在构建 Racket 时使用 --enable-places 和 configure 。 place-enabled? 函数报告地方是否并行运行。

实现和操作系统的限制可能会限制地方的可扩展性。 例如,尽管在 CS 的实现中,地方可以并行地进行垃圾回收,在 3m 的实现中也可以独立地进行垃圾回收, 但垃圾回收可能需要操作一个所有地方都共享的页表, 而这个共享的页表在有足够多的地方时可能会成为一个瓶颈–也许是8或16个左右。

一个 place 是一个并行任务,它实际上是 Racket 虚拟机的一个独立实例,尽管所有地方都在一个操作系统进程中运行。 位置通过 place channels 进行通信,它是双向缓冲通信的端点。

初步估计,地方通道只支持不可变的、透明的值作为消息。 此外,地方信道本身可以跨信道发送,以便在任何现有的线路之外建立新的(可能更直接的)通信线路。 最后,由 shared-flvector, make-shared-flvector, shared-fxvector, make-shared-fxvector, shared-bytes 和 make-shared-bytes 生的可变值可以跨地方通道发送; 这种值的变异对所有共享该值的地方都是可见的,因为它们在 shared memory space 中是允许的。 参见 place-message-allowed? 。

一个 place channel 可以作为一个 synchronizable event (见 事件) ,通过该通道接收一个值。 当通道上有消息可用时,一个 place channel 就 ready for synchronization , 而这个 place channel 的 synchronization result 就是消息(在同步时被移除)。 一个地方也可以用 place-channel-get 来接收消息,并且可以用 place-channel-put 来发送消息。

如果两个 place channels 是相同的底层通道的端点,同时都是或都不是一个 place descriptor, 那么它们就是 equal? 。 在通过 Place channels 发送消息后,place channel 可以是 equal? 而不是 eq? 。

对跨地点通道的消息的限制–因此对地点共享的数据种类的限制–可以比 future 有更大的并行性,甚至包括单独的 garbage collection 的独立地点。 同时,地方的设置和通信成本可能比 futures 更高。

例如,下面的表达式启动了两个地方,向每个地方发送一个消息,然后等待这些地方终止:

(let ([pls (for/list ([i (in-range 2)])
              (dynamic-place "place-worker.rkt" 'place-main))])
   (for ([i (in-range 2)]
         [p pls])
      (place-channel-put p i)
      (printf "~a\n" (place-channel-get p)))
   (map place-wait pls))

"place-worker.rkt" 模块(在一个与上述代码分开的文件中)必须导出每个地方执行的 place-main 函数, 其中 place-main 必须接受一个 place channel 参数:

"place-worker.rkt"

#lang racket
(provide place-main)
 
(define (place-main pch)
  (place-channel-put pch (format "Hello from place ~a"
                                  (place-channel-get pch))))

场所通道和其他 Racket 值一样,会受到 garbage collection的影响, 如果 place channel 的写入端变得不可达, 那么从 place channel 中读取信息被阻塞的 thread 就会被垃圾收集。 然而, 与普通的 channel 阻塞不同, 如果本来不可达的线程在只有同一线程可达的地方通道上相互阻塞,那么这些线程和地方通道都被认为是可达的,而不是不可达的。

当一个地方被创建时,它的 parameter 值通常被设置为创建地方的参数的 initial 值, 除了使用以下参数的 current 值: current-library-collection-paths, current-library-collection-links 和 current-compiled-file-roots 。

一个新创建的地方会被注册到 current custodian

11.5.1 Using Places

procedure

(place-enabled?) → boolean?

如果Racket的配置使 dynamic-place 和 place 创建的地方可以并行运行,则返回 #t ； 如果 dynamic-place 和 place 是用 thread 模拟的,则返回 #f 。

procedure

(place? v) → boolean?

  v : any/c
如果 v 是一个 place descriptor 的值,返回 #t ,否则返回 #f 。 每个 place descriptor 也是一个 place channel 。

procedure

(place-channel? v) → boolean?

  v : any/c
如果 v 是 place channel,则返回 #t ,否则返回 #f 。

procedure

(dynamic-place module-path    
  start-name    
  [#:at location    
  #:named named]) → place?
  module-path : (or/c module-path? path?)
  start-name : symbol?
  location : (or/c #f place-location?) = #f
  named : any/c = #f
创建一个运行程序的 place ,该程序由 module-path 和 start-name 标识。 结果是一个表示新的并行任务的 place descriptor 值;地方描述符会立即返回。 地方描述符值也是一个 place channel ,允许与该地方通信。

由 module-path 指示的模块必须输出一个名称为 start-name 的函数。 该函数必须接受一个参数,它是一个 place channel , 对应于由 place 返回的 place descriptor 的通信的另一端。

如果提供了 location ,它必须是一个 place location , 比如由 create-place-node 产生的分布式地方节点。

当 place 被创建时,初始的 exit handler 会终止这个地方, 使用退出处理程序的参数作为 completion value 。 使用 (exit v) 来立即终止一个具有完成值 v 的地方。 由于完成值被限制为 0 和 255 之间的精确整数,任何其他的 v 值都被转换为 0 。

如果 module-path 和 start-name 指示的函数返回, 那么这个地方就以 completion value 0 终止。

在创建的地方, current-input-port 参数被设置为一个空的输入端口, 而 current-output-port 和 current-error-port 参数的值被连接到创建地方的当前端口。 如果创建地的输出端口是 file-stream ports, 那么创建地的连接端口共享底层流, 否则创建地的一个 thread 将字节从创建地的端口抽到创建地的当前端口。

创建地的大多数 parameters 都有其初始值,但是创建地继承了创建地的以下参数值: current-directory, current-library-collection-paths, current-library-collection-links 和 current-compiled-file-roots 。

module-path 参数不能是 (quote sym) 形式的模块路径, 除非该模块是预定义的(见 module-predefined?)。

dynamic-place 绑定在 protect-out 的意义上是受保护的, 所以可以通过调整代码检查器来阻止对这个操作的访问(见 Code Inspectors)。

Changed in version 8.2.0.7 of package base: 改变了创建的地方,以继承创建地方的 current-directory 值。

procedure

(dynamic-place* module-path    
  start-name    
  [#:in in    
  #:out out    
  #:err err]) → 
place?
(or/c output-port? #f)
(or/c input-port? #f)
(or/c input-port? #f)
  module-path : (or/c module-path? path?)
  start-name : symbol?
  in : (or/c input-port? #f) = #f
  out : (or/c output-port? #f) = (current-output-port)
  err : (or/c output-port? #f) = (current-error-port)
和 dynamic-place 一样,但是接受特定的端口到新地方的端口,当为一个端口提供 #f 时返回一个创建的端口。 in, out 和 err 端口分别连接到 place 的 current-input-port, current-output-port 和 current-error-port 。 任何一个端口都可以是 #f ,在这种情况下, 一个 file-stream port (用于操作系统的管道)被创建并由 dynamic-place* 返回。 err 参数可以是 'stdout, 在这种情况下, 相同的 file-stream port 或作为标准输出的端口也被用于标准错误。 对于每个提供的端口或 'stdout ,不创建管道,相应的返回值是 #f 。

dynamic-place* 的调用者负责关闭所有返回的端口;没有一个是自动关闭的。

dynamic-place* 过程返回四个值:

  • 一个代表所创建位置的位置描述符值;

  • 一个输出端口,它被输送到该地方的标准输入,如果 in 是一个端口,则为 #f ;

  • 一个从该位置的标准输出管道的输入端口,如果 out 是一个端口,则为 #f ;

  • 从地方的标准错误中输送的输入端口,如果 err 是一个端口或 'stdout,则为 #f 。

dynamic-place* 绑定的保护方式与 dynamic-place 相同。

procedure

(place-wait p) → exact-integer?

  p : place?
返回 p 指示的地方的 completion value ,阻塞直到该地方结束。

如果创建了任何抽水线程来连接非 file-stream port 到 p 所指的地方的端口 (见 dynamic-place), place-wait 只在抽水线程完成后返回。

procedure

(place-dead-evt p) → evt?

  p : place?
返回一个 synchronizable event (见 事件) , 当且仅当 p 已经终止时,该事件是 ready for synchronization 。 The synchronization result of a place-dead event is the place-dead event itself.

如果创建了任何抽水线程以将非 file-stream port 连接到 p 的地方的端口(见 dynamic-place), 即使抽水线程仍在运行, place-dead-evt 返回的事件也可能成为就绪。

procedure

(place-kill p) → void?

  p : place?
立即终止该地方,如果该地方还没有完成值,则将该地方的 completion value 为 1 。

procedure

(place-break p [kind]) → void?

  p : place?
  kind : (or/c #f 'hang-up 'terminate) = #f
给地方 p 的主线程发送一个中断;见 中断 。

procedure

(place-channel) → 
place-channel? place-channel?
返回两个 place channels 。 通过第一个通道发送的数据可以通过第二个通道接收,而通过第二个通道发送的数据可以从第一个通道接收。

通常,一个地方通道被当前的 place 用来发送消息到目的地 place; 另一个地方通道被发送到目的地 place (通过一个现有的 place channel)。

procedure

(place-channel-put pch v) → void

  pch : place-channel?
  v : place-message-allowed?
在通道 pch 上发送一条消息 v 。 由于place channel是异步的, place-channel-put 的调用是无阻塞的。

参见 place-message-allowed? 表格中关于 v 的自动强制信息, 例如将一个可变的字符串转换为不可变的字符串。

procedure

(place-channel-get pch) → place-message-allowed?

  pch : place-channel?
返回在通道 pch 上收到的消息,阻塞直到有消息可用。

procedure

(place-channel-put/get pch v) → any/c

  pch : place-channel?
  v : any/c
在通道 pch 上发送一个不可变的消息 v ,然后在同一通道上等待一个消息(也许是一个回复)。

procedure

(place-message-allowed? v) → boolean?

  v : any/c
如果 v 被允许作为一个地方通道上的消息,返回 #t ,否则返回 #f 。

如果 (place-enabled?) 返回 #f, 那么结果总是 #t , 并且不对 v 作为消息进行转换。 否则,以下种类的数据都可以作为消息:

  • numbers, characters, booleans, keywords 和 #<void>;

  • symbols, 其中 uninterned 符号的 eq? 在单个消息中被保存,但在不同的消息中不被保存;

  • strings 和 byte strings, 其中易变的字符串和字节串被自动替换成不可变的变体;

  • paths (用于任何平台);

  • pairs, lists, box, vectors 和包含消息允许值的不可变的 prefab 结构,其中可变的盒被自动替换为不可变的盒, 可变的向量被自动替换为不可变的向量,盒、向量和 prefab 结构的 impersonators 被复制;

  • hash tables 可变的哈希表被自动替换为不可变的变体,哈希表的 impersonator 被复制;

  • place channels, place descriptor 被自动替换为普通场所通道;

  • file-stream ports 和 TCP ports, 其中底层表示(如文件描述符、套接字或句柄)在发送地被复制,在接收地被连接到一个新的端口;

  • 通过 ffi/unsafe 创建或访问的 C pointers ;以及

  • shared-flvector, make-shared-flvector, shared-fxvector, make-shared-fxvector, shared-bytes 和 make-shared-bytes 产生的值。

Changed in version 8.4.0.7 of package base: 在允许的信息中包含了方框。

value

prop:place-location : struct-type-property?

procedure

(place-location? v) → boolean?

  v : any/c
一个 structure type property 和相关的谓词,用于实现 place locations 。 prop:place-location 的值必须是由四个参数组成的过程: place location 本身、模块路径、模块导出的启动函数的符号和地点名称(对于匿名地点可以是 #f )。

一个 place location 可以作为 #:at 参数传递给 dynamic-place , 后者则简单地调用 place location 的 prop:place-location 值。

用 create-place-node 创建的分布式地点注释是 place location 的一个例子。

11.5.2 使用地点的句法支持

本节中的绑定 不 是由 racket/place/dynamic 提供的。

syntax

(place id body ...+)

创建一个地方,评估与地方通道绑定的 id 的 body 表达式。 body 只在 id 和包围模块的顶级绑定上关闭,因为 body 被提升到一个子模块。 place 的结果是一个地方描述符,就像 dynamic-place 的结果。

生成的子模块的名字是 place-body-n, 代表一个整数 n, 子模块输出了一个 main 函数,为新的位置获取一个位置通道。 然而,这个子模块并不打算使用,除了通过 place 形式的扩展。

place 绑定的保护方式与 dynamic-place 相同。

syntax

(place* maybe-port ...
        id
        body ...+)
 
maybe-port = 
  | #:in in-expr
  | #:out out-expr
  | #:err err-expr
和 place 一样,但支持可选的 #:in, #:out 和 #:err 表达式(每种最多一个), 以相同的方式和默认值指定端口,与 dynamic-place* 相同。 place* 形式的结果也与 dynamic-place* 相同。

place* 绑定的保护方式与 dynamic-place 相同。

syntax

(place/context id body ...+)

和 place 一样,但是 body ... 可以有自由词性变量,这些变量会被自动发送到新创建的place。 注意,这些变量的值必须被 place-message-allowed? 所接受,否则会引发 exn:fail:contract exception is raised 。

procedure

(processor-count) → exact-positive-integer?

返回当前机器上可用的并行计算单元(例如,处理器或内核)的数量。

这与 racket/future 的绑定方式相同。

11.5.3 位置记录

地点事件被报告给一个名为 'place 的日志器。 除了它的字符串信息外,每个地方记录的事件都有一个数据值,是一个 place-event prefab 结构的实例:

(struct place-event (place-id action value time)
  #:prefab)

place-id 字段是一个精确的整数,用于识别一个地方。

time 字段是一个不精确的数字,以与 current-inexact-milliseconds 相同的方式表示时间。

action 字段是一个符号:

  • 'create: 一个地方被创建。 这个事件被记录在创建的地方,事件的 value 字段有创建地方的 ID。

  • 'reap: 一个之前在当前地方创建的地方已经退出了 (并且这个事实已经被检测到,可能是通过 place-wait)。 该事件的 value 字段有退出的地方的 ID。

  • 'enter: 一个地方已经开始了,在已开始的地方记录。 该事件的 value 字段有 #f 。

  • 'exit: 一个地方正在退出,记录在退出的地方。 该事件的 value 字段有 #f 。

  • 'put: 一个地方频道的信息已经被发送。 事件的 value 字段是一个正的精确的整数,近似于消息的大小。

  • 'get: 一个地方频道的消息已经被接收。 该事件的 value 字段是一个正的精确的整数,近似于消息的大小。

Changed in version 6.0.0.2 of package base: 增加了通过 'place 增加了通过 place-event 的日志记录。
