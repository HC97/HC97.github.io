8.2 Function Contracts

一个 function contract 包裹着一个过程,以延迟对其参数和结果的检查。 有三种主要的函数契约组合,它们的表现力越来越强,额外的开销也越来越大。 第一个 -> 是最便宜的。 它生成的包装函数可以直接调用原始函数。 用 ->* 构建的合约需要将参数打包成包装函数中的列表, 然后使用 keyword-apply 或 apply 。 最后, ->i 是最昂贵的(和 ->d 一起), 因为它需要将域和范围的契约表达式的评估延迟到函数本身被调用或返回。

case-> 契约是一个专门的契约,旨在匹配 case-lambda 和 unconstrained-domain-> 允许范围检查而不要求域有任何特定的形状(见下面的使用实例)。

syntax

(-> dom ... range)

(-> dom ... ellipsis dom-expr ... range)
 
dom = dom-expr
  | keyword dom-expr
     
range = range-expr
  | (values range-expr ...)
  | any
     
ellipsis = ...
产生一个函数的契约,该函数接受 dom-expr 契约指定的参数, 并返回固定数量的结果或完全未指定的结果(当指定了 any 时为后者)。

每个 dom-expr 是一个函数参数的契约,每个 range-expr 是一个函数结果的契约。

如果域包含 ... ,那么该函数接受与域部分指定的其余合约一样多的参数, 以及任意多的与刚刚在 ... 的合约相匹配的参数。 否则,合约接受的就是指定的参数。

在两个以空格分隔的 . 之间使用 -> , 与把 -> 放在封闭的开括号之后是一样的。 更多信息请参见 列表和 Racket 语法 和 读取成对和列表 。

例如,

(integer? boolean? . -> . integer?)

产生一个关于两个参数的函数的契约。 第一个参数必须是一个整数,第二个参数必须是一个布尔值。 该函数必须产生一个整数。

Examples:
> (define/contract (maybe-invert i b)
    (-> integer? boolean? integer?)
    (if b (- i) i))
> (maybe-invert 1 #t)

-1

> (maybe-invert #f 1)

maybe-invert: contract violation

  expected: integer?

  given: #f

  in: the 1st argument of

      (-> integer? boolean? integer?)

  contract from: (function maybe-invert)

  blaming: top-level

   (assuming the contract is correct)

  at: eval:2:0

一个领域规范可能包括一个关键词。 如果是这样,该函数必须接受相应的(强制性的)关键字参数,而且关键字参数的值必须与相应的契约相匹配。 比如说:

(integer? #:invert? boolean? . -> . integer?)

是一个函数的契约,该函数接受一个是整数的 by-position 参数和一个是布尔值的 #:invert? 参数。

Examples:
> (define/contract (maybe-invert i #:invert? b)
    (-> integer? #:invert? boolean? integer?)
    (if b (- i) i))
> (maybe-invert 1 #:invert? #t)

-1

> (maybe-invert 1 #f)

maybe-invert: arity mismatch;

 the expected number of arguments does not match the given

number

  expected: 1 plus an argument with keyword #:invert?

  given: 2

  arguments...:

   1

   #f

作为一个使用 ... 的例子,这个合同:

(integer? string? ... integer? . -> . any)

在一个函数上坚持认为该函数的第一个和最后一个参数必须是整数(而且必须至少有两个参数),任何其他参数必须是字符串。

Examples:
> (define/contract (string-length/between? lower-bound s1 . more-args)
    (-> integer? string? ... integer? boolean?)
  
    (define all-but-first-arg-backwards (reverse (cons s1 more-args)))
    (define upper-bound (first all-but-first-arg-backwards))
    (define strings (rest all-but-first-arg-backwards))
    (define strings-length
      (for/sum ([str (in-list strings)])
        (string-length str)))
    (<= lower-bound strings-length upper-bound))
> (string-length/between? 4 "farmer" "john" 40)

#t

> (string-length/between? 4 "farmer" 'john 40)

string-length/between?: contract violation

  expected: string?

  given: 'john

  in: the repeated argument of

      (-> integer? string? ... integer? boolean?)

  contract from:

      (function string-length/between?)

  blaming: top-level

   (assuming the contract is correct)

  at: eval:2:0

> (string-length/between? 4 "farmer" "john" "fourty")

string-length/between?: contract violation

  expected: integer?

  given: "fourty"

  in: the last argument of

      (-> integer? string? ... integer? boolean?)

  contract from:

      (function string-length/between?)

  blaming: top-level

   (assuming the contract is correct)

  at: eval:2:0

如果 any 被用作 -> 的最后一个子形式,则不会对函数的结果进行契约检查, 因此任何数量的值都是合法的(甚至在函数的不同调用中的不同数量)。

Examples:
> (define/contract (multiple-xs n x)
    (-> natural? any/c any)
    (apply
     values
     (for/list ([_ (in-range n)])
       n)))
> (multiple-xs 4 "four")

4

4

4

4

如果 (values range-expr ...) 被用作 -> 的最后一个子形式, 该函数必须为每个契约产生一个结果,并且每个值必须与各自的契约相匹配。

Examples:
> (define/contract (multiple-xs n x)
    (-> natural? any/c (values any/c any/c any/c))
    (apply
     values
     (for/list ([_ (in-range n)])
       n)))
> (multiple-xs 3 "three")

3

3

3

> (multiple-xs 4 "four")

multiple-xs: broke its own contract;

 expected 3 values, returned 4 values

  in: the range of

      (->

       natural?

       any/c

       (values any/c any/c any/c))

  contract from: (function multiple-xs)

  blaming: (function multiple-xs)

   (assuming the contract is correct)

  at: eval:2:0

syntax

(->* (mandatory-dom ...) optional-doms rest pre range post)

 
mandatory-dom = dom-expr
  | keyword dom-expr
     
optional-doms = 
  | (optional-dom ...)
     
optional-dom = dom-expr
  | keyword dom-expr
     
rest = 
  | #:rest rest-expr
     
pre = 
  | #:pre pre-cond-expr
  | #:pre/desc pre-cond-expr
     
range = range-expr
  | (values range-expr ...)
  | any
     
post = 
  | #:post post-cond-expr
  | #:post/desc post-cond-expr
->* 契约组合器为接受可选的参数(无论是关键字还是位置)和/或任意多的参数的函数产生契约。 一个 ->* 合约的第一个子句描述了必须的参数,与 -> 合约的参数描述相似。 第二个子句描述了可选参数。 描述的范围可以是 any ,也可以是一连串的契约,表示函数必须返回多个值。

如果存在, rest-expr 契约将管理 rest 参数中的参数。 注意, rest-expr 契约只管理rest参数中的参数,而不是强制性参数中的参数。 例如,这个契约:

(->* () #:rest (cons/c integer? (listof integer?)) any)

并不符合函数

(λ (x . rest) x)

因为合同坚持要求该函数接受零参数(因为合同中没有列出强制性参数)。 ->* 契约不知道关于 rest 参数的契约最终会不允许空参数列表。

pre-cond-expr 和 post-cond-expr 表达式分别在函数被调用和返回时被检查,并且允许在没有与参数(或结果)明确连接的情况下检查环境。 如果使用了 #:pre 或 #:post 关键字, 那么 #f 的结果将被视为失败,任何其他的结果将被视为成功。 如果使用了 #:pre/desc 或 #:post/desc 关键字, 那么表达式的结果必须是一个布尔值、一个字符串或者一个字符串列表, 其中 #t 表示成功,其他任何结果表示失败。 如果结果是一个字符串或字符串列表,那么字符串在每个换行后都应该有一个空格, 并且多个字符串在错误信息中作为行使用;在这种情况下,合同本身会给每个字符串增加一个缩进的空格。 格式要求没有被检查,但它们符合 错误信息约定 的建议。

作为一个例子,合同

(->* () (boolean? #:x integer?) #:rest (listof symbol?) symbol?)

匹配可选择接受布尔值、整数关键字参数 #:x 和任意多的符号,并且返回一个符号的函数。

->i 合同组合器与 ->* 组合器的不同之处在于, 每个参数和结果都被命名,这些名称可以在子合同和前后条件条款中使用。 换句话说, ->i 表达了参数和结果之间的依赖关系。

对 ->i 的可选的第一个关键字参数表明结果合同是否将是一个陪衬物。 如果它是 #:chaperone, 所有的参数和结果的契约必须是 chaperone contracts , 并且 ->i 的结果将是一个 chaperone contract 。 如果它不存在,那么结果契约将不是一个 chaperone contract 。

->i 契约的第一个子形式涵盖了必须的参数,第二个子形式涵盖了可选的参数。 之后是一个可选的 rest-args 契约,以及一个可选的前置条件。 前置条件是用 #:pre 关键字引入的,后面是它所依赖的名字列表。 如果使用 #:pre/name 关键字,提供的字符串将作为错误信息的一部分;类似的还有 #:post/name 。 如果使用了 #:pre/desc 或 #:post/desc , 那么表达式的结果将与 ->* 的处理方式相同。

在前置条件之后是可选的 param-value 非终端,它指定了在函数的动态范围内要分配的参数。 每个赋值都是用 #:param 关键字引入的,后面是它所依赖的名称列表, 确定要设置的参数的 param-expr ,以及将与该参数关联的 value-expr 。

dependent-range 非终端指定了可能的结果合同。 如果它是 any, 那么允许任何值。 否则,结果契约会将一个名字和一个契约配对,或者将多个值返回的名字和契约配对。 在后两种情况下,范围契约后面可以有一个后置条件;如果范围契约是 any ,则不允许后置条件表达。 和前条件一样,后条件必须指定它所依赖的变量。

考虑一下这个样本合同:
(->i ([x number?]
      [y (x) (>=/c x)])
     [result (x y) (and/c number? (>=/c (+ x y)))])
它指定了一个两个参数的函数,都是数字。 第二个参数 (y) 的契约要求它大于第一个参数。结果契约承诺了一个大于两个参数之和的数字。 y 的依赖性说明表明参数契约依赖于第一个参数的值,而 result 的依赖性序列表明契约依赖于两个参数的值。 一般来说,一个空序列(几乎)等同于不添加序列,只是前者比后者更昂贵。 由于 x 的契约不依赖于其他任何东西,所以它不带有任何依赖序列,甚至不带有 () 。

这个例子和前面的例子一样,只是 x 和 y 参数现在是可选的关键字参数,而不是强制性的、按位置的参数:
(->i ()
     (#:x [x number?]
      #:y [y (x) (>=/c x)])
     [result (x y)
      (and/c number?
             (if (and (number? x) (number? y))
                 (>=/c (+ x y))
                 any/c))])
在范围内测试 x 和 y 的条件是必要的, 以涵盖 x 或 y 没有被调用上下文提供的情况 (意味着它们可能被绑定到 the-unsupplied-arg) 。

契约表达式并不总是按顺序被评估的。 首先,如果某个契约表达式没有依赖关系,那么该契约表达式将在 ->i 表达式被评估时被评估, 而不是在函数被调用或返回时被评估。 这些没有依赖关系的契约表达式是按照它们被列出的顺序来评估的。 第二,当契约函数被调用或返回时,依赖性契约子表达式按照满足依赖性的某种顺序被评估。 也就是说,如果一个参数的契约依赖于其他一些契约的值,那么前者首先被评估 (这样一来,该参数在其契约被检查过后,就可以用于其他参数)。 当两个参数(或结果和一个参数)之间没有依赖关系时,那么在源文本中较早出现的契约会被首先评估。

如果具有依赖关系的范围契约的所有标识符位置都是 _ (下划线), 那么范围契约表达式将在函数被调用时而不是返回时被评估。 否则,依赖的范围表达式在函数返回时被评估。

如果有可选的参数没有被提供,那么相应的变量将被绑定到一个特殊的值, 称为 the-unsupplied-arg 值。 例如,在这个契约中:
(->i ([x (y) (if (unsupplied-arg? y)
                 real?
                 (>=/c y))])
     ([y real?])
     any)
关于 x 的契约取决于 y, 但 y 可能没有在调用地点被提供。 在这种情况下, x 上的契约中的 y 的值是 the-unsupplied-arg ,并且 ->i 合同必须检查它并修改 x 上的合同以考虑到 y 没有被提供。

当未提供的参数的契约表达式是依赖性的,而参数在调用地点没有被提供时,契约表达式根本就不会被评估。 例如,在这个契约中, y 的契约表达式只有在 y 被提供时才被评估:
(->i ()
     ([x real?]
      [y (x) (>=/c x)])
     any)
相比之下, x 的表达式总是被评估的(事实上,当 ->i 表达式被评估时, 它也被评估了,因为它没有任何依赖关系)。

(->d (mandatory-dependent-dom ...)
     dependent-rest
     pre-condition
     dependent-range
     post-condition)
(->d (mandatory-dependent-dom ...)
     (optional-dependent-dom ...)
     dependent-rest
     pre-condition
     dependent-range
     post-condition)
 
mandatory-dependent-dom = [id dom-expr]
  | keyword [id dom-expr]
     
optional-dependent-dom = [id dom-expr]
  | keyword [id dom-expr]
     
dependent-rest = 
  | #:rest id rest-expr
     
pre-condition = 
  | #:pre boolean-expr
  | #:pre-cond boolean-expr
     
dependent-range = any
  | [_ range-expr]
  | (values [_ range-expr] ...)
  | [id range-expr]
  | (values [id range-expr] ...)
     
post-condition = 
  | #:post-cond boolean-expr
这个契约在这里是为了向后兼容;任何新的代码都应该使用 ->i 来代替。

这个契约与 ->i 相似,但它是 "宽松的",意味着它在内部不执行契约。例如,使用这个契约
(->d ([f (-> integer? integer?)])
     #:pre
     (zero? (f #f))
     any)
将允许用 #f 来调用 f 触发了 f 的作者通过坚持 f 的契约只接受整数而试图禁止的任何不良行为。

#:pre-cond 和 #:post-cond 关键字是 #:pre 和 #:post 的别名,是为了向后兼容。

syntax

(case-> (-> dom-expr ... rest range) ...)

 
rest = 
  | #:rest rest-expr
     
range = range-expr
  | (values range-expr ...)
  | any
这种契约形式被设计用来匹配 case-lambda 。 case-> 的每个参数都是一个契约,管理 case-lambda 中的一个子句。 如果 #:rest 关键字出现,相应的子句必须接受任意数量的参数。 range 的规定就像 -> 和 ->* 的规定一样。

例如,这个契约与一个有两种情况的函数相匹配,一种是接受一个整数,返回 void,另一种是不接受参数,返回一个整数。
(case-> (-> integer? void?)
        (-> integer?))
这样的契约可以用来保护一个控制对单个共享整数访问的函数。

procedure

(dynamic->* 
  [#:mandatory-domain-contracts mandatory-domain-contracts 
  #:optional-domain-contracts optional-domain-contracts 
  #:mandatory-keywords mandatory-keywords 
  #:mandatory-keyword-contracts mandatory-keyword-contracts 
  #:optional-keywords optional-keywords 
  #:optional-keyword-contracts optional-keyword-contracts 
  #:rest-contract rest-contract] 
  #:range-contracts range-contracts) 
 → contract?
  mandatory-domain-contracts : (listof contract?) = '()
  optional-domain-contracts : (listof contract?) = '()
  mandatory-keywords : (listof keyword?) = '()
  mandatory-keyword-contracts : (listof contract?) = '()
  optional-keywords : (listof keyword?) = '()
  optional-keyword-contracts : (listof contract?) = '()
  rest-contract : (or/c #f contract?) = #f
  range-contracts : (or/c #f (listof contract?))
和 ->* 一样,只是参数和结果的数量可以在运行时计算,而不是在编译时固定。 传递 #f 作为 #:range-contracts 参数会产生一个类似于 any 与 -> 或 ->* 一起使用的契约。

对于许多用途来说, dynamic->* 的结果比 ->* (或 ->) 慢, 但对于某些用途来说,它的速度是相当的。 dynamic->* 返回的契约的名称使用 -> 或 ->* 的语法。

syntax

(unconstrained-domain-> range-expr ...)

构造一个接受一个函数的契约,但对该函数的域不做任何约束。 range-expr 决定了结果的数量和每个结果的契约。

一般来说,这个契约必须与另一个契约结合起来,以确保域实际上是已知的,以便能够安全地调用函数本身。

例如,合同中的

(provide
 (contract-out
  [f (->d ([size natural-number/c]
           [proc (and/c (unconstrained-domain-> number?)
                        (lambda (p)
                          (procedure-arity-includes? p size)))])
          ()
          [_ number?])]))

表示函数 f 接受一个自然数和一个函数。 f接受的函数域必须包括 size 参数的情况, 这意味着 f 可以安全地给它的输入提供 size 参数。

例如,下面是 f 的一个定义,使用上述契约不能责备:

(define (f i g)
  (apply g (build-list i add1)))
