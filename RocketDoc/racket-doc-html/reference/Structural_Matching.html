7.6 Structural Matching

syntax

(unit/new-import-export
  (import tagged-sig-spec ...)
  (export tagged-sig-spec ...)
  init-depends-decl
  ((tagged-sig-spec ...) unit-expr tagged-sig-spec))
与 unit 类似,除了单元的主体是由 unit-expr 产生的一个现有单元决定的。 其结果是一个单元,其实现是 unit-expr, 但其导入、导出和初始化依赖关系与 unit/new-import-export 形式相同(而不是与 unit-expr 产生的单元相同)。

unit/new-import-export 形式的最后一个子句决定了新旧导入和导出之间的连接。 这种连接类似于 compound-unit 传播进口和出口的方式;不同的是, import 和链接子句右侧的连接是基于签名中元素的名字,而不是签名的名字。 也就是说,链接子句右侧的 tagged-sig-spec 不需要作为 tagged-sig-spec 出现在 import 子句中, 但是链接的 tagged-sig-spec 隐含的每个绑定必须由 import 子句中的某个 tagged-sig-spec 所隐含。 同样地, export tagged-sig-spec 所隐含的每个绑定都必须被连接子句中的某个左侧 tagged-sig-spec 所隐含。

syntax

(define-unit/new-import-export unit-id
  (import tagged-sig-spec ...)
  (export tagged-sig-spec ...)
  init-depends-decl
  ((tagged-sig-spec ...) unit-expr tagged-sig-spec))
像 unit/new-import-export 一样, 但像 define-unit 一样将静态信息绑定到 unit-id 。

syntax

(unit/s
  (import tagged-sig-spec ...)
  (export tagged-sig-spec ...)
  init-depends-decl
  unit-id)
与 unit/new-import-export 类似,但链接条款是推断出来的,所以 unit-id 必须有适当的静态信息。

syntax

(define-unit/s name-id
  (import tagged-sig-spec ...)
  (export tagged-sig-spec ...)
  init-depends-decl
  unit-id)
像 unit/s 一样,但像 define-unit 一样将静态信息绑定到 name-id 。
