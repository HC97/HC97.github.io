13.1.1 编码和地域

当一个端口被提供给一个基于字符的操作时,比如 read-char 或 read, 端口的字节被读取并解释为 UTF-8 编码的字符。 因此,读取一个字符可能需要读取多个字节,像 char-ready? 这样的程序可能需要窥探流中的几个字节, 以确定一个字符是否可用。 如果一个字节流不符合有效的 UTF-8 编码, 像 read-char 这样的函数可能需要在字节流中提前窥视一个字节以发现该字节流不是有效的编码。

当一个输入端口产生的字节序列在字符读取上下文中不是有效的 UTF-8 编码时, 那么构成无效序列的字节将被转换为字符 #\uFFFD 。 具体来说,255 和 254 字节总是被转换为 #\uFFFD, 192 至 253 范围内的字节在后面没有形成有效的 UTF-8 编码的字节时产生 #\uFFFD ， 128 至 191 范围内的字节在不属于由 192 至 253 范围内的前一个字节开始的有效编码时被转换为 #\uFFFD 。 换句话说,当读取一个字节序列作为字符时,一组最小的字节被改变为 #\uFFFD 的编码, 这样整个字节序列就是一个有效的 UTF-8 编码。

参见 字节串 以了解促进使用 UTF-8 或其他编码进行转换的程序。 参见 reencode-input-port 和 reencode-output-port , 从使用不同字符编码的端口中获取基于 UTF-8 的端口。

locale 捕捉了关于用户对字符序列的特定语言解释的信息。 特别是,区域设置决定了字符串如何被 "按字母顺序排列",小写字母如何被转换为大写字母,以及字符串如何被比较而不考虑大小写。 像 string-ci=? 这样的字符串操作对当前的 locale 不 敏感, 但是像 string-locale-ci=? 这样的操作(见 字符串) 会产生与当前 locale 一致的结果。

地区设置还指定了将码点序列编码为字节序列的特定编码。Racket 通常会忽略 locale 的这个方面,但有几个明显的例外: 以字节串形式传递给 Racket 的命令行参数会被转换为使用 locale 编码的字符串; 以字节串形式传递给其他进程(通过 subprocess) 的命令行字符串会被转换为使用 locale 编码的字节串; 环境变量使用当地的编码转换为字符串;文件系统路径使用当地的编码转换为字符串(用于显示);最后,Racket 提供了 string->bytes/locale 等函数来专门调用当地的特定编码。

Unix 用户通过设置环境变量(如 LC_ALL 来选择地区设置。 在 Windows 和 Mac OS 上,操作系统提供了其他的机制来设置 locale。 在 Racket 中,当前的 locale 可以通过设置 current-locale 参数来改变。 Racket 内部的 locale 名称是一个字符串,可用的 locale 名称取决于平台及其配置, 但 "" locale 意味着当前用户的默认 locale;在 Windows 和 Mac OS 上, "" 的编码始终是 UTF-8,对本地敏感的操作使用操作系统的本地接口。 (特别是,设置 LC_ALL 和 LC_CTYPE 环境变量并不影响 Mac OS 上的 locale "" 。 如果需要的话,使用 getenv 和 current-locale 来明确安装环境指定的 locale)。 将当前的 locale 设置为 #f ,使得对本地敏感的操作不具有本地敏感性, 这意味着对大小写操作使用 Unicode 映射,对编码使用 UTF-8。

parameter

(current-locale) → (or/c string? #f)

(current-locale locale) → void?
  locale : (or/c string? #f)
一个 parameter ,决定了程序的当前 locale ,如 string-locale-ci=? 。

当通过将该参数设置为 #f 来禁用 locale 敏感性时,字符串将以完全可移植的方式进行比较等,这与标准程序是一样的。 否则,字符串将根据区域设置(在 C 库的 setlocale 意义上)进行解释。 "" 总是当前机器默认 locale 的一个别名,而且是默认的。 "C" 的 locale 也总是可用的;将 locale 设置为 "C" 与用 #f 禁用 locale 敏感性是一样的,只有当字符串操作被限制在前 128 个字符时才是如此。 其他的 locale 名称是针对平台的。

用 write 打印字符串或字符不受参数影响,符号大小写或正则表达式也不受影响(见 正则表达式) 。
