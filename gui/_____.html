procedure

(append-editor-font-menu-items menu) → void?

  menu : (or/c (is-a?/c menu%) (is-a?/c popup-menu%))
将菜单项添加到 menu 中,以实现一套标准的字体处理操作,比如改变字体面孔或样式。 每个菜单项的回调都使用 get-edit-target-object in top-level-window<%> (通过跟踪父类链找到框架,直到达到一个框架);如果结果是一个 editor<%> 对象, change-style in text% 或 change-style in pasteboard% 将在编辑器中被调用。

procedure

(append-editor-operation-menu-items 
  menu 
  [text-only? 
  #:popup-position popup-position]) 
 → void?
  menu : (or/c (is-a?/c menu%) (is-a?/c popup-menu%))
  text-only? : any/c = #t
  popup-position : (or/c #f (list/c (is-a?/c text%) exact-nonnegative-integer?))
   = #f
将菜单项添加到 menu 中,以实现标准的编辑器操作,如剪切和粘贴。 每个菜单项的回调使用 get-edit-target-object in top-level-window<%> (通过跟踪父类链找到框架,直到达到一个框架); 如果结果是一个 editor<%> 对象,则在编辑器中调用 do-edit-operation in editor<%> 。

如果 text-only? 是 #f, 那么插入非文本剪贴的菜单项(比如 Insert Image...)会被附加到菜单上。

如果 popup-position 不是 #f, 那么 append-editor-operation-menu-items 预计将被调用以建立一个弹出菜单, 并且列表中的两个元素应该是鼠标点击弹出菜单的 text% 对象和点击发生的位置。 在这种情况下,当点击落在一个不是 string-snip% 的片段上时, Copy 和 Cut 菜单会被启用,相应的回调会复制和剪切那一个片段。

parameter

(current-text-keymap-initializer)

 → ((is-a?/c keymap%) . -> . any/c)
(current-text-keymap-initializer proc) → void?
  proc : ((is-a?/c keymap%) . -> . any/c)
指定键图初始化过程的参数。 这个过程被调用来初始化 text-field% 对象或由 graphical-read-eval-print-loop 创建的 text% 对象的键盘图。

初始化程序接收一个键位图对象,不返回任何东西。 默认的初始化器将给定的键盘图与一个内部键盘图相连接,该键盘图实现了标准的文本编辑器键盘和鼠标绑定, 用于剪切、复制、粘贴、撤销和全选。 鼠标右键被映射为在释放按钮时弹出一个编辑菜单。

Changed in version 1.51 of package gui-lib: 改变了默认初始化器中的 Unix 键盘绑定,以匹配 Windows,放弃了行首和行尾的绑定。

procedure

(map-command-as-meta-key on?) → void?

  on? : any/c
(map-command-as-meta-key) → boolean?
在 Mac OS 上确定 m: 的解释,对于一个 keymap% 的映射。 也请参见 map-function in keymap% 。

第一种情况:

如果 on? 是 #t, m: 对应于命令键。 如果 on? 是 #f, 那么 m: 对应于 Mac OS 上的任何键。

第二种情况:

如果 m: 对应于 Command,则返回 #t ,否则返回 #f 。

procedure

(open-input-graphical-file filename) → input-port?

  filename : string?
打开 filename (在 'binary 模式下)并检查它是否看起来像一个编辑器格式的 "图形" 文件。 如果该文件看起来不是一个编辑器文件,则返回文件端口,并启用行计数。 否则,该文件会被加载到一个编辑器中,并使用 open-input-text-editor 创建结果端口。

procedure

(open-input-text-editor 
  text-editor 
  [start-position 
  end-position 
  snip-filter 
  port-name 
  expect-to-read-all? 
  #:lock-while-reading? lock-while-reading?]) 
 → input-port
  text-editor : (is-a?/c text%)
  start-position : exact-nonnegative-integer? = 0
  end-position : (or/c exact-nonnegative-integer? 'end) = 'end
  snip-filter : ((is-a?/c snip%) . -> . any/c) = (lambda (s) s)
  port-name : any/c = text-editor
  expect-to-read-all? : any/c = #f
  lock-while-reading? : any/c = #f
创建一个输入端口,从 text-editor 中获取其内容。 start-position 和 end-position 之间的编辑器内容就是端口的内容。 如果 end-position 是 'end, 内容会一直运行到编辑器的末端。 如果一个不是 string-snip% 对象的片段跨越了 start-position 或 end-position, 那么整个片段都会贡献给这个 port。 如果一个 string-snip% 实例跨越了 start-position, 那么只有在 start-position 之后的部分才有贡献; 如果一个 string-snip% 对象跨越了 end-position, 那么只有在 end-position 之前的部分有贡献。

text-editor 中的 string-snip% 实例会在生成的端口中产生一个字符序列。 所有其他类型的狙击都会被传递给 snip-filter 以获得端口的一个 "特殊" 值。 如果 snip-filter 的第一个结果是一个snip, 并且这个 snip 是 readable-snip<%> 的一个实例, 那么这个 snip 将通过 read-special 方法为端口生成一个特殊的值。 如果 snip-filter 返回任何其他种类的snip,它将被复制为特殊结果。 最后,来自 snip-filter 的非 snip 的第一个结果被直接作为特殊结果使用。

port-name 参数用于输入端口的名称。 expect-to-read-all? 参数是一个性能提示;如果整个端口的数据流将被读取,则使用 #t 。

如果 text-editor 发生以下变化,则不得使用结果端口: 插入一个片段(见 after-insert), 删除一个片段(见 after-delete), 分割一个片段(见 after-split-snip), 合并一个片段(见 after-merge-snips), 或改变一个片段的计数(这很罕见,见 recounted)。 get-revision-number 方法可以用来检测任何这些变化。

为了防止这种使用,如果 lock-while-reading? 参数是一个真值, 那么 open-input-text-editor 将 lock text-editor , 并在返回前调用 begin-edit-sequence , 在安全使用上述方法后解 lock 并调用 end-edit-sequence 。 (在某些情况下,如果使用这些方法总是安全的,它将根本不 lock 编辑器或将其放入编辑序列)。

procedure

(open-output-text-editor text-editor    
  [start-position    
  special-filter    
  port-name    
  #:eventspace eventspace]) → output-port
  text-editor : (is-a?/c text%)
  start-position : (or/c exact-nonnegative-integer? (one/of 'end))
   = 'end
  special-filter : (any/c . -> . any/c) = (lambda (x) x)
  port-name : any/c = text-editor
  eventspace : (or/c eventspace? #f) = (current-eventspace)
创建一个输出端口,将其内容传递给 text-editor 。 内容从 start-position 位置开始写入 text-editor , 其中 'end 表示输出应该从 text editor 的当前结束位置开始。

如果提供了 special-filter ， 它将被应用于任何用 write-special 写入端口的值,而结果将被插入其位置。 如果特殊值是一个 snip% 对象,它将被插入到编辑器中。 否则,特殊值被 display 到编辑器中。

如果所产生的输出端口启用了行计数,那么该端口将报告行,从行开始的偏移量,以及该端口写入数据时在编辑器中的位置。

如果 eventspace 不是 #f, 那么当输出端口在 eventspace 的处理程序线程之外的线程中使用时, 内容将通过 eventspace 中的低优先级回调传递给 text-editor 。 因此,如果 eventspace 对应于编辑器 displays 的事件空间,那么从任何线程向输出端口写入都是安全的。

如果 eventspace 是 #f, 请注意该端口只是弱线程安全的。 内容在 edit sequence 中被传递给 text-editor , 但如果,例如,编辑器在一个启用的 editor-canvas% 中显示,编辑序列是不够同步的。 更多信息见 编辑器和线程 。

procedure

(read-editor-global-header in) → boolean?

  in : (is-a?/c editor-stream-in%)
从 in 中读取数据,为从流中读取编辑器进行初始化。 如果读取成功,返回值为 #t ,否则为 #f 。

通过调用编辑器的 read-from-file 方法可以从流中读取一个或多个编辑器。 (要读取的编辑器的数量必须由应用程序事先知道)。 当所有的编辑器被读取后,调用 read-editor-global-footer 。 对 read-editor-global-header 和 read-editor-global-footer 的调用必须在任何对 read-from-file 的调用前面, 而且一次只能用这些方法读取一个流, 或者用 write-editor-global-header 和 write-editor-global-footer 。

当从跨越 Racket 版本的流中读取时,在此过程之前使用 read-editor-version 。

procedure

(read-editor-version in    
  in-base    
  parse-format?    
  [raise-errors?]) → boolean?
  in : (is-a?/c editor-stream-in%)
  in-base : (is-a?/c editor-stream-in-base%)
  parse-format? : any/c
  raise-errors? : any/c = #t
从 in-base 中读取版本信息,其中 in-base 是 in 的基础。 从 in-base 中解析出的版本信息被记录在 in 中,以便以后进行对版本敏感的解析。 如果版本信息被成功读取,并且版本被支持,那么过程结果为真。

如果 parse-format? 为 true,那么 in-base 会被检查出一个初始的 "WXME" 格式指示器。 当 "WXME" 已经被格式化代码消耗时,使用 #f 。

如果 raise-errors? 为真,那么读取过程中的错误会触发一个异常,而不是一个 #f 的结果。

procedure

(text-editor-load-handler [filename]    
  expected-module-name) → any/c
  filename : path = string
  expected-module-name : (or/c symbol? #f)
T这个过程是一个与 current-load 一起使用的加载处理程序。

该处理程序可以识别 Racket 编辑器格式的文件(见 文件格式) ,并对其进行解码以便加载。 它通常在 GRacket 启动时被安装(见 运行 Racket 或 GRacket)。

该处理程序通过文件的前 12 个字符来识别编辑器文件: WXME01‹digit›‹digit› ## 。 这样的文件是通过创建一个 text% 对象来打开加载的, 用 insert-file 将文件加载到该对象中, 然后用 open-input-text-editor 将编辑器内容转换为一个端口。 以这种方式获得一个端口后,内容的读取方式与默认的 Racket 加载程序基本相同。 不同的是,编辑器可能包含 readable-snip<%> 的实例, 这些实例通过片段的 read-special 方法被 "读取"; 详见 open-input-text-editor 。

procedure

(write-editor-global-header out) → boolean?

  out : (is-a?/c editor-stream-out%)
将数据写入 out, 初始化它,以便将编辑器写入流中。 如果写入成功,返回值为 #t ,否则为 #f 。

通过调用编辑器的 write-to-file 方法,可以将一个或多个编辑器写入流中。 当所有的编辑器都被写入时,调用 write-editor-global-footer 。 对 write-editor-global-header 和 write-editor-global-footer 的调用必须在任何对 write-to-file 的调用前面,而且一次只能用这些方法写入一个流, 或用 read-editor-global-header 和 read-editor-global-footer 。

为了支持跨 Racket 版本的流,在这个过程之前使用 write-editor-version 。

参见 文件格式 。

procedure

(write-editor-version out out-base) → boolean?

  out : (is-a?/c editor-stream-out%)
  out-base : (is-a?/c editor-stream-out-base%)
将版本信息写入 out-base ,准备将编辑器信息写入流 out 。

out 参数目前不使用,但 out-base 应该是 out 的基础。 在未来, out 可能会记录关于版本的信息,以便以后进行版本敏感的输出。

如果写入成功,结果是 #t ,否则是 #f 。
