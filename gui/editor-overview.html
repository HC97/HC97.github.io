5 编辑器

编辑器工具箱为两种常见的应用程序提供了一个基础:

  • 需要复杂的文本编辑器的程序 — 简单的文本字段控制对于文本密集型的应用程序是不够的。 许多程序需要能够处理多种字体和非文本项目的编辑器。

  • 需要具有可拖动对象的画布的程序 — 绘图工具箱为绘制线条和方框提供了一个通用的绘图表面, 但许多应用程序需要一个交互式画布,用户可以拖动和调整单个对象的大小。

这两种应用程序都需要一个可扩展的编辑器,可以处理文本、图像、程序员定义的项目,甚至是嵌入式编辑器。 它们之间的区别在于项目的布局。 因此,编辑器工具箱通过两个类提供了两种编辑器:

  • text% — 在一个 text editor 中,项目被自动定位在一个段落流中。

  • pasteboard% — 在一个 pasteboard editor 中,项目被明确定位并可拖动。

这种编辑器架构解决了编辑器的全部现实问题,包括剪切和粘贴、可扩展的文件格式和分层文本样式,同时支持高水平的可扩展性。 不幸的是,该系统因此相当复杂,有效使用编辑器类需要对编辑器工具箱的结构和术语有扎实的了解。 然而,有足够多的应用程序符合上述描述中的一个(或两个),以证明工具箱的深度和复杂性以及使用它所需的学习投资是合理的。

一个简单的例子说明了编辑器是如何工作的。 首先,一个编辑器需要一个 editor-canvas% 来显示其内容。 然后,我们可以创建一个文本编辑器并将其安装到画布中:

(define f (new frame% [label "Simple Edit"]
                      [width 200]
                      [height 200]))
(define c (new editor-canvas% [parent f]))
(define t (new text%))
(send c set-editor t)
(send f show #t)

在这一点上,编辑器的功能是完整的:用户可以在编辑器中输入文本,但没有剪切和粘贴或撤销操作。 我们可以通过菜单栏支持编辑器上的所有标准操作:

(define mb (new menu-bar% [parent f]))
(define m-edit (new menu% [label "Edit"] [parent mb]))
(define m-font (new menu% [label "Font"] [parent mb]))
(append-editor-operation-menu-items m-edit #f)
(append-editor-font-menu-items m-font)
(send t set-max-undo-history 100)

现在,标准的剪切和粘贴操作可以使用,撤销操作也可以使用,用户甚至可以设置字体样式。 编辑器在创建时没有撤销历史堆栈, set-max-undo-history 被用来设置一个非零的堆栈, 所以撤销操作可以被记录下来。 用户也可以通过从 Edit 菜单中选择 Insert Text 来插入一个嵌入式编辑器; 选择该菜单项后,编辑器中会出现一个内有光标的方框。 在方框内用光标打字时,方框内的文字会被拉长,而字体操作则适用于光标处于活动状态的地方。 框外的文字会随着框的大小变化而重新排列。请注意,方框本身可以被复制和粘贴。

编辑器的内容是由 snips 组成的。 从嵌入编辑器的角度看,一个嵌入的编辑器是一个单一的剪贴。 为了对即时文本进行编码,一个片段可以是一个单独的字符,但更常见的是一个片段是同一行中相邻的字符序列。 find-snip 方法从文本编辑器中提取一个snip:

(send t find-snip 0 'after)

上面的表达式返回编辑器中的第一个片段,它可能是一个字符串片段(对于即时文本)或一个编辑器片段(对于嵌入式编辑器)。

编辑器不是永久地附着在任何显示器上。我们可以把文本编辑器从我们的画布中拿出来,而把一个粘贴板编辑器放在画布中:

(define pb (new pasteboard%))
(send c set-editor pb)

安装了粘贴板编辑器后,用户不能再直接在编辑器中输入字符(因为粘贴板不支持直接输入文本)。 然而,用户可以从其他地方剪切文本并将其粘贴到粘贴板上,或者选择 Edit 菜单中的一个 Insert 菜单项。 剪辑在粘贴板编辑器中是可以清楚识别的(与文本编辑器不同),因为每个剪裁都是可以单独拖动的。

我们可以通过明确地创建一个编辑器片段,将旧的文本编辑器(我们最近从画布上删除了它)作为一个嵌入式编辑器插入到粘贴板中:

(define s (make-object editor-snip% t)) ; t is the old text editor
(send pb insert s)

一个单独的片段不能同时插入不同的编辑器中,也不能在同一个编辑器中多次插入:

(send pb insert s) ; no effect

然而,我们可以对片段进行深度拷贝,并将拷贝插入粘贴板中:

(send pb insert (send s copy))

使用编辑器类的应用程序通常会派生出 text% 和 pasteboard% 类的新版本。 例如,要实现一个只允许插入的编辑器(只允许在末尾插入,不允许删除), 可以从 text% 派生出一个新的类, 并覆盖 can-insert? 和 can-delete? 方法:

(define append-only-text%
  (class text%
    (inherit last-position)
    (define/augment (can-insert? s l) (= s (last-position)))
    (define/augment (can-delete? s l) #f)
    (super-new)))

5.1 编辑器结构和术语

编辑器工具箱通过将一个编辑器组件分解成三个功能部分来支持可扩展和可嵌套的编辑器:

  • editor 本身存储文本或粘贴板的状态并处理大多数事件和编辑操作。 editor<%> 接口定义了核心的编辑器功能, 但编辑器是作为 text% 或 pasteboard% 的实例来创建的。

  • 一个 snip 是编辑器中的一个信息段。 每个片段可以包含一个字符序列、一张图片或一个交互式对象(如一个嵌入式编辑器)。 在文本编辑器中,片段被限制在单行上,通常包含单一类型的数据。 snip% 类实现了一个基本的 snip,并作为实现新 snip 的基础;参见 实现新的剪贴画 。 其他狙击类包括用于管理文本的 string-snip% ,用于管理图片的 image-snip% , 以及用于管理嵌入式编辑器的 editor-snip% 。

  • display 在屏幕上显示编辑器。 显示屏可以让用户在一个编辑器周围滚动,或者改变编辑器。 大多数显示器是 editor-canvas% 类的实例, 但 editor-snip% 类也作为嵌入式编辑器的显示器。

这三个部分由一个简单的文字处理器来说明。编辑器与文本文档相对应。 编辑器对象接收键盘和鼠标命令来编辑文本。文本本身被分布在剪贴板中。 每个字符都可以是一个单独的片段,也可以将一行的多个字符组合成一个片段。 显示方式大致与显示文本的窗口相对应。 虽然编辑器管理文本的排列,因为它被显示在一个窗口中,但显示决定了绘制到哪个窗口和显示编辑器的哪个部分。

编辑器中每个可选择的实体是一个 item 。 在粘贴板中,所有的选择和拖动操作都是在剪贴板上进行的,所以剪贴板和项目之间存在着一对一的对应关系。 在一个编辑器中,一个片段包含一个或多个连续的项目,每个项目都属于某个片段。 例如,在一个简单的文本编辑器中,每个字符都是一个项目,但多个相邻的字符可能被归入一个 snip。 一个片段中的项目数就是该片段的 count.

在文本编辑器中,插入点可以出现的每个地方都是一个 position 。 一个有 n 个项目的文本编辑器包含 n+1 个位置:每个项目前一个位置,最后一个项目后一个位置。

粘贴板中的剪贴顺序决定了每个剪贴的绘制平面。 当两个剪贴板内的剪贴重叠时,顺序中较早的剪贴在另一个剪贴的前面 (即前者在后者之后绘制,这样前一个剪贴可能覆盖后一个剪贴的一部分)。

当一个编辑器被绘制成一个显示器时,每个剪贴和位置都有一个 location 。 一个位置或片段的位置是以相对于编辑器左上角的坐标来指定的。 编辑器中的位置只有在编辑器被显示时才有意义。

5.1.1 字符和字形

在有文本的编辑器中,一个 item 对应于一个 Racket 字符。 然而,有些用户会认为是一个字符的东西是由多个 Racket 字符组成的, 例如海盗旗表情符号(使用四个字符的编码)或 "e" 加一个重音修饰符(也有一个字符表示,但可能通过这两个字符表示)。 grapheme 是 Unicode 字素组规范所定义的用户感知字符的近似值。 Racket通过 string-grapheme-count 和 char-grapheme-step 等函数提供了对字素的支持。

在文本编辑器中处理字母词组需要特别小心。 像 grapheme-position in text% 和 position-grapheme in text% 这样的方法可以在项目和石墨烯指数之间进行转换,但大多数操作都是基于项目的位置,而且它们通常不受限制地保留石墨烯簇的序列。 在一个单一的片段内的石墨烯群将呈现为一个石墨烯,但跨越片段边界的石墨烯群将呈现为两个部分石墨烯。 在 insert in text% 方法接受可选的参数,以触发对会跨越现有内容和插入内容的石墨烯序列的检测, 并确保这些序列被保持在一起。

少量的 text% 方法默认是敏感的: delete, insert 一个字符,和 move-position 。

5.1.2 管理员

有两个额外的管理层次来管理 display-编辑和编辑-剪辑的连接。 编辑器从不与 display 直接通信;相反,它总是与一个 editor administrator 通信, 一个 editor-admin% 类的实例,它将信息传递给 display 。 同样地,一个狙击手与一个 snip administrator（ snip-admin% 类的一个实例)进行交流。

管理层使编辑器的层次结构变得灵活,而不强迫编辑器组件的每个部分都包含几个部分的功能。 例如,一个文本编辑器可以是另一个编辑器中的一个单 item ; 如果没有管理员, text% 类也必须包含 display (对于包含的编辑器)和狙击(对于嵌入式编辑器)的所有功能。 使用管理员,一个编辑器类可以同时作为一个包含和嵌入的编辑器, 而不需要直接实现 display 和剪贴功能。

一个剪贴画最多通过一个管理员属于一个编辑器。 一个编辑器在同一时间也只有一个管理员。然而,将编辑器连接到标准 display (即编辑器画布)的管理员可以与其他此类管理员一起工作。 特别是,一个 editor-canvas% 的管理员(每个都有自己的管理员) 可以与其他编 editor-canvas% 的管理员一起工作, 允许一个编辑器同时在多个 editor-canvas% 窗口中显示。

当一个编辑器被多个 canvas 显示时,其中一个 canvas 的管理员被用作编辑器的主要管理员。 为了处理其他画布的用户和更新事件,编辑器的管理员被暂时改变, 然后通过编辑器的 set-admin 方法恢复。 因此,编辑器的 get-admin 方法的返回值取决于调用的上下文。

5.1.3 风格

style 是 style<%> 接口的一个实例,对所有剪贴类所共有的高级显示信息进行参数化。 这包括用于绘制项目的字体、颜色和对齐方式。每个 snip 都有一个样式。

样式是分层次的:每个样式都是根据另一个样式来定义的。 有 一个单一的 root style, 名为 "Basic", 一个编辑器中的所有其他样式都来自于此。 一个基本样式和它的每个派生样式之间的差异被编码为一个 style delta (或简称 delta)。 一个 delta 编码的变化,如

  • 将字体家族改为 X;

  • 通过在字体大小上增加 Y 来扩大字体;

  • 切换字体的粗细;或

  • 改变所有内容以匹配风格描述 Z 。

样式对象从不单独创建;相反,它们总是通过一个 style list ,即 style-list% 类的一个实例来创建。 样式列表管理着样式,为查找特定样式的外部请求提供服务,它还管理着样式之间的层次关系。 一个全局的样式表是可用的,即 the-style-list, 但新的样式表可以被创建,以管理单独的样式层次结构。 例如,每个编辑器通常会有它自己的风格列表。

每个新的样式都是以两种方式之一定义的:

  • 一个 derived style 是以一个基本样式和一个 delta 来定义的。 每个样式(除了根样式)都有一个基本样式, 即使它不以任何方式依赖于基本样式(即,delta 描述了一个固定的样式,而不是对现有样式的扩展)。 (这就是通常的样式继承,就像在 Microsoft Word 等文字处理程序中发现的那样)。

  • 一个 join style 是根据另外两个样式来定义的:一个基本样式和一个 shift style 。 一个连接样式的含义是通过重新解释移位样式来确定的;在重新解释中,基本样式被用作移位样式的 root 样式。 (这类似于多级样式,像 FrameMaker 中的段落和字符样式。 在这个类比中,段落样式是基本样式,而字符样式是转换样式。 然而,FrameMaker 只允许这两个级别;而连接样式支持任何数量的级别)。

通常, 当文本被插入到一个文本编辑器时,它继承了前面的剪贴的样式。 如果文本被插入到一个空的编辑器中,文本通常被分配一个叫做 "Standard" 的样式。 默认情况下, "Standard" 样式是不修改根样式的。 默认的样式名称可以通过覆盖 default-style-name 来改变。

上述情况的例外是,当 change-style in text% 被调用的时候, 有当前的选择 position (当选择是一个 position 而不是一个范围)。 在这种情况下,样式会被记住,如果下一个修改编辑器的动作是插入文本,插入的文本会得到记住的样式。

参见 get-styles-sticky in text% ,了解更多关于插入文本的样式的信息。

5.2 文件格式

为了使编辑器的内容能够被保存到文件中,编辑器类实现了一种特殊的文件格式,称为 WXME 。 (在应用程序或事件空间之间进行剪切和粘贴时也使用这种格式)。 该文件格式没有被记录下来,只是它以 WXME01‹digit›‹digit› ##  开始。 否则, load-file 和 save-file 方法会在内部定义格式。 文件格式对文本编辑器和粘贴板编辑器是一样的。 当一个粘贴板将其内容保存到文件中时,它将从前到后保存剪贴,还包括额外的位置信息。 wxme 库提供了操作 WXME 文件的实用程序。

编辑器的数据是用 editor-stream-in% 和 editor-stream-out% 对象来读写的。 编辑器信息一次只能从一个流中读取或写入。 要将一个或多个编辑器写入一个流中,首先调用函数 write-editor-global-header 将初始化数据写入输出流中。 当所有的编辑器都被写入流中时,调用 write-editor-global-footer 。 同样地,从流中读取编辑器是通过 read-editor-global-header 初始化的, 并通过 read-editor-global-footer 最终完成。 另外,为了支持跨越Racket版本的流,可以在头文件操作之前使用 write-editor-version 和 read-editor-version 。

编辑器文件的数据格式可以被嵌入到另一个文件中,并且可以用新的数据类型来扩展。 编辑器文件格式可以用两种方式来扩展:用特定于剪报或内容的数据,以及用特定于编辑器的全局数据。 这些将在本节的其余部分进行描述。

5.2.1 编码剪贴

广义的剪辑概念允许新的剪辑类型被定义并立即用于任何编辑器类中。 此外,当两个应用程序支持相同种类的剪贴,剪贴数据可以很容易地在它们之间进行剪切和粘贴,并且每个程序都可以读取相同的数据文件。 这种互操作性是由于 snip 系统中内置了一个一致的编码机制。

对snips进行优雅的和可扩展的编码需要解决两个问题:

  • 一个 snip 的编码功能可以与 snip 本身相关联。 为了将一个 snip 从编码表示(例如,作为文件中的字节)转换为一个内存对象,必须为每种类型的 snip 提供一个解码函数。 此外,这种解码器的列表必须提供给高层解码过程。 这种解码映射是通过将一个 snip class 对象与每个 snip 相关联来定义的。 一个 snip 类是 snip-class% 类的一个实例。

  • 一些编辑器可能需要存储关于狙击点的额外信息;这些信息与狙击点本身存储的特定类型信息是正交的。 例如,粘贴板需要记住一个剪贴的 location ,而文本编辑器则不需要这个信息。 如果数据被从一个粘贴板剪切和粘贴到另一个粘贴板,那么相对 location 的信息就需要被维护, 但这些信息不应该妨碍粘贴到编辑器中。 额外的数据通过 editor data 对象与剪贴相关联,这些对象是 editor-data% 类的实例; 解码要求每个编辑器数据对象有一个 editor data class, 它是 editor-data-class% 类的实例。

狙击类、狙击数据和狙击数据类解决了与编码和解码狙击有关的问题。 在一个不需要保存文件或剪切和粘贴的应用程序中,这些问题可以安全地被忽略。

5.2.1.1 剪辑类

每个片段都可以与一个 snip class 相关联。 这个 “class” 不是程序员语言中的类描述;它是一个对象,它提供了一种方法来从编码的狙击规范中创建适当类型的新狙击。

狙击类对象可以被添加到特定事件空间的 snip class list 中,该列表由 get-the-snip-class-list 返回。 当一个狙击手被编码时,狙击手的类别名称与编码相关联;当狙击手需要被解码时,则通过名称搜索狙击手类别列表来找到狙击手的类别。 然后,snip 类将提供一个解码功能,可以从编码中创建一个新的 snip。

如果一个 snip 类的名字是 "((lib ...) (lib ...))" 的形式,那么这个 snip 类的实现就可以按需加载。 这个名字是用 read 来解析的; 如果结果是 ((lib string ...) (lib string ...)) 的形式, 那么第一个元素和 'snip-class 一起使用 dynamic-require 。 如果 dynamic-require 的结果是一个 snip-class% 对象, 那么它将被插入到当前事件空间的 snip 类列表中,并使用新类继续加载或保存。

"((lib ...) (lib ...))" 中的第二个 lib 形式为纯文本的 snip 提供了一个阅读器。 关于这种狙击类如何工作的更多信息,请参见 Snip 类映射 (一般参见 wxme 库)。

一个 snip 类的名字也可以只是 "(lib ...)", 它的使用方式就像 two-lib 形式的第一部分。 然而,这种形式没有为纯文本的 wxme 阅读器提供信息。

关于一个 snip 实现的例子及其相关的 snip-class 实现,请看 实现新的剪贴画 。

5.2.1.2 编辑器数据

当一个片段属于一个编辑器的时候,编辑器可能会以某种专门的方式存储关于片段的额外信息。 当片段要被编码时,这些额外的信息需要被放入一个 editor data 对象中,以便这些额外的信息也能被编码。 在一个文本编辑器中,额外的信息可以与 item 的范围相关联,也可以与剪报相关联。

就像一个片段必须与一个片段类相关联才能被解码(见 剪辑类), 一个编辑器数据对象需要一个 editor data class 来解码。 每个编辑器数据类对象都可以被添加到特定事件空间的 editor data class list 中, 由 get-the-editor-data-class-list 返回。 另外,像Snip类(见 剪辑类), 编辑器数据类名称可以使用 "((lib ...)  (lib ...))" 的形式,以实现按需加载。 相应的模块应该导出一个名为 editor-data-class% 的对象,名为 'editor-data-class 。

为了存储和加载编辑器中的片段或区域的信息:

  • 从 editor-data% 和 editor-data-class% 派生出新的类。

  • 从 text% 或 pasteboard% 类中派生一个新的类, 并覆盖 get-snip-data 和 set-snip-data 方法和/或 get-region-data 和 set-region-data 方法。

    注意: get-region-data 和 set-region-data 方法是为剪切和粘贴编码而调用的, 但不是为文件保存编码而调用的;关于扩展文件格式的信息,请参阅 全局数据: 页眉和页脚 。

5.2.2 全局数据: 页眉和页脚

编辑器文件格式提供了在特殊的页眉和页脚部分添加额外的全局数据。要保存和加载特殊的页眉和/或页脚记录:

  • 为每个页眉/页脚记录选一个名字。 这个名字不应该与任何其他正在使用的页眉/页脚记录的名字冲突,其他人也不应该使用这些名字。 所有以 "wx" 开头的名字都保留给内部使用。 通过用唯一的名字标记额外的页眉和页脚记录,该文件可以在不支持这些记录的安装中安全加载。

  • 从 text% 或 pasteboard% 类派生出一个新的类, 并覆盖 write-headers-to-file, write-footers-to-file, read-header-from-file 和/或 read-footer-from-file 方法。

当一个编辑器被保存时, 会调用 write-headers-to-file 和 write-footers-to-file 方法;此时,派生的 text% 或 pasteboard% 对象有机会保存记录。 要写一条页眉/页脚记录,首先调用 begin-write-header-footer-to-file 方法,这时会提供记录名称。 一旦记录被写入,调用 end-write-header-footer-to-file 。

当一个编辑器被加载并遇到一个页眉/页脚记录时, 会调用 read-header-from-file 或 read-footer-from-file 方法, 将记录名称作为参数。 如果名称与已知的记录类型相匹配,那么数据就可以被加载。

参见 write-headers-to-file 和 read-header-from-file 。

5.3 行末的模糊性

由于编辑器可以在没有换行项的情况下强制断行, 单单一个 position 并不总是能指定一个光标的 location 。 考虑一下软断行(即没有换行)的最后一个 position : 在该行的最后一个 item 和下一行的第一个 item 之间没有 item , 所以两个 location (一个行末,一个行首)映射到同一个 position 。

由于这个原因, position 和 position 方法通常有一个额外的参数。 在 position 设置方法的情况下, 该参数指定光标应该画在页面的左边还是右边(在 location 被双重定义的情况下); #t 意味着光标应该画在右边。 类似地,从 location 计算 position 的方法将接受一个额外的框内布尔值; 如果位置不明确并且来自右侧的位置,则该框内将填入 #t ,否则填入 #f 。

5.4 实现新的剪贴画

为了支持编辑器中新类型的内容,派生出一个 snip% 的子类来实现一种新的 snip 。 在派生一个新的 snip 实现时, snip% 的下列方法必须被重载以创建一个有用的 snip:

  • get-extent

  • draw

  • copy

  • resize 如果狙击点可以被用户调整大小的话

  • partial-offset 如果剪贴画可以包含一个以上的 item

  • split 如果剪贴画可以包含一个以上的 item

  • size-cache-invalid 如果剪贴画缓存了 get-extent 的结果

  • get-text (非必需)

  • find-scroll-step, get-num-scroll-steps 和 get-scroll-step-offset 如果该片段可以包含多个滚动位置

  • set-unmodified 如果狙击手的内部状态可以被用户修改, 当状态第一次改变时,在狙击手的管理员中调用 modified 。

如果一个 snip 可以包含一个以上的 item ,那么这个 snip 的 count 也必须被维护。

要定义一个可以保存或剪切和粘贴的剪贴画类(参见 剪辑类):

  • 创建一个 snip-class% 的实例,实现 read 方法。 将 snip-class% 实例作为 snip-class 从模块中导出, 并使用 "(lib ...)" 形式的类名,如 剪辑类 中所述。

  • 对于 snip 类的每个实例,用 set-snipclass 设置 snip 的类对象。

  • 覆盖 copy 方法。

  • 覆盖 write 方法。

在派生一个新的 snip-class% 类时:

  • 用 set-classname 设置类名。

  • 使用 set-version 来设置版本。

  • 使用 add 方法将该类安装到 get-the-snip-class-list 返回的列表中。 注意,如果同一个名字被多次插入同一个类的列表中,除了第一次插入外,其他都会被忽略。

要定义一个用 open-input-text-editor 专门读取的 snip 类:

  • 使你的 snip% 类实现 readable-snip<%> 。

  • 实现 read-special 方法。

作为一个例子,下面的模块实现了一个画圆的狙击手。点击狙击手会使圆圈变大。 为了能够将 snip 的实例从一个程序/事件空间复制到另一个程序/事件空间, 该模块应该是 "main.rkt" 一个 "circle-snip" 目录, 作为一个 "circle-snip" 包来安装。 这个 snip 后面还有一个 wxme 阅读器的实现, 必须作为 "circle-snip" 目录下的 "wxme-circle-snip.rkt" 文件安装。

  #lang racket/base
  (require racket/class
           racket/snip
           racket/format)
   
  (provide circle-snip%
           (rename-out [circle-snip-class snip-class]))
   
  (define circle-snip%
    (class snip%
      (inherit set-snipclass
               get-flags set-flags
               get-admin)
      (init-field [size 20.0])
      
      (super-new)
      (set-snipclass circle-snip-class)
      (send (get-the-snip-class-list) add circle-snip-class)
      (set-flags (cons 'handles-events (get-flags)))
      
      (define/override (get-extent dc x y        
                                   [w #f]
                                   [h #f]
                                   [descent #f]
                                   [space #f]
                                   [lspace #f]
                                   [rspace #f])
        (define (maybe-set-box! b v) (when b (set-box! b v)))
        (maybe-set-box! w (+ 2.0 size))
        (maybe-set-box! h (+ 2.0 size))
        (maybe-set-box! descent 1.0)
        (maybe-set-box! space 1.0)
        (maybe-set-box! lspace 1.0)
        (maybe-set-box! rspace 1.0))
      
      (define/override (draw dc x y left top right bottom dx dy draw-caret)
        (send dc draw-ellipse (+ x 1.0) (+ y 1.0) size size))
      
      (define/override (copy)
        (new circle-snip% [size size]))
      
      (define/override (write f)
        (send f put size))
      
      (define/override (on-event dc x y editorx editory e)
        (when (send e button-down?)
          (set! size (+ 1.0 size))
          (define admin (get-admin))
          (when admin
            (send admin resized this #t))))))
   
  (define circle-snip-class%
    (class snip-class%
      (inherit set-classname)
      
      (super-new)
      (set-classname (~s '((lib "main.rkt" "circle-snip")
                           (lib "wxme-circle-snip.rkt" "circle-snip"))))
      
      (define/override (read f)
        (define size-b (box 0.0))
        (send f get size-b)
        (new circle-snip% [size (unbox size-b)]))))
   
  (define circle-snip-class (new circle-snip-class%))

5.5 扁平化的文本

在纯文本编辑器中, position 和字符之间存在着简单的关联。 在 editor<%> 对象中,这在很多时候是不正确的,但有时 "获得编辑器的文本" 仍然是有用的。

文本可以以两种形式从一个编辑器中提取:

  • Simple text, 每个 item 有一个字符。 作为字符的 Item 被映射到它们自己,所有其他 item 被映射到一个句号。 换行符由换行符(ASCII 10)表示。

  • Flattened text, 每个 item 可以映射到一个任意的字符串。 作为字符的 Item 仍然被映射到自己身上, 但更复杂的 item 可以用一个由 item 的剪裁决定的有用的字符串来表示。 换行符被映射到特定平台的字符序列(Unix 和 Mac OS 上的换行符,以及 Windows 上的换行符-回车符)。 这种形式被称为 "扁平化",因为编辑器的 item 已经被简化为一个线性的字符序列。

5.6 粗体字的所有权

在一个框架内,只有一个对象可以包含键盘焦点。 当一个框架在多个 display 中包含多个编辑器时, 以及当一个编辑器包含其他编辑器作为 item 时,这个属性必须被维护。

当一个编辑器拥有键盘焦点时,它通常会显示当前的选择或表示插入点的线;该线被称为 caret 。

当一个编辑器包含其他的编辑器时,它就会在所包含的子编辑器中跟踪光标的所有权。 当光标从主编辑器中被拿走时,它将从相应的子编辑器中撤销光标的所有权。

当一个编辑器或剪贴画被画出时,画图方法的一个参数指定了是否应该将光标与数据一起画出,或者是否有一个选择跨越数据。 这个参数可以是以下任何一种:

  • 'no-caret — 根本就不应该画出光标。

  • 'show-inactive-caret — 圆点应该被画成非活动的; 项目可以被识别为本地的当前选择,但是键盘焦点在其他地方。

  • 'show-caret — 应该画出圆点以显示键盘焦点的所有权。

  • (cons start end) — 该光标被一个封闭的区域所拥有,它的选择跨越了当前的编辑器或剪贴画; 在剪贴画的情况下,选择跨越了剪贴画中的元素 start 到 end 位置。

'show-inactive-caret 显示模式对于在没有焦点的文本编辑器中显示选择范围很有用。 这种 'show-inactive-caret 模式与 'no-caret 模式不同; 当编辑器被嵌入时,只有本地活跃的编辑器才会显示其选择。

5.7 剪切和粘贴的时间戳

使用剪贴板的 editor<%> 方法–包括 copy, cut, paste 和 do-edit-operation — 会消耗一个时间戳参数。 这个时间戳通常是从触发剪贴板操作的 mouse-event% 或 key-event% 对象中提取的。 Unix 使用时间戳来同步剪贴板客户端之间的操作。

event% 的所有实例都包括一个时间戳,可以用 get-time-stamp 获得。

如果时间戳是 0,它默认为当前时间。 使用 0 作为时间戳几乎总能正常工作,但在 Unix 中被认为是不礼貌的。

5.8 点击返回

text% 编辑器中的 Clickbacks 有助于创建简单的互动对象,如超文本。 通过将回调函数与编辑器中的 item 范围相关联来定义点击回调。 当用户点击该范围内的 item 时,回调函数被调用。 例如,一个超文本点击回调将把一个范围与一个回调函数联系起来,以改变编辑器中的选择范围。

默认情况下,回调函数是在用户释放鼠标按钮时调用的。 set-clickback 方法接受一个可选的参数,使回调函数在按钮按下时被调用。 这种行为是有用的,例如,对于创建一个弹出式菜单的点击回调。

请注意,在保存文件时不会试图保存回调信息,因为回调将有一个任意的程序与之相关联。

5.9 内部编辑器锁

editor<%> 的实例有三个级别的内部锁定:

  • 写入锁定 — 当一个编辑器被内部锁定为写入时,编辑器的抽象内容不能被改变(例如,插入尝试会无声地失败)。 然而,文本编辑器中的片段仍然可以被分割和合并,而且文本编辑器可以以影响行的流动的方式被改变。 locked-for-write? 方法报告一个编辑器当前是否被锁定用于写作。

  • 流动锁定 — 当一个文本编辑器被内部锁定用于回流时,它被锁定用于写入,编辑器的 snip 内容不能改变, 如果一个snip的 location 还不知道,则不能计算(参见 locations-computed? in editor<%>), 并且编辑器不能被绘制到 display 中。 在流程锁定期间,对未计算的位置信息的请求会产生未定义的结果。 locked-for-flow? 方法报告一个编辑器当前是否被锁定为流动的。

  • 读取锁定 — 当一个编辑器被内部锁定读取时, 不能对该编辑器进行任何操作(例如,对当前选择位置的请求返回一个未定义的值)。 这种极端的状态只在回调到它的片段时使用,用于设置片段的管理员,分割片段,或合并片段。 locked-for-read? 方法报告一个编辑器当前是否被锁定用于阅读。

编辑器的内部锁 不 受对 lock 的调用影响。

报告与编辑器 location 无关的信息的方法不会触发锁。 一个报告 location 信息的方法可能会触发一个流锁或写锁, 如果相关信息在上次修改编辑器后还没有被计算出来的话(见 locations-computed? in editor<%>)。 一个以任何方式修改编辑器的方法,甚至是设置选择位置,都可能触发一个读锁、流锁或写锁。

5.10 编辑器和线程

编辑器并不与任何特定的线程或事件空间相联系,除了它在画布(有一个事件空间)中的显示程度。 编辑器的并发访问在弱的意义上总是安全的,即编辑器不会被破坏。 然而,由于编辑器的访问可以触发锁,并发访问可能会产生合同失败或意外结果。

一个编辑器可靠地支持某些并发模式。 一个相关的模式是在一个线程中更新编辑器,同时编辑器显示在一个由不同(处理程序)线程管理的画布中。 为了确保在编辑器被锁定流动时不执行画布刷新,并确保刷新不会阻止编辑器的修改,以下是保证:

  • 当编辑器的 refresh 方法在 edit sequence (由 begin-edit-sequence 开始并由 end-edit-sequence 结束) 中被调用时,所请求的刷新区域被记录下来,但刷新不会被执行。相反,刷新被推迟到编辑序列的结束。

  • 试图在刷新过程中开始编辑序列,会阻止刷新的完成。

  • on-display-size-when-ready 方法只在编辑器没有被刷新时和编辑序列没有在进行中时调用 on-display-size 。 在第一种情况下, on-display-size 调用被委托给刷新线程,在刷新完成后被调用。 在第二种情况下, on-display-size 调用被委托给编辑序列线程,在编辑序列完成后被调用。

因此,禁用 editor-canvas% 对象(使用 enable) 足以确保后台线程可以修改画布显示的编辑器,只要所有修改都在编辑序列中。 后台修改将最小限度地暂时损害画布的刷新,而刷新不会损害后台线程的修改。

第二个支持的模式是在后台线程中读取一个编辑器,而该编辑器可能在其他线程中被操作。 由于没有与 location 无关的读取会引入锁,因此在后台线程中的这种读取不会损害其他线程。 然而,其他线程可能会干扰背景线程,导致它收到错误的或过时的内容信息。 如果后台线程的工作在编辑器被修改时可以被丢弃,那么这种片面的保证是有用的。
