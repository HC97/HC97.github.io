7 Typed Racket 中的优化

关于 Racket 性能和基准测试的一般信息, 请参见 性能 。

Typed Racket 提供了一个类型驱动的优化器,它可以重写类型良好的程序,从而有可能使其更快。

7.1 关闭优化器

Typed Racket 的优化器默认是打开的。 如果你想停用它(例如为了调试),你必须在指定程序的语言时加入 #:no-optimize 关键字:

#lang typed/racket #:no-optimize

当在沙盒中执行 typed racket 程序时 (见 Sandboxed Evaluation) , 以及环境变量 PLT_TR_NO_OPTIMIZE 被设置为(任何值)时,优化器也被禁用。

7.2 充分利用优化器

Typed Racket 的优化器可以提高各种常见的 Racket 习语的性能。 然而,它在某些习语上的工作比其他习语要好。 通过使用正确的习语来编写程序,你可以帮助优化器帮助你。

为了更好地利用 Typed Racket 优化器,请查阅 the Optimization Coach documentation 。

Typed Racket 优化器会记录主题为 'TR-optimizer 的事件。 参见 记录 以了解如何接收这些日志事件。

7.2.1 数字类型

由于是类型驱动的,优化器会根据你分配给数据的类型做出大部分的决定。 因此,你可以通过编写信息丰富的类型来提高优化器的实用性。

例如,下面的程序都进行了类型检查:
(define (f [x : Real])  : Real  (+ x 2.5))
(f 3.5)
(define (f [x : Float]) : Float (+ x 2.5))
(f 3.5)

但是,第二个程序使用了更多的信息类型: Float 类型只包括 64 位浮点数,而 Real 类型包括精确和 inexact real numbers ， Inexact-Real 类型包括 32 位和 64 位浮点数。 Typed Racket的优化器可以优化后者的程序,使其使用 float 运算,而对前者的程序则无能为力。

因此,为了充分利用 Typed Racket 的优化器,你应该尽可能地使用 Float 类型。 出于类似的原因,在进行浮点计算时,你应该使用浮点字词而不是精确字词。

在算术运算中混合使用浮点数和精确实数时,其结果不一定是 Float 。 例如, (* 2.0 0) 的结果是 0 ,这不是一个 Float 。 这可能会导致错过优化。 为了防止这种情况,当浮点数和精确实数混合使用时,应使用 real->double-flonum 将精确实数转换成浮点数。 在使用 + 或 - 时,则不需要这样做。 当混合不同精度的浮点数时,结果尽可能使用最高精度。

类似地,基于同样的原因, Float-Complex 类型比 Complex 类型更适合。 类型化的 Racket 可以保持浮点 complex numbers 不被装箱;因此,使用 complex numbers 的程序可以比将 complex numbers 表示为两个 real numbers 的同等程序有更好的性能。 和浮点字面一样,浮点 complex 字面(如 1.0+1.0i) 应该比精确 complex 字面(如 1+1i) 更受欢迎。

为了充分利用 Typed Racket 的优化器,你也应该选择矩形坐标而不是极坐标。

请注意,在 Racket BC 上,有可能出现一个成分是精确的, 另一个是 inexact 的复数。 这些值,包括写成 +1.0i 的字面符号,都不具有 Float-Complex 类型。 在 Racket CS 上,这种混合精确性的值是不存在的。

7.2.2 列表

Typed Racket 以不同的方式处理可能为空的列表和已知为非空的列表: 当取一个 Typed Racket 知道为非空的列表的 car 或 cdr 时, 它可以跳过通常在调用 car 和 cdr 时对空列表的检查。

(define (sum [l : (Listof Integer)]) : Integer
  (if (null? l)
      0
      (+ (car l) (sum (cdr l)))))

在这个例子中,Typed Racket 知道,如果我们到达 else 分支, l 就不是空的。 与 car 和 cdr 相关的检查将是多余的,因此被取消了。

除了使用 null? 显式地检查空列表外, 你可以通过使用已知长度的列表类型构造函数来通知 Typed Racket 一个列表是非空的; 如果你的数据存储在固定长度的列表中,你可以使用 List 类型构造函数。

例如,两个 Integer 的列表的类型可以写成:

(define-type List-2-Ints (Listof Integer))

或者写成更精确的

(define-type List-2-Ints (List Integer Integer))

使用第二个定义,所有与 car 和 cdr 有关的检查都可以在这个函数中消除:
(define (sum2 [l : List-2-Ints]) : Integer
  (+ (car l) (car (cdr l))))

7.2.3 向量

除了已知长度的列表之外,Typed Racket 还通过 Vector 类型构造函数支持已知长度的向量。 使用常数索引的已知长度向量访问可以以类似 car 和 cdr 的方式进行优化。

; #(color r g b)
(define-type Color (Vector String Integer Integer Integer))
(define x : Color (vector "red" 255 0 0))
(vector-ref x 0) ; good
(define color-name 0)
(vector-ref x color-name) ; good
(vector-ref x (* 0 10)) ; bad

然而,在许多这样的情况下, structs 比向量更合适。 Typed Racket可以在所有情况下优化结构体访问。

7.2.4 合约边界

当与非类型化代码互操作时(见 Typed-Untyped 交互), 契约被安装在类型化和非类型化模块之间。 契约可能有很大的开销,因此在对性能敏感的代码中应该避免类型-非类型边界的交叉。

类型化的 Racket 为 #lang racket 提供的大多数绑定提供了类型; 在这些情况下,使用 require/typed 是不必要的。

如果你怀疑处于类型化-非类型化边界的契约可能会在你的程序中产生巨大的代价,你可以使用 contract profiler 进一步调查。

如果合约分析器还没有安装,下面的命令将安装它:

  raco pkg install contract-profile
