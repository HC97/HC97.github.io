2.14 服务小程序

 (require web-server/dispatchers/dispatch-servlets)
  package: web-server-lib
web-server/dispatchers/dispatch-servlets 模型 定义了一个运行 servlet 的调度器构造函数。

value

url->servlet/c : contract?

相当于 (-> url? servlet?)

procedure

(make-cached-url->servlet url->path 
  path->serlvet) 
 → 
(->* () ((or/c false/c (listof url?)) (-> servlet? void?)) void?)
url->servlet/c
  url->path : url->path/c
  path->serlvet : path->servlet/c
第一个返回值是一个刷新缓存的过程。 如果它的第一个可选参数是 #f (默认),所有的 servlet 缓存都被刷新。 否则,只有那些 url->path 映射到给定 URL 的 servlet 缓存被刷新。 第二个可选参数是一个过程,在刷新前会对每个缓存值进行调用,它可以用来最终确定 servlet 资源。 请注意,默认值 void 不执行任何最终处理。 特别是,它不会关闭 servlet 的托管人,而是允许 servlet 的托管人管理的资源(如线程)持续存在。

第二个返回值是一个过程,使用 url->path 将 URL 解析为路径, 然后使用 path->servlet 将该路径解析为 servlet,将结果缓存在一个内部表格中。

Changed in version 1.3 of package web-server-lib: 为 URL 列表的第一个返回值增加了可选参数。
Changed in version 1.3: 为 servlet finalizer 程序的第一个返回值增加了可选参数。

procedure

(make url->servlet 
  [#:responders-servlet-loading responders-servlet-loading 
  #:responders-servlet responders-servlet]) 
 → dispatcher/c
  url->servlet : url->servlet/c
  responders-servlet-loading : (url? exn? . -> . can-be-response?)
   = servlet-loading-responder
  responders-servlet : (url? exn? . -> . can-be-response?)
   = servlet-error-responder
这个调度器运行 url->servlet ,使用 url->servlet 来解析 URL 到底层的 servlets。 如果 servlets 在加载时出现错误,那么就使用 responders-servlet-loading 。 其他错误用 responders-servlet 来处理。 如果一个 servlet 引发了对 next-dispatcher 的调用,那么该信号将由该 dispatcher 传播。

2.14.2 Servlet 命名空间

 (require web-server/configuration/namespace)
  package: web-server-lib
本模块提供了一个函数,帮助创建 web-server/dispatchers/dispatch-servlets 的 make 函数所需要的 make-servlet-namespace 程序。

value

make-servlet-namespace/c : contract?

相当于
(->* ()
     (#:additional-specs (listof module-path?))
     namespace?)

procedure

(make-make-servlet-namespace #:to-be-copied-module-specs to-be-copied-module-specs)

 → make-servlet-namespace/c
  to-be-copied-module-specs : (listof module-path?)
这个函数创建了一个函数,当被调用时将构建一个新的 namespace , 该命名空间拥有所有来自 to-be-copied-module-specs 和 additional-specs 的模块, 以及 racket 和 mred, 只要它们已经附在调用站点的 (current-namespace) 上。

例子:
(make-make-servlet-namespace
 #:to-be-copied-module-specs `((lib "database.rkt" "my-module")))

2.14.2.1 为什么这很有用

每个 servlet 都需要一个不同的命名空间,这样, 如果 servlet A 和 servlet B 都使用一个有状态的模块 C,它们就会被相互隔离起来。 我们把 Web Server 看作是 Servlet 的操作系统,所以我们继承了操作系统的隔离要求。

然而,有一些模块必须是共享的。 如果不是这样,那么结构就不能从 Web Server 传递给 servlets,因为 Racket 的结构是生成的。

因为,有时候,用户实际上希望 servlet A 和 B 通过模块 C 进行交互。 通过这个过程,可以创建一个自定义的 make-servlet-namespace ,将模块 C 附加到所有的 servlet 命名空间。 通过其他方式(见 调度器),不同的 servlet 集合可以共享不同的模块集合。

2.14.3 内部 Servlet 表示法

 (require web-server/private/servlet)
  package: web-server-lib

struct

(struct servlet (custodian namespace manager directory handler)
    #:extra-constructor-name make-servlet
    #:mutable)
  custodian : custodian?
  namespace : namespace?
  manager : manager?
  directory : path-string?
  handler : (request? . -> . can-be-response?)
这个结构的实例保存了 servlet 的必要部分:负责 servlet 资源的 custodian 、 servlet 在其中执行的 namespace 、 负责 servlet 延续的 manager 、servlet 的当前 directory , 以及对 servlet 的所有请求的 handler 。
