5 任意精度浮点数 (Bigfloats)

Neil Toronto <ntoronto@racket-lang.org>

 (require math/bigfloat) package: math-lib

这个库为 MPFR 提供了一个类型化的 Racket 接口,MPFR 是一个 C 库,它提供了

  • 一个任意精度的浮点数的 C 类型。

  • 高效且被证明正确的基本和特殊函数。

  • 定义明确的语义,符合最新的 IEEE 754 标准。

MPFR 提供和操作的任意精度浮点数是由 Bigfloat 类型表示,并由谓词 bigfloat? 标识。

除了少数例外,bigfloat 函数将其参数视为精确参数,而不管其精度如何。 从概念上讲,它们使用无限多的比特来计算精确的结果, 并通过使用 (bf-rounding-mode) 进行四舍五入来返回具有 (bf-precision) 比特的结果。 在实践中,它们使用的有限算法已被艰苦地证明等同于概念上的无限过程。

MPFR 是免费的,并且与商业软件的许可兼容。 它与 Windows 和 Mac OS X 的 Racket 一起发布,安装在大多数 Linux 系统上,并且在主要的类 Unix 平台上 easy to install 。

5.1 快速入门

  1. 使用 (bf-precision <some-number-of-bits>) 设置 bigfloat 函数的结果精度。

  2. 使用 bf 将实数和格式良好的字符串转换为 bigfloats。

  3. 使用 bf 函数(如 bf+ 和 bfsin )对大浮点数进行操作。

  4. 使用 bigfloat->real, bigfloat->flonum 和 bigfloat->integer 将 bigfloats 转换为实数。 使用 bigfloat->string 对它们进行格式化显示。

关于例子,请继续阅读 FAQ。

5.2 虚构的问题

为什么使用 math/bigfloat?

有几个原因。

原因: 对于你的应用来说,flonum 的精度要么太高,要么太低。

Examples:
> (flsqrt 3.0)

1.7320508075688772

> pi

3.141592653589793

> (bf-precision 16)
> (bfsqrt (bf 3))

(bf #e1.73206)

> (bf-precision 179)
> pi.bf

(bf #e3.141592653589793238462643383279502884197169399375105819)

一个 flonum 有一个 53 位的显式(我们会说它有 53 位的 precision) 和一个 11 位的 exponent 。 一个 bigfloat 有一个至少 2 位的任意精度和一个 31 位的指数。

原因: 为了有把握地计算大得离谱的数字或小数字。

Examples:
> (bf-precision 128)
> (bfexp (bfexp (bfexp (bf 3))))

(bf "2.050986436051648895105860942072054674579e229520860")

> (bflog (bflog (bflog (bfexp (bfexp (bfexp (bf 3)))))))

(bf 3)

原因: 为了验证你的浮点硬件。

IEEE 754-2008 规定, 符合要求的实现必须对所有操作的结果进行正确的舍入。 粗略地说,结果不能超过半位,其中有问题的位是显值中最没有意义的位。

当然,实现并不总是遵守标准。例如,在我的旧笔记本电脑上, 评估 (exp 400) 的结果是 5.221469689764346e+173 。 请注意显性数字中的最后四个小数位: 4346 。但它们应该是 4144:
> (bf-precision 53)
> (bigfloat->flonum (bfexp (bf 400)))

5.221469689764144e+173

我的新笔记本计算 5.221469689764144e+173 是正确的。

原因: 为了控制最小有效位的舍入。

IEEE 754 为 flonum 结果的最小位提供了不同的四舍五入模式,比如四舍五入到偶数和向零舍入。 我们可以利用这一点来正确实现区间算术,通过向下舍入下限和向上舍入上限。 但是没有一种可移植的方法来设置四舍五入的模式!

MPFR 允许任何操作的舍入模式不同, math/bigfloat 使用参数 bf-rounding-mode 暴露了这种能力。

When shouldn’t I use math/bigfloat?

当你需要原始速度时。Bigfloat 函数可能比 flonum 函数慢数百到数千倍。

这并不是说它们 效率低下 。 例如, bflog 实现了最著名的渐近复杂度的算法。 它只是不能直接在硬件上运行,也不能采取只用固定精度的快捷方式。

Why are there junk digits on the end of (bf 1.1)?

这大约是 flonum 1.1 的值。 使用 (bf #e1.1) 或 (bf "1.1") 来消除这些垃圾。 一般来说,你应该倾向于将精确的有理数和字符串转换为大浮点数。

Why is the last digit of pi.bf not rounded correctly?

除了最后一位以外的所有 位 都是精确的,最后一位被正确地舍入。 这并不能保证最后一位数字也是如此。

一个十进制数字最多代表 log(10)/log(2)≈3.3 位。 这是一个无理数,所以除了小数点之外,小数/位的边界从来都是一致的。 因此,任何 bigfloat 的最后一位小数必须代表少于 3.3 位,所以它经常是错误的。 但是最后一 bit 才是最重要的。

* 如果你在测试一个函数的 flonum 实现和一个 bigfloat 实现,这可能是一个好主意。

从 flonum 字头转换通常是个坏主意*,因为flonum 只有 53 位精度。 更倾向于将精确的有理数和字符串传递给 bf 。

双参数变体将有符号的显著数 sig 和 2 的幂 exp 转换为大浮点数。 一般来说, (bf sig exp) = (bf (* sig (expt 2 exp))), 但是双参数变量要快得多,特别是对于大的 exp 。

Examples:
> (bf 200 56)

(bf 14411518807585587200)

> (bf (* 200 (expt 2 56)))

(bf 14411518807585587200)

bfrandom 函数以这种方式使用双参数变体生成 0 到 1 之间的随机大浮点数:
> (require (only-in math/base random-bits))
> (bf-precision 64)
> (bf (random-bits 64) -64)

(bf #e0.416872969910248753552)

返回一个在[0,1]区间内均匀分布的随机大浮点数。

procedure

(bfcopy x) → Bigfloat

  x : Bigfloat
在当前的 bf-rounding-mode 下, 返回最接近 x 的 (bf-precision)-bit bigfloat 。

以更高的精度计算 bigfloats 的一个常见模式是
> (bf-precision 64)
> (bfcopy
   (parameterize ([bf-precision  (+ (bf-precision) 10)])
     (bf/ (bf+ 1.bf (bfsqrt 5.bf)) 2.bf)))

(bf #e1.61803398874989484821)

这个例子计算黄金比例 (phi.bf) 时比要求的多了 10 位,以弥补三舍五入的误差。

将 bigfloat 分别转换为整数、精确有理数、实数和楞次值。

bigfloat->integer, bigfloat->rational 和 bigfloat->real 的返回值可以用 bf 精确地转换为 x 。 对于前两个函数,如果 x 不是整数或有理数,就会引发一个错误。 另一方面, bigfloat->real 返回 +inf.0, -inf.0 或 +nan.0 , 如果 x 不是有理的大浮点数。

bigfloat->flonum 将 x 舍入到 53 位精度, 以适应flonum中的值,使用 bf-rounding-mode 的当前值。

> (bf-precision 64)
> (bigfloat->integer (bf 21/10))

bigfloat->integer: contract violation

  expected: bfinteger?

  given: (bf #e2.09999999999999999991)

> (bigfloat->integer (bfround (bf 21/10)))

2

> (define x (bf 1/7))

> (bigfloat->flonum x)

0.14285714285714285

> (bigfloat->rational x)

10540996613548315209/73786976294838206464

> (rationalize (bigfloat->rational x) (expt 2 (- (bf-precision))))

1/7

> (bf= x (bf (bigfloat->rational x)))

#t

对精确数转换要小心。 具有大指数的大浮点可能不适合作为整数或精确的有理数放在内存中。 更糟的是,它们可能适合,但却把你所有的 RAM 和交换空间当作午餐。

由 bigfloat->string 返回的字符串包括足够多的数字, string->bigfloat 可以精确地重建 bigfloat。 换句话说, string->bigfloat 是 bigfloat->string 的一个左逆。

> (bigfloat->string (bf 4))

> (bigfloat->string (bf 1/10000))

> (string->bigfloat "0.14285714285714285714")

(bf #e0.142857142857142857141)

> (string->bigfloat "square root of two")

> (string->bigfloat (bigfloat->string pi.bf))

(bf #e3.14159265358979323851)

(bf #e3.14159265358979323851)

5.5 参数

parameter

(bf-precision) → Integer

(bf-precision bits) → void?
  bits : Integer
一个参数,决定了大多数 bigfloat 函数返回的 bigfloats 的精度。 对于不使用 bf-precision 的函数,文档中会指出例外情况。

对于非零的、有理的大浮点数,比特数 bits 包括前面的一位。 例如,为了模拟 64 位浮点,使用 (bf-precision 53) , 即使 flonums 有 52 位的显式,因为这一位是隐含在 flonum 中的。

这个参数有一个保护措施,确保 (bf-precision) 在 bf-min-precision 和 bf-max-precision 之间。

value

bf-min-precision : Exact-Positive-Integer

procedure

(bfcanonicalize x) → Bigfloat

  x : Bigfloat
大浮点数在散列前被规范化,以确保平等意味着相等的散列。 如果 x 是非零并且是有理数,则返回一个新的 bigfloat, 其精度不超过精确编码 x 所需的位数,方法是将所有低阶零从显式中移除并调整指数。

对于零或无理数的l x, 返回 -inf.bf, -0.bf, 0.bf, +inf.bf 或 +nan.bf, 取决于 x 的值。

两个非零的、有理的大浮点数是否 equal?。 当且仅当它们的规范化显数和指数相等时。 两个零或非有理的大浮点数是 equal? 。 当且仅当它们的规范化是 eq? 。

对大浮点数进行规范化处理并不会改变从它们计算出来的答案。

Examples:
> (bf-precision 64)
> (define x (bf 1 -2))

> x

(bf #e0.25)

> (bfcanonicalize x)

(bf #e0.25)

> (bigfloat-precision x)

64

> (bigfloat-precision (bfcanonicalize x))

2
