5 类型化单元

警告: 本节描述的功能是实验性的,可能无法正常工作。 一些功能可能会在下一个版本中改变。

Typed Racket提供了对模块化编程的支持,其单元和签名由 racket/unit 库提供。

5.1 特殊形式

 (require typed/racket/unit) package: typed-racket-lib

下面的特殊形式是由 typed/racket/unit 和 typed/racket 模块提供的, 但不是由 typed/racket/base 提供。 typed/racket/unit 模块另外还提供了 racket/unit 的所有其他绑定方式。

syntax

(define-signature id extension-decl
  (sig-elem ...))
 
extension-decl = 
  | extends sig-id
     
sig-elem = [id : type]
将一个标识符绑定到一个签名上,并在签名环境中用指定的类型绑定注册该标识符。 Typed Racket 中的 Sigantures 只允许规范变量和它们的类型。 变量和语法的定义在 define-signature 形式中是不允许的。 这只是 Typed Racket 中 define-signature 形式的一个限制。

和非类型化 Racket 一样, extends 子句包括了扩展签名的所有元素,任何新签名的实现都可以作为扩展签名的实现。

syntax

(unit
  (import sig-spec ...)
  (export sig-spec ...)
  init-depends-decl
  unit-body-expr-or-defn
  ...)
 
sig-spec = sig-id
  | (prefix id sig-spec)
  | (rename sig-spec (id id) ...)
  | (only sig-spec id ...)
  | (except sig-spec id ...)
     
init-depends-decl = 
  | (init-depend sig-id ...)
Racket unit 形式的类型化版本。 Typed Racket 中的单元表达式不支持用 tag 关键字标记的签名。

syntax

(invoke-unit unit-expr)

(invoke-unit unit-expr (import sig-spec ...))
Racket invoke-unit 形式的类型化版本。

syntax

(define-values/invoke-unit unit-expr
  (import def-sig-spec ...)
  (export def-sig-spec ...))
 
def-sig-spec = sig-id
  | (prefix id def-sig-spec)
  | (rename def-sig-spec (id id) ...)
Racket define-values/invoke-unit 形式的类型化版本。 在类型化的Racket中, define-values/invoke-unit 只允许在模块的顶层。

syntax

(compound-unit
  (import link-binding ...)
  (export link-id ...)
  (link linkage-decl ...))
 
link-binding = (link-id : sig-id)
     
linkage-decl = ((link-binding ...) unit-expr link-id ...)
Racket compound-unit 形式的类型化版本。

syntax

(define-unit unit-id
  (import sig-spec ...)
  (export sig-spec ...)
  init-depends-decl
  unit-body-expr-or-defn
  ...)
Racket define-unit 形式的类型化版本。

syntax

(compound-unit/infer
  (import infer-link-import ...)
  (export infer-link-export ...)
  (link infer-linkage-decl ...))
 
infer-link-import = sig-id
  | (link-id : sig-id)
     
infer-link-export = link-id
  | sig-id
     
infer-linkage-decl = 
((link-binding ...) unit-id
                    tagged-link-id ...)
  | unit-id
Racket compound-unit/infer 形式的类型化版本。

syntax

(define-compound-unit id
  (import link-binding ...)
  (export link-id ...)
  (link linkage-decl ...))
Racket define-compound-unit 形式的类型化版本。

syntax

(define-compound-unit/infer id
  (import link-binding ...)
  (export infer-link-export ...)
  (link infer-linkage-decl ...))
Racket define-compound-unit/infer 形式的类型化版本。

syntax

(invoke-unit/infer unit-spec)

 
unit-spec = unit-id
  | (link link-unit-id ...)
Racket invoke-unit/infer 形式的类型化版本。

syntax

(define-values/invoke-unit/infer maybe-exports unit-spec)

 
maybe-exports = 
  | (export sig-sepc ...)
     
unit-spec = unit-id
  | (link link-unit-id ...)
Racket define-values/invoke-unit/infer 形式的类型化版本。 和上面的 define-values/invoke-unit 形式一样,这种形式只允许在模块的顶层。

syntax

(unit-from-context sig-spec)

5.2 类型

syntax

(Unit
  (import sig-id ...)
  (export sig-id ...)
  optional-init-depend-clause
  optional-body-type-clause)
 
optional-init-depend-clause = 
  | (init-depend sig-id ...)
     
optional-body-type-clause = 
  | type
  | (Values type ...)
具有给定的导入、导出、初始化依赖和主体类型的单元的类型。 省略 init-depend 子句等同于不包含签名的 init-depend 子句。 主体类型是单元主体中最后一个表达式的类型。 如果一个单元只包含定义而没有表达式,那么它的主体类型是 Void 。 省略主体类型等同于指定主体类型为 Void 。

Example:
> (module Unit-Types typed/racket
    (define-signature fact^ ([fact : (-> Natural Natural)]))
    (: use-fact@ (Unit (import fact^)
                       (export)
                       Natural))
    (define use-fact@ (unit (import fact^) (export) (fact 5))))

syntax

UnitTop

所有单元类型的超级类型。这种类型的值不能被链接或调用。 它的主要用途是反射操作 unit?

5.3 与非类型化代码交互

syntax

(require/typed m rt-clause ...)

 
rt-clause = [maybe-renamed t]
  | 
[#:struct name ([f : t] ...)
     struct-option ...]
  | 
[#:struct (name parent) ([f : t] ...)
     struct-option ...]
  | [#:opaque t pred]
  | [#:signature name ([id : t] ...)]
     
maybe-renamed = id
  | (orig-id new-id)
     
struct-option = #:constructor-name constructor-id
  | #:extra-constructor-name constructor-id

require/typed 的 #:signature 子句要求给定的签名,并在签名环境中以指定的绑定方式注册它。 与 require/typed 要求的其它标识符不同,签名不受契约的保护。

签名不是运行时的值,因此不需要被合约保护。

Examples:
> (module UNTYPED-1 racket
    (provide a^)
    (define-signature a^ (a)))
> (module TYPED-1 typed/racket
    (require/typed 'UNTYPED-1
                   [#:signature a^ ([a : Integer])])
    (unit (import a^) (export) (add1 a)))

Typed Racket 将推断出命名的签名是否 extends 了另一个签名。 要求一个扩展了签名环境中不存在的签名的签名是一个错误。

Examples:
> (module UNTYPED-2 racket
    (provide a-sub^)
    (define-signature a^ (a1))
    (define-signature a-sub^ extends a^ (a2)))
> (module TYPED-2 typed/racket
    (require/typed 'UNTYPED-2
                   [#:signature a-sub^
                     ([a1 : Integer]
                      [a2 : String])]))

eval:6:0: Type Checker: Error in macro expansion -- required

signature extends an untyped signature

  required signature: a-sub^

  extended signature: a^

  in: UNTYPED-2

在 Typed Racket 中,从一个包含变量定义的未类型化模块中要求签名是一个错误。

Examples:
> (module UNTYPED racket
    (provide bad^)
    (define-signature bad^ (bad (define-values (bad-ref) (car bad)))))
> (module TYPED typed/racket
    (require/typed 'UNTYPED
                   [#:signature bad^
                     ([bad : (Pairof Integer Integer)]
                      [bad-ref : Integer])]))

eval:8:0: Type Checker: Error in macro expansion -- untyped

signatures containing definitions are prohibited

  in: UNTYPED

5.4 限制

5.4.1 签名形式

与 Racket 的 define-signature 形式不同,在 Typed Racket 中, define-signature 只支持一种签名元素,指定签名中变量的类型。 特别是 Typed Racket 的 define-signature 形式不支持使用 define-syntaxes, define-values 或 define-values-for-export 。 要求在类型化模块中包含定义的非类型化签名将导致一个错误。

Examples:
> (module UNTYPED racket
    (provide bad^)
    (define-signature bad^ ((define-values (bad) 13))))
> (module TYPED typed/racket
    (require/typed 'UNTYPED
                   [#:signature bad^ ([bad : Integer])]))

eval:10:0: Type Checker: Error in macro expansion -- untyped

signatures containing definitions are prohibited

  in: UNTYPED

5.4.2 合约和单元静态信息

在类型化和非类型化上下文之间流动的单元值被包裹在 unit/c 合同中,以保护单元的导入、导出和调用时的结果。 当标识符被额外绑定到单元的静态信息时,比如那些由 define-unit 定义的标识符, 在类型化和非类型化上下文之间流动时,合约的应用会导致静态信息变得不可访问。

Examples:
> (module UNTYPED racket
    (provide u@)
    (define-unit u@ (import) (export) "Hello!"))
> (module TYPED typed/racket
    (require/typed 'UNTYPED
                   [u@ (Unit (import) (export) String)])
    (invoke-unit/infer u@))

eval:12:0: untyped-invoke-unit/infer: unknown unit

definition

  at: u@

  in: (untyped-invoke-unit/infer u@)

然而,当与静态单元信息绑定的标识符从类型化模块流向非类型化模块时,情况会更糟。 因为单元静态信息是作为宏定义绑定在标识符上的,在非类型化上下文中不允许使用类型化的单元。

Examples:
> (module TYPED typed/racket
    (provide u@)
    (define-unit u@ (import) (export) "Hello!"))
> (module UNTYPED racket
    (require 'TYPED)
    u@)

eval:14:0: Type Checker: Macro u@ from typed module used in

untyped code

  in: u@

5.4.3 签名和内部定义上下文

Typed Racket 的 define-signature 形式在顶层和内部定义语境中都是允许的。 正如下面的例子所示,在内部定义上下文中定义签名可能是有问题的。

Example:
> (module TYPED typed/racket
    (define-signature a^ ())
    (define u@
      (let ()
        (define-signature a^ ())
        (unit (import a^) (export) (init-depend a^) 5)))
    (invoke-unit u@ (import a^)))

eval:15:0: Type Checker: type mismatch

  expected: (Unit (import a^) (export) (init-depend a^)

AnyValues)

  given: (Unit (import a^) (export) (init-depend a^)

Positive-Byte)

  in: a^

尽管这个单元导入了一个名为 a^ 的签名,但为导入提供的 a^ 指的是顶层的 a^ 签名, 类型系统会阻止调用这个单元。 这个问题可以通过只在模块的顶层定义签名来避免。

5.4.4 标签签名

Racket 中的各种单元形式允许对签名进行标记,以支持多次导入或导出相同签名的单元的定义。 类型化的 Racket 不支持在上述各种单元形式的任何地方使用标记的签名,使用 tag 关键字。

5.4.5 结构匹配和其他单元形式

Typed Racket 只支持上面描述的那些单元形式。 所有其他由 racket/unit 导出的绑定在类型系统中不被支持。 特别是,包括 unit/new-import-export 和 unit/s 在内的结构匹配形式不被支持。
