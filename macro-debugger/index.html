宏程序调试器: 检查宏程序扩展

Ryan Culpepper


吴华均 译

宏调试器集合包含两个工具:一个用于宏扩展的步进器和一个独立的语法浏览器。 宏扩展器向程序员显示程序的扩展,作为一连串的改写步骤,使用语法浏览器来显示各个术语。 语法浏览器使用颜色和属性面板来显示术语的语法属性,如词法绑定信息和源位置。

1 宏程序步进器

 (require macro-debugger/stepper) package: macro-debugger

procedure

(expand/step stx) → void?

  stx : any/c
展开语法(或 S 表达式),并打开一个宏步进器框架,用于步进扩展。

procedure

(expand-module/step mod) → void?

  mod : module-path?
扩展由 mod 命名的源文件,该文件必须包含一个模块声明,并打开一个宏步进器框架,用于步进扩展。

procedure

(macro-stepper-repl [new-repl?    
  #:eval? eval?]) → void?
  new-repl? : any/c = #f
  eval? : any/c = #t
创建一个宏步进框架,并启动一个 read-eval-print 循环,显示每个输入到 repl 中的表达式的扩展。 如果 new-repl? 为真,则通过调用 (read-eval-print-loop) 创建一个新的 repl; 否则,当前的 repl 被重新使用。 如果 eval? 为真,那么表达式在扩展后被评估;否则,只有编译时的部分被评估。

repl 是通过安装一个自定义的 evaluation handler 来实现的,该处理程序与原始处理程序相连接来进行评估。

2 宏程序扩展工具

 (require macro-debugger/expand)
  package: macro-debugger-text-lib

这个模块提供了类似于 expand 的程序,允许用户指定其扩展应该被隐藏的宏。

警告:由于宏扩展被有选择地隐藏的方式的限制,产生的语法可能无法评估出与原始语法相同的结果。

procedure

(expand-only stx transparent-macros) → syntax?

  stx : any/c
  transparent-macros : (listof identifier?)
展开给定的语法 stx, 但只显示名称出现在 transparent-macros 中的宏的扩展。

Example:
> (syntax->datum
   (expand-only #'(let ([x 1] [y 2]) (or (even? x) (even? y)))
                (list #'or)))

'(let ((x 1) (y 2))

   (let ((or-part (even? x))) (if or-part or-part (#%expression (even? y)))))

procedure

(expand/hide stx hidden-macros) → syntax?

  stx : any/c
  hidden-macros : (listof identifier?)
扩展给定的语法 stx,但隐藏给定标识符列表中的宏的扩展(从概念上讲,是仅扩展的补充)。

Example:
> (syntax->datum
   (expand/hide #'(let ([x 1] [y 2]) (or (even? x) (even? y)))
                (list #'or)))

'(let-values (((x) '1) ((y) '2)) (or (#%app even? x) (#%app even? y)))

procedure

(expand/show-predicate stx show?) → syntax?

  stx : any/c
  show? : (-> identifier? boolean?)
扩展给定的语法 stx, 但只显示其名称满足谓词 show? 的宏的扩展。

Example:
> (syntax->datum
   (expand/show-predicate
    #'(let ([x 1] [y 2]) (or (even? x) (even? y)))
    (lambda (id) (memq (syntax-e id) '(or #%app)))))

'(let ((x 1) (y 2))

   (let ((or-part (#%app even? x)))

     (if or-part or-part (#%expression (#%app even? y)))))

3 Macro 的 Macro 步进器 API

 (require macro-debugger/emit) package: macro-debugger

宏可以通过使用本模块中的程序显式地将信息发送给监听的宏步进器。

procedure

(emit-remark fragment ... [#:unmark? unmark?]) → void?

  fragment : 
(letrec ([emit-arg/c
          (recursive-contract
           (or/c string?
                 syntax?
                 (listof emit-arg/c)
                 (-> emit-arg/c)))])
  emit-arg/c)
  unmark? : boolean? = (syntax-transforming?)
向宏引导器(如果有一个正在监听)发出一个事件,包含给定的字符串和语法对象。 宏引导器通过打印字符串和语法对象,在宏的上下文渲染上显示一个注释。 只有当发出该注释的宏被隐藏策略认为是透明的,才会显示该注释。

默认情况下,备注中的语法对象已经应用了转化器的标记(使用 syntax-local-introduce) , 因此它们在宏引导器中的外观与转化器返回后的外观一致。如果 unmark? 是 #f 的话, 就会抑制未标记的情况。

(define-syntax (mymac stx)
  (syntax-case stx ()
    [(_ x y)
     (emit-remark "I got some arguments!"
                  #'x
                  "and"
                  #'y)
     #'(list 'x 'y)]))
(mymac 37 (+ 1 2))

(在宏观步进中运行上面的片段)。

procedure

(emit-local-step before after #:id id) → void?

  before : syntax?
  after : syntax?
  id : identifier?
发出一个事件,模拟一个从 before 到 after 的局部扩展步骤。

before 和 after 的术语被标记为 syntax-local-introduce , 因此它们在宏步进器中出现,就像该步骤真正由本地扩展产生一样。

id 参数作为该步骤的 “macro” ,用于宏隐藏的目的。

4 Macro 步进器文本界面

 (require macro-debugger/stepper-text)
  package: macro-debugger-text-lib

procedure

(expand/step-text stx [show?]) → void?

  stx : any/c
  show? : 
(or/c (-> identifier? boolean?)
      (listof identifier?))
 = (lambda (x) #t)
展开语法并打印出宏扩展步骤。 如果给定了标识符谓词,它将决定显示哪些宏(如果没有,则显示所有宏)。 也接受一个标识符的列表。

Example:
> (expand/step-text #'(let ([x 1] [y 2]) (or (even? x) (even? y)))
                    (list #'or))

Macro transformation

(let ((x 1) (y 2)) (or (even? x) (even? y)))

  ==>

(let

 ((x 1) (y 2))

 (let:1 ((or-part:1 (even? x))) (if:1 or-part:1 or-part:1 (or:1 (even? y)))))

Macro transformation

(let

 ((x 1) (y 2))

 (let:1 ((or-part:1 (even? x))) (if:1 or-part:1 or-part:1 (or:1 (even? y)))))

  ==>

(let

 ((x 1) (y 2))

 (let:1

  ((or-part:1 (even? x)))

  (if:1 or-part:1 or-part:1 (#%expression:2 (even? y)))))

procedure

(stepper-text stx [show?]) → (symbol? -> void?)

  stx : any/c
  show? : 
(or/c (-> identifier? boolean?)
      (listof identifier?))
 = (lambda (x) #t)
返回一个过程,该过程可以在符号 'next 上调用,以打印下一个步骤, 或者在符号 'all 上调用,以打印出所有剩余步骤。

5 语法浏览器

 (require macro-debugger/syntax-browser)
  package: macro-debugger

procedure

(browse-syntax stx) → void?

  stx : syntax?
创建一个框架,显示给定的语法对象。 关于使用 GUI 的更多信息请见下文。

procedure

(browse-syntaxes stxs) → void?

  stxs : (listof syntax?)
和 browse-syntax 一样,但在同一框架内显示多个语法对象。 两者之间共享着色分区,显示不同语法对象中子词之间的关系。

6 使用 Macro Stepper

6.1 导航

步进器将扩展显示为一个线性的改写过程序列,它为用户提供了向前或向后的控制,以及跳到扩展过程的开始或结束的控制。

如果宏程序显示的是多个扩展,那么它还提供了 "上一个术语" 和 "下一个术语" 按钮, 以便在扩展列表中向上和向下走。 水平线将当前的扩展与其他的扩展划清界限。

6.2 Macro 隐藏

宏隐藏让我们看到,如果某些宏实际上是原始的语法形式,扩展会是什么样子。 宏步进器会跳过你指定为不透明的宏的扩展,但它仍然显示其子项的扩展。

宏引导器的底部面板控制宏隐藏策略。 用户通过在语法浏览器窗格中选择一个标识符,然后点击 "隐藏模块"、"隐藏宏" 或 "显示宏" 中的一项来改变策略。 新的规则出现在策略显示中,用户随后可以使用 "删除 "按钮将其删除。

步进器还提供了更粗略的选项,可以同时隐藏模块的集合。 这些选项的优先级比上面的规则低。

宏隐藏,即使没有标记不透明的宏,也会隐藏某些其他类型的步骤: 内部定义不会被改写为 letrecs,开始形式不会被拼接到模块或块体中,等等。

7 使用语法浏览器

7.1 选择

选择是由粗体字表示的。

用户可以点击子词的任何部分来选择它。 要选择一个括号内的子项,请点击括号中的任何一个。 选定的语法是加粗的。 由于一个语法对象可能会出现在其他多个语法对象中, 所以点击一个出现的语法对象会导致所有出现的语法对象都被加粗。

语法浏览器在右边的属性面板上显示所选语法对象的信息,当该面板显示时。 所选的语法也决定了二级分区所做的高亮显示(见下文)。

7.2 主分区

主分区由前景颜色表示。

如果两个语法子词有相同的标记,主分区总是为它们分配相同的颜色。 在没有不卫生的宏的情况下,这意味着具有相同前景颜色的子词要么存在于原始的预扩展语法中, 要么由同一个宏转换步骤生成。

用黑色表示的语法总是对应于未标记的语法。 这种语法可能是原始的,也可能是由非卫生的宏扩展产生的。

注意:即使是具有相同标记的术语,也可能不会相互 bound-identifier=? , 因为它们可能出现在不同的环境中。

7.3 二级分区

用户可以通过语法菜单选择一个二级分区。 这种分区只适用于标识符。 当用户选择一个标识符时,与所选术语处于同一等价类的所有术语都以黄色高亮显示

可用的二级分区是:

  • bound-identifier=?

  • free-identifier=?

7.4 属性

当显示属性窗格时,它显示所选语法对象的属性。 属性窗格有两个标签页:

  • Term:

    如果选择的是一个标识符,则显示与该语法对象相关的绑定信息。 更多信息请参见 identifier-binding 等。

  • Syntax Object:

    显示源位置信息和语法对象携带的其他属性(见 syntax-property) 。

7.5 解释句法

语法对象的绑定信息可能与它所代表的程序的绑定结构并不相同。 一个程序的绑定结构只有在宏扩展完成后才能确定。

8 寻找无用的 require

 (require macro-debugger/analysis/check-requires)
  package: macro-debugger-text-lib

"Check Requires" 工具可以作为 raco 的一个子命令运行。 例如(从 racket 根目录):

  raco check-requires racket/collects/syntax/*.rkt

  raco check-requires -kbu openssl

每个参数都被解释为一个文件路径,如果它存在的话;否则,它被解释为一个模块路径。 参见 check-requires 以了解输出格式的描述,脚本推荐中的已知限制等。

procedure

(check-requires module-to-analyze    
  [#:show-keep? show-keep?    
  #:show-bypass? show-bypass?    
  #:show-drop? show-drop?    
  #:show-uses? show-uses?]) → void?
  module-to-analyze : module-path?
  show-keep? : boolean? = #t
  show-bypass? : boolean? = #t
  show-drop? : boolean? = #t
  show-uses? : boolean? = #f
分析 module-to-analyze, 检测无用的需求。 每个由 module-to-analyze 导入的模块都被分类为 KEEP, BYPASS, 或 DROP 之一。 对于每个需要的模块,会打印出一行或多行模块的分类和支持信息。 当通过 raco check-requires 运行时,默认只打印 DROP 建议。 通过 show-keep?,show-bypass? 和 show-drop? ,可以根据分类抑制输出。

不分析 for-label 需要的模块。

KEEP req-module at req-phase

在 req-phase 阶段的模块 req-module 的 require 必须被保留, 因为其中定义的绑定被使用。

如果 show-uses? 为真, module-to-analyze 对 req-module 的依赖会被列举出来,每行一个,格式如下:

exp-name at use-phase (mode ...) [RENAMED TO ref-name]

表示在使用阶段 use-phase 使用了名为 exp-name 的出口 (如果 req-phase 为非零,则不一定是它被提供的阶段)。

modes 表示观察到的是哪种依赖关系:作为 reference 使用, 出现在语法模板(quote-syntax)中,等等。

如果 RENAMED TO 子句出现,表明绑定在导入模块时被重新命名, 并且 ref-name 给出了使用的本地名称 (exp-name 是 req-module 提供绑定的名称)。

BYPASS req-module at req-phase

require 被使用,但只用于那些可以通过一个或多个其他模块更直接地获得的绑定。 例如, racket 的使用可能会被绕过,而选择 racket/base, racket/match 和 racket/contract 等等。

一个替换要求的列表被给出,每行一个,格式如下:

TO repl-module at repl-phase [WITH RENAMING]

在 repl-phase 阶段添加 repl-module 的一个需求。 如果 show-uses? 为真,那么在每一行 TO 后面都有一个由 repl-module 满足的依赖关系的列举,其格式与下面 KEEP 中描述的一样。

如果 WITH RENAMING 子句出现, 表明至少有一个替换模块提供了一个不同于模块中本地使用的名字的绑定。 要么应该改变引用,要么应该根据需要在替换模块中使用 rename-in 。

旁路推荐受到以下规则的限制:

  • repl-module 不能涉及从 req-module 跨越一个新的 private 目录

  • repl-module 不能是一个内置的 (“#%”) 模块

  • req-module 不能在 "no-bypass" 白名单中

DROP req-module at req-phase

这个 require 似乎没有被使用,它可能完全可以被放弃。

由于其实现策略的限制, check-requires 偶尔会建议放弃或绕过一个不应该被放弃或绕过的模块。 以下是这种错误建议的典型原因:

  • 该模块的调用有副作用。例如,模块主体可能会更新一个共享表或执行I/O, 或者它可能会转而要求一个模块这样做。(考虑将该模块添加到白名单中)。

  • 在宏的标识符比较中使用了模块的绑定,例如出现在宏的 "字词列表" 中。 在这种情况下,宏应该用 'disappeared-use 属性来注释其扩展, 其中包含与其字面意义比较的标识符;然而,大多数随意编写的宏不会这样做。 另一方面,宏和它们的字面标识符通常由同一个模块提供,所以这个问题有点不常见。

procedure

(show-requires module-name)

 → 
(listof (list/c 'keep   module-path? number?)
        (list/c 'bypass module-path? number? list?)
        (list/c 'drop   module-path? number?))
  module-name : module-path?
和 check-requires 一样,但是把分析结果作为一个列表返回,而不是打印出来。 该过程以下列格式返回每个(非标签)需求的一个元素:

  • (list 'keep req-module req-phase)

  • (list 'bypass req-module req-phase replacements)

  • (list 'drop req-module req-phase)

Example:
> (show-requires 'framework)

'((keep racket/base 0)

  (keep racket/contract/base 0)

  (keep racket/unit 0)

  (keep racket/class 0)

  (keep racket/gui/base 0)

9 显示模块的依赖性

 (require macro-debugger/analysis/show-dependencies)
  package: macro-debugger-text-lib

“Show Dependencies” 工具可以作为 raco 子命令运行。例如(从 racket 根目录):

  raco show-dependencies -bc racket/collects/openssl/main.rkt

  raco show-dependencies -c --exclude racket openssl

每个参数都被解释为一个文件路径,如果它存在的话;否则被解释为一个模块路径。 参见 show-dependencies 获取输出格式的描述。

procedure

(show-dependencies root    
  ...    
  [#:exclude exclude    
  #:exclude-deps exclude-deps    
  #:show-context? show-context?]) → void?
  root : module-path?
  exclude : (listof module-path?) = null
  exclude-deps : (listof module-path?) = null
  show-context? : boolean? = #f
计算 root 模块所需的模块集。 只有当一个 root 模块是另一个 root 模块的依赖关系时,才会包括在输出中。 计算的依赖关系不包括通过 dynamic-require 或 lazy-require 达到的模块, 也不包括通过 define-runtime-module-path-index 引用的模块, 但包括通过 define-runtime-module-path 引用的模块 (因为这隐含地创建了一个 for-label 依赖关系)。

依赖关系被打印出来,每行一个,格式如下:

dep-module [<- (direct-dependent ...)]

表示 dep-module 被一个或多个 root 模块过渡性地要求。 如果 show-context? 为true,那么将显示 direct-dependent ; 它们是可以从(包括)直接需要 dep-module 的 root 模块到达的模块。

依赖关系是通过删除 exclude 中的模块以及 exclude-deps 中的模块(但不等于)所能达到的任何模块而修剪的。

10 Macro 分析器

宏分析器显示哪些宏对程序的 expanded code size 贡献最大。 当你的程序的编译文件比预期的大时,请使用宏分析器。 (宏分析器不报告扩展时间,但扩展时间一般与代码大小成正比)。

  raco macro-profiler module-path ...

宏分析器的工作原理是使用宏调试器扩展文件,并记录每个宏扩展步骤的术语大小差异。 一个术语的大小是通过计算其对、原子等来计算的。

考虑下面的部分宏扩展:

(m (n x)) ⇒ (o (p y) (n x)) ⇒ (f (p y) (n x) z) ⇒ (f y (n x) z)

m 的 direct 成本是 6 –新术语 (p y) 的大小,再加上一个额外的对, 以便将其纳入 o 参数中。 同样地, o 的直接成本是 2。 p 的直接成本是 -4,因为宏的结果比它的用途小。

一个宏的 total 成本包括它的直接成本 加上 m 引入的代码中任何宏的成本, 但 不 包括来自宏参数的成本,如 (n x) 。 所以 m 的总成本是 6+2-4=4,因为 o 和 p 项是由 m 引入的。 相比之下, o 的总成本只有 2,与直接成本相同。

这里有一些已知的限制:

  • 术语大小是编译后代码大小的一个不完美的代表。 例如,一个宏可能会产生一个大的表达式, 但它知道编译器会通过常量传播和死代码消除将其变成一个小的表达式(见 "宏作者的权利法案")。 剖析器会高估这种宏的代码大小成本。

  • 宏分析器使用作用域来确定宏所引入的术语,因此它可能被某些类型的破坏卫生的宏所混淆。

  • 剖析器计算 local-expand 的成本时,假设它只用在宏的参数上,而且其结果被用在宏的结果中。 违反这一假设的宏会有相应的错误的剖析成本。
