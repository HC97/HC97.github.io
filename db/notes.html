6 笔记

本节讨论了与特定数据库系统有关的问题。

6.1 用于 Postgre SQL 和 My SQL 服务器的本地套接字

PostgreSQL 和 MySQL 服务器有时被默认配置为只监听本地套接字(也称为 "unix domain sockets")。 这个库提供了对 Linux 和 Mac OS 上本地套接字通信的支持。 如果本地套接字通信不可用,服务器必须被重新配置为监听TCP端口。

PostgreSQL 服务器的套接字文件位于 postgresql.conf 服务器配置文件中 unix_socket_directory 变量指定的目录中。 例如,在运行 PostgreSQL 8.4 的 Ubuntu 11.04 上, socket 目录是 /var/run/postgresql , socket 文件是 /var/run/postgresql/.s.PGSQL.5432 。 常见的套接字路径可以使用 postgresql-guess-socket-path 函数自动搜索。

MySQL 服务器的 socket 文件位于 my.cnf 配置文件中 socket 变量指定的路径。 例如,在运行 MySQL 5.1 的 Ubuntu 11.04 上,插座位于 /var/run/mysqld/mysqld.sock 。 MySQL 的常见套接字路径可以使用 mysql-guess-socket-path 函数来搜索。

6.2 PostgreSQL 数据库的字符编码

在大多数情况下,数据库的字符编码是不相关的,因为连接函数在创建连接时总是要求翻译成Unicode(UTF-8)。 然而,如果一个 PostgreSQL 数据库的字符编码是 SQL_ASCII,PostgreSQL 将不尊重连接编码; 相反,它将发送未翻译的八位数,这将导致客户端连接中的数据损坏或内部错误。

要把 PostgreSQL 数据库从 SQL_ASCII 转换为合理的编码, pg_dump 数据库,重新编码转储文件(使用像 iconv 这样的工具), 用想要的编码创建一个新的数据库,然后 pg_restore 从重新编码的转储文件。

6.3 PostgreSQL 认证

PostgreSQL 支持大量的 authentication mechanisms, 由 pg_hba.conf 服务器配置文件控制。 这个库目前适用于以下认证方法:

  • plain (和 ldap, pam, radius): 明文密码, 只有在明确允许的情况下(见 postgresql-connect)

  • md5: MD5-hashed password

  • scram-sha-256: 基于密码的挑战/回应协议。 根据服务器配置和是否使用 TLS,这可能对应于 SCRAM-SHA-256 或 SCRAM-SHA-256-PLUS 。

  • peer: 仅适用于本地套接字

不支持 gss, sspi 和 krb5 方法。

Changed in version 1.2 of package db-lib: 增加了 SCRAM-SHA-256 支持。
Changed in version 1.7: 增加了 SCRAM-SHA-256-PLUS 支持。

6.4 PostgreSQL 的时间戳和时区

PostgreSQL 的 timestamp with time zone 类型与 SQL 标准不一致(可能), 与 time with time zone 不一致,并且可能让 PostgreSQL 的新用户感到困惑。

time with time zone 本质上是一个 time 结构,有一个额外的字段来存储时区偏移。 相比之下,一个 timestamp with time zone 除了 timestamp 字段之外没有其他字段。 相反,它表示它的日期时间字段应该被解释为 UTC 时间。 因此,它代表了一个绝对的时间点,不像 timestamp without time zone, 它代表了某个未知时区的本地日期和时间(可能-希望数据库设计者知道,但是 PostgreSQL 不知道)。

当 timestamp with time zone 从一个没有时区信息的源创建时, 会话的 TIME ZONE 设置被用来将源调整为 UTC 时间。 当源包含时区信息时,它被用来将时间戳调整为 UTC 时间。 在这两种情况下,时区信息被 discarded ,只有 UTC 时间戳被存储。 当 timestamp with time zone 被渲染成文本时,它首先被调整到由 TIME ZONE 设置(或由 AT TIME ZONE) 指定的时区,该偏移量被包含在渲染的文本中。

这个库接收二进制格式的时间戳,所以不应用时区调整,也不包括会话的 TIME ZONE 偏移; 因此查询结果中的所有 sql-timestamp 值的 tz 字段为 0 (对于 timestamp with time zone) 或 #f (对于 timestamp without time zone)。 (该库以前的版本以文本形式发送和接收时间戳,因此它们收到的时间戳有调整的时区。)

6.5 MySQL 认证

从 5.5.7 版本开始,MySQL 支持 authentication plugins 。 这个库支持以下插件:

  • caching_sha2_password: 自 MySQL 8.0 版本以来的默认值

  • mysql_native_password: 自 4.1 和 8.0 之前的 MySQL 版本的默认值

  • mysql_old_password: MySQL 4.1 版本之前的默认值

  • mysql_clear_password: 由 LDAP 和 PAM 认证使用。

caching_sha2_password 认证插件有两条 “paths”; 客户端总是先尝试快速路径,但服务器可能会根据服务器认证缓存的状态,要求它通过慢速路径。 快速路径使用一个挑战-回应协议。慢速路径分为以下几种情况:

  • 通过 unix 套接字或通过带有 TLS 的 TCP 连接到 localhost: 客户只是向服务器发送密码。

  • 通过带 TLS 的 TCP 连接,但不连接到 localhost: 如果 allow-cleartext-password? 参数为真,客户端将密码发送给服务器;否则,会产生一个异常。

  • 通过 TCP 连接而不使用 TLS: 这个库不支持;会产生一个异常。

参见 安全地建立数据库连接 。

Changed in version 1.6 of package db-lib: 增加了对 caching_sha2_password 认证的支持。

6.6 MySQL 连接字符集

这个库与 MySQL 服务器进行通信时,所有字符数据都使用 UTF-8。 MySQL 有 two different UTF-8 character sets: utf8 (有时称为 utf8mb3) 是限制在三个字节的非标准版本, 而 utf8mb4 是标准的 UTF-8。 每个字符集都有多种拼写方式,可用的拼写方式和默认的拼写方式可能根据 server version 而有所不同。 这个库初始化一个连接的字符集和拼写方式如下:

  • 如果服务器握手中的排序是 utf8mb4_general_ci 或 utf8mb4_0900_ai_ci, 那么连接就使用该排序,字符集为 utf8mb4;

  • 如果服务器版本至少是 5.5.3,连接使用 utf8mb4_general_ci 排序,字符集为 utf8mb4;否则

  • 连接使用 utf8_general_ci 校对,字符集为 utf8 (utf8mb3)。

这个库以前的版本在每个连接的开始都会发出 SET NAMES utf8 命令。

警告: 如果 client, connection, or result character sets 被改变(例如,使用 SET NAMES) 为 UTF-8 以外的字符集,可能会发生错误或数据损坏。 注意,附加到数据库、表和列的非 UTF-8 字符集不会导致问题; 服务器会自动在用于存储的字符集和用于通信的字符集之间进行转换。

6.7 MySQL CALL 调用存储过程

MySQL CALL 语句只有在最多返回一个结果集且不包含 OUT 或 INOUT 参数时才能执行。

6.8 Cassandra 认证

Cassandra 和 MySQL 一样,支持 authentication plugins 。 本库目前支持的唯一插件是 AllowAllAuthenticator 和 PasswordAuthenticator 。

6.9 SQLite 要求

支持 SQLite 需要适当的本地库。

  • 在Windows上,这个库是 sqlite3.dll 。它包含在 Racket 发行版中。

  • 在 Mac OS 上,这个库是 libsqlite3.0.dylib, 它包含在 Mac OS 10.4 以上的版本中(在 /usr/lib) 。

  • 在 Linux 上,该库是 libsqlite3.so.0 。 它包含在 Debian/Ubuntu 的 libsqlite3-0 包中,以及 Red Hat 的 sqlite 包中。

6.10 基于 FFI 的连接和并发性

Wire-based connections 使用 ports 进行通信,这不会导致其他 Racket 线程阻塞。 相反,FFI 调用会导致所有 Racket 线程阻塞,直到它完成, 所以 FFI-based connections 会降低 Racket 程序的交互性,特别是当使用连接进行长时间的查询时。

这个问题可以通过使用 #:use-place 关键字参数在一个单独的 place 创建基于 FFI 的连接来避免。 这样的连接在查询过程中不会阻塞所有的Racket线程; 缺点是创建和与单独的 place 通信的成本。 在 Racket CS 上,另一个解决方案是在 separate OS thread 中执行查询; 这个解决方案的时间和内存开销可能比独立的地方低。

6.11 ODBC 要求

ODBC 需要适当的驱动管理器本地库,以及你想用 ODBC 连接的每个数据库系统的驱动本地库。

  • 在 Windows 上,驱动管理器是 odbc32.dll, 它自动包含在 Windows 中。

  • 在 Mac OS 上,驱动管理器是 libiodbc.2.dylib (iODBC), 它包含在 Mac OS 10.2 以上的版本中(在 /usr/lib) 。

  • 在Linux上,驱动管理器是 libodbc.so.{2,1} (unixODBC—iODBC 不被支持)。 它可以从Debian/Ubuntu的 unixodbc 包和Red Hat的 unixODBC 包中获得。

此外,你必须安装适当的 ODBC 驱动程序并配置数据源。更多信息请参考特定数据库系统的 ODBC 文档。

6.12 ODBC 状态

ODBC 支持是试验性的。 ODBC 连接的行为可能有很大的不同,这取决于使用的驱动程序,甚至是特定数据源的配置。

6.12.1 DB2 ODBC 驱动

IBM DB2 ODBC 驱动在以下软件配置下进行了测试:

  • Platform: Centos 7.4 on x86_64

  • Database: DB2 Express-C for Linux x64 v11.1

  • Driver: ODBC for DB2 (included with DB2 Express-C)

这个驱动程序似乎需要使用所提供的脚本来设置环境变量 (例如, source /home/db2inst1/sqllib/db2profile)。

已知的问题:

  • 该驱动不支持用于检索 DECIMAL/NUMERIC 字段的标准 SQL_C_NUMERIC 结构。

    修复: 使用 #:quirks '(no-c-numeric) 与 odbc-connect 。

6.12.2 Oracle ODBC 驱动

Oracle ODBC 驱动在以下软件配置下进行了测试:

  • Platform: Centos 7.4 on x86_64

  • Database: Oracle XE 11g (11.2.0)

  • Drivers: Oracle Instant Client ODBC (11.2.0 and 12.2.0)

驱动程序的典型安装需要将 LD_LIBRARY_PATH 环境变量设置为驱动程序安装的 lib 目录 (即包含 libsqora.so 的目录),以便驱动程序能够找到其同级共享库。

已知的问题:

  • 使用 #:strict-parameter-types? #t 选项,参数似乎总是被分配为 varchar 类型。

    修复: 关闭严格参数类型(默认)。

  • 驱动程序不支持参数或结果字段的 SQL_C_BIGINT 格式。因此,传递大的整数作为查询参数可能会失败。

    修复: 使用 #:quirks '(no-c-bigint) 与 odbc-connect 。

  • 一个 TIME 类型的字段会导致驱动程序为 typeid 和 type 参数返回垃圾。 这通常会导致一个错误,信息是 "不支持的类型;typeid: -29936",但 typeid 值是随机的。 (Oracle 似乎没有 TIME 类型, 所以这个错误可能只在一个值被明确地 CAST 为 TIME 时出现–出于某种原因,这不会产生错误。)

  • 试图在连接仍然打开的情况下退出 Racket 可能会导致 Racket 挂起。 具体来说,问题似乎出在驱动的 _fini 函数上。

    修复: 在退出前关闭连接,可以明确地使用 disconnect 或关闭其监护人。

6.12.3 SQL Server ODBC 驱动

Microsoft SQL Server ODBC 驱动在以下软件配置下进行了测试:

  • Platform: Windows 10 on x86_64

  • Database: SQL Server Express 2017

  • Drivers: ODBC Driver 13 for SQL Server, SQL Server Native Client 11.0

已知问题:

  • 如果查询是嵌套的或交错的–即在第一个查询的结果被完全消耗之前执行第二个查询–驱动程序可能会发出错误信号 "连接正忙于另一个命令的结果(SQLSTATE: HY000)"。

    修复: 将 MARS_Connection 数据源选项设置为 Yes (见 this page)。 ODBC 管理器 GUI 没有暴露这个选项,但是可以 by editing the registry 。
