5 启动和声明初始模块 (CS)

正如在 嵌入到程序中 (CS) 中所描述的, Racket CS 的嵌入实例是通过 racket_boot 启动的。 诸如 racket_embedded_load_bytes 等函数有助于用已经编译好的模块来初始化 Racket 命名空间。

对于包含 char* 形式的路径的函数和结构字段,路径在 Windows 上被视为 UTF-8 编码。

5.1 启动和配置

void

 

racket_boot

(

racket_boot_arguments_t* boot_args)

初始化一个 Racket CS 实例。 一个主线程被创建,然后暂停,等待通过 racket_apply, racket_eval 和类似函数的进一步评估。

racket_boot_arguments_t 结构包含了指定 racket_boot 应该如何初始化 Racket 实例的字段。 新的字段可能在未来被添加,但在这种情况下,一个字段的 0 或 NULL 值将意味着向后兼容的默认。

racket_boot_arguments_t 中的字段:

  • const char * boot1_path — 一个包含基本功能的 Chez Scheme 镜像文件的路径。 通常情况下,该文件被称为 "petite.boot" 。 该路径应包含一个目录分隔符,否则 Chez Scheme 将查阅它自己的搜索路径。 racket_get_self_exe_path 和/或 racket_path_replace_filename 函数可能有助于构建路径。

  • long boot1_offset — 读取 boot1_path 中第一个引导图像的偏移量, 这允许引导图像与其他数据结合在一个文件中。 分发的映像是自我终止的,所以不需要大小或结束偏移。

  • long boot1_len — 第一个 Boot Image 的可选长度(以字节为单位), 如果不为零,它将作为加载 Boot 文件的提示。 如果提供了这个提示,它必须至少和 Boot Image 的字节数一样大, 而且它不能长于 Boot Image 偏移后的文件大小。

  • const char * boot2_path — 和 boot1_path 一样,但是用于包含编译器功能的镜像,通常称为 "scheme.boot" 。

  • long boot2_offset — 读取 boot2_path 的偏移量,以获取第二个引导图像。

  • long boot2_len — boot1_len, 一个可选的长度, 以字节为单位,用于第二个引导图像。

  • const char * boot3_path — 与 boot1_path 相似,但用于包含 Racket 功能的镜像,通常称为 "racket.boot" 。

  • long boot3_offset — boot1_len, 在 boot2_path 中读取第三个引导图像的偏移量。

  • long boot3_len — 第三个引导图像的可选长度(字节)。

  • int argc 和 char ** argv — 命令行参数,与独立的 racket 调用的处理方式相同。 如果 argv 是 NULL, 则使用命令行 -n ,它加载引导文件而不采取任何进一步的行动。

  • const char * exec_file — 一个用于 (system-type 'exec-file) 的路径, 通常 argv[0] 使用传递给程序 main 的 argv 。这个字段不能是 NULL 。

  • const char * run_file — 一个用于 (system-type 'run-file) 的路径。 如果该字段为 NULL,则使用 exec_file 的值。

  • const char * collects_dir — 作为主 "collects" 目录使用的路径, 用于定位库集合。 如果这个字段持有 NULL 或 "", 那么图书馆收藏的搜索路径将被初始化为空。

  • const char * config_dir — 一个用作 "etc" 目录的路径, 用于保存配置信息,包括已安装软件包的信息。如果该值为 NULL,则使用 "etc" 。

  • wchar_t * dll_dir — 用于查找 DLL 的路径,例如 iconv 支持。 注意,这个路径使用宽字符,而不是 UTF-8 字节编码。

  • int cs_compiled_subdir — 一个真值表示 use-compiled-file-paths 参数应该被初始化为有一个 "compiled" 的特定平台子目录,这用于 Racket CS 安装时覆盖 Racket BC 安装。

这些函数在初始 Racket 线程中评估 Racket 代码,无论是在内存中的 code 还是从 path 中加载的代码。 其目的是,代码已经被编译了。通常情况下,也包含模块声明。 raco ctool --c-mods 和 raco ctool --mods 命令生成适合用这些函数加载的代码, 而 --c-mods 模式生成的 C 代码会调用 racket_embedded_load_bytes 。

如果 as_predefined 为 true, 那么代码会在创建任何新的 Racket place 时加载到新的地方, 这样代码所声明的模块也会在新的地方加载。

这些函数不是为了在从 Racket 调用的 C 代码中调用。 关于 entry 点与 re-entry 点的讨论,请参见 调用程序 (CS) 。

5.3 启动路径帮助器

char*

 

racket_get_self_exe_path

(

const char* argv0)

返回当前进程的可执行文件的路径。 arg0 参数应该是交付给 main 的可执行文件名,根据操作系统和环境的不同,可能使用也可能不使用。 结果是一个用 malloc 新分配的字符串,它将是一个绝对路径,除非所有寻找绝对路径的尝试都失败。

在 Windows 上, argv0 参数总是被忽略,结果路径是 UTF-8 编码的。

Added in version 8.7.0.11.

char*

 

racket_path_replace_filename

(

const char* path,

 

 

 

 

const char* new_filename)

返回一个类似 path 的路径,但文件名的路径被 new_filename 替换。 new_filename 参数不一定是一个直接的文件名;它可以是以文件名结尾的相对路径。 结果是一个用 malloc 新分配的字符串。

Added in version 8.7.0.11.
